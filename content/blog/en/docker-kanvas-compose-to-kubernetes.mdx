Docker launched Kanvas, a platform that automates the conversion of Docker Compose files into Kubernetes artifacts. It's drawing attention at KubeCon + CloudNativeCon Europe 2026 as a potential challenger to Helm and Kustomize — tools that have dominated Kubernetes configuration management for years.

## The Problem It's Solving

One persistent pain in containerized workflows is the configuration gap between local development and production. You write `docker-compose.yml` locally, but production runs on Kubernetes, which requires Deployments, Services, Ingresses, and more — none of which map cleanly to Compose concepts.

Converting between these formats manually requires Kubernetes knowledge that most application developers don't have. Kanvas automates this:

```yaml
# Your docker-compose.yml
services:
  web:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://db:5432/myapp
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
```

Kanvas reads this and outputs the corresponding Kubernetes manifests.

## How It Compares to Helm and Kustomize

Helm and Kustomize both have mature ecosystems. Kanvas doesn't try to replace them — it occupies a different position:

| | Helm | Kustomize | Kanvas |
|--|------|-----------|--------|
| Starting point | Helm Chart (custom format) | Base YAML | Docker Compose |
| Learning curve | High (template language) | Moderate | Low (if you know Compose) |
| Target user | Kubernetes practitioners | Needs K8s knowledge | Compose users |
| Production track record | Extensive | Extensive | New |

Kanvas's value proposition is specifically for teams that know Compose well but don't have Kubernetes expertise on hand.

## When to Use It

**Good fit:**
- Teams using Compose for local dev and needing a path to Kubernetes
- Frontend or fullstack teams without a dedicated DevOps engineer
- Small-to-medium services where Helm's complexity isn't justified

**Not a good fit:**
- Projects already running Helm in production (migration cost isn't worth it)
- Workloads relying heavily on Kubernetes-native resources (HPA, NetworkPolicy, PodDisruptionBudget)
- Multi-cluster deployments or advanced package management requirements

## What the Auto-Generated Output Looks Like

Kanvas gives you a starting point, not a production-ready manifest. Here's a realistic picture:

```yaml
# Generated Deployment (simplified)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 1  # needs adjustment for production
  selector:
    matchLabels:
      app: web
  template:
    spec:
      containers:
        - name: web
          image: my-app:latest
          # You'll need to add:
          # resources:
          #   requests: { cpu: "100m", memory: "128Mi" }
          #   limits: { cpu: "500m", memory: "512Mi" }
          # livenessProbe: ...
          # readinessProbe: ...
```

Before going to production, you'll need to manually add:
- **Resource limits**: `resources.requests` and `resources.limits`
- **Health checks**: `livenessProbe` and `readinessProbe`
- **Secret management**: Compose environment variables need to move to Kubernetes Secrets or an external secrets manager
- **Persistent storage**: Compose volumes need to become PersistentVolumeClaims

## Context: Cloud Native Growth

According to data shared at KubeCon + CloudNativeCon Europe 2026, the global cloud native developer population has reached 19.9 million. Much of this growth comes from developers who have never directly managed Kubernetes infrastructure.

Kanvas is a strategic move to capture this segment — engineers comfortable with Compose who find Kubernetes intimidating. If Docker can make K8s deployment as approachable as `docker compose up`, it expands its relevance well beyond local development.

Kubernetes 1.35 is also graduating Dynamic Resource Allocation (DRA) binding conditions to beta, improving GPU resource management for AI/ML workloads. There's a reasonable path where Kanvas evolves to simplify AI workload deployment on Kubernetes specifically.

## Takeaway

Kanvas is a well-defined tool for a specific use case: getting from Docker Compose to Kubernetes without deep K8s expertise. It's a bridge, not a replacement for Helm or Kustomize.

For teams considering Kubernetes adoption who are blocked by configuration complexity, Kanvas lowers that barrier meaningfully. Treat the generated output as a starting point, add the production-critical configuration, and you have a workable deployment path without needing a dedicated infrastructure engineer from day one.
