Docker has announced Kanvas, a platform designed to automate the transition from local Docker Compose environments to production-scale Kubernetes deployments. The headline feature: automatic conversion of `docker-compose.yml` into Kubernetes manifests.

This positions Kanvas as a challenger to Helm and Kustomize—tools that have dominated Kubernetes configuration management for years.

## The Core Workflow

The premise is straightforward. You already have a working `docker-compose.yml`. Kanvas reads it and produces the Kubernetes resources you need.

```bash
# Convert Compose to Kubernetes manifests
kanvas convert -f docker-compose.yml -o k8s/

# Review and adjust generated files
ls k8s/
# deployment.yaml  service.yaml  configmap.yaml  secrets.yaml  ...

# Deploy
kubectl apply -f k8s/
```

The appeal is clear to anyone who's spent time manually translating Compose services into Kubernetes Deployments, Services, and StatefulSets.

## How It Handles a Real Example

Take a typical Node.js + PostgreSQL stack:

```yaml
# docker-compose.yml
services:
  app:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data
```

Kanvas converts this to corresponding Kubernetes resources, including:
- `Deployment` for the app
- `StatefulSet` for the database (persistent storage)
- `Service` resources for internal networking
- `Secret` for database credentials (extracted from environment variables)

The automatic extraction of secrets from environment variables into proper Kubernetes `Secret` references is a meaningful quality-of-life improvement over manual conversion.

## Comparison with Helm and Kustomize

**vs. Helm:**
Helm's strength is its chart ecosystem and template flexibility. It can express complex conditional logic, values hierarchies, and release lifecycle management. That power comes with significant complexity—Helm's templating syntax is opaque, and chart authoring requires deep familiarity.

Kanvas takes the opposite philosophy: start from something you already have (Compose) and generate from there. Lower ceiling, much lower floor for getting started.

**vs. Kustomize:**
Kustomize excels at layering patches over existing manifests—managing dev/staging/prod variants without duplication. It doesn't help you create the initial manifests. Kanvas addresses that gap directly.

| Tool | Strength | Limitation |
|------|----------|------------|
| Helm | Rich ecosystem, full lifecycle management | High complexity, steep learning curve |
| Kustomize | Clean overlay strategy for variants | Doesn't generate initial manifests |
| Kanvas | Zero-to-Kubernetes from existing Compose | Early stage, complex configs need tuning |

## What the Generated Output Doesn't Cover

The manifests are a starting point, not a production-ready configuration. Several things will need manual attention:

```yaml
# Kanvas generates this — correctly but incompletely
spec:
  containers:
    - name: app
      image: my-app:latest

# What you need to add before production
      resources:
        requests:          # Affects scheduling decisions
          cpu: 100m
          memory: 256Mi
        limits:
          cpu: 500m
          memory: 512Mi
      livenessProbe:       # How Kubernetes knows the pod is healthy
        httpGet:
          path: /health
          port: 3000
        initialDelaySeconds: 30
        periodSeconds: 10
      readinessProbe:      # When to start sending traffic
        httpGet:
          path: /ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 5
```

NetworkPolicy, RBAC, and pod security settings are absent from generated output. These matter for anything beyond a test environment.

**Secret management needs a separate decision.** Credentials in Compose `environment` blocks get converted to Kubernetes Secrets, but the Secret management strategy—External Secrets Operator, Sealed Secrets, Vault integration—is specific to your infrastructure and not something Kanvas handles.

## When Kanvas Makes Sense

Based on the architecture and stated goals, Kanvas is a good fit for:

- **First-time Kubernetes migration**: Teams moving a Compose-based app to Kubernetes for the first time, without existing Helm charts or Kustomize overlays.
- **Small to medium projects**: Where Helm's overhead isn't justified and you want a clean starting manifest.
- **Rapid prototyping**: Getting a Compose app running in a Kubernetes cluster quickly for validation.

It's less compelling for projects already well-managed with Helm or Kustomize, or large-scale production systems with complex networking requirements.

## Takeaway

Docker Kanvas reduces the friction of moving from Compose to Kubernetes by handling the translation automatically. It's a practical tool for teams with existing Compose definitions who need a Kubernetes starting point without the overhead of writing Helm charts from scratch.

Use it to generate the baseline, then invest time in production hardening: resource requests, health probes, secret management, and network policies. That work is still yours—but you start from a functioning foundation rather than a blank file.
