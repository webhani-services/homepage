Docker released **Kanvas** in early 2026 to address a problem that most development teams know well: local development runs on Docker Compose, but production runs on Kubernetes, and the two require completely different configurations. Kanvas automates the conversion between them, challenging Helm and Kustomize for teams that want a simpler path to Kubernetes.

## The Compose-to-Kubernetes Gap

The friction is real. A typical `docker-compose.yml` for a web service looks like this:

```yaml
version: "3.8"
services:
  api:
    image: myapi:latest
    ports:
      - "4000:4000"
    environment:
      - DATABASE_URL=postgresql://db:5432/myapi
    depends_on:
      - db
  db:
    image: postgres:16
    environment:
      - POSTGRES_DB=myapi
      - POSTGRES_PASSWORD=secret
    volumes:
      - pgdata:/var/lib/postgresql/data
volumes:
  pgdata:
```

Moving this to Kubernetes manually means writing separate Deployment, Service, ConfigMap, Secret, and PersistentVolumeClaim resources — a significant increase in configuration surface area that requires understanding Kubernetes internals before you've shipped anything.

## What Kanvas Generates

Kanvas takes the Compose file and outputs Kubernetes manifests. For the API service above, the generated output looks roughly like:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 1
  selector:
    matchLabels:
      app: api
  template:
    metadata:
      labels:
        app: api
    spec:
      containers:
        - name: api
          image: myapi:latest
          ports:
            - containerPort: 4000
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: api-secrets
                  key: DATABASE_URL
---
apiVersion: v1
kind: Service
metadata:
  name: api
spec:
  selector:
    app: api
  ports:
    - port: 4000
      targetPort: 4000
```

Service dependencies (`depends_on`) are converted to init containers or readiness probes. Volumes become PersistentVolumeClaims. Environment variables flagged as sensitive are moved to Kubernetes Secrets automatically.

## How It Compares to Helm and Kustomize

| Aspect | Helm | Kustomize | Kanvas |
|--------|------|-----------|--------|
| Learning curve | High (templating language) | Medium (overlay config) | Low (reuses Compose knowledge) |
| Starting point | Need to write charts | Need to write overlays | Existing Compose file |
| Complex K8s config | Full flexibility | Full flexibility | Limited, manual adjustment needed |
| Ecosystem maturity | Mature | Mature | Early stage |

Kanvas lowers the floor. If your team already uses Compose for local development, you can bootstrap Kubernetes manifests without learning Helm chart syntax first. The ceiling is lower too — complex multi-environment configurations with custom RBAC, service mesh settings, or advanced network policies will still need Helm or Kustomize.

## Practical Use Cases

**Migrating an existing project to Kubernetes**: Use the generated manifests as a starting point rather than writing from scratch. The output also serves as a learning exercise — it shows how Kubernetes models the same concepts as Compose.

**Standardizing small-to-medium service deployments**: Teams running fewer than 10 services with straightforward networking can often ship with Kanvas-generated manifests plus light customization.

**Rapid environment provisioning**: Build a prototype in Compose, use Kanvas to deploy it to a staging cluster without manual translation.

## What You Still Need to Add

Kanvas output is a starting point, not production-ready configuration. These additions are non-negotiable before going live:

```yaml
# Add to every container spec
resources:
  requests:
    cpu: "100m"
    memory: "128Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
livenessProbe:
  httpGet:
    path: /health
    port: 4000
  initialDelaySeconds: 10
  periodSeconds: 15
readinessProbe:
  httpGet:
    path: /ready
    port: 4000
  initialDelaySeconds: 5
  periodSeconds: 10
```

Also review: NetworkPolicies for inter-service communication restrictions, Ingress configuration for external traffic routing, and HorizontalPodAutoscaler if you need dynamic scaling under load.

## Takeaway

Kanvas doesn't replace Helm or Kustomize for teams running complex Kubernetes environments. What it does well is lower the barrier to entry — teams using Compose can get a working Kubernetes baseline without starting from blank YAML files. Think of it as an on-ramp rather than a destination. The generated output should be reviewed, tested, and extended before it touches production.
