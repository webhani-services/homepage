At KubeCon CloudNativeCon Europe 2026, AWS's Jesse Butler articulated a clear goal: make Kubernetes invisible. The mechanism is EKS Auto Mode, which automates node lifecycle management and reduces the operational surface area that teams have to manage directly.

The pitch isn't new — managed Kubernetes has been around for years — but the scope of what Auto Mode hands off to AWS is broader than previous attempts.

## What EKS Auto Mode Actually Automates

Traditional EKS still requires teams to manage node groups: choosing instance types, configuring scaling policies, applying OS patches, and handling node rotation. Auto Mode removes this:

- **Node provisioning and termination**: AWS selects instance types and manages capacity based on workload demand
- **OS patching**: Nodes are automatically updated to maintain security compliance
- **Cost optimization**: Spot and On-Demand instances are mixed automatically
- **Scaling**: Scale-out and scale-in happen based on Pod resource requests

The node provisioning layer is built on Karpenter, the open-source autoscaler AWS developed as a faster, more flexible replacement for Cluster Autoscaler.

## Enabling Auto Mode

```yaml
# eksctl configuration for EKS Auto Mode
apiVersion: eksctl.io/v1alpha5
kind: ClusterConfig
metadata:
  name: production-cluster
  region: us-east-1
autoMode:
  enabled: true
```

When Auto Mode is enabled, Karpenter is automatically installed and configured. For teams that need finer control, `NodePool` and `EC2NodeClass` resources are still available to constrain behavior.

```yaml
# Customizing node behavior in Auto Mode
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: gpu-workloads
spec:
  template:
    spec:
      requirements:
        - key: karpenter.k8s.aws/instance-family
          operator: In
          values: ["p3", "p4"]
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["on-demand"]  # GPU workloads: no spot
```

## Kro: Higher-Level Resource Abstractions

Alongside Auto Mode, AWS is pushing Kro (Kube Resource Orchestrator) as a way to define higher-level abstractions on top of raw Kubernetes resources.

The problem Kro solves: a "web application" in your organization is really a Deployment + Service + Ingress + HPA + NetworkPolicy + a few ConfigMaps. Creating and maintaining these in sync is repetitive work. Kro lets platform teams define a `ResourceGroup` that bundles these together.

```yaml
apiVersion: kro.run/v1alpha1
kind: ResourceGroup
metadata:
  name: web-app
spec:
  schema:
    spec:
      image: string
      replicas: integer | default=2
      host: string
      port: integer | default=8080
  resources:
    - id: deployment
      template:
        apiVersion: apps/v1
        kind: Deployment
        metadata:
          name: ${schema.metadata.name}
        spec:
          replicas: ${schema.spec.replicas}
          selector:
            matchLabels:
              app: ${schema.metadata.name}
          template:
            spec:
              containers:
                - name: app
                  image: ${schema.spec.image}
                  ports:
                    - containerPort: ${schema.spec.port}
    - id: service
      template:
        apiVersion: v1
        kind: Service
        spec:
          selector:
            app: ${schema.metadata.name}
          ports:
            - port: 80
              targetPort: ${schema.spec.port}
    - id: ingress
      template:
        apiVersion: networking.k8s.io/v1
        kind: Ingress
        spec:
          rules:
            - host: ${schema.spec.host}
              http:
                paths:
                  - path: /
                    pathType: Prefix
                    backend:
                      service:
                        name: ${schema.metadata.name}
                        port:
                          number: 80
```

Application developers then use a simplified API:

```yaml
apiVersion: kro.run/v1alpha1
kind: WebApp  # generated from the ResourceGroup above
metadata:
  name: checkout-service
spec:
  image: "123456789.dkr.ecr.us-east-1.amazonaws.com/checkout:v2.1"
  replicas: 3
  host: "checkout.example.com"
```

## Cedar: Policy as Code for Kubernetes

Cedar is a declarative policy language originally designed for AWS IAM. AWS is now integrating it with Kubernetes to handle access control scenarios that RBAC expresses awkwardly.

RBAC is good at "who can do what to which resource types." Cedar handles richer conditions: time-based restrictions, attribute-based access control, or policies that span multiple resource types in a single rule.

```cedar
// Cedar policy: restrict production changes to business hours
permit(
  principal in Group::"platform-engineers",
  action in [k8s::Action::"update", k8s::Action::"delete"],
  resource in Namespace::"production"
) when {
  context.request.time.hour >= 9 &&
  context.request.time.hour < 18 &&
  context.request.time.dayOfWeek != "Saturday" &&
  context.request.time.dayOfWeek != "Sunday"
};
```

## What This Means for Platform Engineering Teams

The combination of Auto Mode + Kro + Cedar is designed to shift platform engineering work away from infrastructure management and toward building a self-service layer for application developers.

The typical before/after:

**Before**: Platform engineers spend significant time on node group configurations, scaling tuning, OS patch rollouts, and debugging scheduling issues.

**After**: Platform teams define `ResourceGroup` abstractions that application teams consume. Access policies are expressed in Cedar. Node management is largely invisible.

## Caveats Before Migrating

Auto Mode is not a fit for every workload:

- **Custom AMIs**: Auto Mode uses AWS-managed AMIs. If you need specific kernel parameters or custom packages on nodes, Auto Mode won't work.
- **DaemonSet limitations**: Auto Mode nodes have constraints on DaemonSets — check compatibility before migrating.
- **Specific node labels**: Applications relying on specific node labels for scheduling may need adjustments.
- **Full control requirements**: If your SLA or compliance requirements demand precise control over node configuration, managed nodes may still be preferable.

For standard web and API workloads, though, Auto Mode is likely a better default than managing node groups manually.

## Takeaway

EKS Auto Mode, Kro, and Cedar represent AWS's practical answer to Kubernetes complexity. The vision — infrastructure that fades into the background — is directionally right for most teams. The implementation is mature enough to consider for non-production environments now, and production workloads with standard requirements.

If your team is spending disproportionate time on Kubernetes infrastructure instead of application delivery, this is the direction to explore.
