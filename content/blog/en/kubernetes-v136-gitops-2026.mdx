## Kubernetes v1.36 Is Around the Corner

Kubernetes v1.36 is due at the end of April 2026, with a notable number of enhancements and several deprecations and removals. If you're running production clusters, now is the time to review the changelog against your workloads.

But rather than focusing on specific v1.36 features, it's worth zooming out: the more meaningful story in 2026 Kubernetes operations is how GitOps has become the dominant deployment pattern — with ArgoCD leading the way.

## GitOps Is Now the Default

If you're still deploying to Kubernetes with `kubectl apply` in CI/CD pipelines, or pushing directly from a build job to the cluster, you're operating against the direction the industry has settled on.

GitOps is straightforward in principle: **use a Git repository as the single source of truth for cluster state, and continuously reconcile the actual cluster to match what's in Git.**

```
Git Repository (desired state)
         ↓  continuous reconciliation
Kubernetes Cluster (actual state)
```

The practical advantages over imperative deployment approaches:

- **Audit trail**: every infrastructure change is a Git commit with author, timestamp, and diff
- **Rollback**: `git revert` restores any previous infrastructure state — no special tooling needed
- **Drift detection**: the controller continuously detects when actual state diverges from desired state
- **PR-based review**: infrastructure changes go through the same review process as application code

## Why ArgoCD Specifically

FluxCD is a legitimate alternative, but ArgoCD has the broader adoption in 2026. A few concrete reasons:

### Visibility Through the UI

ArgoCD's UI shows the live state of every resource in a deployment — which nodes are Synced, what's OutOfSync, and the full resource tree. When something breaks at 2am, being able to see the current cluster state visually rather than running `kubectl get` chains matters.

### ApplicationSet for Multi-Environment Scale

For teams managing multiple clusters or environments (dev/staging/prod), ApplicationSet generates ArgoCD Applications from a template, reducing the amount of repetitive configuration:

```yaml
apiVersion: argoproj.io/v1alpha1
kind: ApplicationSet
metadata:
  name: my-app
  namespace: argocd
spec:
  generators:
    - list:
        elements:
          - cluster: dev
            url: https://dev.k8s.example.com
          - cluster: staging
            url: https://staging.k8s.example.com
          - cluster: prod
            url: https://prod.k8s.example.com
  template:
    metadata:
      name: "my-app-{{cluster}}"
    spec:
      project: default
      source:
        repoURL: https://github.com/example/k8s-manifests
        targetRevision: HEAD
        path: "apps/my-app/{{cluster}}"
      destination:
        server: "{{url}}"
        namespace: my-app
      syncPolicy:
        automated:
          prune: true
          selfHeal: true
```

One ApplicationSet generates three ArgoCD Applications, each tracking the appropriate path in Git for its environment.

### Multi-Tenancy with Projects

ArgoCD's Project abstraction groups applications by team and enforces access control at that boundary. Multiple teams sharing a single cluster can have isolated views of their own applications without stepping on each other.

## Combining Helm and Kustomize

ArgoCD natively supports both Helm and Kustomize. A common pattern: use Helm charts for the base configuration, and Kustomize overlays for environment-specific differences:

```
k8s-manifests/
├── base/
│   └── my-app/
│       ├── deployment.yaml
│       └── service.yaml
├── overlays/
│   ├── dev/
│   │   └── kustomization.yaml
│   ├── staging/
│   │   └── kustomization.yaml
│   └── prod/
│       └── kustomization.yaml
```

Each overlay applies only the diffs relevant to that environment — replica counts, resource limits, image tags — while the base configuration stays DRY.

## Where the Kubernetes Ecosystem Sits in 2026

Three technologies that were once seen as competing are now understood as complementary:

- **Docker** remains the standard for local development environments
- **Kubernetes** handles production orchestration at scale
- **WebAssembly (Wasm)** is gaining ground at the edge, where full container overhead is impractical

Platform Engineering has also matured: building internal developer platforms (IDPs) that abstract Kubernetes complexity from application developers is now a recognized discipline, with tools like Backstage and Humanitec covering this space.

## Starting With GitOps

If you're migrating from direct CI/CD deployment to GitOps, these steps reduce the risk:

1. **Start with non-production**: pilot ArgoCD on dev/staging before touching production clusters
2. **Manual sync first**: enable automated sync only after you've built confidence in the Git repository state
3. **Establish the PR workflow first**: agree on the review process for infrastructure changes before automating sync

GitOps adoption doesn't require a big-bang migration. Start with one application in one environment, run it long enough to understand how ArgoCD behaves in your organization, then expand from there.
