Next.js 16.2 shipped on March 18, 2026. It's the second release since Turbopack became the default bundler, and the headline numbers are significant: 87% faster startup compared to 16.1, 400-900% improvement in compile times, and 67-100% improvement in server component refresh. This post breaks down what's actually new and what it means day-to-day.

## The Performance Numbers in Context

These gains are relative to Next.js 16.1 — not webpack, and not the pre-Turbopack baseline. Turbopack became the stable default in Next.js 16.0, so 16.2 builds on an already-fast foundation.

**Startup time** (87% faster than 16.1) is the time from running `next dev` to seeing the first ready prompt. On the default application, this drops from measurable seconds to near-instant.

**Compile time** (400-900% faster) refers to the initial compilation of your route tree. The range exists because gains depend on your project size and module graph complexity. Larger projects with many dynamic imports tend to see proportionally bigger wins.

**Server refresh** (67-100% improvement) means the time from saving a Server Component file to seeing the update reflected in your browser. Server Fast Refresh — new in 16.2 — makes this fine-grained: only the changed server subtree re-renders, rather than triggering a full server-side recompilation.

## Server Fast Refresh

Client-side Fast Refresh has existed in Next.js for years. Server Fast Refresh extends the same concept to Server Components. When you save a server file, only the affected component subtree is invalidated and re-fetched, not your entire page.

The practical impact: iterating on deeply nested Server Components — fetching data, modifying server-side logic — now feels similar to editing a client component.

## WASM Worker Support

Web Worker Origin is a new Turbopack feature that improves support for WASM libraries running inside Web Workers. Previously, WASM files used in workers could fail to load correctly due to cross-origin restrictions in certain browser environments. The fix makes Worker + WASM combinations more reliable without requiring manual workarounds.

If your project doesn't use WASM in workers, this doesn't affect you. If it does, the worker origin fix likely removes a class of hard-to-debug loading failures.

## Tree Shaking for Dynamic Imports

```typescript
// Before: all exports were bundled even if only one was used
const { HeavyComponent } = await import('./heavy-module')

// After: unused exports are removed from the chunk
const { LightComponent } = await import('./shared-module')
```

This change removes unused exports from dynamically imported modules. For route segments with large shared dependencies, this can meaningfully reduce chunk sizes. The effect is most pronounced in modules that export many items but only a few are used at the import site.

## TypeScript PostCSS Config

A small but useful ergonomics improvement: `postcss.config.ts` is now supported alongside `postcss.config.js`. If you're already running a TypeScript-first codebase, this removes one configuration file from the list of things that must stay in plain JavaScript.

## React Server Components Payload Performance

A React core contribution from the Next.js team: Server Component payload deserialization is up to 350% faster. This is a React-level change that benefits all frameworks using RSC, not just Next.js. Practically, it means time-to-interactive improves on pages with heavy Server Component data payloads — especially relevant for data-heavy dashboards or content-heavy routes.

## Upgrading

The 16.2 upgrade is low-friction for most projects. The changelog lists over 200 bug fixes alongside the features above. Before upgrading, the main things worth checking:

1. Review any custom Webpack configurations — some escape hatches may behave differently under Turbopack
2. Test WASM workers if you use them — the Worker Origin fix changes loading behavior
3. Verify dynamic import code paths for tree shaking correctness in your specific module graph

The official upgrade guide at `nextjs.org/docs/app/guides/upgrading/version-16` covers breaking changes.

## Takeaway

16.2 is a maintenance-and-performance release in the best sense: the core feature set is stable, and the team is using that stability to push compile speed significantly forward. For teams doing frequent iteration on Server Components, the Server Fast Refresh improvement alone justifies upgrading.
