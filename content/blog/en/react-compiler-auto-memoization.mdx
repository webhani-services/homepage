React Compiler v1.0 shipped in October 2025. By early 2026, it's becoming the baseline for new projects using React 19. The practical effect: `useMemo`, `useCallback`, and `React.memo` — the manual performance toolkit that has been a rite of passage for React developers for years — are increasingly being treated as legacy optimization patterns.

This post examines how the compiler works, what you can safely remove from an existing codebase, and the situations where manual intervention still makes sense.

## What the Compiler Does

The React Compiler is a Babel/SWC plugin that runs at build time. It statically analyzes your components and hooks, identifies values that are safe to memoize automatically, and inserts the equivalent of `useMemo`/`useCallback` calls in the compiled output — without any developer annotation.

The compiler reasons about:
- Pure computations that only depend on props and state
- Stable function references that are passed as event handlers or dependencies
- Component subtrees that don't need to re-render when unrelated state changes

The key insight is that the compiler can make these determinations accurately for code that follows React's rules (pure rendering, no side effects in render). It doesn't require a type system or annotations — it infers from the structure of your code.

## Before and After

A common pattern before React Compiler:

```typescript
function ProductList({ products, onSelect }: Props) {
  const sortedProducts = useMemo(
    () => [...products].sort((a, b) => a.name.localeCompare(b.name)),
    [products]
  );

  const handleSelect = useCallback(
    (id: string) => onSelect(id),
    [onSelect]
  );

  return (
    <ul>
      {sortedProducts.map((p) => (
        <ProductItem key={p.id} product={p} onSelect={handleSelect} />
      ))}
    </ul>
  );
}
```

With React Compiler enabled, the compiled output handles the memoization. The source becomes:

```typescript
function ProductList({ products, onSelect }: Props) {
  const sortedProducts = [...products].sort((a, b) =>
    a.name.localeCompare(b.name)
  );

  return (
    <ul>
      {sortedProducts.map((p) => (
        <ProductItem key={p.id} product={p} onSelect={onSelect} />
      ))}
    </ul>
  );
}
```

Same performance characteristics. Significantly less noise in the source.

## Setting Up React Compiler

```bash
npm install --save-dev babel-plugin-react-compiler
```

```json
// babel.config.json
{
  "plugins": ["babel-plugin-react-compiler"]
}
```

For Next.js 15+, the integration is built-in via `next.config.ts`:

```typescript
import type { NextConfig } from "next";

const config: NextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default config;
```

The compiler runs in "opt-in" mode by default, meaning it only optimizes components it can verify are safe. Components that violate React's rules are skipped without breaking your app.

## What to Remove from an Existing Codebase

When introducing the compiler to an existing codebase, a reasonable approach is:

1. Enable the compiler and run your test suite — confirm nothing breaks
2. Use the React Compiler's ESLint plugin to identify unnecessary manual memoization:

```bash
npm install --save-dev eslint-plugin-react-compiler
```

3. Remove `useMemo` wrapping pure computations that only depend on props/state
4. Remove `useCallback` on handlers passed to child components where those children are also compiled
5. Remove `React.memo` from components that the compiler now handles automatically

Keep manual memoization for:
- External library integration where values need referential stability (e.g., complex chart configurations)
- Expensive computations that are deliberately deferred (like heavy data transformations triggered on demand)
- Any component explicitly opted out with the `"use no memo"` directive

## Opting Out When Needed

The compiler provides an escape hatch for components with intentionally impure behavior:

```typescript
function LegacyWidget({ data }: Props) {
  "use no memo";
  // This component will not be optimized by the compiler.
  // Use for third-party integrations with unusual referential requirements.
}
```

## Practical Impact

The bigger win from removing manual memoization isn't raw performance — it's readability and reduced cognitive overhead. A component with five `useMemo` and three `useCallback` calls forces readers to mentally track dependency arrays and reason about when each value recomputes.

With the compiler handling this, component code returns to expressing what it renders rather than how to optimize it. That's a meaningful reduction in the surface area for bugs.

## Takeaways

- React Compiler v1.0 is production-ready and the default path for new React 19 projects
- It handles memoization automatically for components that follow React's rules
- `useMemo`, `useCallback`, and `React.memo` are still valid for edge cases — but shouldn't be the default
- Introduce the compiler to existing codebases incrementally via the ESLint plugin and test suite validation
- The primary benefit is cleaner, more readable code — performance is maintained, not sacrificed
