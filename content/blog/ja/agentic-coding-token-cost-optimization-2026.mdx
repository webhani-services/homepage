AI コーディングエージェントが実務の道具として定着する一方で、無視できなくなってきたのが Token コストです。業界の試算では、Agentic なワークフローは通常のチャット利用と比べて桁違いの Token を消費し、一つのタスクで数十万〜数百万 Token の入力が API に流れることも珍しくありません。開発者一人あたり月 150〜250 ドル規模、というコスト感の報告も出ています。本記事では、品質を犠牲にせずにコストを下げるための、四つの実務的なレバーを整理します。

## レバー 1: Prompt Caching を効かせる

コスト分析でしばしば指摘されるのが、「請求の大半は再送されるコンテキスト」という事実です。ある監査では、繰り返し送られるコンテキストが請求の 62% を占めていたと報告されています。エージェントは会話のたびに System Prompt やプロジェクト情報を送り直すため、ここが最大の削減対象になります。

Claude の API は、繰り返し使われるプレフィックス（System Prompt や長い前提情報）を Cache し、再送時のコストを大きく下げる Prompt Caching を備えています。Claude Code はこれを自動で活用しますが、自前でエージェントを組む場合は、Cache が効く形にリクエストを設計することが重要です。ポイントは、変わらない部分を前に、変わる部分を後ろに置くことです。

```javascript
// 変わらない前提は前方に固定し、Cache のヒット率を上げる
const messages = [
  { role: "system", content: STABLE_SYSTEM_PROMPT }, // Cache 対象
  { role: "user", content: STABLE_PROJECT_CONTEXT },  // Cache 対象
  { role: "user", content: perRequestQuestion },       // 毎回変わる
];
```

前提情報をリクエストごとに組み替えてしまうと、Cache のプレフィックスが崩れてヒットしなくなります。「安定したものは動かさない」という設計だけで、コストは目に見えて下がります。

## レバー 2: モデル階層をルーティングする

すべての作業に最上位モデルを使う必要はありません。定型的な整形、単純な検索、ボイラープレートの生成といった「頭を使わない作業」には軽量モデルを、設計判断や複雑なデバッグといった「難しい推論」には上位モデルを割り当てる。この使い分けがコストに直結します。

考え方はシンプルで、タスクを難易度で振り分けるルーターを一枚挟むことです。

```javascript
// タスクの性質でモデル階層を振り分ける
function pickModel(task) {
  if (task.kind === "format" || task.kind === "lookup") {
    return "haiku"; // 定型作業は軽量モデル
  }
  if (task.kind === "architecture" || task.kind === "debug") {
    return "opus"; // 難しい推論は上位モデル
  }
  return "sonnet"; // 中間はバランス型
}
```

重要なのは、「上位モデルを使わない」ことではなく、「上位モデルを使うべき場面に集中させる」ことです。安いモデルで済む作業に上位モデルを使い続けるのは、単なる浪費です。

## レバー 3: Context を刈り込む

エージェントは、長いタスクを進めるほど会話履歴が積み上がり、それを毎回送り直すことで Token を消費します。Claude Code には、コンテキストが上限に近づくと履歴を要約する Auto-compaction が備わっていますが、そもそも渡す情報を絞ることのほうが効果的です。

特に効くのが、プロジェクトの前提を記述する `CLAUDE.md` のような設定ファイルの見直しです。ある比較では、3,847 Token あった設定ファイルを、「コードから読み取れない情報だけ」に絞って 312 Token まで削ったところ、約 92% のコンテキスト削減を達成しつつ品質の劣化はなかった、と報告されています。

原則は「モデルがコードから推測できることは書かない」です。ディレクトリ構成やファイル名の規約など、リポジトリを見れば分かることを冗長に書き連ねるより、「なぜその設計なのか」「何を避けるべきか」といった、コードからは読み取れない情報に絞るほうが、コストと品質の両面で有利になります。

もう一つ見落とされがちなのが、Web ページや外部ドキュメントの取り込みです。生の HTML は一ページで数万 Token に達することがあり、それをそのまま渡すと全量に課金されます。本文だけを抽出してから渡す前処理を挟むだけで、無駄な Token を大きく削れます。

## レバー 4: 予算上限を設ける

最後は、暴走を止める仕組みです。エージェントは自律的に動くぶん、想定外のループや過剰な探索でコストが跳ね上がることがあります。ユーザー単位・タスク単位で予算上限を設け、それを超えたら止める、あるいは人間の確認を挟む設計にしておくと、最悪のケースを避けられます。

これは技術的な工夫というより、運用のガードレールです。上限に達したら通知する、上限の 80% で警告する、といった段階的な仕組みを用意しておくと、コストの「見える化」も同時に進みます。

## webhani の視点

これら四つのレバー — Prompt Caching、モデルのルーティング、Context の刈り込み、予算上限 — は、いずれも派手な施策ではありません。しかし、複数の監査で「二週間以内にエージェントコストを 50〜70% 削減した」レバーとして繰り返し挙がっているのがこの組み合わせです。

当社が顧客の AI 活用を支援する際に強調しているのは、「使う量を減らす」のではなく「無駄な量を減らす」という発想です。AI コーディングの価値は生産性にあり、それをコスト懸念で萎縮させるのは本末転倒です。上の四点を仕組みとして組み込めば、生産性を保ったままコストだけを削れます。まずは自分たちの請求のうち、どれだけが「再送コンテキスト」に消えているかを測ることから始めてみてください。
