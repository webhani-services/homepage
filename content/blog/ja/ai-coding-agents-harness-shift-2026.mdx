## 転機が来た

ここ数年、AI を使ったコーディングアシスタントは一つの形に落ち着いていました。エディタの背景で model が動き、型を打つと次の行を提案する。受け入れるか拒否するか。すべてはリアルタイム、ローカルなコンテキストの中で完結していました。

その時代は終わりました。

2026年の中盤、すべての主要な AI ラボが二つのものを配布しています。model と、その model を制御する*エージェントハーネス*です。このハーネスは、スコープを決めたタスクを AI システムに任せ、どのツールにアクセスを許可するか管理し、数時間から数日かけて実行させ、出力をレビューして反復するための runtime 環境です。これは xAI の Grok Build (7月8日ローンチ)、Claude Code の7月のアップデート、そして OpenAI、Google など各社に共通するパターンとして表れています。AI によるコーディングはもはや「インラインの提案」ではなく「ワークフロー」になっています。

これは段階的な改善ではありません。思考のモデルそのものが変わっています。「アシスタントがコーディングを速くする手助けをする」から「スコープが決まったタスクをアシスタントに委譲し、出力をレビューして、ツールへのアクセスを認可・拒否し、反復する」への変化です。

## エージェントハーネスの実装

エージェントハーネスは、四つのプリミティブで構成されます。

**1. 永続的なコンテキストとセッション状態。** エディタを閉じたら忘れてしまうオートコンプリートと異なり、エージェントハーネスは会話履歴、タスク履歴、実行結果を restart 後も保持します。数日後にタスクに戻ってきても、エージェントは完全なコンテキストを持っています。

**2. ツールアクセスと permission ゲート。** エージェントはコードを提案するだけではなく、ファイルを読み書きし、コマンドを実行し、API を呼び出し、コードベースを検査できます。ただし、明示的に許可した操作だけに限ります。permission matrix を制御します。「ソースファイルは読めるが secrets は読めない」「変更を提案できるがテストだけ」「build コマンドは実行できるが deploy はできない」といった具合です。

**3. 承認ワークフロー。** 重要な操作はゲートを通します。エージェントが production ファイルを変更したり、deploy を実行したり、コードを削除する前に、あなたのレビューを待ちます。これはオートコンプリートが暴走することではなく、junior エンジニアが commit 前に許可を求めるのと同じです。

**4. 並列実行とブランチング。** 多くのハーネスでは、同じタスクに複数のワーカーをスピンアップし、独立した調査を並列に実行し、会話を任意の点でブランチできます。これにより throughput が増加しても、human-in-the-loop は保たれます。

技術的には、エージェントハーネスは sandbox (ふつうは container)、ツール実行を制御するエンジン、audit trail、そして operation ごとに permission を認可・拒否・修正する UI を提供します。

## 「オートコンプリートの進化版」ではない理由

現実的なワークフローを考えてみましょう。月曜の朝、microservice に新しい API endpoint を追加し、テストを書き、ドキュメントを更新して、PR のドラフトを出す必要があります。

**旧モデル(オートコンプリート):** ファイルを開く、エージェントが completion を提案する、型を打つ、提案を受け入れる。各ファイルで繰り返し。エージェントにはテストを書く必要があることや、ドキュメント更新が必要なことを知る方法がありません。すべての coordination を自分で行う。提案は「次に来るもの」の best-effort 推測に過ぎません。

**新モデル(エージェントハーネス):** タスクを説明します。「/health endpoint を追加してください。これは service status を返し、database connectivity をチェックします。テストも書いてください。README も更新してください。」エージェントは以下を実行します。
- あなたの API コードを読んで pattern を理解する。
- 既存のテストを確認してスタイルを合わせる。
- endpoint の実装を提案する。
- あなたが提案をレビュー。コード書き込みの permission を付与。
- エージェントは endpoint、テストファイル、README を書く。
- テストを実行して通過を確認。
- draft PR を作成してレビューを待つ。

あなたは各 gate で制御を保っています。エージェントは毎回 permission を求め、あなたは commit 前に結果を見ました。また、context switching を減らしてもっと多くをこなしました。

違いは速度ではなく、*スコープ*と*ガードレール内での自律性*です。オートコンプリートは line-by-line。エージェントハーネスは task-by-task。

## 実践的な採用：小さく始めて permission を段階的に付与する

エージェントハーネスの採用時の誘惑は、すべての permission を付与してエージェントの動作を観察することです。これは間違いです。client チームがこれをやって後悔するのを見ました。

より安全なパスはこれです。

**Phase 1: read-only 調査。** エージェントにコードベースを読ませ、構造を理解させ、質問させ、変更を提案させます。ただし、何も書かせません。これは調査フェーズです。エージェントがあなたのプロジェクトについて何を理解していて、何を誤解しているかを学びます。

**Phase 2: sandbox での書き込み。** 次に、コードを書かせますが、新しいファイルか branch、test directory に限ります。test suite を実行して出力を検証します。production code にはまだ触らせません。

**Phase 3: feature branch への gated write。** エージェントの出力を信頼できるようになったら、新しい機能用の feature branch への書き込みを許可します。すべての書き込みには依然としてあなたの承認が必要です。すべての変更は通常の PR review process を通ります。

**Phase 4: スコープ付き production 更新。** エージェントが信頼性を示したら、狭いゲートを開く。例えば「documentation は更新できる」または「既存の function のテストは書ける」ですが、それ以外はできない、といった具合です。広い write access は付与しません。

**Phase 5: 監視された autonomous 作業。** 多くの成功サイクルの後に初めて、よく scope が決まったタスクではエージェントを attended なしで動作させることを検討できます。この場合でも、audit trail を保持し、時間制限を設定します(例：「このタスクは1時間で完了すべき」)。

## Governance とレビュー

AI エージェントに permission を付与することは「設定して忘れる」ではありません。あなたのチームはいくつかのポリシーが必要です。

- **承認段階。** どの操作が human review を必要とするか決めます。例：読み込みと変更提案は無料；branch への書き込みは human 承認が必要；production への deploy は2人の human が必要。これを文書化します。
- **Audit trail。** エージェントハーネスはすべてのアクション、使用した permission、起きたことをログに記録すべきです。定期的にこれらのログをレビューします。
- **Model と capability の制限。** 異なるタスクは異なる model を必要とするかもしれません。軽量で高速な model は refactoring に十分です。より強い model は architectural 変更に必要かもしれません。これを明示的にします。
- **Rollback procedure。** エージェントが間違いを犯したら、それを undo する明確な方法が必要です。branch を revert する、ファイルを復元する、deploy をキャンセルする。必要になる前に練習します。
- **Escalation paths。** いくつかの決定はエージェントに委譲すべきではありません。エージェントはこれらを認識して help を求めるべきです(例：「この変更は database schema を backward-incompatible な方法で変更します。human input が必要です」)。

## チーム向けチェックリスト

エージェントハーネスを採用する backend チームのための permission tier matrix の具体例です。

| 操作 | Permission Level | Review が必要 |
|------|------------------|-------------|
| ソースファイルの読み取り | 常に許可 | いいえ |
| 設定ファイルの読み取り | 常に許可 | いいえ |
| 変更提案(書き込みなし) | 常に許可 | いいえ |
| テストファイルへの書き込み | 許可 | いいえ |
| ドキュメントへの書き込み | 許可 | いいえ |
| feature branch への書き込み | 許可 | はい(push 前) |
| CI/CD config の修正 | 拒否 | — |
| staging への deploy | 拒否 | — |
| production への deploy | 拒否 | — |
| secrets / credentials へのアクセス | 拒否 | — |

ほとんどが「Denied」と review "No" の状態から始めます。数週間かけてエージェント出力の品質と信頼性を観察してから拡張します。

## Webhani の視点

Webhani は2024年初頭から AI/LLM 採用を client にアドバイスしてきました。エージェントハーネスで繰り返し発生している3つの間違いを見ています。

1. **早すぎるうちに permission を多く付与しすぎる。** チームは「AI magic」を見たくてすべてのゲートを開きます。エージェントが間違いを犯したら(それは必ず起きます)、回復は大変です。狭く始めましょう。

2. **レビュー loop をスキップする。** チームは model が作成したからという理由だけで、エージェント生成コードを信頼できるものとして扱います。私たちの推奨：junior エンジニアのコードのようにエージェント出力をレビューしてください。最初はしっかりレビューします。信頼を段階的に構築します。

3. **実際のメリットを測定しない。** AI エージェントをワークフローに追加するには overhead が生じます。あなたはまだ変更をレビューし、ハーネスをメンテナンスし、permission を最新に保ちます。計測してください：時間を節約していますか？バグを減らしていますか？あなたのチームをより high-leverage な仕事に解放していますか？1ヶ月後に答えが不明確なら、scope を再評価してください。

client 企業での最も成功した採用は次のパターンに従っています。一つの小さく、よく scope が決まったタスクを特定し(例：「このモジュールのテストを書く」または「API ドキュメントを自動生成する」)、上記の段階的 permission model を通して実行し、時間節約と品質への影響を計測して、次のタスクに拡張します。「初日から完全な自律性」がうまくいくことはほぼありません。

## 展望

エージェントハーネスは終点ではありません。ツールはより smart になり、model は改善し、UX は成熟します。ただし、根本的なシフト——オートコンプリートから委譲された、permission-gated なタスク実行へ——は業界に組み込まれています。

採用を検討中のチームへのアドバイスはシンプルです。小さく始め、信頼するが検証し、計測してください。エージェントハーネスは strong ですが、その power は human を loop に保ち、permission について慎重に考えるときにだけ安全です。
