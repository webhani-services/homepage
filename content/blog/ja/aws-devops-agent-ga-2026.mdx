AWS が DevOps Agent の一般提供（GA）を開始しました。生成 AI を活用してインシデントの調査・分析・対応を自動化するサービスです。CloudWatch アラームや PagerDuty、ServiceNow などのイベントをトリガーに、人間が関与する前から調査を開始します。同時期に Security Agent の on-demand ペネトレーションテストも GA となっており、AWS は運用の AI 化を本格的に推進しています。

## DevOps Agent の動作原理

DevOps Agent は、インシデントイベントを受け取ると以下のような流れで自律的に調査を進めます。

1. **イベント受信**: CloudWatch Alarm、PagerDuty アラート、Dynatrace Problem、ServiceNow チケット、または Webhook 経由のカスタムイベント
2. **コンテキスト収集**: 関連するログ、メトリクス、デプロイ履歴、設定変更を自動収集
3. **根本原因分析**: 収集データをもとに原因仮説を生成・検証
4. **レポート生成**: 調査結果、影響範囲、推奨アクションをまとめたレポートを出力
5. **自動アクション（オプション）**: 承認フローを経て、スケールアウトや設定ロールバックなどの対応を実行

重要なのは、このプロセスがオンコール担当者が状況を把握する前から始まることです。担当者がアラートに気づいた時点で、既に初期調査が完了しているという運用が実現します。

## 実際の活用シナリオ

### デプロイ後の異常検知

新しいデプロイ直後にエラーレートが上昇した場合、DevOps Agent は自動的に直前のデプロイ差分、変更されたサービスの依存関係、影響を受けているエンドポイントを特定します。オンコール担当者は「何が起きているか」を調べる時間を省き、「どう対応するか」に集中できます。

### 定期的な異常パターンの発見

特定の時間帯に発生するパフォーマンス劣化など、散発的なインシデントの共通パターンも検出できます。単一インシデントの調査では見落としがちな傾向を、蓄積されたインシデントデータから浮き上がらせます。

### Runbook との連携

既存の Runbook を参照しながら対応ステップを提案する機能も備わっています。組織固有のインシデント対応手順を AI に学習させることで、経験の浅いメンバーでも適切な初動対応が取れるようになります。

## 導入前に確認すべきポイント

便利な機能ですが、導入前に整理しておくべき点があります。

### データアクセス権限の設計

DevOps Agent は調査のためにログ、メトリクス、設定情報へのアクセスが必要です。最小権限の原則を守りながら、必要なリソースへのアクセスを適切にスコープする IAM 設計が重要です。

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:GetLogEvents",
        "logs:DescribeLogGroups",
        "cloudwatch:GetMetricData",
        "cloudwatch:DescribeAlarms"
      ],
      "Resource": [
        "arn:aws:logs:ap-northeast-1:ACCOUNT_ID:log-group:/aws/lambda/production-*",
        "arn:aws:cloudwatch:ap-northeast-1:ACCOUNT_ID:alarm:production-*"
      ]
    }
  ]
}
```

本番環境のログには機密情報が含まれる場合があります。DevOps Agent がアクセスするリソースの範囲と、そのデータがどのように処理されるかを事前に確認してください。

### 自動アクションの承認フロー

DevOps Agent が自動的にインフラ変更を実行できる設定にする場合は、承認ワークフローを必ず設けることを強く推奨します。調査レポートの生成は自動化しつつ、実際のアクション実行には人間の承認を挟む「Human-in-the-loop」構成がリスクを抑えます。

### アラートの質が結果を左右する

DevOps Agent の調査品質は、入力となるアラートとログの質に依存します。ノイズの多いアラートや不十分なログ設計では、有用な洞察を得られません。GA 対応に合わせて、既存のオブザーバビリティ設計を見直す機会にするのが良いでしょう。

## Security Agent との組み合わせ

同時に GA となった Security Agent は、アプリケーションの設計・コード・ランタイム動作を継続的に分析し、オンデマンドでペネトレーションテストを実行します。DevOps Agent がインシデント対応の自動化を担うのに対し、Security Agent は予防的なセキュリティ検証を担います。

両エージェントを組み合わせることで、「検出 → 調査 → 対応」のサイクルと「設計 → 検証 → 修正」のサイクルを両方自動化できます。

## まとめ

AWS DevOps Agent の GA は、インシデント対応の時間軸を大きく変える可能性を持ちます。人間が気づく前から調査が始まり、担当者が現れたときには状況が整理されている — この運用モデルは、特にオンコール負荷が高いチームにとって価値があります。

導入にあたっては、権限設計とデータアクセスのスコープ、自動アクションの承認フロー、既存のオブザーバビリティ品質の三点を事前に整理することが成功の鍵です。webhani では AWS 環境の運用自動化設計についての相談も受け付けています。
