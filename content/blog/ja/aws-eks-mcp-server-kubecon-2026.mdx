## KubeCon EU 2026 での AWS 発表

2026年3月23〜26日、アムステルダムで開催された KubeCon EU 2026 で AWS が注目すべき発表を行った。EKS MCP Server（プレビュー）と AWS DevOps Agent だ。

この2つを組み合わせると何ができるか端的に言えば、「AIアシスタントに自然言語で話しかけて Amazon EKS クラスターを操作する」ことが実用レベルで可能になる。

## EKS MCP Server とは

MCP（Model Context Protocol）は、AIエージェントが外部ツールやAPIと連携するための標準プロトコルだ。EKS MCP Server はこの仕組みを利用して、Claude や GPT などのAIアシスタントが Amazon EKS クラスターと直接対話できるようにする。

```bash
# EKS MCP Serverのセットアップ（プレビュー版）
aws eks install-mcp-server --cluster-name my-cluster --region ap-northeast-1

# AIアシスタントへの自然言語クエリ例
# "本番Namespaceで今日3回以上再起動されたPodを教えて"
# "api-service の Deployment を4レプリカに変更して"
# "backendネームスペース全体のリソース要求と実使用量を比較して"
```

従来、こうした操作には `kubectl` コマンドの知識が必要だった。MCP Server を介することで、コマンドの詳細を知らなくてもクラスターの状態確認や基本的な操作ができる。

## AWS DevOps Agent

EKS MCP Server と組み合わせて発表された AWS DevOps Agent は、Kubernetes 運用に特化したAIエージェントだ。単純なクエリ応答に留まらず、クラスターの状態を継続的に監視して異常を検知し、根本原因の調査と修正案の提案までを行う。

具体的なユースケース：

- **インシデント調査**：PodのOOMKillやサービス障害が発生した際、ログ・イベント・メトリクスを自動収集して分析し、最近のデプロイとの関連を特定する
- **キャパシティプランニング**：リソース使用傾向を分析して、週末のトラフィック急増前にスケールアップのタイミングを提案する
- **コスト最適化**：使用されていないリソースや過剰プロビジョニングを検出し、修正手順とともに報告する

## セキュリティの設計

AIエージェントにクラスター操作を委ねる際のセキュリティは重要な関心事だ。AWSのアプローチでは、操作の範囲を IAM ポリシーで明示的に制限し、すべての操作を CloudTrail でログに残す設計になっている。

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "eks:DescribeCluster",
        "eks:ListNodegroups",
        "eks:DescribeNodegroup",
        "eks:ListPodIdentityAssociations"
      ],
      "Resource": "arn:aws:eks:*:*:cluster/production-*"
    }
  ]
}
```

実用的なスタート地点は、AIアシスタントに読み取り専用の権限を与え、書き込み操作は明示的な人間の承認を必要とする設計だ。エージェントを「調査・提案」に活用し、「適用」のステップは自分のクラスター設定でエージェントの判断への信頼が積み上がるまで人間の手に委ねる。

## Docker Kanvas との違い

2026年初頭に発表された Docker Kanvas は、Docker Compose ファイルを Kubernetes マニフェストに自動変換するツールだ。Helm や Kustomize の複雑さを避けて Kubernetes に入門したいチームに向いている。

EKS MCP Server が既存の Kubernetes 運用の効率化を目指すのに対し、Docker Kanvas は Kubernetes への移行を簡単にするアプローチだ。目的が異なるため競合ではなく補完的な関係にある。

## この流れが示すもの

標準化されたプロトコル（MCP）を通じてAIエージェントが実際のインフラを操作するパターンは、2026年の大きなトレンドの一つだ。EKS MCP Server はそのわかりやすい実例で、同じアプローチがクラウドプロバイダーやインフラカテゴリ全体に広がりつつある。

インフラエンジニアの役割も変化する。`kubectl` コマンドを書いてデバッグする時間が減り、AIエージェントに何を許可するかを定義し、提案された変更をレビューすることに時間を使うようになる。

## 始め方

EKS MCP Server は現時点でプレビュー段階だ。AWS のプレビュープログラムを通じて申し込み、本番以外のクラスターから始めて、最初は読み取り専用の権限に制限するのが安全だ。特定のクラスター構成でエージェントがどう推論するかを理解しながら、段階的に権限を拡張していくアプローチを推奨する。
