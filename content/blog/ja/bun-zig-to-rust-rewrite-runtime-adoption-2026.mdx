## 実装言語を途中で変える、ということ

2026年7月8日、Bun の作者である Jarred Sumner が、Bun を Zig から Rust へ書き直している理由と進め方を詳しく公開しました。同じ週に v1.3.14 も出ています。92件の issue 修正に加えて、画像処理 API の `Bun.Image`、グローバルストアを持つ isolated linker による warm install の高速化（公称でおよそ7倍）、`fetch()` 向けの実験的な HTTP/2・HTTP/3 クライアント、`Bun.serve()` の HTTP/3（QUIC）対応、Linux と macOS の `fs.watch()` 書き直し、そして FreeBSD と Android のビルド。リリースノートとしては十分に厚い内容です。

ただ、この2つのニュースは別々に読むべきものではありません。すでに本番で使われているランタイムが、実装言語を途中で入れ替えると宣言した。これはリリースノートの1行ではなく、プロジェクトの構造に関するシグナルです。慌てて乗り換える理由にも、慌てて捨てる理由にもなりません。読み方を間違えないことが重要です。

## 書き直しが買うもの、支払うもの

まず、Zig を選んだこと自体は当時として不合理ではありませんでした。C との相互運用が素直で、コンパイル時実行が強力で、ランタイムの低レイヤを書くには実際に書きやすい言語です。JavaScriptCore を抱え込んで独自のアロケータやイベントループを組む、という Bun の設計と噛み合っていました。

一方で、Zig はまだ 1.0 に到達していません。言語仕様と標準ライブラリが版ごとに動き、その追従コストはプロダクトの機能開発とは無関係に発生し続けます。エコシステムも小さい。TLS、圧縮、HTTP/2、QUIC、画像処理 — こうした「枯れた実装を借りてきたい」領域で、借りられる先が少ないのです。採用面でも、Zig を書けてランタイム内部を触れる人材は世界的に薄い層です。

Rust に移せば、この裏返しがそのまま利点になります。メモリ安全性がコンパイラの保証として手に入り、crates.io には TLS も QUIC も画像コーデックも実戦投入済みの実装が揃っています。採用母数も桁違いに広い。ランタイムのように「1つの use-after-free が全ユーザーの脆弱性になる」種類のソフトウェアで、安全性を型で担保できることの価値は大きいです。

ただし、正直な対価も書いておきます。書き直しは、複数四半期にわたる機能開発速度への課税です。そして、すでに実戦で叩かれてバグが出尽くしたコードを、まだ叩かれていないコードで置き換える行為でもあります。エッジケースの挙動、性能特性、メモリの使い方 — 旧実装が暗黙に持っていた性質は、仕様書ではなくコードの中にあります。書き直しは必ず新しい回帰を生みます。段階的に進める限りそれは管理可能ですが、ゼロにはなりません。「Rust だから安全になる」は正しく、「Rust に書き直したから今日から安定する」は誤りです。

## ベンチマークを見ずにランタイムを評価する

Bun の議論はほぼ常に速度の話になります。しかし本番でランタイムを選ぶとき、ベンチマークが決め手になった経験は、私たちにはほとんどありません。実務で効いてくるのは次の3つです。

**Node 互換の表面積。** 「Node 互換」は 0 か 1 かではなく、面積です。あなたのアプリが実際に踏む API — `node:crypto` のどの関数か、`worker_threads` を使うか、ネイティブアドオンに依存する ORM を積んでいるか、`AsyncLocalStorage` に乗った APM を入れているか — その具体的な組み合わせで決まります。判定方法は単純で、あなたのテストスイートをそのランタイムで回すことです。他人のベンチマークではなく、自分のコードが答えを持っています。

**セキュリティパッチの供給速度。** ランタイムは攻撃面そのものです。CVE が出てから修正版が出るまでの実績時間、サポートされるメジャーバージョンの本数、脆弱性報告の受付窓口が整備されているか。Node.js には LTS の運用と定期的なセキュリティリリースという、退屈だが予測可能な仕組みがあります。若いランタイムに同じ予測可能性を期待するのは、まだ早い場合があります。

**誰も踏んでいないバグを踏んだときに何が起きるか。** これが一番重要です。Node の奇妙な挙動なら、たいてい15年分の Stack Overflow と issue が先回りしています。新しいランタイムでは、あなたが最初の1人になり得ます。そのとき取れる手は、最小再現を作って issue を立て、修正を待つか、自分でパッチを当てるか、その機能を諦めるかです。この「詰まったときの脱出経路」を事前に持っているかどうかが、本番採用の可否を分けます。

言語の書き直しは、この3点すべてに影響します。移行期間中はパッチ供給が2つの実装に分散し、あなたが踏むバグは「旧実装にはなかった新しいバグ」かもしれません。

## 今日、Bun が正当に居場所を持つ場所

webhani の現場感覚では、Bun の価値が最も確実に出るのは CI です。理由は速さそのものではなく、速さが**掛け算で効く**からです。install はすべてのパイプライン実行で必ず走ります。1日に PR が20本、それぞれで数回 CI が回るなら、install 1回あたりの短縮は月あたり数百回分になります。

v1.3.14 で面白いのは「7倍」という数字ではなく、グローバルストアを持つ isolated linker というアプローチです。パッケージの実体をマシン単位の共有ストアに1つ置き、各プロジェクトの `node_modules` はそこへのリンクで構成する。同じパッケージを何度も展開し直さないので、warm install が速くなる。これは pnpm が同じ発想で先に示した方向で、ディスク使用量にも効きます。数字より構造のほうが、再現性の判断材料として有用です。

スクリプト実行とテスト実行も安全な導入先です。どちらも失敗が即座に、しかもローカルで見える領域だからです。

一方、本番の HTTP サーバ経路はもう少し慎重で構いません。ここで壊れると、壊れ方が静かです。特定のヘッダの扱い、コネクションの寿命、バックプレッシャー、シグナルハンドリング — 障害として顕在化するまでに時間がかかる種類の差異が出ます。`Bun.serve()` 自体は素直な API です。

```ts
// server.ts — Web 標準の Request / Response に沿った素直な形
const server = Bun.serve({
  port: Number(process.env.PORT ?? 3000),

  async fetch(req) {
    const url = new URL(req.url);

    if (url.pathname === "/healthz") {
      return new Response("ok", { status: 200 });
    }

    return Response.json({ path: url.pathname });
  },

  error(err) {
    console.error(err);
    return new Response("internal error", { status: 500 });
  },
});

console.log(`listening on :${server.port}`);
```

ハンドラが `Request` を受けて `Response` を返す形になっているのは重要です。この形なら、ロジックを純粋な関数に切り出しておくかぎり、ランタイムを差し替えてもコアは動きます。ランタイム固有の API は、できるだけ入口の1ファイルに閉じ込めてください。

そして HTTP/3 クライアントと QUIC 対応は、公式が experimental と書いています。experimental とは「動くこともある」という意味です。本番のトラフィックを預ける対象ではありません。

## 書き直しは Bun だけの話ではない

同じ7月8日、TypeScript 7 が GA になりました。コンパイラは Go で書き直されています。Rolldown は Rust、Biome も Rust。JS ツールチェーンは、JavaScript で書かれることをやめつつあります。速いから、というだけの理由です。

ここから来るリスクは、個々のツールの品質ではなく、依存の性質が変わることです。あなたのビルドは今、**手元でパッチを当てられないネイティブバイナリ**に依存し始めています。以前なら `node_modules` の中の .js を1行書き換えて金曜の障害を凌ぐことができました。プリビルドバイナリではそれができません。fork してクロスコンパイル環境を用意する話になります。

対策は地味ですが有効です。ツールチェーンのバージョンは正確にピン留めする。ロックファイルはコミットする。そして、フォールバック経路を持っておく。CI では、こう書けます。

```yaml
# .github/workflows/ci.yml
name: ci
on: [push, pull_request]

jobs:
  test-bun:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7

      - uses: oven-sh/setup-bun@v2
        with:
          bun-version: "1.3.14"   # 範囲指定ではなく厳密に固定する

      - run: bun install --frozen-lockfile
      - run: bun test

  # Bun 側が壊れても本番相当の検証が止まらないようにする第二経路
  test-node:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v7

      - uses: actions/setup-node@v6
        with:
          node-version: "22"
          cache: "npm"

      - run: npm ci
      - run: npm test
```

2つ回すのは無駄に見えますが、これは保険です。Bun 側が回帰で落ちた日に、`test-node` が緑なら「Bun の問題」と即断でき、その日のリリースは止まりません。逆に両方落ちていれば、それはあなたのコードです。切り分けが数分で終わるという価値は、CI 時間の数分より確実に大きい。

`package.json` も、ランタイムに縛られない形で書けます。

```json
{
  "scripts": {
    "dev": "next dev",
    "build": "next build",
    "test": "vitest run",
    "test:bun": "bun test",
    "lint": "biome check ."
  },
  "packageManager": "npm@10.9.0",
  "engines": {
    "node": ">=22"
  }
}
```

ポイントは、`test` を Bun 固有のランナーに直結させないことです。Bun で回したいなら `test:bun` を別に置く。こうしておけば、明日 Bun をやめる判断をしても、消すのは1行です。

## まとめ

- 実装言語の書き直しは、健全な長期投資であり、同時に短期の機能開発とバグ回帰の税でもあります。「Rust になったから安定する」ではなく「安定するまでには時間がかかる」と読んでください。
- ランタイム評価はベンチマークではなく、Node 互換の表面積・セキュリティパッチの実績速度・未知のバグに詰まったときの脱出経路で行います。最良のテストは、自分のテストスイートを回すことです。
- **Bun を入れるなら:** CI の install とテスト、開発用スクリプトと社内ツール。効果が掛け算で効き、失敗が即座に見えます。
- **まだ入れないなら:** 本番の HTTP サービング経路、および experimental と明記された HTTP/3・QUIC。書き直しが一区切りついてから再評価すれば十分です。
- ネイティブツールチェーンは正確なバージョンでピン留めし、フォールバック経路を CI に残してください。手元でパッチを当てられない依存が増えている以上、切り戻せることが唯一の保険です。
