2026年7月9日、OpenAI は GPT-5.6 と同時に「ChatGPT Work」を発表しました。同時に Codex は新しい ChatGPT Desktop App に統合され、従来の Chat App は「ChatGPT Classic」という位置づけになっています。今回の発表で興味深いのは新しい Model そのものよりも、AI Agent の返す成果物の形が変わってきている点です。本記事では ChatGPT Work の特徴と、実務に取り入れる際の考え方を整理します。

## Chat の回答から「完成した成果物」へ

これまでの Chat 型 AI は、質問に対してテキストで回答を返すのが基本でした。ChatGPT Work はこの構造を変えています。ユーザーが渡すのは「質問」ではなく「達成したいゴール」で、Agent は次のような手順で動きます。

1. 連携済みの Application や Workflow から必要な情報を収集する
2. ゴールを実行可能なステップに分解する
3. 各ステップを自律的に実行し、数時間単位のタスクにも取り組み続ける
4. 最終的に Spreadsheet、Slide、Document、Web App といった「完成した成果物」を返す

これは Coding の領域ではすでに馴染みのある発想です。Claude Code や GitHub Copilot の Workspace Agent は、単発の Code 補完から「Issue を渡すと PR が返ってくる」形へと進化してきました。ChatGPT Work は、この Outcome 型のアプローチを Coding 以外の office work 全般に広げたものと捉えると理解しやすいでしょう。

## Codex 統合の実務的な意味

Codex が Desktop App に統合されたことで、Chat・Work・Codex が同一の Interface 上に並ぶようになりました。開発者にとっての実質的な変化は、Coding Agent と汎用 Outcome Agent が同じ Session やコンテキストを共有しやすくなる点です。例えば「API 仕様書を書きながら、対応する実装 PR も並行して進める」といった、Documentation と Coding をまたぐタスクの分断が減ることが期待できます。

一方で、Rollout は Pro・Enterprise・Edu Plan から始まり、Plus・Business Plan へ段階的に広がる形です。組織で導入を検討する場合、まずは限定的な Plan・限定的なユーザーで試験導入し、挙動を見極めてから展開範囲を広げるのが現実的です。

## 導入前に整理すべきガバナンスの論点

Outcome 型 Agent が「連携済みの Application から情報を収集する」という設計である以上、どの Application に、どの範囲でアクセスを許可するかが最初の論点になります。特に社内の Ticket System や Repository、Cloud Storage に接続する場合は、最小権限の原則を徹底する必要があります。

もう一つの論点は、Agent が生成した成果物をそのまま外部に公開してよいかどうかです。数時間かけて自律的に作られた Document や Web App であっても、最終的な公開判断には人によるレビューを挟むべきです。これは Coding Agent の PR に Human Review を必須にするのと同じ発想です。

```yaml
# .github/workflows の一部（概念例）
# AI Agent が作成した PR は、レビューラベルが付くまで Merge をブロックする
name: require-human-review-on-agent-pr

on:
  pull_request:
    types: [labeled, opened, synchronize]

jobs:
  check-review-label:
    if: contains(github.event.pull_request.labels.*.name, 'ai-agent-generated')
    runs-on: ubuntu-latest
    steps:
      - name: Verify human-reviewed label exists
        run: |
          # 'ai-agent-generated' が付いた PR は 'human-reviewed' が
          # 別途付与されるまで Merge を許可しない運用にする
          echo "AI Agent 由来の変更は人によるレビュー完了を必須とする"
```

このような Gate は Vendor に依存しない汎用的な考え方です。ChatGPT Work に限らず、どの Outcome 型 Agent を導入する場合でも「誰が最終承認するか」を運用ルールとして先に決めておくことをお勧めします。

## webhani の視点

webhani では、Outcome 型 AI Agent の導入を検討する Client に対して、次の順序を提案しています。

1. **社外に公開しない社内向けタスク**（定型 Report、社内 Document の下書きなど）から試験導入する
2. Agent に許可する **Data Connection の範囲を最小限に絞り**、後から拡張する
3. 成果物を外部公開・本番反映する前に、**人によるレビューを必須の Gate として運用に組み込む**

新しい Model の性能そのものよりも、こうした「導入プロセスの設計」の方が実務上のリスクを左右します。ChatGPT Work のような Outcome 型 Agent は今後さらに増えていくと見られるため、Vendor 固有の機能に依存しすぎない、共通のガバナンス方針を早めに用意しておくことが重要です。
