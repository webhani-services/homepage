## LLM から Agent へのシフト

2026年の開発者ツール領域での最も大きな変化は、LLM の使い方の変化です。かつては「よりよいオートコンプリート」として使われていたものが、計画を立て、ファイルを読み書きし、コマンドを実行し、結果を検証するエージェントとして機能するようになりました。

Claude Code、Cursor、GitHub Copilot Workspace といったツールの普及により、LLM との対話パターンが根本的に変わっています。2026年4月にリリースされた Claude Opus 4.7 は、SWE-bench Pro で 64.3% のスコアを記録しました（Opus 4.6 の 53.4% から 10.9 ポイント向上）。SWE-bench はオープンソースリポジトリの実際の GitHub Issues を解決する能力を測定するベンチマークです。この改善は、実際のコードベースでの問題解決能力が実質的に向上したことを意味しています。

## Agent に適したタスクとそうでないタスク

すべてのタスクが Agentic ワークフローに向いているわけではありません。この区別が重要です。適切でないタスクに Agent を使うと、一見正しく動作するが、コードレビューで見逃しやすい微妙な問題を含むコードが生成される可能性があります。

**Agent が得意なタスク:**
- 複数のファイルにまたがる一貫した変更（リファクタリング、命名規則の統一など）
- ボイラープレートの生成（テストファイル、型定義、CRUD 操作）
- 既存のコードパターンを参照した類似実装
- コード変更からドキュメントや Changelog を生成

**Agent が苦手なタスク:**
- ビジネスロジックの深い理解が必要な設計判断
- 外部システムの状態に依存する複雑なデバッグ
- 正確性の基準が自明でないセキュリティ関連のコード
- 依存関係が多い長期の複数セッションにまたがる作業

Agent に任せる前に「このタスクの正しさをどう検証するか」を明確にすることが重要です。検証方法が曖昧なまま Agent を走らせると、一見正しく見えてコードレビューを通過してしまうような問題のあるコードが生成されるリスクがあります。

## 効果的なプロンプトの構造

Agentic モードでは、プロンプトの書き方が成果に大きく影響します。曖昧なリクエストは、Agent が意図と相反する前提を立ててしまう原因になります。

```markdown
# タスク
[具体的な変更内容を1〜2文で記述]

# コンテキスト
- 変更対象のファイル: `src/lib/auth.ts`
- 関連する型定義: `src/types/user.ts`
- テストファイル: `src/lib/auth.test.ts`

# 制約
- 既存のテストをすべてパスさせること
- `any` 型を使用しないこと
- エラーハンドリングは `src/lib/errors.ts` の既存パターンに合わせること

# 期待する出力
- 変更されたファイルのリスト
- 変更内容と理由の概要
```

「どう実装するか」ではなく「何を達成するか」と「何を変えてはいけないか」を明確にすることが重要です。Agent は実装戦略を自分で立てます。あなたの役割は解決策の空間を制約することであり、解決策を処方することではありません。

## Context Window の管理

長い Agentic セッションで最もよく遭遇する失敗パターンの一つが、Context ドリフトです。会話が長くなるにつれて、Agent が初期の制約や決定事項を追跡できなくなります。これはモデルのバグではなく、Context Window の管理問題です。

実践的な対策:

```markdown
# セッション開始時のアンカー（新しいセッションの冒頭に貼り付ける）

現在の作業: [簡潔な説明]
完了した作業:
- ✅ 認証モジュールのリファクタリング
- ✅ 型定義の更新

残りの作業:
- [ ] テストの更新
- [ ] ドキュメントの更新

変更不可の制約:
- `any` 型の使用禁止
- エラー処理はカスタム Result 型を使用
- すべてのエンドポイントにリクエストバリデーションを含めること
```

複数セッションにまたがるタスクでは、各セッション終了時に Agent に構造化された進捗サマリーを作成させ、次のセッション開始時にそれをコンテキストとして貼り付ける方法が効果的です。トークンのコストは少しかかりますが、高コストな手戻りを防げます。

## 人間によるレビューの組み込み

Agent が生成したコードをそのままプロダクションに反映することはリスクパターンです。実践的なワークフローは人間のレビューのチェックポイントを含む必要があります。

```bash
# コミット前に変更をレビュー
git diff --stat HEAD

# 大きな変更セットの場合はインタラクティブにステージング
git add -p

# 生成されたコードに対してテストを実行
npm test

# 型チェックを別途実行 — テストは型エラーを検出しない
npx tsc --noEmit
```

注意すべき失敗パターン: テストは通過するが意味的な問題があるコード — 技術的には動作するがテストがカバーしていない不変条件を侵害するリファクタリング、または例外を伝播させる代わりに握りつぶすエラーパスなどです。

Agent の作業から生まれる小さな増分コミットは、一つの大きなコミットより安全です。1000行の diff より 100行の diff をレビューするほうが、どの変更が問題を引き起こしたかを特定しやすくなります。

## チームでの導入パターン

個人での Agent 利用と、チームでの体系的な導入は異なる問題です。

チーム導入で効果的なアプローチ:

**1. コードベースの規約を AI が読める形式で文書化する**

```markdown
# AI アシスタント向けエンジニアリングガイドライン

## TypeScript
- 常に strict モードを使用すること。`any` は絶対に使用しない
- ドメインエラーには throw より `Result<T, E>` を優先する
- すべての public 関数に明示的な戻り値の型アノテーションを付けること

## API Route（Next.js）
- 処理前に Zod でリクエストボディをバリデーションすること
- `src/lib/api.ts` の標準エラーレスポンス形式を使用すること
- Route Handler ではなく Service 層でログを出力すること

## テスト
- ユニットテスト: 純粋関数のみ
- 統合テスト: `.env.test` で設定されたテスト用 DB を使用すること
```

**2. 効果的なプロンプトテンプレートを共有する**

一貫して良い結果を出すプロンプトパターンがあれば、チーム全体が使えるテンプレートとして抽出しましょう。プロンプトはエンジニアリングの成果物です。

**3. Agent の出力に同じコードレビュー基準を適用する**

多くのチームが間違えるポイントです。構文的に正しくコンパイルが通るため、Agent 生成コードが軽いレビューで済まされることがあります。ジュニアエンジニアのコードに適用するのと同じレビュー基準を適用してください。特定の決定が過去になぜ行われたのかというコンテキストを Agent は持っていないため、より厳格に検討すべき場合もあります。

## Claude Code の Background Task モード

GitHub Actions を通じてアクティブなセッションなしで Agent を実行できる Claude Code の Background Task モードは、反復的なタスクのワークフローを変えます。ライブラリアップグレード後の依存関係の型更新、マイグレーションスクリプトの生成、コード変更とドキュメントの同期維持などに活用できます。

```yaml
# .github/workflows/ai-type-update.yml
name: Update types after dependency bump
on:
  pull_request:
    paths:
      - 'package.json'

jobs:
  update-types:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Code to update affected types
        uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            package.json が変更されました。変更されたパッケージの TypeScript 型インポートで
            更新が必要なものがあるか確認してください。`tsc --noEmit` を実行して型エラーを見つけ、
            修正後に "chore: update types for dependency changes" というメッセージでコミットしてください。
```

ほとんどのチームにとってまだ初期段階ですが、パターン自体は有効です: 境界が明確で検証可能なタスクを定義し、CI で実行し、人間が PR をレビューする方式です。

## まとめ

Agentic AI は「試してみるべきか」という段階を過ぎました。今の問いは実践的なものです: どのタスクを委任するか、プロンプトをどう構造化するか、長いセッションで Context をどう管理するか、そして Agent の出力を既存のレビュープロセスにどう統合するか。

モデルの能力は今後も向上し続けます。しかし、タスクを正確に定義し、出力を体系的に検証し、人間の判断が代替不可能な領域を把握するスキルは、モデルがいかに優秀になっても変わらず必要とされます。
