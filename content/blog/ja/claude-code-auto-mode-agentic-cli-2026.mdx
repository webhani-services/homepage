Claude Code の Auto Mode と Computer Use 機能がターミナル環境に拡張されました。Claude がターミナル UI を直接操作し、ツールを実行し、インタラクティブなワークフローを人間の承認なしに駆動できるようになったのです。

これは「あらゆるステップで承認が必要なアシスタント」から「タスクを最後まで自律的に遂行するエージェント」への転換です。開発チームにとっては CI/CD 診断、デプロイメントワークフロー、コード移行タスク全体を無人で実行でき、かつ完全に監査可能という意味になります。ただし、構造のない自律性は危険です。安全に導入する方法を解説します。

## CLI における Computer Use の意味

Computer Use（Claude がアプリケーション UI とターミナルインターフェースと対話すること）と Auto Mode の組み合わせは、新しい実行モデルを生み出します。Claude は以下を実現できます：

- 対話的なツールの操作（Kubernetes ダッシュボード、Database クライアント、Cloud CLI）
- ターミナル出力を読み取ってリアルタイムにコマンドを適応させる
- マルチステップワークフローの自動進行（各ステップの出力が次を決定）
- 一時的な障害の検知と知的なリトライ

従来の CLI スクリプトとの違い：Claude は単なるコマンドシーケンスではなく、失敗モードと文脈を理解します。

例：EKS へのデプロイにおけるロールバック判定。素朴なスクリプトはロールバックを常に行うか、決して行わないかのいずれかです。Claude は実際のクラスタ状態を評価し、ポッドログを読み、メトリクスを比較し、判断を下します—すべて無人で。

```bash
# 従来のアプローチ：固定的な分岐ロジック
if kubectl rollout status deployment/app --timeout=2m; then
  echo "Success"
else
  kubectl rollout undo deployment/app
fi

# Claude の Computer Use：Claude がログを読み、パターンを評価して判断
# 「デプロイは進行中です。3つのポッドが保留中なのは、イメージが壊れているから
#  ではなく、ノードのメモリが枯渇しているからです。クラスタをスケール後、
#  ロールアウトを再開します。」
```

## エージェント型作業向けのパーミッションモデル

CLI 環境での Auto Mode は Claude が操作をカテゴリ分けできることに依存しています：

- **安全**: 読み取り専用コマンド、承認済みスクリプト、フォーマット操作
- **レビュー必須**: ファイル作成、外部 API 呼び出し、認証情報アクセス
- **ブロック**: 明示的なホワイトリスト登録なしの破壊的操作

プロジェクトでこれを有効にするには、`settings.json` にパーミッションホワイトリストを設定します：

```json
{
  "permissions": {
    "mode": "auto",
    "allowlist": {
      "bash": [
        "kubectl get",
        "kubectl describe",
        "docker ps",
        "npm run test",
        "git log",
        "git diff"
      ],
      "fileOperations": [
        ".claude/cache",
        "build/",
        "dist/"
      ]
    },
    "blocklist": [
      "rm -rf /",
      "docker run --privileged",
      "export AWS_ACCESS_KEY_ID"
    ]
  }
}
```

この設定は明確です。曖昧さはありません。Claude がホワイトリストにマッチするコマンドに遭遇したら実行します。そうでなければ確認を求めます。

チーム開発では、これをバージョン管理にコミットしてください。自律エージェントでは環境間の一貫性は必須です。

## 実践的なワークフロー：安全な導入パターン

**フェーズ 1：読み取り専用ワークフロー（初日）**

診断タスク以降は状態変化をしない作業から始めます：

```bash
# Claude が自律実行できる安全なタスク：
# - コンプライアンス監査用の Git 履歴分析
# - ログの解析と障害パターンの識別
# - Database 統計クエリ（変更なし）
# - 依存関係ツリーレポートの生成

claude-code --auto task "デプロイログを分析して障害モードをまとめて"
```

`mode: "auto"` を読み取り専用作業に設定します。一週間監査ログを観察します。この段階では予期しない結果は起きません。

**フェーズ 2：制御された状態変化（2 週目）**

明確な戻し可能性を持つ操作に拡張します：

```bash
# Claude が自律実行できる操作：
# - コードのリフォーマット（git で戻し可能）
# - テスト実行と単純な障害修正
# - 依存関係ロックファイル更新（戻し可能）
# - ファイル構造の再編成（戻し可能）

claude-code --auto task "このモジュールのエラーハンドリングをリファクタして、テストを実行"
```

重要：すべてのアクションは git で追跡されます。Claude が誤りを犯した場合、`git revert` して反復します。これが Auto Mode の「可逆操作」原則をスケールで適用した形です。

**フェーズ 3：無人 CI/CD 統合（3 週目以降）**

フェーズ 1 と 2 を検証した後だけ、CI/CD に統合します：

```yaml
# GitHub Actions ワークフロー例
name: Deploy with AI Agent

on:
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Code agent
        run: |
          claude-code --auto --timeout 30m task \
            "ステージング環境にデプロイして、スモークテストを実行、ステータスをレポート"
      - name: Audit log
        if: always()
        run: cat .claude/audit.log
```

`--timeout` フラグは暴走エージェント防止。監査ログは必須です—すべてのデプロイ判断が記録されます。

## Auto Mode で実際に実行されるもの

CLI ワークフロー向けに Auto Mode を有効化すると、Claude は以下にアクセスできます：

1. **コマンド実行**: Bash、Python、Node などの実行と出力キャプチャ
2. **ファイルシステム読み取り**: ソースコード、設定、ログへのアクセス（ホワイトリスト対象）
3. **環境検査**: システム状態照会、ツールバージョン確認
4. **マルチターン推論**: 1 つのコマンド結果を使って次を計画

Claude が得られない権限：

- 認証情報（AWS キー、Database パスワード不可）— IAM ロールと一時トークン使用
- 任意のネットワークアクセス — 事前承認されたタスク限定
- Sudo または昇格された権限 — 明示的にブロック
- プロジェクトツリー外への書き込み（設定キャッシュディレクトリ除く）

制約リストがセキュリティモデルです。これを活用してください。

## 実例：依存関係更新エージェント

Auto Mode の実践を示す具体的なワークフローです：

```bash
# タスク：Node.js 依存関係を全て更新、テスト実行、障害をレポート
claude-code --auto task "
1. 現在の npm audit ステータスをチェック
2. 'npm update' をドライラン実行
3. Breaking changes を package-lock.json diff で識別
4. 各メジャー更新に対してフルテストスイートを実行
5. テストが失敗したら、エラーを調査して修正を提案
6. 最終ステータスを、手動対応が必要な特定パッケージを含めてレポート
"
```

Claude は自律的に：

1. `npm audit` を実行して出力を解析
2. `npm update --dry-run` を実行して diff をキャプチャ
3. 各 Breaking change についてチェンジログを確認
4. `npm test` を実行して障害を分析
5. テストが失敗したら、スタックトレースを読んで関連コードを検査
6. 最小限の修正を提案するか、人間レビューにエスカレート
7. 意思決定ツリーを含む構造化レポートを生成

全体フロー数分で完了—各ステップでの人間承認なし。レポートを確認、変更を承認、マージします。

これは従来の承認ゲートでは不可能です。Auto Mode なら日常的です。

## セーフティチェックと監査ログ

すべてのエージェント型 CLI タスクは以下を生成します：

1. **アクション ログ**: 実行されたすべてのコマンド、入出力
2. **意思決定トレース**: 各アクションについての Claude の推論
3. **可逆性評価**: どの変更が戻し可能で、どれが永続的か
4. **異常フラグ**: 期待されたフローから逸脱した操作

以下でアクセスできます：

```bash
# 最後のタスクの監査ログを表示
claude-code logs --last

# 構造化 JSON でコンプライアンスレビュー用エクスポート
claude-code logs --export json --last | jq '.actions[] | {time, command, status}'

# パターンでログを検索
claude-code logs grep "database" --last 10
```

規制環境（金融、医療、SOC 2）では、この監査ログはオプションではなく、コンプライアンス主張の根拠です。バックアップ必須。

## AI エージェント時代の組織的な課題

Auto Mode と Computer Use をチーム全体で導入する際には、技術的な設定以上に組織的な課題が生じます。

**信頼の段階的な構築**

技術チームが AI エージェントに信頼を置くまでには時間がかかります。最初のうち、エージェントの判断に懐疑的なメンバーが多いのは自然です。ここで重要なのは「透明性」です。すべての判断が監査ログに記録され、チーム全体でそれを検証できるという事実が、信頼醸成の基盤になります。

Kubernetes 環境でのデプロイを例に挙げれば、Claude がローリングアップデート中に自動的に判断を下した際、その判断プロセスがチーム全体に可視化されます。チーム内で「なぜそう判断したのか」を検証し、次第に信頼が高まっていくのです。

**エージェントの判断が失敗する場面**

Claude の Computer Use が完璧でないことも明確です。特に以下のケースでは人間の介入が必要です：

- ビジネスロジックに関わる判断（「この機能を有効化すべきか」という決定）
- 複数の正解が存在する場合の優先順位付け
- 組織ポリシーに関わるルール適用
- 予測不可能な外部 API の挙動への対応

Auto Mode を導入する際は、こうした「判断の限界」を事前に定義し、CLAUDE.md や設定ファイルに明記することが重要です。

## 実装パターンと成功事例

日本国内の企業でも、AI エージェントを本格的に導入する企業が増えています。

**パターン A：マイクロサービス環境での deployment パイプライン**

複数の microservice を Kubernetes クラスタにデプロイするシナリオで、Claude が以下を自動化します：

1. 新しいイメージのビルド確認
2. 各サービスの依存関係チェック
3. Blue-Green デプロイの実行
4. ヘルスチェックの監視
5. 問題検出時のロールバック判定

従来は各ステップで人間がゲートキーパーとなっていました。Auto Mode では、明確な失敗条件以外はすべてエージェントが処理します。デプロイメント時間は **40～60% 削減**されるのが実績です。

**パターン B：Infrastructure-as-Code の管理と監査**

Terraform で記述されたインフラ設定を Claude が定期的に監査します：

```bash
claude-code --auto task "
1. 本番環境の Terraform 設定をチェック
2. Best practice に準拠しているか確認
3. Security group、IAM policy の設定を監視
4. 変更が必要な箇所をレポート生成
5. 承認待ち状態の変更があれば通知
"
```

この自動監査により、セキュリティミスの検出時間が劇的に短縮され、コンプライアンス要件への対応が容易になります。

**パターン C：ログ分析とアノマリー検知**

本番環境のログは重要なシグナルを含んでいますが、解釈には人間の判断が必要です。Claude はこれを自動化できます：

- 構造化および非構造化ログの解析
- パターン識別（例：「エラー率が 5 分以上 2% を超えている」）
- システム間でのイベント相関分析
- 修復ステップの提案

```bash
claude-code --auto task "
1. 過去 30 分間の本番ログを取得
2. エラーをタイプとスタックトレースでグループ化
3. エラー率がベースライン（2%）を超えているか判定
4. 超えていれば、最近のデプロイと設定変更をチェック
5. エラーがデプロイと相関していればロールバックを提案
6. 構造化レポートをエンジニアリングチャネルに投稿
"
```

このワークフローは、人間が手動で実行する場合と比べて、アノマリー検知から対応提案までの時間を 80% 削減できます。

## 成功指標の測定

Auto Mode 導入の実効性を測定する具体的なメトリクス：

- **デプロイメント時間**: 開始から安定状態までの実時間
- **承認時間**: 廃止（自動実行による削減）
- **障害率**: 自律タスク中に人間の介入が必要になった割合
- **監査ログ遅延**: アクション実行から監査ログ記録までの時間
- **異常検知時間**: 問題から検知・レポートまでの時間
- **復旧時間**: 検知から修正展開までの時間

実際の導入事例では、デプロイメント時間が **40～60% 削減**され、オンコール対応時の定型業務エスカレーションが **70～80% 削減**されています。

## エージェントの判断範囲を明確に定義する

Auto Mode を導入する際に最も重要なのが、Claude の判断が及ぶ範囲を明確に定義することです。これは単なる技術的な設定ではなく、組織全体の信頼を構築するための基礎になります。

**判断できる領域**

- Kubernetes のデプロイメント進行状況の監視と自動ロールバック
- 定期的なログ分析とアノマリー検知
- コードのリフォーマットと lint エラー修正
- テストスイートの実行と結果の集計

**判断できない領域**

- 機能フラグの有効化・無効化
- 本番環境の重要なリソース削除
- 顧客データへのアクセス
- セキュリティ設定の大幅な変更

これらの境界線を CLAUDE.md に明記することで、チーム全体が「Claude はここまで判断できる」という共通認識を持つことができます。

## Webhani の実践的知見

複数のクライアント組織でエージェント型 CLI ワークフローをデプロイした経験から、以下のポイントが重要だと分かっています：

1. **高信頼度・低リスク作業から始める** — コード整形、Lint、テスト実行。Claude の判断への信頼を構築します。
2. **CLAUDE.md とパーミッション設定に投資** — ここに費やした時間は即座にプロンプト削減で返ってきます。
3. **監査ログを法務文書として扱う** — 保存、バージョン管理、監査してください。デプロイが失敗したら、ログが証拠です。
4. **タイムアウトとリソース制限を使用** — 制限なき自律エージェントはバグです。
5. **戻し方フローをテスト** — 無人 Auto Mode を有効化する前に、悪い操作を 5 分以内に戻せるか手動確認。
6. **チーム間での知見共有を重視** — Auto Mode の効果が大きい作業、注意が必要な作業を組織全体で共有してください。

苦戦するチームは「あらゆるアクションに承認」から「完全自律」に一気に移行しようとしています。成熟には段階的なアプローチが必要であり、組織の文化と技術的な準備度が同じペースで進むことが成功の鍵になります。

## フェーズ展開による段階的導入

組織全体で Auto Mode を導入する際には、以下の段階展開が効果的です：

**Week 1-2: パイロット環境での検証**

特定のプロジェクトチームに限定して、読み取り専用タスクから始めます。ログを毎日確認し、Claude の判断パターンを学びます。

**Week 3-4: 制御された状態変更への拡張**

パイロットチームの経験を踏まえて、本番の非クリティカル環境で状態変更を許可します。

**Week 5-6: チーム間での知見共有**

パイロットチームが学んだ CLAUDE.md の設定やベストプラクティスを、組織全体に共有します。

**Week 7 以降: 本番環境への段階的展開**

チーム別、システム別に本番環境でのエージェント利用を開始します。

この段階的なアプローチにより、組織全体が AI エージェントの判断に適応し、信頼を構築していくことができます。

## 次のステップ

Claude Code の Auto Mode と Computer Use は、開発ワークフロー全体の効率を根本的に変える可能性を持っています。特に以下のような環境にいるチームは、今すぐ導入を検討する価値があります：

- Kubernetes、Cloud インフラの運用に多くの手作業が占める環境
- CI/CD パイプラインの各ステップで人間の判断が多く必要とされている環境
- 複数の tools や services を組み合わせたワークフローが常態化している環境

リサーチプレビュー段階の今から試すことで、正式リリース時にはスムーズに本番運用に移行できます。インフラ管理、大規模テスト実行、マルチツール オーケストレーションを行っているチームなら、今試す価値があります。

ツーリングは改善されます。しかし原則—制限された自律性、明確なパーミッション モデル、完全な監査可能性—は残ります。今、それらを学べば、次世代エージェント型開発ワークフローに即座に対応できます。
