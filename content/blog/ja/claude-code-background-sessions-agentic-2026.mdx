## Claude Code のバックグラウンドセッション

Claude Code に `/bg` コマンドが追加され、アクティブなセッションをバックグラウンドの Agent View に移動できるようになりました。長時間かかるコーディングタスクを Claude に委任しながら、自分は別の作業を進められるようになります。

これは単なる UI の変更ではありません。開発ワークフローをどう設計するかという問いへの、具体的な答えです。AI コーディングツールとのインタラクションモデルが「対話型」から「委任型」へとシフトしています。

## /bg の基本的な使い方

Claude Code のターミナルセッション内で `/bg` を入力すると、現在のタスクが Agent View に移行します。

```
# Claude Code セッション内でタスクを指示した後
/bg
```

バックグラウンド移行後にできること:
- **新しいセッションを開始** して別タスクを進める
- **Agent View で進捗を確認** する（リアルタイム更新）
- **ターミナルを閉じても継続** — Remote Control でセッション外からアクセス可能

Agent View は複数のバックグラウンドタスクを並列で管理し、それぞれのステータスをインライン更新で表示します。

## 実用的なワークフロー

### パターン 1: 大規模リファクタリングのオフロード

TypeScript の strict モード移行など、時間のかかるリファクタリングを Claude に任せながら、自分は別の作業を進めます。

```
# セッション 1: Claude に strict モード移行を依頼
> tsconfig.json の strict: true を有効にして、型エラーをすべて修正してください。
  src/ 以下のファイルが対象です。既存のロジックは変更せず、
  型の問題のみを修正してください。any 型の使用は避け、
  適切な型定義を追加してください。
  修正が完了したら npm run type-check を実行して確認してください。

/bg  # バックグラウンドに移行

# セッション 2: 別の開発タスクを開始
> 新しいコンポーネントのテストを書いてください。
```

### パターン 2: テストスイートの実行と分析

CI で発生したテスト失敗を調査させながら、ほかの作業を続けます。

```
> npm test の出力を確認して、失敗しているテストを修正してください。
  E2E テストは除いて、Unit テストのみ対象です。
  テストの assertions は変更せず、テストが正しい場合は
  実装側を修正してください。完了後に npm test を再実行して確認してください。

/bg
```

### パターン 3: 長時間ビルドの監視

`docker build` など時間のかかるコマンドの完了待ちを Claude に任せ、完了時にアクションを実行させます。

```
> docker compose build を実行して、ビルドが成功したら
  docker compose up -d を実行し、コンテナのヘルスチェックを確認してください。
  失敗した場合は正確なエラーを報告して停止してください。

/bg  # 完了まで Claude が監視する
```

## バックグラウンドタスクへの効果的な指示

バックグラウンドセッションを活用するには、タスクの指示を明確にすることが重要です。Claude が後続の質問をできない状況では、判断を自律的に下します。指示が明確なほど、その判断の精度が上がります。

```
# 曖昧な指示（バックグラウンドには不向き）
> 型エラーを直してください

# 具体的な指示（バックグラウンドに適している）
> src/components/ 以下の TypeScript 型エラーをすべて修正してください。
  以下の制約を守ってください:
  - 既存のロジックとテストは変更しない
  - any 型の使用は禁止
  - 関数のシグネチャ変更が必要な場合は最小限に留める
  - 修正完了後に npx tsc --noEmit を実行してエラーが 0 になることを確認する
```

明確にすべき要素は「スコープ」「制約」「成功基準」「不確実な場合の行動方針」の 4 つです。

## Dispatch と Remote Control

より高度な使い方として、Dispatch と Remote Control の 2 つの機能があります。

**Dispatch** は API 経由でタスクをキューに入れる仕組みです。CI/CD パイプラインや外部スクリプトから Claude Code にタスクを送信できます。PR が作成されるたびに自動でコードレビューを依頼するような使い方が想定されます。

**Remote Control** はターミナル外からセッションを操作する仕組みです。タスクを開始してターミナルを閉じ、後からセッションに戻って結果を確認できます。ラップトップとデスクトップ間でセッションを引き継ぐ用途にも使えます。

## 次のステップ: Chyros

「Chyros」は計画中の Claude Code 機能の内部コード名で、常時起動のバックグラウンドデーモンとして動作することを目標としています。現在の `/bg` コマンドがアクティブなセッションをバックグラウンドに送る方式であるのに対し、Chyros はターミナルセッションを必要とせず非同期で独立実行するモデルです。

まだリリースされていませんが、Claude Code が向かう方向は明確です。常時動作しているエージェントに作業を割り当て、自分のスケジュールに合わせて結果を受け取る形態です。

## 現在の制約と注意点

- バックグラウンドセッションは Claude Code の最新バージョンが必要です
- Dispatch 機能はまだベータ段階で、すべてのユーザーに完全提供されていません
- 複数のバックグラウンドタスクが同じファイルを編集する場合はコンフリクトが発生する可能性があります。同時に複数のエージェントを実行する場合は、ディレクトリを分けてスコープを限定してください

## まとめ

Claude Code のバックグラウンドセッションは、長時間かかる規則に従ったタスク（リファクタリング、テスト修正、コードマイグレーション、ビルド監視）に最も効果を発揮します。製品方向性に関わる曖昧な判断が必要なタスクは、引き続き対話型で進めるのが適切です。

明確にスコープが定まった、制約の整った指示から始めてください。指示が具体的であるほど、自律的な実行の信頼性が高まります。
