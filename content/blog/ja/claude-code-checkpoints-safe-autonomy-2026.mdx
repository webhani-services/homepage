Anthropic が 2026年7月初旬にリリースした Claude Code v2.1.200 には、変更履歴の上では小さく見えて、実際にはエージェント運用のあり方を変える 2 つの変更が含まれています。1 つ目は、`AskUserQuestion` ダイアログがデフォルトで自動継続しなくなったこと。エージェントが勝手に選択して先へ進むのではなく、人間の回答を待つようになりました。2 つ目は、これまで「default」と呼ばれていたパーミッションモードが、CLI・VS Code 拡張・JetBrains プラグインを通じて **Manual** という名称に変わったことです。あわせて、Anthropic は自律開発向けの **チェックポイント** 機能を展開しています。

一見するとこれらは「手間が増えた」ように読めます。しかし実際には、より長い自律実行を信頼できるものにするためのガードレールです。本質的な問いは「エージェントをどこまで自律させられるか」ではなく、「誤った判断をどれだけ安く取り消せるか」です。チェックポイントはこの問いに答えます。

## チェックポイントが実際に提供するもの

チェックポイントとは、意味のある作業単位を始める前にエージェントが取得する「復元可能なスナップショット」です。エージェントが誤った方向に 3 ステップ進んでしまった場合（契約を壊すリファクタ、誤ったファイルの編集、失敗テストの読み違えなど）、絡み合った変更を手作業で戻す代わりに、チェックポイントまで巻き戻します。

これは自律性の経済性を変えます。従来、エージェントの誤りのコストは「気づくまでにどれだけ逸れたか」に比例していました。チェックポイントがあれば、そのコストは「直近のスナップショットまでの距離」で上限が決まります。「信頼するには全 diff を読まないといけない」から「走らせておいて、悪い区間だけ数秒で巻き戻せばいい」への転換です。

チェックポイントは Git の代替ではありません。Git は永続的でレビュー可能な履歴、チェックポイントはエージェント自身の探索のためのセッション内の一時的な安全網です。両方を使います。

```bash
# 永続的な境界: エージェントにタスクを渡す前にコミット
git add -A && git commit -m "checkpoint: 依存関係アップグレード前"

# エージェントは作業しながらセッション内チェックポイントを作成する。
# 区間が失敗したら、git コミットまで全部戻すのではなく、
# エージェントのチェックポイントまで巻き戻す。
```

## デフォルト名が「Manual」であることの意味

デフォルトのパーミッションモード名を「Manual」に変えたのは、単なるラベル変更ではなくシグナルです。運用者に対して明確にこう伝えています——「初期状態では、エージェントは行動する前に確認を取る」。自律性はプロジェクトごとに意図的にオプトインするものであり、暗黙のうちに引き継ぐものではありません。

これは正しいデフォルトだと考えます。チームがコーディングエージェントを導入するときに最もよく見る失敗は、テストもなく、明確なパーミッション境界もなく、ロールバックの予行演習もしていないリポジトリで、いきなり完全自律に飛びつくことです。エージェントは一見もっともらしい何かを実行し、誰もレビューしていない場所に着地し、チームは数ヶ月ツールへの信頼を失います。

有用なメンタルモデルは 3 段階で、`settings.json` に落とし込めます。

```json
{
  "permissions": {
    "mode": "manual",
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint)",
      "Bash(git diff:*)",
      "Bash(git log:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(curl:*)"
    ]
  }
}
```

読み取り専用かつ冪等なコマンドは `allow` に入れ、テスト実行のたびに割り込まれないようにします。破壊的・外向きのコマンドは `deny` に。それ以外はすべて手動確認にフォールバックします。`AskUserQuestion` の変更もこれを補強します。エージェントが本当の判断点に到達したら、推測せずに立ち止まって尋ねるのです。

## ロールバック前提の自律ループ

クライアントに推奨しているパターンは、チェックポイントとコミットをループの背骨として扱い、後付けにしません。

1. **クリーンなベースラインをコミットする。** エージェントは常に「そのまま出荷できる状態」から始めるべきです。
2. **タスクを狭くスコープする。** 「このモジュールのエラーハンドリングをリファクタし、既存テストをグリーンに保つ」は「エラーハンドリングを改善する」より優れています。狭いスコープは、巻き戻し先のチェックポイントが近いことを意味します。
3. **作業しながらチェックポイントを取らせる。** 各サブステップが巻き戻しポイントになります。
4. **成果は雰囲気ではなくテストでゲートする。** テストスイートが通って初めてループは「完了」です——エージェントがあなた抜きで検証できるシグナルです。
5. **全キーストロークではなく最終 diff をレビューする。** グリーンなスイートとクリーンなベースラインがあれば、ライブ配信を見張る代わりに、一貫した 1 つの変更をレビューできます。

```bash
# 「無人だが境界のある」実行に向く形
git commit -am "baseline: feature ブランチ クリーン"

claude --permission-mode manual \
  "src/lib/http.ts のリトライロジックを指数バックオフに
   リファクタして。既存テストは全て通すこと。
   public な関数シグネチャは変更しないこと。"

# エージェントはステップ間でチェックポイントを取りながら作業する。
# 成功: diff を 1 つレビュー。失敗: 巻き戻して再スコープ。
npm run test && git diff
```

「public な関数シグネチャは変更しないこと」という制約はここで実際に効いています。プロンプト内の明示的な境界は、タスク固有の制限にはパーミッション設定より安く、エージェントに明確な失敗条件を与えます。

## それでも自律が破綻する場面

チェックポイントは誤りを安く取り消せるようにしますが、エージェントの判断そのものを健全にするわけではありません。以下は依然として人間の判断です。

- **ビジネス文脈を伴う判断。** その機能が存在すべきか、2 つの妥当な挙動のどちらが正しいか、競合する制約をどう優先するか。
- **不可逆な副作用。** ローカルワークスペースの外に出るもの——メール送信、本番データの変更、パッケージ公開。送信済みのリクエストはチェックポイントでは巻き戻せません。
- **静かなスコープ拡大。** バグ修正を頼まれたエージェントが「親切に」隣接コードをリファクタしてしまうことがあります。狭いプロンプトと厳格な `deny` リストがこれを抑えます。

`AskUserQuestion` がデフォルトで待機するようになった変更は、まさに 1 つ目のカテゴリを狙っています。本当の分岐点で、エージェントはもっともらしい推測に踏み切るのではなく、人間のために立ち止まるようになりました。

## Webhani の見解

重要なのは「エージェントがより自律的になっている」という捉え方ではなく、「エージェントを取り巻くツーリングが、自律性を可逆かつ監査可能にするのが上手くなっている」という捉え方です。チェックポイント、Manual デフォルト、ブロッキングな質問ダイアログは、すべて同じ方向を指しています——境界のある、回復可能な空間の中でエージェントを速く動かす、という方向です。

今から始めるチームには 3 点を勧めます。Manual デフォルトを維持し、自信がつくにつれて `allow` リストを段階的に広げること。エージェントのタスクごとにクリーンなベースラインをコミットし、チェックポイントに戻り先を用意すること。そして依存する前に、意図的にロールバックを一度予行演習すること——テストされていない復旧経路は復旧経路ではありません。この 3 つを押さえれば、自律コーディングは「信仰の飛躍」ではなくなり、ワークフローの当たり前で退屈な一部になります。それこそが目指すべき状態です。
