## 何が変わったのか

Claude Code に「Dynamic Workflows」と呼ばれる新機能がリサーチプレビューとして公開されました。これまでの AI コーディングツールは、ファイル単位・関数単位のタスクを一つずつ進める「ペアプログラマー」的な動き方をしていました。Dynamic Workflows はその前提を根本から変えます。

単一の Session 内で数百の並列 Subagent を起動し、大規模な Codebase 全体にわたるタスクを Claude が自律的に完遂できるようになります。

---

## Dynamic Workflows のアーキテクチャ

Dynamic Workflows が実行するプロセスは次の 4 フェーズに分かれます。

**1. 計画フェーズ**  
Claude がタスク全体を分析し、並列実行可能なサブタスクに分解します。依存関係のマッピングも行い、どの Agent が何を担当するかを決定します。

**2. 並列実行フェーズ**  
数百の Subagent が同時に起動し、それぞれが担当ファイルやモジュールを処理します。Agent 間の競合は Claude が管理します。

**3. 検証フェーズ**  
既存の Test Suite を実行し、すべての変更が仕様を満たしているかを確認します。テスト通過が完了条件のバーとなります。

**4. 統合フェーズ**  
検証済みの変更をまとめてユーザーに提示し、最終確認を求めます。

重要なのは、Claude が「指示された通りに実行する」だけでなく、**テストの全通過を自ら完了基準として採用する**点です。これにより、人間が逐一確認しなくても品質のバーが担保されます。

---

## ユースケース：Codebase スケールの Migration

最もインパクトが大きいユースケースが、Codebase スケールの Migration です。たとえば次のようなシナリオを考えてみてください。

- 10万行を超える Node.js プロジェクトを ESM に移行する
- 古い ORM から新しい ORM に全体を変換する
- 非推奨 API の呼び出しをすべてのファイルで置換する

これらは人間のエンジニアが手作業で進めると数週間かかるタスクです。Dynamic Workflows を使えば、指示とテストを用意するだけで Claude が全工程を処理します。

```bash
# Claude Code での典型的な指示例
> このプロジェクト全体を CommonJS から ESM に移行してください。
  既存のテストは引き続き全通過させること。

# Claude が実行するプロセス（自律的）
[計画立案]
  - 200個の .js ファイルを解析
  - require() → import() の変換箇所を特定
  - __dirname / __filename の代替パターンを選定

[Subagent 200個を並列起動]
  Agent-001: src/index.js → 処理完了
  Agent-002: src/utils/logger.js → 処理完了
  ...
  Agent-200: tests/integration.test.js → 処理完了

[Test Suite 実行]
  300/300 テスト通過

[完了報告]
  全 300 テストが通過しました。変更内容をレビューしてください。
```

## テスト Suite がカギになる理由

Dynamic Workflows が画期的なのは、AI が自分でチェックポイントを設けられる点です。

従来の AI コーディングでは、「正しいコードを書く」ことが目標でした。Dynamic Workflows では、**「既存のテストが全通過する」ことが完了条件**です。これは実務上、非常に重要な意味を持ちます。

テストを書いているプロジェクトほど、AI が安全に大規模変更を加えられます。逆に言えば、テストカバレッジが低いプロジェクトでは Dynamic Workflows の恩恵は限定的です。AI ツールの進化に合わせて、テスト戦略も見直す時期に来ています。

---

## 従来の AI コーディングとの比較

| 観点 | 従来のアプローチ | Dynamic Workflows |
|------|----------------|------------------|
| 作業単位 | ファイル / 関数 | Codebase 全体 |
| 並列性 | なし | 数百の Subagent |
| 完了判断 | ユーザーが逐一確認 | Test Suite 自動検証 |
| 人間の役割 | 逐次レビュー | 仕様定義・最終確認 |
| 最適なタスク | 機能追加、バグ修正 | Migration、一括変換 |

## 実践的な準備

Dynamic Workflows を最大限に活用するためには、プロジェクト側の準備が必要です。

**テストカバレッジを高める**  
テストが AI の「判断基準」になります。Unit Test だけでなく Integration Test も整備しておくと、より複雑な Migration に対応できます。

**テストを仕様として書く**  
「このファイルが何をすべきか」がテストから明確に読み取れるようにしておくと、Claude の計画フェーズの精度が上がります。

**段階的に試す**  
最初からプロジェクト全体を対象にするのではなく、単一モジュールや小さなサブシステムから Dynamic Workflows を試すと、挙動を把握しやすくなります。
