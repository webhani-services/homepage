自律的にコードを書くエージェントをチームに導入するとき、最初に問題になるのは「どこまで自由にやらせるか」です。Subagent が別の Subagent を無制限に生成し、承認なしに危険な操作を実行できてしまうと、レビューの網をすり抜ける経路が生まれます。2026年7月の Claude Code アップデートは、この統制のための実用的な仕組みをいくつか追加しました。本記事では、パラメータ単位の権限ルール、Dynamic workflow size 設定、Subagent 起動前の分類器チェックを整理します。

## Tool(param:value) — パラメータ単位の権限ルール

これまでの権限ルールは Tool 単位でした。`Bash` を許可するか拒否するか、という粒度です。新しく `Tool(param:value)` 構文が加わり、Tool の入力パラメータの値でマッチングできるようになりました。`*` ワイルドカードも使えます。

代表的な使い方は、Subagent が使うモデルの制限です。コスト管理のために、Opus を使う Subagent の生成をブロックしたいケースがあります。

```json
{
  "permissions": {
    "deny": [
      "Agent(model:opus)"
    ]
  }
}
```

`.claude/settings.json` にこのルールを置くと、`model: opus` を指定した Subagent 起動リクエストが拒否されます。逆に、特定のモデルだけを許可する運用も可能です。

```json
{
  "permissions": {
    "allow": [
      "Agent(model:sonnet)",
      "Agent(model:haiku)"
    ],
    "deny": [
      "Agent(model:*)"
    ]
  }
}
```

この例では、Sonnet と Haiku の Subagent は許可し、それ以外のモデル指定はすべて拒否します。`deny` が `allow` より優先されるわけではなく、より具体的なルールが評価されるため、明示的に許可したモデル以外がブロックされる形になります。

同じ考え方は Bash コマンドの引数にも応用できます。たとえば本番環境への `kubectl` 操作だけを止めたい場合、コマンド文字列のパターンでマッチさせられます。ワイルドカードを使うことで「特定のフラグを含む呼び出しだけを拒否する」といった細かい制御が現実的になります。

## Subagent の起動前チェック — 分類器による事前評価

Auto モード（許可判断を自動化するモード）には、以前まで一つ穴がありました。Subagent が起動した後で、その中からブロック対象の操作をリクエストできてしまう経路です。親エージェントの権限チェックをすり抜けて、子エージェントが危険な操作に到達する余地がありました。

2026年7月のアップデートで、Subagent の起動リクエスト自体が起動前に分類器で評価されるようになりました。子エージェントに何をやらせようとしているのかを、生成する前に判定します。これにより「まず Subagent を作ってしまえば、あとは中で自由にできる」という抜け道が塞がれました。

実務的な意味は明確です。権限ルールを親エージェントに設定していても、Subagent 経由で回避されないという保証が一段強まります。CI 上で無人実行するようなワークフローでは、この事前チェックの有無が安全性を大きく左右します。

## Subagent の再委譲を抑制

もう一つの改善は、Subagent が受け取ったタスクをまるごと別の Subagent に再委譲しにくくなった点です。以前は、タスクを渡された Subagent が「これは自分の仕事ではない」と判断して、そのまま次の Subagent に丸投げする挙動が見られました。委譲の連鎖が深くなると、実際に誰が何をやったのかが追いにくくなり、トークン消費も膨らみます。

再委譲が抑制されると、タスクを受け取った Subagent がその場で作業を完了する傾向が強まります。結果として、実行経路が浅くなり、ログの追跡性が上がります。チームで運用する場合、この「経路の浅さ」はレビューコストに直結するため、地味ですが重要な変更です。

## Dynamic workflow size — 動的ワークフローの規模を制御する

`/config` に `Dynamic workflow size` 設定が追加されました。Claude が動的に生成するワークフローの規模（生成する Subagent の数）を small / medium / large で指定します。

重要なのは、これが「強制的な上限」ではなく「助言的なガイドライン」である点です。large を選んでも、タスクの性質によっては少数の Subagent で済むことがありますし、small を選んでも必要なら増えます。あくまで「だいたいこのくらいの規模で動いてほしい」という方向づけです。

チームでの使い分けの目安は次の通りです。

- **small**: 単純な修正、レビュー、調査。トークンコストを抑えたい日常タスク。
- **medium**: 複数ファイルにまたがる実装や、並行して進められる中規模タスク。
- **large**: 大規模なリファクタリング、網羅的な監査、移行作業など、並列化で時間を稼ぎたい場面。

CI で無人実行するパイプラインでは small〜medium に寄せ、対話的に大きな作業を任せるときだけ large に上げる、という運用が現実的です。規模の指定はコストの予測可能性に直結するため、プロジェクトの `.claude/settings.json` でデフォルトを固定しておくとチーム内のばらつきが減ります。

## webhani の視点 — 統制は「後付け」では効かない

これらの機能に共通するのは、エージェントの自律性とレビュー可能性のバランスを取るための仕組みだという点です。私たちが顧客の開発チームに AI コーディングを導入する際、最初に整備するのは権限の境界です。何を許可し、何を止めるかを最初に決めておかないと、エージェントの出力を後からレビューで捕まえる運用になり、必ず取りこぼしが出ます。

具体的には、以下を推奨しています。

1. `.claude/settings.json` の `deny` リストに、本番環境への書き込み系操作とコスト影響の大きいモデル指定を明示する。
2. `Dynamic workflow size` のデフォルトをプロジェクト単位で固定し、CI では small〜medium に寄せる。
3. Auto モードを使う場合は、分類器の事前チェックが有効なバージョンであることを確認する。

権限ルールを設定ファイルに書いておくことは、単なる安全策ではありません。「このプロジェクトではエージェントに何を任せるか」という意思決定をコードとして残すことでもあります。チームの誰が実行しても同じ境界が適用されるため、レビューの負荷が下がり、監査もしやすくなります。

## まとめ

2026年7月の Claude Code アップデートは、エージェントを個人の道具からチームの道具へと押し上げる方向の変更です。`Tool(param:value)` による細かい権限制御、Subagent 起動前の分類器チェック、再委譲の抑制、`Dynamic workflow size` — これらはどれも派手さはありませんが、複数人で自律エージェントを運用するときに効いてきます。導入初期のうちに権限の境界を設定ファイルとして固め、規模のデフォルトを決めておくことが、安全に自律性を引き出す近道です。
