## エージェントは「コードを書く道具」から「仕事をこなす同僚」へ

ここ2年、AIアシスタントの進化はコーディング支援を中心に語られてきた。だが2026年に入り、その重心は明確に移り始めている。Anthropic の Claude Cowork は、タスクを一つ渡すと、ファイル、カレンダー、メール、メッセージングアプリ、Web、そして接続した外部ツールを横断しながら、完了まで作業を進める仕組みだ。IDE の中でコードを補完する段階から、業務そのものを代行する段階へと踏み出している。

webhani ではクライアントの業務自動化を数多く手がけてきたが、この変化は単なる機能追加ではなく、AI をどうシステムに組み込むかという設計思想の転換だと捉えている。本稿では、Cowork が象徴するエージェント型ワークフローの実務的な意味と、導入時に本当に効いてくる論点を整理する。

## 「ツールを横断する」ことの技術的な意味

従来のチャット型 AI は、ユーザーが貼り付けた文脈の中だけで回答を返す閉じた存在だった。エージェント型では前提が変わる。モデル自身が「次に何のツールを呼ぶべきか」を判断し、実行結果を受け取って次の行動を決める。このループを人間の逐次確認なしに回せることが本質だ。

その接続層を標準化しているのが MCP（Model Context Protocol）である。Cowork のようなエージェントが外部ツールと安全に対話できるのは、ツールの能力を宣言的に記述する共通インターフェースがあるからだ。自社ツールをエージェントに繋ぐ場合も、この宣言的な記述が起点になる。

```json
// エージェントに接続するツールを宣言的に定義する例（MCP サーバー設定）
{
  "mcpServers": {
    "internal-crm": {
      "command": "node",
      "args": ["./servers/crm-server.js"],
      "env": { "CRM_API_BASE": "https://crm.internal.example.com" }
    },
    "billing": {
      "command": "python",
      "args": ["-m", "billing_mcp"],
      "env": { "BILLING_SCOPE": "read-only" }
    }
  }
}
```

重要なのは、ツールごとに権限スコープを明示できる点だ。上の例では請求システムを `read-only` に限定している。エージェントが自律的に動く以上、「何ができないか」を設計段階で縛ることが、そのまま安全設計になる。

## 自律実行のデフォルト化が意味すること

2026年7月のアップデートで、Claude Code は Amazon Bedrock、Google Vertex AI、Microsoft Foundry の各基盤上で自律実行モード（auto mode）をデフォルト有効にした。あわせて Bedrock 側のモデルも Opus 4.8 に更新されている。

デフォルトが「都度確認」から「自律実行」に変わったことは、地味だが象徴的だ。これまでは人間が一手ごとに承認するのが安全側の初期設定だった。それが逆転したということは、モデルの判断精度と、失敗時のロールバック設計が実用水準に達したという開発元側の判断を意味する。

ただし実務では、デフォルトを鵜呑みにしないことを勧めたい。自律実行の恩恵が大きい作業（定型的なリファクタリング、ログ調査、テスト生成）と、慎重な確認が必要な作業（本番データの更新、外部への送信）を仕分けし、後者には明示的なガードを置く。エージェントの生産性は「どこまで任せるか」ではなく「どこで止めるか」を設計できているかで決まる。

## 企業導入で効くのはモデル性能ではなくガバナンス

Anthropic は同時期に、Claude Code と Cowork を政府機関向けデスクトップ（FedRAMP High 認証）でパブリックベータとして提供し始めた。ここで前面に出ているのは、モデルの賢さではなく、改ざん検知が可能な監査ログ、管理者による制御、支出ガバナンスといった運用機能だ。

この優先順位は示唆的だ。エージェントを組織で使う際、経営や情報システム部門が最初に問うのは「どれだけ賢いか」ではなく、次の三点である。

- **誰が、いつ、何をエージェントに実行させたか追跡できるか**（監査可能性）
- **エージェントが触れられる範囲を職務単位で制限できるか**（最小権限）
- **暴走やコスト超過をどう検知・遮断するか**（支出とレート制御）

webhani がクライアントにエージェント基盤を提案する際も、この三点を満たせない構成は本番投入を見送るよう助言している。生産性の話は、統制が取れて初めて意味を持つ。

## 現場でどう組み立てるか

エージェント型ワークフローを実務に落とすうえで、私たちが有効だと考えるのは次の順序だ。

1. **狭く始める。** まずは読み取り専用・低リスクなタスク（社内ドキュメント検索、レポート下書き）から着手し、失敗コストの低い領域でエージェントの挙動を観察する。
2. **権限を宣言的に固定する。** どのツールに、どのスコープでアクセスさせるかをコード（設定ファイル）として管理し、レビュー対象にする。
3. **観測点を先に用意する。** エージェントの各アクションをログ化し、後から追跡できる状態を、機能追加より先に作る。
4. **人間の承認ゲートを明示的に置く。** 不可逆な操作（削除、送信、決済）の前には、自律実行であっても停止して確認を挟む。

```text
[タスク投入] → [計画立案] → [read-only ツールで調査]
     → [承認ゲート: 不可逆操作の直前で停止] → [実行] → [監査ログ記録]
```

この形にしておけば、モデルが Opus 4.8 から次世代へ更新されても、統制の枠組みは変えずに性能だけを取り込める。

## まとめ

Claude Cowork が示しているのは、AI エージェントが「賢い補完機能」から「業務を横断して完遂する実行主体」へ移行したという事実だ。技術的な鍵は MCP による標準化された接続層であり、ビジネス的な鍵は監査・権限・支出のガバナンスにある。

webhani の見立てとして、2026年後半のエージェント導入は、モデルの性能比較よりも「どこまで安全に任せられる構成を組めるか」で成否が分かれる。狭く始め、権限を宣言的に縛り、観測と承認ゲートを先に用意する——この地味な設計こそが、エージェント時代の生産性を実際に引き出す土台になる。
