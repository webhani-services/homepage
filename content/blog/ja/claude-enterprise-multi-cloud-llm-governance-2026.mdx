1年前、クライアントが持ち込むLLMの相談はほとんどがプロンプティングの話でした。どのモデルを使うか、Temperatureをいくつにするか、手強いタスクからどうやってより良い答えを引き出すか。2026年になってもそれらの問いが消えたわけではありませんが、エンジニアリングマネージャーの眠りを妨げる問いはもう別のところにあります。それは組織的な問いです。誰が、どのモデルを、どのクラウド経由で、どの予算で呼べるのか。そして何かがおかしくなったとき、私たちはどうやってそれに気づくのか。

この変化は、モデルの届けられ方が実際に変わったことと結びついています。たとえばAnthropicのClaudeモデルは、いまやベンダー自身のAPIだけでなく、Amazon BedrockやGoogle Cloud内のマネージド提供としても利用でき、Anthropicはコスト可視化のための管理者向け統制を追加し続けています。主要なモデルプロバイダー全般でおおむね同じ状況です。その結果、一つの会社が*同じ*モデルファミリーを三つの異なる正面玄関から呼ぶことになり、各玄関はそれぞれの課金、IAMモデル、ロギングを持ちます。ガバナンスとは、その玄関たちを一つに束ねる仕事です。本記事は、その作業のためにクライアントと一緒に使っているフレームワークです。

## マルチクラウドのLLMアクセスは、計画しなくても生まれる

誰も最初からモデルアクセスを断片化しようとはしません。それは少しずつ起こります。データチームはすでにGoogle Cloudで働いているので、同じVPC内に留まるためにパイプラインはそこでモデルを呼びます。アプリケーションチームはAWSに深く入っているので、トラフィックをプライベートネットワークに保ち既存のIAMロールを引き継ぐために、本番サービスはBedrockを使います。その間、プロダクトマネージャーはプロトタイピングのためにベンダー直のAPIに登録し法人カードで決済したものの、そのキーはいつのまにか四ヶ月動き続ける「一時的な」社内ツールを静かに支えています。

一つひとつの判断は局所的には妥当です。データと同じクラウド内で推論すればEgressコストとデータレジデンシーの頭痛を避けられます。プラットフォーム標準の提供を使えば、すでに信頼している監査ログ、鍵管理、ネットワークポリシーを再利用できます。しかし組織全体で合算すると、互いに無関係な三本の課金ライン、三つの権限体系、そして「今月AIにいくら、何に使ったのか」に答える単一の場所がない状態になります。

よくある間違いは、この状況を「なくすべき問題」として扱うことです。前述の理由が本物なので、全員を一つの経路に統合することはできません。目標は断片化をなかったことにするのではなく、断片化の*上に*ガバナンスのレイヤーを載せることです。

## ゲートウェイパターン：正面玄関は一つ、裏側は複数

もっとも有用なアーキテクチャ上の一手は、各サービスにプロバイダーを直接呼ばせるのではなく、アプリケーショントラフィックを社内ゲートウェイに流すことです。ゲートウェイは薄いサービスです。自作でも既製品でも、正規化されたリクエストを受け取り、どの裏側を使うか決め、適切な資格情報を付け、何が起きたかを記録します。

概念的にはこうなります。

```typescript
// すべての社内呼び出し元は、プロバイダーSDKではなくゲートウェイを叩く。
async function complete(req: LlmRequest, ctx: CallerContext) {
  const route = pickRoute(req.model, ctx.team); // 例: bedrock | gcp | vendor
  const started = Date.now();

  const res = await route.client.send(req);

  await recordUsage({
    team: ctx.team,
    model: req.model,
    route: route.name,
    inputTokens: res.usage.inputTokens,
    outputTokens: res.usage.outputTokens,
    latencyMs: Date.now() - started,
  });

  return res;
}
```

ここに特別なものはありません。そしてそれこそが要点です。すべての呼び出しが一つの関数を通れば、以前は不可能だった四つのことが些細になります。チーム単位のコスト帰属、呼び出し元に触れないルート切り替え、一箇所でのポリシー適用、統合された利用ログの確保です。`pickRoute`がマルチクラウドの現実を吸収する地点です。データチームのリクエストはGoogle Cloudへ、本番リクエストはBedrockへ解決され、呼び出し元はそれを知る必要も気にする必要もありません。

ゲートウェイはフェイルオーバーの継ぎ目も与えます。ある裏側がエラーを返したりレートリミットに掛かったりしたら、`pickRoute`は呼び出し元のコード変更なしに同じモデルの別プロバイダーへ切り替えられます。私たちはこれをまず信頼性の機能として、次にコストの機能として捉えます。

## コスト可視化：大きなダッシュボードより帰属が先

クライアントが「AIの請求が制御不能だ」と言うとき、問題はほぼ常に絶対値ではありません。誰もそれを分解できないことが問題です。月4万ドルが「本番推論2.8万、分析バッチ9千、実験3千」ときれいに割れれば、それは扱える経営の入力値です。同じ4万ドルが不透明な一行なら、不安と誤った判断の源になります。

プロバイダー側のコスト統制、つまりクラウドやベンダーがいま公開する管理画面は、必要ですが十分ではありません。それは*その*プロバイダーがいくら請求したかは教えてくれますが、分析チームの夜間ジョブが原因だとは教えてくれません。プロバイダーはあなたの組織図を知らないからです。帰属はあなたの側で起こさねばならず、上の`recordUsage`呼び出しがまさにそれを捉えます。チーム別・モデル別のトークン数を自前のストアに持っていれば、月次集計は単純なクエリです。

```sql
SELECT team,
       model,
       sum(input_tokens)  AS input_tokens,
       sum(output_tokens) AS output_tokens,
       count(*)           AS calls
FROM llm_usage
WHERE created_at >= date_trunc('month', now())
GROUP BY team, model
ORDER BY output_tokens DESC;
```

このトークン数に各ルートの公開トークン単価を掛ければ、クラウドの請求書を待たずにチーム別の推定コストが得られます。この数字は請求書とセント単位までは一致しません。プロバイダーごとに丸めやまとめ方が違うからです。しかし方向は正確で、何より実時間で得られます。暴走するジョブを請求書ではなく二日目に捕まえられるのは、まさにこの点のおかげです。

## プロンプトではなくゲートウェイに置くべきガードレール

すべての呼び出しが一箇所を通るので、ゲートウェイはサービスごとに散らすと苦しくなるいくつかの統制の、正しい住処でもあります。

- **牙のある予算。** 誰かにメールを送るだけの緩い予算は助言にすぎません。ゲートウェイでチーム別支出を月次上限と比べて追跡し、上限に達したら非必須リクエストを拒否または格下げしてください。社内ツールを鈍らせるほうが、事後に五桁の超過を発見するよりずっとましです。
- **チーム別のモデルAllowlist。** 安いモデルで足りるタスクに、すべてのチームが最も高価なモデルを呼べる必要はありません。それを人が忘れるコードレビューの慣習ではなく、`pickRoute`のポリシーとして刻んでください。
- **機微データのためのロギング境界。** ゲートウェイは、特定のリクエスト区分を自社クラウドとネットワーク内の裏側だけへ送るよう強制するのに自然な場所です。規制データを望まない経路へ漏らさず守ります。

私たちはこれらの統制を意図的にプロンプトと個々のサービスの外に置きます。プロンプト水準の規則は簡単に迂回され監査もできず、サービス別の規則は二チームがそれぞれ実装した瞬間からずれ始めます。ゲートウェイは、一度書いたポリシーが実際に保たれる唯一のチョークポイントです。

## いま始めるチームへの推奨

始めたばかりなら、重厚なプラットフォームを建てたい衝動を抑えるべきです。死角をなくす最小のものから始めてください。

1. **すべてのアプリケーショントラフィックの前に薄いゲートウェイを置く。** 百行のサービスでも、利用ログと帰属を始めるには十分です。要件がはっきりしたら、後で成熟したオープンソースのゲートウェイを採用すればよいのです。
2. **初日から自前のストアに利用量を記録する。** 今月捉え損ねたトークン数は永遠に失われます。最小の労力で最大のてこを得る一歩です。
3. **持っているプロバイダー側のコスト統制をすべてオンにする。** ただし主たる防衛線ではなく、安全網として。主たる防衛線はあなた自身の帰属です。
4. **どのチームがどのクラウドでどのモデルを使ってよいかを書き出し、** そのリストをルーティングに刻む。書かれていないポリシーはポリシーではありません。

2026年にLLMガバナンスをうまく扱う組織は、最も賢いプロンプトを持つところではありません。モデルアクセスはインフラだと早くに結論づけ、すでにデータベースやクラウドコンピュートに適用していた帰属・予算・アクセスの規律を同じように与えたところです。ゲートウェイは、実在するチームが必ず生み出す断片化を凍らせずに、そこへ到達する方法です。

*LLMの支出が説明しづらくなった、あるいはモデルアクセスがいつのまにかクラウドをまたいで断片化した——ゲートウェイ設計からコスト帰属、アクセスポリシーまで、これはまさにwebhaniがクライアントの絡んだ糸をほどく類のアーキテクチャ作業です。*
