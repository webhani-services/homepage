これまでの AI アシスタントとの会話は、セッションをまたいで完全にリセットされるのが常だった。「私は TypeScript エンジニアです」「このプロジェクトは Next.js を使っています」——そのような背景説明を毎回繰り返す手間は、AI を日常的に使う開発者なら誰もが感じてきた摩擦だ。

Anthropic は 2026年3月、Claude に **Persistent Memory（永続的な記憶）** 機能を段階的に展開し始めた。これにより Claude は、会話セッションをまたいで、ユーザーの名前・好み・作業スタイル・進行中のプロジェクト情報などを記憶・活用できるようになった。

## 何が変わるのか

従来の Claude は各会話が独立したコンテキストウィンドウ内で完結していた。どれだけ詳しいシステムプロンプトを設定しても、新しい会話を始めれば白紙に戻る仕組みだった。Memory 機能は、ユーザーが明示的に「覚えておいてほしい」と伝えた情報（または Claude が重要と判断した情報）を、会話をまたいで保持する。

具体的には以下のような情報が対象になる：

- **ユーザーの基本情報**: 名前、職種、使用言語・フレームワーク
- **作業スタイルの好み**: コードコメントの量、説明の詳細度、好みのアーキテクチャパターン
- **進行中のプロジェクト**: リポジトリ名、技術スタック、開発中の機能
- **これまでの意思決定**: 「このプロジェクトでは PostgreSQL を選択した」「認証は Clerk を使う予定」

## 開発者にとっての実際の変化

最も効果を感じるのは**繰り返しのコンテキスト設定が不要になる**点だ。

**Memory 導入前:**

```
あなた: 私は TypeScript を使う Web エンジニアです。今は Next.js 16 +
        Neon PostgreSQL + Clerk のスタックで SaaS を開発しています。
        データベースのスキーマについて質問があります。

Claude:  Next.js SaaS アプリの PostgreSQL スキーマについてですね。
```

**Memory 導入後:**

```
あなた: DB スキーマについて相談したいのですが。

Claude:  先日お話しした SaaS プロジェクトのことですね。
         Neon PostgreSQL と Clerk を使うあのプロジェクトでしたら...
```

この変化は一見小さく見えるが、日常的に AI を使う開発者にとっては無視できない効率化だ。毎日 10 分の文脈説明が不要になれば、週 50 分の節約になる。コンテキストを把握した状態から始められるため、Claude の返答の質も上がる傾向にある。

## Claude Sonnet 4.6 と 1M トークンコンテキストウィンドウ

3月のアップデートでは Memory 機能に加え、**Claude Sonnet 4.6** のリリースも行われた。注目点は **1M トークンのコンテキストウィンドウ（ベータ）** の対応だ。

コンテキストウィンドウ 1M トークンは、約 75 万文字分のテキストを一度に処理できることを意味する。大規模なコードベース、長期的なプロジェクト文書、複数ファイルにわたる設計書——これらを丸ごとコンテキストに含めた質問が現実的になる。

```typescript
// 大規模コードベースの一括分析例
const response = await anthropic.messages.create({
  model: "claude-sonnet-4-6",
  max_tokens: 8096,
  messages: [
    {
      role: "user",
      content: `以下がリポジトリ全体のソースコードです:

      ${entireRepositoryContent}

      N+1 クエリパターンをすべて特定し、改善案を提案してください。`,
    },
  ],
});
```

ただし、1M トークン規模のコンテキストは現状ベータであり、コストとレイテンシを慎重に考慮する必要がある。すべての用途に適しているわけではなく、日常的な質問では必要なコードスニペットに絞ったプロンプトが引き続き効率的だ。

## プライバシーとセキュリティの考慮

Memory 機能を使う上で、プライバシーの観点から把握しておくべき点がある。

- **メモリの内容は確認・削除が可能**: Claude の設定画面から、何が記憶されているかを確認・編集・削除できる
- **機密情報の扱いに注意**: 顧客データ、API キー、内部システムの詳細などを Claude に「覚えさせる」ことは避けるべきだ
- **組織アカウントでの制御**: チームプランでは、管理者がメモリ機能の範囲を制御できる
- **Memory は個人単位**: あるユーザーのメモリは他のチームメンバーには共有されない

企業での利用においては、どの情報を AI に記憶させてよいかのポリシーを明示的に定めることが重要だ。

## Claude Code における /loop コマンド

同時期にリリースされた **Claude Code の `/loop` コマンド** も、開発ワークフローに直接影響するアップデートだ。

```bash
# 5分ごとにデプロイのステータスを確認
/loop 5m check the deploy status and notify if failed

# 開発中に影響を受けるテストを定期実行
/loop 2m run the affected test suite and summarize failures
```

これは現在のセッション内で動作する軽量な cron ジョブとして機能する。CI/CD パイプラインの監視、テストの定期実行、長時間処理の完了待ちといったユースケースに使える。別途ポーリングスクリプトを書かずに Claude Code セッション内で完結できる。

## 実用的な活用パターン

Memory 機能を有効活用するために、以下のアプローチが効果的だ：

1. **プロジェクト開始時に明示的に登録**: 「このプロジェクトは〜というスタックを使っています。以降の会話でも参考にしてください」と明示する
2. **コーディングスタイルの設定**: 好みのコード規約、コメントスタイル、エラーハンドリングパターンを一度伝えれば以降は自動適用される
3. **定期的な棚卸し**: 月に一度、何が記憶されているか確認し、古い情報（終了したプロジェクトや変更した技術選定）を更新する

## まとめ

Claude の Memory 機能は、AI をツールとして使う上での具体的な摩擦を取り除くアップデートだ。「毎回自己紹介する手間」という非常に明確な問題を解決している。

劇的な変革ではないが、AI を日常的に使う開発者にとっては着実に効いてくる改善だ。メモリの内容が透明に管理でき、削除も簡単という設計は信頼しやすい。チームでの利用ポリシーと合わせて、積極的に活用を検討する価値がある。
