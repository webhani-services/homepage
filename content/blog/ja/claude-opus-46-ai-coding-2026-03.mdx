2026 年 2 月、AI Coding Assistant の競争において一つの節目が訪れました。Claude Opus 4.6 が SWE-bench Verified で 80.8% を達成し、Gemini 3.1 Pro が Google として久しぶりにベンチマーク上位に返り咲きました。GPT-5.3-Codex も同週にリリースされ、コーディング特化モデルとして実力を示しています。

ただし、2026 年 3 月中旬の時点では大きな発表がなく、市場は一定の安定期に入りつつあります。週ごとにモデルが入れ替わる時代から、主要モデルの成熟期へと移行している印象です。

## 主要モデルの現状

### Claude Opus 4.6 と Sonnet 4.6

Opus 4.6 は SWE-bench Verified で 80.8%、Sonnet 4.6 は 79.6% を達成しています。注目すべきは Sonnet 4.6 のコストパフォーマンスで、Opus の 5 分の 1 のコストで 1.2% 差のスコアを実現しています。

実務での使い分け：

- **Opus 4.6**：複雑なバグ調査、大規模リファクタリング、アーキテクチャレベルの設計支援
- **Sonnet 4.6**：日常的なコード補完、ドキュメント生成、テストコード作成

Anthropic の研究者 Nicholas Carlini 氏が報告した事例では、16 体の Opus 4.6 エージェントが Rust で C コンパイラをゼロから書き、Linux カーネルをコンパイルできることを検証しました。エージェントが自律的に複雑なシステムを構築できる水準に達していることを示す一例です。

### GPT-5.3-Codex

2026 年 2 月 5 日にリリースされた GPT-5.3-Codex は、Codex CLI と Codex Web 向けに最適化されたコーディング専用モデルです。特に既存コードベースへの変更提案と、マルチファイルにわたるリファクタリングに強みを持ちます。

### Gemini 3.1 Pro

Google は Gemini 3.1 Pro でベンチマーク上位に復帰しました。長いコンテキストウィンドウと画像理解の組み合わせが強みで、設計書や ER 図を参照しながらコードを生成するユースケースで優位性を発揮します。

## Claude Code の実態

Anthropic が 2025 年末に出荷した Claude Code は、2026 年 3 月時点で唯一「実際にコードを書き、テストし、コミットする」自律的なコーディング環境として機能しています。

```bash
# Claude Code の基本的な使い方
claude code "認証モジュールのユニットテストを追加してください"

# ファイルを指定して改善を依頼
claude code --file src/auth/login.ts "このファイルのエラーハンドリングを改善してください"
```

実際のプロジェクトで有効な活用パターンを整理します。

### パターン 1：テスト駆動での機能追加

```
指示: "src/api/users.ts に対して、以下の仕様でユニットテストを書いてください。
テストが通るように実装も修正してください。

仕様:
- メールアドレスが不正な場合は ValidationError を throw する
- 既存のメールアドレスが存在する場合は ConflictError を throw する
- 正常ケースでは作成したユーザーオブジェクトを返す"
```

このように仕様を明確にして渡すことで、テストと実装を同時に生成させることができます。

### パターン 2：バグ調査の起点

```
指示: "以下のスタックトレースを見て、原因を特定してください。
修正案を提示する前に、まず原因の仮説をリストアップしてください。

[スタックトレース貼り付け]"
```

「修正してください」と一言で頼むより、仮説の列挙を求めることで見落としが減ります。

### パターン 3：レビューコメントへの対応

```
指示: "PR #142 のレビューコメントを読んで、指摘された箇所を修正してください。
変更内容を変更前・変更後で説明してください。"
```

## コスト感覚

API 利用の目安（2026 年 3 月時点）：

| モデル | 入力 (1M tokens) | 出力 (1M tokens) | SWE-bench |
|--------|-----------------|-----------------|-----------|
| Claude Opus 4.6 | $15 | $75 | 80.8% |
| Claude Sonnet 4.6 | $3 | $15 | 79.6% |
| GPT-5.3-Codex | $10 | $50 | 非公開 |

コーディングタスクのほとんどは Sonnet 4.6 で十分です。Opus 4.6 は「どうしても解決できない複雑な問題」に絞って使うのが費用対効果の観点から合理的です。

## 実務で注意すべき点

### 1. 生成コードを盲目的に信頼しない

SWE-bench スコアが高くても、実際の業務コードベースはベンチマークとは異なります。特に：
- 社内独自の設計パターンを反映できないことが多い
- テストのないコードに変更を加えると、意図しない副作用が発生する可能性がある
- セキュリティ要件（認証、入力検証）は必ず人間がレビューする

### 2. コンテキストの質が出力の質を決める

```
# 良くない指示
"ログイン機能を作ってください"

# 良い指示
"Next.js App Router の Route Handler として、
メールアドレスとパスワードでのログイン API を実装してください。
- 認証には JWT を使用（既存の lib/auth.ts を参照）
- エラーレスポンスは src/types/api.ts の ApiError 型に合わせる
- バリデーションには zod を使用"
```

### 3. 自律エージェントは監視が必要

Claude Code のような自律エージェントは、指示を読み違えてファイルを意図せず変更することがあります。`git status` と差分の確認を習慣化し、大きなタスクは小さなステップに分割して進めることを推奨します。

2026 年 3 月時点で、AI Coding Assistant は「実験的ツール」から「日常的な開発ツール」へと確実に移行しています。しかし、モデルスコアの競争よりも、チームとしてどう組み込むかのワークフロー設計の方が重要になってきています。

webhani では、特にテストカバレッジの低い既存コードベースに対して AI を活用したリファクタリングを支援しています。適切な指示設計と人間によるレビューの組み合わせが、現時点での最も現実的なアプローチです。
