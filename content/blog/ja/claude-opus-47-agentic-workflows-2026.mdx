## Claude Opus 4.7 の概要

Claude Opus 4.7 は 2026年4月16日にリリースされました。今回の更新で特筆すべきは SWE-bench Pro スコアです。前バージョン比で 10.9 ポイント向上し、1,865 件のテストセットにおいて約 200 件以上のソフトウェアエンジニアリングタスクを自律的に解決できるようになりました。

SWE-bench Pro は GitHub の実際の Issue を AI が自律的に解決する能力を測定するベンチマークです。単純なコード補完とは異なり、問題の理解、関連コードの調査、変更の実装、テスト通過まで一連のエンジニアリングサイクルを自律的にこなせるかを評価します。

## Agentic Workflow へのシフト

同時期に開催された Code with Claude 2026 カンファレンスでは、より重要な文脈が示されました。「LLM を使う」から「LLM で動くシステムを設計する」へのパラダイムシフトが主要テーマとなりました。

エージェントワークフローの特性は次の通りです：

- **Goal Orientation**: 最終目標から逆算してサブタスクを自律的に計画・実行する
- **Memory Retention**: 実行コンテキストをステップをまたいで保持・参照する
- **External System Interaction**: API 呼び出し、コード実行、ファイル操作など外部アクションを取る
- **Self-Correction**: 実行結果を評価し、目標に向けて計画を修正する

従来の「質問 → 回答」モデルから、「目標 → 計画 → 実行 → 検証 → 完了」のループ構造に変化しています。

## 実践的なエージェント設計パターン

### Plan → Execute → Verify ループ

```python
import anthropic
from typing import Any

client = anthropic.Anthropic()

TOOLS = [
    {
        "name": "execute_bash",
        "description": "Execute a bash command and return the output",
        "input_schema": {
            "type": "object",
            "properties": {
                "command": {"type": "string", "description": "Bash command to run"}
            },
            "required": ["command"]
        }
    },
    {
        "name": "write_file",
        "description": "Write content to a file",
        "input_schema": {
            "type": "object",
            "properties": {
                "path": {"type": "string"},
                "content": {"type": "string"}
            },
            "required": ["path", "content"]
        }
    }
]

def agent_loop(goal: str, max_steps: int = 15) -> str:
    messages: list[dict] = [
        {"role": "user", "content": goal}
    ]

    for _ in range(max_steps):
        response = client.messages.create(
            model="claude-opus-4-7-20260416",
            max_tokens=8192,
            tools=TOOLS,
            messages=messages
        )

        messages.append({"role": "assistant", "content": response.content})

        # エージェントが自律的に「完了」と判断した場合
        if response.stop_reason == "end_turn":
            for block in response.content:
                if hasattr(block, "text"):
                    return block.text
            return "Completed"

        # Tool 呼び出しを処理して結果を返す
        tool_results = []
        for block in response.content:
            if block.type == "tool_use":
                result = dispatch_tool(block.name, block.input)
                tool_results.append({
                    "type": "tool_result",
                    "tool_use_id": block.id,
                    "content": result
                })

        if tool_results:
            messages.append({"role": "user", "content": tool_results})

    return "Max steps reached"


def dispatch_tool(name: str, params: dict[str, Any]) -> str:
    import subprocess

    if name == "execute_bash":
        result = subprocess.run(
            params["command"], shell=True,
            capture_output=True, text=True, timeout=30
        )
        return (result.stdout + result.stderr)[-2000:]

    if name == "write_file":
        with open(params["path"], "w") as f:
            f.write(params["content"])
        return f"Written to {params['path']}"

    return f"Unknown tool: {name}"
```

`stop_reason == "end_turn"` でエージェントが自律的に「完了」と判断します。ツールを呼び続けるか、テキストで結論を述べるかはモデルが決定します。Tool 結果は `-2000` 文字でトリムしていることに注目してください。詳細なツール出力はコンテキストウィンドウを消費します。必要であればモデルが追加情報を要求します。

### 状態の永続化

マルチステップのタスクでは中間状態の管理が重要になります。

```python
import json
from pathlib import Path

def save_checkpoint(session_id: str, messages: list, completed: list[str]) -> None:
    path = Path(f".agent_sessions/{session_id}.json")
    path.parent.mkdir(exist_ok=True)
    path.write_text(json.dumps({
        "messages": messages,
        "completed_steps": completed
    }, ensure_ascii=False, indent=2))

def load_checkpoint(session_id: str) -> dict | None:
    path = Path(f".agent_sessions/{session_id}.json")
    if not path.exists():
        return None
    return json.loads(path.read_text())
```

セッションが中断されても、`messages` の履歴を復元してエージェントループを再開できます。長時間タスクや、外部処理待ちが発生するワークフローで特に重要です。

## Claude Code と Cursor/Windsurf における位置づけ

Claude は現在、Cursor、Windsurf、Claude Code の三つの主要な AI コーディングツールのバックエンドとして稼働しています。Opus 4.7 の改善はこれらのツールを通じた体験として直接現れます：

- **Claude Code**: CLI 完全統合型。Git 操作、ファイル変更、テスト実行まで一元的にこなす
- **Cursor**: エディタ統合型。コードベース全体をコンテキストに含めた補完
- **Windsurf**: 連続実装タスクをフローで処理する設計

大規模リファクタリングや複数ファイルにまたがるバグ修正が、以前より高い精度で完了するようになっています。以前は「どのファイルを編集すべきか？」「どのテストを実行すべきか？」といった確認プロンプトが複数回必要だったタスクが、中断なしに完了するケースが増えています。

## 本番投入時の三原則

**1. イテレーション上限の設定**: 制御なしのエージェントループはコストと実行時間の上振れを招きます。`max_steps` は必ず設定してください。上限に達した際にアラートを出す仕組みも組み込んでおくと、問題の早期発見につながります。

**2. コード実行のサンドボックス化**: `execute_bash` 等のツールを与える場合、Docker コンテナや gVisor などのサンドボックス内で実行することが必須です。ネットワークアクセスを制限し、本番データへの読み取り専用マウントを基本設定にしてください。

**3. Human-in-the-loop の設計**: 本番データへの書き込み、外部 API への送信など、不可逆な操作には確認ステップを設けてください。エージェントが「間違った行動」を取るリスクはゼロではなく、自律実行時にその誤りが連鎖する可能性があります。

## まとめ

Claude Opus 4.7 の SWE-bench 改善は、エンジニアリングタスクの自律実行が実用水準に近づいていることを示しています。

しかし能力の向上は「何でも任せられる」を意味しません。サンドボックス、コスト管理、Human-in-the-loop の三点が本番運用の基盤です。エージェントシステムを自社サービスに組み込む際には、この三点を設計の起点に据えることを推奨します。
