## Sonnet 5が変えたモデル選定の前提

2026年6月30日、AnthropicはClaude Sonnet 5をリリースしました。特徴は「Opus 4.8に近いエージェント性能を、大幅に低いコストで実現する」という位置づけです。計画立案、ツール利用、ターミナル操作、長時間の多段階コーディングといった、これまで上位モデルでなければ安定しなかった領域を、Sonnetクラスの価格帯でこなせるようになりました。

導入価格は2026年8月31日まで入力$2/M・出力$10/Mトークン、以降は$3/$15に移行します。すでにClaude Free・Proプラン、Claude Code、APIのデフォルトモデルとして採用されており、多くの開発者は意識せずにこのモデルを使い始めることになります。

Webhaniが顧客のAI導入を支援する際、常に議論になるのが「どのタスクにどのモデルを割り当てるか」というコスト設計です。Sonnet 5の登場は、この設計をやり直す良いタイミングです。

## なぜモデル選定が経営課題になるのか

AIコーディングを個人の生産性ツールとして使う分には、モデルの違いは体感差でしかありません。しかし、チームやプロダクトにAIエージェントを組み込む段階になると、話は変わります。

- 月間のトークン消費量がプロダクトコストの一部になる
- タスクの難易度と投入モデルのミスマッチが、コストと品質の両方に響く
- 長時間の自律実行（エージェント型ワークフロー）は、失敗時のやり直しコストも含めて設計する必要がある

Sonnet 5がOpus級の性能を持つということは、「これまでOpusでしか任せられなかったタスク」の一部をSonnetに移せる可能性を意味します。これはコスト構造そのものを見直す機会です。

## タスクベースのモデルルーティング

Webhaniでは、AIエージェントを本番導入する際に、タスクの性質でモデルを振り分ける設計を推奨しています。考え方はシンプルです。

```text
タスク分類          推奨モデル層          理由
------------------  -------------------  --------------------------------
単純な変換・整形     軽量モデル           速度優先、品質差が出にくい
通常の実装・レビュー Sonnet 5             コストと性能のバランスが良い
高難度の設計判断     Opus 4.8             アーキテクチャ判断の精度を優先
```

擬似コードで表すと、ルーティングロジックは次のようなイメージになります。

```typescript
type TaskComplexity = "trivial" | "standard" | "critical";

function selectModel(complexity: TaskComplexity): string {
  switch (complexity) {
    case "trivial":
      return "claude-haiku-4-5";
    case "standard":
      return "claude-sonnet-5";
    case "critical":
      return "claude-opus-4-8";
  }
}
```

重要なのは、この判定を人間の勘だけに頼らないことです。タスクの見積もり難易度、影響範囲（本番コードか実験コードか）、失敗時のやり直しコストを基準に、機械的に振り分けるルールをチームで合意しておくと、コストの予測可能性が上がります。

## エージェント型ワークフローでの注意点

Sonnet 5は長時間の多段階コーディングに強くなったとされていますが、これは「任せっぱなしにしてよい」という意味ではありません。エージェントが自律的に複数ファイルを変更するワークフローでは、次の設計が引き続き必須です。

1. **チェックポイントの設置**：長時間タスクを一括で任せず、区切りごとに人間のレビューを挟む
2. **差分の可視化**：エージェントが何を変更したかを、コミット単位で追いやすくする
3. **ロールバック手段の確保**：想定外の変更が入った場合に、素早く元に戻せる状態を保つ

モデルの性能が上がるほど、こうしたガードレールの重要性は下がるのではなく、むしろ「任せる範囲が広がる分だけ」重要性が増します。

## Webhaniの推奨アプローチ

顧客企業のAI導入支援では、まず既存のAIコーディングタスクを棚卸しし、上記のようなタスク分類を作るところから始めます。その上で、Sonnet 5のような「性能とコストのバランスが良いモデル」をデフォルトに据え、本当に高難度な設計判断だけをOpus級に回す、という二段構えの運用を提案しています。

導入価格が2026年8月末で変わる点も踏まえ、コスト試算は現時点の価格だけでなく、9月以降の価格を前提にシミュレーションしておくことをお勧めします。

## まとめ

Claude Sonnet 5は、単なる「新しいモデル」ではなく、AIコーディングのコスト構造を見直す契機です。Opus級の性能をSonnet価格で使えるということは、これまでコストを理由に見送っていたタスクの自動化が現実的になったことを意味します。一方で、性能が上がるほど検証設計の重要性は増します。Webhaniでは、モデル選定とガードレール設計をセットで提案し、顧客のAI導入を支援していきます。
