## 「安いフロンティア」という位置づけ

Anthropic は 6 月 30 日、Claude Sonnet 5 をリリースしました。話題の中心は性能そのものよりも、価格とエージェント運用のバランスにあります。導入価格は入力 $2/百万トークン、出力 $10/百万トークン（2026 年 8 月末まで、その後は $3 / $15）。数ヶ月前なら Opus クラスでしか実現できなかった自律的なタスク遂行を、この価格帯で回せるようになった、というのが最大の変化です。

webhani ではクライアントのエージェント基盤を複数運用していますが、そこで一貫して問題になるのは「1 回の推論の賢さ」ではなく「ループを何十回も回したときの総コスト」です。エージェントはツールを呼び、結果を読み、次の行動を決める、という反復を繰り返します。この反復回数が多いワークロードほど、モデル単価がそのまま利益率に効いてきます。Sonnet 5 が注目されるのは、この「反復コスト」を大きく下げる位置に着地したからです。

## 何が変わったのか

Anthropic の説明によれば、Sonnet 5 は「これまでで最もエージェント的な Sonnet」を目指して設計されています。ブラウザやターミナルといったツールを使いながら、複数ステップのタスクを人間の介入なしに走らせる能力が向上しました。実務でよく聞く評価は「タスクを最後まで終わらせる」という点です。従来モデルが途中で手を止めていた場面でも、計画を立て直して完遂まで持っていく傾向が強い、という報告があります。

技術面では以下が押さえどころです。

- **1M トークンのコンテキストがデフォルト** — 大規模コードベースの読み込みやログ分析に有利
- **最大 128k トークンの出力** — 長い生成物を一度に返せる
- **アダプティブ思考（adaptive thinking）** — 問題の難易度に応じて内部推論の深さを調整
- **新しいトークナイザー** — ここに落とし穴があります

## 新トークナイザーという「隠れコスト」

見落とされがちなのが、Sonnet 5 が新しいトークナイザーを採用した点です。同じ入力テキストでも、Sonnet 4.6 と比べておよそ 30% 多くトークンを消費します。つまり「単価は下がったが、トークン数は増える」という関係です。

これは移行時のコスト試算をそのまま持ち越せないことを意味します。単価だけを見て「Opus から乗り換えれば安くなる」と結論づけると、想定より請求額が膨らむケースがあります。実際に見積もる際は、単価の差分だけでなくトークン増加分を掛け合わせる必要があります。

```typescript
// 移行コストの概算：単価だけでなくトークン増加率を織り込む
interface ModelCost {
  name: string;
  inputPerM: number; // $/百万トークン
  outputPerM: number;
  tokenMultiplier: number; // 旧トークナイザー比の消費倍率
}

const sonnet46: ModelCost = {
  name: "Sonnet 4.6",
  inputPerM: 3,
  outputPerM: 15,
  tokenMultiplier: 1.0,
};

const sonnet5: ModelCost = {
  name: "Sonnet 5 (標準価格)",
  inputPerM: 3,
  outputPerM: 15,
  tokenMultiplier: 1.3, // 約 30% 増
};

function estimateCost(
  model: ModelCost,
  inputTokens: number,
  outputTokens: number
): number {
  const adjustedInput = inputTokens * model.tokenMultiplier;
  const adjustedOutput = outputTokens * model.tokenMultiplier;
  return (
    (adjustedInput / 1_000_000) * model.inputPerM +
    (adjustedOutput / 1_000_000) * model.outputPerM
  );
}

// 1 回のエージェントループ想定
const inputTokens = 40_000;
const outputTokens = 2_000;

console.log(estimateCost(sonnet46, inputTokens, outputTokens));
console.log(estimateCost(sonnet5, inputTokens, outputTokens));
// 単価が同じでもトークン増加分だけ Sonnet 5 の方が高くなる
```

ポイントは、トークナイザーが変わるとコストだけでなくコンテキストウィンドウの「実効容量」も変わることです。1M トークンの枠があっても、同じ文書がより多くのトークンを占めるため、体感的に入る情報量は目減りします。長文を大量に扱う設計では、この点を前提にチャンク分割の閾値を見直す価値があります。

## エージェント運用で効く実践パターン

**アダプティブ思考の effort 制御** — 簡単なタスクにまで深い推論を回すと、出力トークンが増えてコストに跳ね返ります。タスクの難易度に応じて推論の深さを指定し、単純な分類やフォーマット変換では浅く抑えるのが定石です。

**プロンプトキャッシング** — エージェントはシステムプロンプトやツール定義を毎回同じ内容で送ります。この固定部分をキャッシュに載せると、2 回目以降の入力コストを大幅に削減できます。反復回数の多いエージェントほど効果が大きい。

**モデルの階層化は依然有効** — Sonnet 5 が安くなったとはいえ、すべてを Sonnet 5 で回す必要はありません。定型的なサブタスクは Haiku、難所は Sonnet 5、それでも解けない最難関だけ Fable 5、という三段構えは引き続き機能します。Sonnet 5 の登場でこの「真ん中」が厚くなった、という理解が実態に近い。

**トークン予算の可視化** — エージェントを本番投入する前に、1 タスクあたりの平均トークン消費を計測しておくことを勧めます。新トークナイザーにより旧来の実測値が使えないため、移行後に必ず取り直してください。

## 移行判断のチェックリスト

webhani がクライアントに Sonnet 5 への移行を勧めるかどうかは、以下で判断しています。

- エージェントのループ回数が多く、モデル単価が利益率に直結している
- タスクが「途中で止まる」問題を抱えており、完遂率の改善が価値になる
- 移行後にトークン消費を再計測し、コスト増を織り込める体制がある

逆に、単発の短い生成が中心で反復が少ないワークロードでは、移行の旨みは限定的です。単価の安さは反復回数が多いほど効いてくるためです。

## まとめ

Claude Sonnet 5 の本質は「賢くなった」ことよりも「エージェントを安く回せる位置に来た」ことにあります。ただし新トークナイザーによるトークン増加という副作用があり、単価だけを見た移行判断は危険です。単価差とトークン増加率の両方を掛け合わせ、実測でコストを取り直す。この手順を踏めば、Sonnet 5 はエージェント基盤の運用コストを現実的に下げる選択肢になります。webhani ではこの検証プロセスをクライアント案件の標準手順として組み込んでいます。
