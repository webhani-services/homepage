## チャットから「実行環境」へ

AIアシスタントの使い方が、この1年で大きく変わりました。以前は「良いプロンプトを書けるかどうか」が成果を分けていましたが、2026年時点では「同じ品質の作業を、何度でも再現できる仕組みを組めるか」が問われるようになっています。

Anthropicのユーザー利用動向を紹介する複数の記事でも、Claudeを一問一答のチャットボットではなく「Workflowを回すための環境」として扱う人ほど得られる価値が大きい、という指摘が出ています（[Let's Data Science](https://letsdatascience.com/news/users-treat-claude-as-workflow-operating-system-1c5926ac)）。私たちが顧客企業の開発チームを支援する際にも、この視点の有無で導入効果が大きく変わると感じています。

この記事では、単発のプロンプトから「再現可能なWorkflow」へ移行するための構成要素と、実務での組み立て方を整理します。

## 役割分担で考える5つの構成要素

Claude Codeのようなagenticなツールでは、機能を5つの層で捉えると設計がしやすくなります。それぞれ「いつ効くか」が違うため、混同すると期待した挙動になりません。

- **CLAUDE.md（常時Context）**: プロジェクトの前提・規約・禁止事項を常に読ませる
- **Skills（オンデマンドの知識・手順）**: 特定のTaskのときだけ呼び出す手順書
- **MCP（外部システム接続）**: Issue Tracker、Database、監視Toolなどファイルの外へ接続する
- **Subagent（Contextの分離）**: レビューやテストなど、独立したContextで実行させる
- **Hooks（自動化・ガードレール）**: Tool呼び出しやSession終了時にルールを強制する

覚え方はシンプルです。CLAUDE.mdは「常に効く前提」、Skillsは「必要なときの手順」、MCPは「外への接続」、Subagentは「文脈の隔離」、Hooksは「自動の強制」。この5つを役割で分けておくと、あとから機能を足すときに迷いません。

## CLAUDE.md: 前提を毎回書かない

もっとも費用対効果が高いのがCLAUDE.mdです。使用するFrameworkのバージョン、命名規則、テストの実行方法、触ってはいけないファイルなどを一度書いておけば、毎回のプロンプトで説明する必要がなくなります。

```markdown
# Project規約（抜粋）

## Tech Stack
- Next.js 16 App Router / TypeScript strict
- Styleは Tailwind のみ。CSS Modules は使わない

## 禁止事項
- `src/generated/` 配下は手で編集しない（自動生成）
- 環境変数は `.env.local` に直書きしない

## テスト
- 変更後は必ず `npm test` を実行してから完了とする
```

ポイントは「守ってほしいこと」だけでなく「やってはいけないこと」を明記することです。禁止事項を書いておくと、Modelが良かれと思って壊す事故を減らせます。

## Skills: 手順を再利用可能にする

同じ作業を何度も頼むなら、その手順をSkillとして切り出します。「Blog記事を生成する」「Release Noteを作る」といった定型Taskを、実行条件と手順の形で保存しておくと、毎回同じ品質で再現できます。

CLAUDE.mdが「常に読まれる前提」であるのに対し、Skillsは「条件に合ったときだけ読み込まれる手順」です。常時Contextを膨らませずに専門知識を持たせられるため、Context圧迫を避けつつ振る舞いを安定させられます。

## MCP: ファイルの外へつなぐ

MCP（Model Context Protocol）は、AIをファイルシステムの外の世界へ接続する仕組みです。Issue TrackerからTicketを読む、PostgreSQLにクエリを投げる、監視Toolでエラーを確認する、といった「実際の業務Workflow」を回せるようになります。

ここで重要なのは権限設計です。書き込み可能なMCP接続を安易に増やすと、意図しない変更のリスクが上がります。私たちが顧客に導入する際は、まず読み取り専用の接続から始め、効果を確認しながら書き込み権限を段階的に開放することを勧めています。

## Subagentとhooks: 隔離と強制

**Subagent**は、独立したContext・プロンプト・Tool権限を持つ補助エージェントです。メインのエージェントが計画と統合を担い、コードレビューやテスト実行、Security Checkといった限定的なTaskをSubagentに任せると、メインの文脈を汚さずに専門作業を並行できます。

**Hooks**は、Tool呼び出しやSession終了といったイベントにフックして、ルールを機械的に強制する仕組みです。よくあるパターンは次のようなものです。

- 終了前に必ずテストを走らせる
- 自動生成ファイルへの編集をブロックする
- Commit前にLintを実行する
- 依存関係の変更後にSecurity Scanを走らせる

Hooksの価値は「Modelの善意に頼らない」点にあります。人間のレビューと同じで、忘れる前提でルールを仕組み化しておくほうが安定します。

## webhaniの視点: 小さく始めて計測する

これらを一気に全部導入する必要はありません。私たちが顧客のチームに提案している順序は次のとおりです。

1. **CLAUDE.mdだけ整える**: これだけで体感の安定性が大きく変わる
2. **頻出Taskを1つSkill化する**: 効果が見えやすいものから
3. **読み取り専用のMCPを1本つなぐ**: まずは安全な接続で価値を確認
4. **Hooksでテスト実行を強制する**: 品質の下限を機械的に担保
5. **必要になったらSubagentで隔離する**: レビューやテストから

大切なのは、導入前後で「手戻りの回数」や「レビュー指摘の数」を計測しておくことです。効果を数字で確認できないと、チームへの定着は進みません。

## まとめ

AIアシスタントの成果は、もはや単発のプロンプトの巧拙では決まりません。CLAUDE.mdで前提を固定し、Skillsで手順を再利用し、MCPで外部へつなぎ、Subagentで文脈を隔離し、Hooksでルールを強制する——この5つを役割で分けて組み立てることで、作業は「再現可能なWorkflow」になります。

まずはCLAUDE.mdから始めてみてください。もっとも手軽で、もっとも効果を実感しやすい一歩です。
