Cloudflare が自社製 AI 推論エンジン「Infire」の詳細を公開しました。Cloudflare Workers AI のバックエンドとして機能するこのエンジンは、グローバルに分散した Cloudflare のネットワーク上で LLM を効率的に動かすために開発されました。

## Infire が解決しようとした課題

LLM の推論をグローバル Edge で提供することには、データセンター集中型と異なる制約があります。

**GPU の断片化**: 地理的に分散したポップ（Point of Presence）では、各拠点が保有する GPU リソースが限られます。モデルが複数の GPU にまたがる場合、従来の推論エンジンではオーバーヘッドが大きくなります。

**コールドスタートの問題**: LLM は起動に時間がかかります。ユーザーリクエストに対して数十秒のウォームアップ時間が発生すると、Web サービスとしての実用性が損なわれます。

**メモリ効率**: GPU メモリは高価なリソースです。同じメモリ上で複数のモデルやリクエストを効率的に扱えるかどうかが、コストと応答速度の両面に直結します。

## Infire の主要な技術アプローチ

### マルチ GPU 推論の効率化

Infire は、モデルの重みを複数の GPU に分割して並列推論するテンソル並列処理（Tensor Parallelism）を、Edge 環境向けに最適化しています。GPU 間の通信オーバーヘッドを最小化するために、独自の通信プロトコルと同期ロジックを実装しています。

標準的なテンソル並列実装は高帯域幅の GPU 間接続を前提としていますが、Infire は Edge ハードウェアプロファイルに合わせて同期オーバーヘッドを削減します。

### KV Cache の共有

LLM の推論では、過去のトークンの計算結果（KV Cache）を再利用することで処理速度が向上します。Infire は複数のリクエスト間でこのキャッシュを共有するアーキテクチャを採用しており、同一のシステムプロンプトを使う API コール（例：同じ AI アシスタントへの複数ユーザーからのリクエスト）でメモリを効率的に利用できます。

```python
# Cloudflare Workers AI API 経由の利用例
import httpx

async def call_assistant(user_message: str, account_id: str, token: str):
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"https://api.cloudflare.com/client/v4/accounts/{account_id}/ai/run/"
            "@cf/meta/llama-4-scout-17b-16e-instruct",
            headers={"Authorization": f"Bearer {token}"},
            json={
                "messages": [
                    # 共有システムプロンプトは KV Cache 再利用の恩恵を受ける
                    {"role": "system", "content": "You are a helpful support agent."},
                    {"role": "user", "content": user_message}
                ]
            }
        )
        return response.json()["result"]["response"]
```

### モデルの高速起動

Infire は、よく使われるモデルの重みをメモリに常駐させる「Hot Standby」方式を採用しています。リクエストが来た時点で起動処理が始まる従来の方式と異なり、コールドスタート時間を数十秒から数百ミリ秒単位に短縮しています。

## Web 開発者への実践的な影響

Infire の改善は、Cloudflare Workers AI を使うアプリケーションに直接反映されます。

```javascript
// Cloudflare Worker でのストリーミング LLM 応答
export default {
  async fetch(request, env) {
    const { question } = await request.json();

    const stream = await env.AI.run(
      "@cf/meta/llama-4-scout-17b-16e-instruct",
      {
        messages: [
          { role: "system", content: "You are a helpful assistant." },
          { role: "user", content: question }
        ],
        stream: true,
      }
    );

    return new Response(stream, {
      headers: {
        "Content-Type": "text/event-stream",
        "Cache-Control": "no-cache",
      }
    });
  }
};
```

**レイテンシの改善**: コールドスタートの短縮により、初回リクエストでも応答が早くなります。特にトラフィックが散発的なアプリケーションでの改善が顕著です。

**コスト効率**: KV Cache 共有による GPU メモリの効率利用は、同じ推論能力をより低コストで提供できることを意味します。

## Edge AI アーキテクチャの設計指針

Infire の設計思想から、Edge で AI 推論を扱うアーキテクチャの実践的な指針を導けます。

**短いシステムプロンプトを共有する**: KV Cache 共有の恩恵を最大化するには、システムプロンプトを短く、かつ複数のリクエスト間で共通にすることが効果的です。

**ストリーミングを基本とする**: Edge での LLM 推論は、生成速度とネットワーク転送の関係から、ストリーミングレスポンスが TTFB（Time To First Byte）の改善に有効です。

**地理的な近さを活用する**: Cloudflare の場合、ユーザーに最も近い PoP での処理が自動的に行われますが、機密データの処理地域に要件がある場合は Workers の地域設定を明示的に行う必要があります。

## 他の推論エンジンとの比較

| エンジン | 用途 | 主な特徴 |
|---------|------|---------|
| vLLM | セルフホスト・クラウド | PagedAttention, 高スループット |
| TensorRT-LLM | NVIDIA GPU 環境 | CUDA 最適化, 量子化対応 |
| llama.cpp | エッジデバイス・CPU | 軽量, ポータビリティ |
| Infire | Cloudflare Edge | マルチテナント, 低コールドスタート |

Infire の差別化ポイントは、マルチテナント環境でのリソース共有効率です。専用 GPU クラスターが持てないサービスや、スパイクしやすいトラフィックパターンを持つアプリケーションに適しています。

## セキュリティとデータプライバシー

Edge で LLM を動かす際に気になるのは、テナント間のデータ分離です。Cloudflare は KV Cache の共有がリクエストのコンテンツを他テナントに露出させないよう、テナント境界を KV Cache のキーに組み込む方式を採用しています。

機密性の高い業務データを扱う場合は、Cloudflare AI Gateway のゼロデータ保持オプションや、プライベートクラウドでの自己ホスト推論エンジンとの組み合わせも検討する価値があります。

## まとめ

Infire の公開は、LLM 推論の「インフラ層」が成熟してきたことを示しています。1 年前は「Edge で LLM を動かす」こと自体が実験的でしたが、今や実用的なレイテンシとコストで提供できる段階に達しました。

webhani が扱うプロジェクトで Cloudflare Workers AI を評価する際の判断基準として、以下を挙げます。

- **適している**: チャットボット、軽量な文書分類、コンテキスト依存の入力補完
- **他の選択肢を検討**: 長文生成・複雑な推論（GPT-4o / Claude Sonnet クラスの能力が必要なケース）、データ所在地の厳格な要件がある場合

Edge AI は「小さく始めて必要に応じてスケール」する設計に最も向いています。最初からクラウド規模の推論インフラを構築するより、Cloudflare Workers AI で機能を実装して負荷を計測してからアーキテクチャを判断するアプローチが、コストと開発速度の両面で合理的です。
