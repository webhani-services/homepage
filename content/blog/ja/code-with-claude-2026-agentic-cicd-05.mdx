2026年5月6日、AnthropicはサンフランシスコでCode with Claudeを開催しました。このイベントは単なる製品発表の場ではなく、Claude Codeをプロダクション環境でどう運用するかに焦点を当てた実践的な開発者向けカンファレンスでした。

3つのトラック（Claude Code、Claude Platform、Managed Agents）を通じて提示されたのは、AIがオートコンプリートから本格的なCI/CDコンポーネントへと移行するための具体的な知見です。

## 「長期タスク」という概念の変化

これまでのAIコーディングツールの利用は、主にインラインの補完やチャットベースの質問応答に限られていました。Code with Claude 2026では、**ロングホライズンタスク**と呼ばれる概念が中心テーマの一つになっていました。

ロングホライズンタスクとは、単一のプロンプトに収まらない、複数ステップにわたる作業です。たとえば：

- 新機能の仕様書を読み込み、複数のファイルにわたる実装を行い、テストを追加してPRを作成する
- セキュリティ脆弱性のレポートを受け取り、影響範囲を特定し、修正を実装して動作確認まで行う
- 複数のリポジトリにまたがるAPIの変更を一貫した形で適用する

これらのタスクをClaude Codeが自律的に処理できるようになった背景には、エージェントの制御フローと状態管理の改善があります。

## Parallel Agentの実践的なアーキテクチャ

イベントで特に注目を集めたのが、複数のClaude Codeエージェントを並列で動かすパターンです。

単一のエージェントがすべてを担当するモデルでは、タスクが複雑になるにつれてコンテキストウィンドウが圧迫され、エラーが増加します。Parallel Agentアーキテクチャでは、タスクを並列処理可能な単位に分割し、それぞれを独立したエージェントが担当します。

```bash
# Claude Code CLIでのParallel Agent起動例
# 複数のマイクロサービスに対して同時にレビューを実施

claude --parallel \
  --task "review src/auth-service for security issues" \
  --task "review src/payment-service for security issues" \
  --task "review src/notification-service for security issues" \
  --output-format json \
  --max-turns 20
```

CI/CDパイプラインでは、この並列化が特に効果的です。たとえばモノレポ環境で、変更の影響を受けるサービスのみを並列でテスト・レビューするパターンが紹介されました。

```yaml
# .github/workflows/ai-review.yml
name: AI Code Review

on:
  pull_request:
    branches: [main]

jobs:
  detect-changed-services:
    runs-on: ubuntu-latest
    outputs:
      services: ${{ steps.detect.outputs.services }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - id: detect
        run: |
          CHANGED=$(git diff --name-only origin/main...HEAD | \
            grep '^src/' | cut -d'/' -f2 | sort -u | jq -R -s -c 'split("\n")[:-1]')
          echo "services=$CHANGED" >> $GITHUB_OUTPUT

  ai-review:
    needs: detect-changed-services
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: ${{ fromJson(needs.detect-changed-services.outputs.services) }}
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Code review
        run: |
          claude review src/${{ matrix.service }} \
            --focus "bugs,security,performance" \
            --output pr-comment
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

## 運用制御：本番環境で必要なもの

Code with Claude 2026が強調したのは、エージェントの能力だけではありませんでした。**本番環境でエージェントを運用するための制御機構**が同様に重要なテーマでした。

### パーミッション管理

Claude Codeにはファイルシステムへの書き込み、コマンドの実行、外部APIの呼び出しなどの能力があります。プロダクション環境では、これらをきめ細かく制御する必要があります。

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(git:*)",
      "Bash(npm run test:*)",
      "Bash(npm run lint:*)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(curl:*)",
      "WebFetch"
    ]
  }
}
```

### エビデンストレイル

エージェントが何を変更したか、なぜその判断をしたかのトレーサビリティは、特に規制業種では必須です。Claude Codeの実行ログを構造化された形で保存し、後から監査できる形にしておく必要があります。

### コスト管理

Parallel Agentを大規模に動かすとAPI利用コストが急増します。イベントではPricing SignalというコンセプトのもとPR単位、サービス単位でのコスト上限を設定するパターンが紹介されました。

## マルチリポジトリ対応

GitHub-Scale EngineeringセッションではGitHub規模のコードベース（複数のリポジトリにまたがる何千ものサービス）に対してClaude Codeをどう適用するかが議論されました。

キーとなるパターンは**コンテキストの事前ロード**です。エージェントが作業を始める前に、関連するAPIドキュメント、テスト慣習、コーディングスタイルをコンテキストとして明示的に渡します。

```bash
# リポジトリのコンテキストをClaude Codeに渡す
claude --context-file .claude/project-context.md \
       --context-file docs/api-contracts.md \
       --context-file docs/coding-standards.md \
       "Refactor the auth module to support OAuth 2.1"
```

## webhaniの視点

Code with Claude 2026が示した方向性は明確です。Claude Codeは「開発者の横に座るペアプログラマー」から「CI/CDパイプラインの中で自律的に動くエージェント」へと進化しています。

この変化に対応するために、チームがまず着手すべきことは：

1. **タスク境界の定義**：エージェントに委譲するタスクの範囲を明確にする。境界が曖昧なタスクは失敗しやすい
2. **パーミッションの最小化**：エージェントに必要最小限の権限のみを付与し、`.claude/settings.json`で明示的に管理する
3. **オブザーバビリティの確立**：エージェントの実行ログを構造化して保存し、コスト・実行時間・成功率を測定する
4. **小さく始める**：いきなりロングホライズンタスクから始めるのではなく、PRのコードレビューやテスト追加など限定的なタスクから始める

AIエージェントをCI/CDに組み込む際の最大のリスクは技術的な失敗ではなく、制御の欠如です。まず制御の仕組みを構築し、その上でエージェントの自律性を段階的に拡張していくアプローチが、本番環境では最も安全です。
