## 独自路線から「歩み寄り」へ

Deno はもともと、Node.js の反省から生まれたランタイムだった。標準で TypeScript を実行でき、パーミッションモデルを持ち、Web 標準 API を第一に据える——設計思想は明快で魅力的だった。ただ現場での採用は、既存の npm エコシステムとの距離が壁になっていた。

2026 年のアップデートは、その距離を積極的に詰めにきている。Deno 2.4 では `deno bundle` が復活し、Node.js の API やグローバルへの互換性がさらに改善された。フレームワーク側でも、Deno 純正の Fresh が 2.0 beta で Vite 統合へ舵を切った。独自路線を貫くより、既存ツールに歩み寄る——この現実路線を webhani の採用視点で読み解く。

## Deno 2.4：`deno bundle` の復活が意味すること

かつて Deno には `deno bundle` があったが、一度非推奨・削除されていた。2.4 で改めて公式バンドラーとして戻ってきたのは、実務的な要請の反映だ。単一ファイルへのバンドルは、CLI ツールの配布、エッジ環境へのデプロイ、依存を固めた成果物の生成といった場面で今も需要が高い。

```bash
# エントリポイントから単一の JS にバンドルする
deno bundle main.ts --output dist/app.js

# 実行時の権限は明示的に付与する（Deno のセキュリティモデル）
deno run --allow-net --allow-read dist/app.js
```

注目すべきは、バンドラーが戻ってきても Deno のパーミッションモデルは変わっていない点だ。`--allow-net` や `--allow-read` を明示しない限り、ネットワークやファイルへのアクセスは拒否される。サプライチェーン攻撃が現実的な脅威になっている今、「デフォルトで何もできない」という前提は、Node.js に対する明確な差別化要素として意味を増している。

Node.js 互換の改善も実務では大きい。既存の npm パッケージや Node のグローバル（`process` など）が動く範囲が広がれば、「Deno は良さそうだが手持ちの資産が動かない」という導入時の摩擦が減る。

## Fresh 2.0：独自バンドラーを捨て、Vite に乗る

より象徴的なのは Fresh 2.0 beta だ。Fresh は Deno 純正のフルスタックフレームワークで、これまで独自のビルドパイプラインを持っていた。2.0 beta では、オプションで Vite 統合を導入し、HMR（ホットリロード）、起動時間の短縮、React のエイリアス解決、そして Vite プラグインエコシステム全体を取り込めるようにした。

これは技術的な妥協ではなく、賢い判断だと考える。Vite は既に事実上の標準ビルドツールであり、Vite 8 では Rolldown（Rust 製バンドラー）によってビルド性能も底上げされている。ここに独自実装で対抗し続けるより、成熟したエコシステムに乗るほうが、開発者にとっての体験も、保守の持続性も高い。

```tsx
// Fresh のルート例：ファイルベースルーティングは維持しつつ、
// ビルド基盤に Vite を採用できるようになった
export default function Page(props: { data: { title: string } }) {
  return (
    <main class="p-8">
      <h1 class="text-2xl font-bold">{props.data.title}</h1>
      <p>Fresh は島（islands）単位でのみクライアント JS を送る。</p>
    </main>
  );
}
```

Fresh の「アイランドアーキテクチャ」——インタラクティブな部分だけにクライアント JS を送り、それ以外はサーバーレンダリングのみで済ませる考え方——は据え置かれている。設計思想は保ちつつ、ビルド基盤だけを実績あるものへ差し替えた形だ。

## webhani の採用視点：どこで Deno を選ぶか

現実路線への転換は歓迎すべきだが、だからといって Deno をあらゆる場面で選ぶべきという話にはならない。私たちがクライアントに助言する際の判断軸は次のとおりだ。

- **新規の CLI ツール／小さなエッジ関数**：TypeScript を追加設定なしで走らせ、`deno bundle` で単一成果物にできる。Deno の強みが素直に効く領域。
- **セキュリティ要件が厳しいスクリプト実行**：明示的パーミッションモデルにより、外部依存を走らせる際の攻撃面を制御しやすい。
- **既存の大規模 Node.js 資産の移行**：互換性は改善しているが、依存ツリー全体の検証が必要。段階的に、周辺のツール群から試すのが現実的。
- **フロントエンドの新規プロジェクト**：Fresh 2.0 の Vite 統合により選択肢に入るが、beta である点は本番投入前に必ず確認する。

一言でいえば、Deno は「Node.js の完全な置き換え」ではなく「特定の要件に強い選択肢」として捉えるのが適切だ。TypeScript ネイティブ実行、明示的パーミッション、Web 標準準拠が効く場面を見極めて投入する。

## まとめ

Deno 2.4 の `deno bundle` 復活と Fresh 2.0 の Vite 統合は、いずれも「独自性を薄める」方向の変化だ。だがこれは後退ではなく、エコシステムの現実に歩み寄る成熟だと見るべきだろう。独自の設計思想（TypeScript ネイティブ、パーミッションモデル、Web 標準）という核は保ちながら、周辺の摩擦を減らしている。

webhani としては、Deno を導入候補から外していた組織こそ、この現実路線を機に再評価する価値があると考える。まずは小さな CLI ツールやエッジ関数で `deno bundle` を試し、フロントエンドは Fresh 2.0 の安定版化を見極めてから——という段階的なアプローチを勧める。
