Docker が新しいプラットフォーム **Docker Kanvas** を発表しました。このツールの核心は、`docker-compose.yml` を Kubernetes のマニフェストへ自動変換する機能です。Helm や Kustomize といった既存ソリューションへの代替を目指すポジショニングも注目されています。

## Docker Kanvas とは

Kanvas は、ローカルの Docker Compose 環境から本番スケールの Kubernetes 環境への移行を自動化するプラットフォームです。開発者がすでに慣れ親しんでいる Compose の定義をそのまま活かしながら、Kubernetes に必要なリソース（Deployment、Service、ConfigMap、Secret など）を生成します。

典型的なワークフロー：

```bash
# ローカル開発はこれまで通り
docker compose up

# Kanvas で Kubernetes リソースを生成
kanvas convert -f docker-compose.yml -o k8s/

# 生成されたマニフェストを確認・調整
ls k8s/
# deployment.yaml  service.yaml  configmap.yaml  secrets.yaml  ...

# Kubernetes へデプロイ
kubectl apply -f k8s/
```

## Helm・Kustomize との比較

既存のエコシステムとの比較が気になるところです。

**Helm との比較:**
Helm はテンプレートエンジンとしての成熟度が高く、チャートのエコシステムも豊富です。Kanvas はその複雑さを避けるアプローチで、`docker-compose.yml` を起点に置くことで学習コストを大幅に下げることを目指しています。Helm が「すべてを設定可能にする」思想なら、Kanvas は「Compose から自動で始める」思想です。

**Kustomize との比較:**
Kustomize は既存のマニフェストへのパッチ適用を得意とします。環境ごとの差分管理には強力ですが、最初のマニフェストを作成する作業はゼロではありません。Kanvas は「最初の生成」フェーズを担うツールとして、Kustomize とは異なる役割を持ちます。

| ツール | 強み | 弱み |
|--------|------|------|
| Helm | 豊富なエコシステム、再利用性 | 学習コスト高、テンプレート複雑 |
| Kustomize | シンプルなパッチ戦略 | 初期マニフェスト生成は手動 |
| Kanvas | Compose から自動生成 | 成熟度・エコシステムはまだ発展途上 |

## 実際の変換例

簡単な Node.js + PostgreSQL の Compose 定義を例に考えてみます：

```yaml
# docker-compose.yml
services:
  app:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data

volumes:
  postgres_data:
```

Kanvas はこれを以下のような Kubernetes リソースへ変換します：

- App 用 `Deployment`
- Database 用 `StatefulSet`（永続ストレージ）
- 内部通信用 `Service`
- データベース認証情報を格納した `Secret`（環境変数から自動抽出）

機密情報が自動的に `Secret` リソースへの参照に変換される点は、手動変換と比較した際の実用的な改善です。

## 生成後に必要な作業

生成されたマニフェストはベースラインであり、本番利用には追加の設定が必要です。

```yaml
# Kanvas が生成する（正しいが不完全な）状態
spec:
  containers:
    - name: app
      image: my-app:latest

# 本番適用前に手動追加が必要な設定
      resources:
        requests:
          cpu: 100m
          memory: 256Mi
        limits:
          cpu: 500m
          memory: 512Mi
      livenessProbe:
        httpGet:
          path: /health
          port: 3000
        initialDelaySeconds: 30
        periodSeconds: 10
      readinessProbe:
        httpGet:
          path: /ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 5
```

NetworkPolicy、RBAC、Pod セキュリティポリシーも生成結果には含まれないため、テスト環境以上を目標とするなら手動で追加が必要です。

## webhani としての視点

webhani ではクラウドインフラの構築・移行支援を提供しており、「Docker Compose で動いていたものを Kubernetes に移したい」というニーズは実際に多く見られます。現時点での見立てとしては、Kanvas は以下のシナリオで特に効果を発揮すると考えています：

1. 小〜中規模のプロジェクトで Kubernetes を初めて導入する場合
2. プロトタイプを素早く Kubernetes 環境で試したい場合
3. Helm の複雑さを避けたいチームが最初のマニフェストを生成する起点として

一方、大規模プロダクション環境や、既存の Helm Chart があるプロジェクトでは、当面 Helm/Kustomize が主力であり続けるでしょう。

## まとめ

Docker Kanvas は「ローカル開発と本番 Kubernetes の間のギャップを埋める」という明確な課題に取り組んでいます。Compose の普及率を考えると、このアプローチには実用的な価値があります。

Helm や Kustomize を今すぐ置き換えるものではなく、「Compose から Kubernetes への最初の一歩」を支援するツールとして活用するのが適切です。KubeCon 後の公式フィードバックを見ながら、プロジェクトへの適用を検討することをお勧めします。
