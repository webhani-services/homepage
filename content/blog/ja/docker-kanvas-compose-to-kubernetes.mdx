Docker が Kanvas を正式に発表しました。ローカル開発環境（Docker Compose）とクラウドの本番環境（Kubernetes）のギャップを埋めることを目的としたプラットフォームです。KubeCon + CloudNativeCon Europe 2026 でも注目を集めており、Helm や Kustomize に対する挑戦状とも受け取られています。

## Docker Kanvas が解決しようとする問題

「ローカルでは動くのに本番で動かない」問題の根本原因の一つは、開発環境と本番環境の設定形式が根本的に異なることです。

開発者は `docker-compose.yml` でサービスを定義し、ローカルで動作を確認します。一方、Kubernetes では Deployment、Service、Ingress など複数の YAML ファイルが必要で、Compose の概念と一対一で対応していません。この変換作業は手間がかかり、Kubernetes の知識がない開発者には高い障壁です。

Kanvas はこの変換を自動化します。

```yaml
# docker-compose.yml（開発環境）
services:
  web:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://db:5432/myapp
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
```

このような Compose ファイルから、Kanvas は対応する Kubernetes マニフェストを生成します。

## Helm / Kustomize との比較

Helm と Kustomize はそれぞれ成熟したエコシステムを持ちます。Kanvas の差別化ポイントは「Compose からの移行パス」です。

| 側面 | Helm | Kustomize | Kanvas |
|------|------|-----------|--------|
| 出発点 | Helm Chart（独自形式）| Base YAML | Docker Compose |
| 学習コスト | 高い（テンプレート言語）| 中程度 | 低い（Compose 知識で可）|
| 対象ユーザー | Kubernetes 経験者 | Kubernetes 知識が必要 | Compose ユーザー |
| 本番運用実績 | 豊富 | 豊富 | 新規 |

Kanvas は Kubernetes の専門知識がない開発者でも、既存の Compose 設定から Kubernetes への移行を始められる設計です。

## 実際の評価: どんなプロジェクトに向いているか

**向いているケース:**
- Compose でローカル開発、Kubernetes で本番という構成を採用している
- Kubernetes の設定を手書きできるエンジニアが少ないチーム
- マイクロサービスの数が少なく、Helm の複雑さを持ち込みたくない
- フロントエンドチームが自分たちのサービスを Kubernetes にデプロイしたい

**向いていないケース:**
- Helm Chart を既に本番運用している（移行コストが高い）
- Kubernetes 独自のリソース（HPA、PodDisruptionBudget、NetworkPolicy など）を多用している
- マルチクラスタや高度なパッケージ管理が必要

## 自動生成されるマニフェストの限界と補完

Kanvas の自動変換はあくまで出発点です。生成されたマニフェストには本番利用に向けた追加設定が必要です。

```yaml
# Kanvas が生成するベースの Deployment（例）
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 1  # 本番では適切な値に変更が必要
  selector:
    matchLabels:
      app: web
  template:
    spec:
      containers:
        - name: web
          image: my-app:latest
          # 以下は追加が必要:
          # resources:
          #   requests: { cpu: "100m", memory: "128Mi" }
          #   limits: { cpu: "500m", memory: "512Mi" }
          # livenessProbe: ...
          # readinessProbe: ...
```

本番運用に向けて追加が必要な主な設定:
- **リソース制限**: `resources.limits` と `resources.requests`
- **ヘルスチェック**: `livenessProbe` と `readinessProbe`
- **シークレット管理**: Compose の環境変数は Kubernetes Secret または外部シークレット管理への移行が必要
- **永続化ストレージ**: Compose の volumes は PersistentVolumeClaim への変換が必要

## KubeCon 2026 の文脈で見る

KubeCon + CloudNativeCon Europe 2026 で発表された調査によると、Cloud Native 開発者の人口は 1,990 万人に達しました。この成長の多くは、Kubernetes を直接触ったことのない開発者によるものです。

Docker Kanvas はこの層——Compose には慣れているが Kubernetes は敷居が高いと感じているエンジニア——を取り込む戦略的な製品です。

また、Kubernetes 1.35 では Dynamic Resource Allocation（DRA）の binding conditions が beta に昇格し、AI/ML ワークロードの GPU リソース管理が改善されています。将来的に Kanvas が AI ワークロードの Kubernetes 移行を簡素化する方向に進化する可能性もあります。

## まとめ

Docker Kanvas は「Compose から Kubernetes へ」の移行障壁を下げることに特化したツールです。Helm や Kustomize を置き換えるものではなく、それらへの移行前の踏み台として、あるいは Kubernetes 専門家が少ない小規模チームの代替手段として位置づけが明確です。

特に、フロントエンドや小規模なバックエンドチームが自律的に Kubernetes を活用できる環境づくりに貢献する可能性があります。本番への投入には追加設定が不可欠ですが、Kubernetes 入門の障壁を下げるアプローチとして注目する価値があります。
