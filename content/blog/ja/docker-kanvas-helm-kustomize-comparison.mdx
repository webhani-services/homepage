DockerとLayer5が共同開発したDocker Kanvasが注目を集めています。Docker ComposeファイルをKubernetesリソースに自動変換し、インフラ構成をビジュアルで確認できる機能を持ちます。Helm/Kustomizeとの比較と実際の活用場面を整理します。

## Docker Kanvasとは

Docker KanvasはDocker Compose定義からKubernetesリソースを自動生成するInfrastructure as Codeツールです。Layer5との協業で開発されており、Mesheryのビジュアライゼーション基盤を活用しています。

主な機能：
- Docker ComposeからDeployment/Service/PVC/ConfigMapを自動生成
- サービス間の依存関係をグラフィカルに可視化
- マルチクラスター管理のサポート

## 既存ツールとの位置づけ

### Helmとの比較

Helmはパッケージマネージャーとして成熟しており、Chartによる再利用性と依存関係管理が強みです。一方でGoテンプレート言語の複雑さと、大規模なChartのデバッグのしにくさが課題です。

```yaml
# Helmテンプレートの例 — 強力だが冗長になりやすい
containers:
  - name: {{ .Values.app.name }}
    image: "{{ .Values.image.repository }}:{{ .Values.image.tag | default .Chart.AppVersion }}"
    {{- if .Values.resources }}
    resources:
      {{- toYaml .Values.resources | nindent 12 }}
    {{- end }}
```

### Kustomizeとの比較

KustomizeはYAMLのbase + overlaysモデルで環境差分を管理します。kubectl組み込みで追加インストール不要という利点がありますが、複雑なパラメータバリエーションの表現にはHelmが優ります。

Docker Kanvasはこれらとは異なるポジションを持ちます。Docker Composeという開発者に馴染みやすいフォーマットを入口にして、Kubernetes採用の障壁を下げることを狙いとしています。

## 実際の変換フロー

```yaml
# docker-compose.yml（既存）
services:
  api:
    image: myapi:latest
    ports:
      - "8080:8080"
    environment:
      - DB_URL=postgresql://db:5432/appdb
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data

volumes:
  pgdata:
```

Kanvasはこれを元に以下のKubernetesリソースを生成します：

- `api`用の`Deployment`と`Service`
- `db`用の`Deployment`と`Service`
- `pgdata`用の`PersistentVolumeClaim`
- `DB_URL`用の`Secret`または`ConfigMap`

ビジュアルエディタでリソース間の依存関係を確認しながら、HPA（Horizontal Pod Autoscaler）やNetworkPolicy、リソースリミットを追加してからマニフェストをエクスポートできます。

## 採用時の判断基準

### Kanvasが向いているケース

- Docker Composeからの移行プロジェクト
- 小〜中規模のサービス（マイクロサービス15個以下）
- Kubernetesの習熟度が低いチームのオンボーディング
- インフラ構成のドキュメント化・可視化が目的

### Helm/Kustomizeが依然として優れているケース

- 複雑なパラメータバリエーションが必要なパッケージ配布（Helmの強み）
- 既存KubernetesのCIパイプラインへの統合（Kustomizeの強み）
- 大規模でガバナンスが求められる本番環境

## 現実的な評価

Docker Kanvasは「Docker ComposeユーザーのためのKubernetes入口」として優れたポジションを持っています。しかし本番グレードの環境では、Kanvasで生成した基本マニフェストをHelmチャート化するか、Kustomizeのoverlaysで環境差分を管理するハイブリッドアプローチが現実的です。

インフラ依存関係のビジュアライゼーションは、チームのオンボーディングやドキュメント目的で実質的な価値を提供します。YAMLを読み込まなくても依存グラフを把握できることは、新メンバーの理解を大きく助けます。

## まとめ

Docker Kanvasは新しいカテゴリのツールとして興味深い存在ですが、Helm/Kustomizeを置き換えるものではありません。Docker ComposeからKubernetesへの移行を検討しているチームにとっては、学習コストを大幅に下げる有益なツールです。すでにHelm/KustomizeでKubernetesを運用しているチームには、可視化レイヤーとしての活用が現実的です。
