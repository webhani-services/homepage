## Docker Kanvasとは

2026年1月、DockerはKanvasをリリースしました。これはDocker Composeファイルを自動的にKubernetes Manifestに変換するツールで、HelmやKustomizeが占めてきた領域に直接競合します。

KanvasはローカルのDocker Compose環境とKubernetes本番環境のギャップを埋めることを目的としています。

### 基本的な変換フロー

```yaml
# docker-compose.yml（入力）
services:
  app:
    image: myapp:latest
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: postgresql://db:5432/mydb
  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data
```

このCompose FileをKanvasに渡すと、対応するKubernetes Deployment、Service、PersistentVolumeClaimのManifestが自動生成されます。

## HelmやKustomizeとの違い

| | Helm | Kustomize | Docker Kanvas |
|---|---|---|---|
| 主な用途 | パッケージング・配布 | 環境差分管理 | Compose→K8s変換 |
| 学習コスト | 高い | 中程度 | 低い（Compose知識で使える） |
| テンプレート | HelmChart | Overlay | 自動生成 |
| 対象ユーザー | DevOps Engineer | DevOps Engineer | 開発者全般 |

Kanvasの強みは**Compose知識だけでKubernetesにDeployできる**点です。特に「ローカルはDocker Composeで動かしているが、本番はK8sにDeployしたい」という開発者にとって、設定の重複を減らせます。

ただし、既存のHelmチャートエコシステムとの互換性はなく、高度なK8sリソース管理にはまだHelmやKustomizeが必要な場面もあります。

## AWSが進めるAI-Assisted Kubernetes運用

Kanvasと並行して、AWSは2026年にKubernetes運用のAI化を積極的に推進しています。

### EKS MCP Server（Preview）

AWSがPreviewとして提供しているEKS MCP Serverは、Claude CodeのようなAI Coding AssistantがKubernetesクラスターを自然言語で操作できるようにするインターフェースです。

```bash
# Claude CodeからEKSクラスターを操作する例（概念）
claude "production EKSクラスターのweb-appデプロイメントを
image tag v2.1.0にRolling Updateしてください。
現在のPodの状態も確認してください。"
```

このような自然言語指示でClaudeがkubectlコマンドを実行し、Deploymentを更新します。

### AWS DevOps Agent（Preview）

AWS DevOps Agentは、Kubernetesクラスターの問題診断・設定最適化・ルーティン作業を自然言語で処理するAgentです。主な機能：

- Podのクラッシュ原因特定と修正提案
- Resource Requestsの最適化推奨
- セキュリティポリシーの確認

いずれもまだPreviewであり、本番クラスターの操作への適用は慎重な評価が必要です。

## K8s本番採用率82%が意味すること

CNCFの2025年 Annual Cloud Native Surveyによると、Containerユーザーの82%がKubernetesを本番環境で運用しており、そのうち66%がGenAIワークロードにも利用しています。

この数字は、K8sがすでにニッチな選択肢ではなく**クラウドネイティブ開発の標準基盤**であることを示しています。

## 2026年のKubernetes運用で変わること

### Compose→K8s変換の簡略化

Docker KanvasによってCompose知識をそのままK8sに活かせるようになり、開発者がK8s設定のボイラープレートと格闘する時間が減ります。

### 自然言語によるクラスター操作

EKS MCP ServerやDevOps Agentにより、専門的なkubectl知識がなくても基本的なK8s操作を実行できる体制が整いつつあります。ただし、本番操作は依然として専門知識を持ったエンジニアのレビューが不可欠です。

### Multi-Cloudへの対応

AWS・Azure・GCPにまたがるマルチクラウド運用が当たり前になりつつあり、特定ベンダーへの依存を避ける設計が重要性を増しています。

## まとめ

Docker KanvasはCompose→K8sの橋渡しツールとして実用的な価値があります。一方でHelmを完全に置き換えるものではなく、用途に応じた使い分けが現実的です。

AI-Assisted K8s運用（EKS MCP Server、DevOps Agent）はまだPreviewですが、運用自動化の方向性は明確です。Kubernetes運用の複雑さが解消されれば、まだK8sを未採用の18%のチームにとっても採用障壁が下がります。
