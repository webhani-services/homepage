Dockerは新しいプラットフォーム「Kanvas」を発表しました。ローカルの開発環境とクラウドのプロダクション環境のギャップを埋めることを目的としており、Docker ComposeファイルをKubernetesのマニフェストに自動変換する機能を中核に据えています。HelmやKustomizeが確立したKubernetesのデプロイメント管理領域に、Dockerが本格参入する動きです。

## Kanvas が解決しようとする問題

ローカル開発でDocker Composeを使い、本番環境でKubernetesを使うチームにとって、最も摩擦の大きい作業の一つは「Compose定義からKubernetesマニフェストへの変換と同期」です。

既存のアプローチとしては、`kompose`（オープンソース）を使った変換が一般的でしたが、Composeの更新のたびに手動での変換と調整が必要でした。KanvasはこのプロセスをDocker Desktopのワークフローに統合することで、変換を継続的かつ自動化することを目指しています。

## 実際の変換フロー

典型的なComposeファイルを例に取ります。

```yaml
# docker-compose.yml
services:
  api:
    image: my-api:latest
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: postgresql://db:5432/app
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}

volumes:
  db-data:
```

Kanvasはこれを解析し、Deployment、Service、PersistentVolumeClaim、ConfigMapなどのKubernetesリソースを生成します。生成されるマニフェストはComposeが表現している内容を正確に変換した最低限の構成です。

デフォルトでは含まれない項目: リソース制限、Liveness/Readiness Probe、HorizontalPodAutoscaler、環境ごとの設定上書き。これらは手動で追加する必要があります。

## Helm / Kustomize との比較

| | Kanvas | Helm | Kustomize |
|---|---|---|---|
| 入力形式 | docker-compose.yml | Chart (templates) | base + overlays |
| 学習曲線 | Compose知識があれば低い | Helm固有の知識が必要 | Kubernetesの基礎知識が必要 |
| カスタマイズ性 | 現状は限定的 | 高い | 高い |
| エコシステム | Docker Desktop統合 | 豊富なChartリポジトリ | kubectl組み込み |
| 本番成熟度 | 初期段階 | 非常に成熟 | 成熟 |

KanvasはComposeを起点とするため、Docker Composeを使いこなしているチームの入門障壁を下げる効果があります。一方で、Helmが持つ豊富なパッケージエコシステムや、Kustomizeの柔軟なオーバーレイ機構には現段階では及びません。

## Kanvasが有効なシナリオ

**Kanvasを選ぶ場合:**
- 小規模プロジェクトまたはPoC（概念実証）
- Composeから初めてKubernetesに移行するチーム
- Docker Desktopを開発環境の中心に置いているチーム

**既存ツールが引き続き有効な場合:**
- 複雑なデプロイメント要件（複数環境、Blue/Greenデプロイ等）
- 豊富なHelmチャートエコシステムを活用したいケース
- 既にHelmまたはKustomizeで確立したワークフローを持つチーム

## 現時点での不足点

KanvasがHelmとKustomizeに対して本格的な競合になるかどうかは、以下の機能の充実にかかっています。

1. **Secret管理の統合**: KubernetesのSecret、Sealed Secrets、External Secrets Operatorとの連携
2. **マルチ環境サポート**: staging/production等の環境ごとの設定上書き
3. **GitOpsとの統合**: ArgoCD/FluxとのCI/CDパイプラインへの組み込み

これらはプロダクション環境での大規模導入において必須の要素ですが、実験や評価の障壁にはなりません。

## 実践的な結論

Docker Composeで開発しながらまだKubernetesに移行していないチームには、Kanvasは転換期の障壁を下げる実験として価値があります。すでにHelmやKustomizeをプロダクションで運用しているチームには、現時点での移行コストに見合うメリットはありません。

Dockerが持つ開発者コミュニティのリーチを考えると、ComposeユーザーをKubernetesに引き込む橋渡しとして一定の役割を果たす可能性はあります。成熟を待ちながら注目すべきツールです。
