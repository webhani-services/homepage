Kubernetes のアップグレードは、長い間「一方通行」でした。バージョンアップして何か壊れたら、バックアップからのリストア、あるいはクラスターの作り直し — どちらも遅く、危険です。特に本番環境での深夜オペはやりたくない選択肢でした。

その状況が変わります。Amazon EKS がバージョンロールバック機能をリリースしました。アップグレードから 7 日以内であれば、control plane のバージョンを前の状態に戻せます。スナップショットからの復元ではなく、実際に本番で動いていた前のバージョンへの完全な復帰です。

## ロールバックの仕組み

EKS クラスターをアップグレードするとき、AWS は前のバージョンを記録しています。アップグレード後に問題が生じた場合 — API 互換性エラー、ワークロードのエラー、予期しない動作 — ロールバックを実行して前の状態に戻せます。

ロールバックには安全弁が組み込まれています：

1. **ロールバック準備状態の確認**: ロールバックを実行する前に、EKS が安全性を自動判定します。チェック項目は API 互換性、ノード間のバージョンスキュー、add-on の互換性、クラスター全体の健全性です。
2. **7 日間のウィンドウ**: アップグレードから 7 日以内にロールバックを開始する必要があります。その後、前のバージョンは利用できなくなります。
3. **完全なバージョン復帰**: 部分的な復元ではなく、前の Kubernetes バージョンに完全に戻ります。検証済みで運用実績のある状態です。

EKS Auto Mode クラスターの場合、control plane とマネージドノードが一緒にロールバックします。ノードのロールバックは Pod Disruption Budget (PDB) を尊重するため、ワークロード設定によっては時間がかかる場合があります。

## なぜこれが重要か

根本的な問題：**チームはサポート終了前の Kubernetes バージョンにアップグレードしたい**。古いバージョンはセキュリティリスクになるからです。しかしアップグレードは危険が伴う — API の変更、スケジューラー動作の変化、ワークロード側の予期しない不具合など。従来は入念なステージング検証しか対策がなく、それでも本番完全には不安があります。

ロールバック機能があれば、その計算が変わります。こんなことができるようになります：

- より自信を持ってアップグレードできる
- 新しい API サーバーに対してスモークテストと統合テストを実行する
- 限定的な期間内（24～48 時間程度が一般的）で重要メトリクスを監視する
- 推測ではなく、本番のシグナルに基づいてロールバック判定を下す

## ロールバック準備状態の確認

ロールバックを実行する前に、EKS が自動チェックを走らせ、ロールバックが可能かどうかを報告します：

- **API 互換性**: ワークロードマニフェストは前の API バージョンで動作するか？
- **Add-on 互換性**: インストール済みの EKS add-on (CoreDNS、VPC CNI、kube-proxy など) は前の control plane バージョンと互換性があるか？
- **バージョンスキュー**: ノード側 Kubernetes バージョンはロールバック対象バージョンとのサポート範囲内にあるか？
- **クラスター健全性**: 未解決のエラーやクラスター全体の問題はないか？

これらのチェックは自動実行され、ロールバック判定の流れで報告されます。問題がフラグされた場合は、ロールバック前に対処するか、新しいバージョンで原因を修正するか決定できます。

## 実践的なアップグレードランブック

ロールバックをゲートに組み込んだ、現実的なアップグレードワークフロー：

1. **Control plane をアップグレード**（メンテナンスウィンドウまたはオフピーク時）
   ```bash
   aws eks update-cluster-version \
     --name my-cluster \
     --region ap-northeast-1 \
     --kubernetes-version 1.31
   ```

2. **アップグレード進捗を監視**: クラスターが新しいバージョンに到達するまで待機

3. **スモークテスト**（直後に実行）：
   - テスト用 Pod をデプロイして API サーバーと通信可能か確認
   - 重要なワークロードが起動しているか確認
   - ログから API 廃止予定警告を確認

4. **統合テストを 24～48 時間かけて実行**：
   - 主要なアプリケーションフローを動作確認
   - 可視性パイプライン (メトリクス、ログ、トレース) を検証
   - SLO ダッシュボードで異常を確認

5. **ロールバック判定ゲート**（2～3 日目）：
   - 重大な問題が浮上したら、ロールバック準備状態を確認：
     ```bash
     aws eks describe-cluster-version-rollback-status \
       --cluster-name my-cluster \
       --region ap-northeast-1
     ```
   - ロールバック安全な場合は実行：
     ```bash
     aws eks rollback-cluster-version \
       --cluster-name my-cluster \
       --region ap-northeast-1
     ```
   - ロールバック非推奨 (add-on 互換性問題、健全性問題) の場合は新バージョンで原因対処

6. **ノードロールバック** (必要な場合): Auto Mode クラスターの場合、ノードは control plane と並行してロールバックし、PDB を尊重します。

7. **アップグレード完了判定**（7 日目）: 7 日間のウィンドウが終わってロールバック実行がなければ、前のバージョンは利用不可になり、アップグレードが確定します。

## 重要な制限事項

- **7 日間のウィンドウ**: 迅速な判断が必要です。ロールバックは長期的なエスケープハッチではなく、発見した問題への安全弁です。
- **全体ロールバック**: 個別コンポーネントや add-on だけではなく、クラスター全体をロールバックします。
- **PDB 対応ノードロールバック**: Auto Mode の場合、ノードは disruption budget を守りながらロールバックします。厳しい PDB を持つ大規模クラスターは数時間かかる可能性があります。
- **コスト**: 標準的な EKS とコンピュート費用が発生します。ロールバック操作自体に追加料金はありません。
- **前のバージョンのみ**: 直前のバージョンにのみロールバック可能。複数バージョン前にはロールバックできません。

## 利用可能性

この機能は今すぐ、追加料金なしで、EKS が利用できるすべての商用 AWS リージョンで利用可能です。EKS コンソール、AWS CLI、AWS SDK から実行できます。

## ビジネス観点

CNCF の最近の Cloud Native Survey によると、本番環境で実行されるコンテナワークロードの約 82% が Kubernetes です。その規模ではアップグレードは常に発生し、アップグレード リスクを低減する仕組みは実務的な価値があります。

ロールバック機能はステージング検証の代替ではなく、可視性の上に成り立つ仕組みです。「祈りながらアップグレード」から「高速に検証して実データで判定する」へ、アップグレードの心構えが変わります。
