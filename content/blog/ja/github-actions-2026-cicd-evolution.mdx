2026年、GitHub Actions に大きな変化が訪れた。2025年末から段階的に進んでいた価格体系の見直しと、バックエンドアーキテクチャの刷新が相次いで実施された。毎日 7,100 万件以上のジョブを処理するプラットフォームの変化は、現在 GitHub Actions を使っているチームに直接影響する。

## 価格体系の変化

### ホステッドランナーの値下げ

2026年1月1日より、GitHub の管理するホステッドランナーの価格が最大 **39% 引き下げ**られた。特に Linux ランナーが値下げの恩恵を受けており、従量課金で Actions を使っているチームには直接コスト削減につながる変化だ。

### セルフホストランナーの有料化

一方、2026年3月より **セルフホストランナーが無料ではなくなった**。これまでセルフホストランナーは無料で利用できたが、今後は実行分数に応じた課金が発生する。

この変化は、主にコスト理由でセルフホストランナーを採用していたチームに影響する：

```yaml
# セルフホストランナーを使用していた典型的な設定
jobs:
  build:
    runs-on: self-hosted  # ← 有料化対象
    steps:
      - uses: actions/checkout@v4
      - run: npm ci && npm test
```

セルフホストランナーを使い続ける理由が「特殊なハードウェア」「セキュリティ要件」「カスタムソフトウェア」にある場合は引き続き有効な選択肢だ。しかし「ホステッドランナーより安いから」という理由だけであれば、ホステッドランナーへの移行を再検討する価値がある。

## アーキテクチャの刷新

GitHub は 2025年中にジョブキューとランナー管理の基盤インフラを全面的に再設計し、すべての Actions ジョブが新アーキテクチャ上で動作するようになった。

この変更の主な目的：
- **可用性の向上**: インフラ障害への耐性強化
- **スロットリングの削減**: 高負荷時のジョブ滞留を減少
- **パフォーマンスの改善**: 全体的なジョブ実行時間の短縮

現在 GitHub Actions は 1日 **7,100 万ジョブ**以上を処理しており、新アーキテクチャはその規模に最適化されている。

## 主要な機能追加

### キャッシュ上限の撤廃

従来の 10GB キャッシュ制限が撤廃された。モノレポや大規模な依存関係を持つプロジェクトでは、キャッシュが溢れてキャッシュヒット率が下がるという問題が頻繁に発生していた。

```yaml
- uses: actions/cache@v4
  with:
    path: |
      ~/.npm
      ~/.cache/cypress
      node_modules
    key: ${{ runner.os }}-deps-${{ hashFiles('package-lock.json') }}
    # 以前は 10GB 上限に引っかかる大規模プロジェクトが多かった
```

### 再利用可能ワークフローのネスト対応強化

再利用可能ワークフローのネスト上限が **10レベル**に、1回の実行あたりのワークフロー呼び出し上限が **50件**に拡張された。

```yaml
# ワークフローの呼び出し連鎖が深いモノレポ構成に対応
jobs:
  setup:
    uses: ./.github/workflows/setup.yml

  test:
    needs: setup
    uses: ./.github/workflows/test-matrix.yml  # このワークフロー内でさらに呼び出し可能

  deploy:
    needs: test
    uses: ./.github/workflows/deploy.yml
```

大規模なモノレポや、共通ワークフローを複数レベルで再利用するアーキテクチャにとって、これは実用上の制約が大幅に緩和されることを意味する。

### 組織内ワークフローテンプレート

組織レベルのワークフローテンプレートが、`.github` リポジトリから **非公開（non-public）** で提供できるようになった。

```
.github リポジトリ構造:
.github/
  workflow-templates/
    node-ci.yml          # 組織メンバーのみが利用可能
    deploy-production.yml
    security-scan.yml
```

チーム全体が共通の CI パターンから作業を始められるようになり、「各リポジトリで微妙に違う CI 設定」という問題が解消しやすくなる。

### スケジュールジョブのタイムゾーン対応

Q1 2026 の追加として、スケジュールジョブがタイムゾーンを指定できるようになった。これまでは UTC 固定だったため、「毎日午前 9時（JST）」のようなスケジュールを組む際に計算が必要だった。

```yaml
on:
  schedule:
    # JST 午前 9:00 に実行 (UTC+9)
    - cron: '0 9 * * 1-5'
      timezone: 'Asia/Tokyo'
```

日本や韓国のチームで「なぜこの時間に実行されるのか」と混乱していたケースが解消される。

## セルフホストからホステッドへの移行判断

セルフホストランナーを使っているチームが移行を検討する際の判断軸：

**ホステッドランナーへの移行が適している場合:**
- セルフホストの主な動機がコスト削減だった
- 特殊なハードウェア（GPU、大容量メモリ）が不要
- カスタムソフトウェアのインストールが `actions/setup-*` や Docker で代替可能

**セルフホストを維持すべき場合:**
- セキュリティ要件で GitHub のホステッドインフラが使用不可（オンプレミスネットワーク内での実行が必須）
- 特殊なハードウェア（GPU 推論、ARM など）が必要
- 非常に大規模なビルドで専用リソースが必要

移行時は、現在の月次コストと新価格でのホステッドランナーコストを比較するのが第一歩だ。

## 実用的な対応チェックリスト

現在 GitHub Actions を使っているチームが今月確認すべき点：

- [ ] セルフホストランナーを使用している場合、有料化後のコストを試算する
- [ ] ホステッドランナーへの移行可否を評価する
- [ ] 再利用可能ワークフローが複雑になりすぎていないか見直す（今なら整理しやすい）
- [ ] キャッシュ設定を最適化する（上限撤廃を活かして依存関係キャッシュを拡充）
- [ ] スケジュールジョブのタイムゾーン設定を更新する

## まとめ

2026年の GitHub Actions は、価格体系の合理化と機能的な成熟の両面で大きく前進した。

セルフホストランナーの有料化は、コスト理由だけで採用していたチームには再検討が必要なインパクトだ。一方、ホステッドランナーの値下げ、キャッシュ上限撤廃、再利用可能ワークフローの強化は、ホステッドで完結できるチームには純粋なメリットになる。

変化が多い時期だが、チームの具体的なユースケースに照らして判断すれば、おおむね合理的な選択ができるはずだ。
