## ダッシュボードから「強制力」を持つ仕組みへ

コード品質の可視化ツールは以前から数多く存在する。CodeQL によるスキャン結果を眺め、カバレッジのグラフを確認する——ここまでは、あくまで「参考情報」の域を出ないことが多かった。開発者が無視しようと思えば無視できる、ダッシュボード止まりの機能だったからだ。

2026年7月20日、GitHub Code Quality が Public Preview から一般提供（GA）へ移行する。この変化で重要なのは新機能の追加そのものより、**保守性・信頼性・カバレッジの閾値を下回った PR のマージを実際にブロックできる**品質ゲートが、有償の正式機能として組み込まれる点だ。ダッシュボードだった仕組みが、強制力を持つゲートに変わる。webhani として、この移行が現場に与える影響を整理したい。

## GA で何が変わるのか

GitHub Code Quality は GitHub Enterprise Cloud と GitHub Team プランで利用可能で、GitHub Enterprise Server では提供されない。GA 後の課金体系はアクティブなコミッター単位の月額ライセンスとなり、これには次の機能が含まれる。

- 保守性・信頼性・カバレッジに基づく **PR マージブロックの品質ゲート**
- Ruleset との統合、Organization 全体への展開
- CodeQL ベースの決定論的解析（GitHub Actions 上で実行）
- AI を用いた検出支援と Copilot Autofix の生成（従量課金が別途発生）

注目すべきは、AI 関連機能が従量課金として別枠になっている点だ。品質ゲート自体のライセンス費用に加え、AI による検出支援や自動修正提案を使うほど課金が積み上がる構造になっている。

```yaml
# Ruleset でマージ条件に Code Quality のステータスチェックを組み込むイメージ
name: main branch protection
target: branch
enforcement: active
rules:
  - type: required_status_checks
    parameters:
      required_status_checks:
        - context: "code-quality/maintainability"
        - context: "code-quality/reliability"
        - context: "code-quality/coverage"
```

## なぜ「導入前の監査」が必要なのか

品質ゲートが実際に PR をブロックする以上、導入時の設計を誤ると開発速度そのものを止めかねない。webhani がクライアントに推奨する監査項目は次の通りだ。

- **既存コードベースの現在地を先に測る**: ゲートを有効化する前に、まず計測のみを行うモードでベースラインを把握する。いきなり閾値でブロックを始めると、既存の技術的負債がある巨大リポジトリでは PR が一切マージできなくなるリスクがある。
- **閾値の粒度をリポジトリごとに調整する**: 新規プロジェクトと長期運用されているレガシーコードでは、求めるべき保守性・カバレッジの水準が異なる。全社一律の閾値は現実的でないことが多い。
- **AI 従量課金の上限を確認する**: Copilot Autofix の自動生成は便利だが、大量の既存指摘に対して一括で修正提案を走らせると、想定外の課金が発生する可能性がある。段階的な有効化を検討すべきだ。
- **コミッター数ベースの課金モデルを組織構造に照らして試算する**: 外部委託や短期契約の開発者が多い組織では、アクティブなコミッター数の変動を踏まえたコスト試算が必要になる。

## 開発フローへの実務的な影響

品質ゲートが機能する最大の意義は、コードレビューの負荷分散にある。これまで人間のレビュアーが指摘していた保守性や重複コードの問題を、機械的なチェックとして PR の入口で弾けるようになれば、レビュアーはより設計判断やビジネスロジックの妥当性といった、人間にしか判断できない領域に集中できる。

一方で、閾値設計を誤ったゲートは「レビュアーの代わり」ではなく「開発のボトルネック」になる。特に移行期のレガシーコードベースでは、段階的な閾値引き上げロードマップを事前に用意しておくことが欠かせない。

## webhani としての見解

GitHub Code Quality の GA 移行は、コード品質管理が「あると便利な機能」から「組織のガバナンス基盤」へ格上げされる転換点だと捉えている。特に PR ベースの開発フローを持つ組織にとって、マージ時点で品質基準を強制できる仕組みは、レビュープロセスの標準化に直結する。

ただし、有償化のタイミングでもあるため、既存の CI パイプラインに CodeQL や他の静的解析ツールをすでに組み込んでいる組織は、機能の重複がないか、コスト対効果は見合うかを個別に精査すべきだ。webhani では、GA 移行前の計測フェーズから伴走し、閾値設計とロールアウト計画の策定を支援することを推奨している。

## まとめ

2026年7月20日の GA 移行で、GitHub Code Quality はダッシュボードから実際に PR をブロックする強制力のある仕組みへと変わる。導入自体は開発フローの品質を底上げする好機だが、閾値設計を誤れば開発速度の低下という代償を払うことになる。既存コードベースの計測、段階的な閾値設定、AI 従量課金の見積もり——この3点を有効化前に済ませておくことが、円滑な移行の鍵になる。
