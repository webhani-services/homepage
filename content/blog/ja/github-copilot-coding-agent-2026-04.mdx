GitHub Copilot は 2026 年初頭にかけて、補完ツールから「自律的に動くエージェント」へと大きく変わりました。中でも注目は **Coding Agent** — GitHub Issue を Copilot にアサインすると、コードを書き、テストを実行し、PR を開いてレビューを待つ、という一連の作業を自動でこなします。

## Coding Agent の仕組み

Coding Agent は Issues → Code → Tests → PR という流れを自律的に処理します。

```
1. Issue を @copilot にアサイン
2. Copilot がリポジトリをクローン・解析
3. 変更箇所を特定してコードを実装
4. テストを実行し、エラーがあれば自己修正
5. Draft PR を作成してレビュー依頼
```

GitHub の Copilot は VS Code と JetBrains IDEs の両方で、ファイルの編集対象を自己決定し、ターミナルコマンドを実行し、エラーをループで修正する **フルエージェントループ** が動くようになりました。

## Agentic Code Review

2026 年 3 月にリリースされた **Agentic Code Review** も重要な変更です。従来の「コメントを貼るだけ」のレビューではなく、PR 全体のコンテキストを収集した上で問題を指摘し、そのまま Coding Agent に修正 PR を作らせることができます。

```yaml
# .github/copilot-instructions.yml の例
review:
  auto_fix: true          # 指摘事項を自動修正
  assign_agent: true      # 修正を Coding Agent に委譲
  scope:
    - security
    - performance
    - test_coverage
```

## Copilot CLI の GA

2026 年 3 月、**Copilot CLI** が有料プラン向けに GA になりました。ターミナルから直接エージェントを起動でき、Plan モードと Autopilot モードを切り替えられます。

```bash
# Plan モード：実行前に計画を確認
gh copilot suggest --plan "Add rate limiting to /api/users endpoint"

# Autopilot モード：確認なしで自律実行
gh copilot run --autopilot "Refactor the payment module to use the new Stripe SDK"
```

Plan モードでは変更内容を事前に提示するため、予期しない変更を防ぎやすくなっています。

## 実務での使いどころ

現時点では、Coding Agent が特に効果を発揮するのは次のようなケースです。

**向いているタスク**
- 明確な仕様がある機能追加（新しい API エンドポイントの追加など）
- ライブラリのバージョンアップと関連する修正
- テストカバレッジの追加
- ドキュメント更新

**まだ人の判断が必要なケース**
- 設計上のトレードオフを含む変更
- ビジネスロジックが複雑なリファクタリング
- セキュリティ上の判断が必要な実装

実際に使ってみると、Coding Agent は「完璧なコードを書く」のではなく「実装の叩き台を素早く用意する」ツールとして機能します。PR のコードを必ずレビューし、マージ前に人が確認する前提で運用するのが現実的です。

## JetBrains IDE の強化

2026 年 3 月、JetBrains IDE 向けに Custom Agents・Sub-agents・Plan Agent が GA になりました。`auto-approve` でMCP ツールを承認なしで呼び出せるため、IntelliJ や PyCharm から直接 DB 操作やファイル生成を含むワークフローを組めます。

```kotlin
// Plan Agent の設定例 (JetBrains)
copilot {
    agent {
        name = "api-generator"
        tools = listOf("filesystem", "terminal", "database-schema")
        autoApprove = setOf("filesystem.read", "terminal.run_tests")
    }
}
```

## チームでの運用上の注意点

Coding Agent を導入する際には、いくつかの運用ルールを決めておくと混乱が少ないです。

1. **Branch 命名規則** — Agent が作成する Branch を識別しやすくする（例: `copilot/issue-123-add-rate-limiting`）
2. **レビュー担当の明確化** — Agent が開いた PR は必ず人がレビューするルールを明文化
3. **スコープの制限** — `copilot-instructions.yml` で操作を許可するディレクトリやファイルを限定
4. **テストの充実** — Agent のアウトプット品質はテストの網羅率に比例する

## まとめ

GitHub Copilot の Coding Agent は、現時点では「完全な自動化」よりも「タスクの実装コストを下げるアシスタント」として評価するのが妥当です。Issue から PR まで一気に進むフローは試す価値がありますが、最終的な判断は常に人が行う設計にしておくことが大切です。

Agentic Architecture は急速に成熟しており、2026 年を通じてさらに能力が上がることが見込まれます。今のうちに小さいタスクから試して、チームの運用フローに組み込む方法を模索しておくと良いでしょう。
