## また新しい Model、また評価サイクル

xAI は今週、Grok 4.5 を投入しました。汎用 Chat Model としてではなく、明確に Coding と Agentic Task に照準を合わせた位置付けです。xAI の API、xAI 自身の Grok Build という Agent 製品、そして全 Plan で使える Cursor から利用可能で、報道によれば Cursor Team と共同で Training されたとのこと——数週間後に取ってつけたように追加された Provider ではなく、Launch 時点から統合が緊密である理由がここにあります。

すでに Coding Assistant を標準化しているチームにとっては、発表をざっと眺めて読み飛ばしたくなるところです。しかし私たちはそれを誤った判断だと考えます。Grok 4.5 が必ずしも自社の Stack に入るべきだからではなく、新しい Frontier Model が出るたびに取るべき正しい対応は、毎回同じ「軽量な評価」であって、無視することでも全面的な乗り換えでもないからです。

## 今回、実際に新しい点

今回の Release で、一般的な Marketing Noise と切り分けて見るべき点が3つあります。

**価格**: Grok 4.5 は Input 100万 Token あたり $2、Output 100万 Token あたり $6 で Launch しました。比較対象となる Frontier Coding Model と比べて積極的な価格設定であり、しかも長い Context と短い出力という構成のワークフロー（大規模な Codebase の Context を入力し、対象を絞った Diff を出力する——Agentic Coding Task の大半に当てはまる形）に有利な非対称性を持っています。

**スループット**: xAI はデフォルト設定でおよそ秒間 80 Token を報告しており、推論の深さ（Low / Medium / High）も設定可能です。対話的な Coding Session では、生成速度は生の性能と同じくらい体感的な反応の良さを左右します。Benchmark Score が高くても、遅い Model は使っていて悪く感じられることがあります。

**配布方法**: 自分で設定する Third-Party Model としてではなく、Launch 時点で Cursor に組み込まれている点は、見た目以上に重要です。Model Provider が「API 経由で自分で組み込んでもらう」ことより「開発者がすでに開いている Tool の中で使える」ことを優先する傾向を強めていることを示しています。

## 私たちが新しい Model Release をどう評価しているか

私たちは Launch 時の Benchmark だけを根拠に新しい Frontier Model を Client の実務に採用することはありません。それらは Provider が選んだ Benchmark であり、実際の Codebase や Task 構成を反映しているとは限らないからです。私たちの評価 Checklist は次の通りです。

1. **合成 Benchmark ではなく、自分たちの Repository で実行する**: Backlog から実際の Ticket を3〜5件選ぶ（Bug 修正、小さな機能追加、Refactoring など）。同じ Ticket を現在使っている Model と新しい Model の両方で走らせる。
2. **Pass/Fail だけでなく、Working Diff に至るまでの Token 数を測る**: 少ない反復回数で正しい解決策にたどり着く Model は、単純な正答率が示す以上の価値があります。Agentic Workflow では反復のコストが積み重なるためです。
3. **Tool 呼び出しの信頼性を推論の質と切り離して確認する**: Agentic Coding は、File 読み書きや Shell 実行といった Tool を正しい順序で呼び出せるかどうかに大きく依存します。これは Code の質とは別の能力であり、独立したテストが必要です。
4. **見出しの Rate ではなく、自分たちの実際の利用パターンで価格を試算する**: Turn ごとに大きな Context Window を読み直す Input 重視の Agentic Session は、同じ価格表の下でも短く独立した Completion とはまったく異なる挙動を示します。

```typescript
// 公開 Benchmark ではなく、自分たちの Ticket で
// Model 同士を比較するための最小限の Harness
interface EvalTicket {
  id: string;
  prompt: string;
  repoSnapshot: string;
  expectedDiffPattern: RegExp;
}

async function evaluateModel(
  model: string,
  tickets: EvalTicket[]
): Promise<{ id: string; iterations: number; passed: boolean }[]> {
  const results = [];
  for (const ticket of tickets) {
    const { diff, iterations } = await runAgenticSession(model, ticket);
    results.push({
      id: ticket.id,
      iterations,
      passed: ticket.expectedDiffPattern.test(diff),
    });
  }
  return results;
}
```

## ほとんどのチームがすでに直面している Multi-Model の現実

私たちが関わるチームの中で、単一の Model だけを使い続けているところはごく少数です。一般的なパターンは、日常の Agentic Coding には主力 Model を使いつつ、特定の Task 種別には別の Model へ Route するというものです——Cost 意識が強い大量処理はより安価な Model へ、複雑な Architecture 判断はその時点でその軸に強い Model へ、といった具合です。Grok 4.5 の価格設定と Cursor Native な提供形態は、こうした構成の「Cost 意識・高 Volume」枠の有力な候補になり得ます。必ずしも最難関の推論 Task を担っている Model を置き換える必要はありません。

だからこそ、統合 Layer での Vendor Lock-in は積極的に避ける価値があります。Agentic Coding の構成が特定 Vendor の SDK に直結していると、Grok 4.5 のような新規参入者を評価するだけで、乗り換える価値があるかを試す前に統合 Code を書き直す羽目になります。Model に依存しない Interface——Gateway 的な抽象化であれ、Provider をまたいで一貫した Prompt / Tool Call の形式であれ——を経由させておけば、評価のコストは低く保てます。

## webhani としての推奨事項

- **四半期に半日を新しい Model 評価に確保する**: 毎回同じ固定 Ticket Set を使い、Provider 自身の Benchmark 主張だけでなく、Release 間で比較可能な結果を得る。
- **すべての Release を追いかけない**: 意味のある差別化がある Launch（新しい Provider、目立つ価格変動、新しい Tool 使用能力）を評価対象とし、細かい Point Release まで追わない。
- **「Coding が得意か」と「Agentic な Tool 使用が得意か」を分けて考える**: 両者は相関はあるが同一ではなく、Agentic Workflow は前者より後者の軸で失敗することの方が多いです。
- **統合 Layer を Model に依存させない構成にする**: Frontier Model の勢力図がどれだけ速く動いても、評価コストを低く保てます。

## まとめ

Grok 4.5 は Marketing 目的だけの発表ではなく、正当な新規参入です——競争力のある価格、Cursor Native な配布、Coding 優先の設計はいずれも実務上意味を持ちます。しかし、この分野を Release ごとに観察してきて得られるより持続的な教訓は、個々の Model そのものより、最新の Model を安価かつ繰り返し評価できるチームの能力の方が重要だということです。その能力を一度身につけてしまえば、今後 Grok であれ GPT であれ Claude であれ、新しい Release は議論の対象ではなく半日で下せる判断になります。
