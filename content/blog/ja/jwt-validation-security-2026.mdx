2026年に公開された重大な脆弱性 CVE-2026-29000 は、広く使われる Java セキュリティライブラリ（pac4j）の JWT authenticator が暗号化された JWT を処理する際の認証バイパスを記述しています。これは、同じ根本原因を共有する JWT 問題の長い系譜に連なります——トークンの暗号自体は問題なかったのに、その周りのコードが「検証すべきもの」を信頼してしまったのです。

JWT は予測可能な形で失敗し、そのほぼすべては「攻撃者が署名アルゴリズムを破った」ではありません。検証の抜け——ライブラリや呼び出し側のコードが、拒否すべきトークンを受け入れてしまう——です。本稿では繰り返し現れる失敗パターンと、正しい検証がどう見えるかを Node.js の例で解説します。原則はどの言語にも通用します。

## ヘッダーは攻撃者が制御できる

最も内面化すべきこと：JWT のヘッダーはトークンを送る者が供給します。特に `alg` フィールドは「事実」ではなく「主張（claim）」です。歴史的に、2 つの古典的バイパスはどちらもヘッダーに存在します。

1 つ目は `alg: none` バイパス。一部のライブラリは「このトークンはアルゴリズムなし」と告げられると、親切にも署名検証を完全にスキップしていました。2 つ目は RS256-to-HS256 の取り違え。RSA 署名トークン（公開鍵で検証）を期待するエンドポイントに HMAC 署名トークンを渡すと、素朴な検証器は RSA の *公開* 鍵を HMAC の *秘密鍵* として使ってしまう——公開鍵なので、攻撃者は自由に偽造できます。

両者への防御は同じです。トークンに「自分をどう検証するか」を決めさせないこと。サーバー側でアルゴリズムを固定します。

```javascript
import jwt from "jsonwebtoken";

// 誤り — トークンのヘッダーが主張する alg を信頼してしまう
jwt.verify(token, key); // none / 取り違えたアルゴリズムを受け入れうる

// 正しい — サーバーが許容するアルゴリズムを指定する
const payload = jwt.verify(token, publicKey, {
  algorithms: ["RS256"],       // 固定し、それ以外はすべて拒否
  issuer: "https://auth.example.com",
  audience: "api.example.com",
});
```

`algorithms` の許可リストは任意ではありません。これがなければヘッダーが決める——そしてヘッダーは攻撃者の入力です。

## 暗号化と署名は異なる保証

pac4j のアドバイザリは暗号化された JWT に関わるもので、この 2 種類のトークンが日常的に混同されるため、掘り下げる価値があります。

**署名された** JWT（JWS）は *完全性と真正性* を保証します——誰が発行したか、改ざんされていないかが分かります——が、ペイロードは誰でも読めます（暗号化ではなく base64）。**暗号化された** JWT（JWE）は *機密性* を保証します——ペイロードは隠されます——が、暗号化だけでは送信者を認証しません。暗号化されていても署名や完全性が適切にチェックされていないトークンは、検証パスに抜けがあれば依然として偽造やリプレイが可能です。

実践的な要点：暗号化は認証ではありません。システムが JWE を使うなら、コードは依然として「トークンが信頼できる発行者から来た」ことを適切な仕組みで検証しなければならず、「復号に成功した」を「真正である」と扱ってはなりません。この領域のバイパスバグは、信頼できないトークンを拒否すべきところで有効なセッションを返してしまう検証分岐から生じるのが典型です。

## 依存するすべてのクレームを検証する

署名チェックはトークンが改変されていないことを証明します。それが *あなた宛て* か、*まだ有効* か、*信頼できる相手から* かについては何も語りません。これらは別々のチェックであり、省略することが実システムの侵害される場所です。

```javascript
import jwt from "jsonwebtoken";

function verifyAccessToken(token) {
  const payload = jwt.verify(token, publicKey, {
    algorithms: ["RS256"],
    issuer: "https://auth.example.com",   // 誰が署名したか
    audience: "api.example.com",          // 誰宛てか
    clockTolerance: 5,                    // 小さなクロックずれ許容（秒）
    maxAge: "15m",                        // 古すぎるトークンを拒否
  });

  // exp/nbf は上の verify() が強制するが、アプリケーション層の
  // クレームはあなたの責任：
  if (payload.token_use !== "access") {
    throw new Error("access トークンのエンドポイントに refresh トークンが提示された");
  }
  if (!payload.scope?.split(" ").includes("api:write")) {
    throw new Error("スコープ不足");
  }
  return payload;
}
```

ここで避けるべき 2 つの誤り。access トークンが期待される場所で refresh トークンを使わないこと——`token_use` 相当のクレームを確認します。そして `sub` だけで認可しないこと——`iss` と `aud` を検証し、*別の* サービス向けに発行された有効なトークンが、あなたのサービスに対してリプレイされないようにします。

## ローテーション、失効、そしてステートレスの罠

JWT が魅力的なのはステートレスだからです——署名を検証すれば完了、データベース参照なし。まさにその性質が弱点でもあります——ステートレスなトークンは期限切れ前に失効できません。トークンが漏れれば `exp` まで有効です。

実践上重要な緩和策は 2 つ。access トークンを短命に保ち（日単位ではなく分単位）、長命性は refresh トークンに頼ることで、漏れた access トークンのウィンドウを小さくします。高価値な操作には失効リスト——トークン識別子（`jti`）をキーにした小さく高速なストア（Redis が定番）——を維持し、機密ルートで確認します。

```javascript
async function assertNotRevoked(payload, store) {
  if (payload.jti && (await store.exists(`revoked:${payload.jti}`))) {
    throw new Error("トークンは失効済み");
  }
}
```

これは参照を再導入し、「ステートレス」の魅力を部分的に打ち消します——が、必要な場所だけです。ほとんどの読み取りエンドポイントはステートレスのままにでき、即時失効を尊重すべき少数の操作だけが参照コストを払います。

## メンテナンスされたライブラリに頼り、最新に保つ

pac4j の CVE は運用上の点も強調します。JWT 処理はまさに「自分で書くべきでない」種類のコードであり、まさに「パッチを当て続けるべき」種類の依存関係です。微妙な検証ロジック——アルゴリズム固定、鍵選択、JWE 処理——こそ、自作実装も古いライブラリも同じように誤る場所です。信頼できるライブラリを使う場合でもアルゴリズムを明示的に固定し、認証依存関係のセキュリティアドバイザリを「バックログ」ではなく「今すぐパッチ」項目として扱ってください。

## Webhani の見解

数ヶ月ごとにまた別の JWT CVE が来ますが、パターンはほとんど変わりません——暗号は持ちこたえたのに、検証分岐が誤ったものを信頼した。これは朗報です。防御が既知で安価だという意味だからです。ヘッダーが選べないよう、アルゴリズムをサーバー側で固定する。署名だけでなく、リクエストごとに `iss`・`aud`・`exp`・トークン用途クレームを検証する。署名・暗号化・両方のどれが必要かを理解し、「復号できた」を「真正」と決して扱わない。トークンを短命に保ち、即時性が重要な場所にだけ失効を追加し、認証ライブラリにパッチを当て続ける。どれも奇抜ではありません。システムが「相手が誰か」を決める境界で、一貫して適用される規律です。
