Kubernetes v1.36が2026年4月末にリリースされます。同時に、AIを活用したクラスター運用ツールの台頭が、従来のYAML中心の運用プロセスを大きく変えつつあります。v1.36の主要変更点とAI-powered Kubernetesツールの現状を整理します。

## Kubernetes v1.36 の主要変更点

### kubelet API AuthorizationのGA昇格

v1.36で最も実用的な変更の一つは、fine-grained kubelet API authorizationのGA（General Availability）昇格です。

kubelet APIは、Pod logs、exec、メトリクス取得、ノードレベルの操作を制御します。従来はクラスター全体の広い権限を持つユーザーやService Accountであれば、kubelet APIにも広範囲にアクセスできました。これは、クラスター侵害時の横展開リスクを高めていました。

GA昇格により、Webhook authorizationモードが安定版として使用可能になります：

```yaml
# /var/lib/kubelet/config.yaml
authorization:
  mode: Webhook
  webhook:
    cacheAuthorizedTTL: 5m0s
    cacheUnauthorizedTTL: 30s
```

既存クラスターへの影響：CI/CDパイプラインや監視エージェントが持つkubelet APIへのアクセス権限を事前に棚卸しする必要があります。Webhookモードに切り替える前に、ステージング環境での検証を強く推奨します。

### Suspended JobのリソースLimit変更（Beta昇格）

一時停止（suspend）状態のJobのPod templateにあるリソース要求・上限を変更できる機能がBetaに昇格します。

```yaml
apiVersion: batch/v1
kind: Job
spec:
  suspend: true  # Job停止中
  template:
    spec:
      containers:
      - name: data-processor
        image: processor:latest
        resources:
          requests:
            cpu: "8"
            memory: "16Gi"
          limits:
            cpu: "8"
            memory: "16Gi"
```

これまではリソース設定を変更するためにJobを削除して再作成する必要がありました。大規模なデータ処理JobをいったんSuspendし、リソース要件を調整してから再開できるようになり、バッチ処理ワークフローで実用的な改善になります。

## AI-Powered Kubernetes運用ツールの台頭

v1.36の機能強化と並行して、AIを活用したKubernetes運用ツールが急速に普及しています。

### 従来の課題

Kubernetesの運用で最も時間がかかるのは、問題の特定と根本原因分析です。`kubectl describe`、`kubectl logs`、メトリクスダッシュボードを往復しながら、複数のシグナルを手動で相関付けるプロセスには経験と時間が必要でした。単純でない障害では10〜30分の診断時間が一般的です。

### AIによるクラスター状態推論

新世代のツールは、クラスターの状態を複数のソースから収集し、LLMを使って相関分析を行います。

```
アラート: Pod payment-service OOMKilled（10分以内に3回）

AI分析結果:
- 過去24時間でメモリ使用量が2.1GB → 3.8GBに増加
- 相関シグナル: 同時刻にDBクエリのp99レイテンシが450ms → 2.1sに上昇
- 相関シグナル: 14:00のデプロイ後にアップストリームトラフィックが40%増加
- 仮説: 高い同時接続数でメモリリークの可能性（コネクションプール関連）
- 推奨調査: コネクションプール設定の確認、パターンXに一致するクエリの分析
```

従来のアラートは単一のメトリクス閾値に基づくものでしたが、AIツールは複数のシグナルを組み合わせて文脈のある診断を提供します。エンジニアは仮説の検証と実際の対処に集中できます。

### DockSecとDockerfile最適化

KubernetesとDockerのエコシステムでは、Trivy（CVE検出）、Hadolint（Dockerfile解析）、Docker Scout（依存関係スキャン）を統合し、AIレイヤーで相関分析を行うツールが普及しています。3つの別々の脆弱性レポートの代わりに、実際の設定で実質的なリスクの高い問題を優先順位付けして提供します。

## VMはなくならない

2020年頃に「KubernetesがVMを完全に置き換える」という予測がありましたが、2026年の現実は異なります。ベアメタルクラスターの大規模管理は複雑でコストが高く、VMはハイブリッドクラウド環境やレガシーワークロードとの統合で依然として重要な役割を担っています。

Kubernetes on VM on Cloud Providerという構成は依然として標準パターンです。インフラ予算の判断において、KubernetesとVMは競合関係ではなく補完関係として捉えるべきです。

## v1.36アップグレードの実務的な推奨事項

kubelet API authorization変更は、アップグレード前に準備が必要な変更です。ステージングクラスターで先に既存のCI/CDシステムと監視エージェントのkubelet APIアクセスパターンを把握した後、Webhookモードに切り替えてください。事前の棚卸しなしに切り替えると、予期せぬアクセス拒否が発生します。

一時停止Job機能はリスクが低く、アップグレード後にバッチワークロードで試してみることで、Job再作成なしにリソースを調整できる実質的なメリットを確認できます。

AI-powered運用ツールについては、既存のオブザーバビリティスタックを置き換えるのではなく、追加レイヤーとして試験導入することを推奨します。AIは「回答」ではなく「仮説」を提示するツールとして扱うことで、誤診断のリスクを抑えられます。
