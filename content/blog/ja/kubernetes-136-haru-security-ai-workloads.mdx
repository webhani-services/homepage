## Kubernetes v1.36「Haru」概要

Kubernetes v1.36「Haru（春）」は2026年最初のメジャーリリースとして公開されました。70のEnhancementを含み、セキュリティの強化とAI Workloadへの対応が重点領域です。

今回のリリースは「手動設定なしにデフォルトで安全な動作をする」という方向性をより強く打ち出しており、新規Clusterでは多くのセキュリティ設定が自動的に有効になります。

## GAになった主要機能

### User Namespaces

コンテナのUser NamespacesがGA（一般提供）になりました。これはホストOSとコンテナの間でUID/GIDを分離する機能で、コンテナ内でrootとして動作するProcessがホストOSでもrootになれないようにします。

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: isolated-runner
spec:
  hostUsers: false  # User Namespacesを有効化
  containers:
  - name: runner
    image: ci-runner:latest
    securityContext:
      runAsUser: 0  # コンテナ内ではroot (UID 0)...
      # ...しかしHostではUID 65536以降にマッピングされる
```

これによりコンテナブレイクアウト攻撃のリスクが大幅に低下します。CI Runner、Script実行環境、ユーザー提供コードを実行するWorkloadでの採用が特に推奨されます。`securityContext.runAsNonRoot`の単独使用と比べて格段に強い隔離を提供します。

### Mutating Admission Policies

CEL（Common Expression Language）ベースのMutating Admission PoliciesがGAになりました。Webhookを使わず、CEL表現式でAdmission時にResourceを変更できます。

```yaml
apiVersion: admissionregistration.k8s.io/v1
kind: MutatingAdmissionPolicy
metadata:
  name: default-resource-limits
spec:
  matchConstraints:
    resourceRules:
    - apiGroups: [""]
      apiVersions: ["v1"]
      operations: ["CREATE"]
      resources: ["pods"]
  mutations:
  - patchType: "ApplyConfiguration"
    applyConfiguration:
      expression: |
        Object{
          spec: Object.spec{
            containers: request.object.spec.containers.map(c,
              c.with({
                resources: c.resources.with({
                  limits: c.?resources.?limits.orValue({}).with({
                    "memory": c.?resources.?limits.?memory.orValue("256Mi"),
                    "cpu": c.?resources.?limits.?cpu.orValue("500m")
                  })
                })
              })
            )
          }
        }
```

WebhookベースのAdmission ControllerをCEL Policyに置き換えることで、Webhookのデプロイ・TLS管理の手間がなくなり、Admission Webhookのレイテンシ問題も解消されます。

### Fine-Grained Kubelet API Authorization

Kubelet APIがEndpoint単位の認可に対応しました。以前は`system:kubelet-api-admin`ロールが全操作に必要でしたが、v1.36からMonitoring Agent、Log Collector、CI Systemが必要な特定Endpointの権限のみ要求できるようになりました。

マルチテナントやマルチチームのClusterで最小権限の原則を実践しやすくなります。

## AI Workload対応の改善

### Dynamic Resource Allocation（DRA）の改善

GPU、FPGAなど特殊ハードウェアのSchedulingを担うDynamic Resource Allocation（DRA）のAPIが改善され、Production ML WorkloadでのReliabilityが向上しています。

```yaml
apiVersion: resource.k8s.io/v1alpha3
kind: ResourceClaim
metadata:
  name: training-gpu
spec:
  resourceClassName: nvidia-gpu
---
apiVersion: v1
kind: Pod
metadata:
  name: distributed-training
spec:
  resourceClaims:
  - name: gpu
    resourceClaimName: training-gpu
  containers:
  - name: trainer
    image: pytorch:2.4
    resources:
      claims:
      - name: gpu
```

### Topology-Aware Scheduling

分散学習JobでGPU間の通信帯域（NVLink、NVSwitch）が重要なWorkloadでは、Topology-Aware Schedulingが効果を発揮します。同一Training GroupのPodが同じInterconnect TopologyのNodeに配置されるため、学習Throughputに直接影響します。

## Upgradeの前に確認すること

**非推奨APIの使用状況を確認してください。** Kubernetesのдепrecationは予測可能なスケジュールに従いますが、Upgrade前に把握することが重要です:

```bash
# 非推奨API呼び出しの確認
kubectl get --raw /metrics | grep apiserver_requested_deprecated_apis

# hostUsers: true のPodを確認（User Namespaces影響対象）
kubectl get pods -A -o jsonpath='{range .items[*]}{.metadata.namespace}/{.metadata.name}: hostUsers={.spec.hostUsers}{"\n"}{end}'
```

**セキュリティDefaultの変更をStagingでテストしてください。** v1.36では複数のセキュリティRelatedなDefaultが強化されています。暗黙的に緩和された設定に依存していたWorkloadが異なる動作をする可能性があります。

**Upgradeパス。** Kubernetesはn-1 Minor Versionからのupgradeをサポートしています。v1.34またはv1.35からであれば`kubeadm`で標準パスを取れます:

```bash
# 現在のバージョン確認
kubectl version --short

# Node upgradeの前にdrainする
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
```

## まとめ

v1.36は手動設定を減らしながらデフォルトで安全な動作を実現する方向で進化しています。実務上注目すべき3点:

1. **User Namespaces GA** — CI RunnerやUntrusted実行環境に今すぐ採用を検討
2. **CEL Admission Policies** — Webhookベースのコントローラーを段階的に移行。運営コスト削減効果が蓄積される
3. **DRA改善** — KubernetesでML Training Jobを運用しているチームは現行のGPU Scheduling対比でDRAをテスト

新規ClusterはすぐにV1.36を採用することを推奨します。既存ClusterはDeprecation確認後、Stagingで検証してから本番に適用してください。
