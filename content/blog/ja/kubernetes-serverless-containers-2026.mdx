## 「全部Kubernetes」の時代は終わりつつある

数年前まで、コンテナで本番環境を組むなら第一選択はKubernetesでした。しかし2026年の状況を見ると、その前提が少しずつ変わってきています。複数の業界レポートによると、Kubernetesの本番ワークロードにおける採用は約68%で頭打ちとなり、2024年のピークからわずかに下がったとされています。一方でチームは、ワークロードを従来型のクラスタとserverless containerプラットフォームに分散させる傾向を強めています（[The New Stack](https://thenewstack.io/aws-kubernetes-invisible-simplicity/)）。

ここで言うserverless containerとは、AWS Fargate、Google Cloud Run、Azure Container Appsといった、クラスタの管理を意識せずコンテナを動かせるサービス群です。さらにNomadやAWS ECSといった、より軽量な選択肢も合わせて22%程度まで成長しているという指摘もあります。

この記事では、「Kubernetesかserverless containerか」を二者択一で語るのではなく、どういう基準で使い分けるべきかを実務の視点から整理します。

## なぜ「頭打ち」なのか

Kubernetesの採用が頭打ちになった理由は、Kubernetesが劣化したからではありません。むしろ逆で、多くのチームが「自分たちの規模にはKubernetesはオーバースペックだった」と気づき始めたことが大きいと考えています。

Kubernetesは強力ですが、その強力さには運用コストが伴います。クラスタのアップグレード、ノードの管理、ネットワークポリシーの設計、権限の設定——これらを安全に回し続けるには、専任に近い体制が必要です。小〜中規模のチームにとって、この運用負荷は無視できません。

AWSもこの課題を認識しており、KubeConでは、KarpenterやKro、Cedarといったツールで「Kubernetesの複雑さを見えなくする」取り組みが紹介されました（[The New Stack](https://thenewstack.io/aws-kubernetes-invisible-simplicity/)）。つまり、Kubernetesそのものを使いやすくする流れと、そもそもKubernetesを直接触らない選択肢を増やす流れが、同時に進んでいるわけです。

## 使い分けの判断基準

私たちが顧客のインフラ設計を支援する際、次の観点で選択肢を整理しています。

### Serverless Containerが向くケース

- **チームの運用リソースが限られている**: クラスタ運用に人を割けないなら、管理を肩代わりしてくれるサービスが合理的
- **トラフィックの変動が大きい**: リクエストに応じて自動でスケールし、アイドル時にコストを抑えられる
- **ワークロードが比較的シンプル**: 単純なWeb APIやバッチ処理など、複雑なオーケストレーションを必要としない

```yaml
# Fargateタスク定義の一例（概念的な抜粋）
# ノード管理が不要で、必要なCPU/メモリだけ宣言する
family: web-api
cpu: "512"
memory: "1024"
containerDefinitions:
  - name: api
    image: myapp/api:latest
    portMappings:
      - containerPort: 8080
```

### Kubernetesが向くケース

- **多数のサービスを協調させる**: マイクロサービスが数十以上あり、複雑な通信制御が必要
- **細かい制御が必要**: スケジューリング、ネットワーク、ストレージを細かく制御したい
- **マルチクラウド・オンプレを跨ぐ**: 環境を問わない共通基盤としての価値が効く
- **すでに運用ノウハウがある**: チームにKubernetesの知見が蓄積されている

## ハイブリッドという現実解

重要なのは、これが「どちらか一方」の話ではないという点です。実際には、多くのチームが両方を併用しています。定常的に動く中核サービスはKubernetesで運用しつつ、変動の大きい周辺ワークロードやバッチ処理はserverless containerに逃がす、といった構成が現実的です。

私たちが設計を支援する際も、「まず全部Kubernetesで組む」という発想からは離れることを勧めています。ワークロードごとに特性を見て、運用負荷と制御の必要性のバランスで配置を決めるほうが、長期的なコストと安定性の両面で有利です。

## 移行を検討する際の注意点

すでにKubernetesで運用している場合、serverless containerへの部分移行を検討することもあるでしょう。その際に気をつけたい点を挙げておきます。

まず、**ロックインの度合い**です。Cloud Runやfargateはそれぞれのクラウドに固有の部分があり、移植性はKubernetesより下がります。マルチクラウドを重視するなら、この点は無視できません。

次に、**コスト構造の違い**です。serverless containerはアイドル時のコストを抑えやすい一方、常時高負荷なワークロードでは、専有リソースを確保したほうが安くなる場合があります。実際のトラフィックパターンに基づいて試算することが欠かせません。

最後に、**観測性の設計**です。プラットフォームを跨ぐと、ログやメトリクスの集約方法を統一しておかないと、障害時の調査が難しくなります。移行前に監視基盤の方針を決めておくことをお勧めします。

## まとめ

Kubernetesの採用が頭打ちになったのは、Kubernetesが不要になったからではなく、「全部Kubernetes」という発想が見直されているからです。serverless containerという選択肢が成熟し、ワークロードの特性に応じて使い分けるのが当たり前になりつつあります。

判断の軸はシンプルです。運用リソースが限られ、ワークロードがシンプルならserverless containerを、多数のサービスの協調や細かい制御が必要ならKubernetesを。そして多くの場合、答えは「両方をうまく組み合わせる」ことになります。自分たちの規模と体制に正直になって選ぶことが、長続きするインフラへの近道です。
