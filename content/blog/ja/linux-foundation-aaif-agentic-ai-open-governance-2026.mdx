Linux Foundationは2026年4月、**Agentic AI Foundation（AAIF）**の設立を発表しました。初期のプロジェクト寄贈として、AnthropicのModel Context Protocol（MCP）、BlockのgooseエージェントフレームワーK、そしてAGENTS.md仕様が含まれています。AI Agentを活用したアプリケーションを構築しているチームにとって、このガバナンスの変化は見た目以上に重要な意味を持ちます。

## AAIFとは何か

Agentic AI Foundationは、AI Agentが依存するツールと標準に対して、中立的でオープンなガバナンスを提供するためにLinux Foundationが立ち上げた新組織です。Cloud Nativeの世界でCNCFが果たした役割と同様に、AI Agentエコシステムにおける標準化の基盤になることが期待されています。「個別企業が管理するツール」から「コミュニティが維持するオープン標準」への移行点と言えるでしょう。

寄贈される主なプロジェクト：

- **Model Context Protocol（MCP）** — AnthropicがリリースしたAI ModelをツールやデータソースやExternal Serviceに接続するための標準。すでに2,300以上のPublic Serverが存在し、Claude、Cursor、VS Code、その他200以上のClientがネイティブサポートしています。
- **goose** — BlockのOpen Source AI Agentフレームワーク。Localでコーディングやエンジニアリングタスクを自律実行するよう設計されています。
- **AGENTS.md** — AI AgentのBehavior、制約、Capabilityを機械可読な形式で記述するSpecification。

## オープンガバナンスが変えること

MCPがAnthropic単独の管理下にあった頃、Enterprise採用には正当な懸念がありました。単一ベンダーが自社製品の利益に沿った方向にSpecを変更できるという問題です。Linux FoundationへのオープンガバナンスはこのRiskを直接解消します。

CNCFの例が参考になります。2016年にKubernetesがCNCFへ移管する前、Enterprise チームはGoogleが実質的に管理するProjectに本番Infrastructureを構築することに消極的でした。中立ガバナンスへの移行が本格投資と標準化を解放し、現在のCloud Nativeエコシステムへとつながりました。

AAIFはAgentic AIツールに同様のTrajectoryをもたらします。MCPの上に構築するチームは、AnthropicのロードマップだけでなくコミュニティのInputによって進化するSpecの上に立つことになります。

## MCP実装への具体的な影響

既にMCPを利用しているチームへの近期的な実務影響は限定的です。Protocolが初日から変わるわけではありません。ただし長期的な含意は大きいです：

**Multi-vendorの安定性** — 主要なCloud ProviderやToolingベンダーは、自らが参加できるガバナンスのSpecに対してより深く投資します。MCPのIntegrationはさらに急速に広がるでしょう。

**RFC経由のSpec進化** — セキュリティFixや認証の改善、新機能は、AnthropicによるUnilateralな決定ではなくPublic RFCプロセスを経て実装されます。

**Enterprise採用の加速** — 大企業のLegalや調達チームはオープンガバナンスの標準を好みます。規制産業でのMCP採用における一般的な障壁が取り除かれます。

## 最も注目すべきAGENTS.md

MCPよりも議論が少ないAGENTS.mdですが、AI Agentの実際の動作を定義するという意味で同様に重要です。AgentのAllowed Actions、Tool Access、スコープ制約、Behavior ガイドラインを構造化して記述する方法を提供します：

```markdown
# AGENTS.md

## Identity
name: webhani-deploy-agent
version: 1.0.0
description: webhaniの本番環境のDeployment自動化を担うAgent

## Capabilities
tools:
  - name: read_file
    scope: ./src/**
  - name: run_command
    allowed: ["npm run build", "npm run test"]
    denied: ["rm -rf", "git push --force"]

## Constraints
- ./srcと./config以外のFileを変更しないこと
- 本番環境へのDeployには必ずHuman承認を必要とすること
- すべてのTool Invocationをaudit logに記録すること

## Escalation
on_uncertainty: request_human_review
on_error: halt_and_notify
```

この種の構造化Specificationにより、OrchestrationフレームワークはPrompt指示に頼らず制約をプログラム的に適用できます。LLMが逆張り条件下でPromptを無視または誤解するリスクを大幅に低減できます。

## 今すぐAI Agentスタックを見直す

AAIFの設立は、現在または計画中のAI Agent Infrastructureを見直す良い機会です：

1. **Tool ConnectivityにMCPを使っているか？** カスタムIntegration Layerを構築している場合、長期ガバナンスが明確になった今、MCP移行を検討する価値があります。

2. **制約はCodeで定義されているか？** Agent BehaviorへのAd-hocなPrompt指示は脆弱です。AGENTS.md方式のSpecificationにより、監査可能で適用可能な定義が持てます。

3. **Governance上のリスクは？** 規制産業に属する場合、Linux Foundation由来のSpecは内部採用の議論で実践的な根拠になります。

4. **どのフレームワークがAAIF準拠か？** Claude Code、Cursor、Windsurf、VS CodeはすでにMCPをネイティブサポートしています。AAIFのMembership拡大に伴い、goose周辺のエコシステムも成長するでしょう。

## まとめ

Agentic AI FoundationはMCPが今日行うことを変えるのではなく、それが何になるかを誰がコントロールするかを変えます。本番AI Agentシステムを構築するチームにとって、オープンガバナンスは長期的なVendor Lockリスクを低減し、このStack層が本格投資に値する成熟段階に達したことを示します。AGENTS.mdをExistingドキュメントと並ぶFirst-class Artifactとして扱い始める時期が来ています。
