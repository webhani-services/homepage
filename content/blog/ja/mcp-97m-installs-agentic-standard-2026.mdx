2026年3月25日、AnthropicのModel Context Protocol（MCP）が9700万インストールを突破しました。AIのInfrastructure標準としては、これは異例のスピードです。Kubernetesが同等の普及密度に達するまで約4年かかったのに対し、MCPはわずか16ヶ月でこの数字を達成しました。

この数字が示すのは単なる人気ではありません。MCPがどのようにしてこの普及を遂げたかを見ることで、AI Agentのアーキテクチャがどこに向かっているのかが見えてきます。

## MCPが解決する問題

MCPは、AI ModelとExternalのToolやData Source、Serviceとの接続方法を標準化するOpen Protocolです。MCP以前は、AI ApplicationはそれぞれCustomのIntegration Layerを実装する必要がありました。Language ModelをDatabase、API、ファイルシステム、検索Indexに接続するための独自Codeを書く必要があったのです。

この問題は組み合わせ爆発を引き起こします。N個のModelとM個のToolがある場合、N×M個のCustom Integrationが必要になります。MCPはこれをN+Mに削減します。各ModelはMCP Client Interfaceを一度実装し、各ToolはMCP Server Interfaceを一度実装するだけで、相互に連携できます。

```typescript
// MCPサーバーの最小実装例（TypeScript）
import { McpServer } from "@modelcontextprotocol/sdk/server/mcp.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { z } from "zod";

const server = new McpServer({ name: "db-tools", version: "1.0.0" });

server.tool(
  "query_users",
  { sql: z.string().describe("ユーザーテーブルへのSELECTクエリ") },
  async ({ sql }) => {
    const results = await db.query(sql);
    return {
      content: [{ type: "text", text: JSON.stringify(results, null, 2) }],
    };
  }
);

const transport = new StdioServerTransport();
await server.connect(transport);
```

MCP Client側（通常はAI Agent Runtime）も同様にシンプルです。Modelは利用可能なToolのListを受け取り、構造化されたJSONでToolを呼び出し、ResponseをProcessします。Transport Layer（stdio、HTTP、WebSocket）は抽象化されています。

## 急速な普及を支えた3つの要因

**主要なAI ProviderがすべてMCP Supportを標準搭載しました。** 2026年3月中旬までに、OpenAI、Google DeepMind、Cohere、MistralがすべてのAgent FrameworkにMCP Integrationを組み込みました。これはOptionalではなく、標準的なInterfaceになったのです。

**ガバナンスが中立的な組織に移管されました。** 2025年12月、AnthropicはMCPをLinux Foundation傘下のAgentic AI Foundation（AAIF）に寄贈しました。Anthropic、Block、OpenAIが共同設立したこの組織に管理が移ることで、特定企業に依存しない標準として開発者の信頼を得ました。

**Server Ecosystemが予想を超えた速度で成長しました。** 現在、Database、Cloud Provider、CRM、Developer Tool、Analytics Platformをカバーする5,800以上のCommunityおよびEnterprise MCP Serverが存在します。既存のPostgreSQLやSalesforceに数行のConfigでAI AgentをConnectできるようになれば、普及が加速するのは自然な流れです。

## 実践的な影響

社内AI Toolingを構築しているチームにとって、MCPの標準化は以前は曖昧だった判断を明確にします。「独自のTool Calling抽象化を構築すべきか、業界標準の登場を待つべきか」という問いへの答えは出ています。標準はMCPです。

実践的な意味合いとしては、内部SystemにAccessするAI Agentを構築する場合、各Systemに対してMCP Serverを実装することを推奨します。すべてのMCP対応ModelとすぐにCompatibilityが得られ、将来のAI Projectで再利用できるAssetが構築できます。

```python
# Python MCP Serverの実装例
from mcp.server import Server
from mcp.server.stdio import stdio_server
from mcp import types

app = Server("analytics-tools")

@app.list_tools()
async def list_tools() -> list[types.Tool]:
    return [
        types.Tool(
            name="get_monthly_revenue",
            description="指定した年の月別売上データを返します",
            inputSchema={
                "type": "object",
                "properties": {
                    "year": {"type": "integer", "description": "対象年"}
                },
                "required": ["year"],
            },
        )
    ]

@app.call_tool()
async def call_tool(name: str, arguments: dict) -> list[types.TextContent]:
    if name == "get_monthly_revenue":
        data = await fetch_revenue(arguments["year"])
        return [types.TextContent(type="text", text=str(data))]

async def main():
    async with stdio_server() as streams:
        await app.run(*streams, app.create_initialization_options())
```

## まだ解決されていない課題

MCPはIntegration Interfaceの問題を解決しますが、すべてを解決するわけではありません。MCP ServerとBackend System間のAuthenticationは依然としてServer実装側で処理されます。MCP仕様にはAuth Layerが組み込まれていないため、本番環境で複数のMCP Serverを運用するチームはSecretsの管理とAccess Controlを慎重に設計する必要があります。

ObservabilityもまだAttentionが必要な領域です。AI Agentが一つのSessionで数十回のTool Callを行う場合、予期しない動作のDebuggingにはMCP RequestとResponseの完全なTraceが必要です。MCP RuntimeとのOpenTelemetry Integrationは改善されつつありますが、まだ標準化されていません。

## まとめ

MCPは、もはや「試してみる価値がある」という段階を超え、標準的なInfrastructureとして定着しました。9700万インストールという数字は、開発者の好奇心だけでなく、組織がMCPをAI Integration Layerとして本番環境にDeployしていることを示しています。

社内SystemにAI Capabilityを組み込む方法を評価しているチームにとって、今MCPから始めることは、業界全体がSupportし、維持し、発展させるFoundationの上に構築することを意味します。初期のAI Agent開発を悩ませていたIntegrationの組み合わせ爆発問題は、概ね解決されています。残る課題は、良いToolを構築し、うまくConnectすることです。
