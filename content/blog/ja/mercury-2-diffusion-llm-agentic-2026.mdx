本番で動かしている言語モデルの多くは、構造的に同じ特徴を持っています。左から右へ、トークンを1つずつ書いていく方式です。各トークンは直前のトークンが確定するまで待ちます。この順序性が、私たちが頼りにしている自然な文章生成の源であると同時に、戦い続けているレイテンシの源でもあります。Inception LabsのMercury 2は別のアプローチを取ります。次のトークンを順番に予測するのではなく、文章全体をまず下書きし、それを少数のデノイジングステップで磨き上げていく。画像生成の拡散モデルが使っている反復的なクリーンアップの発想を、テキストに応用したものです。公表されているスループットは毎秒1,000トークン超で、速度最適化された自己回帰システムの中でも最速級と比べておよそ5倍。推論の質は保ったままとされています。

この数字は確かに興味深いのですが、スループットそのものが本題ではありません。本題は、それがシステム設計に何をもたらすかです。以下では、webhaniがこの種のモデルをクライアントのアーキテクチャ向けに実際どう評価するか、エージェントの内側のループに何を任せるか、そしてどこでフロンティアモデルにエスカレーションするかを整理します。

## 並列リファインメントと次トークン予測の違い

標準的なTransformerデコーダは構造上シーケンシャルです。N番目のトークンを生成するには、1からN-1までが確定している必要があります。リクエストをバッチ化することも、speculative decodingを使うことも、量子化することもできますが、1つのレスポンス内部の依存チェーンそのものからは逃れられません。1回の生成にかかる実時間は、その長さと本質的に結びついています。

拡散テキストモデル、いわゆるdLLMはこのチェーンを断ち切ります。出力全体のノイズだらけ、あるいはマスクされた下書きから始め、固定回数のリファインメントステップで全ポジションをほぼ並列に鮮明化していく。コストを支配するのはトークン数ではなくステップ数です。長い出力がはるかに速く返ってくるのはこのためです。Mercury 2は、現代的なモデルに期待される実用的なインターフェースをそのまま備えています。tool calling、JSON schemaで制約した構造化出力、そしてマルチモーダル拡張です。Inception Labsの公表価格によれば、入力が100万トークンあたり約$0.25、出力が約$0.75で、執筆時点では early access 経由で利用できます。

トレードオフは正直に見ておくべきです。自己回帰モデルのエコシステムは成熟しており、何年もかけて積み上げたツール群、プロンプトのパターン、ファインチューニングのレシピがその周りに揃っています。拡散方式のテキスト生成は、シーケンシャルなデコードが持つ決定性や逐次的な制御性をある程度手放す代わりに、並列性を得ています。そして最難関の推論タスクでは、フロンティアの自己回帰モデルがいまだ優位です。これらはdLLMを失格にする理由ではありません。どこに向けて使うべきかを教えてくれているだけです。

## レイテンシは見栄えの指標ではなく設計上の関心事

チャットボットの1ターンであれば、数百ミリ秒の追加レイテンシはUXの細部にすぎません。ところがエージェントシステムでは積み上がります。典型的なエージェントのループは、計画し、ツールを呼び、結果を観測し、また繰り返す、という流れで、その各周回には少なくとも1回のモデル生成が含まれます。エンドツーエンドのレイテンシは全ステップの合計です。

```
total_latency ≈ Σ (ステップiの生成トークン数 / スループット) + ツール・ネットワークのオーバーヘッド
```

エージェントが1つのタスクを終えるのに8回や12回のステップを踏むとき、ステップあたりの生成速度はステップ数だけ掛け算されます。ここでこそ、5倍のスループット改善はベンチマーク上の一行ではなく、2秒の応答と10秒の応答を分ける差になります。同じ論理がリアルタイム音声などのインタラクティブUXにも当てはまります。ユーザー入力から音声出力までの予算は容赦がないからです。

ですから、どんな高速モデルに手を伸ばす前にも、自分のワークロードが本当にレイテンシ律速なのかを測ってください。簡単な概算です。各エージェントステップが約400トークンを生成し、タスク平均が10ステップだとします。毎秒200トークンならおよそ20秒が純粋な生成時間です。毎秒1,000トークンなら約4秒。ツール呼び出しとネットワークのオーバーヘッドがどちらの場合も合計3秒を足すなら、23秒から7秒になります。逆に、各ステップが30トークンしか生成しない短いものなら、生成部分はすでに小さく、高速化は実時間をほとんど動かしません。何かを変える前に、自分の実トラフィックでこの計算をしてください。

## 2層ルーター — ループには高速モデル、難所にはフロンティアモデル

私たちが最もよく勧めるのは「モデルを置き換える」ではなく「モデルを階層化する」というパターンです。ループ内部の高頻度で相対的に低リスクな仕事、たとえばルーティング、分類、下書き生成、ツール引数の組み立てには高速なdLLMを使う。本当に深さが要るステップ、たとえば最終的な統合や難しい複数制約の判断には、フロンティアの自己回帰モデルを取っておく。ルーターがターンごとにどちらの層を使うかを決めます。

以下はプロバイダ非依存のスケッチです。`generate` インターフェースは意図的に汎用にしてあり、実際に使うSDKへマッピングできます。

```typescript
type Role = "system" | "user" | "assistant" | "tool";

interface Message {
  role: Role;
  content: string;
}

interface ToolSpec {
  name: string;
  description: string;
  parameters: Record<string, unknown>;
}

interface GenerateRequest {
  model: string;
  messages: Message[];
  tools?: ToolSpec[];
}

interface GenerateResult {
  text: string;
  usage: { inputTokens: number; outputTokens: number };
}

// プロバイダ非依存のクライアントが generate(req) を提供すると仮定する。
declare function generate(req: GenerateRequest): Promise<GenerateResult>;

const FAST_MODEL = "mercury-2";        // 拡散方式、スループット最適化
const FRONTIER_MODEL = "frontier-ar";  // 自己回帰、推論の深さに最適化

interface Turn {
  messages: Message[];
  tools?: ToolSpec[];
  // オーケストレータがこのターンについて既に持っているシグナル。
  isFinalAnswer: boolean;
  estimatedComplexity: number; // 0..1、軽量なヒューリスティックか分類器から
}

function pickModel(turn: Turn): string {
  // 最終統合と明らかに難しいターンはフロンティアモデルへ。
  if (turn.isFinalAnswer || turn.estimatedComplexity > 0.7) {
    return FRONTIER_MODEL;
  }
  // それ以外（ルーティング、分類、下書き）は高速パスを通す。
  return FAST_MODEL;
}

async function runTurn(turn: Turn): Promise<GenerateResult> {
  const model = pickModel(turn);
  const result = await generate({
    model,
    messages: turn.messages,
    tools: turn.tools,
  });

  // エスカレーション: 高速モデルの下書きが軽量な品質チェックを
  // 通らなければ、諦める前に一度だけフロンティアモデルで再試行する。
  if (model === FAST_MODEL && !passesQualityGate(result.text)) {
    return generate({
      model: FRONTIER_MODEL,
      messages: turn.messages,
      tools: turn.tools,
    });
  }

  return result;
}

function passesQualityGate(text: string): boolean {
  // ドメイン固有: schema検証、必須フィールドの有無、
  // 明らかな拒否や途中切れがないこと。軽く決定的に保つ。
  return text.trim().length > 0;
}
```

肝心なのは `pickModel` の形です。たいていの本番エージェントには、明らかに軽いターンがいくつかと、本当に難しいターンが少しあります。単純なシグナルでルーティングするだけで、利得の大部分を取り込めます。`runTurn` のエスカレーション・フォールバックは安全弁です。高速モデルが決定的なゲートを通らないものを出したら、悪い結果をそのまま流すのではなく、フロンティアモデルに1回だけ追加コストを払う。これで一般的なケースは速いまま、まれな難所は正しく保てます。

`estimatedComplexity` は高度である必要はありません。入力長、特定キーワードの有無、現在の推論チェーンの深さ、あるいは小さな専用分類器、どれでも機能します。粗いヒューリスティックから始め、エスカレーション率を測り、そこから調整していきましょう。

## まだdLLMを使わない場面

速さには文脈がついて回ります。タスクがエッジケースの正しさが支配する短い単発のリクエストで、レイテンシが既に許容範囲なら、階層化は不要な複雑さを足すだけです。特定の自己回帰プロバイダに密結合したツール群、評価ハーネス、ファインチューニングのパイプラインに依存しているなら、移行コストがスループットの利得を上回るかもしれません。そして最難関のフロンティア推論、誤答のコストが高く分解しにくい類の問題には、私たちはいまだ最上位の自己回帰モデルをデフォルトにします。階層化の要点は、まさに全体で1つを選ばなくてよいという点にあります。選ぶのはターンごとです。

## まとめ

- Mercury 2のような拡散LLMは、自己回帰モデルが逃れられない唯一のコスト、すなわち1レスポンス内部の逐次依存を攻めます。効果が見えるのはマルチステップのエージェントとインタラクティブUXで、レイテンシが多数の生成の合計になる領域です。
- 移行する前に測る。ステップあたりトークン数とタスクあたりステップ数を掛けてください。その積が小さければ、速いモデルでも実時間はほとんど変わりません。
- 置き換えではなく階層化する。軽く高頻度なターンは高速なdLLMへ、難しいターンや最終ターンはフロンティアモデルへルーティングし、決定的な品質ゲートをフォールバックに置く。
- トレードオフは正直に。エコシステムの成熟度、制御性、最上位の推論は、最難関の仕事ではいまだフロンティアの自己回帰モデルに分があります。dLLMが居場所を得るのはループの中であって、必ずしも頂点ではありません。
