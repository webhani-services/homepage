Next.js 16.2 が 2026 年 3 月 18 日に正式リリースされました。今回のリリースで最も注目すべき点は、`next dev` の起動速度が従来比で約 400% 向上したことです。大規模なプロジェクトで毎朝感じていた「開発サーバーが上がりきるまでの待ち時間」が、実質的に体感できないレベルまで短縮されています。

## 主な変更点

### 1. dev サーバーの起動速度改善

今回最大のインパクトを持つ変更です。内部的には Turbopack のコード変換パイプラインが刷新され、初回コンパイルのモジュール解析処理が並列化されました。プロジェクト規模にもよりますが、500 以上のモジュールを持つアプリケーションでも数秒以内の起動が現実的になっています。

また、レンダリング速度も約 50% 向上しており、Server Components の初回 HTML 生成が体感的にも速くなっています。

### 2. Agent-ready な `create-next-app`

`create-next-app` に AI Agent 向けの設定オプションが追加されました。具体的には、エージェントが自律的にプロジェクトの状態を把握しやすくするための構造的なメタデータファイルが生成されます。Claude Code や GitHub Copilot Workspace といったツールがプロジェクトのコンテキストを迅速に読み込めるようになります。

```bash
npx create-next-app@latest my-app --agent-ready
```

このオプションを付けると、`agent.json`（プロジェクト構造の要約）と `.agent/context.md`（主要なアーキテクチャ上の決定事項）が生成されます。

### 3. ブラウザログの転送機能

開発中、ブラウザのコンソールに出力されたログをターミナルに転送できるようになりました。これにより、AI Coding Assistant がクライアントサイドのエラーや警告をリアルタイムで参照しながらデバッグを支援できます。

```typescript
// next.config.ts
const nextConfig = {
  experimental: {
    browserLogForwarding: true,
  },
};
```

### 4. 実験的 Agent DevTools

`--experimental-agent-devtools` フラグを付けて dev サーバーを起動すると、ローカルで稼働している AI Agent（Claude Code 等）が Next.js の内部状態（ルーティングツリー、Server/Client Component の境界、キャッシュの状態）をリアルタイムで問い合わせできる専用の IPC エンドポイントが有効化されます。

```bash
next dev --experimental-agent-devtools
```

この機能はまだ実験段階ですが、AI を使った開発ループを短縮するポテンシャルを持っています。

## Turbopack の改善

### Server Fast Refresh

従来の Hot Module Replacement (HMR) はクライアントサイドに限定されていましたが、Server Fast Refresh ではサーバーサイドのモジュール変更も細粒度でリロードできます。Server Component を編集した際にページ全体のリロードが走っていた問題が解消されます。

### Web Worker Origin と WASM 対応強化

Workers 内で WASM ライブラリを使う場合に問題になっていたオリジン制約が緩和され、`crypto-wasm` や `@tensorflow/tfjs-backend-wasm` などのライブラリを Worker 内で扱いやすくなっています。

### Subresource Integrity（SRI）サポート

JavaScript ファイルに SRI ハッシュが自動付与されるようになりました。CDN 経由でのスクリプト改ざんリスクを軽減できます。

```html
<!-- 自動生成される形式 -->
<script
  src="/_next/static/chunks/main.js"
  integrity="sha256-abc123..."
  crossorigin="anonymous"
></script>
```

### Dynamic Import のツリーシェイキング

`import()` で動的にロードされるモジュールから未使用のエクスポートが除去されるようになりました。分割されたチャンクのサイズが削減され、特に多数の動的インポートを持つ大規模アプリでバンドルサイズの改善が見込めます。

### 再設計された 500 エラーページ

ビルトインの 500 エラーページが刷新され、スタックトレースやエラーコンテキストが見やすく整理されました。プロダクション環境ではユーザー向けのシンプルなページ、開発環境では詳細なデバッグ情報が表示されます。

## 移行時の注意点

16.1 から 16.2 への移行は基本的に破壊的変更がありません。ただし、以下の点に注意してください。

- `experimental.turbopack` の設定は 16.0 で既に `turbopack`（トップレベル）に移動済みです。16.2 以降はレガシーの `experimental.turbopack` は無視されるようになりました
- Server Fast Refresh はデフォルト有効です。既存のカスタム HMR 設定と競合する場合は `turbopack.hotUpdate` で制御できます

```typescript
// next.config.ts
const nextConfig = {
  turbopack: {
    hotUpdate: {
      serverComponents: false, // Server Fast Refresh を無効にする場合
    },
  },
};
```

## webhani の見解

今回のリリースで最も実務的な価値があるのは dev 起動速度の改善です。ページ数が 100 を超えるプロジェクトでは、毎日の累積待ち時間は無視できません。Agent DevTools はまだ実験段階ですが、AI Coding Assistant を活用した開発ワークフローを本格的に整備している場合は今から試す価値があります。

Turbopack の SRI サポートは、セキュリティ要件が厳しいエンタープライズ案件では特に歓迎される機能です。CDN 経由で Next.js アプリを配信している場合は、この機能を有効化することを推奨します。

次回の 16.3 では Agent DevTools の安定化と、キャッシュ層のさらなる最適化が予定されています。プロジェクトのアップグレードは早めに済ませておくと良いでしょう。
