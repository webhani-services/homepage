Next.js 16.2が2026年3月18日にリリースされました。TurbopackがDefault Bundlerになってから2回目のReleaseで、主要な数字は注目に値します：16.1比87%のStartup高速化、Compile時間400-900%改善、Server Component Refresh 67-100%改善です。何が実際に新しくなったのか、日々の開発にどう影響するかを整理します。

## パフォーマンス数値の文脈

これらの改善はNext.js 16.1比であり、webpackやTurbopack導入前との比較ではありません。TurbopackはNext.js 16.0でStable Defaultになっているため、16.2はすでに高速な基盤の上に構築されています。

**Startup time**（16.1比87%高速化）は`next dev`実行からReadyプロンプトが表示されるまでの時間です。Default Applicationで計測すると、数秒かかっていたものがほぼ即座になります。

**Compile time**（400-900%改善）はRoute treeの初回Compilationにかかる時間です。幅があるのはプロジェクトサイズとModule graphの複雑さによるためです。Dynamic importが多い大規模プロジェクトほど効果が大きい傾向があります。

**Server refresh**（67-100%改善）はServer ComponentファイルをSaveしてからBrowserに反映されるまでの時間です。16.2で追加されたServer Fast Refreshにより、変更されたServer subtreeのみがRe-renderされ、Page全体のServer-side RecompilationがTriggerされなくなりました。

## Server Fast Refresh

Client-side Fast RefreshはNext.jsに数年前から存在します。Server Fast RefreshはこれをServer Componentsに拡張したものです。Server Fileを保存すると、Page全体ではなく影響を受けたComponent subtreeのみがInvalidateされ、再取得されます。

実際の影響：Data fetchingやServer-side Logicの変更を伴う、深くNestされたServer Componentの開発が、Client Componentの編集と同様のスピードで行えるようになります。

## Web Worker Origin（WASMサポート改善）

Web Worker Originは、Web Worker内で動作するWASMライブラリのサポートを向上させる新しいTurbopack機能です。以前は、Worker内で使用されるWASMファイルが、特定のBrowser環境でのCross-origin制限により、正しくLoadされない場合がありました。このFixにより、Worker + WASMの組み合わせが手動のWorkaroundなしに安定して動作します。

WASMをWorker内で使用していないプロジェクトには影響しません。使用している場合、このFixはデバッグが困難なLoadingエラーの一類を解消する可能性があります。

## Dynamic ImportのTree Shaking

```typescript
// Before: 使用しているものだけでも全ExportがBundleされていた
const { HeavyComponent } = await import('./heavy-module')

// After: 未使用ExportはChunkから除外される
const { LightComponent } = await import('./shared-module')
```

この変更により、動的にImportされるModuleから未使用のExportが除外されます。共有依存関係が大きいRoute Segmentでは、Chunk sizeの削減に効果があります。多くのItemをExportするModuleから一部だけを使用するパターンで最も効果が大きくなります。

## TypeScript PostCSS Config

小さいですが実用的な改善：`postcss.config.ts`が`postcss.config.js`と同様にサポートされました。TypeScript-firstのコードベースを運用している場合、Plain JavaScriptで残す必要があったConfigファイルが一つ減ります。

## React Server Components Payloadのパフォーマンス

Next.jsチームからReact本体へのContribution：Server Component PayloadのDeserializationが最大350%高速化しました。これはRSCを使用するすべてのFrameworkに恩恵をもたらすReactレベルの変更です。データが多いDashboardやコンテンツが豊富なRouteで、Time-to-interactiveの改善が体感できます。

## アップグレードについて

16.2へのアップグレードはほとんどのプロジェクトでスムーズです。200以上のBug fixがChangelogに含まれています。アップグレード前に確認しておくこと：

1. カスタムWebpack ConfigurationのReview — Escape hatchの一部がTurbopackで異なる動作をする場合があります
2. WASM WorkerのTest — Worker Origin fixがLoadingの挙動を変更します
3. Tree Shakingの正確性確認 — Dynamic importのCode pathがModule graphで期待通りに動作しているか確認

Breaking changesは`nextjs.org/docs/app/guides/upgrading/version-16`で確認できます。

## まとめ

16.2は良い意味でのMaintenance-and-performanceリリースです：Core Feature setが安定し、そのStabilityを活かしてCompile speedを大幅に向上させています。Server Componentの頻繁なIterationを行うチームにとって、Server Fast Refreshだけでもアップグレードする価値があります。
