Next.js 16.2 のリリースノートには、`next dev` 起動速度の約 400% 向上と React Server Components Payload デシリアライズの約 350% 高速化という2つの数字が並んでいます。起動速度の改善は主に Turbopack の並列化によるものですが、RSC デシリアライズの改善はより興味深い経緯があります。JSON パースの特定のアルゴリズムを変更したことが直接の原因です。

## RSC Payload とデシリアライズのコスト

React Server Component がレンダリングされると、サーバーはその出力を JSON ライクな Wire Format に Serialize し、クライアントがそれを Deserialize してコンポーネントツリーを再構築します。この処理はページナビゲーション、Server Action のレスポンス、Streaming の更新ごとに実行されます。コンポーネントツリーが深いアプリケーションや、データフェッチが多いページでは、このコストが積み重なります。

デシリアライズはクライアント側、つまりブラウザの JavaScript エンジン上で実行されます。16.1 以前の実装では、この処理が測定可能なほど遅かったことが今回の改善のきっかけです。

## Reviver コールバック付き JSON.parse の問題

旧実装では `JSON.parse` に Reviver コールバックを渡していました。

```javascript
// 旧実装の概念的な表現
function deserializeRSCPayload(rawJson) {
  return JSON.parse(rawJson, (key, value) => {
    // パース中、すべてのキーと値のペアに対して呼び出される
    if (isSpecialRSCType(value)) {
      return reconstructRSCNode(value);
    }
    return value;
  });
}
```

Reviver コールバックは JSON 内のすべてのキーと値のペア（深くネストされたものも含む）に対してエンジンから呼び出されます。このコールバックは任意の JavaScript 関数であり、エンジンの C++ パースループの中から実行されるため、エンジンがこのパスを最適化するのは困難です。1 回あたりのオーバーヘッドは小さくても、RSC Payload は密度が高く、50 個のコンポーネントを持つページでは数千のネストされたキーを持つ Payload が生成されることがあります。

## 解決策: 先にパース、次にウォーク

16.2 ではパースと変換を分離するアプローチに変更されました。

```javascript
// 新実装の概念的な表現
function deserializeRSCPayload(rawJson) {
  // Step 1: 素のJSON.parse — エンジンが完全に最適化、コールバックなし
  const parsed = JSON.parse(rawJson);

  // Step 2: 純粋なJavaScriptによる再帰的なウォーク
  return walkRSCTree(parsed);
}

function walkRSCTree(node) {
  if (node === null || typeof node !== "object") {
    return node;
  }
  if (isSpecialRSCType(node)) {
    return reconstructRSCNode(node);
  }
  if (Array.isArray(node)) {
    return node.map(walkRSCTree);
  }
  const result = {};
  for (const key of Object.keys(node)) {
    result[key] = walkRSCTree(node[key]);
  }
  return result;
}
```

着目すべき点は、Reviver なしの `JSON.parse` が V8・SpiderMonkey・JavaScriptCore において最もアグレッシブに最適化されたコードパスの一つだということです。エンジンはこれをほぼネイティブな操作として扱います。Reviver を渡した瞬間、すべてのノードに対してインタープリタの JavaScript に戻ってしまいます。

まず素の `JSON.parse` を実行し、得られたオブジェクトツリーを別の JavaScript パスでウォークすることで、エンジン最適化パースの恩恵を最大限に受けられます。ウォーク自体も通常の JavaScript のオブジェクトトラバーサル、つまり JIT コンパイル可能で予測しやすい処理です。C++ のパース状態マシンと混在しません。

結果として、代表的な RSC Payload でデシリアライズが 350% 高速化されました。

## Streaming への影響

RSC Payload はしばしば Streaming HTTP レスポンスとしてチャンク単位で届きます。クライアントは各チャンクが届き次第処理するため、デシリアライズのレイテンシは Streaming ページの Time to Interactive に直結します。

16.1 では、10 チャンクの RSC データを Streaming するページは Reviver ベースのデシリアライズを 10 回実行していました。各呼び出しはノードごとに Reviver 呼び出しのオーバーヘッドを支払っていました。16.2 では各チャンクが高速な `JSON.parse` パスと別のウォークを通過し、チャンクあたりのレイテンシが下がることで Streaming ページのインタラクティブ化が早くなります。

この改善が最も顕著なのは以下のケースです。

- 深いコンポーネントツリーと多くの Server Components を持つページ
- 大きな Serialize されたデータ構造を扱うダッシュボードやデータ量の多いページ
- Streaming を使用しているルート（`loading.tsx`、`Suspense` 境界）

## dev 起動速度の改善について

`next dev` の 400% 高速化は RSC とは別の改善で、Turbopack のモジュールバンドリングパイプラインの最適化によるものです。依存関係グラフの解決が CPU コアをまたいでより積極的に並列化され、インクリメンタルキャッシュ層も再起動時に効果的になっています。

Next.js 16.1 との比較では、デフォルトアプリケーションで約 87% の改善が報告されています。400% という数字はより広いベースラインとの比較であり、測定対象が異なるため矛盾しているわけではありません。

実務感覚で言うと、16.1 で 12〜15 秒かかっていたプロジェクトは 16.2 では 2〜4 秒で起動するケースが多いです。

## 今回のリリースに含まれるセキュリティ修正

16.2 には複数のサブシステムにまたがるセキュリティパッチも含まれています。

| 対象領域 | 問題の種類 |
|---|---|
| React Server Components | XSS、SSRF |
| App Router | キャッシュポイズニング、認可バイパス |
| Middleware | リクエストバイパス |
| Proxy 設定 | DoS |

App Router のキャッシュポイズニングや認可バイパスは、実際の攻撃パターンが確立されているクラスの脆弱性です。プロダクションで Next.js を運用しているなら、パフォーマンス改善とは別にセキュリティ修正だけでもアップグレードの理由になります。

## アップグレードの注意点

16.1 から 16.2 へのアップグレードに破壊的変更はありません。RSC デシリアライズの変更はフレームワーク内部の変更であり、アプリケーションコードの修正は不要です。

```bash
npm install next@latest
# または
pnpm update next@latest
```

アップグレード後に確認すべき点は以下の通りです。

**1. Server Component を多用しているルートをテストする。** デシリアライズロジックは変更されています。Next.js チームによって正確性は担保されていますが、カスタムの RSC Payload ハンドリングやコンポーネントツリーのエッジケースがある場合は明示的にテストしてください。

**2. デプロイ前にフルテストスイートを実行する。** パフォーマンス改善は受動的で設定不要です。ただし、セキュリティパッチとあわせて考えれば、プロダクションへの迅速なデプロイが推奨されます。

**3. Turbopack の設定場所を確認する。** `next.config.ts` の `experimental.turbopack` 以下に設定が残っている場合は、トップレベルの `turbopack` キーに移動してください。

```typescript
// next.config.ts — 16.2 時点での正しい形式
const nextConfig = {
  turbopack: {
    // Turbopack の設定をここに記述
  },
};
```

## プロダクションアプリへの実際の影響

RSC デシリアライズの改善は、開発体験だけでなくクライアントサイドのレンダリングパフォーマンスに直接影響します。

- **ナビゲーション速度**: App Router のクライアントサイドナビゲーションは RSC Payload のデシリアライズを伴います。350% の高速化はナビゲーション完了とインタラクティブ化の短縮につながります。
- **Server Actions**: Action のレスポンスには RSC Payload が含まれます。Server Actions を使ったフォームや Mutation がクライアント側でより速く反応するようになります。
- **初回ページロード**: SSR + Streaming のページでは、各 Streaming チャンクが高速パースの恩恵を受けます。

効果の大きさは Payload のサイズに比例します。Server Components が少ないシンプルなページでは控えめな改善にとどまりますが、大きなデータセットを Server Components 経由で Serialize するダッシュボードページでは顕著な改善が見込めます。

## まとめ

RSC デシリアライズの修正は、振り返ってみればシンプルな最適化です。Reviver の使用をやめ、別のパスで処理する。しかし JavaScript エンジンが Parsing コールバックをどのように扱うかを理解していなければ、この改善機会に気づくことはできません。350% の向上は実測値であり、エンジンをまたいで再現可能で、アプリケーションコードの変更も不要です。

セキュリティ修正のためにアップグレードし、パフォーマンス改善は同じパッケージとして受け取ってください。
