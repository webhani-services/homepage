Next.js は 2026年7月21日、新設した月次 Security Release Program の第1弾として Next.js 16.2 と 15.5 に対する Patch を公開し、9件の脆弱性（High 4件、Medium 5件）を修正しました。そのうち2件は React Server Components 固有のもので、High 相当の Denial of Service と Medium 相当の Source Code 漏えいです。影響範囲は Server Components・App Router・Middleware/Proxy にまたがり、DoS・認可バイパス・XSS・SSRF・Cache Poisoning が含まれます。

現状特に問題が起きていなくても、本番で Next.js を運用しているなら20分ほど時間を割く価値があります。Routing・Rendering・Middleware・Caching とこれだけ広い領域を持つ Framework は、まさにこの種の脆弱性が蓄積しやすい構造です。また、これまでの不定期な公表から予測可能な月次サイクルへ移行したこと自体も有用で、突発的な CVE への後追い対応ではなく、決まった日付を起点に Patch レビューの習慣を組み込めるようになります。

## 今回のリリースがいつも以上に重要な理由

今回の内容を「一応確認したことにして流す」のではなく、真剣に扱うべき理由がいくつかあります。

- **Middleware / Proxy の不具合は Edge で悪用可能**であり、多くの場合 Application のコードが実行される前に影響します。Middleware の認可バイパスは、堅牢だと思っていた Route 保護を無効化します。
- **RSC の Source Code 漏えい**は、Client Bundle に届くはずのなかった Server 専用ロジック — 環境変数の扱いや内部 API の形、Business Rule — を露出させる可能性があります。
- **App Router の Cache Poisoning** は、ある User 向けのレスポンスが別の User に配信されるという、単なるクラッシュより深刻な失敗モードにつながります。

いずれも特殊な攻撃経路を必要としません。公表から1週間程度で自動スキャンに引っかかるタイプの不具合です。

## Step 1: 現在の Version を確認する

```bash
npx next --version
```

16.2.x または 15.5.x（あるいはそれ以前）を使っている場合は影響を受けるため、更新が必要です。無関係な Breaking Change まで一緒に取り込んでしまわないよう、更新前に該当 Patch Version の Release Note を確認してください。

```bash
npm install next@latest
# 確認済みの Patch Version に明示的に固定する場合
npm install next@16.2.5
```

Deploy 前に Test Suite に加え、認証が必要な Route と Middleware を使う Route に対して Smoke Test を行ってください。Security Patch であっても、依存している挙動に影響が出る可能性はあります。

## Step 2: Framework 側の Patch だけに頼らない

Patch は今回公表された特定の不具合を塞ぐものであり、Framework 側の不具合と組み合わさって初めて問題化する Application 側の実装ミスまでは解決しません。今回のリリースとは関係なく点検しておく価値がある実装パターンを二つ挙げます。

**Middleware の認可チェックは fail-closed にする**

```typescript
// middleware.ts
export function middleware(request: NextRequest) {
  const session = request.cookies.get("session");

  // 悪い例: 明示的に "invalid" のときだけ Redirect し、それ以外は素通り
  if (session?.value === "invalid") {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  // 良い例: 有効な Session であることを明示的に要求し、それ以外はすべて拒否
  if (!session || !isValidSession(session.value)) {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  return NextResponse.next();
}
```

Framework 側のバイパス系不具合は、前者のパターンに対してより危険です。バグが生む想定外の分岐が、すでに「許可」側にフォールバックしているためです。後者のパターンであれば、仮に Middleware にバイパスが存在しても fail-closed のロジックが被害範囲を限定する、多層防御として機能します。

**Server Components で機微な Data を習慣的に Props に流さない**

```typescript
// 避けたい書き方: DB のレコードをそのまま Client Component に渡し、
// 「その Field を描画していないから大丈夫」に依存する
<UserProfile user={dbUser} />

// 推奨: Server 境界で明示的に整形してから渡す
<UserProfile
  user={{
    name: dbUser.name,
    avatarUrl: dbUser.avatarUrl,
  }}
/>
```

RSC の Serialization に関する不具合は、まさに「その Field は描画していないから安全」という前提が崩れる典型的な脆弱性クラスです。Server / Client の境界で Data を明示的に整形しておけば、Framework 側で Serialization の不具合が発生しても、漏えいさせる機微な情報がそもそも存在しないことになります。

## Step 3: 月次サイクルを運用に組み込む

今回から不定期な公表ではなく計画的な Program になったため、Patch Tuesday と同じ扱いにするのが妥当です。定期的な予定として Calendar に登録し、新しい Security Release が公開されたタイミングで Renovate や Dependabot による依存更新 PR が自動的に届くよう Gate しておきます。すでに `next` を CI Gate 付きの自動更新 Workflow に乗せている Team は、こうした Release をほぼ手作業なしで吸収できます。一方、Version を手動で固定し「誰かが気づいたら更新する」運用のままの Team は、1ヶ月後も同じ脆弱性に晒され続けることになります。

## まとめ

- 今すぐ Next.js の Version を確認してください。今回の Release は 16.2.x と 15.5.x の両方が対象で、どちらも更新が必要です。
- Middleware の認可ロジックと RSC への Props 渡しは、Framework 側の不具合と Application 側の実装習慣が組み合わさる典型箇所です。Version 更新だけでなく両方を見直してください。
- Next.js の月次 Security Release Program は予測可能性という点で明確な改善です。個別の CVE 公表を追いかけるのではなく、この Schedule を軸に更新 Process を組み立てることを推奨します。
