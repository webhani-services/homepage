たいていのデータベースのリリースノートは、あなたが一生触らない機能の長いリストです。PostgreSQL 18には、それとは違う二つがあります。ほぼすべてのアプリが引き継ぐデフォルトとデータモデリングの習慣を変えるからです。非同期I/Oサブシステム、そして組み込みの`uuidv7()`関数です。どちらも華やかではありません。そしてどちらも、あなたの読み取りがどれだけ速いか、インデックスがどれだけ上手に歳を取るかを静かに左右します。本記事は、その二つが何をするのか、いつ役立つのか、頼る前に何を確認すべきかを実務目線でたどるガイドです。

## Async I/O：なぜ生まれ、どこに現れるか

歴史の大半、PostgreSQLはOSにデータブロックを一度に一つずつ要求し、次を頼む前に各読み取りが返るのを待っていました。速いローカルSSDでは、この直列化はほとんど気づきません。しかしクラウドストレージ——一回の読み取りが無視できないレイテンシを持つネットワーク接続ボリューム——では、一件ずつ待つやり方が多くのスループットを取りこぼします。ディスクは複数のリクエストを同時に処理できるのに、旧モデルはそれらを頼むことを拒んでいたのです。

PostgreSQL 18は非同期I/Oを導入し、エンジンが複数の読み取りリクエストを同時に投げたまま待ち時間を重ねられるようにします。もっとも明確な効果は、ストレージから大量のデータを流す操作で出ます。大きなテーブルのシーケンシャルスキャン、ビットマップヒープスキャン、そしてVacuumです。あなたのワークロードがクラウドベースのストレージ上でこうしたスキャンに支配されているなら、今回のリリースは、一行のクエリも書き換えずにそれらが速くなる番です。

挙動は`io_method`設定で制御されます。知っておくべき値が三つあります。

```sql
-- 現在の設定を確認
SHOW io_method;

-- 利用可能な値:
--   sync           従来の同期動作（非同期なし）
--   worker         専用のバックグラウンドI/Oワーカープロセスによる非同期
--   io_uring       Linux io_uringインターフェースによる非同期（Linux専用）
```

Linuxでは`io_uring`がバッチ処理をカーネルまで押し込み、たいてい最も効率的な選択肢ですが、カーネルのサポートとプラットフォームのビルド方法に依存します。`worker`は移植性のある経路です。ヘルパープロセスで読み取りを並行発行し、複数のプラットフォームで動きます。18へ移るときは、あなたのビルドがどのメソッドをデフォルトにするか、そしてマネージドのクラウドPostgresを使うならプロバイダーがその選択を公開しているかどうかを確認してください。

すべてのクライアントに伝える注意を一つ。非同期I/Oがどこでもタダだと仮定しないでください。レイテンシの高いストレージ上の読み取り中心・スキャン中心のワークロードで最も効きます。すでにRAMに収まる小さなデータベースや、ワーキングセットが完全にキャッシュされている場合は、差がほとんど出ないこともあります。重ねる物理I/Oがほとんどないからです。他人のハードウェアで出たベンチマークを信じるのではなく、最も重いスキャンについて前後で`EXPLAIN (ANALYZE, BUFFERS)`を取り、実際のワークロードを測ってください。

## UUIDv7：ランダムUUIDが静かに作った問題を直す

二つ目の機能は範囲がより小さいですが、チームが絶えず行うこと——主キー型の選択——に触れます。長年の論争は、自動採番の整数かUUIDかでした。UUIDはいくつかの実務的な理由で勝ります。データベースの往復なしにアプリコードで生成でき、行数を漏らさず、シャードやサービスをまたいで決して衝突しません。ずっと引っかかっていたのは、ランダム版のUUIDv4でした。

UUIDv4は完全にランダムです。これを主キーに使うと、新しい行がインデックスのソート順の予測できない位置に落ちます。B-treeインデックスは、新しいエントリが末尾近くに追記されるときに最も幸せです。最近書いたページがキャッシュに熱く残り、密に詰まるからです。ランダムキーは挿入をインデックス全体にばらまき、冷たいページに触れ、ページ分割を増やし、インデックスを必要以上に大きくキャッシュに優しくない状態に残します。挿入の多いテーブルでは、これは実在する測定可能な税です。

UUIDv7は、ミリ秒精度のタイムスタンプを上位ビットに置き、下位ビットだけにランダム性を入れてこれを解きます。結果は依然としてグローバルに一意で、データベースの外で生成しても安全ですが、近い時刻に作られた値どうしが近くにソートされます。挿入がふたたびおおむね順次的になり、これはまさにB-treeが望むものです。PostgreSQL 18はネイティブジェネレーターを備えるので、もう拡張やアプリ側ライブラリは要りません。

```sql
-- 一つ生成
SELECT uuidv7();

-- 主キーのデフォルトとして使う
CREATE TABLE orders (
    id          uuid PRIMARY KEY DEFAULT uuidv7(),
    customer_id bigint NOT NULL,
    total_cents integer NOT NULL,
    created_at  timestamptz NOT NULL DEFAULT now()
);
```

うれしい副次効果があります。タイムスタンプが先頭ビットに宿るため、近い時刻に挿入された行がインデックス内で物理的にまとまり、`id`で並べるクエリがおおむね時系列で結果をタダで返します。新しい順の一覧のための別の`created_at`インデックスをしばしば避けられます。誇張はしないでください。厳密な時間順が必要なら、本物のタイムスタンプ列を持ちそれで並べてください。しかし「最新のN行をくれ」というページネーションなら、UUIDv7主キーで並べるのはたいてい十分で、より安上がりです。

## 既存のUUIDv4キーを移行すべきか

すぐに続く問いであり、私たちの答えはたいていノーです。少なくとも単独のプロジェクトとしては。大きな既存テーブルの主キーを書き換えるのは高く、危険です。テーブルとすべてのインデックスを書き直し、外部キーとの慎重な調整が要り、得られるのは*未来の*挿入パターンの改善だけで、過去のものではありません。すでに書かれた行はすでに散らばっており、型を変えても全面再構築なしには集まり直しません。

実用的な経路は、新しいテーブルと、別の理由ですでに再構築中のテーブルにUUIDv7を導入することです。まっさらなサービスなら、初日から`uuidv7()`をデフォルト主キーにしてください。UUIDの使い勝手とともに、順次整数に近い挿入挙動が得られます。ほぼすべての利益をほぼ無コストで取り込めます。既存の熱いテーブルの専用移行は、インデックスの肥大化や挿入の競合を測定し、それを具体的にキーのランダム性に帰属できる場合に取っておいてください。

## 短い導入チェックリスト

クライアントのPostgreSQL 18への移行を手伝うとき、データベース水準のレビューはいくつかの確認に収れんします。

1. **`io_method`を確認し、重要なスキャンをテストする。** 最も重いシーケンシャル・ビットマップスキャンを特定し、18で`EXPLAIN (ANALYZE, BUFFERS)`を取り、ベースラインと比べる。Changelogよりあなたの数字を信じる。
2. **新しい主キーは`uuidv7()`をデフォルトにする。** bigintを好む具体的な理由がない限り。UUIDの分散に優しい性質を保ちつつ、インデックス肥大化のペナルティを落とせます。
3. **既存キーを反射的に一括移行しない。** UUIDv4からUUIDv7への変換は、日常的なアップグレード手順ではなく、証拠に裏打ちされた狙い撃ちの修正として扱う。
4. **マネージドサービスのサポートを再確認する。** マネージドのクラウドPostgresで動かすなら、設計をその上に載せる前に、プロバイダーが実際にどの`io_method`値と機能を公開しているか確認する。表面のバージョン番号は、すべてのつまみがあなたに開かれていることを保証しません。

PostgreSQL 18は、最も価値あるデータベース機能がしばしば最も地味だということをよく示しています。Async I/Oは、ほとんどのクラウドアプリが実際に走るストレージで既存のクエリを速くし、UUIDv7は、あらゆるスキーマが下す判断に長く貼りついていた税を取り除きます。どちらもアプリの書き直しを要求しません。どこに効くのかを理解し、あなたのワークロードで利益を検証するだけです。
