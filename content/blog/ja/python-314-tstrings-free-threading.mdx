Python 3.14 は 2025 年 10 月にリリースされ、2026 年 2 月には 3.14.3 のパッチが公開されました。今回の主要な変更は **t-strings (PEP 750)**、**Free-threaded Python (PEP 779)**、**deferred annotations (PEP 649)** の 3 つです。どれも実務での使い方が変わる可能性があるため、順に整理します。

## t-strings (PEP 750)：テンプレート文字列

`t-strings` は f-strings と同じ構文を持ちますが、即座に文字列を生成するのではなく、**テンプレートオブジェクト**を返します。これにより、補間される値を処理前に検査・変換できます。

```python
from string.templatelib import Template

name = "Alice'; DROP TABLE users; --"

# f-string は即座に文字列化される（危険）
query_f = f"SELECT * FROM users WHERE name = '{name}'"

# t-string はテンプレートオブジェクトを返す
query_t = t"SELECT * FROM users WHERE name = '{name}'"
print(type(query_t))  # <class 'string.templatelib.Template'>
```

### SQL インジェクション防止への応用

t-strings を活用すると、ライブラリ側で補間値を安全に処理できます。

```python
def safe_query(template: Template) -> tuple[str, list]:
    """t-string をパラメータ化クエリに変換する"""
    parts = []
    params = []
    for item in template:
        if isinstance(item, str):
            parts.append(item)
        else:
            # Interpolation オブジェクト — 値を直接埋め込まず、プレースホルダーに置換
            parts.append("?")
            params.append(item.value)
    return "".join(parts), params

name = "Alice'; DROP TABLE users; --"
sql, params = safe_query(t"SELECT * FROM users WHERE name = '{name}'")
print(sql)     # SELECT * FROM users WHERE name = '?'
print(params)  # ["Alice'; DROP TABLE users; --"]
```

同様のパターンは HTML エスケープや Shell コマンド生成にも応用できます。ライブラリが t-strings に対応すれば、使う側が意識しなくてもインジェクションが防げる構造になります。

### HTML テンプレートでの活用

```python
def html(template: Template) -> str:
    from html import escape
    result = []
    for item in template:
        if isinstance(item, str):
            result.append(item)
        else:
            result.append(escape(str(item.value)))
    return "".join(result)

user_input = "<script>alert('xss')</script>"
safe_html = html(t"<p>Hello, {user_input}!</p>")
print(safe_html)
# <p>Hello, &lt;script&gt;alert(&#x27;xss&#x27;)&lt;/script&gt;!</p>
```

## Free-threaded Python (PEP 779)：GIL の廃止

Python 3.14 では **Free-threaded Python** が正式サポートになりました。これは GIL (Global Interpreter Lock) を取り除き、真の並列スレッド実行を可能にするものです。

ただし、Free-threaded は**オプトイン**です。通常インストールの Python 3.14 は従来通り GIL を持ちます。Free-threaded ビルドは別途インストールが必要です。

```bash
# uv で Free-threaded ビルドをインストール
uv python install 3.14t

# または pyenv で
pyenv install 3.14t
```

### 実際の並列処理の変化

```python
import threading
import time

def cpu_intensive_task(n):
    """CPU バウンドな計算"""
    result = 0
    for i in range(n):
        result += i * i
    return result

# Free-threaded Python では、これが真に並列で動く
threads = [
    threading.Thread(target=cpu_intensive_task, args=(10_000_000,))
    for _ in range(4)
]

start = time.perf_counter()
for t in threads:
    t.start()
for t in threads:
    t.join()
elapsed = time.perf_counter() - start

print(f"Elapsed: {elapsed:.2f}s")
# GIL あり: ~4.0s (実質シリアル)
# Free-threaded: ~1.2s (真の並列)
```

### 注意点

Free-threaded Python はまだ **experimental** に近い位置づけです。

- C 拡張ライブラリが GIL を前提に実装されている場合、動作が不安定になる可能性がある
- numpy、pandas などの主要ライブラリは対応を進めているが、全てではない
- データ競合が起きやすくなるため、スレッドセーフな設計が必要

CPU バウンドな処理で multiprocessing から threading への移行を検討している場合、まずベンチマークを取り、使用ライブラリの対応状況を確認してから判断することを推奨します。

## Deferred Annotations (PEP 649)

アノテーションの評価が遅延するようになりました。これは `from __future__ import annotations` と同様の効果を、import なしで実現します。

```python
# Python 3.14 以前：前方参照でエラーになるケース
class Node:
    def children(self) -> list[Node]:  # NameError: 'Node' is not defined
        return self._children

# Python 3.14 以降：アノテーションが遅延評価されるため問題なし
class Node:
    def children(self) -> list[Node]:  # OK
        return self._children
```

循環参照を含む型アノテーションや、起動時のアノテーション処理コストが高いケースで恩恵があります。

## Tail-call インタープリタによる性能改善

3.14 では新しい tail-call インタープリタが導入されました。対応コンパイラ（LLVM 19+、GCC 13+）でビルドした場合、Python 全体の速度が 5〜15% 改善するケースがあります。

```bash
# CPython をソースからビルドする場合、自動で適用される
# 公式 macOS / Windows バイナリにも実験的 JIT が含まれる
python3.14 -c "import sys; print(sys._base_executable)"
```

## まとめ

Python 3.14 で実務に直結する変更は主に 2 つです。

| 機能 | 使いどころ | 注意点 |
|------|-----------|--------|
| t-strings | SQL/HTML のインジェクション防止 | ライブラリの対応が必要 |
| Free-threaded | CPU バウンド処理の並列化 | 別ビルドが必要、実験的 |

t-strings はライブラリエコシステムが充実してくる今後 1〜2 年が本格普及のタイミングです。Free-threaded Python は本番投入前に十分な検証が必要ですが、長期的には Python の並行処理の在り方を変える可能性があります。
