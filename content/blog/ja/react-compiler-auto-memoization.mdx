React Compiler v1.0は2025年10月にリリースされました。2026年初頭には、React 19を使う新規プロジェクトにおける標準的なSetupになりつつあります。実際の変化として、`useMemo`、`useCallback`、`React.memo`——何年もReact開発者の通過儀礼だった手動のPerformance最適化ツール群——が徐々にLegacy Patternとして扱われるようになっています。

本記事では、Compilerの動作原理、既存Codebaseから安全に削除できる記述、そして手動での介入がまだ意味を持つケースを整理します。

## Compilerが行うこと

React CompilerはBuild時に実行されるBabel/SWCのPluginです。ComponentとHookを静的解析し、自動的にMemoizeしても安全な値を特定し、Compileされた出力に`useMemo`/`useCallback`相当の処理を挿入します——開発者によるAnnotationは不要です。

Compilerが分析する対象:
- PropsとStateのみに依存する純粋な計算
- Event HandlerやDependencyとして渡される安定したFunction Reference
- 無関係なState変更時に再Renderの必要がないComponent Subtree

重要な点として、ReactのRuleに従ったCode（純粋なRendering、Render内にSide Effectなし）であれば、Compilerはこれらを正確に判断できます。Type SystemやAnnotationは不要で、Codeの構造から推論します。

## Before / After

React Compiler導入前の典型的なパターン:

```typescript
function ProductList({ products, onSelect }: Props) {
  const sortedProducts = useMemo(
    () => [...products].sort((a, b) => a.name.localeCompare(b.name)),
    [products]
  );

  const handleSelect = useCallback(
    (id: string) => onSelect(id),
    [onSelect]
  );

  return (
    <ul>
      {sortedProducts.map((p) => (
        <ProductItem key={p.id} product={p} onSelect={handleSelect} />
      ))}
    </ul>
  );
}
```

React Compilerを有効にすると、MemoizationはCompile済み出力が担います。Sourceは次のようになります:

```typescript
function ProductList({ products, onSelect }: Props) {
  const sortedProducts = [...products].sort((a, b) =>
    a.name.localeCompare(b.name)
  );

  return (
    <ul>
      {sortedProducts.map((p) => (
        <ProductItem key={p.id} product={p} onSelect={onSelect} />
      ))}
    </ul>
  );
}
```

Performance特性は同一です。Sourceの記述量が大幅に減ります。

## React CompilerのSetup

```bash
npm install --save-dev babel-plugin-react-compiler
```

```json
// babel.config.json
{
  "plugins": ["babel-plugin-react-compiler"]
}
```

Next.js 15以降では、`next.config.ts`に組み込み済みです:

```typescript
import type { NextConfig } from "next";

const config: NextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default config;
```

CompilerはデフォルトでOpt-inモードで動作します。最適化が安全と確認できたComponentのみを処理し、ReactのRuleに違反するComponentはスキップします——Appが壊れることはありません。

## 既存Codebaseから削除できるもの

既存Codebaseに導入する際の現実的な手順:

1. Compilerを有効にしてTest Suiteを実行——破損がないことを確認する
2. React CompilerのESLint Pluginで不要な手動Memoizationを検出する:

```bash
npm install --save-dev eslint-plugin-react-compiler
```

3. PropsとStateのみに依存する純粋な計算を包む`useMemo`を削除する
4. Compilerでも処理されるChild Componentに渡されるHandlerの`useCallback`を削除する
5. Compilerが自動処理するようになったComponentの`React.memo`を削除する

手動Memoizationを残すべきケース:
- 参照安定性が必要な外部Library連携（複雑なChart Configurationなど）
- 意図的に遅延させる重い計算（オンデマンドで発火するデータ変換など）
- `"use no memo"` DirectiveでOpt-outしたComponent

## Opt-outが必要な場合

意図的に非純粋な振る舞いをするComponentのためのEscape Hatch:

```typescript
function LegacyWidget({ data }: Props) {
  "use no memo";
  // このComponentはCompilerによる最適化の対象外になります。
  // 特殊な参照要件を持つThird-party連携に使用してください。
}
```

## 実際のメリット

手動Memoizationを排除する最大の恩恵は、Raw Performanceよりも可読性と認知負荷の低減です。`useMemo`が5つ`useCallback`が3つあるComponentは、読む人にDependency Arrayの追跡と各値の再計算タイミングの推論を強います。

CompilerがこれらをHandleすることで、Component Codeは「どう最適化するか」ではなく「何をRenderするか」の表現に戻ります。これはBugの温床となるSurface Areaの有意な削減です。

## まとめ

- React Compiler v1.0は本番Ready、React 19新規プロジェクトの標準パスへ
- ReactのRuleに従ったComponentに対してMemoizationを自動で処理する
- `useMemo`、`useCallback`、`React.memo`はEdge Caseには有効——ただしDefaultであるべきではない
- 既存Codebaseへの導入はESLint PluginとTest Suiteを活用して段階的に進める
- 主なメリットはよりCleanで読みやすいCode——Performanceは維持される
