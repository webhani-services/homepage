React 19.2のリリースにより、React CompilerがStableステータスに到達しました。これは単なるバージョンアップではなく、Reactの最適化アプローチそのものが変わる転換点です。同時に、TypeScriptは2026年のフロントエンド開発において「導入するかどうか」ではなく「いかに使いこなすか」の段階に移行しています。

## React Compilerとは何か、何が変わるのか

React CompilerはReact 19.0でExperimentalとして導入され、19.1でベータを経て、19.2でStableに昇格しました。コンパイラの役割は、開発者が手動で書いていた`useMemo`、`useCallback`、`React.memo`をビルド時に自動的に最適化することです。

### これまでの問題

従来のReact開発では、不必要な再レンダリングを防ぐために開発者が手動でメモ化を管理する必要がありました。

```tsx
// コンパイラ前: 手動でメモ化
const UserCard = React.memo(({ user, onDelete }: Props) => {
  const formattedDate = useMemo(
    () => formatDate(user.createdAt),
    [user.createdAt]
  );
  const handleDelete = useCallback(
    () => onDelete(user.id),
    [onDelete, user.id]
  );

  return (
    <div>
      <span>{user.name}</span>
      <span>{formattedDate}</span>
      <button onClick={handleDelete}>削除</button>
    </div>
  );
});
```

この書き方には問題があります。依存配列を間違えると最適化が効かなくなるか、逆に古い値を参照し続けるバグが発生します。レビューコストが高く、経験が浅いメンバーが書いたコードは特にミスが起きやすい部分でした。

### コンパイラ後の世界

React Compilerを有効にすると、このようなコードが自動で最適化されます。

```tsx
// コンパイラ後: 余分なメモ化なしに書ける
const UserCard = ({ user, onDelete }: Props) => {
  const formattedDate = formatDate(user.createdAt);
  const handleDelete = () => onDelete(user.id);

  return (
    <div>
      <span>{user.name}</span>
      <span>{formattedDate}</span>
      <button onClick={handleDelete}>削除</button>
    </div>
  );
};
```

コンパイラはソースコードを静的に解析し、どの値が変化したときにどの計算やコンポーネントを再実行すべきかを判断して、ビルド出力に最適化を挿入します。

## 導入時に確認すべき点

### Next.jsプロジェクトでの有効化

Next.js 15.x以降ではReact Compilerのサポートが組み込まれています。

```js
// next.config.mjs
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default nextConfig;
```

有効化した後は、既存の`useMemo`や`useCallback`が不要になるケースが多くあります。ただし、既存コードをすぐに全削除する必要はありません。コンパイラはこれらが残っていても正しく動作します。不要なフックの削除は段階的に行うのが現実的です。

### コンパイラが苦手なパターン

コンパイラはReactのルールに従ったコードで最大の効果を発揮します。次のようなパターンはコンパイラの最適化が効きにくくなります。

- Propsや状態を直接変更する（immutabilityの違反）
- `useEffect`内で複数の副作用を混在させている
- カスタムフックが同じ入力に対して異なる参照を返す

ESLint plugin `eslint-plugin-react-compiler`を導入することで、コンパイラが最適化できないパターンを事前に検出できます。

```bash
npm install --save-dev eslint-plugin-react-compiler
```

```json
// .eslintrc.json
{
  "plugins": ["react-compiler"],
  "rules": {
    "react-compiler/react-compiler": "error"
  }
}
```

## TypeScript：「使うかどうか」から「どう使うか」へ

2026年のフロントエンド開発現場では、TypeScriptはもはや選択肢ではなく前提です。新規プロジェクトでJavaScriptだけで始めるという判断は、チームへの説明コストが高くなってきています。

重要なのは、TypeScriptを「型を書くための言語」として捉えるのではなく、**コードの構造を設計するためのツール**として活用することです。

### 型による設計の具体例

```typescript
// Bad: any を使って型チェックを回避
function processPayment(data: any) {
  return fetch('/api/payment', {
    method: 'POST',
    body: JSON.stringify(data),
  });
}

// Good: 型で入力の仕様を明示する
type PaymentRequest = {
  amount: number;
  currency: 'JPY' | 'USD' | 'EUR';
  customerId: string;
  description?: string;
};

async function processPayment(data: PaymentRequest): Promise<{ transactionId: string }> {
  const res = await fetch('/api/payment', {
    method: 'POST',
    body: JSON.stringify(data),
    headers: { 'Content-Type': 'application/json' },
  });
  return res.json();
}
```

`PaymentRequest`型を定義することで、呼び出し側で通貨コードのタイポや数値型の混入をコンパイル時に検出できます。ドキュメントとしての役割も果たします。

### React 19の型定義改善

React 19ではServer ComponentsとActions APIに合わせて型定義が大幅に改善されています。

```typescript
// Server Actions の型
'use server';

export async function createPost(
  prevState: { error: string | null },
  formData: FormData
): Promise<{ error: string | null }> {
  const title = formData.get('title');
  if (typeof title !== 'string' || title.length === 0) {
    return { error: 'タイトルは必須です' };
  }
  // 保存処理...
  return { error: null };
}
```

`useActionState`フックと組み合わせることで、サーバーサイドのバリデーションエラーをフォームに反映する型安全なパターンが書けます。

## 実プロジェクトへの適用戦略

### 新規プロジェクト

React 19.2 + TypeScript strict mode + React Compilerを最初から有効にした構成をベースラインとして始めることを推奨します。

```bash
npx create-next-app@latest my-app \
  --typescript \
  --tailwind \
  --eslint \
  --app
```

`tsconfig.json`では`strict: true`を必ず有効にしてください。後から厳格モードに移行するコストは非常に高いです。

### 既存プロジェクト

段階的な移行が現実的です。

1. **React Compilerの有効化**（低リスク）: `next.config.mjs`に1行追加するだけ。既存コードへの影響は最小限
2. **TypeScript strictの段階的適用**: `strict: false`のまま`noImplicitAny`から始め、一ファイルずつ型を改善
3. **不要なuseMemo/useCallbackの削除**: コンパイラが安定したタイミングで、段階的に整理

## まとめ

React Compiler Stableの到達は、Reactが「パフォーマンス最適化の責任をフレームワークが担う」方向に舵を切った重要なマイルストーンです。TypeScriptの標準化とあわせて、フロントエンドの開発体験は確実に向上しています。

一方で、コンパイラはReactのルールを守ったコードを前提とします。`eslint-plugin-react-compiler`でルール違反を事前に検出し、クリーンなコードベースを維持することが、コンパイラの恩恵を最大限に受けるための前提条件です。
