React Server Components（RSC）のプロトコルに、CVSS 10.0 と評価される Remote Code Execution（RCE）の脆弱性が見つかりました。研究者によって「React2Shell」と名付けられ、CVE-2025-55182（React 本体）および CVE-2025-66478（Next.js）として追跡されています。App Router を使う Next.js 15.x / 16.x と React 19.x が影響を受けるため、当社の標準スタックである Next.js 16 を含め、多くのプロジェクトが対象になります。本記事では、慌てずに対処するための確認手順と、根本的な考え方を整理します。

## 何が問題なのか

React2Shell は、RSC が受け取るリクエストの Deserialization 処理に起因する脆弱性です。RSC はクライアントとサーバーの間で、シリアライズされたペイロードをやり取りしてコンポーネントの状態やアクションを伝えます。この処理において、攻撃者が細工したリクエストを送ると、サーバー側で意図しないコードが実行され得る、というのが問題の核心です。

深刻なのは、認証前（pre-authentication）に単一の HTTP リクエストだけで成立し得る点です。ログインや特別な権限を必要とせず、公開されているエンドポイントに到達できれば攻撃が試行可能になります。CVSS 10.0 という最大値の評価は、この「到達性の高さ」と「影響の大きさ」を反映したものです。

さらに、公開直後から実際の悪用が観測されている点も見逃せません。セキュリティベンダー各社の報告によれば、公開から数時間のうちに複数の攻撃グループによる Exploit の試行が確認され、その多くは Coin Miner の設置を狙ったものでした。「まだ攻撃されていないから大丈夫」という前提は、この脆弱性には通用しません。

## まず影響範囲を確認する

対応の第一歩は、自分たちのアプリが影響バージョンを使っているかの確認です。Next.js の場合、公式が提供する診断・更新用のコマンドが用意されています。

```bash
# 影響の有無をチェックし、推奨バージョンへ更新する
npx fix-react2shell-next
```

このツールは、現在の Next.js バージョンを検出し、決められた対応バージョンへの Bump を提案・実行します。手動で確認したい場合は、主要な対応バージョンの目安として以下を参照してください（マイナーバージョンごとにパッチが分かれています）。

- 15.0.x 系 → 15.0.5
- 15.1.x 系 → 15.1.9
- 15.5.x 系 → 15.5.7
- 16.0.x 系 → 16.0.7

React 単体では、19.0.1 / 19.1.2 / 19.2.1 で修正が入っています。フレームワークを介さず React を直接利用している場合は、こちらの系列に合わせて更新します。

CI に組み込んでいる依存監査があれば、そのアラートも併せて確認します。`package-lock.json` を固定している環境では、更新後にロックファイルの再生成まで確実に行い、デプロイされる成果物が本当にパッチ済みかを確認することが重要です。

## パッチだけで終わらせない — 多層防御

バージョン更新が最優先ですが、それだけを対処の全てにすべきではありません。今回のように「1 リクエストで RCE」という性質の脆弱性は、パッチ適用までの空白時間や、見落とされたサブシステムから被害が広がります。以下の観点を併せて点検しておくと安心です。

まず、WAF やリバースプロキシの層で、RSC のエンドポイントに向かう不審なペイロードを検知・遮断できるかを確認します。緊急の緩和策として、パッチ適用が完了するまでの時間稼ぎに使えます。

次に、サーバープロセスの権限最小化です。アプリケーションを実行するユーザーの権限を絞り、Container であれば読み取り専用のファイルシステムや不要な Capability の削除を徹底しておくと、万一 Exploit が成立しても、その後の横展開を難しくできます。

そして、Egress（外向き通信）の監視です。今回の実被害の多くは Coin Miner の設置でした。サーバーから見慣れないマイニングプールや外部ホストへの通信が発生していないかを監視しておくと、侵害の兆候を早期に捉えられます。

```javascript
// 例: 監視すべき「侵害後の挙動」の観点をチェックリスト化しておく
const postExploitSignals = [
  "見慣れない外部ホストへの Egress 通信",
  "想定外のプロセス起動（マイニングバイナリなど）",
  "アプリ実行ユーザーによる予期しないファイル書き込み",
  "CPU 使用率の恒常的な高止まり",
];
```

## webhani の視点

こうしたクリティカルな脆弱性への対応で差が出るのは、「どれだけ速く、漏れなく更新できるか」という運用体制です。技術的にはコマンド一発で済む更新であっても、対象となるプロジェクトの棚卸しができていなければ、パッチが当たらないアプリが残ります。

日頃から、自社・顧客のどのプロジェクトがどのフレームワークのどのバージョンで動いているかを把握できる状態を作っておくこと。そして、依存関係の監査を CI に組み込み、こうしたアラートが自動で上がる仕組みを持っておくこと。この二点があれば、React2Shell のような事案が起きても、対応を「調査」ではなく「実行」から始められます。

当社では、フレームワークのバージョン管理と脆弱性対応のプロセス整備も含めた開発・運用の支援を行っています。今回の React2Shell は、パッチ適用の速さそのものよりも、「速く動ける準備ができているか」が問われる事案だと捉えています。まずは `npx fix-react2shell-next` で自分たちの現状を確認し、そのうえで運用体制の見直しに着手することをおすすめします。
