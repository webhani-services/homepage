TypeScript 7.0 の RC が 2026 年 6 月に公開され、GA も間近とされています。最大の変化は、これまで TypeScript 自身で書かれていた Compiler が Go に移植されたことです。コードネームは Project Corsa。Microsoft は、型チェックが 6.0 比でしばしば約 10 倍速くなると報告しています。本記事では、この変化を CI で試すための具体的な手順と、移行前に押さえておくべき点を整理します。

## 何が変わったのか — 「作り直し」ではなく「移植」

まず誤解しやすい点を整理します。Project Corsa は、型チェッカーを設計し直したわけではありません。既存の TypeScript 実装（Strada と呼ばれる従来コードベース）を、ファイル単位でほぼ忠実に Go へ移植する方針が採られました。型チェックの挙動を同じに保つことを優先した「トランスプラント（移植）」であり、型システムの意味論が大きく変わったわけではない、というのが重要な前提です。

では、なぜ速くなるのか。理由は主に二つです。一つは、JavaScript ランタイム上ではなくネイティブバイナリとして動くこと。もう一つは、Worker Thread 間で共有メモリを使った並列処理が効くことです。Microsoft はこの二点の合わせ技で、約 10 倍という数字を説明しています。

## CI で試す — まずは型チェックだけ

RC 時点では、標準パッケージの `tsc` バイナリがそのまま Go ネイティブ Compiler になっています。以前のプレビューでは `tsgo` という別名で提供されていましたが、RC ではその名前で呼び分ける必要はありません（`tsgo` バイナリは nightly の `@typescript/native-preview` に残っています）。

導入の第一歩としては、いきなり全面移行せず、CI の型チェック工程だけを新しい `tsc` に差し替えて計測するのが安全です。

```bash
# RC を開発依存として試験導入する
npm install -D typescript@rc

# 型チェックのみを実行（出力は生成せず、型エラーだけ確認）
npx tsc --noEmit
```

`--noEmit` で型チェックだけを走らせれば、既存のビルドパイプライン（Bundler や Transpile 工程）に手を入れずに、型チェックの速度と結果だけを比較できます。CI のログでジョブ時間を before / after で比べれば、自分たちのコードベースで実際にどれだけ短縮されるかが分かります。大規模なコードベースほど、この差は体感しやすくなります。

## 移行前に押さえるべきこと

「移植であって作り直しではない」とはいえ、メジャーバージョンの更新には注意点があります。TypeScript 7.0 に至る流れでは、その手前の 6.0 で既定値の変更が予告されており、strict 系の設定がより厳しくなる方向、古いターゲットやモジュール解決方式の整理が進む方向で議論されてきました。長く運用しているプロジェクトほど、緩い設定に依存している箇所が表面化する可能性があります。

したがって、実務での移行はおおむね次の順序を勧めます。

- まず RC を `--noEmit` で回し、型エラーが新たに出ないかを確認する
- 出た場合は、それが「本当のバグの顕在化」なのか「設定の厳格化による指摘」なのかを切り分ける
- ビルド成果物（Emit 結果）を使う工程は、型チェック工程の検証が済んでから差し替える

RC はあくまで RC です。プロダクションのビルドを一足飛びに置き換えるのではなく、まずは型チェックという「壊れても被害の小さい工程」で速度と互換性を確かめる。この段階的なアプローチが、移行のリスクを最小化します。

## webhani の視点

Compiler の高速化は、一見すると地味な改善に見えます。しかし、型チェックは開発者がローカルで何度も回し、CI でも毎回走る工程です。ここが 10 倍速くなるということは、開発者の待ち時間と CI の実行コストが恒常的に下がるということです。フィードバックループが短くなれば、その積み重ねは開発体験そのものを変えます。

当社が顧客のフロントエンド基盤を支援する際に重視しているのも、こうした「日々効いてくる」土台の改善です。TypeScript 7.0 は、新しい構文や派手な機能ではなく、既存の型システムをそのままに実行基盤だけを速くする、という堅実なアップデートです。だからこそ、移行のハードルは比較的低く、得られる効果は継続的です。まずは検証環境で `tsc --noEmit` の速度を測り、自分たちのコードベースでの効き目を確かめることから始めてみてください。
