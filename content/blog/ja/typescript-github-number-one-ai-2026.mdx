## 何が起きたか

GitHub の Octoverse 2025 レポートによると、TypeScript が 2025 年 8 月に Python と JavaScript の両方を抜き、GitHub で最も使われる言語になりました。1 年間で 100 万人以上の新規開発者が TypeScript を採用し、成長率は前年比 66% という記録的な数値です。

GitHub はこの変化を「10 年以上で最も重大な言語シフト」と表現しています。しかし、この数字よりも興味深いのは「なぜ起きたか」という理由です。

---

## Convenience Loop とは何か

TypeScript の急成長には、AI コーディングツールとの相乗効果が大きく関与しています。これを「Convenience Loop」と呼びます。

仕組みはシンプルです。

1. AI ツールが TypeScript を書きやすくする
2. 開発者が TypeScript を選びやすくなる
3. TypeScript のコードが GitHub 上でさらに増える
4. LLM の学習データに TypeScript が増える
5. AI が TypeScript をさらに得意になる
6. → 1 に戻る

AI が言語を「摩擦なし」に感じさせると、開発者がその言語に流れ込みます。それがさらに多くの学習データを生み、AI がその言語をさらに得意にします。この自己強化サイクルが TypeScript を加速させています。

## TypeScript が AI との相性がよい理由

AI ツールが TypeScript を好む技術的な理由があります。

**型情報がエラーを即時に捕捉する**  
GitHub の調査によると、LLM が生成したコードのコンパイルエラーの 94% は型チェックの失敗です。TypeScript は型違反を即座にフラグアップするため、AI が書いたコードの品質を自動的に検証できます。

```typescript
// AI が生成した型エラーを TypeScript が即時検出する例
interface User {
  id: number;
  name: string;
  email: string;
}

function sendEmail(user: User) {
  // TS2339: Property 'mail' does not exist on type 'User'
  // TypeScript が誤りを即座に指摘 — JavaScriptではランタイムまで気づかない
  console.log(user.mail);
}
```

**インターフェース定義が AI のコンテキストになる**  
型定義ファイルは AI にとって、コードの意図を伝える最も正確な情報源です。`interface Order { items: OrderItem[]; total: number; currency: string; }` という定義があれば、AI は関連する関数やコンポーネントを正確に推論できます。

**補完と型推論の精度が上がる**  
TypeScript の型情報は、IDE の補完精度を高めます。AI が提案するコードの適切さを型システムがリアルタイムで評価できるため、人間のレビュー負荷が下がります。

## JavaScript との棲み分け

TypeScript の台頭は JavaScript の終わりを意味しません。二つは共存しており、役割が分かれています。

| ユースケース | TypeScript | JavaScript |
|------------|------------|------------|
| Web アプリ、API | ◎ | △ |
| 大規模チーム開発 | ◎ | △ |
| スクリプト・自動化 | ○ | ◎ |
| 素早いプロトタイプ | △ | ◎ |
| ブラウザ直接実行 | △（トランスパイル必要） | ◎ |

TypeScript が強いのは「型が仕様書になる」場面、つまり複数人が長期間メンテナンスするプロジェクトです。特に AI ツールを活用する場面では、型情報が AI の正確性を高めるため、TypeScript の優位性は増します。

## Next.js と主要フレームワークのデフォルト化

TypeScript の成長を後押しする要因の一つが、主要フレームワークのデフォルト設定です。

Next.js、SvelteKit、Qwik、Astro など、ほぼすべての主要フロントエンドフレームワークが TypeScript をデフォルトでスキャフォールドするようになりました。つまり、新しいプロジェクトを始めると、最初から TypeScript で始まります。

```bash
# Next.js の新規プロジェクト作成 — デフォルトで TypeScript
npx create-next-app@latest my-app
# ✓ Would you like to use TypeScript? … Yes

# 生成される tsconfig.json と .ts/.tsx ファイル構成
my-app/
├── tsconfig.json
├── src/
│   ├── app/
│   │   ├── page.tsx     ← .tsx がデフォルト
│   │   └── layout.tsx
│   └── components/
│       └── Button.tsx
```

フレームワークのデフォルトが TypeScript になることで、「TypeScript を選ぶ」ではなく「JavaScript を選ばなければ TypeScript」という状況が生まれています。

## AI × TypeScript の実践的な活用

AI ツールと TypeScript を組み合わせる際に、実際に効果が高いパターンをいくつか紹介します。

**型定義を先に書く**  
コード実装より先にインターフェースと型定義を書くと、AI がより正確なコードを生成します。AI に対する「仕様書」として型が機能します。

```typescript
// 先に型定義を書いてから AI に実装を依頼する
interface PaymentResult {
  success: boolean;
  transactionId: string;
  amount: number;
  currency: "JPY" | "USD" | "EUR";
  timestamp: Date;
}

// この型定義があることで、AI は適切な実装を生成しやすくなる
async function processPayment(
  amount: number,
  currency: PaymentResult["currency"]
): Promise<PaymentResult> {
  // AI がここを実装 — 型制約が正確性を担保する
}
```

**Strict モードを有効にする**  
`strict: true` を有効にすることで、AI が生成したコードの型エラーをより厳密に検出できます。AI との協働では、緩い型より厳しい型の方が信頼性が高まります。

```json
// tsconfig.json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

webhani が新規プロジェクトで TypeScript を推奨する理由は以前から変わりません——型が仕様書になり、リファクタリングが安全になり、チーム規模が大きくなっても品質を保てるからです。ただし今は、もう一つの理由が加わりました。AI ツールとの相性が良いため、AI 活用の恩恵を最大化できるからです。

JavaScript から TypeScript への移行を迷っているチームがあれば、今がその踏み切りのタイミングです。AI ツールのサポートがあれば、移行のコストは以前より大幅に下がっています。
