2025年の GitHub Octoverse レポートに、注目すべき数字が含まれていました。TypeScript が Python と JavaScript を抜き、GitHub 上で最も多く使われるプログラミング言語の座に就いたのです。

この変化の背景には、単純な技術的優位性以上の要因があります。AI Coding Assistant の普及がプログラマーの言語選択そのものを変え始めています。

## 数字で見る変化

GitHub Octoverse 2025 のデータから主要な指標を整理します。

- TypeScript が Python・JavaScript を抜いて **GitHub 上の言語使用率第 1 位**に（2025年8月）
- 新規 GitHub 開発者の **80% が最初の 1 週間で Copilot を使用**
- LLM SDK を使用するパブリックリポジトリが **110 万件以上**（過去 12 ヶ月で 178% 増加）
- 開発者の **84% が AI ツールを使用または使用予定**、51% が毎日使用

GitHub のレポートはこの現象を「Convenience Loop」と呼んでいます。AI ツールを使い始めた開発者が TypeScript を選ぶと、AI のアシスタンスがより正確になり、それが TypeScript を使い続ける動機になる。この正のフィードバックループが、言語シェアの急速なシフトを生んでいます。

## なぜ TypeScript が AI と相性が良いのか

LLM がコードを生成する際の本質的な制約があります。それは型情報です。

2025 年の学術研究によると、LLM が生成するコードのコンパイルエラーの **94% が型チェックの失敗**でした。型アノテーションがあると、AI は文字列を期待するパラメータに数値を渡すような基本的なミスを避けられます。

```typescript
// TypeScript: AI が型制約を理解して正確なコードを生成できる
async function processPayment(
  amount: number,
  currency: 'JPY' | 'USD' | 'EUR',
  userId: string
): Promise<PaymentResult> {
  // AI は currency が必ず 3 種類のいずれかであることを把握している
}

// JavaScript: AI が currency に 'YEN' や 'jpy' を渡す可能性がある
function processPayment(amount, currency, userId) {
  // 型情報なし → AI の推測に頼るしかない
}
```

TypeScript が理論的に優れているという話ではありません。AI ツールがより多くのシグナルを受け取るとより良い出力を生成するという、実用的な観察です。

## 段階的な移行アプローチ

TypeScript への移行は一度に全部やる必要はありません。段階的なアプローチが現実的です。

### フェーズ 1：設定の追加（1〜2日）

```bash
npm install -D typescript @types/node
npx tsc --init
```

`tsconfig.json` の初期設定：

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": false,
    "allowJs": true,
    "checkJs": false,
    "outDir": "./dist"
  },
  "include": ["src/**/*"]
}
```

`allowJs: true` により、既存の `.js` ファイルはそのまま残しながら、新規ファイルから TypeScript で書き始められます。

### フェーズ 2：新規ファイルを TypeScript で書く（継続的）

新しい機能やモジュールは最初から `.ts` で書きます。既存ファイルの移行は優先度が低いタスクとして扱います。

### フェーズ 3：strict モードへの移行（3〜6 ヶ月）

`"strict": true` に変更し、エラーを順次修正します。

```bash
# strict 有効化後のエラー数を確認
npx tsc --noEmit --strict 2>&1 | grep "error TS" | wc -l

# エラーの多いファイルを特定
npx tsc --noEmit --strict 2>&1 | grep "error TS" \
  | awk -F'(' '{print $1}' \
  | sort | uniq -c | sort -rn \
  | head -20
```

## AI Coding Tools との組み合わせで変わること

TypeScript プロジェクトで GitHub Copilot や Claude Code を使うと、体感できる違いがあります。

**補完精度の向上**：型情報があると、AI は引数の型・戻り値・nullable の扱いを正確に推測できます。「それらしいが間違っている」コードの頻度が下がります。

**リファクタリングの安全性**：AI が提案する変更を型チェッカーが自動的に検証するため、「テストを実行して初めてエラーがわかる」状態から「実行前にエラーが見える」状態に変わります。

**オンボーディング効率**：新しい開発者（と AI アシスタント）が、すべての境界で型コントラクトが明示的なコードベースをより速く把握できます。

## TypeScript が適さないケース

バランスのとれた視点のために補足します。TypeScript が常に答えとは限りません。

- **使い捨てスクリプト**：一度実行して削除されるスクリプトには型設定のコストが不要です
- **非常に小さなプロジェクト**：200行程度のユーティリティに完全な TypeScript ツールチェーンは過剰です
- **活発に開発されていない安定した JavaScript コードベース**：移行のための移行は避けてください

TypeScript が最も効果的なのは、成長するプロジェクト、複数人が保守するコード、AI 支援開発の恩恵を受けるコードです。多くの本番 Web アプリケーションがここに該当します。

## チームへの実践的なアドバイス

TypeScript を採用していないチームにとって、今が移行を検討する好機です。

1. **採用市場への影響**：新しい開発者の多くが TypeScript に慣れた状態で入ってきます。JavaScript のみのプロジェクトは適応コストが発生します。

2. **AI ツールの投資効率**：Copilot や Claude Code などへの投資が、TypeScript 環境でより大きなリターンをもたらします。

3. **長期的なメンテナンスコスト**：型情報はコードベースが大きくなるほどドキュメントとして機能します。

## まとめ

TypeScript が GitHub 首位に立ったことは、技術的な優位性と AI ツールの普及が重なった結果です。「いつか移行しよう」を先送りにするほど、移行コストは上がっていきます。

まず `allowJs: true` で新規ファイルだけ TypeScript で書き始める小さな一歩が、チームにとって最も低リスクな出発点になります。
