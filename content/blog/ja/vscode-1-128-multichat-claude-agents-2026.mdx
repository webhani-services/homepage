## VS Code 1.128が届いた

2026年7月8日、Visual Studio Code 1.128がリリースされました。今回の目玉のひとつが、Claudeエージェントセッション内でのマルチチャット対応です。1つのセッションの中に複数の関連チャットを持たせ、アプローチを比較したり、途中のターンから分岐したり、並行して作業を走らせたりできるようになりました。

一見すると地味なUIの改善に見えますが、AIコーディングの実務では「文脈の分岐」を扱えるかどうかが効率を大きく左右します。本稿では、この機能が何を解決するのか、Webhaniの視点で整理します。

## マルチチャットが解決する問題

これまでのAIコーディングでは、1つの会話に文脈が積み上がっていきました。便利な反面、次のような場面で扱いにくさがありました。

- 実装Aと実装Bを見比べたいが、同じ会話で試すと文脈が混ざる
- ある時点まで戻って別の方針を試したいが、それまでの経緯は保ちたい
- テストの作成と本体の実装を同時に進めたい

VS Code 1.128のマルチチャットは、これらを「1つのセッションに複数のチャットをぶら下げる」構造で解決します。各チャットは独自の履歴・タイトル・モデル選択を保持し、再起動後も親セッションとともに復元されます。ピアチャットはセッションの下にまとまり、トップレベルのセッション一覧を散らかしません。

## 3つの操作モデル

公式の説明にもとづくと、主な操作は次の3つです。

### 1. ピアチャットの追加

同じセッションに並列のチャットを足せます。たとえばメインのチャットでExpressアプリに `/health` エンドポイントを追加しながら、ピアチャットで同時にそのテストを書く、といった分担ができます。

```text
Claude セッション: user-service の改善
├─ メインチャット   : /health エンドポイントを追加
├─ ピアチャット #1  : 上記のテストを並行して作成
└─ ピアチャット #2  : 代替実装（別方式）を検討
```

### 2. フォーク（分岐）

会話の途中のターンから分岐できます。1.128では、フォークはトップレベルの新しいセッションではなく、同じセッション内の新しいピアチャットとして作られるようになりました。フォークしたチャットは分岐点までの会話を引き継ぎ、兄弟チャットとは独立して動き、タイトルは自動生成されます。

「ここまでの理解は共有したいが、この先は別方針で」という場面に噛み合います。

### 3. 並行実行（同時ターン送信）

複数のチャットにターンを同時に送れます。人間が1つずつ返信を待つのではなく、並行して複数の作業を進められるため、待ち時間が減ります。

## 実務での使いどころ

Webhaniが顧客プロジェクトでAIコーディングを使う際、この機能が効きそうな場面を挙げます。

- **実装と検証の分離**：本体実装のチャットと、テスト・レビュー観点のチャットを分けることで、文脈の混線を避けられます。
- **設計の比較検討**：同じ要件に対して2つのアプローチを別チャットで並走させ、出力を突き合わせて選ぶ、という進め方が自然になります。
- **調査と実装の両立**：ライブラリの挙動を調べるチャットと、実装を進めるチャットを分けておくと、調査で会話が長くなっても実装側の文脈が汚れません。

一方で注意点もあります。並列にすればするほど、レビューすべき出力の総量は増えます。AIが並行して生成したコードを、人間が一つずつ検証しきれるかは別問題です。Webhaniでは「並列で速く作る」ことと「検証の目を通す」ことをセットで設計することを推奨しています。並列化はスループットを上げますが、品質ゲートを緩めてよい理由にはなりません。

## 運用面での考え方

チームで使う場合、次の点を事前に決めておくと混乱が減ります。

1. **チャットの命名規則**：自動生成タイトルは便利ですが、後から履歴を追う際にはチーム内で「何のためのチャットか」がわかる命名を意識すると振り返りが楽になります。
2. **モデルの使い分け**：各チャットが独自にモデルを選べるため、軽い調査は速く安いモデル、難しい実装は上位モデル、という使い分けが可能です。コストと品質のバランスを設計に組み込めます。
3. **検証の担当を明確に**：並列で走らせた分、最終的に人間がレビューする責任範囲を曖昧にしないこと。生成量が増えるほど、この線引きが重要になります。

## まとめ

VS Code 1.128のマルチチャットは、AIコーディングを「1本の会話」から「分岐・並列できる作業空間」へと広げる変更です。フォークが同一セッション内のピアチャットになったことで、文脈を保ちながら方針を分けやすくなりました。

ただし、並列化そのものが成果を保証するわけではありません。速く作れるようになるほど、検証の設計が効いてきます。Webhaniでは、こうしたツールの進化を追いながら、「並列で作り、確実に検証する」ワークフローの設計を顧客に提案していきます。
