## 전환점이 왔습니다

지난 몇 년간 AI를 사용한 코딩 어시스턴트는 하나의 형태로 정착했습니다. 에디터 배경에서 모델이 작동하고, 타이핑하면 다음 줄을 제안합니다. 받아들이거나 거절합니다. 모든 것은 실시간이고 로컬 컨텍스트 내에서 완결됩니다.

그 시대는 끝났습니다.

2026년 중반, 모든 주요 AI 연구소가 두 가지를 배포하고 있습니다. 모델과 그 모델을 제어하는 *에이전트 하네스*입니다. 이 하네스는 범위가 정해진 작업을 AI 시스템에 위임하고, 어떤 도구에 액세스를 허용할지 관리하며, 수 시간에서 수 일에 걸쳐 실행하고, 출력을 검토하고 반복하기 위한 런타임 환경입니다. 이는 xAI의 Grok Build (7월 8일 출시), Claude Code의 7월 업데이트, 그리고 OpenAI, Google 등 각사에 공통적으로 나타나는 패턴입니다. AI를 사용한 코딩은 더 이상 "인라인 제안"이 아니라 "워크플로우"가 되었습니다.

이것은 점진적인 개선이 아닙니다. 사고 방식 자체가 변하고 있습니다. "어시스턴트가 코딩을 빠르게 하는 것을 돕는다"에서 "범위가 정해진 작업을 어시스턴트에 위임하고, 출력을 검토하고, 도구에 대한 액세스를 승인하거나 거부하고, 반복한다"로의 변화입니다.

## 에이전트 하네스의 실제 구성

에이전트 하네스는 네 가지 기본 요소로 구성됩니다.

**1. 지속적인 컨텍스트와 세션 상태.** 에디터를 닫으면 잊어버리는 자동완성과 달리, 에이전트 하네스는 재시작 후에도 대화 기록, 작업 기록, 실행 결과를 유지합니다. 며칠 후에 작업으로 돌아와도 에이전트는 전체 컨텍스트를 갖고 있습니다.

**2. 도구 액세스와 permission 게이트.** 에이전트는 코드를 제안하는 것뿐만 아니라 파일을 읽고 쓰고, 명령을 실행하고, API를 호출하고, 코드베이스를 검사할 수 있습니다. 하지만 명시적으로 허용한 작업만 수행합니다. permission 매트릭스를 제어합니다. "소스 파일은 읽을 수 있지만 secrets는 읽을 수 없다", "변경을 제안할 수 있지만 테스트만", "빌드 명령은 실행할 수 있지만 배포는 할 수 없다" 같은 식입니다.

**3. 승인 워크플로우.** 중요한 작업은 게이트를 통합니다. 에이전트가 프로덕션 파일을 수정하거나, 배포를 실행하거나, 코드를 삭제하기 전에 당신의 검토를 기다립니다. 이는 자동완성이 통제 불능이 되는 것이 아니라 주니어 엔지니어가 커밋 전에 허가를 요청하는 것과 같습니다.

**4. 병렬 실행과 분기.** 많은 하네스에서는 동일한 작업에서 여러 워커를 실행하고, 독립적인 조사를 병렬로 실행하고, 언제든지 대화를 분기할 수 있습니다. 이는 처리량을 증가시키면서도 human-in-the-loop을 유지합니다.

기술적으로 에이전트 하네스는 샌드박스(보통 컨테이너), 도구 실행을 제어하는 엔진, 감사 추적(audit trail), 그리고 작업별로 permission을 승인하거나 거부하거나 수정할 수 있는 UI를 제공합니다.

## "자동완성의 진화" 아님을 증명하는 예

현실적인 워크플로우를 생각해보세요. 월요일 아침, 마이크로서비스에 새로운 API endpoint를 추가하고, 테스트를 작성하고, 문서를 업데이트하고, PR 초안을 내야 합니다.

**구 모델(자동완성):** 파일을 열고, 어시스턴트가 완성을 제안하고, 타이핑하고, 제안을 받아들입니다. 각 파일에서 반복합니다. 어시스턴트는 테스트를 작성해야 한다거나 문서 업데이트가 필요하다는 것을 알 방법이 없습니다. 모든 조율을 직접 합니다. 제안은 "다음에 올 것"에 대한 최선의 추측일 뿐입니다.

**신 모델(에이전트 하네스):** 작업을 설명합니다. "/health endpoint를 추가해주세요. 이것은 서비스 상태를 반환하고 데이터베이스 연결을 확인합니다. 테스트도 작성해주세요. README도 업데이트해주세요." 에이전트는 다음을 수행합니다.
- 당신의 API 코드를 읽어 패턴을 이해합니다.
- 기존 테스트를 확인해 스타일을 맞춥니다.
- endpoint 구현을 제안합니다.
- 당신이 제안을 검토합니다. 코드 작성 permission을 부여합니다.
- 에이전트가 endpoint, 테스트 파일, README를 작성합니다.
- 테스트를 실행해 통과하는지 확인합니다.
- draft PR을 생성하고 검토를 기다립니다.

당신은 각 게이트에서 제어권을 유지했습니다. 에이전트는 매번 permission을 요청했고, 당신은 커밋 전에 결과를 봤습니다. 또한 context switching을 줄이면서 더 많은 작업을 완료했습니다.

차이는 속도가 아니라 *범위*와 *보호장치 내에서의 자율성*입니다. 자동완성은 line-by-line입니다. 에이전트 하네스는 task-by-task입니다.

## 실무 도입: 작게 시작해서 permission을 단계적으로 부여하기

에이전트 하네스 도입 시의 유혹은 모든 permission을 부여하고 에이전트의 작동을 관찰하는 것입니다. 이것은 실수입니다. 클라이언트 팀이 이렇게 하는 것을 봤고 후회했습니다.

더 안전한 경로는 다음과 같습니다.

**Phase 1: read-only 탐색.** 에이전트에게 코드베이스를 읽게 하고, 구조를 이해하게 하고, 질문하게 하고, 변경을 제안하게 합니다. 하지만 아무것도 쓰게 하지 마세요. 이것은 조사 단계입니다. 에이전트가 당신의 프로젝트에 대해 무엇을 이해하고 있는지, 무엇을 오해하고 있는지 배웁니다.

**Phase 2: sandboxed 작성.** 다음으로 에이전트가 코드를 작성하도록 하되, 새로운 파일이나 branch, test 디렉토리에만 국한합니다. test suite를 실행해서 출력을 검증합니다. 아직 프로덕션 코드는 건드리지 못하게 합니다.

**Phase 3: feature branch에 대한 gated write.** 에이전트의 출력을 신뢰할 수 있게 되면, 새로운 기능용 feature branch에 대한 작성을 허용합니다. 모든 작성에는 여전히 당신의 승인이 필요합니다. 모든 변경은 일반적인 PR review 프로세스를 통과합니다.

**Phase 4: scoped 프로덕션 업데이트.** 에이전트가 신뢰성을 입증했다면, 좁은 게이트를 엽니다. 예를 들어 "문서를 업데이트할 수 있다" 또는 "기존 함수의 테스트는 작성할 수 있다"이지만, 그 외의 작업은 못하게 하는 식입니다. 광범위한 write access는 부여하지 마세요.

**Phase 5: 모니터링된 autonomous 작업.** 많은 성공 사이클 후에야 비로소 잘 범위가 정해진 작업에서 에이전트를 unattended로 실행하는 것을 고려할 수 있습니다. 이 경우에도 audit trail을 유지하고 시간 제한을 설정하세요(예: "이 작업은 1시간 내에 완료되어야 한다").

## Governance와 검토

AI 에이전트에게 permission을 부여하는 것은 "설정하고 잊기"가 아닙니다. 당신의 팀은 몇 가지 정책이 필요합니다.

- **승인 단계.** 어떤 작업이 human review를 필요로 하는지 결정합니다. 예: 읽기 및 변경 제안은 자유; branch에 대한 작성은 human 승인 필요; 프로덕션 배포는 두 명의 human 필요. 이를 문서화합니다.
- **감사 추적.** 에이전트 하네스는 취한 모든 action, 사용한 permission, 발생한 일을 기록해야 합니다. 이 로그를 정기적으로 검토합니다.
- **Model과 capability 제한.** 다양한 작업은 다양한 모델을 필요로 할 수 있습니다. 가볍고 빠른 모델은 refactoring에 충분합니다. 더 강한 모델은 architectural 변경에 필요할 수 있습니다. 이를 명시적으로 합니다.
- **Rollback 절차.** 에이전트가 실수를 하면(할 것입니다), 이를 취소하는 명확한 방법이 필요합니다. branch를 revert하거나, 파일을 복원하거나, 배포를 취소합니다. 필요하기 전에 연습합니다.
- **Escalation path.** 일부 결정은 에이전트에게 위임해서는 안 됩니다. 에이전트는 이를 인식하고 도움을 요청해야 합니다(예: "이 변경은 데이터베이스 스키마를 backward-incompatible 방식으로 수정합니다. human input이 필요합니다").

## 팀용 체크리스트

에이전트 하네스를 도입하는 backend 팀을 위한 permission 계층 매트릭스의 구체적인 예입니다.

| 작업 | Permission Level | 검토 필요 |
|-----|------------------|---------|
| 소스 파일 읽기 | 항상 허용 | 아니오 |
| 설정 파일 읽기 | 항상 허용 | 아니오 |
| 변경 제안(작성 없음) | 항상 허용 | 아니오 |
| 테스트 파일 작성 | 허용 | 아니오 |
| 문서 작성 | 허용 | 아니오 |
| feature branch 작성 | 허용 | 예(push 전) |
| CI/CD config 수정 | 거부 | — |
| staging 배포 | 거부 | — |
| 프로덕션 배포 | 거부 | — |
| secrets/credentials 액세스 | 거부 | — |

대부분이 "Denied"과 검토 "아니오" 상태로 시작합니다. 수주에 걸쳐 에이전트 출력의 품질과 신뢰성을 관찰한 후에야 확대합니다.

## Webhani의 관점

Webhani는 2024년 초부터 AI/LLM 도입을 클라이언트에게 조언해왔습니다. 에이전트 하네스에서 반복적으로 발생하는 3가지 실수를 봤습니다.

1. **너무 빨리 permission을 많이 부여하기.** 팀은 "AI 마법"을 보고 싶어서 모든 게이트를 열어줍니다. 에이전트가 실수를 하면(할 것입니다), 복구는 고통스럽습니다. 좁게 시작하세요.

2. **검토 루프 건너뛰기.** 팀은 모델이 만들었다는 이유만으로 에이전트 생성 코드를 신뢰할 수 있는 것으로 취급합니다. 우리의 권장: junior 엔지니어 코드처럼 에이전트 출력을 검토하세요. 처음에는 철저히 검토합니다. 신뢰를 단계적으로 구축합니다.

3. **실제 이점 측정 안 하기.** AI 에이전트를 워크플로우에 추가하면 overhead가 생깁니다. 여전히 변경을 검토하고, 하네스를 유지보수하고, permission을 최신 상태로 유지합니다. 측정하세요: 시간을 절약하고 있나요? 버그를 줄이고 있나요? 팀을 더 높은 leverage 작업으로 해방하고 있나요? 한 달 후 답이 불명확하다면 범위를 재평가하세요.

클라이언트 회사에서 가장 성공한 도입은 이 패턴을 따릅니다. 하나의 작고 범위가 잘 정해진 작업을 특정합니다(예: "이 모듈의 테스트를 작성한다" 또는 "API 문서를 자동 생성한다"), 위의 단계적 permission 모델을 거쳐서 실행하고, 시간 절약과 품질 영향을 측정한 다음, 다음 작업으로 확대합니다. "초일부터 완전한 자율성"이 잘 작동하는 경우는 거의 없습니다.

## 미래 전망

에이전트 하네스는 종착점이 아닙니다. 도구는 더 스마트해지고, 모델은 개선되고, UX는 성숙합니다. 하지만 근본적인 변화——자동완성에서 위임된, permission-gated 작업 실행으로——는 업계에 내장되어 있습니다.

도입을 고려 중인 팀에게 조언은 간단합니다. 작게 시작하고, 신뢰하되 검증하고, 측정하세요. 에이전트 하네스는 강력하지만, 그 강력함은 human을 루프에 유지하고 permission에 대해 신중하게 생각할 때만 안전합니다.
