## 숫자가 보여주는 모순

2026년, 개발자의 84%가 AI 코딩 툴을 사용합니다. AI가 생성하는 코드는 전체 커밋 코드의 41%를 차지하며, 2027년에는 65%에 달할 것으로 전망됩니다. 시장의 내러티브는 생산성 혁명입니다.

그런데 METR 연구가 있습니다.

2025년 7월에 발표된 이 연구는 경험 많은 오픈소스 개발자 16명이 자신의 프로젝트(평균 22,000 GitHub 스타, 100만 줄 이상의 코드)에서 246개의 실제 작업을 수행하는 방식으로 진행되었습니다. 결과는 **AI 툴을 사용한 개발자들이 그렇지 않은 개발자보다 작업 완료에 19% 더 오래 걸렸다**는 것이었습니다.

더 흥미로운 것은 인식의 격차입니다. 연구 후 같은 개발자들은 AI가 자신을 평균 20% 빠르게 만들었다고 응답했습니다. 실제로는 더 느려졌는데 더 빨라졌다고 느낀 것입니다.

## 왜 실제로는 느려지는가

인식과 성과 사이의 불일치에는 몇 가지 확인 가능한 원인이 있습니다.

**Prompt 반복 작업의 비용**

AI와 작업하려면 반복적인 Prompt 조정이 필요합니다. 그럴듯한 출력이 아닌 실제로 필요한 출력을 얻기 위한 지정-수신-평가-재지정 사이클에는 시간이 듭니다. 복잡한 작업일수록 이 반복 비용이 구현 자체에서 절약되는 시간을 초과할 수 있습니다.

**검증 부하**

AI가 생성한 코드는 검토 없이 Merge할 수 없습니다. 별도 데이터에 따르면 AI 생성 코드에는 사람이 작성한 코드보다 2.74배 더 많은 보안 취약점이 포함되어 있습니다. 코드 작성 시간 절약분이 검증 시간으로 상쇄됩니다. 보안에 민감한 코드 경로에서는 이 상쇄가 완전할 수 있습니다.

**자동화 편향**

AI 출력을 권위 있고 완성된 것처럼 보이기 때문에 줄어든 주의력으로 수용하는 심리적 경향이 있습니다. *그럴듯하게 보이는* 코드와 *올바른* 코드는 다른 것이지만, 시각적 유사성 때문에 Review에서 구분하기 쉽지 않습니다.

**도구 활용 습관 퇴화**

AI에 크게 의존하는 개발자들은 Debugger, Profiler, 전문 분석 도구를 덜 사용하는 경향이 있습니다. 장기적으로 AI가 아직 대체할 수 없는 디버깅과 진단 기술의 격차가 생깁니다.

## 2026년 2월 업데이트가 보여주는 변화

METR은 2026년 2월에 후속 연구를 발표했는데, 새로운 방법론적 문제가 드러났습니다. 개발자들이 이제 AI 툴에 너무 의존해서 **AI가 금지된 Control Group 참여를 거부**하고 있습니다.

이것은 중요한 신호입니다. AI 툴이 "유용한 추가물"에서 점점 더 많은 개발자에게 "전제 조건"으로 전환되고 있습니다. AI 없이 생산성을 측정하는 것이 인터넷 없이 생산성을 측정하는 것과 유사해지고 있습니다.

METR의 평가는 개발자들이 2025년 초 대비 2025년 말 AI 툴로 *더 빨라졌을 가능성이 높다*는 것이지만, 이제 실험 설계로는 그것을 명확히 측정할 수 없습니다.

## 효과를 내는 팀의 특징

METR 연구에는 비판도 있습니다. 실제 개발 환경은 실험실 조건과 다른 요소가 많습니다. Deloitte의 2026년 소프트웨어 산업 전망은 제대로 적용된 AI로 30~35% 생산성 향상을 예측합니다. "제대로 적용된"과 "습관적으로 채택된" 사이의 격차가 METR 수치가 사는 곳입니다.

실제로 생산성 향상을 보고하는 팀들은 공통적인 패턴을 보입니다.

**작업 유형별 구분**

AI는 잘 정의된 패턴 중심의 작업에서 일관된 속도 향상을 제공합니다.

```
AI 고효율 작업:
- 확립된 패턴 내 Boilerplate 생성
- 기존 구현에서 Test 케이스 생성
- Library API 사용법 및 문서 조회
- Refactoring 후보 제안

AI 저효율 작업:
- 중요한 Trade-off가 있는 Architecture 결정
- 사람의 명확화가 필요한 모호한 요구사항
- 깊은 도메인 추론이 필요한 Security 로직
- 문서화되지 않은 동작과의 새로운 통합
```

**구조화된 검증**

AI 출력을 서명이 필요한 승인된 코드가 아닌 검토되지 않은 코드로 취급하는 팀은 문제가 쌓이기 전에 잡습니다.

```typescript
// AI 생성 코드에서 확인해야 할 일반적인 문제:

// 1. Null/오류 처리 누락
async function getUser(id: string) {
  const user = await db.users.findOne({ id });
  // AI는 종종 누락: user가 null이면 어떻게 되는가?
  if (!user) throw new NotFoundError(`User ${id} not found`);
  return user;
}

// 2. 변환 과정에서 데이터 손실
// 3. 설정 가능해야 할 하드코딩된 값
// 4. 경계 함수에서 입력 검증 누락
// 5. 로깅 없는 포괄적 오류 처리
```

**AI 출력을 입력으로 사용하기**

가장 효과적인 팀은 AI 생성 코드를 승인할 최종 결과물이 아닌 생각의 출발점 초안으로 취급합니다. 질문이 "이게 맞아 보이는가?"에서 "내가 다르게 작성했을 부분은 어디이고, 왜인가?"로 바뀝니다.

## 의존성 신호

2026년 업데이트의 방법론적 문제 — AI 없이 일하기를 거부하는 개발자들 — 는 생각해볼 가치가 있습니다. AI 이전과 이후를 측정하는 것이 Google 이전과 이후를 측정하는 것만큼 무의미해지고 있습니다.

이는 Engineering 조직이 역량 개발에 대해 어떻게 생각해야 하는지에 영향을 미칩니다. 비판적 추론, 보안 판단, Architecture 사고가 AI 위임으로 퇴화하는 기술이고, 동시에 그 기술들이 AI 출력을 유용하게 검토하는 데 필요하다면, 이 기술을 의도적으로 유지하는 팀이 복리 효과를 얻습니다.

## Webhani의 시각

METR 연구는 AI 툴을 사용하지 말라는 주장이 아닙니다. 습관이 아닌 의도를 가지고 사용하라는 주장입니다.

Workflow 재설계 없는 채택이 19% 감속을 만드는 것입니다. METR 연구의 개발자들은 AI를 사용할 수 있고 생산적으로 느껴지기 때문에 사용했습니다. 어떤 작업을 위임하고 결과를 어떻게 검증할지에 대한 의도적인 프레임워크를 가지고 있지 않았습니다.

Webhani에서는 AI 툴 채택을 도구 선택 문제만큼이나 Workflow 설계 문제로 취급합니다. 어떤 작업을 위임하는지, 출력물을 어떻게 검증하는지, 팀이 검증을 의미있게 만드는 추론 기술을 어떻게 유지하는지 — 이것들은 엔지니어링 결정입니다.

84% 채택률은 대화가 "AI를 사용해야 하는가"에서 "실제로 도움이 되는 방식으로 사용하고 있는가"로 이동했음을 의미합니다.
