Belitsoft의 최신 조사에 따르면 2026년 현재 React 개발자가 작성하는 코드의 42%가 AI 툴로 생성되며, 이 비율은 2027년 말까지 65%에 이를 것으로 전망됩니다. 또한 개발자의 90% 이상이 업무에서 AI 코딩 툴을 정기적으로 사용하고 있습니다. AI는 이제 "보조 도구"가 아닌 개발 워크플로우의 핵심 요소로 자리 잡았습니다.

## 실제 현장에서 달라진 점

GitHub Copilot, Claude Code, Cursor 같은 툴이 개발 현장에 깊이 침투했습니다. 모든 작업에 AI가 효과적인 것은 아니지만, 특정 영역에서는 생산성이 크게 향상됩니다.

### Boilerplate 및 Scaffold 자동 생성

타입이 있는 컴포넌트 Scaffold, API Client 코드, 폼 유효성 검사 로직 생성에서 AI는 특히 강력합니다.

```tsx
interface Column<T> {
  key: keyof T;
  header: string;
  render?: (value: T[keyof T], row: T) => React.ReactNode;
}

interface DataTableProps<T> {
  data: T[];
  columns: Column<T>[];
  pageSize?: number;
}

export function DataTable<T extends { id: string | number }>({
  data,
  columns,
  pageSize = 20,
}: DataTableProps<T>) {
  const [page, setPage] = useState(0);
  const total = Math.ceil(data.length / pageSize);
  const rows = data.slice(page * pageSize, (page + 1) * pageSize);

  return (
    <div className="overflow-x-auto">
      <table className="min-w-full text-sm">
        <thead>
          <tr>
            {columns.map((col) => (
              <th key={String(col.key)} className="px-4 py-2 text-left font-medium">
                {col.header}
              </th>
            ))}
          </tr>
        </thead>
        <tbody>
          {rows.map((row) => (
            <tr key={row.id}>
              {columns.map((col) => (
                <td key={String(col.key)} className="px-4 py-2">
                  {col.render ? col.render(row[col.key], row) : String(row[col.key])}
                </td>
              ))}
            </tr>
          ))}
        </tbody>
      </table>
    </div>
  );
}
```

이 코드는 프롬프트 몇 줄로 생성됩니다. 중요한 것은 이 설계가 우리 아키텍처에 맞는지 판단하는 것 — 이 부분은 여전히 사람이 해야 합니다.

### 구현과 Test를 동시에 생성

구현 코드와 Test를 동시에 생성하는 것이 자연스러운 워크플로우가 되었습니다.

```typescript
import { render, screen } from "@testing-library/react";
import userEvent from "@testing-library/user-event";

test("모든 행이 표시됩니다", () => {
  const data = [
    { id: 1, name: "Alice", role: "Engineer" },
    { id: 2, name: "Bob", role: "Designer" },
  ];
  const columns = [
    { key: "name" as const, header: "이름" },
    { key: "role" as const, header: "역할" },
  ];
  render(<DataTable data={data} columns={columns} />);
  expect(screen.getByText("Alice")).toBeInTheDocument();
  expect(screen.getByText("Bob")).toBeInTheDocument();
});

test("다음 페이지로 이동합니다", async () => {
  const data = Array.from({ length: 25 }, (_, i) => ({
    id: i,
    name: `사용자 ${i}`,
    role: "Dev",
  }));
  render(
    <DataTable
      data={data}
      columns={[{ key: "name", header: "이름" }]}
      pageSize={10}
    />
  );

  expect(screen.getByText("1 / 3")).toBeInTheDocument();
  await userEvent.click(screen.getByRole("button", { name: /다음/i }));
  expect(screen.getByText("2 / 3")).toBeInTheDocument();
});
```

구현과 Test를 하나의 프롬프트로 생성하면, TDD처럼 동작을 먼저 생각하게 됩니다. 이는 부수적인 효과지만 품질에 긍정적입니다.

### Code Review의 초점 변화

Boilerplate가 AI로 생성되면, Reviewer는 문법보다 설계 적합성과 비즈니스 로직의 정확성에 집중하게 됩니다. 이 변화는 긍정적이지만, AI가 자주 놓치는 미묘한 문제에 더 주의를 기울여야 합니다.

## AI 생성 코드의 주요 위험 요소

| 위험 요소 | 예시 | 대응 방법 |
|-----------|------|-----------|
| 보안 취약점 | 미이스케이프 사용자 입력, CSRF 누락 | 보안 체크리스트 유지 |
| 성능 문제 | N+1 Query, 불필요한 Re-render | 배포 전 Profile |
| 과도한 추상화 | useState 하나를 감싸는 Hook | YAGNI 원칙 적용 |
| 구식 패턴 | Deprecated API 사용 | 프롬프트에 Library 버전 명시 |

## 실용적인 권장 사항

팀에서 AI 툴 사용을 표준화할 때 고려할 점:

- **프롬프트에 제약 조건을 명시합니다**: "TypeScript strict mode", "Zod 유효성 검사", "외부 의존성 없음"
- **항상 Error 처리 경로를 Review합니다**: AI는 Happy Path를 중심으로 최적화하는 경향이 있습니다
- **구현과 Test를 함께 생성합니다**: Test 생성을 별도 단계로 두지 않습니다
- **아키텍처 제약을 문서화합니다**: CLAUDE.md나 설정 파일에 규칙을 명시하면 AI 생성 코드가 프로젝트 패턴에 맞게 생성됩니다

42%라는 수치는 중요하지만, AI를 잘 활용하는 팀과 그렇지 않은 팀의 품질 격차는 점점 커지고 있습니다. 핵심은 어떤 코드가 Merge될지에 대한 사람의 판단을 유지하는 것입니다.
