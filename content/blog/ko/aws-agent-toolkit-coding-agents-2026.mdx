AI Coding Agent는 일반적인 소프트웨어 엔지니어링에서 효과적입니다. 버그를 수정하고, 테스트를 추가하고, 코드를 리팩토링합니다. 하지만 AWS 인프라 환경에 그대로 배포하면, 실행은 되지만 본프로덕션 수준이 아닌 코드를 생성합니다.

문제는 Agent가 AWS 코드를 쓸 수 없다는 게 아닙니다. AWS API의 깊이와 폭입니다. DynamoDB 스캔을 해야 한다는 지시를 받으면, Agent는 스캔 API를 반복 호출합니다. 코드는 작동합니다—테이블이 백만 개 행을 넘을 때까지. 그다음부턴 응용프로그램이 조용히 불완전한 결과를 반환합니다. 페이지네이션을 모르기 때문입니다.

AWS Agent Toolkit for AWS(2026년 출시, AWS Labs MCP 서버의 후속)는 이 문제를 SDK 베스트프랙티스를 온디맨드로 제공함으로써 해결합니다. 현재 작업과 관련 있는 스킬만 로드되므로 Agent의 토큰 윈도우는 효율적이면서도, 본프로덕션급 코드가 나옵니다.

## 세 가지 계층

Toolkit은 계층 구조를 가지고 있습니다.

**Agent Skills** — 언어별 지시, 코드 예제, 참고 자료를 담은 큐레이션된 패키지. DynamoDB Pagination 스킬은 Agent에게 "수동 루프 대신 페이지네이터를 써라"고 가르칩니다. Python Async 스킬은 asyncio 이벤트 루프 소유권과 `gather` vs `wait` 의미를 다룹니다. 스킬이 설치되면 Agent는 추측하지 않고 올바른 패턴을 압니다.

**AWS MCP Server** — 모든 AWS 서비스를 호출할 수 있는 완전 관리형 MCP 서버. 중요한 점은 IAM 기반 가드레일을 존중합니다. "이 작업은 S3 읽기 전용 권한만 갖는다"고 정의하면, MCP 서버가 강제합니다. CloudWatch와 CloudTrail이 모든 작업을 기록하므로 Agent가 뭘 건드렸는지 감시할 수 있습니다. 다단계 작업은 샌드박스 실행되므로 실패 시 정확히 어디서 멈췄는지 봅니다.

**Plugins** — AWS Core (풀스택 앱), AWS Data Analytics (데이터 파이프라인), AWS Agents (Amazon Bedrock AgentCore에서 Agent 구축).

Toolkit은 20개 이상의 스킬을 포함하며 가장 흔한 SDK 함정을 다룹니다. 핵심: Agent가 자동으로 모두 로드하지 않습니다. 대신 설정 파일(`.agentconfig` 등)에 사용 가능한 스킬을 나열합니다. Agent가 작업을 만나면 스킬 레지스트리를 검색해 관련 스킬을 내려받고 지식을 context에 포함합니다.

이는 단순 기능이 아닌 context-engineering 패턴입니다. 스킬 온디맨드는 토큰 예산을 보존하면서도 Agent가 정확한 코드를 작성하는 데 필요한 지식을 정확히 전달합니다.

## 문제: DynamoDB 사례

교육받지 않은 Agent가 만드는 코드입니다.

```javascript
// ❌ 페이지네이션 없음 — 최대 1MB 또는 아이템 제한만 반환
const params = {
  TableName: 'Orders'
};
const response = await dynamodb.scan(params).promise();
console.log(response.Items.length); // 불완전할 수 있음
```

Agent는 scan이 아이템을 반환한다는 것만 알고 있습니다. DynamoDB 스캔이 페이지네이션된다는 사실은 모릅니다. 테이블이 충분히 크면 조용히 실패합니다.

DynamoDB Pagination 스킬을 로드한 같은 Agent:

```javascript
// ✓ 페이지네이터 API — 페이지네이션을 투명하게 처리
const { DynamoDBClient, ScanCommand } = require('@aws-sdk/client-dynamodb');
const { paginateScans } = require('@aws-sdk/lib-dynamodb');

const client = new DynamoDBClient({});
const paginator = paginateScans({ client }, { TableName: 'Orders' });

let allItems = [];
for await (const page of paginator) {
  allItems = allItems.concat(page.Items || []);
}
console.log(allItems.length); // 완전함
```

페이지네이터는 `ExclusiveStartKey`, 재시도 로직, 상태 관리를 내부적으로 처리합니다. Agent는 스킬에서 "리스트 작업은 페이지네이터를 기본으로 써라"고 배웁니다.

## Python 비동기 패턴

RDS 인스턴스가 사용 가능해질 때까지 기다리기. 스킬 없을 때:

```python
# ❌ 수동 폴링 — sleep과 retry 로직을 다시 발명
import time
while True:
    response = rds_client.describe_db_instances(DBInstanceIdentifier='prod-db')
    status = response['DBInstances'][0]['DBInstanceStatus']
    if status == 'available':
        break
    print(f"Current status: {status}. Waiting...")
    time.sleep(30)
```

Python Async 스킬을 가지면:

```python
# ✓ Waiter API — SDK가 retry, backoff, timeout을 관리
waiter = rds_client.get_waiter('db_instance_available')
waiter.wait(DBInstanceIdentifier='prod-db')
```

Waiter는 지수 백오프, 최대 시도, 지연 로직을 캡슐화합니다. Agent는 폴링을 다시 발명하지 않고 SDK의 waiter를 사용하며, 이는 본프로덕션 사용으로 테스트되고 튜닝되었습니다.

## IAM 범위 지정과 MCP 관찰성

MCP 서버는 서비스 수준에서 IAM 권한을 강제합니다. Agent에 S3 읽기만 허용하는 IAM 역할을 할당합니다. MCP 서버가 모든 API 호출을 그 역할로 검사합니다. Agent가 DynamoDB에서 삭제하려 하면, AWS에 도달하기 전에 호출이 실패합니다.

이건 지침이 아닙니다. 아키텍처 보장입니다.

또한 모든 작업이 CloudTrail에 기록됩니다. Agent가 어느 API를 호출했고, 어느 리소스에 대해, 언제했는지 정확히 감시할 수 있습니다. 본프로덕션에서 필수입니다—자율 시스템이 뭘 했고 왜 했는지 알아야 합니다.

## 도입 체크리스트

- [ ] Agent Toolkit 설치 (`npm install @aws-sdk/agent-toolkit` 또는 동등)
- [ ] `.agentconfig` 생성 (예: `skills: ["dynamodb-pagination", "s3-transfer-manager", "lambda-async"]`)
- [ ] 작업 유형별 IAM 역할 설정 (읽기 전용, 쓰기 제한 등)
- [ ] MCP 서버를 로컬 또는 컨테이너에서 시작
- [ ] 낮은 위험 작업부터 시작: 보일러플레이트 생성, 구성 조회
- [ ] CloudTrail 감시 대시보드 설정

## 왜 이게 중요한가

클라우드 인프라에서 AI Agent의 병목은 코드 생성 속도가 아닙니다. SDK 베스트프랙티스를 따르고, 보안 경계를 존중하고, 비용 효율적인 리소스 선택을 하는 코드를 생성하는 것입니다. 페이지네이션을 빼먹거나 비동기 패턴을 안 쓰는 Agent는 본프로덕션에서 놀라운 일을 합니다.

스킬 온디맨드는 이를 해결하는 패턴입니다. 필요한 순간에 필요한 지식만 상황에 맞게 전달함으로써, Agent가 문제를 추론하는 능력을 유지하면서 올바른 AWS 프리미티브로 추론하도록 합니다.
