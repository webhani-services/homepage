## Release가 아니라 반복(Iteration)을 위해 만들어진 Deployment Mode

AWS는 2026년 6월 말 CloudFormation Express Mode를 발표했습니다. 이는 특정한 Pain Point를 겨냥한 것입니다. "Resource 설정이 완료된" 상태와 "CloudFormation이 해당 Resource의 완전한 안정성을 확인 완료한" 상태 사이의 간극은, 반복적인 Infrastructure 변경 작업에서 가장 느린 부분인 경우가 많습니다. 표준 CloudFormation Deployment는 성공을 보고하기 전에 긴 Stabilization Check — Load Balancer가 Health Check를 통과하는지, Database가 연결을 수락하는지, ECS Service가 Steady State에 도달했는지 — 를 기다립니다. Express Mode는 대신 CloudFormation이 Resource 설정 자체가 적용되었음을 확인하는 즉시 완료로 간주하며, AWS에 따르면 반복적인 Workflow에서 Deployment 시간을 최대 4배 단축할 수 있습니다. CloudFormation과 CDK 모두에서 모든 Commercial Region에 걸쳐 추가 비용 없이 사용할 수 있습니다.

AWS는 AI Agent Workflow를 명시적인 Target Use Case로 언급하고 있으며, 이 포지셔닝은 단순한 Marketing이 아니라 Infrastructure Tooling에서 "충분히 빠르다"는 것의 의미가 실질적으로 바뀌고 있다는 신호로 받아들일 가치가 있습니다.

## Stabilization Wait이 존재하는 이유 — 그리고 이를 생략하는 것의 진짜 Trade-off

표준 Mode의 Stabilization Check는 관료적인 Overhead가 아닙니다. Resource가 "생성됨"이라고 보고하는 것과 그 Resource가 실제로 Traffic을 처리할 준비가 된 것은 서로 다른 상태이기 때문에 존재합니다. RDS Instance는 존재하면서도 여전히 초기화 중일 수 있습니다. ECS Service는 Task 정의가 업데이트되었지만 여전히 이전 Task를 Drain하는 중일 수 있습니다. Stabilization을 기다리는 것이 CloudFormation이 의미 있는 "완료"를 알려주는 방식입니다.

Express Mode가 전제하는 것은, 반복적인 Development 중에는 모든 변경마다 그 보장이 필요하지 않다는 것입니다. 필요한 것은 설정이 수락되었음을 확인하고, Resource 식별자를 받아 다음 Step으로 넘어가는 것입니다. AWS는 이것이 Development 반복, 개별 Component Test, 그리고 하위 Stack이 ARN이나 Endpoint 참조만 필요로 하고 Traffic 준비 완료의 증명은 필요로 하지 않는 Dependent Stack Deployment에 적합하다고 명시합니다.

```bash
# 표준 Deploy — 완전한 Stabilization을 기다림
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name my-service-dev

# Express Mode — 설정 적용이 확인되는 즉시 반환
aws cloudformation deploy \
  --template-file template.yaml \
  --stack-name my-service-dev \
  --deployment-mode EXPRESS
```

이는 개발 Loop에서는 정당한 Trade-off지만, Health를 기준으로 Gating하는 Production Rollout에서는 위험한 Trade-off입니다. 기억해야 할 차이는, Express Mode가 알려주는 것은 "원하는 설정이 수락되었다"는 것이지 "System이 건강하다"는 것이 아니라는 점입니다.

## 실제로 도움이 되는 곳

**AI Agent가 주도하는 Infrastructure 변경**: CDK Stack을 반복하는 Agent — IAM Policy 조정, 환경 변수 추가, 새 Queue 연결 — 는 빠른 Confirm/Fail Loop에서 큰 이득을 얻습니다. 시도할 때마다 완전한 Stabilization을 몇 분씩 기다려야 한다면 Agent 주도의 Infra 작업은 현실적이지 않습니다. 빠른 "설정 수락" Signal이 있어야 Code를 반복하듯 Infra를 반복할 수 있습니다.

**Dependent Stack 연쇄**: Stack B가 Deploy를 시작하기 위해 Stack A의 출력(VPC ID, Security Group ARN 등)을 필요로 한다면, Stack A의 실제 Traffic 준비 상태는 Stack B의 Unblock 여부와 무관합니다. Express Mode는 정확성을 훼손하지 않으면서 식별자를 더 빠르게 얻게 해줍니다.

**Local 및 CI Development 환경**: 한 시간 만에 삭제되는 일회성 Dev 환경이나 Preview 환경은 실제 Traffic을 처리하는 Production Stack과 같은 수준의 Stabilization 보장이 필요하지 않습니다. Express Mode는 빠른 Create-Test-Destroy Cycle에 자연스럽게 맞습니다.

**의도적으로 쓰지 않을 곳**: Production Release를 Gating하는 Deployment, 실제 Health Check가 중요한 변경(이전 Task가 Drain되기 전에 새 Task 정의가 ALB Health Check를 통과할 수 있음을 증명해야 하는 경우), 또는 "Deployment 성공"이 Traffic 승격의 Signal로 쓰이는 Pipeline Stage. 이런 경우에는 표준 Mode의 Stabilization Wait이 실질적인 역할을 하고 있으므로, 속도를 위해 이를 생략해서는 안 됩니다.

## Deployment Pipeline에 어떻게 녹여낼 것인가

이 기능을 도입하는 팀에 권하고 싶은 Pattern은, 안쪽 Loop에는 Express Mode를, 바깥쪽 Gate에는 표준 Mode를 쓰는 것입니다.

1. **Local 및 PR Preview 환경**: 기본값으로 Express Mode. 몇 시간만 존재하는 대상이라면 Stabilization 보장보다 빠른 Feedback이 더 중요합니다.
2. **Staging Deploy**: 표준 Mode. Staging은 애초에 Stabilization Check에서만 드러나는 문제(영원히 통과하지 못하는 Health Check, Rollback Loop에 빠진 Service)를 잡아내기 위해 존재합니다.
3. **Production**: 예외 없이 항상 표준 Mode. 실제 Traffic을 받기 전에 Rollout이 실제로 건강한지 알려주는 Signal은 4배의 속도 향상과 맞바꿀 가치가 없습니다.
4. **Agent 주도 IaC 작업에 한해서는**: Sandbox나 일회성 Account에서는 Express Mode를 기본값으로 하고, "설정 수락"을 Agent가 다음 Step으로 진행하기에 충분한 Signal로 취급합니다. 공유 환경에 도달하기 전에는 사람이 Review하는 별도의 표준 Mode Deploy를 거치도록 합니다.

## 정리

Express Mode는 Infrastructure Tooling이 실제 반복 방식 — `cdk deploy`를 입력하고 기다리는 사람뿐 아니라 Agent가 주도하는 반복까지 포함해서 — 을 따라잡는 좋은 사례입니다. 기능 자체는 단순하지만, 그 주변의 판단은 단순하지 않습니다. 속도와 맞바꾸는 보장(완전한 Stabilization)이 정확히 무엇인지 파악하고, 그 Trade-off를 부분적으로만 안정화된 Resource가 실질적인 피해를 줄 수 없는 환경에만 국한시켜야 합니다.
