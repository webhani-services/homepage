AWS가 DevOps Agent의 정식 출시(GA)를 발표했습니다. 온콜 엔지니어가 상황 파악을 시작하기 전에 인시던트 조사를 자동으로 시작하는 AI 기반 서비스입니다. CloudWatch 알람, PagerDuty 알림, Dynatrace 이슈, ServiceNow 티켓, 또는 커스텀 Webhook을 트리거로 즉시 동작합니다. 같은 시기에 Security Agent의 온디맨드 침투 테스트도 GA가 되면서, AWS는 운영 영역의 AI 자동화를 본격적으로 추진하고 있습니다.

## DevOps Agent 동작 방식

인시던트 이벤트가 들어오면 에이전트는 사람이 개입하기 전에 다음 과정을 자율적으로 수행합니다.

1. **이벤트 수신**: CloudWatch, PagerDuty, Dynatrace, ServiceNow 또는 커스텀 Webhook
2. **컨텍스트 수집**: 관련 로그, 메트릭, 배포 이력, 설정 변경 사항 자동 수집
3. **근본 원인 분석**: 수집 데이터를 바탕으로 원인 가설 생성 및 검증
4. **보고서 생성**: 조사 결과, 영향 범위, 권장 조치를 담은 구조화된 보고서 출력
5. **자동 조치 (선택)**: 인간 승인 후 스케일 아웃, 설정 롤백 등 실행

핵심은 타이밍입니다. 온콜 엔지니어가 알람을 확인할 시점에 이미 초기 조사가 완료되어 있습니다. 새벽 3시에 아무것도 없는 상태에서 시작하는 대신, AI가 준비한 상황 브리핑을 받고 시작할 수 있습니다.

## 실용적인 활용 시나리오

### 배포 후 이상 감지

새 배포 직후 오류율이 급등했을 때, 에이전트는 직전 배포 Diff, 변경된 서비스 의존성, 영향받은 Endpoint를 자동으로 연관 분석합니다. 엔지니어는 "무슨 일이 일어났나"를 파악하는 데 소비하는 시간을 줄이고 "어떻게 대응할 것인가"에 집중할 수 있습니다.

### 반복적인 이상 패턴 발견

특정 시간대나 특정 부하 조건에서 간헐적으로 발생하는 성능 저하는 수동 디버깅으로 잡기가 매우 어렵습니다. 에이전트는 여러 인시던트에 걸쳐 컨텍스트를 축적하여 개별 조사에서 놓치기 쉬운 패턴을 드러냅니다.

### Runbook 기반 대응 제안

기존 Runbook을 참조하여 조치 단계를 제안하는 기능도 있습니다. 잘 관리된 Runbook을 보유한 팀은 그만큼 더 높은 품질의 자동화된 안내를 받을 수 있습니다. AI가 시니어 엔지니어의 지식을 경험이 적은 온콜 담당자에게 효과적으로 전달하는 역할을 합니다.

## 도입 전 반드시 확인할 사항

### 권한 범위 설계

DevOps Agent가 작업하려면 로그, 메트릭, 설정 정보에 대한 접근이 필요합니다. 최소 권한 원칙을 지키면서, 분석이 필요한 특정 리소스와 환경으로 권한 범위를 제한하세요.

```json
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "logs:GetLogEvents",
        "logs:DescribeLogGroups",
        "cloudwatch:GetMetricData",
        "cloudwatch:DescribeAlarms"
      ],
      "Resource": [
        "arn:aws:logs:ap-northeast-1:ACCOUNT_ID:log-group:/aws/lambda/production-*",
        "arn:aws:cloudwatch:ap-northeast-1:ACCOUNT_ID:alarm:production-*"
      ]
    }
  ]
}
```

운영 로그에는 민감한 정보가 포함될 수 있습니다. 특히 엄격한 데이터 레지던시나 컴플라이언스 요건이 있는 경우, 에이전트가 접근하는 데이터 범위와 처리 방식을 사전에 명확히 파악하세요.

### 자동 조치에는 Human-in-the-loop 필수

자동 조사는 리스크가 낮습니다. 자동 조치는 그렇지 않습니다. 보고서와 권장 사항은 자동으로 생성하되, 인프라 변경 실행에는 반드시 인간의 명시적 승인을 요구하도록 구성하세요. 정보 수집은 자동화하고, 실행 결정은 인간이 내리는 구조가 안전합니다.

### 옵저버빌리티 품질이 결과를 결정합니다

에이전트 출력 품질은 입력 데이터의 품질에 의해 제한됩니다. 노이즈가 많은 알람, 불충분한 로그 커버리지, 일관성 없는 태깅은 얕은 분석 결과를 낳습니다. GA 도입을 기회 삼아 기존 옵저버빌리티 설계를 재점검하는 것이 좋습니다. 수동 워크플로우에서는 참아낼 수 있었던 빈틈이 자동화에서는 병목이 됩니다.

## DevOps Agent + Security Agent 조합

Security Agent도 같은 시기에 GA가 됐습니다. 애플리케이션 설계, 코드, 런타임 동작을 지속적으로 분석하여 온디맨드 침투 테스트를 실행하고 취약점을 식별합니다. DevOps Agent가 사후 인시던트 대응을 담당한다면, Security Agent는 선제적 보안 검증을 담당합니다.

두 에이전트를 조합하면 두 가지 사이클을 모두 자동화할 수 있습니다.
- **DevOps Agent**: 감지 → 조사 → 대응
- **Security Agent**: 설계 → 검증 → 수정

두 에이전트 모두 인간의 보안 판단을 대체하지는 않지만, 감지에서 조치까지 걸리는 시간을 크게 단축합니다.

## 정리

AWS DevOps Agent GA는 인시던트 대응을 "엔지니어가 처음부터 시작"에서 "AI가 준비한 브리핑을 검토"하는 방식으로 전환할 수 있는 기회입니다. 온콜 부담이 높은 팀에게 특히 새벽 대응에서 큰 가치를 발휘합니다.

도입 성공을 위해 사전에 정리해야 할 세 가지:
1. IAM 권한 범위 — 최소 권한, 환경별 스코핑
2. 자동 조치에 대한 승인 게이트 구성
3. 옵저버빌리티 품질 점검 — 잘못된 입력은 에이전트도 해결 못합니다
