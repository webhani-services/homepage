AWS가 **DevOps Agent**의 일반 출시(GA)를 발표했습니다. AWS 환경에서 트러블슈팅, Deployment 분석, 운영 작업 자동화를 지원하는 AI 기반 도구입니다. Preview 기간을 거쳐 GA가 된 만큼 이제 프로덕션 환경 도입을 진지하게 검토할 수 있는 단계입니다.

## AWS DevOps Agent가 하는 일

AWS DevOps Agent는 CloudWatch Logs, X-Ray, ECS Deployment 이력, Parameter Store 등 기존 AWS 가시성 스택에 연결해 자연어 지시를 받아 다음 작업을 수행합니다.

**1. 근본 원인 분석(Root Cause Analysis)**: 알림이나 증상이 주어지면 Log, Metric, Trace를 종합 분석해 문제의 원인과 발생 시점을 특정합니다.

**2. Deployment 분석**: CodeDeploy 또는 ECS Deployment 이벤트와 성능 변화를 연관 지어 어떤 릴리즈가 문제를 일으켰는지 식별합니다.

**3. 운영 실행**: IAM 권한 범위 내에서 스케일링 정책 조정, Parameter Store 업데이트, Lambda 재배포 같은 작업을 직접 실행할 수 있습니다.

## 기존 도구와의 차이

CloudWatch Alarms는 "무언가 잘못됐다"는 사실을 알려줍니다. Runbook은 "무엇을 해야 하는가"를 문서화합니다. 그 사이에 있는 "왜 이런 일이 일어났는가"의 분석은 여전히 엔지니어의 수작업이었습니다.

DevOps Agent는 이 간극을 메웁니다.

```
기존: 알림 발생 → 엔지니어가 콘솔 접속 → Logs 확인 → 
      Trace 추적 → 가설 수립 → 수동 대처

DevOps Agent: 알림 발생 → Agent에게 분석 요청 → 
              Logs/Trace/Metric 자동 교차 분석 → 
              원인 제시 → 수정 방안 제안 (실행도 가능)
```

## 구체적 시나리오

ECS 서비스의 Latency가 새벽 2시에 급등했다고 가정합니다. Agent에게 조사를 요청하면 이런 응답을 받을 수 있습니다.

```
분석: 01:47 AM부터 p99 Latency가 120ms에서 2400ms로 증가.

발견: RDS 인스턴스 db.api-primary의 Connection 수가 
      01:45 AM 기준 최대치(100/100)에 도달.
      쿼리 대기 시간이 2ms에서 1800ms로 증가.

추정 원인: 01:43 AM Deployment에서 Parameter Store의 
           DB_POOL_SIZE가 50에서 10으로 변경됨.

권장 조치: /prod/api/DB_POOL_SIZE를 50으로 복원.

이 변경을 실행하시겠습니까? [예 / 아니오]
```

수동으로 20-30분 걸리던 교차 서비스 분석이 압축됩니다.

## IAM 권한 설정: 핵심

실행 권한을 가진 DevOps Agent는 강력하면서도 위험합니다. 권한 범위를 잘못 설정하면 잘못된 조치를 자동 실행할 수 있습니다.

권장 권한 계층:

| 작업 유형 | 접근 방식 |
|-----------|-----------|
| Read (Logs, Metric, Trace) | 자동 실행 허용 |
| 설정 변경 (Parameter Store, 스케일링) | Agent 제안 → 사람 승인 후 실행 |
| 인프라 변경 (인스턴스 종료 등) | 원칙적으로 자동 실행 금지 |

AI 기능이 뛰어나도 잘못된 작업의 영향 범위(Blast Radius)는 변하지 않습니다. 권한 범위 설계는 별도로 신중하게 다루어야 합니다.

## 가시성 기반이 전제 조건

DevOps Agent는 참조할 수 있는 데이터의 질에 직결됩니다. Application Log가 비구조화된 텍스트이거나 X-Ray/OpenTelemetry 계측이 부족하면 Agent의 분석도 피상적이 됩니다.

DevOps Agent를 의미 있게 활용하기 위한 사전 조건:

1. **구조화 Log**: CloudWatch Logs Insights가 효율적으로 쿼리할 수 있는 JSON 형식
2. **분산 Tracing**: 모든 서비스 경계에 걸친 X-Ray 또는 OpenTelemetry Trace
3. **의미 있는 Metric**: CPU/Memory뿐 아니라 비즈니스 레벨 지표를 포함한 커스텀 CloudWatch Metric

이것들이 갖춰져야 Agent가 제대로 기능합니다.

## 비용 관점

DevOps Agent는 내부적으로 Bedrock 모델을 호출하므로 쿼리당 비용이 발생합니다. 일반적인 인시던트 한 건당 조사 비용은 수 달러 수준입니다. 새벽에 엔지니어가 1시간 수동 조사하는 비용과 비교하면, 빈도가 높고 시간에 민감한 인시던트에서 경제성이 명확합니다.
