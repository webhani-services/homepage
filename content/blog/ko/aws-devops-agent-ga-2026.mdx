## 무엇이 출시되었나

2026년 4월, AWS DevOps Agent가 정식 출시(GA)되었습니다. Preview 기간 동안 AWS는 고객들이 MTTR(평균 복구 시간)을 최대 75% 단축하고, 인시던트 해결 속도가 3~5배 빨라졌다고 보고했습니다. 이 수치는 자체 환경에서 검증해야 하지만, 핵심 패턴은 분명합니다. 인시던트 대응 시간의 대부분은 수정에 쓰이는 것이 아니라 **정보 수집**에 소비됩니다.

AWS DevOps Agent는 다음 세 가지 영역을 담당합니다.

- **인시던트 조사**: CloudWatch 알람을 트리거로 관련 로그, 메트릭, 트레이스를 자동 수집
- **근본 원인 파악**: 여러 서비스에 걸친 이벤트를 상관 분석하여 장애 원인 특정
- **예방적 감지**: 기준 패턴으로부터의 이탈을 지속적으로 모니터링하여 문제가 심각해지기 전 알림

## 어떤 문제를 해결하는가

일반적인 온콜 인시던트 대응 흐름은 이렇습니다.

1. 알람 수신
2. CloudWatch에서 로그 수동 검색
3. X-Ray에서 트레이스 확인
4. 최근 배포 및 설정 변경 이력 조회
5. 가설을 세우고 조치

2~4단계는 정보 수집입니다. 시간이 걸리고, 반복적이며, 시스템이 장애 상태일 때는 극도로 스트레스를 줍니다. AWS DevOps Agent는 이 단계들을 자동화하여, 엔지니어가 5단계에 바로 집중할 수 있는 환경을 만듭니다.

새벽 3시에 페이징을 받았을 때, 이미 관련 데이터가 정리된 요약을 확인하고 바로 조치에 들어갈 수 있다는 것은 온콜 경험을 크게 바꿉니다.

## 작동 방식

AWS DevOps Agent는 Amazon Bedrock 위에 구축되어 있으며 다음 서비스들과 통합됩니다.

- **Amazon CloudWatch** — 로그, 메트릭, 알람 읽기
- **AWS X-Ray** — 분산 트레이스 분석
- **AWS Config** — 영향 받은 리소스의 설정 변경 이력 조회
- **AWS Systems Manager** — 복구 조치 실행 (명시적 권한 부여 필요)

인시던트 발생 시 Agent는 자연어로 조사 요약을 생성합니다. 엔지니어는 Slack이나 AWS Console을 통해 대화 형식으로 추가 조사를 지시할 수 있습니다.

```
엔지니어: "us-east-1 API 서버에서 5xx 에러가 급증하고 있습니다.
           최근 30분간 무슨 일이 있었나요?"

Agent: "조사 결과를 알려드립니다.
        - 14:32 UTC: 새로운 ECS 태스크 정의 배포 완료
        - 14:35 UTC: /api/orders 엔드포인트 P99 레이턴시 200ms → 2.1s 상승
        - 14:36 UTC: RDS 연결 수가 설정 상한(100)에 도달
        
        배포 직후 Connection Pool이 고갈된 것으로 보입니다.
        새 태스크 정의에서 DB 연결 풀 설정이 변경되었을 가능성이 있습니다.
        태스크 정의 Diff에서 데이터베이스 연결 설정을 확인해보세요."
```

## 도입 전 고려사항

### IAM 설계

Agent에게 복구 조치 실행 권한을 부여하려면 최소 권한 원칙을 철저히 적용해야 합니다. 처음에는 읽기 전용 조사 권한만 부여하고, 분석 신뢰도를 검증한 후 단계적으로 복구 권한을 확장하는 접근을 권장합니다.

### 비용 구조

AWS DevOps Agent 사용은 Amazon Bedrock 요금이 발생합니다. 분석하는 로그 양과 인시던트 빈도에 따라 비용이 변동됩니다. 전체 도입 전에 CloudWatch Logs 데이터 볼륨을 확인하고 Bedrock 토큰 소비량을 추정해두는 것이 중요합니다.

### False Positive 관리

AI 기반 근본 원인 분석은 특이한 케이스에서 잘못된 원인을 제시할 수 있습니다. 초기 도입 시에는 Agent의 결론을 조사의 출발점으로 활용하고, 제안된 원인을 직접 검증한 후 조치를 취하는 습관을 유지해야 합니다.

## 현실적인 기대치

AWS DevOps Agent가 온콜 엔지니어를 대체하지는 않습니다. 변화는 인시던트의 **정보 수집 단계**에서 일어납니다. 로그 검색에 20분을 쓰는 대신, 2분 안에 구조화된 요약을 받아 바로 해결에 집중할 수 있게 됩니다.

인시던트가 잦고 온콜 부하가 높은 분산 시스템을 AWS에서 운영하는 팀에게는 실질적인 생산성 향상입니다. 반면 인시던트가 드물고 아키텍처가 단순한 소규모 팀에게는 설정 비용 대비 효과가 즉시 나타나지 않을 수 있습니다.

**실용적인 접근법**: 먼저 Staging 환경이나 낮은 심각도의 알람 일부에 적용해 파일럿을 진행하세요. 실제 워크로드에서 요약의 정확도를 검증한 후 전체 프로덕션 인시던트 대응에 통합하는 것을 권장합니다.
