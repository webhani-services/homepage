## Anthropic의 선택

Anthropic이 Bun runtime을 Claude Code, Claude Agent SDK, 그리고 향후 AI 코딩 제품의 핵심 인프라로 채택했습니다. 표면적으로는 단순한 도구 선택처럼 보이지만, AI 코딩 에이전트 맥락에서는 명확한 성능 전략이 반영된 결정입니다.

Runtime의 선택은 에이전트가 얼마나 빠르게 반복할 수 있는지, 테스트가 얼마나 신속하게 실행되는지, 장시간 세션에서 도구가 얼마나 빠르게 응답하는지를 결정합니다. Bun이 무엇을 제공하는지, 그리고 AI 중심 워크플로에서 왜 중요한지 살펴보겠습니다.

---

## Bun이란 무엇인가

Bun은 단일 바이너리로 제공되는 JavaScript 올인원 툴킷입니다.

- **Runtime** — JavaScriptCore 기반, 최고 속도 설계
- **Package manager** — npm 호환, cold install 약 25배 빠름
- **Bundler** — 대부분의 워크로드에서 esbuild보다 빠름
- **Test runner** — Jest 호환 API
- **Script runner** — 추가 설정 없이 `ts-node`, `tsx` 대체

Node.js가 Node + npm + 빌드 도구 + 테스트 러너를 조합하는 방식이라면, Bun은 이 모든 것을 하나의 바이너리에 통합합니다. 설정이 적고, 의존성이 줄어들고, 전반적으로 빠릅니다.

```bash
# Bun 설치
curl -fsSL https://bun.sh/install | bash

# Node.js 워크플로를 그대로 대체
bun install          # npm install 대체 (cold 기준 ~25배 빠름)
bun run build        # npm run build 대체
bun test             # Jest 호환 테스트 실행
bun run server.ts    # TypeScript 직접 실행 (트랜스파일 불필요)
```

## AI 코딩 에이전트에서 속도가 중요한 이유

Claude Code 같은 AI 코딩 에이전트는 작업 중에 다음 루프를 반복합니다.

1. 파일을 읽고 컨텍스트를 파악
2. 변경 사항 적용
3. 테스트 실행
4. 결과 해석
5. 다음 액션 결정

이 루프가 빠를수록 에이전트는 같은 시간에 더 많은 반복을 처리할 수 있습니다. 테스트 실행에 2초가 걸리고 Claude가 100번 실행한다면 200초가 소요됩니다. 이를 0.8초로 줄이면 세션당 2분 이상을 절약합니다.

Dynamic Workflows처럼 수백 개의 Subagent를 병렬로 실행하는 경우, Runtime의 처리량이 전체 작업 완료 시간에 직접적인 영향을 미칩니다.

## 벤치마크 비교

일반적인 워크로드 기준 비교 (환경에 따라 다를 수 있음):

| 작업 | Node.js | Bun |
|------|---------|-----|
| Package install (cold cache) | ~10초 | ~0.4초 |
| TypeScript 실행 (ts-node) | ~800ms | ~80ms |
| 테스트 실행 (Jest vs bun test) | ~2.5초 | ~0.8초 |
| HTTP 서버 Cold start | ~180ms | ~20ms |

이 수치가 프로덕션 전체를 대표하지는 않지만, AI 코딩 에이전트가 의존하는 빠른 반복 루프에서는 실질적인 차이입니다.

## 개발자에게 미치는 실질적인 영향

Bun이 AI 코딩 인프라의 기본 Runtime이 된다면 실무에서 어떤 의미를 가질까요?

**Node.js 호환성은 충분히 성숙**  
Bun은 Node.js API의 대부분을 구현합니다. npm 패키지 대부분이 변경 없이 동작합니다. 갭은 주로 네이티브 모듈(`.node` 파일)과 일부 Node.js 내부 API에 있습니다. 표준 도구를 사용하는 TypeScript 프로젝트라면 마이그레이션 비용은 낮습니다.

**TypeScript 설정 없이 직접 실행**  
Bun은 기본 실행에 `tsconfig.json` 없이도 TypeScript를 직접 실행합니다. 스크립트와 유틸리티 코드에서는 `ts-node` 의존성을 완전히 제거할 수 있습니다.

```typescript
// bun run server.ts — 트랜스파일 단계 불필요
import { serve } from "bun";

serve({
  port: 3000,
  fetch(req: Request): Response {
    return new Response("OK", { status: 200 });
  },
});
```

**통합 Test runner**  
Bun의 내장 테스트 러너는 Jest 호환 API를 사용합니다. 새 프로젝트에서는 별도의 Jest나 Vitest 의존성을 제거할 수 있습니다.

```typescript
import { describe, expect, test } from "bun:test";

describe("formatAmount", () => {
  test("JPY 금액을 올바르게 포맷", () => {
    expect(formatAmount(1000, "JPY")).toBe("¥1,000");
  });

  test("0 처리", () => {
    expect(formatAmount(0, "JPY")).toBe("¥0");
  });
});
```

## 더 큰 그림: AI 인프라로서의 Runtime

Anthropic이 Bun을 선택한 것은 단순히 Claude Code를 빠르게 만들기 위한 것이 아닙니다. AI 코딩 인프라가 향하는 방향을 보여주는 결정입니다.

AI 코딩 에이전트는 점점 전체 개발 파이프라인(작성, 테스트, 빌드, 배포)을 담당하게 됩니다. 에이전트가 사용하는 Runtime은 단순한 실행 환경이 아니라 에이전트 오케스트레이션의 기반이 됩니다. 빠르고 통합된 Runtime은 에이전트의 추론 단계와 도구 호출 간의 오버헤드를 줄입니다.

## 실전 마이그레이션 경로

이미 Node.js 프로젝트를 운영 중인 팀의 현실적인 접근 방법입니다.

**기존 프로덕션 서비스는 아직 마이그레이션하지 마세요**  
Bun은 많은 워크로드에서 프로덕션 준비가 됐지만, 네이티브 모듈 의존성과 일부 Node.js 내부 API는 확인이 필요합니다. 전환 전 의존성 트리를 감사하세요.

**스크립트와 테스트 러너부터 시작하세요**  
개발 스크립트에서 `ts-node`를 교체하고, 독립적인 패키지에서 Jest를 `bun test`로 전환하세요. 리스크가 낮고 속도 향상이 큰 변경입니다.

**새로운 AI 도구 프로젝트에 Bun을 사용하세요**  
Claude Code나 유사 에이전트와 상호작용하는 스크립트나 서비스를 새로 만든다면 Bun으로 시작하는 것이 합리적입니다.

## 정리

- Anthropic이 Bun을 Claude Code와 Claude Agent SDK의 기반 Runtime으로 채택
- Bun은 Runtime, Package manager, Bundler, Test runner를 단일 바이너리로 제공
