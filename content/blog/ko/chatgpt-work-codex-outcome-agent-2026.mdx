2026년 7월 9일, OpenAI는 GPT-5.6과 함께 "ChatGPT Work"를 발표했습니다. 같은 발표에서 Codex는 새로운 ChatGPT Desktop App으로 통합되었고, 기존 Chat App은 "ChatGPT Classic"이라는 위치로 재정의됐습니다. 이번 발표에서 주목할 부분은 새 Model 자체보다 AI Agent가 반환하는 결과물의 형태가 바뀌고 있다는 점입니다. 이 글에서는 ChatGPT Work의 특징과 실무에 도입할 때의 접근 방식을 정리합니다.

## Chat 답변에서 "완성된 결과물"로

지금까지 Chat형 AI는 질문에 Text로 답하는 구조가 기본이었습니다. ChatGPT Work는 이 구조를 바꿉니다. 사용자가 전달하는 것은 "질문"이 아니라 "달성하고 싶은 Goal"이며, Agent는 다음 순서로 동작합니다.

1. 연동된 Application과 Workflow에서 필요한 정보를 수집합니다
2. Goal을 실행 가능한 단계로 분해합니다
3. 각 단계를 자율적으로 실행하며, 수 시간 단위의 작업도 이어갑니다
4. 최종적으로 Spreadsheet, Slide, Document, Web App 같은 "완성된 결과물"을 반환합니다

이런 방식은 Coding 영역에서는 이미 익숙합니다. Claude Code와 GitHub Copilot의 Workspace Agent는 단발성 Code 자동완성에서 "Issue를 전달하면 PR이 돌아오는" 형태로 진화해왔습니다. ChatGPT Work는 이 Outcome형 접근을 Coding 외의 일반 업무 전반으로 확장한 것으로 이해하면 됩니다.

## Codex 통합이 실무에 갖는 의미

Codex가 Desktop App에 통합되면서 Chat·Work·Codex가 하나의 Interface 위에 놓이게 됐습니다. 개발자 입장에서 실질적인 변화는 Coding Agent와 범용 Outcome Agent가 같은 Session과 Context를 공유하기 쉬워졌다는 점입니다. 예를 들어 "API 명세를 작성하면서 동시에 대응하는 구현 PR도 진행"하는 것처럼, Documentation과 Coding을 오가는 작업의 단절이 줄어들 것으로 기대됩니다.

다만 Rollout은 Pro·Enterprise·Edu Plan부터 시작해 Plus·Business Plan으로 단계적으로 확대되는 방식입니다. 조직 차원에서 도입을 검토한다면, 제한된 Plan과 소수 사용자로 먼저 시험해보고 동작을 확인한 뒤 범위를 넓히는 것이 현실적입니다.

## 도입 전 정리해야 할 Governance 쟁점

Outcome형 Agent가 "연동된 Application에서 정보를 수집"하는 구조인 이상, 어떤 Application에 어느 범위까지 접근을 허용할지가 가장 먼저 결정해야 할 부분입니다. 특히 사내 Ticket System, Repository, Cloud Storage에 연결하는 경우 최소 권한 원칙을 철저히 지켜야 합니다.

또 하나의 쟁점은 Agent가 만든 결과물을 그대로 외부에 공개해도 되는가입니다. 수 시간에 걸쳐 자율적으로 만들어진 Document나 Web App이라 해도, 최종 공개 여부는 사람이 판단해야 합니다. 이는 Coding Agent가 만든 PR에 Human Review를 필수로 두는 것과 같은 원칙입니다.

```yaml
# .github/workflows 일부 (개념 예시)
# AI Agent가 작성한 PR은 Review Label이 붙기 전까지 Merge를 차단
name: require-human-review-on-agent-pr

on:
  pull_request:
    types: [labeled, opened, synchronize]

jobs:
  check-review-label:
    if: contains(github.event.pull_request.labels.*.name, 'ai-agent-generated')
    runs-on: ubuntu-latest
    steps:
      - name: Verify human-reviewed label exists
        run: |
          # 'ai-agent-generated' Label이 붙은 PR은
          # 'human-reviewed' Label이 별도로 붙기 전까지 Merge 불가
          echo "AI Agent가 만든 변경 사항은 사람의 Review 완료가 필수"
```

이런 Gate는 특정 Vendor에 종속되지 않는 범용적인 접근입니다. ChatGPT Work뿐 아니라 어떤 Outcome형 Agent를 도입하든 "누가 최종 승인할 것인가"를 운영 규칙으로 먼저 정해두는 것을 권장합니다.

## webhani의 관점

webhani는 Outcome형 AI Agent 도입을 검토하는 Client에게 다음 순서를 제안합니다.

1. **외부에 공개하지 않는 사내 업무**(정형 Report, 사내 Document 초안 등)부터 시험 도입합니다
2. Agent에 허용하는 **Data Connection 범위를 최소한으로 좁히고**, 이후 필요에 따라 확장합니다
3. 결과물을 외부에 공개하거나 Production에 반영하기 전, **사람에 의한 Review를 필수 Gate로 운영에 포함**시킵니다

새 Model의 성능 자체보다 이런 "도입 Process 설계"가 실무 Risk를 좌우합니다. ChatGPT Work 같은 Outcome형 Agent는 앞으로도 계속 늘어날 것으로 보이는 만큼, 특정 Vendor 기능에 지나치게 의존하지 않는 공통 Governance 방침을 미리 마련해두는 것이 중요합니다.
