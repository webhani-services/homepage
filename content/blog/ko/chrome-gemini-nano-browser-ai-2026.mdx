Google이 Chrome에 Gemini Nano(약 4GB의 경량 LLM)를 기본 설치하고, 웹 페이지의 JavaScript에서 직접 호출할 수 있는 AI API 세트를 단계적으로 공개하고 있습니다. 서버에 요청을 보내지 않고 브라우저 내에서 자연어 처리가 가능해졌다는 점은 웹 애플리케이션 설계에 새로운 선택지를 제공합니다.

## 어떤 API가 있는가

현재 제공 중이거나 Origin Trial 단계에 있는 API는 다음과 같습니다.

- **Prompt API**: 텍스트 프롬프트를 보내고 LLM 응답을 받는 범용 API
- **Translation API**: 온디바이스 텍스트 번역
- **Summarization API**: 긴 텍스트 요약
- **Rewrite API**: 텍스트 톤 변환 및 재작성

이 API들은 모두 클라이언트 사이드에서 완결되므로 사용자 입력이 서버로 전송되지 않습니다.

## Prompt API 사용 방법

```javascript
// Origin Trial 등록이 필요합니다 (2026년 현재)
const session = await window.ai.languageModel.create({
  systemPrompt: "You are a helpful assistant for a Japanese e-commerce site.",
});

const response = await session.prompt("반품 정책에 대해 알려주세요");
console.log(response);

// 스트리밍도 지원합니다
const stream = await session.promptStreaming("긴 질문...");
for await (const chunk of stream) {
  process.stdout.write(chunk);
}
```

API 구조는 서버 사이드 AI SDK와 유사하게 설계되어 있어, 이미 AI 기능을 구현한 팀이라면 학습 비용이 낮습니다.

## Translation API 실용 예제

```javascript
const translator = await window.translation.createTranslator({
  sourceLanguage: "en",
  targetLanguage: "ja",
});

// 사용자가 입력하는 영어 텍스트를 실시간으로 번역
document.getElementById("input").addEventListener("input", async (e) => {
  const translated = await translator.translate(e.target.value);
  document.getElementById("output").textContent = translated;
});
```

DeepL이나 Google Translate API로의 HTTP 요청을 브라우저 내 처리로 대체하면 네트워크 의존성이 줄고, 응답 시간도 50~200ms 수준으로 개선됩니다.

## 하드웨어 요구사항과 폴백 처리

Gemini Nano는 GPU 또는 NPU가 있는 환경에 최적화되어 있습니다. 프로덕션 구현 시 능력 확인은 필수입니다.

```javascript
const capabilities = await window.ai.languageModel.capabilities();

if (capabilities.available === "readily") {
  // 모델 사용 가능
} else if (capabilities.available === "after-download") {
  // 다운로드 완료 후 사용 가능
} else {
  // 이 기기에서는 사용 불가 → 서버 API로 폴백
  return callServerSideAPI(input);
}
```

모바일 기기나 저사양 노트북에서는 사용 불가 케이스가 많습니다. 반드시 서버 사이드 API를 폴백으로 유지해야 합니다.

## 프라이버시 측면

개인정보나 사내 문서처럼 민감한 텍스트를 다루는 경우, 온디바이스 처리는 서버 사이드 API가 갖는 데이터 전송 리스크를 없애줍니다. 이는 추상적인 장점이 아니라 실제 보안 설계 관점에서 의미 있는 차이입니다.

오프라인 환경에서도 동작하므로 네트워크 연결이 불안정한 환경(현장 작업, 지하 등)을 대상으로 한 애플리케이션에서도 가치를 발휘합니다.

## 실제로 활용할 수 있는 곳

**적합한 경우:**
- 리치 텍스트 에디터에서의 자동완성 및 문장 개선 제안
- 다국어 UI에서 실시간 번역 미리보기
- 지원 티켓 등 사용자 입력 내용의 경량 요약

**서버 사이드 API를 유지해야 하는 경우:**
- 정확도가 중요한 프로덕션 번역
- 긴 문서 처리 (Gemini Nano의 컨텍스트 윈도우가 제한적)
- iOS/Safari 사용자 비중이 높은 서비스 (현재 Chrome 전용)
- 모바일 중심 트래픽 (하드웨어 요건 미충족 케이스가 많음)

## Origin Trial 현황

2026년 5월 기준, 대부분의 API가 Chrome Origin Trial 단계에 있습니다.

```html
<!-- Origin Trial 토큰을 메타 태그로 지정 -->
<meta http-equiv="origin-trial" content="YOUR_TOKEN_HERE">
```

W3C 표준화가 진행 중이며, 다른 브라우저로의 확장도 논의되고 있습니다. 현시점에서는 반드시 폴백을 구현하고, Chrome Desktop 사용자에 대한 점진적 개선으로 접근하는 것이 현실적입니다.

## webhani의 판단

브라우저 네이티브 AI는 서버 사이드 추론을 대체하는 솔루션이 아닙니다. 가벼운 AI 기능(입력 보조, 번역, 요약)을 인프라 비용 없이 제공할 수 있는 새로운 런타임 레이어로 보는 것이 적절합니다.

능력 확인 → 서버 사이드 폴백 구현 → Chrome Desktop 사용자에 대한 개선 기능 제공이라는 순서로 구현하면, 지금 당장 가치를 얻으면서도 범용성 문제를 회피할 수 있습니다.
