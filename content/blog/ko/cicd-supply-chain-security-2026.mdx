CI/CD 파이프라인을 노리는 공급망 공격이 지속적으로 증가하고 있습니다. 2026년에도 GitHub 토큰 탈취, npm 패키지 오염, 워크플로우 취약점 악용 등 여러 심각한 보안 사고가 보고되었습니다. 실제 공격 패턴과 바로 적용할 수 있는 방어 방법을 정리합니다.

## 실제 공격 패턴

### Pwn Request를 통한 Token 탈취

Kubernetes CI 워크플로우에서 발견된 "Pwn Request" 취약점은 공격자가 Pull Request 이벤트를 통해 악의적인 워크플로우 실행을 트리거하여 Repository 토큰을 탈취하는 방식이었습니다. 근본 원인은 신뢰할 수 없는 코드를 과도한 권한으로 실행하는 워크플로우였습니다.

### 패키지 오염

두 가지 일반적인 패턴이 나타납니다.

1. **타이포스쿼팅**: 인기 패키지와 유사한 이름의 패키지 게시
2. **계정 탈취**: 메인테이너 계정을 침해하여 정상 패키지에 악성 코드 삽입

SAP 관련 라이브러리를 포함한 여러 엔터프라이즈 패키지가 영향을 받았습니다.

### 자격 증명 노출

실수로 Repository에 커밋된 시크릿 — `.env` 파일, AWS 자격 증명, API 토큰 — 은 노출 후 몇 분 내에 자동화된 도구에 의해 수집됩니다. 이는 여전히 가장 흔한 초기 침투 경로 중 하나입니다.

## 구현할 만한 방어 수단

### 1. 워크플로우 권한 제한

```yaml
# .github/workflows/ci.yml
jobs:
  build:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      id-token: write  # OIDC 사용 시에만
    steps:
      - uses: actions/checkout@v4
```

Job 레벨에서 `permissions`를 명시적으로 설정하세요. GitHub의 기본값은 대부분의 워크플로우가 필요로 하는 것보다 넓습니다. 최소 권한 원칙을 적용하세요.

### 2. 서드파티 Action을 SHA로 고정

```yaml
# 위험: 태그는 덮어쓸 수 있음
- uses: actions/setup-node@v4

# 안전: Commit SHA는 불변
- uses: actions/setup-node@60edb5dd545a775178f52524783378180af0d1f8
```

GitHub Actions의 태그는 변경 가능합니다. 침해된 Action 메인테이너가 기존 태그에 악성 코드를 Push할 수 있습니다. SHA 고정으로 이를 방지합니다.

### 3. Secret Scanning 활성화

GitHub의 내장 Secret Scanning이 일반적인 자격 증명 패턴을 Repository에 도달하기 전에 탐지합니다. Pre-commit 훅으로 보완하세요.

```bash
pip install detect-secrets
detect-secrets scan > .secrets.baseline
```

```yaml
# .pre-commit-config.yaml
repos:
  - repo: https://github.com/Yelp/detect-secrets
    rev: v1.4.0
    hooks:
      - id: detect-secrets
        args: ['--baseline', '.secrets.baseline']
```

### 4. 의존성 지속적 감사

```bash
# CI에서 실행
npm audit --audit-level=high

# 지속적인 모니터링
npx snyk monitor
```

`package-lock.json`을 커밋하여 Pull Request에서 예상치 못한 의존성 변경을 확인할 수 있도록 하세요.

### 5. 장기 자격 증명을 OIDC로 교체

```yaml
- name: Configure AWS Credentials
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: arn:aws:iam::123456789012:role/GitHubActions
    aws-region: ap-northeast-2
```

OIDC를 사용하면 `AWS_ACCESS_KEY_ID`와 `AWS_SECRET_ACCESS_KEY`를 GitHub Secret으로 저장할 필요가 없습니다. 워크플로우는 Job 외부에서 재사용할 수 없는 단기 자격 증명을 받습니다.

## webhani의 관점

CI/CD 파이프라인은 코드, 자격 증명, 프로덕션 접근이 교차하는 지점에 위치하기 때문에 공격자에게 매우 가치 있는 타깃입니다. 파이프라인이 침해되면 종종 완전한 시스템 침해로 이어집니다.

대부분의 팀은 CI/CD 보안을 제품 작업이 아닌 인프라 오버헤드로 느껴 미루는 경향이 있습니다. 하지만 여기에 나열된 수정 사항들은 대부분 일회성 설정 변경으로, 구현 비용이 낮고 지속적인 보호를 제공합니다.

권한 제한과 Action SHA 고정부터 시작하세요. 이 두 가지 변경만으로도 워크플로우 중단을 최소화하면서 광범위한 공급망 위험을 제거할 수 있습니다.
