## LLM에서 Agent로의 전환

2026년 개발자 도구 분야에서 가장 큰 변화는 새로운 모델의 등장이 아니라 모델 활용 방식의 변화입니다. 고급 자동완성으로 시작했던 LLM이 이제는 계획을 세우고, 파일을 읽고 쓰며, 명령어를 실행하고, 결과를 검증하고 실패 시 재시도하는 Agent로 발전했습니다.

Claude Code, Cursor, GitHub Copilot Workspace 모두 이 방향으로 이동하고 있습니다. 2026년 4월에 출시된 Claude Opus 4.7은 SWE-bench Pro에서 64.3%를 기록했습니다 — 오픈소스 저장소의 실제 GitHub Issue를 해결하는 능력을 측정하는 벤치마크로, Opus 4.6의 53.4%에서 10.9포인트 향상된 수치입니다. 이 개선은 벤치마크 최적화만이 아닌, 실제 코드베이스에서의 추론 능력이 실질적으로 향상되었음을 반영합니다.

실제 개발 작업에서 무엇이 중요한지 정리합니다.

## Agent에 적합한 작업과 그렇지 않은 작업

모든 작업이 Agentic 워크플로우에 적합한 것은 아닙니다. 이 구분이 중요합니다. 잘못된 작업에 Agent를 사용하면 겉으로는 올바르게 보이지만 코드 리뷰에서 잡아내기 어려운 미묘한 문제가 있는 코드가 생성됩니다.

**적합한 작업:**
- 여러 파일에 걸친 일관성이 필요한 리팩토링
- 기존 패턴을 따르는 보일러플레이트 생성 (테스트, 타입 정의, CRUD 핸들러)
- 코드베이스의 기존 패턴과 유사한 기능 구현
- 코드 변경사항에서 문서 또는 Changelog 생성

**적합하지 않은 작업:**
- 깊은 비즈니스 도메인 지식이 필요한 설계 결정
- 외부 시스템 상태에 의존하는 복잡한 디버깅
- 정확성 기준이 명확하지 않은 보안 관련 코드
- 많은 의존관계가 있는 장기 다중 세션 작업

Agent에 위임하기 전 핵심 질문: "이것을 어떻게 검증할 것인가?" 명확히 답할 수 없다면, Agent는 그럴듯해 보이지만 코드 리뷰를 통과할 수도 있는 잘못된 코드를 생성할 수 있습니다.

## Agentic 작업을 위한 프롬프트 구조화

프롬프트 구조는 일반 LLM 대화보다 Agentic 모드에서 더 큰 영향을 미칩니다. 모호한 요청은 Agent가 의도와 충돌하는 가정을 하게 만듭니다.

```markdown
# 작업
[1-2문장으로 구체적인 변경 내용 설명]

# 컨텍스트
- 대상 파일: `src/lib/auth.ts`
- 관련 타입: `src/types/user.ts`
- 테스트 파일: `src/lib/auth.test.ts`

# 제약 조건
- 변경 후 기존 테스트가 모두 통과해야 함
- `any` 타입 사용 금지
- `src/lib/errors.ts`의 에러 처리 패턴을 따를 것

# 기대 출력
- 수정된 파일 목록
- 변경 내용과 이유에 대한 간략한 요약
```

핵심은 어떻게 구현할지가 아니라 무엇을 달성하고 무엇을 변경해서는 안 되는지를 명시하는 것입니다. Agent는 자체적인 구현 전략을 세웁니다. 여러분의 역할은 해결책 공간을 제약하는 것이지 해결책을 처방하는 것이 아닙니다.

## Context Window 관리

긴 Agentic 세션에서 가장 흔한 실패 유형은 컨텍스트 드리프트입니다 — 대화가 길어질수록 Agent가 초기 제약 조건이나 결정사항을 추적하지 못하게 됩니다. 이는 모델 버그가 아니라 컨텍스트 창 관리 문제입니다.

실용적인 해결 방법:

```markdown
# 세션 앵커 (새 세션 시작 시 붙여넣기)

현재 작업: [간략한 설명]
완료된 작업:
- ✅ 인증 모듈 리팩토링
- ✅ 타입 정의 업데이트

남은 작업:
- [ ] 테스트 스위트 업데이트
- [ ] 문서 업데이트

변경 불가 제약:
- `any` 타입 사용 금지
- 에러 처리는 커스텀 Result 타입 사용
- 모든 엔드포인트에 요청 유효성 검사 포함
```

다중 세션 작업의 경우, 각 세션 종료 시 Agent가 구조화된 진행 요약을 생성하게 하고, 다음 세션 시작 시 이를 컨텍스트로 붙여넣으세요. 토큰 비용이 약간 들지만 비용이 큰 역추적을 방지합니다.

## 인간 리뷰 통합

Agent가 생성한 코드를 바로 프로덕션에 반영하는 것은 위험 패턴입니다. 실용적인 워크플로우는 인간 리뷰 체크포인트를 포함해야 합니다:

```bash
# 커밋 전 변경사항 검토
git diff --stat HEAD

# 큰 변경 세트의 경우 대화형으로 스테이징
git add -p

# 생성된 코드에 대해 테스트 실행
npm test

# 타입 체크 별도 실행 — 테스트는 타입 에러를 잡지 못함
npx tsc --noEmit
```

주의할 실패 패턴: 테스트는 통과하지만 의미론적 문제가 있는 코드 — 기술적으로는 작동하지만 테스트가 커버하지 않는 불변성을 위반하는 리팩토링, 또는 예외를 전파하는 대신 삼키는 에러 경로입니다.

Agent 작업에서 소규모 증분 커밋이 하나의 큰 커밋보다 안전합니다. 1000줄 diff보다 100줄 diff를 검토할 때 어떤 변경이 문제를 유발했는지 파악하기 쉽습니다.

## 팀 도입 패턴

개인적인 Agent 사용과 팀 단위의 체계적 도입은 다른 문제입니다.

팀 도입에서 효과적인 접근법:

**1. 코드베이스 규칙을 AI가 읽을 수 있는 형식으로 문서화**

```markdown
# AI 어시스턴트를 위한 엔지니어링 가이드라인

## TypeScript
- 항상 strict 모드 사용; `any` 절대 사용 금지
- 도메인 에러에는 throw 대신 `Result<T, E>` 선호
- 모든 public 함수에 명시적 반환 타입 어노테이션 필수

## API Route (Next.js)
- 처리 전 Zod로 요청 본문 유효성 검사
- `src/lib/api.ts`의 표준 에러 응답 형식 사용
- Route Handler가 아닌 Service 레이어에서 로깅

## 테스트
- 단위 테스트: 순수 함수만
- 통합 테스트: `.env.test`에 설정된 테스트 DB 사용
```

**2. 효과적인 프롬프트 템플릿 공유**

일관적으로 좋은 결과를 내는 프롬프트 패턴이 있다면, 팀 전체가 사용할 수 있는 템플릿으로 추출하세요. 프롬프트는 엔지니어링 산출물입니다.

**3. Agent 출력에 동일한 코드 리뷰 기준 적용**

대부분의 팀이 잘못하는 부분입니다. 구문적으로 올바르고 컴파일이 되기 때문에 Agent 생성 코드가 가벼운 리뷰를 받는 경우가 많습니다. 주니어 엔지니어의 코드에 적용하는 것과 동일한 리뷰 기준을 적용하세요 — 특정 결정이 역사적으로 왜 내려졌는지 Agent는 맥락이 없기 때문에 더 엄격하게 검토해야 할 수도 있습니다.

## CI에서의 Claude Code Background Task

GitHub Actions를 통해 활성 세션 없이 Agent 작업을 실행하는 Claude Code의 기능은 반복적인 작업을 위한 워크플로우를 변화시킵니다. 라이브러리 업그레이드 후 의존성 타입 업데이트, 마이그레이션 스크립트 생성, 코드 변경과 문서 동기화 유지 등에 활용할 수 있습니다.

```yaml
# .github/workflows/ai-type-update.yml
name: Update types after dependency bump
on:
  pull_request:
    paths:
      - 'package.json'

jobs:
  update-types:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Code to update affected types
        uses: anthropics/claude-code-action@v1
        with:
          prompt: |
            package.json이 변경되었습니다. 수정된 패키지의 TypeScript 타입 임포트
            업데이트가 필요한지 확인하세요. `tsc --noEmit`을 실행해 타입 에러를 찾고,
            수정 후 "chore: update types for dependency changes" 메시지로 커밋하세요.
```

대부분의 팀에게 아직 초기 단계이지만, 패턴 자체는 유효합니다: 경계가 명확하고 검증 가능한 작업을 정의하고, CI에서 실행하고, 사람이 PR을 리뷰하는 방식입니다.

## 마치며

Agentic AI는 "시도해볼까?" 단계를 지나쳤습니다. 이제 질문은 실용적입니다: 어떤 작업을 위임할 것인가, 프롬프트를 어떻게 구조화할 것인가, 긴 세션에서 컨텍스트를 어떻게 관리할 것인가, 그리고 Agent 출력을 기존 리뷰 프로세스에 어떻게 통합할 것인가.

모델 성능은 계속 향상될 것입니다. 하지만 작업을 정확하게 정의하고, 출력을 체계적으로 검증하며, 인간의 판단이 대체 불가능한 영역을 파악하는 능력은 모델이 아무리 발전해도 변함없이 필요합니다.
