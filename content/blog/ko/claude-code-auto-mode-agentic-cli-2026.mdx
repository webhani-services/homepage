Claude Code의 Auto Mode와 Computer Use 기능이 이제 터미널 환경으로 확대되었습니다. Claude가 터미널 UI를 직접 조작하고, 도구를 실행하며, 사람의 승인 없이 대화형 워크플로우를 끝까지 처리할 수 있게 된 것입니다.

이는 "모든 단계에서 승인이 필요한 어시스턴트"에서 "작업을 처음부터 끝까지 자율적으로 수행하는 에이전트"로의 전환입니다. 개발팀 입장에서는 CI/CD 진단, 배포 워크플로우, 코드 마이그레이션 전체를 무인으로 실행할 수 있으면서도 완전히 감시 가능하다는 의미입니다. 다만 구조 없는 자율성은 위험합니다. 안전하게 도입하는 방법을 설명합니다.

## CLI에서 Computer Use의 의미

Computer Use(Claude가 애플리케이션 UI와 터미널 인터페이스와 상호작용함)와 Auto Mode의 결합은 새로운 실행 모델을 만듭니다. Claude는 다음을 수행할 수 있습니다:

- 대화형 도구 조작 (Kubernetes 대시보드, Database 클라이언트, Cloud CLI)
- 터미널 출력을 읽고 실시간으로 명령 적응
- 각 단계의 출력이 다음 단계를 결정하는 다단계 워크플로우 자동 진행
- 일시적 장애 감지 및 지능형 재시도

기존 CLI 스크립팅과의 차이점: Claude는 단순 명령 시퀀스만 이해하는 게 아니라, 실패 모드와 문맥을 이해합니다.

예시: EKS에 배포하되 헬스체크 실패 시 롤백. 단순 스크립트는 항상 롤백하거나 절대 롤백하지 않습니다. Claude는 실제 클러스터 상태를 평가하고, Pod 로그를 읽고, 메트릭을 비교하여 판단합니다—모두 무인으로.

```bash
# 기존 방식: 고정 분기 로직
if kubectl rollout status deployment/app --timeout=2m; then
  echo "Success"
else
  kubectl rollout undo deployment/app
fi

# Claude의 Computer Use: Claude가 로그를 읽고 패턴을 평가해서 판단
# "배포가 진행 중입니다. 3개 Pod이 보류 중인 이유는 이미지가 손상되었기 때문이 아니라
#  노드 메모리가 부족하기 때문입니다. 클러스터를 먼저 스케일링한 후 롤아웃을 재개합니다."
```

## 에이전트형 작업의 권한 모델

CLI 환경에서 Auto Mode는 Claude가 작업을 다음과 같이 분류할 수 있는 능력에 의존합니다:

- **안전**: 읽기 전용 명령, 승인된 스크립트, 포매팅 작업
- **검토 필요**: 파일 생성, 외부 API 호출, 자격증명 접근
- **차단**: 명시적 화이트리스트 등록 없는 파괴적 작업

프로젝트에서 이를 활성화하려면 `settings.json`에 권한 화이트리스트를 설정합니다:

```json
{
  "permissions": {
    "mode": "auto",
    "allowlist": {
      "bash": [
        "kubectl get",
        "kubectl describe",
        "docker ps",
        "npm run test",
        "git log",
        "git diff"
      ],
      "fileOperations": [
        ".claude/cache",
        "build/",
        "dist/"
      ]
    },
    "blocklist": [
      "rm -rf /",
      "docker run --privileged",
      "export AWS_ACCESS_KEY_ID"
    ]
  }
}
```

이 설정은 명확합니다. 모호함이 없습니다. Claude가 화이트리스트와 일치하는 명령을 만나면 실행합니다. 아니면 확인을 요청합니다.

팀 개발에서는 이를 버전 관리에 커밋하세요. 자율 에이전트에서 환경 간 일관성은 필수입니다.

## 실전 워크플로우: 안전한 도입 패턴

**Phase 1: 읽기 전용 워크플로우 (첫째 날)**

상태 변경 없는 진단 작업부터 시작합니다:

```bash
# Claude가 자율적으로 처리할 수 있는 안전한 작업:
# - 컴플라이언스 감시를 위한 Git 이력 분석
# - 로그 파싱 및 장애 패턴 식별
# - Database 통계 쿼리 (변경 없음)
# - 종속성 트리 리포트 생성

claude-code --auto task "배포 로그를 분석해서 장애 모드를 정리해줘"
```

읽기 전용 작업에 대해 `mode: "auto"`를 설정합니다. 일주일간 감시 로그를 관찰합니다. 이 단계에서는 예상 밖의 결과가 발생하지 않습니다.

**Phase 2: 제어된 상태 변경 (둘째 주)**

명확한 복구 경로를 가진 작업으로 확대합니다:

```bash
# Claude가 자율적으로 수행할 수 있는 작업:
# - 코드 포매팅 (git으로 복구 가능)
# - 테스트 실행 및 단순 장애 수정
# - 종속성 락파일 업데이트 (복구 가능)
# - 파일 구조 재조정 (복구 가능)

claude-code --auto task "이 모듈의 에러 처리를 리팩터링하고 테스트를 실행해줘"
```

핵심: 모든 작업이 git으로 추적됩니다. Claude가 실수하면 `git revert`하고 반복합니다. 이것이 Auto Mode의 "가역적 작업" 원칙을 규모에 맞춰 적용한 형태입니다.

**Phase 3: 무인 CI/CD 통합 (셋째 주 이후)**

Phase 1과 2를 검증한 후에만 CI/CD에 통합합니다:

```yaml
# GitHub Actions 워크플로우 예시
name: Deploy with AI Agent

on:
  workflow_dispatch:

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Run Claude Code agent
        run: |
          claude-code --auto --timeout 30m task \
            "스테이징에 배포하고 스모크 테스트를 실행한 후 상태를 보고해줘"
      - name: Audit log
        if: always()
        run: cat .claude/audit.log
```

`--timeout` 플래그는 폭주 에이전트 방지. 감시 로그는 필수입니다—모든 배포 결정이 기록됩니다.

## Auto Mode에서 실제 실행되는 것

CLI 워크플로우에 대해 Auto Mode를 활성화하면 Claude는 다음에 접근할 수 있습니다:

1. **명령 실행**: Bash, Python, Node 등의 실행 및 출력 캡처
2. **파일시스템 읽기**: 소스 코드, 설정, 로그 접근 (화이트리스트 대상)
3. **환경 검사**: 시스템 상태 조회, 도구 버전 확인
4. **다단계 추론**: 한 명령의 결과를 사용해 다음 계획 수립

Claude가 얻지 못하는 권한:

- 자격증명 (AWS 키, Database 암호 불가) — IAM 역할 및 임시 토큰 사용
- 임의 네트워크 접근 — 사전 승인된 작업만
- Sudo 또는 상승된 권한 — 명시적으로 차단
- 프로젝트 트리 외 쓰기 (설정 캐시 디렉토리 제외)

제약 목록이 보안 모델입니다. 이를 활용하세요.

## 실제 예시: 종속성 업데이트 에이전트

Auto Mode의 실제 작동을 보여주는 구체적 워크플로우입니다:

```bash
# 작업: Node.js 종속성 전부 업데이트, 테스트 실행, 장애 보고
claude-code --auto task "
1. 현재 npm audit 상태 확인
2. 'npm update' 드라이런 실행
3. package-lock.json diff로 Breaking Change 식별
4. 각 메이저 업데이트에 대해 전체 테스트 스위트 실행
5. 테스트 실패 시 에러 조사 및 수정 제안
6. 최종 상태를 수동 개입이 필요한 특정 패키지와 함께 보고
"
```

Claude는 자율적으로:

1. `npm audit` 실행해서 출력 파싱
2. `npm update --dry-run` 실행해서 diff 캡처
3. 각 Breaking Change의 체인지로그 확인
4. `npm test` 실행해서 장애 분석
5. 테스트 실패 시 스택 트레이스를 읽고 관련 코드 검사
6. 최소한의 수정 제안하거나 사람 검토로 에스컬레이션
7. 의사결정 트리를 포함한 구조화된 리포트 생성

전체 흐름이 몇 분 만에 완료됩니다—각 단계에서 사람 승인 없이. 리포트를 검토하고, 변경을 승인하고, 병합합니다.

기존 승인 게이트웨이로는 불가능합니다. Auto Mode로는 일상입니다.

## 안전 점검 및 감시 로그

모든 에이전트형 CLI 작업은 다음을 생성합니다:

1. **작업 로그**: 실행된 모든 명령, 입출력
2. **결정 트레이스**: 각 작업에 대한 Claude의 추론
3. **가역성 평가**: 어떤 변경이 복구 가능하고 어떤 것이 영구적인지
4. **이상 플래그**: 예상 흐름에서 벗어난 작업

다음을 통해 접근합니다:

```bash
# 마지막 작업의 감시 로그 확인
claude-code logs --last

# 컴플라이언스 검토를 위한 구조화된 JSON 내보내기
claude-code logs --export json --last | jq '.actions[] | {time, command, status}'

# 패턴으로 로그 검색
claude-code logs grep "database" --last 10
```

규제 환경 (금융, 의료, SOC 2)에서는 이 감시 로그가 선택사항이 아니라 컴플라이언스 주장의 근거입니다. 백업 필수입니다.

## AI 에이전트 도입 시 조직적 과제

Auto Mode와 Computer Use를 팀 전체에 도입할 때는 기술 설정 이상으로 조직적 과제가 발생합니다.

**신뢰의 점진적 구축**

기술팀이 AI 에이전트를 신뢰하기까지는 시간이 필요합니다. 초기에 에이전트의 판단에 회의적인 팀원이 많은 것은 자연스럽습니다. 여기서 핵심은 투명성입니다. 모든 판단이 감시 로그에 기록되고, 팀 전체가 이를 검증할 수 있다는 사실이 신뢰 형성의 기초가 됩니다.

Kubernetes 환경의 배포를 예로 들면, Claude가 롤링 업데이트 중 자동으로 판단을 내렸을 때, 그 판단 과정이 팀 전체에 시각화됩니다. 팀이 "왜 그렇게 판단했는가"를 검증하고, 점차 신뢰가 높아집니다.

**에이전트 판단의 한계 인식**

Claude의 Computer Use가 완벽하지 않다는 것도 명확합니다. 특히 다음의 경우 인간 개입이 필요합니다:

- 비즈니스 로직 관련 판단 ("이 기능을 활성화할 것인가" 결정)
- 여러 정답이 존재할 때의 우선순위 결정
- 조직 정책 준수 관련 규칙 적용
- 예측 불가능한 외부 API 동작 대응

Auto Mode를 도입할 때는 이러한 "판단의 한계"를 사전에 정의하고, CLAUDE.md나 설정 파일에 명기하는 것이 중요합니다.

## 실제 구현 패턴과 성공 사례

일본을 포함한 여러 국가의 기업에서 AI 에이전트를 본격적으로 도입하기 시작했습니다.

**패턴 A: Microservice 환경에서의 Deployment 파이프라인**

여러 microservice를 Kubernetes 클러스터에 배포하는 시나리오에서 Claude는 다음을 자동화합니다:

1. 새 이미지 빌드 확인
2. 각 서비스 간 종속성 체크
3. Blue-Green 배포 실행
4. 헬스체크 모니터링
5. 문제 감지 시 롤백 판단

기존에는 각 단계에서 인간이 게이트키퍼 역할을 했습니다. Auto Mode에서는 명확한 실패 조건을 제외하고 모든 것이 에이전트에 의해 처리됩니다. 배포 시간은 실제로 **40~60% 단축**됩니다.

**패턴 B: Infrastructure-as-Code 관리 및 감시**

Terraform으로 작성된 인프라 설정을 Claude가 주기적으로 감시합니다:

```bash
claude-code --auto task "
1. 프로덕션 환경의 Terraform 설정 점검
2. Best Practice 준수 확인
3. Security Group, IAM Policy 설정 감시
4. 변경 필요 부분을 리포트로 생성
5. 승인 대기 중인 변경 사항 알림
"
```

이 자동 감시를 통해 보안 오류 감지 시간이 크게 단축되고, 컴플라이언스 요구사항 대응이 용이해집니다.

**패턴 C: 로그 분석 및 성능 모니터링**

프로덕션 로그를 실시간으로 분석하고 성능 이상을 감지합니다:

```bash
claude-code --auto task "
1. 지난 1시간의 에러 로그를 수집
2. 에러 패턴을 분류 및 집계
3. 임계값을 초과한 항목 식별
4. 영향받은 사용자 범위 추정
5. 초기 대응 절차 자동 실행
"
```

## Webhani의 실전 경험

여러 클라이언트 조직에 에이전트형 CLI 워크플로우를 배포한 경험에서 다음 포인트가 중요함을 알았습니다:

1. **높은 신뢰도 · 낮은 위험 작업부터 시작** — 코드 포매팅, Lint, 테스트 실행. Claude의 판단에 대한 신뢰를 구축합니다.
2. **CLAUDE.md와 권한 설정에 투자** — 여기에 들인 시간은 즉시 프롬프트 감소로 돌아옵니다.
3. **감시 로그를 법적 문서처럼 다루기** — 저장하고, 버전 관리하고, 감시하세요. 배포가 잘못되면 로그가 증거입니다.
4. **타임아웃과 리소스 제한 사용** — 제한 없는 자율 에이전트는 버그입니다.
5. **되돌리기 흐름 테스트** — 무인 Auto Mode를 활성화하기 전에, 나쁜 배포를 5분 이내에 복구할 수 있는지 수동으로 확인하세요.
6. **팀 간 지식 공유 문화** — 어떤 작업이 자동화 가능한지, 어디서 주의가 필요한지를 팀 전체가 이해해야 합니다.

어려움을 겪는 팀은 "모든 작업 승인"에서 "완전 자율"로 한 번에 이동하려고 합니다. 성숙에는 점진적 접근이 필요하며, 조직 문화와 기술적 준비가 동일한 속도로 진행되는 것이 성공의 열쇠입니다.

## 단계별 롤아웃 전략

조직 전체에 Auto Mode를 도입할 때 다음과 같은 단계 전개가 효과적입니다:

**1주-2주: 파일럿 환경 검증**

특정 프로젝트팀에 한정하여 읽기 전용 작업부터 시작합니다. 매일 로그를 확인하고 Claude의 판단 패턴을 학습합니다.
