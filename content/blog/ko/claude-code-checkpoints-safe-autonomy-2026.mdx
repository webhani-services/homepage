Anthropic이 2026년 7월 초에 릴리스한 Claude Code v2.1.200에는 Changelog 상으로는 사소해 보이지만 실제로는 Agent 운영 방식을 바꾸는 두 가지 변경이 포함되어 있습니다. 첫째, `AskUserQuestion` 대화상자가 기본적으로 자동 진행하지 않게 되었습니다. Agent가 임의로 선택해 넘어가는 대신 사람의 답변을 기다립니다. 둘째, 지금까지 "default"라 불리던 Permission Mode가 CLI·VS Code Extension·JetBrains Plugin 전반에서 **Manual**이라는 이름으로 바뀌었습니다. 여기에 더해 Anthropic은 자율 개발을 위한 **Checkpoint** 기능을 배포하고 있습니다.

언뜻 보면 이 변경들은 "번거로움이 늘었다"처럼 읽힙니다. 하지만 실제로는 더 긴 자율 실행을 신뢰할 수 있게 만드는 Guardrail입니다. 본질적인 질문은 "Agent를 얼마나 자율적으로 만들 수 있는가"가 아니라 "잘못된 판단을 얼마나 싸게 되돌릴 수 있는가"입니다. Checkpoint가 이 질문에 답합니다.

## Checkpoint가 실제로 제공하는 것

Checkpoint는 의미 있는 작업 단위를 시작하기 전에 Agent가 확보하는 "복원 가능한 Snapshot"입니다. Agent가 잘못된 방향으로 세 단계를 진행했을 때(Contract를 깨는 Refactoring, 잘못된 파일 편집, 실패 Test 오독 등), 얽혀 있는 변경을 수동으로 되돌리는 대신 Checkpoint까지 되감습니다.

이는 자율성의 경제학을 바꿉니다. 기존에는 Agent 실수의 비용이 "알아차리기까지 얼마나 벗어났는가"에 비례했습니다. Checkpoint가 있으면 그 비용은 "직전 Snapshot까지의 거리"로 상한이 정해집니다. "신뢰하려면 모든 diff를 읽어야 한다"에서 "일단 돌려두고, 잘못된 구간만 몇 초 만에 되감으면 된다"로의 전환입니다.

Checkpoint는 Git의 대체재가 아닙니다. Git은 영속적이고 Review 가능한 History이고, Checkpoint는 Agent 자신의 탐색을 위한 Session 내 임시 안전망입니다. 둘 다 사용합니다.

```bash
# 영속적 경계: Agent에게 Task를 넘기기 전에 Commit
git add -A && git commit -m "checkpoint: 의존성 Upgrade 전"

# Agent는 작업하면서 Session 내 Checkpoint를 생성한다.
# 구간이 실패하면 Git Commit까지 전부 되돌리는 대신
# Agent Checkpoint까지 되감는다.
```

## 기본 이름이 "Manual"인 것의 의미

기본 Permission Mode 이름을 "Manual"로 바꾼 것은 단순한 Label 변경이 아니라 Signal입니다. 운영자에게 명확히 이렇게 전합니다 — "초기 상태에서 Agent는 행동하기 전에 확인을 받는다." 자율성은 Project마다 의도적으로 Opt-in 하는 것이지, 암묵적으로 물려받는 것이 아닙니다.

이것이 올바른 Default라고 봅니다. 팀이 Coding Agent를 도입할 때 가장 자주 보는 실패는, Test도 없고 명확한 Permission 경계도 없고 Rollback 예행연습도 하지 않은 Repository에서 곧바로 완전 자율로 뛰어드는 것입니다. Agent는 그럴듯해 보이는 무언가를 실행하고, 아무도 Review하지 않은 지점에 착지하며, 팀은 몇 달간 도구에 대한 신뢰를 잃습니다.

유용한 Mental Model은 3단계이며 `settings.json`에 담을 수 있습니다.

```json
{
  "permissions": {
    "mode": "manual",
    "allow": [
      "Bash(npm run test:*)",
      "Bash(npm run lint)",
      "Bash(git diff:*)",
      "Bash(git log:*)"
    ],
    "deny": [
      "Bash(rm -rf:*)",
      "Bash(git push --force:*)",
      "Bash(curl:*)"
    ]
  }
}
```

Read-only이고 Idempotent한 Command는 `allow`에 넣어 Test 실행마다 방해받지 않게 합니다. 파괴적·외부로 나가는 Command는 `deny`에. 그 외에는 전부 수동 확인으로 Fallback합니다. `AskUserQuestion` 변경도 이를 보강합니다. Agent가 진짜 판단 지점에 도달하면 추측하지 않고 멈춰서 묻습니다.

## Rollback을 전제로 한 자율 Loop

Client에게 권하는 Pattern은 Checkpoint와 Commit을 Loop의 척추로 다루고 나중에 덧붙이지 않습니다.

1. **깨끗한 Baseline을 Commit한다.** Agent는 항상 "그대로 출하 가능한 상태"에서 시작해야 합니다.
2. **Task를 좁게 Scope한다.** "이 Module의 Error Handling을 Refactoring하고 기존 Test를 Green으로 유지"는 "Error Handling을 개선"보다 낫습니다. 좁은 Scope는 되감을 Checkpoint가 가깝다는 뜻입니다.
3. **작업하면서 Checkpoint를 찍게 한다.** 각 Sub-step이 되감기 지점이 됩니다.
4. **결과는 분위기가 아니라 Test로 Gate한다.** Test Suite가 통과해야 비로소 Loop가 "완료"입니다 — Agent가 당신 없이 검증할 수 있는 Signal입니다.
5. **모든 Keystroke가 아니라 최종 diff를 Review한다.** Green Suite와 깨끗한 Baseline이 있으면 Live 중계를 지켜보는 대신 일관된 하나의 변경을 Review할 수 있습니다.

```bash
# "무인이지만 경계가 있는" 실행에 적합한 형태
git commit -am "baseline: feature Branch 깨끗함"

claude --permission-mode manual \
  "src/lib/http.ts의 Retry 로직을 Exponential Backoff로
   Refactoring해줘. 기존 Test는 모두 통과시킬 것.
   public 함수 Signature는 변경하지 말 것."

# Agent는 Step 사이에 Checkpoint를 찍으며 작업한다.
# 성공: diff 하나 Review. 실패: 되감고 재Scope.
npm run test && git diff
```

"public 함수 Signature는 변경하지 말 것"이라는 제약은 여기서 실제로 효과를 냅니다. Prompt 내의 명시적 경계는 Task별 제한에 있어 Permission 설정보다 저렴하고, Agent에게 명확한 실패 조건을 부여합니다.

## 그래도 자율이 무너지는 지점

Checkpoint는 실수를 싸게 되돌릴 수 있게 하지만, Agent의 판단 자체를 건전하게 만들지는 않습니다. 다음은 여전히 사람의 몫입니다.

- **Business 맥락을 동반한 판단.** 그 기능이 존재해야 하는지, 두 가지 타당한 동작 중 무엇이 옳은지, 상충하는 제약을 어떻게 우선순위화할지.
- **되돌릴 수 없는 Side Effect.** Local Workspace 밖으로 나가는 것 — Email 발송, Production Data 변경, Package 배포. 이미 보낸 Request는 Checkpoint로 되감을 수 없습니다.
- **조용한 Scope 확장.** Bug 수정을 부탁받은 Agent가 "친절하게" 인접 Code를 Refactoring하기도 합니다. 좁은 Prompt와 엄격한 `deny` List가 이를 억제합니다.

`AskUserQuestion`이 기본적으로 대기하게 된 변경은 바로 첫 번째 범주를 겨냥합니다. 진짜 분기점에서 Agent는 그럴듯한 추측으로 밀어붙이는 대신 사람을 위해 멈춥니다.

## Webhani의 견해

중요한 것은 "Agent가 더 자율적이 되고 있다"는 관점이 아니라, "Agent를 둘러싼 Tooling이 자율성을 되돌릴 수 있고 감사 가능하게 만드는 데 능숙해지고 있다"는 관점입니다. Checkpoint, Manual Default, Blocking 질문 대화상자는 모두 같은 방향을 가리킵니다 — 경계가 있고 회복 가능한 공간 안에서 Agent를 빠르게 움직이게 한다는 방향입니다.

지금 시작하는 팀에는 세 가지를 권합니다. Manual Default를 유지하고, 자신감이 붙는 만큼 `allow` List를 단계적으로 넓히세요. Agent Task마다 깨끗한 Baseline을 Commit해 Checkpoint가 되돌아갈 곳을 마련하세요. 그리고 의존하기 전에 의도적으로 Rollback을 한 번 예행연습하세요 — Test하지 않은 복구 경로는 복구 경로가 아닙니다. 이 세 가지를 잡으면 자율 Coding은 "신앙의 도약"이 아니게 되고, Workflow의 당연하고 지루한 일부가 됩니다. 그것이 바로 지향해야 할 상태입니다.
