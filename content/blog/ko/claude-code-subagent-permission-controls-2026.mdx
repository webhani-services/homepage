자율적으로 코드를 작성하는 Agent를 팀에 도입할 때 가장 먼저 부딪히는 문제는 "어디까지 자유를 줄 것인가"입니다. Subagent가 다른 Subagent를 제한 없이 생성하고, 승인 없이 위험한 동작을 실행할 수 있다면 Review 게이트를 우회하는 경로가 생깁니다. 2026년 7월 Claude Code 업데이트는 바로 이 통제를 위한 실용적인 장치를 여러 개 추가했습니다. 이 글에서는 파라미터 단위 권한 Rule, Dynamic workflow size 설정, Subagent 실행 전 Classifier 검사를 정리합니다.

## Tool(param:value) — 입력 파라미터를 매칭하는 권한 Rule

지금까지 권한 Rule은 Tool 단위로 동작했습니다. `Bash`를 허용할지 거부할지 정도의 입도였습니다. 새로 추가된 `Tool(param:value)` 문법은 Tool의 입력 파라미터 값으로 매칭할 수 있게 해주며, `*` Wildcard도 지원합니다.

가장 명확한 사용 사례는 Subagent가 사용하는 Model을 제한하는 것입니다. Cost 관리를 위해 Opus로 Subagent가 생성되는 것을 막고 싶은 경우가 있습니다.

```json
{
  "permissions": {
    "deny": [
      "Agent(model:opus)"
    ]
  }
}
```

`.claude/settings.json`에 이 Rule을 넣으면 `model: opus`를 지정한 Subagent 실행 Request가 거부됩니다. 반대로 특정 Model만 허용하는 방식도 가능합니다.

```json
{
  "permissions": {
    "allow": [
      "Agent(model:sonnet)",
      "Agent(model:haiku)"
    ],
    "deny": [
      "Agent(model:*)"
    ]
  }
}
```

이 예시에서는 Sonnet과 Haiku Subagent는 허용하고, 그 외 모든 Model 지정은 거부합니다. 더 구체적인 allow Rule이 넓은 Wildcard deny보다 우선 평가되므로, 명시적으로 허용한 Model 외에는 모두 차단됩니다.

같은 발상은 Bash Command의 인자에도 적용됩니다. 예를 들어 Production 환경에 대한 `kubectl` 호출만 막고 싶다면 Command 문자열 패턴으로 매칭할 수 있습니다. Wildcard를 쓰면 "특정 Flag를 포함한 호출만 거부"하는 것처럼 세밀한 제어가 현실적으로 가능해집니다.

## 실행 전 검사 — Subagent가 생성되기 전에 평가

Auto Mode(허용 판단을 자동화하는 Mode)에는 이전까지 한 가지 구멍이 있었습니다. Subagent가 먼저 실행된 뒤, 그 내부에서 차단 대상 동작을 Request할 수 있는 경로였습니다. 부모 Agent의 권한 검사를 우회해 자식 Agent가 위험한 동작에 도달할 여지가 있었습니다.

2026년 7월 업데이트에서 Subagent 실행 Request 자체가 실행 *전에* Classifier로 평가되도록 바뀌었습니다. 자식 Agent에게 무엇을 시키려는지를 생성하기 전에 판정합니다. "일단 Subagent를 만들어 두면 그 안에서는 자유롭게 할 수 있다"는 우회로가 막힌 것입니다.

실무적 의미는 분명합니다. 권한 Rule을 부모 Agent에 설정해 두었더라도 Subagent를 통해 우회되지 않는다는 보장이 한층 강해집니다. CI에서 무인으로 실행하는 Workflow에서는 이 사전 검사의 유무가 안전성을 크게 좌우합니다.

## Subagent 재위임 억제

또 다른 개선은 Subagent가 받은 Task를 통째로 다른 Subagent에게 다시 위임하기 어려워진 점입니다. 이전에는 Task를 받은 Subagent가 "이건 내 일이 아니다"라고 판단해 그대로 다음 Subagent에게 넘기는 동작이 보였습니다. 위임 체인이 깊어지면 실제로 누가 무엇을 했는지 추적하기 어렵고 Token 소비도 커집니다.

재위임이 억제되면 Task를 받은 Subagent가 그 자리에서 작업을 완료하는 경향이 강해집니다. 결과적으로 실행 경로가 얕아지고 Log 추적성이 올라갑니다. 팀에서 운영할 때 이 "경로의 얕음"은 Review Cost에 직결되므로, 눈에 띄지 않지만 중요한 변경입니다.

## Dynamic workflow size — 동적 Workflow의 규모 제어

`/config`에 `Dynamic workflow size` 설정이 추가되었습니다. Claude가 동적으로 생성하는 Workflow의 규모(생성하는 Subagent 수)를 small / medium / large로 지정합니다.

중요한 점은 이것이 "강제 상한"이 아니라 "권고성 가이드라인"이라는 것입니다. large를 선택해도 Task 성격에 따라 소수의 Subagent로 끝날 수 있고, small을 선택해도 필요하면 늘어납니다. 어디까지나 "대략 이 정도 규모로 동작해 달라"는 방향 제시입니다.

팀에서의 사용 기준은 다음과 같습니다.

- **small**: 단순 수정, Review, 조사. Token Cost를 낮추고 싶은 일상 Task.
- **medium**: 여러 File에 걸친 구현이나 병렬 진행이 가능한 중규모 Task.
- **large**: 대규모 Refactoring, 전수 감사, Migration 등 병렬화로 시간을 벌고 싶은 상황.

CI에서 무인 실행하는 Pipeline에서는 small~medium으로 두고, 대화형으로 큰 작업을 맡길 때만 large로 올리는 운영이 현실적입니다. 규모 지정은 Cost의 예측 가능성에 직결되므로, Project의 `.claude/settings.json`에서 Default를 고정해 두면 팀 내 편차가 줄어듭니다.

## webhani의 관점 — 통제는 "나중에 붙이면" 듣지 않는다

이 기능들의 공통점은 Agent의 자율성과 Review 가능성 사이의 균형을 잡기 위한 장치라는 점입니다. 저희가 고객 개발팀에 AI 코딩을 도입할 때 가장 먼저 정비하는 것은 권한의 경계입니다. 무엇을 허용하고 무엇을 막을지를 먼저 정해 두지 않으면, Agent의 출력을 나중에 Review로 잡아내는 운영이 되어 반드시 누락이 생깁니다.

구체적으로 다음을 권장합니다.

1. `.claude/settings.json`의 `deny` List에 Production 환경 쓰기 계열 동작과 Cost 영향이 큰 Model 지정을 명시한다.
2. `Dynamic workflow size` Default를 Project 단위로 고정하고, CI에서는 small~medium으로 둔다.
3. Auto Mode를 쓸 경우, Classifier 사전 검사가 활성화된 Version인지 확인한다.

권한 Rule을 Config File에 적어 두는 것은 단순한 안전장치가 아닙니다. "이 Project에서 Agent에게 무엇을 맡길 것인가"라는 의사결정을 Code로 남기는 일이기도 합니다. 팀의 누가 실행해도 같은 경계가 적용되므로 Review 부담이 줄고 감사도 쉬워집니다.

## 정리

2026년 7월 Claude Code 업데이트는 Agent를 개인 도구에서 팀 도구로 끌어올리는 방향의 변경입니다. `Tool(param:value)`를 통한 세밀한 권한 제어, Subagent 실행 전 Classifier 검사, 재위임 억제, `Dynamic workflow size` — 이들은 모두 화려하지는 않지만 여러 명이 자율 Agent를 운영할 때 효과를 냅니다. 도입 초기에 권한 경계를 Config File로 굳히고 규모 Default를 정해 두는 것이, 안전하게 자율성을 끌어내는 지름길입니다.
