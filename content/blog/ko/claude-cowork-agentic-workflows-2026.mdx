## Agent는 "코드를 쓰는 도구"에서 "일을 끝내는 동료"로

지난 2년간 AI Assistant의 발전은 주로 코딩 지원을 중심으로 이야기되어 왔습니다. 그러나 2026년에 들어서면서 무게 중심이 분명히 이동하고 있습니다. Anthropic의 Claude Cowork는 Task를 하나 넘기면 파일, 캘린더, 메일, 메신저 App, Web, 그리고 연결된 외부 Tool을 넘나들며 완료할 때까지 작업을 진행합니다. IDE 안에서 코드를 자동완성하던 단계에서, 업무 자체를 대행하는 단계로 나아간 것입니다.

webhani는 Client의 업무 자동화를 다수 진행해 왔는데, 이 변화는 단순한 기능 추가가 아니라 AI를 System에 어떻게 녹여 넣을지에 대한 설계 사상의 전환이라고 봅니다. 이 글에서는 Cowork가 상징하는 Agent 기반 Workflow의 실무적 의미와, 도입 시 실제로 효과를 내는 논점을 정리합니다.

## "Tool을 넘나든다"는 것의 기술적 의미

기존 Chat 방식 AI는 사용자가 붙여넣은 Context 안에서만 답하는 닫힌 존재였습니다. Agent 방식에서는 전제가 바뀝니다. Model 스스로 "다음에 어떤 Tool을 호출할지"를 판단하고, 실행 결과를 받아 다음 행동을 결정합니다. 이 Loop를 사람이 매 단계 확인하지 않고 돌릴 수 있다는 점이 본질입니다.

그 연결 계층을 표준화한 것이 MCP(Model Context Protocol)입니다. Cowork 같은 Agent가 외부 Tool과 안전하게 대화할 수 있는 이유는, Tool의 능력을 선언적으로 기술하는 공통 Interface가 있기 때문입니다. 자사 Tool을 Agent에 연결할 때도 이 선언적 기술이 출발점이 됩니다.

```json
// Agent에 연결할 Tool을 선언적으로 정의하는 예 (MCP Server 설정)
{
  "mcpServers": {
    "internal-crm": {
      "command": "node",
      "args": ["./servers/crm-server.js"],
      "env": { "CRM_API_BASE": "https://crm.internal.example.com" }
    },
    "billing": {
      "command": "python",
      "args": ["-m", "billing_mcp"],
      "env": { "BILLING_SCOPE": "read-only" }
    }
  }
}
```

주목할 점은 Tool마다 권한 Scope를 명시할 수 있다는 것입니다. 위 예시에서는 Billing System을 `read-only`로 제한하고 있습니다. Agent가 자율적으로 움직이는 이상, "무엇을 못 하게 할지"를 설계 단계에서 묶는 것이 곧 안전 설계가 됩니다.

## 자율 실행의 Default화가 뜻하는 것

2026년 7월 Update에서 Claude Code는 Amazon Bedrock, Google Vertex AI, Microsoft Foundry 각 기반 위에서 자율 실행 Mode(auto mode)를 Default로 활성화했습니다. 함께 Bedrock 쪽 Model도 Opus 4.8로 갱신되었습니다.

Default가 "매번 확인"에서 "자율 실행"으로 바뀐 것은 소소하지만 상징적입니다. 지금까지는 사람이 한 수마다 승인하는 것이 안전한 초기 설정이었습니다. 이것이 뒤집혔다는 건, Model의 판단 정확도와 실패 시 Rollback 설계가 실용 수준에 도달했다는 개발사 측의 판단을 뜻합니다.

다만 실무에서는 Default를 그대로 믿지 말기를 권합니다. 자율 실행의 이점이 큰 작업(정형적 Refactoring, Log 조사, Test 생성)과, 신중한 확인이 필요한 작업(Production 데이터 갱신, 외부 전송)을 나누고, 후자에는 명시적 Guard를 둡니다. Agent의 생산성은 "어디까지 맡길까"가 아니라 "어디서 멈출까"를 설계했는지로 갈립니다.

## 기업 도입에서 효과를 내는 건 Model 성능이 아니라 Governance

Anthropic은 같은 시기에 Claude Code와 Cowork를 정부기관용 Desktop(FedRAMP High 인증)에서 Public Beta로 제공하기 시작했습니다. 여기서 전면에 나온 것은 Model의 똑똑함이 아니라, 변조 탐지가 가능한 Audit Log, 관리자 제어, 지출 Governance 같은 운영 기능입니다.

이 우선순위는 시사적입니다. Agent를 조직에서 쓸 때 경영진과 정보 System 부서가 먼저 묻는 것은 "얼마나 똑똑한가"가 아니라 다음 세 가지입니다.

- **누가, 언제, 무엇을 Agent에게 실행시켰는지 추적할 수 있는가** (감사 가능성)
- **Agent가 접근할 수 있는 범위를 직무 단위로 제한할 수 있는가** (최소 권한)
- **폭주나 Cost 초과를 어떻게 탐지·차단하는가** (지출과 Rate 제어)

webhani가 Client에게 Agent 기반을 제안할 때도, 이 세 가지를 충족하지 못하는 구성은 Production 투입을 보류하도록 조언합니다. 생산성 이야기는 통제가 잡힌 뒤에야 의미가 있습니다.

## 현장에서 어떻게 조립할 것인가

Agent 기반 Workflow를 실무에 녹일 때 저희가 유효하다고 보는 순서는 다음과 같습니다.

1. **좁게 시작한다.** 우선 Read-only·저위험 Task(사내 Document 검색, Report 초안)부터 착수해, 실패 Cost가 낮은 영역에서 Agent의 거동을 관찰합니다.
2. **권한을 선언적으로 고정한다.** 어떤 Tool에, 어떤 Scope로 접근시킬지를 코드(설정 File)로 관리하고 Review 대상으로 삼습니다.
3. **관측점을 먼저 마련한다.** Agent의 각 Action을 Log화해 나중에 추적 가능한 상태를, 기능 추가보다 먼저 만듭니다.
4. **사람의 승인 Gate를 명시적으로 둔다.** 비가역 조작(삭제, 전송, 결제) 앞에서는 자율 실행이라도 멈추고 확인을 끼웁니다.

```text
[Task 투입] → [계획 수립] → [read-only Tool로 조사]
     → [승인 Gate: 비가역 조작 직전 정지] → [실행] → [Audit Log 기록]
```

이 형태로 두면 Model이 Opus 4.8에서 차세대로 갱신되어도, 통제 Framework는 바꾸지 않고 성능만 받아들일 수 있습니다.

## 정리

Claude Cowork가 보여주는 것은, AI Agent가 "똑똑한 자동완성 기능"에서 "업무를 넘나들며 완결하는 실행 주체"로 이동했다는 사실입니다. 기술적 열쇠는 MCP를 통한 표준화된 연결 계층이고, Business적 열쇠는 감사·권한·지출의 Governance에 있습니다.

webhani의 판단으로는, 2026년 후반의 Agent 도입은 Model 성능 비교보다 "어디까지 안전하게 맡길 수 있는 구성을 짤 수 있는가"로 성패가 갈립니다. 좁게 시작하고, 권한을 선언적으로 묶고, 관측과 승인 Gate를 먼저 마련하는 이 수수한 설계야말로 Agent 시대의 생산성을 실제로 끌어내는 토대가 됩니다.
