## AI 에이전트와 메모리 문제

Claude Code를 몇 주, 몇 달씩 같은 프로젝트에 사용하다 보면 눈치채는 패턴이 있습니다. 처음엔 괜찮은데, 시간이 지날수록 에이전트가 엉뚱한 제안을 하거나 이미 리팩토링된 코드 구조를 기준으로 답변하는 경우가 생깁니다.

이 문제의 원인은 에이전트 메모리가 자동으로 정리되지 않는다는 점입니다. 세션마다 새로운 정보가 쌓이는데, 오래된 정보나 중복된 정보가 제거되지 않으면 에이전트의 컨텍스트는 점점 노이즈로 가득 차게 됩니다.

Anthropic은 이 문제를 해결하기 위해 Managed Agents API에 **Dreaming**이라는 기능을 리서치 프리뷰로 도입했습니다. 인간이 수면 중에 기억을 통합하고 정리하는 과정에서 착안한 이 기능은, 에이전트의 메모리를 주기적으로 압축하고 최적화합니다.

## AI 에이전트 메모리 관리의 실제 문제점

Managed Agents API를 사용하면 에이전트는 세션을 넘어 기억을 유지할 수 있습니다. 코드베이스의 구조, 팀이 사용하는 기술 스택, 과거에 해결한 버그 패턴 등이 메모리에 저장됩니다. 이론적으로는 매우 유용합니다.

하지만 실제로는 다음과 같은 문제가 발생합니다.

**중복 항목 누적**: 에이전트는 "이 프로젝트는 TypeScript strict mode를 사용한다"는 정보를 여러 세션에 걸쳐 반복해서 기록할 수 있습니다. 내용은 같지만 별도의 메모리 항목으로 저장되어 불필요한 토큰을 소비합니다.

**오래된 정보 잔존**: 6개월 전에 Redux를 사용하다가 Zustand로 마이그레이션했다면, 에이전트의 메모리에는 Redux 관련 아키텍처 노트가 그대로 남아 있습니다. 이 정보는 더 이상 유효하지 않을 뿐만 아니라, 에이전트가 잘못된 제안을 하도록 유도할 수 있습니다.

**컨텍스트 비대화**: 메모리 항목이 늘어날수록 매 API 호출 시 소비되는 토큰이 증가합니다. 비용 상승은 물론, 너무 많은 정보가 컨텍스트에 포함되면 모델의 응답 품질도 떨어집니다.

## Dreaming이 해결하는 방법

Dreaming은 에이전트의 메모리 저장소에 대해 다음 세 가지 작업을 수행합니다.

**1. 중복 통합**: 의미적으로 유사한 항목들을 하나의 깔끔한 항목으로 병합합니다. 같은 내용이 반복 기록된 항목들이 단일 항목으로 정리됩니다.

**2. 오래된 항목 삭제**: 설정한 기간 동안 참조되지 않은 항목이나 신뢰도가 낮아진 항목을 자동으로 제거합니다.

**3. 컨텍스트 압축**: 정보량은 유지하면서 토큰 수를 줄이는 방향으로 항목을 재구성합니다.

핵심은 이 작업이 에이전트가 "쉬는 시간"에 백그라운드로 실행된다는 점입니다. 사용자 입장에서는 별도의 작업 없이, 다음 세션을 시작하면 더 가볍고 정확한 메모리를 가진 에이전트와 대화하게 됩니다.

## Managed Agents API 코드 예시

TypeScript로 Dreaming 기능을 활성화한 에이전트를 만들고 운용하는 예시입니다.

```typescript
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

// Dreaming이 활성화된 에이전트 생성
async function createAgentWithDreaming() {
  const agent = await client.beta.agents.create({
    model: "claude-opus-4-5",
    name: "dev-assistant",
    description: "장기 개발 프로젝트용 지속 에이전트",
    instructions: `
      당신은 이 코드베이스를 깊이 이해하는 개발 어시스턴트입니다.
      세션이 바뀌어도 이전에 학습한 프로젝트 맥락을 유지하고,
      일관된 방향으로 도움을 제공하세요.
    `,
    memory: {
      enabled: true,
      dreaming: {
        enabled: true,
        trigger: "session_end", // 세션 종료 후 자동 실행
      },
    },
  });

  console.log(`에이전트 생성 완료: ${agent.id}`);
  return agent;
}

// 세션 실행
async function runSession(agentId: string, message: string) {
  const session = await client.beta.agents.sessions.create(agentId, {
    metadata: {
      project: "my-saas-app",
      sprint: "sprint-12",
    },
  });

  const response = await client.beta.agents.sessions.messages.create(
    agentId,
    session.id,
    {
      role: "user",
      content: message,
    }
  );

  return response;
}

// 수동으로 Dreaming 실행 (예: CI/CD 파이프라인 또는 주간 배치 작업)
async function runDreaming(agentId: string) {
  const result = await client.beta.agents.dream(agentId, {
    // "conservative" | "balanced" | "aggressive"
    // 처음에는 conservative로 시작하는 것을 권장합니다
    consolidation_level: "balanced",
    // 며칠 이상 참조되지 않은 항목을 삭제할지 설정
    stale_threshold_days: 21,
  });

  console.log("Dreaming 완료:");
  console.log(`  이전: ${result.entries_before}개 항목`);
  console.log(`  이후: ${result.entries_after}개 항목`);
  console.log(`  병합: ${result.merged_count}개`);
  console.log(`  삭제: ${result.deleted_count}개`);

  return result;
}

// 현재 메모리 상태 확인
async function checkMemoryState(agentId: string) {
  const memory = await client.beta.agents.memory.list(agentId, {
    limit: 10,
    order: "desc",
  });

  console.log(`현재 메모리 항목 수: ${memory.data.length}`);
  memory.data.forEach((entry, i) => {
    console.log(`[${i + 1}] ${entry.content.substring(0, 80)}...`);
  });
}

// GitHub Actions 등에서 주간 실행하는 유지보수 작업 예시
async function weeklyMaintenanceJob() {
  const AGENT_ID = process.env.AGENT_ID!;

  console.log("주간 에이전트 메모리 최적화 시작...");
  await runDreaming(AGENT_ID);
  await checkMemoryState(AGENT_ID);
  console.log("완료.");
}
```

코드에서 주목할 점은 두 가지입니다.

첫째, `trigger: "session_end"` 설정은 세션이 끝날 때마다 자동으로 Dreaming을 실행합니다. 이 작업은 비동기로 백그라운드에서 진행되므로 사용자 응답에 영향을 주지 않습니다.

둘째, `client.beta.agents.dream()` 호출을 통해 수동으로 Dreaming을 트리거할 수 있습니다. 주간 CI 작업이나 스프린트 종료 시점에 더 깊은 수준의 통합을 실행하는 데 유용합니다.

## Claude Code 장기 프로젝트 활용 팁

실제로 Claude Code를 6개월 이상 사용하는 팀이라면 Dreaming 도입 여부를 진지하게 고민해볼 만합니다.

**에이전트 메모리를 의도적으로 구축하세요.** 초기 세션에서 에이전트에게 프로젝트의 핵심 컨텍스트를 체계적으로 알려주면, 이 정보가 메모리에 저장됩니다. Dreaming은 이 초기 지식을 유지하면서 불필요한 노이즈를 제거합니다.

**스프린트 종료 시점에 수동 Dreaming을 실행하세요.** 스프린트마다 코드베이스가 변화하므로, 스프린트 종료 후 Dreaming을 실행하면 오래된 정보가 정리되고 새로운 아키텍처 상태가 반영됩니다. GitHub Actions나 스크립트로 자동화하면 관리 부담이 줄어듭니다.

**`consolidation_level`은 단계적으로 높이세요.** `"conservative"`로 시작해서 Dreaming 결과를 모니터링하다가, 중요한 정보가 삭제되지 않는다는 확신이 생기면 `"balanced"`나 `"aggressive"`로 조정하는 것이 안전합니다.

**메모리에 민감한 정보가 들어가지 않도록 주의하세요.** 에이전트 메모리는 API Key, 비밀번호, 개인정보가 포함되지 않도록 설계해야 합니다. 에이전트 Instructions에 이에 대한 명확한 지침을 포함하는 것을 권장합니다.

## webhani의 관점과 권장사항

Dreaming은 "에이전트를 일회성 도구가 아니라 성장하는 동료"로 만들기 위한 인프라입니다. 단순히 메모리를 저장하는 기능을 넘어, 그 메모리의 품질을 시간이 지나도 유지하는 메커니즘입니다.

**지속 에이전트를 사용해야 하는 경우:**
- 3개월 이상 지속되는 프로젝트
- 동일한 팀이 같은 코드베이스를 반복적으로 다루는 환경
- 도메인 지식이 복잡하고 매번 컨텍스트를 설명하는 비용이 큰 경우
- Claude Code를 개발 워크플로우의 핵심 도구로 사용하는 경우

**Stateless 에이전트가 적합한 경우:**
- 단일 작업으로 명확히 정의된 태스크 (문서 번역, 데이터 변환 등)
- 민감한 데이터를 다루어 세션 간 정보 유지가 불가한 경우
- 코드베이스 변화가 너무 빠르게 기억이 금방 무효화되는 경우
- 재현성이 중요해서 메모리를 변수로 두고 싶지 않은 경우

**현재 상태 주의사항:**

Dreaming은 현재 리서치 프리뷰 단계입니다. Managed Agents API 자체에 접근하려면 Anthropic에 별도 신청이 필요하며, API 인터페이스는 GA(일반 출시) 전에 변경될 수 있습니다. 프로덕션 환경 적용 시에는 안정 버전 출시 이후를 권장합니다.

## 마무리

메모리 관리는 AI 에이전트가 단기 도구에서 장기 파트너로 진화하는 데 반드시 해결해야 하는 과제입니다. Dreaming은 그 과제에 대한 Anthropic의 첫 번째 실용적인 답변입니다.

수면 중 기억을 통합하는 인간의 뇌처럼, Dreaming은 에이전트가 "쉬는 동안" 스스로 지식을 정리합니다. 이 기능이 Claude Code에 완전히 통합되는 날, 개발자와 AI 에이전트의 협업 방식은 지금과 크게 달라질 것입니다.

webhani는 내부 프로젝트에서 이 기능을 이미 실험 중이며, 실제 사용 경험과 팁을 앞으로도 공유할 예정입니다.
