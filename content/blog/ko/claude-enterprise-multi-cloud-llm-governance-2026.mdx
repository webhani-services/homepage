1년 전만 해도 Client가 가져오는 LLM 관련 질문은 대부분 Prompting에 관한 것이었습니다. 어떤 모델을 쓸지, Temperature를 얼마로 둘지, 고집 센 작업에서 더 나은 답을 어떻게 끌어낼지. 2026년에도 그 질문들이 사라진 것은 아니지만, 이제 Engineering Manager의 밤잠을 설치게 하는 질문은 따로 있습니다. 조직적인 질문입니다. 누가 어떤 모델을, 어느 Cloud를 통해, 어떤 예산으로 호출할 수 있는가. 그리고 무언가 잘못됐을 때 우리는 그것을 어떻게 알아차리는가.

이 변화는 모델이 전달되는 방식이 실제로 바뀐 것과 맞물려 있습니다. 예를 들어 Anthropic의 Claude 모델은 이제 Vendor 자체 API뿐 아니라 Amazon Bedrock과 Google Cloud 안의 관리형 서비스로도 접근할 수 있고, Anthropic은 비용 가시성을 위한 Admin 측 통제 기능을 계속 추가하고 있습니다. 주요 모델 Provider 전반이 대체로 비슷한 상황입니다. 그 결과 한 회사가 *같은* 모델 계열을 세 개의 서로 다른 정문으로 호출하게 되고, 각 정문은 저마다의 청구, IAM 모델, Logging을 가집니다. Governance는 그 정문들을 하나로 묶는 일입니다. 이 글은 그 작업에 우리가 Client와 함께 쓰는 Framework입니다.

## Multi-Cloud LLM 접근은 계획하지 않아도 생긴다

누구도 처음부터 모델 접근을 파편화하려고 하지 않습니다. 그것은 점진적으로 벌어집니다. Data 팀은 이미 Google Cloud에서 일하니, 같은 VPC 안에 머물려고 Pipeline이 거기서 모델을 호출합니다. Application 팀은 AWS에 깊이 들어가 있으니, Private Network에 Traffic을 유지하고 기존 IAM Role을 물려받으려고 Production Service가 Bedrock을 씁니다. 그사이 Product Manager는 Prototyping을 위해 Vendor의 직접 API에 가입하고 법인카드로 결제했는데, 그 Key가 어느새 넉 달째 돌아가는 "임시" 내부 도구를 조용히 돌리고 있습니다.

각각의 결정은 국소적으로는 합리적입니다. Data와 같은 Cloud 안에서 추론하면 Egress 비용과 Data Residency 골칫거리를 피할 수 있습니다. 플랫폼 기본 제공 방식을 쓰면 이미 신뢰하는 Audit Logging, Key 관리, Network Policy를 재사용합니다. 그러나 조직 전체로 합산하면 서로 무관한 세 개의 청구 라인, 세 개의 권한 체계, 그리고 "이번 달 AI에 얼마를, 무엇에 썼는가"에 답할 단일 지점이 없는 상태가 됩니다.

흔한 실수는 이 상황을 없애야 할 문제로 취급하는 것입니다. 앞의 이유들이 실제이기 때문에 모두를 한 경로로 통합할 수는 없습니다. 목표는 파편화를 없는 척하는 것이 아니라, 파편화 *위에* Governance Layer를 얹는 것입니다.

## Gateway 패턴: 정문은 하나, 뒷단은 여럿

가장 유용한 아키텍처 결정 하나는, 각 Service가 Provider를 직접 호출하게 두는 대신 Application Traffic을 내부 Gateway로 흐르게 하는 것입니다. Gateway는 얇은 Service입니다. 직접 만들든 기성품을 쓰든, 정규화된 요청을 받아 어떤 뒷단을 쓸지 정하고, 알맞은 자격 증명을 붙이고, 무슨 일이 있었는지 기록합니다.

개념적으로는 이렇게 생겼습니다.

```typescript
// 모든 내부 호출자는 Provider SDK가 아니라 Gateway를 친다.
async function complete(req: LlmRequest, ctx: CallerContext) {
  const route = pickRoute(req.model, ctx.team); // 예: bedrock | gcp | vendor
  const started = Date.now();

  const res = await route.client.send(req);

  await recordUsage({
    team: ctx.team,
    model: req.model,
    route: route.name,
    inputTokens: res.usage.inputTokens,
    outputTokens: res.usage.outputTokens,
    latencyMs: Date.now() - started,
  });

  return res;
}
```

여기에 특별한 것은 없습니다. 그리고 그게 핵심입니다. 모든 호출이 함수 하나를 지나게 되면, 이전에는 불가능했던 네 가지가 사소해집니다. 팀 단위 비용 귀속, 호출자를 건드리지 않는 경로 전환, 한곳에서의 Policy 적용, 통합된 사용 Log 확보입니다. `pickRoute`가 Multi-Cloud 현실을 흡수하는 지점입니다. Data 팀의 요청은 Google Cloud로, Production 요청은 Bedrock으로 풀리고, 호출자는 그것을 알 필요도 신경 쓸 필요도 없습니다.

Gateway는 Failover 이음새도 줍니다. 한 뒷단이 오류를 뱉거나 Rate Limit에 걸리면, `pickRoute`는 호출자 코드 변경 없이 같은 모델의 다른 Provider로 넘어갈 수 있습니다. 우리는 이것을 우선 신뢰성 기능으로, 그다음에 비용 기능으로 봅니다.

## 비용 가시성: 더 큰 Dashboard보다 귀속이 먼저다

Client가 "AI 청구서가 통제 불능"이라고 말할 때, 문제는 거의 항상 절대 숫자가 아닙니다. 아무도 그것을 분해하지 못한다는 게 문제입니다. 월 4만 달러가 "Production 추론 2.8만, Analytics Batch 9천, 실험 3천"으로 깔끔하게 쪼개지면 그것은 다룰 수 있는 경영 입력값입니다. 같은 4만 달러가 불투명한 한 줄이면 불안과 잘못된 결정의 원천이 됩니다.

Provider 측 비용 통제, 즉 Cloud와 Vendor가 이제 노출하는 Admin 화면은 필요하지만 충분하지 않습니다. 그것은 *그* Provider가 얼마를 청구했는지 알려주지만, Analytics 팀의 야간 작업이 원인이라는 것은 알려주지 못합니다. Provider는 여러분의 조직도를 모르기 때문입니다. 귀속은 여러분 쪽에서 일어나야 하고, 위의 `recordUsage` 호출이 정확히 그것을 담아냅니다. 팀별·모델별 Token 수를 자체 저장소에 갖고 있으면 월별 집계는 단순한 Query입니다.

```sql
SELECT team,
       model,
       sum(input_tokens)  AS input_tokens,
       sum(output_tokens) AS output_tokens,
       count(*)           AS calls
FROM llm_usage
WHERE created_at >= date_trunc('month', now())
GROUP BY team, model
ORDER BY output_tokens DESC;
```

이 Token 수에 각 경로의 공개된 Token 단가를 곱하면, Cloud Invoice를 기다리지 않고도 팀별 추정 비용을 얻습니다. 이 숫자가 청구서와 센트 단위까지 맞지는 않습니다. Provider마다 반올림과 묶음 방식이 다르기 때문입니다. 그러나 방향은 정확하고, 무엇보다 실시간으로 얻을 수 있습니다. 폭주하는 작업을 Invoice가 아니라 둘째 날에 잡아낼 수 있게 해주는 것이 바로 이 지점입니다.

## Prompt가 아니라 Gateway에 있어야 할 Guardrail

모든 호출이 한곳을 지나므로, Gateway는 Service마다 흩어놓으면 괴로운 몇 가지 통제의 올바른 거처이기도 합니다.

- **이빨이 있는 예산.** 누군가에게 메일만 보내는 무른 예산은 권고일 뿐입니다. Gateway에서 팀별 지출을 월 상한과 비교해 추적하고, 상한에 닿으면 비필수 요청을 거부하거나 하향 조정하십시오. 내부 도구를 느리게 만드는 편이 사후에 다섯 자리 초과를 발견하는 것보다 훨씬 낫습니다.
- **팀별 모델 Allowlist.** 값싼 모델로 충분한 작업에 모든 팀이 가장 비싼 모델을 부를 수 있어야 하는 것은 아닙니다. 그것을 사람들이 잊어버리는 Code Review 관습이 아니라 `pickRoute`의 Policy로 새기십시오.
- **민감 데이터를 위한 Logging 경계.** Gateway는 특정 요청 범주를 자사 Cloud와 Network 안의 뒷단으로만 보내도록 강제하기에 자연스러운 자리입니다. 규제 데이터가 원치 않는 경로로 새지 않게 지킵니다.

우리는 이 통제들을 의도적으로 Prompt와 개별 Service 밖에 둡니다. Prompt 수준 규칙은 손쉽게 우회되고 Audit이 불가능하며, Service별 규칙은 두 팀이 각자 구현하는 순간 어긋나기 시작합니다. Gateway는 한 번 쓴 Policy가 실제로 유지되는 유일한 병목입니다.

## 지금 시작하는 팀에 권하는 것

이제 막 시작한다면, 무거운 Platform을 짓고 싶은 충동을 눌러야 합니다. 사각지대를 없애는 가장 작은 것부터 시작하십시오.

1. **모든 Application Traffic 앞에 얇은 Gateway를 두십시오.** 100줄짜리 Service라도 사용량 Logging과 귀속을 시작하기에 충분합니다. 요구사항이 분명해지면 나중에 성숙한 Open Source Gateway를 채택하면 됩니다.
2. **첫날부터 자체 저장소에 사용량을 기록하십시오.** 이번 달에 담지 못한 Token 수는 영영 사라집니다. 가장 적은 노력으로 가장 큰 지렛대를 얻는 단계입니다.
3. **가진 Provider 측 비용 통제를 모두 켜십시오.** 단, 주된 방어선이 아니라 안전망으로 삼으십시오. 주된 방어선은 여러분 자신의 귀속입니다.
4. **어느 팀이 어느 Cloud에서 어느 모델을 써도 되는지 적어 두고,** 그 목록을 Routing에 새기십시오. 적히지 않은 Policy는 Policy가 아닙니다.

2026년에 LLM Governance를 잘 다루는 조직은 가장 영리한 Prompt를 가진 곳이 아닙니다. 모델 접근이 곧 Infra라고 일찍 결론짓고, 이미 Database와 Cloud Compute에 적용하던 귀속·예산·접근 규율을 똑같이 부여한 곳입니다. Gateway는 실제 팀이 늘 만들어내는 파편화를 얼리지 않고도 거기에 도달하는 방법입니다.

*LLM 지출을 설명하기 어려워졌거나 모델 접근이 어느새 Cloud를 가로질러 파편화됐다면, Gateway 설계부터 비용 귀속, 접근 Policy까지 — 이것이 바로 webhani가 Client의 얽힌 실타래를 푸는 종류의 아키텍처 작업입니다.*
