## 최고의 성능, 최고의 가격

Claude Fable 5는 SWE-bench Verified에서 약 95%의 점수를 기록하고 있습니다. 이는 현재 이용 가능한 코딩 모델 중 최고의 성능입니다. 하지만 가격도 함께 높습니다. 입력 토큰당 $10/백만 토큰, 출력 토큰당 $50/백만 토큰입니다. 팀이 매일 여러 번의 API 호출을 하는 환경에서는 이 비용이 무시할 수 없습니다.

webhani의 고객사들과 운영을 함께하며 배운 교훈은 단순합니다. Fable 5는 "모든 곳에 사용하는 모델"이 아니라 "언제 사용할 것인가를 결정하는 모델"이라는 점입니다.

## 난이도별 계층화 라우팅

현실적인 접근법은 작업의 복잡도에 따라 모델을 다르게 선택하는 것입니다.

**가벼운 작업** — 보일러플레이트 코드 생성, 기존 패턴의 간단한 확장, 포맷 변환. Claude Haiku나 이전 Opus로 충분합니다. 빠른 실행 시간, 낮은 토큰 비용.

**중간 수준 작업** — 버그 분석 초기 단계, 설계 검증, 테스트 작성. Claude Opus 또는 Sonnet이 비용과 추론 능력의 균형을 잘 맞춥니다.

**복잡한 작업** — 다단계 최적화 문제, 레거시 시스템의 깊은 디버깅, 새로운 아키텍처 설계. 여기서 Fable 5의 높은 가격이 정당화됩니다. 문제가 진정으로 어려울 때 모델의 우수한 추론 능력이 투자값을 증명합니다.

이러한 계층화 전략을 도입하면 일일 작업의 70-80%를 저비용 모델로 처리하고, 나머지 20-30%의 어려운 문제만 Fable 5에 할당할 수 있습니다. 토큰 소비량을 제어하면서도 팀의 생산성을 유지할 수 있습니다.

## 라우팅 시스템 구현

이 전략을 실제로 운영하려면 작업의 난이도를 평가하고 적절한 모델을 선택하는 라우터가 필요합니다. TypeScript로 구현한 예시입니다.

```typescript
interface CodeTask {
  description: string;
  codeContext?: string;
  estimatedComplexity?: number;
}

interface ModelConfig {
  model: string;
  maxInputTokens: number;
  costPerMInputTokens: number;
  costPerMOutputTokens: number;
}

const modelTiers: Record<string, ModelConfig> = {
  haiku: {
    model: "claude-3-5-haiku-20241022",
    maxInputTokens: 8000,
    costPerMInputTokens: 0.8,
    costPerMOutputTokens: 4,
  },
  sonnet: {
    model: "claude-3-5-sonnet-20241022",
    maxInputTokens: 200000,
    costPerMInputTokens: 3,
    costPerMOutputTokens: 15,
  },
  opus: {
    model: "claude-3-opus-20250219",
    maxInputTokens: 200000,
    costPerMInputTokens: 15,
    costPerMOutputTokens: 75,
  },
  fable: {
    model: "claude-fable-5",
    maxInputTokens: 300000,
    costPerMInputTokens: 10,
    costPerMOutputTokens: 50,
  },
};

function assessTaskComplexity(task: CodeTask): number {
  let score = 0;

  // 코드 양 평가
  if (task.codeContext) {
    const lineCount = task.codeContext.split("\n").length;
    score += Math.min(lineCount / 100, 20); // 최대 20점
  }

  // 키워드 기반 평가
  const complexKeywords = [
    "memory leak",
    "concurrency",
    "optimization",
    "race condition",
    "performance bottleneck",
    "architecture",
    "multi-tenant",
  ];
  const keywordMatches = complexKeywords.filter((kw) =>
    task.description.toLowerCase().includes(kw)
  ).length;
  score += keywordMatches * 15; // 키워드당 15점

  // 사용자 지정 난이도
  if (task.estimatedComplexity) {
    score += task.estimatedComplexity;
  }

  return Math.min(score, 100);
}

function routeTask(task: CodeTask): string {
  const complexity = assessTaskComplexity(task);

  if (complexity < 30) {
    return "haiku";
  } else if (complexity < 60) {
    return "sonnet";
  } else if (complexity < 85) {
    return "opus";
  } else {
    return "fable";
  }
}

async function solveTask(task: CodeTask): Promise<string> {
  const modelKey = routeTask(task);
  const config = modelTiers[modelKey];

  console.log(`Task complexity: ${assessTaskComplexity(task)}/100`);
  console.log(`Selected model: ${config.model}`);

  // API 호출 (구현 생략)
  // const response = await client.messages.create({
  //   model: config.model,
  //   max_tokens: 2000,
  //   messages: [{ role: "user", content: task.description }],
  // });

  return `Using ${config.model}`;
}

// 사용 예시
const debugTask: CodeTask = {
  description:
    "레거시 Node.js 앱의 메모리 누수 원인 파악. 멀티테넌트 요청 처리 중 Race Condition 의심.",
  codeContext: "// 수백 줄의 코드...",
  estimatedComplexity: 30,
};

routeTask(debugTask); // → "fable" 반환
```

이 라우터는 세 가지 관점에서 난이도를 평가합니다.

1. **코드량** — 다루는 코드가 클수록 강력한 모델의 이점이 큽니다.
2. **문제의 성질** — 병렬 처리, 최적화, 메모리 관리 같은 키워드는 높은 복잡도를 나타냅니다.
3. **명시적 지정** — 팀의 경험을 반영하여 엔지니어가 난이도를 미리 표시할 수 있습니다.

## 비용 제어 패턴

**Prompt Caching** — 동일한 문서나 코드를 반복 참조할 때 Prompt Caching을 활용하면 2회차 이후 90%의 비용 절감이 가능합니다. 대규모 코드베이스를 여러 번 분석해야 할 때 특히 효과적입니다.

**Context 스코핑** — 전체 로그 파일을 보내지 말고 관련 부분만 추출하세요. Fable 5의 Context Window는 넉넉하지만, 모든 토큰은 비용입니다. 정확하게 필요한 정보만 포함시키세요.

**Batch 처리** — 독립적인 여러 작업이 있다면 Batch API를 활용하세요. Batch 요금은 동기 호출 대비 약 50% 저렴합니다.

**단계적 에스컬레이션** — 먼저 Opus로 시도해보세요. Opus가 한계에 도달했을 때만 Fable 5를 투입하세요. "최대 성능부터 시작"하는 것보다 "필요할 때 업그레이드"하는 패턴이 총비용이 더 낮습니다.

## Fable 5를 사용해야 할 때

webhani 팀은 다음의 기준을 만족할 때만 Fable 5를 지정합니다.

- 기존 모델들이 해당 작업을 시도했지만 실패했습니다.
- 문제로 인해 8시간 이상의 엔지니어링 시간이 소요될 가능성이 있습니다.
- 영향이 Production 환경에 미치고 있습니다.

고객사 프로젝트의 월간 모델 이용률을 보면, Fable 5는 전체 API 호출의 5-15% 수준에 머물러 있습니다. 나머지 85-95%는 Opus와 Sonnet으로 처리되며, 대부분의 팀이 이 비율을 목표로 해야 합니다.

## 결론

Fable 5는 강력합니다. 하지만 그 강력함에는 대가가 따릅니다. 모델 사용 결정은 비즈니스 논리에 기반해야 합니다. 즉, "이 토큰 비용이 절약할 엔지니어링 시간보다 큰가?"라는 질문에 답해야 합니다.

계층화된 라우팅과 에스컬레이션 로직을 구현하면 Fable 5의 가치를 최대화하면서도 전체 비용을 예측 가능한 수준으로 유지할 수 있습니다. webhani는 이제 이러한 모델 선택 프레임워크를 고객사의 LLM 비용 전략의 일부로 제안하고 있습니다. 진정한 승리는 최고성능 모델에 접근할 수 있다는 것이 아니라, 정확히 언제 그것을 사용해야 하는지 아는 것입니다.
