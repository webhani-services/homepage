Anthropic이 Claude Managed Agents에 'Dreaming'이라는 기능을 추가했습니다. Agent가 유휴 상태일 때 비동기적으로 실행되며, 세션 간 Persistent Memory를 자동으로 정리합니다. 중복 항목을 병합하고, 오래된 정보를 삭제하는 방식입니다. 현재 Research Preview 단계이지만, 장기간 운용되는 Agent 시스템을 설계하고 있다면 지금 파악해 두는 것이 좋습니다.

## Persistent Memory가 어려운 이유

Agent에 세션 간 Memory를 부여하는 방법은 크게 두 가지입니다.

**외부 Store 방식**: Agent가 Tool을 통해 메모를 읽고 씁니다. Vector DB나 KV Store에 저장합니다.  
**Context 주입 방식**: 이전 세션의 Summary를 새 세션의 System Prompt에 삽입합니다.

두 방법 모두 공통적인 문제가 있습니다. 운용 기간이 길수록 Memory가 오염되고 비대해집니다.

- 동일한 정보가 조금씩 다른 형태로 중복 저장됩니다
- 과거엔 정확했지만 현재는 낡은 정보가 계속 남아있습니다
- Memory 양이 늘수록 관련 정보 검색 정확도가 떨어집니다

주기적인 수동 정리 없이는 이 문제가 계속 악화됩니다. Dreaming은 이 정리 작업을 자동화하는 접근입니다.

## Dreaming 동작 원리

Dreaming은 Agent가 활성 Request를 처리하지 않는 시간에 실행되는 Background Process입니다.

1. **중복 감지**: 의미적으로 중복된 항목을 식별합니다
2. **통합**: 중복 항목을 하나로 병합하고 최신 정보를 우선합니다
3. **노후화 판정**: 타임스탬프, 참조 빈도, 내용 모순을 기반으로 오래된 항목을 감지합니다
4. **삭제**: 노후화된 항목을 Memory Store에서 제거합니다

중요한 점은 이 Process가 사용자와의 대화 중에는 실행되지 않는다는 것입니다. 실시간 세션에서의 Latency에 영향을 주지 않습니다.

## Dreaming이 해결하지 못하는 것

Dreaming은 Memory 정리를 담당합니다. 전체 Memory Architecture 설계를 대체하진 않습니다. Production Agent 시스템을 구축하고 있다면 여전히 직접 설계해야 할 것들이 있습니다.

- **무엇을 저장할지**: 모든 중간 결과가 영구 저장될 필요는 없습니다
- **Retrieval 로직**: 어떤 Memory를 언제, 어떤 순서로 활용할지
- **범위 분리**: 사용자별, 프로젝트별, 도메인별 Memory 격리

Dreaming을 Agent Memory의 가비지 컬렉션이라고 생각하면 됩니다. 필수 인프라이지만, Agent가 무엇을 기억할 가치가 있는지는 알려주지 않습니다.

## Outcomes API와 기타 업데이트

Dreaming과 함께 Managed Agents에 추가된 다른 기능들도 Production 통합에서 실용적입니다.

- **API Rate Limit 상향**: 다수의 병렬 Agent가 필요한 워크로드 대응
- **Multi-Agent Orchestration API**: API로 Agent 간 협조 제어 가능
- **Outcomes API**: Webhook 또는 Polling으로 Agent 실행 결과 수신
- **Microsoft 365 연동 강화**: Excel/PowerPoint Add-in이 전체 대화 컨텍스트를 공유

Outcomes API는 장기 실행 작업에서 특히 유용합니다. Agent 완료 결과를 Polling 대신 Webhook으로 받을 수 있어, Orchestration 코드가 단순해집니다.

```python
from fastapi import FastAPI, Request

app = FastAPI()

@app.post("/agent-webhook")
async def receive_outcome(request: Request):
    payload = await request.json()

    agent_id = payload["agent_id"]
    status = payload["status"]  # "completed" | "failed" | "cancelled"

    if status == "completed":
        result = payload.get("result")
        await store_result(agent_id, result)
    elif status == "failed":
        error = payload.get("error")
        await handle_failure(agent_id, error)

    return {"ok": True}
```

## 자동 정리를 고려한 Memory 설계

Dreaming을 사용하거나 도입할 계획이라면 다음 패턴이 도움됩니다.

**중요도 태그를 명시적으로 관리하세요.** 규정 준수 요건이나 사용자 핵심 설정 같은 항목에 보호 플래그를 달면, 자동 정리로 인한 의도치 않은 삭제 위험을 줄일 수 있습니다.

**임시 데이터는 저장하지 마세요.** 단일 세션의 중간 결과는 장기 Memory에 저장할 필요가 없습니다.

```python
from dataclasses import dataclass
from datetime import datetime
from enum import Enum

class MemoryTier(Enum):
    PERSISTENT = "persistent"   # 사용자 설정, 핵심 컨텍스트 — 자동 삭제 대상 제외
    STANDARD = "standard"       # 학습된 사실, 일반 메모
    EPHEMERAL = "ephemeral"     # 세션 한정, 적극 삭제 대상

@dataclass
class MemoryEntry:
    content: str
    tier: MemoryTier
    created_at: datetime
    tags: list[str]

def should_persist(entry: MemoryEntry) -> bool:
    return entry.tier != MemoryTier.EPHEMERAL
```

**항목을 세분화해서 저장하세요.** 사실 하나당 항목 하나로 저장하면 병합과 노후화 판단이 훨씬 쉬워집니다.

## webhani의 시각

Agent Memory 관리 비용은 초기 설계에서 과소평가되기 쉽습니다. Agent 운용 기간이 길어질수록 Memory 정리가 눈에 띄지 않는 운영 부담으로 쌓입니다. Dreaming은 이 실제 문제를 다룹니다.

다만 노후화 판단 기준의 세부 내용은 아직 완전히 공개되지 않았습니다. 고객 데이터나 규정 준수 워크플로우처럼 업무상 중요한 Agent에는, Dreaming이 활성화된 상태에서도 Memory Store 정기 감사를 계속 권장합니다. 자동화는 도움이 되지만, 무엇이 여전히 관련성이 있는지 최종 판단은 사람의 컨텍스트가 필요합니다.
