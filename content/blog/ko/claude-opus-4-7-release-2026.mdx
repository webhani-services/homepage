Anthropic이 2026년 4월 출시한 **Claude Opus 4.7**이 SWE-Bench Verified에서 **87.6%** 를 기록했습니다. 이 벤치마크는 실제 오픈소스 프로젝트의 GitHub Issue를 AI가 자율적으로 해결할 수 있는지 평가합니다. 이전 세대인 Opus 4.6 대비 큰 폭의 향상이며, 개발팀의 워크플로우에 실질적인 변화를 가져올 수준입니다.

## SWE-Bench란 무엇인가

SWE-Bench Verified는 Django, Flask, scikit-learn 등 실제 오픈소스 프로젝트의 GitHub Issue를 사용합니다. AI는 다음 과정을 자율적으로 수행해야 합니다.

1. Issue 내용을 이해한다
2. 코드베이스에서 관련 코드를 찾는다
3. 패치를 생성한다
4. 기존 테스트를 통과시킨다

87.6%는 대상 Issue의 약 9건 중 8-9건을 정확히 해결한다는 의미입니다. 6개월 전만 해도 이 수치는 70% 미만이었습니다.

## 역할 변화: 구현자 → 리뷰어

실질적인 영향은 엔지니어가 사라진다는 게 아닙니다. 시간 배분이 바뀝니다.

**기존 워크플로우:**
```
엔지니어가 Issue 읽기 → 관련 파일 찾기 → 
수정안 작성 → 적용 → 테스트 → 반복
```

**Opus 4.7 활용 워크플로우:**
```
Issue를 AI에 전달 → AI가 패치 생성 → 
엔지니어가 diff 리뷰 → 승인 또는 피드백 제공
```

병목이 "코드 작성"에서 "코드 평가"로 이동합니다. 이는 스킬 전환이지, 인력 감소가 아닙니다.

## 현재 Opus 4.7이 잘 작동하는 영역

모든 작업이 높은 SWE-Bench 점수의 혜택을 균등하게 받지는 않습니다. 특히 효과적인 영역은 다음과 같습니다.

- **타입 에러 수정**: TypeScript, Python의 기계적인 타입 불일치 해결
- **테스트 생성**: 기존 함수에 대한 Unit test 자동 작성
- **API 마이그레이션**: Deprecated 라이브러리 호출을 최신 API로 교체
- **문서 동기화**: 함수 시그니처 변경에 맞춰 JSDoc, README 업데이트

반면 다음 작업은 여전히 인간의 판단이 필요합니다.

- 아키텍처 결정 (무엇을 만들어야 하는가)
- 이해관계자 확인이 필요한 모호한 요구사항
- 프로덕션 컨텍스트에 따른 성능 최적화 트레이드오프

## 효과적인 프롬프트 패턴

Opus 4.7은 개방형 지시보다 명시적인 컨텍스트를 제공할 때 더 안정적인 결과를 냅니다.

### 권장 패턴

```
다음 Issue를 수정해 주세요.

## Issue
[Issue 내용]

## 관련 파일
- src/services/payment.ts — 결제 처리 로직
- src/types/order.ts — 주문 타입 정의

## 제약 조건
- 공개 API 인터페이스는 변경하지 말 것
- 새 테스트는 Vitest 형식으로 작성
- 번들 크기를 늘리지 말 것
```

"결제 버그 고쳐줘" 같은 모호한 지시와 비교하면 훨씬 일관된 결과를 얻을 수 있습니다.

### 작업을 단계별로 분리

큰 리팩토링 작업은 명시적 단계로 나누는 것이 좋습니다.

```
Step 1: 현재 구현의 문제점을 나열해 주세요 (코드 작성 없이)
Step 2: 접근 방식을 제안해 주세요 (코드 작성 없이)
Step 3: 제가 확인 후 구현해 주세요
```

이렇게 하면 방향을 수정할 기회가 생깁니다.

## 비용 고려사항

Opus 4.7은 Sonnet이나 Haiku보다 토큰당 비용이 높습니다. 작업 유형에 맞게 모델을 라우팅하는 것이 중요합니다.

| 작업 유형 | 권장 모델 |
|-----------|-----------|
| 복잡한 Bug 수정, 리팩토링 | Opus 4.7 |
| 일상적인 Code completion | Sonnet 4.6 |
| 보일러플레이트 생성 | Haiku 4.5 |

Claude Code 같은 통합 도구는 작업 복잡도에 따라 이 라우팅을 자동화해 줍니다.

## webhani의 관점

벤치마크 점수 향상은 중요한 신호이지만, 팀에게 실질적인 질문은 항상 같습니다. "이것이 우리가 소프트웨어를 만드는 방식을 바꾸는가, 그렇다면 어떻게 안전하게 적용할 것인가."

저희가 클라이언트와 함께 취하는 접근법은, 잘 정의된 저위험 작업(테스트 생성, Deprecation 처리, 문서 업데이트)부터 시작해 AI 생성 코드에 대한 신뢰를 쌓은 후 프로덕션 핵심 로직으로 확장하는 것입니다.

Opus 4.7은 AI 코드 지원이 "유용한 실험"에서 "운영 도구"로 전환되었다는 가장 명확한 신호입니다. 지금 리뷰 스킬과 통합 워크플로우를 구축하는 팀이 복리 효과의 이점을 누릴 것입니다.

- 엔지니어 역할이 구현에서 리뷰와 판단으로 이동합니다
- 명시적이고 제약이 있는 프롬프트가 더 안정적인 결과를 냅니다
- 작업 복잡도에 따른 모델 라우팅으로 품질을 유지하면서 비용을 절감합니다
