Anthropic이 2026년 6월 30일 공개한 Claude Sonnet 5의 핵심은 성능 그 자체가 아니라 경제성입니다. 얼마 전까지 상위 Model에서만 가능했던 자율 작업을 이제 Sonnet 가격대에서 돌릴 수 있게 됐습니다. 이 변화는 일회성 고난도 문제보다, 요청 수가 쌓이는 Workload에서 의미가 큽니다. 이 글에서는 Sonnet 5의 위치를 정리하고, Opus와의 역할 분담 그리고 비용을 눌러 두는 구현 요령을 다룹니다.

## 무엇이 달라졌나

Anthropic 발표에 따르면 Sonnet 5는 Agent 용도를 전제로 설계됐습니다. 계획 수립, Browser·Terminal 같은 Tool 활용, 여러 단계에 걸친 자율 실행의 신뢰도가 이전 세대인 Sonnet 4.6보다 개선됐습니다. Anthropic이 제시한 Agentic Coding Benchmark에서 Sonnet 5는 63.2%로, Opus 4.8의 69.2%와 Sonnet 4.6의 58.1% 사이에서 Opus와의 격차를 상당히 좁혔습니다. 그러면서도 Sonnet 가격대를 유지한다는 점이 특징입니다.

가격은 2026년 8월 31일까지 도입가로 Input 100만 Token당 $2, Output 100만 Token당 $10이며, 이후 $3 / $15로 전환됩니다(Anthropic 공개 수치). 같은 작업을 Opus로 돌릴 때와 비교하면 자릿수가 하나 달라지는 경우도 드물지 않습니다.

여기서 새겨 둘 점은 "똑똑해졌다"가 아니라, **Agent를 상시 가동하는 운영이 이제 채산이 맞는다**는 것입니다. 이 이점은 사내 자동화나 1차 고객 응대처럼 호출 수가 누적되는 Workflow에서 드러나지, 고립된 고난도 작업에서 드러나는 것이 아닙니다.

## Opus와 Sonnet 5 나눠 쓰기

모든 작업에 상위 Model을 쓰는 것은 운영에서 정답이 아닌 경우가 많습니다. 비용과 Latency가 그대로 운영 부담이 되기 때문입니다. webhani는 다음 기준으로 분배하기를 권합니다.

- **Sonnet 5를 기본값으로**: CRUD 처리, 정형 Code 생성, Log 조사, 문서 요약, 1차 지원 응답처럼 양이 많은 작업
- **Opus로 승격**: 설계 판단, 복잡한 장애의 근본 원인 분석, 실패 비용이 큰 단발성 작업

구현 측면에서는 먼저 Sonnet 5로 처리하고, 확신도가 임계값 아래로 떨어지거나 특정 조건에 해당할 때만 Opus로 Escalation하는 2단 구성이 유효합니다.

```typescript
function pickModel(task: { tokensEstimate: number; needsDesignJudgment: boolean }): string {
  // 설계 판단을 동반하거나, 입력이 크고 실패 비용이 높은 것만 Opus로.
  if (task.needsDesignJudgment || task.tokensEstimate > 40_000) {
    return "claude-opus-4-8";
  }
  return "claude-sonnet-5";
}

async function runWithEscalation(prompt: string, task: Parameters<typeof pickModel>[0]) {
  const first = await callClaude(pickModel(task), prompt);
  // 자가 보고 확신도가 낮으면 상위 Model로 재실행.
  if (first.selfConfidence < 0.6 && pickModel(task) !== "claude-opus-4-8") {
    return callClaude("claude-opus-4-8", prompt);
  }
  return first;
}
```

위 Code는 개념을 보여 주는 의사 구현입니다. `selfConfidence`를 얻는 방식은 작업마다 다르지만, 구조화 출력으로 확신도를 반환받거나 별도의 검증 단계를 두는 방법이 현실적입니다.

## Prompt Caching과 Batch로 비용 깎기

Sonnet 5는 Prompt Caching으로 최대 90%, Batch 처리로 50%의 비용 절감이 가능하다고 알려져 있습니다. Agent를 운영한다면 이 두 가지를 쓰느냐 마느냐가 청구액을 크게 가릅니다.

**Prompt Caching**은 System Prompt, Tool 정의, 참조 문서처럼 "매번 동일하고 긴 앞부분"을 Cache하는 방식입니다. Agent는 같은 System Prompt를 반복해서 보내므로 Cache Hit율이 높아지기 쉽고, 궁합이 좋습니다.

```typescript
// 길고 고정된 Prompt 부분을 Cache 대상으로 표시한다.
const system = [
  {
    type: "text",
    text: LONG_SYSTEM_PROMPT_WITH_TOOL_SPECS, // 수천 Token의 고정 구간
    cache_control: { type: "ephemeral" },
  },
];

// 두 번째 이후 요청부터 이 고정 블록은 Cache에서 읽힌다.
```

**Batch 처리**는 즉시 응답이 필요 없는 Workload에 적합합니다. 야간 Log 분류, 대량 문서 요약, Test Case 일괄 생성처럼 "몇 시간 안에 돌아오면 되는" 처리를 Batch로 묶으면 실시간 API의 절반 비용으로 돌릴 수 있습니다.

운영 설계 순서로는 먼저 "정말 실시간이 필요한가"를 가려내고, 불필요한 것을 Batch로 몰고, 남은 실시간 처리는 Caching으로 Header 부분을 압축하는 흐름이 이해하기 쉽습니다.

## Agent 설계에서의 실무 주의점

Model이 싸져도 설계가 허술하면 비용은 부풀어 오릅니다. webhani가 현장에서 중시하는 지점은 다음과 같습니다.

1. **단계 수에 상한을 둔다.** 자율 실행은 편리하지만 무한 Loop나 과도한 Tool 호출은 곧 비용입니다. `maxSteps`를 명시하고 초과 시 Fallback시킵니다.
2. **Tool 입도를 맞춘다.** 하나의 Tool에 기능을 몰아넣으면 Model이 잘못된 인자를 넘기기 쉬워 재시도가 늘어납니다. 단일 기능에 이름이 자명한 Tool로 나누는 편이 결과적으로 Token 소비를 줄입니다.
3. **출력을 구조화한다.** 자유 서술 출력은 후속 처리에서 Parsing이 필요하고 실패 시 재실행 비용이 듭니다. 처음부터 JSON Schema로 묶는 편이 안정적입니다.
4. **관측 가능성을 확보한다.** 어느 단계에서 몇 Token을 썼는지 기록하지 않으면 최적화의 수가 보이지 않습니다. Tracing은 Agent 운영의 전제로 봐야 합니다.

## 정리

Claude Sonnet 5의 의미는, 상위 Model에서만 돌아가던 Agent를 일상 운영에 올릴 수 있는 가격대로 끌어내렸다는 데 있습니다. 실무에서는 Sonnet 5를 기본값으로 양이 많은 작업을 맡기고, 설계 판단이나 실패 비용이 큰 국면에서만 Opus로 승격하는 2단 구성이 현실적입니다. 여기에 Prompt Caching과 Batch를 조합하면 Agent 운영 비용은 무리 없이 관리 가능한 범위에 들어옵니다.
