## 모델 선택의 기준이 바뀌었습니다

2026년 6월 30일 Anthropic이 Claude Sonnet 5를 출시했습니다. 핵심은 "Opus 4.8에 근접한 에이전트 성능을 훨씬 낮은 비용으로 제공한다"는 점입니다. 계획 수립, 도구 사용, 터미널 조작, 장시간에 걸친 다단계 코딩처럼 기존에는 상위 모델에서만 안정적으로 처리되던 작업을 Sonnet 가격대에서 처리할 수 있게 되었습니다.

도입 가격은 2026년 8월 31일까지 입력 $2/M, 출력 $10/M 토큰이며 이후 $3/$15로 전환됩니다. 이미 Claude Free·Pro Plan, Claude Code, API의 기본 모델로 채택되어 있어 많은 개발자가 별도의 선택 없이 이 모델을 사용하게 됩니다.

webhani가 고객사의 AI 도입을 지원할 때 반복적으로 논의되는 주제가 바로 "어떤 Task에 어떤 모델을 배정할 것인가"라는 비용 설계입니다. Sonnet 5의 등장은 이 설계를 다시 점검할 좋은 계기입니다.

## 왜 모델 선택이 경영 이슈가 되는가

AI 코딩을 개인 생산성 도구로만 쓸 때는 모델 차이가 체감 수준에 머무릅니다. 하지만 팀이나 Product에 AI Agent를 내재화하는 단계에서는 이야기가 달라집니다.

- 월간 토큰 소비량이 Product 비용의 실질적인 항목이 됩니다
- Task 난이도와 투입 모델의 불일치가 비용과 품질 모두에 영향을 줍니다
- 장시간 자율 실행되는 Agent Workflow는 실패 시 재작업 비용까지 포함해 설계해야 합니다

Sonnet 5가 Opus급 성능을 갖췄다는 것은, 기존에는 Opus에만 맡길 수 있었던 Task 일부를 Sonnet으로 옮길 수 있다는 의미입니다. 이는 비용 구조 자체를 재검토할 기회입니다.

## Task 기반 모델 라우팅

webhani는 AI Agent를 프로덕션에 도입할 때, Task의 성격에 따라 모델을 분배하는 설계를 권장합니다.

```text
Task 분류            권장 모델 Tier         이유
------------------  -------------------  --------------------------------
단순 변환·정형화     경량 모델            속도 우선, 품질 차이가 크지 않음
일반 구현·리뷰       Sonnet 5             비용과 성능의 균형이 좋음
고난도 설계 판단     Opus 4.8             Architecture 판단의 정확도 우선
```

의사 코드로 표현하면 라우팅 로직은 다음과 같습니다.

```typescript
type TaskComplexity = "trivial" | "standard" | "critical";

function selectModel(complexity: TaskComplexity): string {
  switch (complexity) {
    case "trivial":
      return "claude-haiku-4-5";
    case "standard":
      return "claude-sonnet-5";
    case "critical":
      return "claude-opus-4-8";
  }
}
```

중요한 것은 이 판단을 감(感)에만 맡기지 않는 것입니다. Task의 예상 난이도, 영향 범위(프로덕션 코드인지 실험 코드인지), 실패 시 재작업 비용을 기준으로 기계적인 분배 Rule을 팀 차원에서 합의해두면 비용 예측 가능성이 높아집니다.

## Agent Workflow에서 변하지 않는 것

Sonnet 5가 장시간 다단계 코딩에 강해졌다고 해서 "맡겨두면 된다"는 뜻은 아닙니다. Agent가 자율적으로 여러 파일을 변경하는 Workflow에서는 다음 설계가 여전히 필수입니다.

1. **Checkpoint 설치**: 장시간 Task를 한 번에 맡기지 않고 구간마다 사람의 Review를 끼워 넣습니다
2. **Diff 가시화**: Agent가 무엇을 변경했는지 Commit 단위로 쉽게 추적할 수 있게 합니다
3. **Rollback 수단 확보**: 예상치 못한 변경이 발생했을 때 빠르게 되돌릴 수 있는 상태를 유지합니다

모델 성능이 올라갈수록 이런 Guardrail의 중요성은 줄어드는 것이 아니라, 오히려 맡길 수 있는 범위가 넓어지는 만큼 더 커집니다.

## webhani의 권장 접근

고객사의 AI 도입 지원에서는 먼저 기존 AI 코딩 Task를 전수 조사해 위와 같은 Task 분류 체계를 만드는 것부터 시작합니다. 그 위에 Sonnet 5처럼 "성능과 비용의 균형이 좋은 모델"을 기본값으로 두고, 정말 고난도인 설계 판단만 Opus급으로 보내는 이중 구조를 제안합니다.

도입 가격이 2026년 8월 말에 바뀌는 점을 고려해, 비용 시뮬레이션은 현재 가격뿐 아니라 9월 이후 가격을 전제로도 함께 계산해두는 것을 권장합니다.

## 마무리

Claude Sonnet 5는 단순한 "새 모델 출시"가 아니라 AI 코딩의 비용 구조를 재검토할 계기입니다. Opus급 성능을 Sonnet 가격에 쓸 수 있다는 것은, 그동안 비용 때문에 미뤄왔던 자동화 Task가 현실적인 선택지가 되었다는 뜻입니다. 다만 성능이 올라갈수록 검증 설계의 중요성도 함께 커집니다. webhani는 모델 선택과 Guardrail 설계를 함께 제안하며 고객의 AI 도입을 지원합니다.
