## "저렴한 Frontier"라는 포지셔닝

Anthropic은 6월 30일 Claude Sonnet 5를 릴리스했습니다. 핵심은 순수 성능이 아니라 가격과 Agent 동작 사이의 균형입니다. 도입 가격은 Input $2/백만 Token, Output $10/백만 Token(2026년 8월 말까지, 이후 $3 / $15)입니다. 몇 달 전이라면 Opus급 Model에서만 가능했던 자율적 Task 수행을 이 가격대에서 돌릴 수 있게 되었다는 점이 가장 큰 변화입니다.

webhani는 Client의 Agent Platform을 여러 개 운영하고 있는데, 여기서 반복적으로 문제가 되는 것은 "1회 추론이 얼마나 똑똑한가"가 아니라 "Loop를 수십 번 돌린 뒤 전체가 얼마인가"입니다. Agent는 Tool을 호출하고, 결과를 읽고, 다음 행동을 결정하는 과정을 반복합니다. 반복 횟수가 많은 Workload일수록 Model 단가가 그대로 이익률에 반영됩니다. Sonnet 5가 주목받는 이유는 바로 이 "반복 Cost"를 크게 낮추는 지점에 안착했기 때문입니다.

## 무엇이 바뀌었나

Anthropic은 Sonnet 5를 "지금까지 가장 Agent다운 Sonnet"으로 설계했다고 설명합니다. Browser나 Terminal 같은 Tool을 사용하면서 여러 Step으로 이루어진 Task를 사람의 개입 없이 실행하는 능력이 향상되었습니다. 실무에서 가장 많이 듣는 평가는 "Task를 끝까지 완수한다"는 점입니다. 기존 Model이 중간에 멈추던 상황에서도 계획을 다시 세워 완료까지 끌고 가는 경향이 강하다는 보고가 있습니다.

기술적으로 짚어둘 지점은 다음과 같습니다.

- **1M Token Context가 기본값** — 대규모 Codebase 로딩과 Log 분석에 유리
- **최대 128k Token Output** — 긴 산출물을 한 번에 반환
- **Adaptive Thinking** — 문제 난이도에 따라 내부 추론 깊이를 조정
- **새로운 Tokenizer** — 여기에 함정이 있습니다

## 새 Tokenizer라는 "숨은 Cost"

놓치기 쉬운 부분이 Sonnet 5가 새 Tokenizer를 채택했다는 점입니다. 같은 Input Text라도 Sonnet 4.6 대비 약 30% 더 많은 Token을 소비합니다. 즉 "단가는 내려갔지만 Token 수는 늘어난다"는 관계입니다.

이것은 Migration 시 기존 Cost 산정을 그대로 옮길 수 없다는 뜻입니다. 단가만 보고 "Opus에서 갈아타면 싸진다"고 결론 내리면 예상보다 청구액이 늘어나는 경우가 생깁니다. 실제로 견적을 낼 때는 단가 차이뿐 아니라 Token 증가율을 함께 곱해야 합니다.

```typescript
// Migration 견적: 단가만이 아니라 Token 증가율까지 반영
interface ModelCost {
  name: string;
  inputPerM: number; // $/백만 Token
  outputPerM: number;
  tokenMultiplier: number; // 구 Tokenizer 대비 소비 배율
}

const sonnet46: ModelCost = {
  name: "Sonnet 4.6",
  inputPerM: 3,
  outputPerM: 15,
  tokenMultiplier: 1.0,
};

const sonnet5: ModelCost = {
  name: "Sonnet 5 (표준 가격)",
  inputPerM: 3,
  outputPerM: 15,
  tokenMultiplier: 1.3, // 약 30% 증가
};

function estimateCost(
  model: ModelCost,
  inputTokens: number,
  outputTokens: number
): number {
  const adjustedInput = inputTokens * model.tokenMultiplier;
  const adjustedOutput = outputTokens * model.tokenMultiplier;
  return (
    (adjustedInput / 1_000_000) * model.inputPerM +
    (adjustedOutput / 1_000_000) * model.outputPerM
  );
}

// Agent Loop 1회 가정
const inputTokens = 40_000;
const outputTokens = 2_000;

console.log(estimateCost(sonnet46, inputTokens, outputTokens));
console.log(estimateCost(sonnet5, inputTokens, outputTokens));
// 단가가 같아도 Token 증가분만큼 Sonnet 5가 더 비싸진다
```

두 번째 결과도 있습니다. Tokenizer가 바뀌면 Context Window의 *실효 용량*도 바뀝니다. 1M Token 예산이 넉넉해 보여도 같은 문서가 더 많은 Token을 차지하므로 실제로 담을 수 있는 정보량은 줄어듭니다. 긴 문서를 대량으로 다루는 설계라면 이 점을 전제로 Chunk 분할 임계값을 재검토할 가치가 있습니다.

## Agent 운영에서 효과 있는 실전 패턴

**Adaptive Thinking의 effort 제어** — 간단한 Task에까지 깊은 추론을 돌리면 Output Token이 늘어 Cost로 되돌아옵니다. Task 난이도에 맞춰 추론 깊이를 지정하고, 단순 분류나 Format 변환에서는 얕게 억제하는 것이 정석입니다.

**Prompt Caching** — Agent는 System Prompt와 Tool 정의를 매번 같은 내용으로 보냅니다. 이 고정 부분을 Cache에 올리면 두 번째 호출부터 Input Cost를 크게 절감할 수 있습니다. 반복이 많은 Agent일수록 효과가 큽니다.

**Model 계층화는 여전히 유효** — Sonnet 5가 저렴해졌다고 해서 모든 것을 Sonnet 5로 돌릴 필요는 없습니다. 정형적인 Sub-task는 Haiku, 어려운 지점은 Sonnet 5, 그래도 안 풀리는 최난도만 Fable 5로 가는 3단 구성은 계속 유효합니다. Sonnet 5의 등장으로 이 "가운데 층"이 두꺼워졌다고 이해하는 편이 실제에 가깝습니다.

**Token 예산 가시화** — Agent를 Production에 투입하기 전에 Task당 평균 Token 소비를 계측해 두기를 권합니다. 새 Tokenizer 때문에 기존 실측값을 쓸 수 없으므로 Migration 후 반드시 다시 측정하세요.

## Migration 판단 Checklist

webhani가 Client에게 Sonnet 5 Migration을 권하는지는 다음으로 판단합니다.

- Agent의 Loop 횟수가 많아 Model 단가가 이익률에 직결된다
- Task가 "중간에 멈추는" 문제를 안고 있고, 완수율 개선이 가치가 된다
- Migration 후 Token 소비를 재계측하고 Cost 증가를 반영할 체계가 있다

반대로 단발성 짧은 생성이 중심이고 반복이 적은 Workload에서는 Migration의 이점이 제한적입니다. 단가의 저렴함은 반복 횟수가 많을수록 효과를 발휘하기 때문입니다.

## 정리

Claude Sonnet 5의 본질은 "똑똑해진 것"보다 "Agent를 저렴하게 돌릴 수 있는 지점에 온 것"입니다. 다만 새 Tokenizer로 인한 Token 증가라는 부작용이 있어, 단가만 보고 Migration을 판단하는 것은 위험합니다. 단가 차이와 Token 증가율을 함께 곱하고, 실측으로 Cost를 다시 잡는다 — 이 순서를 지키면 Sonnet 5는 Agent Platform의 운영 Cost를 현실적으로 낮추는 선택지가 됩니다. webhani는 이 검증 Process를 Client 안건의 표준 절차로 포함하고 있습니다.
