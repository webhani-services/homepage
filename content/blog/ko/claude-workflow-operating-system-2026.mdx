## 챗봇에서 실행 환경으로

AI 어시스턴트를 쓰는 방식이 지난 1년 사이 크게 달라졌습니다. 예전에는 "좋은 Prompt를 쓸 수 있는가"가 성과를 갈랐지만, 2026년 현재는 "같은 품질의 작업을 몇 번이든 재현하는 구조를 만들 수 있는가"가 핵심이 되었습니다.

Anthropic 사용 패턴을 다룬 여러 글에서도, Claude를 일문일답 챗봇이 아니라 "Workflow를 돌리기 위한 실행 환경"으로 다루는 사람일수록 얻는 가치가 크다는 지적이 나옵니다([Let's Data Science](https://letsdatascience.com/news/users-treat-claude-as-workflow-operating-system-1c5926ac)). 저희가 고객사의 개발팀을 지원할 때도 이 관점의 유무에 따라 도입 효과가 크게 달라진다고 느낍니다.

이 글에서는 단발성 Prompt에서 "재현 가능한 Workflow"로 넘어가기 위한 구성 요소와 실무 조립 방법을 정리합니다.

## 역할로 나눈 5가지 구성 요소

Claude Code 같은 agentic Tool에서는 기능을 5개 층으로 이해하면 설계가 쉬워집니다. 각각 "언제 작동하는가"가 다르기 때문에, 혼동하면 기대한 동작이 나오지 않습니다.

- **CLAUDE.md (상시 Context)**: 프로젝트 전제·규약·금지 사항을 항상 읽히기
- **Skills (온디맨드 지식·절차)**: 특정 Task일 때만 불러오는 절차서
- **MCP (외부 시스템 연결)**: Issue Tracker, Database, 모니터링 Tool 등 파일 밖으로 연결
- **Subagent (Context 격리)**: 리뷰나 테스트처럼 독립된 Context에서 실행
- **Hooks (자동화·가드레일)**: Tool 호출이나 Session 종료 시 규칙을 강제

외우는 법은 간단합니다. CLAUDE.md는 "항상 작동하는 전제", Skills는 "필요할 때의 절차", MCP는 "밖으로의 연결", Subagent는 "문맥 격리", Hooks는 "자동 강제". 이 5개를 역할로 나눠 두면 나중에 기능을 추가할 때 헷갈리지 않습니다.

## CLAUDE.md: 전제를 매번 쓰지 않기

비용 대비 효과가 가장 큰 것이 CLAUDE.md입니다. 사용하는 Framework 버전, 네이밍 규칙, 테스트 실행 방법, 건드리면 안 되는 파일 등을 한 번 적어 두면 매번 Prompt에서 설명할 필요가 없어집니다.

```markdown
# Project 규약 (발췌)

## Tech Stack
- Next.js 16 App Router / TypeScript strict
- Style은 Tailwind만. CSS Modules는 쓰지 않는다

## 금지 사항
- `src/generated/` 아래는 손으로 편집하지 않는다 (자동 생성)
- 환경 변수는 `.env.local`에 직접 적지 않는다

## 테스트
- 변경 후 반드시 `npm test`를 실행한 뒤 완료로 간주한다
```

핵심은 "지켰으면 하는 것"뿐 아니라 "하면 안 되는 것"을 명시하는 것입니다. 금지 사항을 적어 두면 Model이 좋은 의도로 무언가를 망가뜨리는 사고를 줄일 수 있습니다.

## Skills: 절차를 재사용 가능하게

같은 작업을 반복해서 시킨다면 그 절차를 Skill로 떼어 냅니다. "Blog 글을 생성한다", "Release Note를 만든다" 같은 정형 Task를 실행 조건과 절차 형태로 저장해 두면 매번 같은 품질로 재현됩니다.

CLAUDE.md가 "항상 읽히는 전제"라면, Skills는 "조건에 맞을 때만 불러오는 절차"입니다. 상시 Context를 부풀리지 않고도 전문 지식을 갖게 할 수 있어, Context 압박을 피하면서 동작을 안정시킬 수 있습니다.

## MCP: 파일 밖으로 연결하기

MCP(Model Context Protocol)는 AI를 파일시스템 밖 세계와 연결하는 구조입니다. Issue Tracker에서 Ticket을 읽고, PostgreSQL에 쿼리를 던지고, 모니터링 Tool에서 에러를 확인하는 등 "실제 업무 Workflow"를 돌릴 수 있게 됩니다.

여기서 중요한 것은 권한 설계입니다. 쓰기 가능한 MCP 연결을 무심코 늘리면 의도치 않은 변경 위험이 커집니다. 저희가 고객사에 도입할 때는 먼저 읽기 전용 연결로 시작하고, 효과를 확인하면서 쓰기 권한을 단계적으로 개방하도록 권장합니다.

## Subagent와 Hooks: 격리와 강제

**Subagent**는 독립된 Context·Prompt·Tool 권한을 가진 보조 에이전트입니다. 메인 에이전트가 계획과 통합을 맡고, 코드 리뷰·테스트 실행·Security Check 같은 한정된 Task를 Subagent에 맡기면 메인 문맥을 오염시키지 않고 전문 작업을 병행할 수 있습니다.

**Hooks**는 Tool 호출이나 Session 종료 같은 이벤트에 걸어 규칙을 기계적으로 강제하는 구조입니다. 자주 쓰는 패턴은 다음과 같습니다.

- 종료 전 반드시 테스트를 실행한다
- 자동 생성 파일 편집을 차단한다
- Commit 전에 Lint를 실행한다
- 의존성 변경 후 Security Scan을 돌린다

Hooks의 가치는 "Model의 선의에 기대지 않는다"는 점에 있습니다. 사람의 리뷰와 마찬가지로, 잊어버린다는 전제로 규칙을 시스템화하는 편이 안정적입니다.

## webhani의 관점: 작게 시작하고 계측하기

이 모든 것을 한 번에 도입할 필요는 없습니다. 저희가 고객 팀에 제안하는 순서는 다음과 같습니다.

1. **CLAUDE.md만 정비한다.** 이것만으로도 체감 안정성이 크게 달라집니다.
2. **자주 하는 Task 하나를 Skill로 만든다.** 효과가 눈에 잘 보이는 것부터.
3. **읽기 전용 MCP를 하나 연결한다.** 우선 안전한 연결로 가치를 확인.
4. **Hooks로 테스트 실행을 강제한다.** 품질의 하한선을 기계적으로 담보.
5. **필요해지면 Subagent로 격리한다.** 리뷰나 테스트부터.

중요한 것은 도입 전후로 "재작업 횟수"나 "리뷰 지적 수"를 계측해 두는 것입니다. 효과를 숫자로 확인하지 못하면 팀 정착이 진행되지 않습니다.

## 정리

AI 어시스턴트의 성과는 더 이상 단발성 Prompt의 솜씨로 결정되지 않습니다. CLAUDE.md로 전제를 고정하고, Skills로 절차를 재사용하고, MCP로 외부와 연결하고, Subagent로 문맥을 격리하고, Hooks로 규칙을 강제한다 — 이 5개를 역할로 나눠 조립하면 작업은 "재현 가능한 Workflow"가 됩니다.
