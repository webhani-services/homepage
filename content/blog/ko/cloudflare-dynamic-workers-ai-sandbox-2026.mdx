AI Agent가 코드를 생성하고 실행하는 시스템을 만들 때, 실행 환경 격리는 피할 수 없는 문제입니다. 지금까지 많은 팀이 Docker Container나 E2B, Modal 같은 Managed Service를 활용해 왔습니다. Cloudflare가 Beta로 공개한 Dynamic Workers는 이 문제를 다른 방식으로 접근합니다. JavaScript와 TypeScript 중심의 Agent Pipeline이라면 진지하게 검토할 가치가 있습니다.

## Dynamic Workers의 동작 방식

일반적인 Cloudflare Worker는 배포 시점에 코드가 고정됩니다. Dynamic Workers는 이 제약을 없앱니다. 실행 중인 Worker가 런타임에 제공된 코드 문자열로 새 Worker를 생성할 수 있고, 그 Worker는 독립된 V8 Isolate 안에서 실행됩니다.

핵심 특성은 격리입니다. 동적으로 생성된 Worker는 부모 Worker의 Global Scope, 다른 Request, 명시적으로 주어지지 않은 Binding에 접근할 수 없습니다. 진짜 Sandbox입니다.

```typescript
// AI가 생성한 코드를 안전하게 실행하는 Worker
export default {
  async fetch(request: Request): Promise<Response> {
    const { code, input } = await request.json() as {
      code: string;
      input: unknown;
    };

    const result = await runInSandbox(code, input);
    return Response.json({ result });
  },
};

async function runInSandbox(code: string, input: unknown): Promise<unknown> {
  const worker = await DynamicWorker.create({
    script: code,
    limits: {
      cpuMs: 5000,    // 실행당 최대 CPU 시간
      memoryMb: 128,  // Isolate당 최대 Memory
    },
  });

  return await worker.run(input);
}
```

동적으로 생성된 Worker는 실행 후 소멸합니다. 각 호출은 공유 State 없이 완전히 초기화된 상태에서 시작합니다.

## Docker 대비 성능 차이

성능 차이는 실질적입니다. Docker Container Cold Start는 Image 크기와 Host 환경에 따라 500ms에서 수 초가 걸립니다. V8 Isolate는 수 밀리초 안에 초기화됩니다. AI Agent가 수십 개의 코드 실행 Request를 병렬로 처리할 때 이 차이가 누적되어 Latency와 비용에 직접 영향을 줍니다.

| 지표 | Docker Container | Dynamic Workers |
|------|-----------------|-----------------|
| Cold Start | 500ms ~ 3초 | 5ms ~ 30ms |
| Sandbox당 Memory | 50MB ~ 500MB+ | 5MB ~ 20MB |
| Scale-out | 수 초 | 거의 즉시 |

"100배 빠르다"는 수치는 Docker Container가 처음부터 시작하는 경우와 비교한 것입니다. 벤치마크가 보여주지 않는 것도 있습니다. Isolate 초기화는 빠르지만 Zero는 아니고, Script가 클수록 JIT Compilation 시간이 늘어납니다. 실제 Workload로 직접 측정하고 판단하는 것이 중요합니다.

## 사전 등록 없는 Durable Workflow

Dynamic Workers는 단일 실행에는 적합하지만, AI Agent의 Multi-step Workflow — 코드 실행 후 결과를 기다리고 다음 단계로 진행 — 에는 영속적인 State가 필요합니다. `@cloudflare/dynamic-workflows` 패키지가 이 문제를 해결합니다.

표준 Cloudflare Workflows는 Workflow Class를 사전에 등록해야 합니다. `@cloudflare/dynamic-workflows`는 사전 등록 없이 Runtime에서 Workflow를 완전히 동적으로 정의하고 실행할 수 있습니다.

```typescript
import { DynamicWorkflow } from "@cloudflare/dynamic-workflows";

interface AgentTask {
  taskId: string;
  steps: Array<{
    code: string;
    input: unknown;
  }>;
}

export default {
  async fetch(request: Request, env: Env): Promise<Response> {
    const task = await request.json() as AgentTask;

    const workflow = await DynamicWorkflow.create(env.WORKFLOWS, {
      id: task.taskId,
      steps: task.steps.map((step, i) => ({
        name: `step-${i}`,
        run: async (ctx) => {
          // 각 Step은 독립된 Dynamic Worker Sandbox에서 실행
          const worker = await ctx.createWorker(step.code);
          return await worker.run(step.input);
        },
        retries: {
          limit: 3,
          backoff: "exponential",
          delay: "1 second",
        },
      })),
    });

    return Response.json({
      workflowId: workflow.id,
      status: "started",
    });
  },
};
```

Workflow는 `sleep`, `resume`, `retry`를 기본 지원합니다. 사람의 승인을 기다리거나 외부 Webhook을 대기하는 시나리오에서도 Durable Object가 State를 영속화해 중단 후 재개가 가능합니다.

## Cloudflare Sandboxes와 Mesh

Agent Architecture에서 Dynamic Workers를 보완하는 두 가지 기능이 있습니다.

**Cloudflare Sandboxes**는 Agent를 위한 영속적인 Shell과 Filesystem 환경을 제공합니다. Dynamic Workers가 Stateless 코드 실행에 최적화되어 있다면, Sandboxes는 File I/O, 복수 명령어에 걸친 Shell Session, Build Tool이 필요한 경우에 적합합니다. Dynamic Workers로 생성된 함수 실행과 Sandboxes의 실제 Filesystem 접근이 서로를 보완하는 구조입니다.

**Cloudflare Mesh**는 Agent별 Network 접근 제어를 추가합니다. 각 Agent가 접근할 수 있는 External Endpoint를 Whitelist로 설정하고 나머지를 기본 차단할 수 있습니다.

```typescript
const worker = await DynamicWorker.create({
  script: agentGeneratedCode,
  network: {
    allowedOrigins: [
      "https://api.openai.com",
      "https://your-internal-api.example.com",
    ],
    // 명시되지 않은 외부 접속은 기본 차단
    defaultPolicy: "deny",
  },
});
```

AI Agent Pipeline에서 이 제어는 특히 중요합니다. Agent가 생성한 코드가 필요한 API만 호출하고 인터넷 전체에 무제한 접근하는 것을 방지할 수 있습니다.

## 도입 전에 파악해야 할 제약사항

**언어 지원**: JavaScript, TypeScript, WASM만 가능합니다. Python은 현재 지원하지 않습니다. Agent가 데이터 분석이나 ML 작업을 위해 Python 코드를 생성하는 경우 Dynamic Workers를 적용할 수 없습니다. 이런 경우 E2B나 Modal이 더 적합합니다.

**실행 모델 제약**: Cloudflare Workers의 실행 제한을 그대로 받습니다. Request당 CPU 시간 상한(기본 50ms, Unbound Plan은 최대 30초), 외부 Network 접근 제한 등이 해당됩니다. 몇 분 이상 걸리는 장시간 처리에는 맞지 않습니다.

**Cold Start는 거의 Zero이지 완전한 Zero가 아닙니다**: V8 Isolate 초기화에 수 밀리초가 소요됩니다. Script가 클수록 Compilation 시간이 늘어납니다. Latency에 민감한 Path에서는 이 점을 염두에 두어야 합니다.

**Beta 단계**: API가 변경될 수 있습니다. 중요한 Production Path에 도입할 때는 API 변경에 대응할 계획을 세워두어야 합니다.

## 무엇을 선택할 것인가

| 요구사항 | 적합한 선택 |
|---------|------------|
| AI 생성 JS/TS 코드를 빠르게 대량 실행 | Dynamic Workers |
| Python 또는 다중 언어 코드 실행 | E2B / Modal |
| Agent에 영속 Filesystem 또는 Shell 필요 | Cloudflare Sandboxes |
| Sleep/Resume가 있는 Multi-step Workflow | @cloudflare/dynamic-workflows |
| Agent Network 접근 제어 | Cloudflare Mesh |
| 장시간 실행 작업 (수 분 이상) | Docker / 전용 VM |

Dynamic Workers는 Agent Pipeline이 JavaScript 또는 TypeScript 코드를 생성하고, 실행당 Latency와 Overhead를 최소화하려는 요구사항에 가장 잘 맞습니다. 거의 Zero에 가까운 Cold Start, Isolate별 Resource 제한, Mesh를 통한 Network 제어의 조합은 전용 Container Infrastructure 없이도 규모 있게 운영할 수 있는 Sandbox를 제공합니다.

제약은 명확합니다. JavaScript/TypeScript만 실행 가능합니다. Python이 필수라면 Docker나 E2B 경로가 맞습니다. 하지만 JS/TS 기반 Agent Workload라면 Beta 단계인 지금 바로 테스트해볼 가치가 있습니다.
