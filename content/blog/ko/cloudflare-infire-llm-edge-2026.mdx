Cloudflare가 Workers AI를 구동하는 자체 AI 추론 엔진 Infire의 상세 내용을 공개했습니다. Infire가 해결하는 핵심 문제는 Cloudflare의 글로벌 분산 네트워크에서 LLM을 효율적으로 실행하는 것입니다. 이는 중앙화된 데이터센터에서의 추론과는 다른 엔지니어링 과제입니다.

## Edge 추론의 과제

Edge에서 LLM 추론을 실행하면 단일 지역 데이터센터에는 없는 제약이 생깁니다.

**분산된 GPU 리소스**: 각 Edge 거점은 제한된 GPU 용량을 보유합니다. 표준 추론 엔진은 지리적으로 분산된 소규모 GPU 클러스터에서의 모델 샤딩을 효율적으로 처리하지 못합니다.

**콜드 스타트 지연**: LLM은 로딩이 느립니다. 배치 처리에서 30초 콜드 스타트는 허용 가능하지만, 사용자 대면 웹 애플리케이션에서는 허용할 수 없습니다.

**메모리 효율**: GPU 메모리는 고가 리소스입니다. 여러 테넌트와 모델을 서비스하는 엔진은 동시 워크로드 전체에서 메모리를 효율적으로 사용해야 합니다.

## Infire의 핵심 기술

### Edge GPU 클러스터를 위한 텐서 병렬화 최적화

Infire는 텐서 병렬화(모델 가중치를 여러 GPU에 분산)를 Edge 배포의 소규모 GPU 클러스터에 맞게 튜닝했습니다. 표준 텐서 병렬 구현은 고대역폭 인터커넥트를 가정하지만, Infire는 Edge 하드웨어 프로파일에 맞춰 동기화 오버헤드를 줄입니다.

### 요청 간 KV Cache 공유

LLM 추론에서 KV(키-값) 캐시는 이전에 계산된 어텐션 상태를 저장하여 반복된 컨텍스트 처리를 빠르게 합니다. Infire는 동일한 System Prompt를 공유하는 요청 간에 KV 캐시를 공유합니다.

```python
import httpx

async def query_assistant(user_message: str, account_id: str, token: str):
    async with httpx.AsyncClient() as client:
        response = await client.post(
            f"https://api.cloudflare.com/client/v4/accounts/{account_id}/ai/run/"
            "@cf/meta/llama-4-scout-17b-16e-instruct",
            headers={"Authorization": f"Bearer {token}"},
            json={
                "messages": [
                    # 공유 System Prompt는 KV 캐시 재사용으로 혜택을 받습니다
                    {"role": "system", "content": "You are a helpful support agent."},
                    {"role": "user", "content": user_message}
                ]
            }
        )
    return response.json()["result"]["response"]
```

같은 어시스턴트에 여러 사용자가 요청하는 멀티 테넌트 SaaS 제품에서 GPU 메모리 소비를 줄이고 응답 시간을 단축시킵니다.

### Hot Model Standby

Infire는 자주 사용하는 모델 가중치를 메모리에 상주시켜 요청별 로딩을 피합니다. 이로써 콜드 스타트 지연이 수십 초 범위에서 수백 밀리초 수준으로 줄어들어, 간헐적 트래픽을 갖는 애플리케이션에서도 실용적인 프로덕션 경험을 가능하게 합니다.

## 실제 활용

Infire의 개선은 Workers AI 개발자 경험에 직접 반영됩니다.

```javascript
// 스트리밍 LLM 응답을 처리하는 Cloudflare Worker
export default {
  async fetch(request, env) {
    const { question } = await request.json();

    const stream = await env.AI.run(
      "@cf/meta/llama-4-scout-17b-16e-instruct",
      {
        messages: [
          { role: "system", content: "You are a helpful assistant." },
          { role: "user", content: question }
        ],
        stream: true,
      }
    );

    return new Response(stream, {
      headers: {
        "Content-Type": "text/event-stream",
        "Cache-Control": "no-cache",
      }
    });
  }
};
```

**지연 시간**: 콜드 스타트 감소는 간헐적 트래픽 패턴의 애플리케이션에 가장 눈에 띄는 변화입니다.

**비용**: KV 캐시 공유로 동일한 GPU 메모리가 공유 System Prompt에서 더 많은 동시 사용자를 처리할 수 있습니다.

## 주목할 설계 원칙

Infire 아키텍처에서 도출할 수 있는 실용적인 설계 원칙들:

**짧고 공유 가능한 System Prompt 유지하기.** KV 캐시 공유는 요청 간 System Prompt가 동일할 때만 효과가 있습니다. 요청마다 다른 긴 System Prompt는 이 최적화를 무력화합니다.

**첫 토큰부터 스트리밍하기.** Edge 추론에서 스트리밍은 체감 지연 시간을 크게 줄입니다. 전체 생성이 완료될 때까지 기다리지 마세요.

**데이터 거주 요건 파악하기.** Cloudflare는 자동으로 가장 가까운 PoP에서 요청을 처리합니다. 데이터에 지리적 제한이 있다면 Workers 지역 라우팅을 명시적으로 설정해야 합니다.

## 다른 추론 엔진과의 비교

| 엔진 | 주요 용도 | 차별화 포인트 |
|------|---------|-------------|
| vLLM | 셀프 호스트 / 클라우드 | PagedAttention, 고처리량 |
| TensorRT-LLM | NVIDIA GPU 클러스터 | CUDA 네이티브 최적화 |
| llama.cpp | Edge 디바이스 / CPU | 이식성, 최소 의존성 |
| Infire | Cloudflare Edge | 멀티 테넌트 효율, 낮은 콜드 스타트 |

Infire의 틈새는 분산된 GPU 제한 환경에서의 멀티 테넌트 리소스 공유입니다. 전용 GPU 클러스터의 vLLM과 원시 처리량 경쟁을 하는 것이 아니라, 다른 배포 모델을 위해 설계되었습니다.

## 보안 고려사항

멀티 테넌트 KV 캐시 공유는 테넌트 간 데이터 누출 가능성에 대한 의문을 제기합니다. Cloudflare의 설계는 KV 캐시 키잉 체계에 테넌트 경계를 포함시켜 교차 테넌트 캐시 접근을 방지합니다. 고도로 민감한 워크로드에는 Cloudflare AI Gateway의 제로 데이터 보존 옵션을 활용할 수 있습니다.

## Workers AI 활용 판단 기준

**Workers AI(Infire 기반)가 적합한 경우:**
- 챗봇 및 대화형 UI 기능
- 경량 문서 분류
- 웹 애플리케이션에서 컨텍스트 인식 입력 제안
- 인프라 관리 없이 글로벌 저지연이 필요한 애플리케이션

**다른 대안을 고려해야 하는 경우:**
- GPT-4o 또는 Claude 수준의 추론 능력이 필요한 경우
- 소규모 모델 한계를 초과하는 문서 길이나 추론 복잡성
- 엄격한 데이터 거주 요건으로 멀티 지역 Edge 배포가 문제가 되는 경우

Infire 공개는 Edge AI 추론이 실험적 단계에서 프로덕션 실용 단계로 이동했음을 의미합니다. 이전에 전담 ML 플랫폼 팀이 운영해야 했던 인프라 복잡성이 이제 간단한 API 호출로 가능해졌습니다. 작게 시작해서 필요에 따라 확장하는 접근 방식이 Edge AI에는 가장 잘 맞습니다.
