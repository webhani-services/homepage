# Cloudflare의 LLM 인프라 설계: Prefill/Decode 분리와 커스텀 추론 엔진

대규모 LLM을 글로벌 분산 서비스로 운영하는 것은 일반적인 웹 애플리케이션 운영과 근본적으로 다른 접근이 필요합니다. 2026년 5월, Cloudflare가 자신들의 LLM 추론 인프라 설계 세부 내용을 공개했습니다. 커스텀 추론 엔진 Infire, Prefill/Decode 단계 분리 아키텍처, 그리고 모델 가중치를 15~22% 압축하는 Unweight 기술이 핵심입니다.

self-hosted LLM 배포나 고부하 AI 애플리케이션을 고려하는 팀에게 실용적인 시사점이 많습니다.

## LLM 추론의 두 단계 문제

LLM 추론에는 계산 특성이 근본적으로 다른 두 단계가 있습니다.

**Prefill 단계**: 입력 프롬프트 전체를 한 번에 읽고 KV Cache(어텐션 상태의 키-값 캐시)를 구축합니다. 병렬 처리가 가능하고 높은 FLOPS가 필요한 집약적인 계산 단계입니다.

**Decode 단계**: 이전 토큰에 의존해 출력 토큰을 하나씩 순차 생성합니다. 병렬화가 어렵고 메모리 대역폭에 의존하는 단계입니다.

```
추론 요청 타임라인

─── Prefill ──────────────────────── Decode ───────────────────────────────
전체 프롬프트 읽기 → KV Cache 구축   |  토큰 1 → 2 → 3 → ... 순차 생성
높은 FLOPS, 짧은 버스트              |  높은 메모리 BW, 긴 꼬리
GPU 병렬화 효과적                    |  순차적, 병렬화 어려움
```

두 단계를 같은 GPU에서 실행하면 GPU 활용도가 불균형해집니다. 최적 하드웨어 구성이 서로 다른데 하나의 머신으로 타협해야 합니다.

## Disaggregated Prefill/Decode (단계 분리)

Cloudflare 아키텍처의 핵심은 **Prefill과 Decode를 별도의 머신에서 처리**하는 것입니다.

```
기존 방식 (단일 머신)
Request → [GPU Server: Prefill → Decode] → Response

Cloudflare 방식 (단계 분리)
Request → [Prefill Server] ──KV Cache 전송──→ [Decode Server] → Response
```

KV Cache를 네트워크로 전송하는 비용이 있지만, 이를 통해 세 가지 중요한 것이 가능해집니다.

**하드웨어 최적화 독립성**: Prefill 머신은 높은 FLOPS를 위해, Decode 머신은 높은 메모리 대역폭을 위해 각각 최적화할 수 있습니다. 타협이 필요 없습니다.

**스케일링 독립성**: 긴 프롬프트가 늘어나 Prefill 부하가 높아지면 Decode 용량은 그대로 두고 Prefill 서버만 확장할 수 있습니다. 세밀한 자원 제어가 가능합니다.

**GPU 활용률 향상**: Prefill 전용 머신은 Decode 완료를 기다리지 않고 연속으로 Prefill 작업을 배치 처리할 수 있어 GPU 활용률이 크게 향상됩니다.

## Infire 엔진

Cloudflare는 기존 프레임워크(vLLM, TGI 등) 대신 Rust로 **독자적인 추론 엔진 Infire**를 구축했습니다.

### Paged KV Cache 관리

vLLM의 PagedAttention과 유사하게, GPU 메모리를 고정 크기 페이지로 분할해 관리합니다. 이는 가변 길이 시퀀스를 동시에 처리할 때 발생하는 메모리 단편화를 제거합니다.

```
PagedAttention: GPU 메모리 레이아웃

┌────────────────────────────────────────────────────┐
│ Page 0  │ Page 1  │ Page 2  │ Page 3  │ Page 4 │...│
│ (req_a) │ (req_b) │ (req_a) │ (req_c) │ (req_b)│...│
└────────────────────────────────────────────────────┘

서로 다른 요청이 동일한 물리 메모리 페이지를 공유
사전 할당으로 인한 메모리 낭비 없음
```

### Flash Attention 통합

FlashAttention-2에 상응하는 구현을 사용해 긴 컨텍스트 처리 시 GPU HBM 메모리 사용량을 O(n²)에서 O(n)으로 줄입니다. 128K+ 컨텍스트 윈도우를 가진 모델에서 큰 영향을 미칩니다.

### 효율적인 멀티 GPU 분산

Infire는 외부 오케스트레이터 없이 직접 텐서 병렬화(Tensor Parallelism)를 관리해 GPU 간 통신 오버헤드를 최소화합니다. 70B 이상의 대형 모델을 여러 GPU에 분산해야 할 때 특히 효과적입니다.

## Unweight: 무손실 모델 압축

Cloudflare의 **Unweight** 시스템은 정확도 손실 없이 모델 가중치를 15~22% 압축합니다. 정밀도를 희생하는 양자화(8bit, 4bit)와는 다른 접근 방식입니다.

가중치 텐서의 통계적 중복성을 분석해 정보를 보존하면서 데이터 양을 줄입니다.

```
15~22% 가중치 압축의 실질적 효과

콜드 스타트 개선
  더 작은 모델 → GPU HBM 로드 시간 단축 → 빠른 서비스 시작

메모리 효율화
  VRAM 여유 확보 → 더 큰 배치 크기 → 단위 시간당 더 많은 요청 처리

KV Cache 전송 효율화 (단계 분리 설계와 시너지)
  더 작은 모델 상태 → Prefill → Decode 간 전송 데이터 감소
```

16비트 정밀도의 70B 파라미터 모델(~140GB)에서 20% 압축을 적용하면 약 28GB가 확보됩니다. 이는 8×H100 구성에서 배치 크기를 의미 있게 늘릴 수 있는 여유입니다.

## AI Gateway: 통합 API 레이어

이 추론 인프라 위에서 Cloudflare는 **AI Gateway**를 운영합니다. 70개 이상의 AI 모델에 단일 API로 접근하며, 캐싱, 속도 제한, 비용 가시화, 자동 폴백을 제공합니다.

```typescript
// Cloudflare AI Gateway를 통해 여러 Provider에 통합 접근
const response = await fetch(
  `https://gateway.ai.cloudflare.com/v1/${ACCOUNT_ID}/${GATEWAY_ID}/anthropic/v1/messages`,
  {
    method: 'POST',
    headers: {
      'x-auth-token': CF_API_TOKEN,
      'Content-Type': 'application/json',
    },
    body: JSON.stringify({
      model: 'claude-opus-4-7',
      messages: [{ role: 'user', content: 'Hello' }],
      max_tokens: 1024,
    }),
  }
)
```

프라이머리 Provider에 장애가 발생하면, 애플리케이션 코드 변경 없이 자동으로 폴백 모델로 라우팅합니다.

## self-hosted LLM 배포에 대한 시사점

Cloudflare가 구현한 패턴들은 독점 기술이 아닙니다. 오픈소스 추론 생태계가 이미 같은 방향으로 발전하고 있습니다.

```bash
# vLLM에서 Disaggregated Prefill 활성화 (Cloudflare 방식과 동일한 개념)
# Prefill (producer) 인스턴스
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-70B-Instruct \
  --kv-transfer-config \
  '{"kv_connector":"PyNcclConnector","kv_role":"kv_producer","kv_rank":0}'

# Decode (consumer) 인스턴스
python -m vllm.entrypoints.openai.api_server \
  --model meta-llama/Llama-3.1-70B-Instruct \
  --kv-transfer-config \
  '{"kv_connector":"PyNcclConnector","kv_role":"kv_consumer","kv_rank":1}'
```

프라이버시 요구사항이나 대용량 처리에서의 비용 최적화를 위해 self-hosted LLM을 검토하는 팀이라면, vLLM, SGLang, LMDeploy가 Paged Attention과 FlashAttention을 이미 구현하고 있으며, 2026년에는 Disaggregated Prefill 지원도 추가되고 있습니다.

Cloudflare의 사례가 보여주는 것은 이 기술들이 연구 벤치마크에서만이 아니라, 글로벌 멀티테넌트 규모에서도 실제로 작동한다는 검증입니다.

## 핵심 정리

- LLM 추론의 Prefill과 Decode 단계는 하드웨어 요구사항이 다르며, 함께 실행하면 GPU 효율이 낮아짐
- Cloudflare는 두 단계를 별도 머신에 분리해 하드웨어 최적화와 독립적 스케일링을 실현
- Rust 기반 Infire 엔진이 Paged KV Cache, FlashAttention, 멀티 GPU 텐서 병렬화를 처리
- Unweight는 정확도 손실 없이 모델 가중치를 15~22% 압축해 콜드 스타트와 처리량을 개선
- 동일한 아키텍처 패턴이 vLLM, SGLang으로 self-hosted 배포에서도 활용 가능
