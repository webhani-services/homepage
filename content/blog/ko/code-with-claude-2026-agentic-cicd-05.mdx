2026년 5월 6일, Anthropic은 샌프란시스코에서 Code with Claude를 개최했습니다. 단순한 제품 발표가 아닌, Claude Code를 프로덕션 환경에서 실제로 어떻게 운영할 것인가에 초점을 맞춘 개발자 컨퍼런스였습니다. Claude Code, Claude Platform, Managed Agents의 세 가지 트랙으로 구성되었으며, "에이전틱 시스템으로 워크플로우 재설계하기"부터 GitHub 규모의 엔지니어링, 선제적 에이전트 워크숍까지 실질적인 세션들로 채워졌습니다.

핵심 메시지는 명확했습니다. Claude Code는 더 이상 단순한 자동완성 도구가 아닙니다. 이 이벤트에서 답하려 했던 질문은 이것이었습니다: "어떻게 CI/CD 구성 요소로서 대규모로 운영할 것인가?"

## 장기 태스크: 무엇이 달라졌나

오늘날 대부분의 팀은 AI 코딩 어시스턴트를 인라인 자동완성이나 채팅 기반 질문응답에 주로 사용합니다. Code with Claude 2026은 다른 작동 방식에 초점을 맞췄습니다. 단일 프롬프트에 담기지 않는 여러 단계의 작업인 장기 태스크(Long-Horizon Tasks)가 그것입니다.

장기 태스크의 예시:
- Feature 스펙을 읽고, 여러 파일에 걸쳐 변경사항을 구현하고, 테스트를 추가하고, PR을 생성하기
- 보안 취약점 보고서를 받고, 영향 받는 코드 경로를 파악하고, 수정을 구현하고, 패치를 검증하기
- 여러 서비스에 걸쳐 Breaking API 변경을 일관되게 적용하기

새로운 아이디어가 아닙니다. 달라진 것은 에이전트가 각 단계마다 사람의 개입 없이 이를 완료하는 신뢰성입니다. 이 개선은 더 나은 에이전틱 제어 흐름, 안정적인 도구 사용, 그리고 턴 간 상태 관리 개선에서 비롯됩니다.

## 병렬 에이전트: 아키텍처 패턴

컨퍼런스에서 가장 실용적으로 적용 가능한 패턴은 Claude Code 에이전트를 직렬이 아닌 병렬로 실행하는 것이었습니다.

복잡한 코드베이스를 처리하는 단일 에이전트는 근본적인 제약에 직면합니다. 태스크가 커질수록 Context Window가 포화 상태가 되어 출력 품질이 저하됩니다. 병렬 에이전트는 작업을 독립적으로 처리 가능한 단위로 분할하여 이를 방지합니다.

```bash
# 예시: 마이크로서비스 전반에 걸친 병렬 보안 리뷰
claude --parallel \
  --task "review src/auth-service for security vulnerabilities" \
  --task "review src/payment-service for security vulnerabilities" \
  --task "review src/notification-service for security vulnerabilities" \
  --output-format json \
  --max-turns 20
```

CI/CD에서 이 패턴은 모노레포 구조에 잘 맞습니다. 특정 PR에서 영향받는 서비스만 리뷰하면 됩니다.

```yaml
# .github/workflows/ai-review.yml
name: AI Code Review

on:
  pull_request:
    branches: [main]

jobs:
  detect-changes:
    runs-on: ubuntu-latest
    outputs:
      services: ${{ steps.detect.outputs.services }}
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0
      - id: detect
        run: |
          CHANGED=$(git diff --name-only origin/main...HEAD | \
            grep '^src/' | cut -d'/' -f2 | sort -u | \
            jq -R -s -c 'split("\n")[:-1]')
          echo "services=$CHANGED" >> $GITHUB_OUTPUT

  review:
    needs: detect-changes
    runs-on: ubuntu-latest
    strategy:
      matrix:
        service: ${{ fromJson(needs.detect-changes.outputs.services) }}
    steps:
      - uses: actions/checkout@v4
      - name: Claude Code 리뷰 실행
        run: |
          claude review src/${{ matrix.service }} \
            --focus "bugs,security,performance" \
            --output github-comment
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
```

## 운영 제어: 프로덕션에서 실제로 필요한 것

컨퍼런스 어젠다의 절반은 역량에 관한 것이었고, 나머지 절반은 운영에 관한 것이었습니다. CI/CD 파이프라인에서 실행되는 에이전트를 어떻게 통제할 것인가?

### Permission 범위 지정

Claude Code는 파일을 작성하고, 명령을 실행하고, 외부 API를 호출할 수 있습니다. 프로덕션에서는 이를 명시적으로 제한해야 합니다.

```json
// .claude/settings.json
{
  "permissions": {
    "allow": [
      "Bash(git:*)",
      "Bash(npm run test:*)",
      "Bash(npm run lint:*)",
      "Bash(npm run build:*)"
    ],
    "deny": [
      "Bash(rm:*)",
      "Bash(curl:*)",
      "WebFetch",
      "Bash(git push:*)"
    ]
  }
}
```

거부 목록은 허용 목록만큼 중요합니다. CI 컨텍스트에서는 일반적으로 에이전트가 원격에 푸시하거나, 아웃바운드 HTTP 요청을 하거나, 파괴적인 파일 시스템 작업을 수행하는 것을 원하지 않습니다.

### Evidence Trail(감사 추적)

규제 산업이나 감사 가능성이 필요한 팀의 경우, 에이전트가 *무엇을* 변경했고 *왜* 특정 결정을 내렸는지 파악하는 것이 필수입니다. 에이전트 실행의 구조화된 로깅은 나중에 생각할 것이 아니라 1급 요구사항으로 다뤄야 합니다.

### 비용 거버넌스

대규모로 병렬 에이전트를 실행하면 API 비용이 빠르게 증가할 수 있습니다. 컨퍼런스에서는 Pricing Signal 개념을 소개했습니다. PR별 또는 서비스별 비용 상한을 설정하여 대규모 리팩터링이나 인시던트 대응 중 비용이 폭발하는 것을 방지합니다.

## 멀티 레포지토리 Context 관리

GitHub 규모 엔지니어링 세션에서는 여러 레포지토리와 수천 개의 서비스에 걸쳐 Claude Code를 적용하는 방법을 다뤘습니다.

핵심 패턴은 명시적인 Context 사전 로딩입니다. 에이전트가 작업을 시작하기 전에, 해당 영역에 적용되는 관련 API 계약, 테스트 관례, 코딩 표준을 제공합니다.

```bash
# 크로스 레포지토리 태스크 시작 전 Context 사전 로딩
claude \
  --context-file .claude/project-context.md \
  --context-file docs/api-contracts.md \
  --context-file docs/coding-standards.md \
  --context-file docs/test-patterns.md \
  "Update the user service to support the new authentication token format"
```

이를 하지 않으면 에이전트가 팀의 관례에서 벗어난 가정을 하게 됩니다. 기술적으로는 맞지만 스타일이 일관되지 않은 결과물이 생성되어 리뷰 부담이 늘어납니다.

## 멘탈 모델의 전환

Code with Claude 2026에서 가장 중요한 교훈은 특정 기능이나 API가 아닙니다. AI 코딩 도구에 대한 생각 방식의 전환입니다.

이전 멘탈 모델: 개발자가 더 빠르게 작업할 수 있도록 돕는 대화형 어시스턴트.

새로운 멘탈 모델: 소프트웨어 딜리버리 파이프라인에서 정의된 태스크를 독립적으로 처리하는 자율적인 참여자. 사람은 매 단계를 안내하는 대신 결과물을 검토합니다.

이 전환에는 새로운 엔지니어링 작업이 필요합니다. 태스크 경계 정의, Permission 아키텍처, 관찰 가능성 인프라, 비용 거버넌스가 그것입니다. Claude Code를 더 유능한 자동완성 도구로 접근하는 팀은 그 잠재력을 충분히 활용하지 못할 것입니다. 적절한 제어와 함께 자율 운영을 위해 설계하는 팀이 훨씬 더 큰 레버리지를 얻을 것입니다.

## 시작하는 방법

이 모델을 도입하려는 팀을 위한 제안:

1. **먼저 좁은 태스크 경계를 정의하기**: 명확한 입력과 검증 가능한 출력이 있는 태스크(코드 리뷰, 테스트 생성, 의존성 업데이트)부터 시작하기. 기본 신뢰성을 확립하기 전까지 개방형 태스크는 피하기.
2. **확장 전에 계측하기**: 중요한 경로에 에이전트를 실행하기 전에 로깅, 비용 추적, 성공률 측정을 먼저 설정하기.
3. **기본 거부(Default-Deny) Permission**: 가장 제한적인 Permission 세트로 시작하고, 특정 필요를 정당화할 수 있을 때만 Permission을 추가하기.
4. **에이전트 결과물을 주니어 기여자처럼 리뷰하기**: 해당 코드베이스에서 에이전트의 동작에 대한 신뢰를 확립하기 전까지, 결과물을 샘플 검토가 아닌 전체 리뷰가 필요한 것으로 취급하기.

Agentic CI/CD를 위한 인프라는 이제 충분히 성숙했습니다. 병목은 기술적 역량이 아닌 조직 설계에 있습니다.
