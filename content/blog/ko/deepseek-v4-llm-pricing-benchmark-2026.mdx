"최고 수준의 성능에는 최고 수준의 가격이 따른다" — LLM 시장을 오랫동안 지배해온 이 전제를, 2026년 3월 초 출시된 DeepSeek V4가 다시 한번 흔들고 있습니다.

입력 토큰 100만 개당 $0.30, 출력 토큰 100만 개당 $0.50이라는 가격은 현재 시장에서 프로덕션 등급 모델 중 가장 경쟁력 있는 수준에 속합니다. 그런데 더 주목할 만한 것은 Benchmark 성능입니다. SWE-bench Verified에서 81%를 달성해 전 버전 V3(69%)에서 12포인트 향상되었습니다. SWE-bench를 실제 코딩 능력의 지표로 삼는 팀이라면, V4는 3~5배 비싼 모델과 경쟁할 수 있는 위치에 놓입니다.

## V4에서 무엇이 달라졌나

V4의 개선 사항은 이해할 가치가 있는 몇 가지 핵심 영역에 집중되어 있습니다.

**Architecture**: V4는 Mixture-of-Experts(MoE) 설계를 채택하여 Forward Pass마다 파라미터의 일부만 활성화합니다. DeepSeek가 V3에서 추론 비용을 크게 줄인 것과 동일한 원칙을 V4에서 더욱 발전시킨 것입니다. 동등한 파라미터 수의 Dense Model 대비 비용당 추론 능력이 향상된다는 실질적인 효과가 있습니다.

**Coding과 Structured Output**: SWE-bench Verified에서 69%에서 81%로 향상된 것은 주로 Multi-step Code 생성 태스크에서의 지시 따르기 능력 개선 덕분입니다. JSON Mode 신뢰성과 Function Calling 정확도도 향상되었는데, 이는 출력 파싱 실패가 하위 단계로 연쇄되는 Agentic 워크플로우에서 특히 중요합니다.

**Context Caching**: Cache Hit은 입력 토큰 100만 개당 $0.028로, 기본 요금의 약 10분의 1 수준입니다. 긴 System Prompt나 반복 Context를 사용하는 Application(RAG Pipeline, 프로젝트 수준 Context를 가진 Code Assistant 등)에서는 대규모로도 Persistent Context가 경제적으로 실현 가능해집니다.

## 실용적인 비용 분석

V4를 프로덕션에서 평가할 때 적절한 질문은 "이것이 최고의 모델인가?"가 아니라 "이 워크로드에서 이 가격으로 충분한 성능을 발휘하는가?"입니다.

요약, 분류, Code 생성, 문서 Q&A 등 일반적인 유즈케이스 대부분에서 V4는 모델 능력이 병목이 되는 임계값을 훨씬 상회합니다. 실제 병목은 대부분 Retrieval 품질, Prompt 설계, Latency에 있고 모델 자체의 지능이 아닙니다.

월 1,000만 건의 요청을 처리하는 RAG Pipeline을 가정해봅시다. 각 요청이 2,000개의 입력 토큰과 500개의 출력 토큰을 처리한다면, V4 가격 기준으로:

```
Input:  1,000만 × 2,000 / 100만 × $0.30 = $6,000
Output: 1,000만 × 500   / 100만 × $0.50 = $2,500
합계:                                      $8,500/월
```

Cache Hit으로 반복 Context 비용이 약 90% 절감될 경우, Prompt Heavy Application의 실제 합계는 훨씬 낮아집니다. 출력 토큰 100만 개당 $3~$5 수준의 모델과 비교하면, 이 요청 규모에서 월 $30,000~$50,000의 차이가 발생할 수 있습니다.

이런 비용 구조는 이전에는 API 비용이 장벽이었던 중규모 프로덕트에서도 LLM 통합을 경제적으로 실현 가능하게 만듭니다.

## V4가 적합하지 않은 경우

모든 시나리오에서 V4가 최선의 선택은 아닙니다.

복잡한 Multi-step 추론 — 장기 계획 수립, 적대적 문제 해결, 상충하는 정보 통합 — 에서는 Claude Opus 4.6이나 GPT-5 같은 모델이 여전히 유의미한 우위를 보입니다. V4의 SWE-bench 81% 스코어는 강력하지만, 많은 단계에 걸쳐 정밀한 추론이 필요한 태스크에서는 현재 최고 수준과의 격차가 실재합니다.

Latency와 Availability도 고려 사항입니다. V4는 DeepSeek 자체 API Infrastructure를 통해 제공되므로, 그들의 Rate Limit과 지역적 제약에 의존합니다. 엄격한 Latency SLA나 중국 이외 지역의 Data Residency 요건이 있는 팀은 Third-party V4 호스팅이나 Fallback Logic 추가 등 Architecture 상의 대응이 필요할 수 있습니다.

## webhani의 관점

webhani에서는 모델 선택을 다른 Infrastructure 의사결정과 동일하게 접근합니다. 요건에서 시작하고, 적합성을 평가하며, 사용하지 않는 기능에 비용을 지불하지 않는 것입니다.

Content 생성, Developer Tooling, Structured Data 추출을 포함한 대부분의 고객 프로젝트에서 V4는 성능과 경제성의 매력적인 조합을 제공합니다. V4, Claude Opus 4.6, Gemini 3.1 Pro를 대표적인 Enterprise 워크로드로 비교 평가한 결과, 프론티어 수준의 추론이 필요하지 않은 태스크에서 V4는 상위 모델 대비 5~10% 이내의 성능을 일관되게 발휘하면서 비용은 크게 낮았습니다.

우리의 권고: 비용이 중요한 프로덕션 워크로드에는 V4를 기본 선택지로 삼고, Benchmark 격차가 측정 가능한 비즈니스 임팩트로 이어지는 태스크에만 고가 모델을 사용하세요.

DeepSeek의 궤적에서 얻을 수 있는 더 큰 교훈은 특정 모델에 관한 것이 아닙니다. 비용 대비 성능의 프론티어는 대부분의 가격 전제가 예상하는 것보다 빠르게 이동하고 있습니다. 비용 효율성을 처음부터 일급 설계 제약으로 삼아 LLM Infrastructure를 구축하세요.
