## 독자 노선에서 "다가서기"로

Deno는 본래 Node.js에 대한 반성에서 나온 Runtime이었습니다. 기본으로 TypeScript를 실행할 수 있고, Permission Model을 가지며, Web 표준 API를 우선에 둡니다 — 설계 사상은 명확하고 매력적이었습니다. 다만 현장에서의 채택은 기존 npm Ecosystem과의 거리가 벽이 되어 왔습니다.

2026년의 Update는 그 거리를 적극적으로 좁히러 옵니다. Deno 2.4에서는 `deno bundle`이 부활했고, Node.js의 API와 Global에 대한 호환성이 한층 개선되었습니다. Framework 쪽에서도 Deno 순정 Fresh가 2.0 beta에서 Vite 통합으로 방향을 틀었습니다. 독자 노선을 고수하기보다 기존 Tool에 다가서는 이 현실 노선을 webhani의 채택 시각으로 풀어봅니다.

## Deno 2.4: `deno bundle` 부활이 뜻하는 것

한때 Deno에는 `deno bundle`이 있었지만, 한 번 Deprecated·삭제되었습니다. 2.4에서 다시 공식 Bundler로 돌아온 것은 실무적 요청의 반영입니다. 단일 File로의 Bundle은 CLI Tool 배포, Edge 환경 Deploy, 의존을 고정한 산출물 생성 같은 장면에서 지금도 수요가 높습니다.

```bash
# Entry point에서 단일 JS로 Bundle한다
deno bundle main.ts --output dist/app.js

# 실행 시 권한은 명시적으로 부여한다 (Deno의 Security Model)
deno run --allow-net --allow-read dist/app.js
```

주목할 점은 Bundler가 돌아와도 Deno의 Permission Model은 바뀌지 않았다는 것입니다. `--allow-net`이나 `--allow-read`를 명시하지 않는 한 Network나 File 접근은 거부됩니다. Supply Chain 공격이 현실적 위협이 된 지금, "기본적으로 아무것도 못 한다"는 전제는 Node.js에 대한 분명한 차별화 요소로서 의미가 커집니다.

Node.js 호환 개선도 실무에서는 큽니다. 기존 npm Package나 Node의 Global(`process` 등)이 동작하는 범위가 넓어지면 "Deno는 좋아 보이지만 보유 자산이 안 돈다"는 도입 시 마찰이 줄어듭니다.

## Fresh 2.0: 독자 Bundler를 버리고 Vite에 올라타기

더 상징적인 것은 Fresh 2.0 beta입니다. Fresh는 Deno 순정 Full-stack Framework로, 지금까지 독자적인 Build Pipeline을 가지고 있었습니다. 2.0 beta에서는 Option으로 Vite 통합을 도입해 HMR(Hot Reload), 기동 시간 단축, React의 Alias 해석, 그리고 Vite Plugin Ecosystem 전체를 받아들일 수 있게 했습니다.

이것은 기술적 타협이 아니라 현명한 판단이라고 봅니다. Vite는 이미 사실상의 표준 Build Tool이고, Vite 8에서는 Rolldown(Rust 제 Bundler)으로 Build 성능도 끌어올렸습니다. 여기에 독자 구현으로 계속 맞서기보다 성숙한 Ecosystem에 올라타는 편이 개발자 경험도, 유지보수 지속성도 높습니다.

```tsx
// Fresh Route 예: File 기반 Routing은 유지하면서,
// Build 기반에 Vite를 채택할 수 있게 되었다
export default function Page(props: { data: { title: string } }) {
  return (
    <main class="p-8">
      <h1 class="text-2xl font-bold">{props.data.title}</h1>
      <p>Fresh는 Island 단위로만 Client JS를 보낸다.</p>
    </main>
  );
}
```

Fresh의 "Islands Architecture" — Interactive한 부분에만 Client JS를 보내고, 그 외에는 Server Rendering만으로 끝내는 사고방식 — 은 그대로 유지됩니다. 설계 사상은 지키면서 Build 기반만 실적 있는 것으로 갈아 끼운 형태입니다.

## webhani의 채택 시각: 어디서 Deno를 고를까

현실 노선으로의 전환은 환영할 일이지만, 그렇다고 Deno를 모든 장면에서 골라야 한다는 이야기는 아닙니다. 저희가 Client에게 조언할 때의 판단 축은 다음과 같습니다.

- **신규 CLI Tool / 작은 Edge Function**: TypeScript를 추가 설정 없이 돌리고 `deno bundle`로 단일 산출물을 만들 수 있습니다. Deno의 강점이 순수하게 통하는 영역.
- **Security 요건이 엄격한 Script 실행**: 명시적 Permission Model 덕에 외부 의존을 돌릴 때의 공격면을 제어하기 쉽습니다.
- **기존 대규모 Node.js 자산 이전**: 호환성은 개선되었지만 의존 Tree 전체의 검증이 필요합니다. 단계적으로 주변 Tool 무리부터 시도하는 것이 현실적.
- **Frontend 신규 Project**: Fresh 2.0의 Vite 통합으로 선택지에 들지만, beta라는 점은 Production 투입 전 반드시 확인합니다.

한마디로 Deno는 "Node.js의 완전한 대체"가 아니라 "특정 요건에 강한 선택지"로 보는 것이 적절합니다. TypeScript Native 실행, 명시적 Permission, Web 표준 준수가 통하는 장면을 가려내 투입합니다.

## 정리

Deno 2.4의 `deno bundle` 부활과 Fresh 2.0의 Vite 통합은 모두 "독자성을 옅게 하는" 방향의 변화입니다. 하지만 이것은 후퇴가 아니라 Ecosystem의 현실에 다가서는 성숙으로 봐야 할 것입니다. 독자적 설계 사상(TypeScript Native, Permission Model, Web 표준)이라는 핵심은 지키면서 주변의 마찰을 줄이고 있습니다.

webhani로서는 Deno를 도입 후보에서 빼 두었던 조직이야말로 이 현실 노선을 계기로 재평가할 가치가 있다고 봅니다. 우선 작은 CLI Tool이나 Edge Function에서 `deno bundle`을 시도하고, Frontend는 Fresh 2.0의 안정판화를 가늠한 뒤에 — 라는 단계적 접근을 권합니다.
