Docker Compose로 운영해 온 Service를 Kubernetes로 옮기는 판단은 2026년 시점에도 많은 팀이 마주하는 주제입니다. Compose는 단일 File로 완결되는 편리함이 매력이지만, Scale이나 가용성, Rollout 제어가 필요해지면 Kubernetes의 이점이 커집니다. 이 글에서는 변환 Tool의 활용처와, 자동 변환으로는 메워지지 않는 지점의 손질을 실무 관점에서 정리합니다.

## Compose와 Kubernetes의 구조적 차이

먼저 짚어야 할 것은 둘의 "입도" 차이입니다. Docker Compose에서는 Service 정의도 Network도 Volume도 하나의 YAML에 모여 있습니다. 반면 Kubernetes는 Deployment, Service, Volume, Secret처럼 역할별로 별개의 Manifest를 요구합니다.

대응 관계를 대략 보이면 다음과 같습니다.

- Compose의 `services` → Kubernetes의 Deployment (상태를 가지면 StatefulSet, Node 상주면 DaemonSet)
- Compose의 `ports`(공개 Port) → Service (Service 간 통신·외부 공개)
- Compose의 `volumes` → PersistentVolumeClaim
- Compose의 `environment` / Secret 상당 → ConfigMap / Secret

이 "1 대 다"의 전개가 수작업으로는 품이 들고, 변환 Tool이 유용한 부분입니다.

## 변환 Tool의 활용처

대표적인 선택지로 Kompose와 Compose Bridge가 있습니다.

**Kompose**는 `kompose convert`로 Compose File에서 Kubernetes Manifest를 자동 생성합니다. 다만 공개 자료에서도 70~80% 정도의 자동 변환에 그친다고 하며, 나머지는 수동으로 메우는 전제로 쓰는 것이 현실적입니다. 생성된 Manifest를 초안으로 삼고 Health Check, Resource 제한, Secret, NetworkPolicy를 나중에 보완합니다.

**Compose Bridge**는 Application Service에 대해 지정한 Instance 수를 유지하는 Deployment, Service 간 통신을 위한 Service, 공개 Port용 Service 등을 생성합니다.

어느 쪽을 쓰든 중요한 것은 "자동 생성된 Manifest를 그대로 Production에 올리지 않는" 것입니다. 변환은 어디까지나 출발점이고, Production을 견디는 구성으로 만드는 것은 그 이후의 작업입니다.

## 자동 변환으로 메워지지 않는 지점

webhani가 이전 지원에서 반드시 확인하는 것은 다음 4가지입니다. 이 부분이 빠진 채 Production에 올리면 장애나 Scale 불량의 온상이 됩니다.

### 1. Health Check(Probe)

Compose의 `healthcheck`는 Kubernetes의 Liveness / Readiness Probe에 대응하지만, 자동 변환으로는 충분히 반영되지 않는 경우가 많습니다. Readiness Probe가 없으면 아직 준비되지 않은 Pod로 Traffic이 흘러갑니다.

```yaml
# Deployment의 containers[]에 추가하는 Probe 예
livenessProbe:
  httpGet:
    path: /healthz
    port: 8000
  initialDelaySeconds: 10
  periodSeconds: 10
readinessProbe:
  httpGet:
    path: /ready
    port: 8000
  initialDelaySeconds: 5
  periodSeconds: 5
```

### 2. Resource 제한(requests / limits)

Compose에서는 생략하기 쉽지만, Kubernetes에서는 `requests` / `limits`를 설정하지 않으면 Scheduling이 불안정해지고, 다른 Pod를 끌어들이는 형태로 Node가 압박됩니다.

```yaml
resources:
  requests:
    cpu: "250m"
    memory: "256Mi"
  limits:
    cpu: "500m"
    memory: "512Mi"
```

### 3. Secret 관리

Compose의 `environment`에 직접 적었던 인증 정보는 Kubernetes에서는 Secret으로 분리합니다. 자동 변환은 환경 변수를 그대로 ConfigMap에 떨구는 경우가 있으므로, 기밀 정보가 평문으로 남아 있지 않은지 반드시 확인합니다.

### 4. Network Policy

Compose의 Network 분리는 Kubernetes에서는 NetworkPolicy로 명시적으로 정의해야 합니다. 기본값으로는 Pod 간 통신이 전부 허용되므로, 필요한 경로만 허용하는 설계로 고칩니다.

## Helm과 Kustomize의 역할 분담

변환 후 Manifest를 환경별로 운영하려면 Helm이나 Kustomize를 씁니다. 둘은 경합한다기보다 역할이 다릅니다.

- **Helm**: Package 관리. 복잡한 Application을 하나의 Chart로 정의·설치·Upgrade할 수 있고, 값 교체(`values.yaml`)로 재사용성이 높습니다.
- **Kustomize**: 구성 관리. 원본 Manifest를 고치지 않고 개발·Staging·Production 같은 환경 차이를 Overlay로 표현합니다.

실무에서는 Compose에서 생성한 Manifest를 Base로 삼아 환경 차이를 Kustomize Overlay로 관리하거나, Helm Chart로 만들어 `values`로 전환하는 두 접근으로 정착하는 경우가 많습니다. 팀이 Helm 운영에 익숙하다면 Chart화, 단순하게 유지하고 싶다면 Kustomize라는 선택이 이해하기 쉽습니다.

## 이전 진행 방식

한 번에 전 Service를 옮기는 것은 권장하지 않습니다. webhani는 다음 순서를 권합니다.

1. **Stateless한 Service부터 옮긴다.** Web Application이나 API처럼 상태를 갖지 않는 것은 Deployment화의 Risk가 작습니다.
2. **Data Store는 신중하게.** PersistentVolume 설계와 Backup 전략을 굳힌 뒤 옮깁니다. 안이한 StatefulSet화는 피합니다.
3. **Probe·Resource 제한·Secret을 이전의 필수 항목으로 삼는다.** 변환 Tool의 출력을 그대로 쓰지 않습니다.
4. **Production 전환 전에 부하 시험을 한다.** Scale 거동과 Probe 임계값은 실제 부하를 걸지 않으면 타당성을 판단할 수 없습니다.

## 정리

Docker Compose에서 Kubernetes로의 이전은 변환 Tool로 7~8할을 기계적으로 진행할 수 있지만, Production을 견디는 구성으로 만드는 것은 나머지 손질에 달려 있습니다. Health Check, Resource 제한, Secret 관리, Network Policy — 이 4가지를 이전의 필수 항목으로 다루고, 환경 차이는 Helm이나 Kustomize로 정리합니다. 이 순서를 밟으면 이전은 착실하게 진행됩니다.
