Docker가 새로운 플랫폼 **Docker Kanvas**를 발표했습니다. 핵심 기능은 `docker-compose.yml`을 Kubernetes 매니페스트로 자동 변환하는 것입니다. Helm과 Kustomize의 대안으로 포지셔닝하고 있어 주목받고 있습니다.

## Docker Kanvas란?

Kanvas는 로컬 Docker Compose 환경에서 프로덕션 규모의 Kubernetes 환경으로의 전환을 자동화하는 플랫폼입니다. 개발자가 이미 익숙한 Compose 정의를 활용하여 Kubernetes에 필요한 리소스(Deployment, Service, ConfigMap, Secret 등)를 생성합니다.

```bash
# Compose를 Kubernetes 매니페스트로 변환
kanvas convert -f docker-compose.yml -o k8s/

# 생성된 파일 확인 및 조정
ls k8s/
# deployment.yaml  service.yaml  configmap.yaml  secrets.yaml  ...

# Kubernetes에 배포
kubectl apply -f k8s/
```

## 실제 변환 예시

Node.js + PostgreSQL 스택을 예로 들겠습니다.

```yaml
# docker-compose.yml
services:
  app:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://user:pass@db:5432/mydb
    depends_on:
      - db

  db:
    image: postgres:16
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: pass
      POSTGRES_DB: mydb
    volumes:
      - postgres_data:/var/lib/postgresql/data
```

Kanvas는 이를 다음과 같은 Kubernetes 리소스로 변환합니다:

- App용 `Deployment`
- Database용 `StatefulSet` (영구 스토리지)
- 내부 네트워크를 위한 `Service`
- 데이터베이스 자격증명을 위한 `Secret` (Environment Variable에서 자동 추출)

환경 변수에서 Secret을 자동으로 추출하는 기능은 수동 변환 대비 실용적인 개선입니다.

## Helm, Kustomize와의 비교

**Helm과 비교:**
Helm의 강점은 풍부한 Chart 에코시스템과 템플릿 유연성입니다. 복잡한 조건부 로직과 릴리즈 라이프사이클 관리를 표현할 수 있습니다. 단, 학습 비용이 높고 Template 문법이 복잡합니다.

Kanvas는 반대 철학을 취합니다. 이미 가지고 있는 것(Compose)에서 시작하여 생성합니다. 최대 활용 범위는 Helm보다 낮지만, 시작 장벽도 훨씬 낮습니다.

**Kustomize와 비교:**
Kustomize는 기존 매니페스트에 패치를 레이어링하는 데 뛰어납니다. Dev/Staging/Prod 변형을 중복 없이 관리할 수 있습니다. 다만 초기 매니페스트 생성은 도와주지 않습니다. Kanvas는 바로 이 간극을 채웁니다.

| 도구 | 강점 | 한계 |
|------|------|------|
| Helm | 풍부한 에코시스템, 전체 라이프사이클 관리 | 높은 복잡도, 가파른 학습 곡선 |
| Kustomize | 깔끔한 오버레이 전략 | 초기 매니페스트 생성 불가 |
| Kanvas | 기존 Compose에서 즉시 시작 | 초기 단계, 복잡한 설정은 수동 조정 필요 |

## 생성 결과물의 한계

생성된 매니페스트는 시작점이지 프로덕션 준비 완료 설정이 아닙니다. 수동으로 추가해야 할 항목들이 있습니다.

```yaml
# Kanvas가 생성하는 것 — 맞지만 불완전함
spec:
  containers:
    - name: app
      image: my-app:latest

# 프로덕션 적용 전 추가해야 할 설정
      resources:
        requests:          # 스케줄링에 영향을 주는 CPU/메모리 요청
          cpu: 100m
          memory: 256Mi
        limits:
          cpu: 500m
          memory: 512Mi
      livenessProbe:       # Pod가 정상 상태인지 Kubernetes가 확인하는 방법
        httpGet:
          path: /health
          port: 3000
        initialDelaySeconds: 30
        periodSeconds: 10
      readinessProbe:      # 트래픽을 언제 보낼지 결정하는 기준
        httpGet:
          path: /ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 5
```

**Secret 관리는 별도 결정이 필요합니다.** Compose `environment`의 자격 증명은 Kubernetes Secret으로 변환되지만, Secret 관리 전략(External Secrets Operator, Sealed Secrets, Vault 연동 등)은 인프라마다 다르며 Kanvas가 처리하지 않습니다.

NetworkPolicy, RBAC, Pod 보안 설정은 생성 결과물에 포함되지 않습니다. 테스트 환경 이상을 목표로 한다면 반드시 추가해야 합니다.

## Kanvas가 적합한 상황

다음 시나리오에서 Kanvas는 효과적입니다:

- **첫 Kubernetes 도입**: Compose 기반 앱을 처음으로 Kubernetes에 마이그레이션하는 팀
- **소~중규모 프로젝트**: Helm의 복잡도가 불필요하고 깔끔한 시작 매니페스트가 필요한 경우
- **빠른 프로토타이핑**: Compose 앱을 Kubernetes 환경에서 빠르게 테스트하고 싶을 때

이미 Helm이나 Kustomize로 잘 관리되고 있는 프로젝트나 복잡한 네트워크 요구사항이 있는 대규모 프로덕션 시스템에는 효용이 제한적입니다.

## 정리

Docker Kanvas는 변환을 자동화하여 Compose에서 Kubernetes로 이동하는 마찰을 줄여줍니다. 기존 Compose 정의를 가진 팀이 Helm 차트를 처음부터 작성하지 않고 Kubernetes를 시작할 수 있는 실용적인 도구입니다.

베이스라인 생성에 활용하고, 이후 프로덕션 강화 작업에 시간을 투자하세요. Resource Request, Health Probe, Secret 관리, NetworkPolicy 설정은 여전히 개발자의 몫이지만, 빈 파일이 아닌 동작하는 기반에서 시작할 수 있습니다.
