Docker가 Kanvas를 발표했습니다. Docker Compose 파일을 Kubernetes 아티팩트로 자동 변환하는 플랫폼입니다. KubeCon + CloudNativeCon Europe 2026에서 주목받으며 Helm과 Kustomize에 대한 도전으로 받아들여지고 있습니다.

## 해결하려는 문제

컨테이너 기반 워크플로우의 오래된 고민 중 하나는 로컬 개발과 프로덕션 사이의 설정 형식 차이입니다. 개발자는 `docker-compose.yml`로 서비스를 정의하지만, 프로덕션은 Kubernetes에서 실행됩니다. Kubernetes는 Deployment, Service, Ingress 등 여러 리소스가 필요하고, Compose 개념과 일대일로 대응하지 않습니다.

이 변환 작업은 Kubernetes 지식이 없는 개발자에게 높은 장벽입니다. Kanvas는 이 과정을 자동화합니다:

```yaml
# docker-compose.yml
services:
  web:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://db:5432/myapp
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
```

Kanvas는 이 파일을 읽고 대응하는 Kubernetes 매니페스트를 생성합니다.

## Helm, Kustomize와의 비교

Helm과 Kustomize는 각각 성숙한 생태계를 가지고 있습니다. Kanvas는 이들을 대체하는 것이 아니라 다른 위치를 차지합니다:

| | Helm | Kustomize | Kanvas |
|--|------|-----------|--------|
| 시작점 | Helm Chart (자체 형식) | Base YAML | Docker Compose |
| 학습 곡선 | 높음 (템플릿 언어) | 중간 | 낮음 (Compose 지식으로 가능) |
| 대상 사용자 | Kubernetes 전문가 | K8s 지식 필요 | Compose 사용자 |
| 프로덕션 실적 | 풍부 | 풍부 | 신규 |

Kanvas의 가치 제안은 Compose는 잘 알지만 Kubernetes 전문성이 없는 팀을 위한 것입니다.

## 적합한 사용 시나리오

**잘 맞는 경우:**
- 로컬 개발에 Compose를 사용하고 Kubernetes로의 경로가 필요한 팀
- 전담 DevOps 엔지니어가 없는 Frontend 또는 Fullstack 팀
- Helm의 복잡성이 필요하지 않은 소규모 서비스

**맞지 않는 경우:**
- 이미 Helm을 프로덕션에서 운영 중인 프로젝트 (마이그레이션 비용 불필요)
- HPA, NetworkPolicy, PodDisruptionBudget 등 Kubernetes 전용 리소스를 많이 사용하는 워크로드
- 멀티 클러스터나 고급 패키지 관리가 필요한 경우

## 자동 생성 결과물의 실제

Kanvas가 주는 것은 시작점이지 프로덕션 준비 완료 매니페스트가 아닙니다:

```yaml
# 생성된 Deployment (간략화)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web
spec:
  replicas: 1  # 프로덕션에서는 조정 필요
  selector:
    matchLabels:
      app: web
  template:
    spec:
      containers:
        - name: web
          image: my-app:latest
          # 추가로 필요한 설정:
          # resources:
          #   requests: { cpu: "100m", memory: "128Mi" }
          #   limits: { cpu: "500m", memory: "512Mi" }
          # livenessProbe: ...
          # readinessProbe: ...
```

프로덕션 투입 전 수동으로 추가해야 할 사항:
- **Resource 제한**: `resources.requests`와 `resources.limits`
- **헬스 체크**: `livenessProbe`와 `readinessProbe`
- **Secret 관리**: Compose의 환경 변수는 Kubernetes Secret 또는 외부 시크릿 관리 도구로 이전 필요
- **영구 스토리지**: Compose의 Volume은 PersistentVolumeClaim으로 변환 필요

## KubeCon 2026의 맥락

KubeCon + CloudNativeCon Europe 2026에서 공개된 데이터에 따르면, 전 세계 Cloud Native 개발자 인구는 1,990만 명에 달했습니다. 이 성장의 상당 부분은 Kubernetes를 직접 다뤄본 적 없는 개발자들에 의한 것입니다.

Kanvas는 이 계층, 즉 Compose에는 익숙하지만 Kubernetes는 진입 장벽이 높다고 느끼는 엔지니어들을 타겟으로 한 전략적 제품입니다.

Kubernetes 1.35에서는 Dynamic Resource Allocation(DRA)의 binding conditions가 beta로 승격되어 AI/ML 워크로드의 GPU 리소스 관리가 개선됩니다. 향후 Kanvas가 AI 워크로드의 Kubernetes 배포를 단순화하는 방향으로 발전할 가능성도 있습니다.

## 정리

Kanvas는 특정 용도에 명확하게 정의된 도구입니다: 깊은 Kubernetes 지식 없이 Docker Compose에서 Kubernetes로 이동하는 것. Helm이나 Kustomize의 대안이 아닌 보완재입니다.

Kubernetes 도입을 설정 복잡성 때문에 미루고 있는 팀에게 Kanvas는 그 장벽을 실질적으로 낮춥니다. 생성된 결과물을 시작점으로 삼아 프로덕션에 필요한 설정을 추가하면, 처음부터 전담 인프라 엔지니어 없이도 실용적인 배포 경로를 만들 수 있습니다.
