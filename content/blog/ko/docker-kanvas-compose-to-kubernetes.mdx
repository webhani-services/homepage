## Compose와 Kubernetes 사이의 간격

많은 팀이 로컬 개발에는 Docker Compose를, 프로덕션에는 Kubernetes를 사용합니다. 이 둘 사이의 간격은 단순하지 않습니다. Compose의 Service 정의는 Kubernetes Deployment, Service, PersistentVolumeClaim, ConfigMap에 자동으로 대응되지 않으며, 변환 작업에는 Kubernetes 지식과 시간이 필요합니다.

Docker Kanvas는 이 문제를 직접 해결하려는 접근입니다. Compose 파일을 입력하면 Kubernetes 매니페스트가 출력됩니다. 생성된 리소스 간의 관계를 시각적으로 보여주는 Canvas UI도 함께 제공됩니다.

## 변환 방식

Kanvas는 Compose의 각 요소를 Kubernetes 리소스에 의미적으로 매핑합니다.

```yaml
# docker-compose.yml — Kanvas의 입력
version: '3.8'
services:
  api:
    image: myapp/api:latest
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://db:5432/myapp
    depends_on:
      - db
    deploy:
      replicas: 2
      resources:
        limits:
          memory: 512M

  db:
    image: postgres:16
    environment:
      POSTGRES_DB: myapp
      POSTGRES_USER: admin
      POSTGRES_PASSWORD: secret
    volumes:
      - db_data:/var/lib/postgresql/data

volumes:
  db_data:
```

이 입력으로 Kanvas가 생성하는 것:

- 2개 레플리카와 메모리 제한이 설정된 `api` Deployment
- 포트 3000을 노출하는 Service
- `db` 컨테이너를 위한 Deployment
- `db_data` 볼륨을 위한 PersistentVolumeClaim
- `environment`에서 발견된 자격 증명을 담은 Secret

Canvas UI는 이 리소스들을 그래프로 렌더링하여, 클러스터에 적용하기 전에 의존 관계를 확인할 수 있게 합니다.

## Helm, Kustomize와의 위치 비교

Kanvas는 Helm을 대체하려는 것이 아닙니다. 비교는 더 세밀하게 봐야 합니다.

| 관점 | Docker Kanvas | Helm | Kustomize |
|---|---|---|---|
| 입력 형식 | Docker Compose 파일 | Go 템플릿 차트 | Base YAML + Overlay |
| 학습 비용 | 낮음 (Compose를 알면) | 중~높음 | 중간 |
| 환경 차이 관리 | 개발 중 | Chart 값 + values 파일 | Overlay 패치 |
| 에코시스템 성숙도 | 초기 단계 | 성숙 (Artifact Hub) | 성숙 (kubectl 내장) |
| 시각화 | 있음 | 없음 (기본) | 없음 |

Kanvas는 Helm 차트 작성에 수 주를 투자하지 않고 Kubernetes에 진입하려는 Compose 사용자를 위한 입구로 이해하는 것이 적절합니다. 이미 성숙한 차트 라이브러리와 배포 파이프라인을 가진 팀에게 Helm의 대체제가 되는 것은 아닙니다.

## 실무 활용 시나리오

**소~중규모 서비스의 Kubernetes 마이그레이션**

3~5개 서비스를 Compose로 운영하다가 스케일링 한계에 부딪혀 Kubernetes가 필요한 팀. 전담 플랫폼 엔지니어 없이 Helm 차트를 처음부터 작성하기는 어렵습니다. Kanvas는 몇 분 만에 동작하는 베이스라인을 생성합니다. 여기에 Liveness Probe, Resource Request, RBAC를 추가하는 작업이 필요하지만, 매니페스트를 처음부터 작성하는 것보다 훨씬 빠릅니다.

**실전 피드백을 통한 Kubernetes 학습**

생성된 매니페스트는 읽을 수 있는 형태입니다. Compose 필드가 Kubernetes 리소스로 어떻게 변환되는지 따라가면서 매핑을 배울 수 있습니다. 빈 YAML 파일에서 시작하는 것보다 실질적인 학습 경로입니다.

**프로토타입의 신속한 클러스터 배포**

PoC 애플리케이션을 Kubernetes 클러스터에 빠르게 배포하되, 실제 프로덕션 배포 전에 매니페스트를 정리할 의도가 있는 경우.

## 생성 결과물의 한계

생성된 매니페스트는 시작점이지 프로덕션 준비 완료 설정이 아닙니다. 수동으로 추가해야 할 항목들이 있습니다.

```yaml
# Kanvas가 생성하는 것 — 맞지만 불완전함
spec:
  containers:
    - name: api
      image: myapp/api:latest
      resources:
        limits:
          memory: 512Mi

# 프로덕션 적용 전 추가해야 할 설정
      resources:
        requests:          # 누락: 스케줄링에 영향을 주는 CPU/메모리 요청
          cpu: 100m
          memory: 256Mi
        limits:
          cpu: 500m
          memory: 512Mi
      livenessProbe:       # 누락: Pod가 정상 상태인지 Kubernetes가 확인하는 방법
        httpGet:
          path: /health
          port: 3000
        initialDelaySeconds: 30
        periodSeconds: 10
      readinessProbe:      # 누락: 트래픽을 언제 보낼지 결정하는 기준
        httpGet:
          path: /ready
          port: 3000
        initialDelaySeconds: 5
        periodSeconds: 5
```

**Secret 관리는 별도 결정이 필요합니다.** Compose `environment`의 자격 증명은 Kubernetes Secret으로 변환되지만, Secret 관리 전략(External Secrets Operator, Sealed Secrets, Vault 연동 등)은 인프라마다 다르며 Kanvas가 처리하지 않습니다.

**NetworkPolicy, RBAC, Pod 보안 설정**은 생성 결과물에 포함되지 않습니다. 테스트 환경 이상을 목표로 한다면 반드시 추가해야 합니다.

## 에코시스템 성숙도

Helm의 가장 큰 강점은 에코시스템입니다. Artifact Hub에는 Database, Message Queue, Observability Stack 등 수천 개의 커뮤니티 차트가 있습니다. PostgreSQL을 Helm으로 배포하는 것은 검증된 차트로 `helm install`을 실행하는 일입니다. Kanvas는 자신의 Compose 정의를 변환하는 데 특화되어 있으며, 커뮤니티 인프라 템플릿은 아직 갖추어지지 않았습니다.

## 정리

Docker Kanvas는 변환을 자동화하고 결과를 시각적으로 보여줌으로써, Compose에서 Kubernetes로 이동하는 마찰을 줄여줍니다. 기존 Compose 정의를 가진 팀이 Helm 차트를 처음부터 작성하지 않고 Kubernetes를 시작할 수 있는 실용적인 도구입니다.

베이스라인 생성에 활용하고, 이후 프로덕션 강화 작업에 시간을 투자하세요. Resource Request, Health Probe, Secret 관리, NetworkPolicy 설정은 여전히 개발자의 몫이지만, 빈 파일이 아닌 동작하는 기반에서 시작할 수 있습니다.
