이번 달 인프라 분야에서 주목할 두 가지 소식이 있습니다. Docker가 Compose와 Kubernetes 사이의 간극을 자동화하는 Kanvas 플랫폼을 발표했고, Kubernetes 1.33이 AI/ML 워크로드를 위한 GPU 리소스 관리를 대폭 강화했습니다.

두 가지 모두 많은 인프라 팀이 실제로 겪는 문제를 해결합니다.

## Docker Kanvas: Compose → Kubernetes 자동 변환

Kanvas가 해결하려는 문제는 익숙합니다. 로컬에서 `docker compose up`으로 잘 동작하던 서비스가 Kubernetes에 배포하면 설정 차이로 문제가 생기는 상황입니다. Compose는 단일 호스트 오케스트레이션, Kubernetes는 멀티 노드 분산 시스템으로 설계 철학이 다르기 때문에 변환 작업은 항상 수동으로, 오류가 발생하기 쉬운 방식으로 이루어졌습니다.

기존의 `kompose` 같은 도구는 출력 품질이 낮아 상당한 수동 정리가 필요했습니다. Kanvas는 더 명확한 기준으로 프로덕션에 가까운 Kubernetes 아티팩트를 생성하고, 시각적 인터페이스로 설정을 관리할 수 있게 합니다.

```yaml
# 시작점: docker-compose.yml
services:
  api:
    build: ./api
    ports:
      - "3000:3000"
    environment:
      - DATABASE_URL=postgres://postgres:password@db:5432/myapp
      - NODE_ENV=production
    depends_on:
      db:
        condition: service_healthy
    healthcheck:
      test: ["CMD", "curl", "-f", "http://localhost:3000/health"]
      interval: 30s
      timeout: 10s
      retries: 3

  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_PASSWORD: password
      POSTGRES_DB: myapp
    volumes:
      - pgdata:/var/lib/postgresql/data
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres"]
      interval: 10s
      retries: 5

volumes:
  pgdata:
```

Kanvas는 이것을 파싱해 Deployment, Service, ConfigMap, Secret, PersistentVolumeClaim을 생성합니다.

```yaml
# Kanvas 출력: Deployment (간략 버전)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api
spec:
  replicas: 2
  selector:
    matchLabels:
      app: api
  template:
    spec:
      containers:
        - name: api
          image: myregistry/api:latest
          ports:
            - containerPort: 3000
          env:
            - name: DATABASE_URL
              valueFrom:
                secretKeyRef:
                  name: app-secrets
                  key: database-url
          livenessProbe:
            httpGet:
              path: /health
              port: 3000
            initialDelaySeconds: 30
            periodSeconds: 30
          resources:
            requests:
              memory: "256Mi"
              cpu: "250m"
            limits:
              memory: "512Mi"
              cpu: "500m"
```

생성된 결과물은 시작점입니다. Security Context(비루트 사용자), Network Policy, Pod Disruption Budget, 적절한 Secret 관리(Sealed Secrets 또는 External Secrets Operator)는 여전히 수동으로 구성해야 합니다. Kanvas는 보일러플레이트를 크게 줄이지만 Kubernetes 전문 지식을 대체하지는 않습니다.

## Kubernetes 1.33: AI 워크로드를 위한 GPU 공유

AI/ML 팀에게 더 직접적인 영향을 주는 변화는 Kubernetes 1.33의 GPU 리소스 관리 개선입니다. 핵심은 GPU 공유 지원으로, 여러 워크로드가 하나의 GPU를 분할해서 사용할 수 있게 됩니다.

기존 모델에서는 GPU 용량의 10%만 사용하는 작은 추론 서비스도 GPU 전체를 독점했습니다. 수십 개의 추론 엔드포인트를 운영하는 팀에게 이는 상당한 낭비였습니다.

```yaml
# Kubernetes 1.33: GPU 타임슬라이싱 설정
# NVIDIA Device Plugin ConfigMap으로 적용
apiVersion: v1
kind: ConfigMap
metadata:
  name: nvidia-device-plugin-config
  namespace: kube-system
data:
  config.yaml: |
    version: v1
    sharing:
      timeSlicing:
        resources:
          - name: nvidia.com/gpu
            replicas: 4  # GPU 1개를 가상 GPU 4개로 노출
```

```yaml
# GPU 1/4을 요청하는 추론 Pod
apiVersion: v1
kind: Pod
metadata:
  name: text-embedder
spec:
  containers:
    - name: embedder
      image: myapp/embeddings:latest
      resources:
        limits:
          nvidia.com/gpu: 1  # 가상 GPU 1개 = 물리 GPU 1/4
        requests:
          memory: "4Gi"
          cpu: "2"
```

보고된 60% 비용 절감은 현재 많은 소규모 추론 워크로드를 전용 GPU에서 실행하는 경우를 가정합니다. 실제 절감 효과는 워크로드 프로필에 따라 다르며, GPU 수요가 크게 변동하는 경우 타임슬라이싱 효과가 줄어듭니다.

## 표준 Metric을 활용한 GPU 관측성

Kubernetes 1.33은 DCGM Exporter를 통해 GPU 메트릭을 표준 Prometheus 형식으로 제공합니다.

```yaml
# GPU 메트릭 수집을 위한 ServiceMonitor
apiVersion: monitoring.coreos.com/v1
kind: ServiceMonitor
metadata:
  name: dcgm-exporter
  namespace: monitoring
spec:
  selector:
    matchLabels:
      app: dcgm-exporter
  endpoints:
    - port: metrics
      interval: 15s
```

이를 활용하면 GPU 활용률이 낮은 Pod를 찾아 통합이나 스팟 인스턴스 이전을 결정할 수 있습니다.

```promql
# GPU 활용률 20% 미만인 추론 Pod 찾기
# (통합 또는 스팟 인스턴스 이전 후보)
avg by (pod, namespace) (
  DCGM_FI_DEV_GPU_UTIL{namespace=~"ml-.*"}
) < 20

# GPU 메모리 압박 경보
avg by (node) (
  DCGM_FI_DEV_FB_USED / DCGM_FI_DEV_FB_TOTAL * 100
) > 85
```

## Kanvas와 Helm vs. Kustomize

Docker Kanvas는 Raw Kubernetes Manifest를 생성합니다. 이후 어떻게 관리할지는 별도의 결정입니다.

**Kanvas + Kustomize**는 템플릿 언어 없이 환경별 오버레이를 원하는 팀에 적합합니다. 생성된 Manifest가 Base 레이어가 되고, 오버레이가 Staging과 Production 차이를 처리합니다.

```
k8s/
├── base/           # Kanvas 출력
│   ├── deployment.yaml
│   ├── service.yaml
│   └── kustomization.yaml
└── overlays/
    ├── staging/
    │   └── kustomization.yaml  # Replica, Image Tag 패치
    └── production/
        └── kustomization.yaml  # Resource, HPA 추가
```

**Kanvas + Helm**은 여러 프로젝트에서 인프라를 재사용하거나 Chart로 배포하는 경우에 적합합니다. 변환 작업이 추가되지만 규모가 커질수록 투자 가치가 있습니다.

대부분의 내부 애플리케이션에서는 Kustomize로 시작하는 것이 현실적입니다.

## 도입 판단 가이드

**Docker Kanvas 도입을 검토할 때:**
- 로컬에서 Docker Compose를 사용하고 Kubernetes로 프로덕션 배포 시 설정 차이로 인한 장애가 반복되는 팀
- Kubernetes YAML이 특정 인원에게만 집중되어 있어 팀 전체가 관리할 수 없는 상황

**Kubernetes 1.33 GPU 공유를 활용할 때:**
- 소규모~중간 규모의 추론 서비스를 여러 개 운영하며 GPU 비용이 가시적인 예산 항목인 팀
- 워크로드 GPU 사용 패턴이 안정적이고 예측 가능한 경우 (변동이 큰 배치 작업에는 타임슬라이싱 효율이 낮음)

**도입을 보류할 때:**
- Kanvas: Helm/Kustomize로 이미 잘 관리되는 Kubernetes 설정이 있다면 레이어 추가가 도움이 되지 않습니다
- GPU 공유: 워크로드가 GPU를 거의 포화 상태로 사용한다면 공유 오버헤드가 이득보다 클 수 있습니다

두 도구 모두 실제 인프라 문제를 해결합니다. 도입 가치는 그 특정 문제가 현재 팀의 주요 Pain Point와 일치하는지에 달려 있습니다.
