Docker의 Kanvas 플랫폼은 클라우드 개발에서 오랜 마찰 지점이었던 문제를 다룹니다. 바로 로컬 개발용 Docker Compose와 프로덕션용 Kubernetes 사이의 개념적 간극입니다. Kanvas가 실제로 무엇을 하고, 어떤 상황에 적합한지 살펴봅니다.

## 문제의 본질

대부분의 개발팀은 로컬 환경에서 Docker Compose를 사용합니다. 형식이 직관적이고, 작은 YAML 파일 하나로 멀티 컨테이너 환경을 시작할 수 있으며, `.env` 파일에서 환경 변수가 깔끔하게 매핑됩니다.

프로덕션은 다른 이야기입니다. Kubernetes는 Deployment, Service, Ingress, ConfigMap, PersistentVolumeClaim 등을 별도로 정의해야 합니다. 이 두 세계의 간극은 개발자에게 인지 부담을 주고, 로컬에서는 나타나지 않는 배포 실패를 만들어냅니다.

## Kanvas가 하는 일

Kanvas는 `docker-compose.yml`과 Kubernetes API 사이에 위치합니다. 개발자는 익숙한 Compose 파일을 작성하고 유지하며, Kanvas가 이를 Kubernetes Manifest로 변환하고 배포를 관리합니다.

```yaml
# 기존 docker-compose.yml
services:
  api:
    image: myapp-api:latest
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: ${DATABASE_URL}
      REDIS_URL: ${REDIS_URL}
    depends_on:
      - db
      - cache

  db:
    image: postgres:16
    volumes:
      - pg_data:/var/lib/postgresql/data

  cache:
    image: redis:7-alpine

volumes:
  pg_data:
```

Kanvas는 이 파일을 읽고, 대응하는 Kubernetes 리소스를 생성하여 클러스터에 배포합니다. 시작하는 데 Helm Chart 문법이나 Kustomize Overlay가 필요하지 않습니다.

## Helm, Kustomize와의 관계

Kanvas는 Helm이나 Kustomize의 대체제가 아니라 대안적인 진입점입니다.

**Helm**은 패키징과 릴리스 관리에 강점이 있지만 학습 곡선이 가파릅니다. Chart 템플릿, Values 파일, 릴리스 라이프사이클 관리는 올바르게 구성하기가 쉽지 않습니다.

**Kustomize**는 `kubectl`에 내장되어 있어 환경별 오버레이를 깔끔하게 처리하지만, Docker Compose가 아닌 Kubernetes Manifest에서 시작합니다.

**Kanvas**는 대부분의 개발자가 이미 있는 곳에서 시작합니다. 프로젝트가 성숙하여 Kubernetes 네이티브 기능이 필요해지면, Compose 기반 토대 위에 Kubernetes 전용 설정을 덧붙이거나, 패키징 사례가 필요할 때 Helm으로 마이그레이션할 수 있습니다.

## 현재의 한계

Kanvas는 비교적 새로운 제품으로, 중요한 워크로드에 도입하기 전에 파악해야 할 실제 한계가 있습니다.

**Stateful 서비스 처리.** Compose의 `volumes`는 Kubernetes PersistentVolumeClaim에 매핑되지만, Storage Class 선택과 Access Mode 설정은 프로덕션 데이터베이스 워크로드에서 수동 조정이 필요한 경우가 많습니다. 자동 변환이 데이터베이스에 대한 프로덕션 준비 완료 Persistence 설정을 만들어낸다고 가정하지 마세요.

**멀티 클러스터 배포.** 현재 Tooling은 단일 클러스터 배포에 초점을 맞춥니다. 멀티 리전이나 연합 클러스터 시나리오는 별도 Tooling이 필요합니다.

**CI/CD 통합 성숙도.** GitHub Actions 및 ArgoCD 통합 패턴이 존재하지만, 커뮤니티 생태계는 Helm만큼 깊지 않습니다. 기존 파이프라인에 Kanvas를 연결할 때 더 많은 수동 설정이 필요합니다.

**복잡한 네트워킹.** Compose의 `ports`와 `networks`는 단순한 경우에는 Kubernetes Service와 NetworkPolicy에 합리적으로 매핑되지만, 고급 트래픽 관리(가중 라우팅, Circuit Breaking, mTLS)는 여전히 Kanvas가 추상화하지 않는 Service Mesh 설정이 필요합니다.

## 적합한 사용 사례

Kanvas가 잘 맞는 경우:

- 자체 배포를 직접 관리해야 하지만 Kubernetes 전문 지식이 깊지 않은 애플리케이션 팀
- 전담 인프라 엔지니어가 없는 소규모 팀이나 스타트업
- 상당한 운영 투자 없이 클러스터에서 실행해야 하는 내부 Tooling 및 비핵심 서비스

대규모 마이크로서비스나 규제가 엄격한 프로덕션 환경의 경우, Kanvas로 초기 Compose-to-Kubernetes 변환을 하고 그 위에 Kubernetes 네이티브 설정을 덧붙이는 하이브리드 접근 방식이 Kanvas를 완전한 솔루션으로 취급하는 것보다 현실적입니다.

## Platform Engineering과의 연결

2026년 들어 Platform Engineering이 상당한 모멘텀을 얻고 있습니다. 인프라 팀이 Internal Developer Platform(IDP)을 구축하여 애플리케이션 개발자가 Kubernetes를 직접 다루지 않고도 개발할 수 있는 환경을 만드는 접근 방식입니다. Kanvas는 이 패턴과 자연스럽게 맞아떨어집니다. Docker Compose를 개발자 향 인터페이스로, Kanvas를 그 아래의 변환 및 배포 레이어로 삼는 구조입니다.

처음부터 사용자 정의 추상화 레이어를 구축하지 않고 Platform Engineering 실천을 시작하려는 소규모 조직에게는, Kanvas가 사전에 상당한 인프라 투자 없이도 개발자 인터페이스의 기반이 될 수 있습니다.

## 정리

Docker Kanvas는 Kubernetes의 복잡성을 없애주지는 않습니다. 일부를 유예하고 진입점을 낮출 뿐입니다. 이미 Docker Compose를 로컬에서 실행하고 Kubernetes에 배포할 때 마찰을 겪고 있다면, 먼저 비핵심 서비스에서 Kanvas를 평가해볼 가치가 있습니다. 변환 결과물이 어떻게 생겼는지 파악하고, 기대치에 맞는지 검증한 다음, 프로덕션 워크로드에 적용하기 전에 얼마나 멀리 데려가줄지 충분히 이해하고 결정을 내리세요.
