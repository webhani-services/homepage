## Docker Kanvas: Compose를 Kubernetes로 자동 변환

2026년 1월, Docker가 Kanvas를 출시했습니다. Docker Compose 파일을 Kubernetes Manifest로 자동 변환하는 도구로, Helm과 Kustomize가 차지해온 영역에 직접 도전합니다.

Kanvas가 해결하려는 핵심 문제: 대부분의 개발자는 이미 로컬 개발용 Docker Compose 설정을 보유하고 있습니다. 이를 Kubernetes로 전환하려면 지금까지는 더 가파른 학습 곡선을 가진 다른 포맷으로 설정을 처음부터 다시 작성해야 했습니다. Kanvas는 이 변환 단계를 제거합니다.

### 변환 방식

```yaml
# Input: docker-compose.yml
services:
  app:
    image: myapp:latest
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: postgresql://db:5432/mydb
  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data
```

이 파일을 Kanvas에 전달하면 대응하는 Kubernetes Deployment, Service, PersistentVolumeClaim Manifest가 생성됩니다. Compose를 아는 개발자라면 처음부터 YAML을 작성하지 않고도 Kubernetes에 배포할 수 있습니다.

## Helm, Kustomize와의 비교

| | Helm | Kustomize | Docker Kanvas |
|---|---|---|---|
| 주요 용도 | 패키지 배포 | 환경별 설정 관리 | Compose→K8s 변환 |
| 학습 비용 | 높음 | 중간 | 낮음 (Compose 지식 활용) |
| 템플릿 방식 | Helm Chart | Patch Overlay | 자동 생성 |
| 대상 사용자 | DevOps 엔지니어 | DevOps 엔지니어 | 전체 개발자 |

Kanvas가 Helm을 완전히 대체하는 것은 아닙니다. Helm의 Chart 에코시스템과 Release 관리 기능은 복잡한 배포에서 여전히 유효합니다. Kanvas는 K8s 내부를 깊이 파고들 시간 없이 Compose 기반 애플리케이션을 Kubernetes에 올려야 하는 개발자를 위한 **진입 도구**로 보는 게 적절합니다.

## AWS의 AI 기반 Kubernetes 운영

Kanvas와 함께, AWS도 두 가지 Preview 제품을 통해 AI 네이티브 Kubernetes 운영을 추진하고 있습니다.

### EKS MCP Server

EKS MCP Server는 Claude Code 같은 AI Coding Assistant가 자연어로 Kubernetes 클러스터를 조작할 수 있는 Interface를 제공합니다:

```bash
# Claude Code를 통한 EKS 클러스터 조작 예시
claude "production EKS 클러스터의 web-app Deployment를
image tag v2.1.0으로 Rolling Update해주세요.
업데이트 후 현재 Pod 상태도 확인해주세요."
```

자연어 지시를 기반으로 Claude가 필요한 kubectl 명령을 실행합니다. kubectl 문법을 외우지 않아도 원하는 작업을 수행할 수 있습니다.

### AWS DevOps Agent

AWS DevOps Agent는 Kubernetes 클러스터의 문제 진단, Resource Request 최적화, 보안 정책 확인 등을 자연어 쿼리로 처리하는 Agent입니다:

- Pod 크래시 원인 특정 및 수정 제안
- Resource Request 최적화 권고
- 보안 정책 확인

두 제품 모두 아직 Preview 단계로, Production 클러스터 운영에는 신중한 평가가 필요합니다.

## Kubernetes 채택률 82%가 의미하는 것

CNCF 2025 Annual Cloud Native Survey에 따르면, Container 사용자의 82%가 Kubernetes를 Production 환경에서 운영하며, 그 중 66%가 GenAI 워크로드에도 활용하고 있습니다.

K8s는 더 이상 특화된 인프라 선택지가 아닌 **클라우드 네이티브 개발의 표준 플랫폼**이 됐습니다. 이 기반 위에서 Kanvas나 AI 기반 운영 도구를 평가해야 합니다.

## 실무에서 기대할 수 있는 변화

**Docker Kanvas**는 Compose 기반 애플리케이션을 Kubernetes로 올리는 마찰을 줄여줍니다. 단순한 아키텍처와 기존 Compose 설정이 있는 팀에게는 실질적인 시간 절감이 됩니다. 복잡한 네트워킹, RBAC, 다환경 Overlay가 필요한 경우에는 Helm이나 Kustomize가 여전히 필요합니다.

**AI 기반 K8s 운영** (EKS MCP Server, DevOps Agent)은 아직 Production Ready가 아니지만, 방향성은 명확합니다. 자연어 Interface가 클러스터 운영의 진입 장벽을 낮추면, DevOps 리소스가 부족한 팀이 가장 큰 혜택을 받습니다.

**Multi-cloud 운영**의 중요성도 계속 높아지고 있습니다. Kubernetes 스킬은 AWS, Azure, GCP 전반에 걸쳐 활용되므로 플랫폼 지식에 대한 투자는 여러 환경에서 복리로 돌아옵니다.

## 마무리

Docker Kanvas는 로컬 Compose 개발과 Kubernetes Production 사이를 잇는 실용적인 도구입니다. Helm을 대체하는 것이 아니라 명확한 공백을 채웁니다.

AWS의 AI 기반 운영 제품은 아직 Preview 단계지만 방향성은 분명합니다. Kubernetes 운영의 복잡성이 AI 도구로 해소될 때, 아직 K8s를 도입하지 않은 18%의 팀들의 채택 장벽도 낮아질 것입니다.
