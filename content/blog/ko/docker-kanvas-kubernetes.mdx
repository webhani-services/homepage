Docker가 새로운 플랫폼 'Kanvas'를 출시했습니다. 로컬 개발 환경(Docker Compose)과 Production 환경(Kubernetes) 사이의 간극을 자동화로 해소하는 것이 핵심 목표입니다. `docker-compose.yml`을 Kubernetes Manifest로 자동 변환하는 기능을 Docker Desktop 워크플로에 통합해, Helm과 Kustomize가 주도하던 Kubernetes 배포 관리 영역에 Docker가 본격 진입한 것입니다.

## Kanvas가 해결하려는 문제

Docker Compose로 로컬 개발을 하고 Kubernetes로 Production을 운영하는 팀이라면 공통적으로 겪는 마찰이 있습니다. 서비스가 추가되거나 설정이 변경될 때마다 Compose 정의와 Kubernetes Manifest를 별도로 유지해야 하는 것입니다.

기존에는 오픈소스 도구 `kompose`로 변환하는 방법이 일반적이었지만, Compose 변경마다 수동 변환과 조정이 필요했습니다. Kanvas는 이 과정을 Docker Desktop에 통합해 지속적이고 자동화된 변환을 목표로 합니다.

## 변환 예시

아래와 같은 Compose 파일이 있다면:

```yaml
# docker-compose.yml
services:
  api:
    image: my-api:latest
    ports:
      - "8080:8080"
    environment:
      DATABASE_URL: postgresql://db:5432/app
    depends_on:
      - db
  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: ${DB_PASSWORD}

volumes:
  db-data:
```

Kanvas는 이를 분석해 Deployment, Service, PersistentVolumeClaim, ConfigMap 등의 Kubernetes 리소스를 생성합니다. 생성된 Manifest는 Compose가 표현하는 내용을 정확하게 변환한 최소한의 구성입니다.

기본적으로 포함되지 않는 항목들: Resource Limit, Liveness/Readiness Probe, HorizontalPodAutoscaler, 환경별 설정 오버라이드. 이 부분은 직접 추가해야 합니다.

## Helm / Kustomize 비교

| | Kanvas | Helm | Kustomize |
|---|---|---|---|
| 입력 형식 | docker-compose.yml | Chart Template | Base + Overlay |
| 학습 곡선 | 낮음 (Compose 지식 활용) | 보통 (Helm 고유 개념) | 낮음~보통 |
| 커스터마이징 | 제한적 (현재) | 매우 높음 | 높음 |
| 생태계 | Docker Desktop 통합 | 대형 Chart Repository | kubectl 내장 |
| Production 성숙도 | 초기 단계 | 매우 성숙 | 성숙 |

Kanvas는 Compose 중심으로 개발해온 팀의 Kubernetes 전환 장벽을 낮춥니다. 반면 복잡한 배포 요구사항에 대한 Helm의 유연성과 풍부한 생태계, Kustomize의 Overlay 방식에는 현재 미치지 못합니다.

## Kanvas가 적합한 상황

**Kanvas를 고려할 만한 상황:**
- 소규모 프로젝트 또는 PoC(개념 검증)
- Compose에서 Kubernetes로 처음 마이그레이션하는 팀
- Docker Desktop을 개발 환경 중심으로 사용하는 팀

**기존 도구를 유지하는 것이 나은 상황:**
- 다중 환경(staging/production/preview) 배포가 복잡한 경우
- Helm Chart 생태계를 적극 활용 중인 경우
- Helm 또는 Kustomize로 이미 확립된 워크플로를 보유한 팀

## 현재 부족한 부분

Kanvas가 Production에서 Helm, Kustomize와 진지하게 경쟁하려면 다음이 필요합니다.

1. **Secret 관리**: Kubernetes Secret, Sealed Secrets, External Secrets Operator 통합
2. **다중 환경 설정**: Base Compose를 복제하지 않고 환경별 오버라이드 지원
3. **GitOps 통합**: ArgoCD나 Flux가 Kanvas 생성 Manifest를 깔끔하게 소비할 수 있는 패턴

이것들이 실험을 막는 장애물은 아니지만, 대규모 Production 도입의 필수 조건입니다.

## 실용적인 결론

Docker Compose로 개발하면서 아직 Kubernetes로 마이그레이션하지 않은 팀이라면, Kanvas는 전환 장벽을 낮추는 실험 가치가 있습니다. 이미 Helm이나 Kustomize를 Production에서 운영 중이라면, 지금 당장 전환할 이유는 없습니다.

Docker가 보유한 개발자 커뮤니티의 영향력과 Desktop 배포 채널을 고려하면, Kanvas가 많은 개발자를 Kubernetes로 이끄는 진입로 역할을 할 가능성은 충분합니다. 성숙해가는 과정을 지켜볼 만한 도구입니다.
