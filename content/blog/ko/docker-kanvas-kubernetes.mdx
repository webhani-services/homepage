Docker가 새로운 Platform인 Kanvas를 출시했습니다. 로컬 Docker Compose 개발 환경과 Kubernetes Production 환경 사이의 간극을 줄이는 것이 목표입니다. 핵심 기능은 Compose 파일을 Kubernetes Artifact로 자동 변환하는 것으로, 수년간 Kubernetes Configuration 관리를 지배해온 Helm과 Kustomize에 도전장을 내밀고 있습니다.

## Kanvas가 해결하려는 문제

`docker compose up`으로 로컬에서 잘 동작하는 Application을 Kubernetes에 배포하려면 상당한 작업이 따릅니다. Deployment, Service, PersistentVolumeClaim, ConfigMap 등 다양한 Kubernetes YAML을 작성하고 유지해야 합니다. 전담 Platform Engineer가 없는 팀에게는 이 비용이 Cloud 전환을 지연시키거나 불안정한 Configuration으로 이어집니다.

기존 선택지의 한계:

- **Kompose**: Compose를 Kubernetes YAML로 변환하는 OSS 도구. 간단하지만 출력 품질이 낮아 Production 사용 전 수동 조정이 필요합니다.
- **Helm**: 강력한 Template 기능을 가진 Kubernetes Package Manager. 독자적인 DSL과 높은 학습 비용이 진입 장벽입니다.
- **Kustomize**: `kubectl`에 내장된 Overlay 방식. 소규모에는 깔끔하지만 규모가 커지면 복잡해집니다.

Kanvas는 다른 접근법을 취합니다. Compose를 Source of Truth로 유지하면서 그로부터 배포 가능한 Kubernetes Configuration으로 가는 자동화된 경로를 제공합니다.

## 동작 방식

일반적인 Compose 파일을 예로 들면:

```yaml
# docker-compose.yml
services:
  web:
    image: my-app:latest
    ports:
      - "3000:3000"
    environment:
      DATABASE_URL: postgres://db:5432/myapp
    depends_on:
      - db

  db:
    image: postgres:16
    volumes:
      - db-data:/var/lib/postgresql/data

volumes:
  db-data:
```

Kanvas는 다음 Kubernetes Resource를 자동 생성합니다:

- 각 Service에 대한 `Deployment`
- Port 노출 여부에 따른 `Service` (ClusterIP 또는 LoadBalancer)
- Named Volume에 대한 `PersistentVolumeClaim`
- 환경 변수를 위한 `ConfigMap` 또는 `Secret`
- 해당하는 경우 `Ingress` 설정

변환 기능 외에도 생성된 Manifest를 시각적으로 검토하고 편집할 수 있는 Web UI와, 로컬 설정과 Production 설정의 차이를 관리하는 기능이 포함되어 있습니다. 예를 들어 개발 환경에서만 필요한 Port Forward가 Production에는 포함되지 않도록 구조적으로 관리할 수 있습니다.

## Kanvas vs. Helm vs. Kustomize

| | Kanvas | Helm | Kustomize |
|---|---|---|---|
| 학습 비용 | 낮음 | 높음 | 중간 |
| Compose 호환성 | 우수 | 미흡 | 미흡 |
| Template 유연성 | 중간 | 높음 | 중간 |
| GitOps 통합 | 계획 중 | 성숙 | 성숙 |
| 커뮤니티 성숙도 | 초기 | 성숙 | 성숙 |

Helm의 강점은 Template 능력과 방대한 Public Chart Ecosystem입니다. Prometheus, PostgreSQL, cert-manager 등 공통 인프라 Component에 대해 관리되는 Helm Chart가 존재합니다. Kanvas는 이 영역에서 경쟁하지 않습니다.

Kustomize는 Base Configuration에 환경별 Overlay를 적용하는 계층적 Configuration 관리에 강점이 있으며 `kubectl`에 직접 통합되어 있습니다.

Kanvas가 경쟁하는 지점은 다릅니다. `docker-compose.yml`을 가진 팀이 새로운 Configuration 패러다임 학습 없이 Kubernetes를 시작할 수 있게 하는 것입니다.

## Kanvas가 적합한 경우

**처음으로 Compose에서 Kubernetes로 전환하는 팀**: 기계적인 변환 작업을 자동화하여 팀이 YAML 문법보다 Kubernetes 개념 이해에 집중할 수 있습니다.

**전담 Platform Engineer가 없는 Application 팀**: Application 개발자가 깊은 Kubernetes 전문 지식 없이도 Infrastructure Configuration에 참여할 수 있습니다.

**Compose가 로컬 개발 표준인 프로젝트**: Compose를 정규 Configuration으로 유지하고 Kubernetes Manifest를 파생시켜 Single Source of Truth를 유지합니다. 환경 간 설정 불일치 위험을 줄일 수 있습니다.

## Helm이나 Kustomize를 유지해야 하는 경우

이미 Helm Chart나 Kustomize Overlay를 관리하고 있다면 Kanvas로의 전환 비용이 이점을 초과할 가능성이 높습니다. Helm의 Template 유연성은 멀티 테넌트 구성, 복잡한 Rollout 전략(Canary, Blue-Green), 팀 간 Chart 공유에서 중요합니다.

## 실무적 평가

현재 Kanvas는 카테고리를 대체하는 도구는 아닙니다. GitOps 통합은 계획 중이지만 아직 성숙하지 않았고, 커뮤니티도 Helm과 Kustomize에 비해 초기 단계입니다. 두 도구를 이미 활용하고 있는 팀에게는 전환 동기가 부족합니다.

중요한 점은 Kanvas가 생성하는 Kubernetes YAML이 표준 형식이라는 것입니다. Kanvas를 사용하다가 다른 도구로 전환해도 생성된 파일을 그대로 활용할 수 있습니다. Vendor Lock-in 위험이 낮아 실험적 채택이 용이한 설계입니다.

신규 프로젝트나 Kubernetes를 처음 시작하는 팀에게 Kanvas는 첫 단계로 검토할 가치가 있습니다. 익숙한 Compose 문법에서 시작하여 Kubernetes Primitive를 처음부터 작성하지 않고 점진적으로 학습할 수 있습니다.
