2026년 Docker는 기존 Linux Container에 더해 WebAssembly(Wasm) Module을 네이티브로 실행하는 기능을 지원합니다. 이제 `docker run` 명령으로 WASI 호환 Wasm 바이너리를 직접 실행할 수 있습니다.

이것은 Linux Container의 대체재가 아닙니다. Wasm Container는 밀리초 단위로 시작하고 작은 바이너리로 패키징되며 Linux 커널 없이 동작하는 다른 공간을 차지합니다. 이 트레이드오프가 유효한 상황이 어디인지가 실무적인 질문입니다.

## Docker의 Wasm 실행 원리

Docker는 containerd를 기반으로 하며, Wasm 지원은 `runwasi`라는 shim이 담당합니다. 이를 통해 Docker는 Wasm Module을 Linux Container와 동일한 방식으로 First-class Runtime 대상으로 처리합니다.

```bash
# Docker로 Wasm 이미지 실행
docker run --platform wasi/wasm \
  --runtime=io.containerd.wasmedge.v1 \
  my-org/my-wasm-app:latest
```

현재 여러 Wasm Runtime이 지원됩니다.

| Runtime | 특징 |
|---|---|
| `io.containerd.wasmedge.v1` | 고성능, Production 지향 |
| `io.containerd.wasmtime.v1` | Bytecode Alliance 구현, 안정성 우선 |
| `io.containerd.spin.v2` | Fermyon Spin 기반, HTTP/이벤트 처리 전용 |

## Linux Container vs. Wasm Container

| 특성 | Linux Container | Wasm Container |
|---|---|---|
| Cold start | 100ms ~ 수초 | 1~10ms |
| 이미지 크기 | 수십MB ~ GB | 수백KB ~ 수MB |
| System call | 직접 (Linux 커널 경유) | WASI 경유 (Sandbox) |
| CPU 아키텍처 의존성 | 있음 (amd64/arm64) | 없음 (Architecture 중립) |
| 에코시스템 호환성 | 높음 | 제한적 (Rust/Go/C/C++ 주류) |

Wasm Container의 핵심 강점은 **시작 속도**와 **이식성**입니다. 같은 `.wasm` 바이너리가 x86, ARM 어디서든 재컴파일 없이 동작합니다.

## Rust로 Wasm Container 만들기

WASI 호환 Rust 서비스의 Dockerfile 예시입니다.

```dockerfile
# Build 단계
FROM rust:1.80-slim AS builder

RUN rustup target add wasm32-wasip1

WORKDIR /app
COPY . .
RUN cargo build --target wasm32-wasip1 --release

# Runtime 단계
FROM wasmedge/slim-runtime:0.11.0
COPY --from=builder /app/target/wasm32-wasip1/release/my_app.wasm /app.wasm

ENTRYPOINT ["wasmedge", "/app.wasm"]
```

```rust
// src/main.rs — WASI 호환 Request Handler
use std::io::{self, Read};

fn main() {
    let mut input = String::new();
    io::stdin().read_to_string(&mut input).unwrap();

    let output = process(&input);
    println!("{}", output);
}

fn process(input: &str) -> String {
    serde_json::json!({
        "status": "ok",
        "input_length": input.len()
    })
    .to_string()
}
```

빌드 결과는 단일 `.wasm` 파일이며, 이런 서비스는 보통 2MB 이하입니다.

## Docker Compose에서 Wasm과 Linux Container 혼용

실제 시스템에서는 Wasm Container와 Linux Container를 용도에 맞게 함께 사용하는 패턴이 일반적입니다.

```yaml
# docker-compose.yml
services:
  # Infrastructure용 표준 Linux Container
  postgres:
    image: postgres:17
    environment:
      POSTGRES_PASSWORD: secret

  redis:
    image: redis:8-alpine

  # 경량 처리용 Wasm Container
  request-validator:
    image: my-org/validator:latest
    platform: wasi/wasm
    runtime: io.containerd.wasmedge.v1
    ports:
      - "8080:8080"

  # 메인 API는 Linux Container
  api-gateway:
    build: ./api-gateway
    ports:
      - "3000:3000"
    depends_on:
      - request-validator
      - postgres
```

Validator는 거의 즉시 시작하고 최소한의 메모리를 사용합니다. API Gateway는 Node.js 에코시스템, 파일 I/O, Database 연결이 필요한 모든 것을 처리합니다.

## Kubernetes와 Wasm

Kubernetes 1.35 이상에서는 `RuntimeClass`를 통해 Wasm 워크로드를 명시적으로 타겟팅할 수 있습니다.

```yaml
# RuntimeClass 정의
apiVersion: node.k8s.io/v1
kind: RuntimeClass
metadata:
  name: wasmedge
handler: wasmedge

---
# Deployment에서 사용
apiVersion: apps/v1
kind: Deployment
metadata:
  name: wasm-validator
spec:
  template:
    spec:
      runtimeClassName: wasmedge  # Wasm Runtime 지정
      containers:
        - name: validator
          image: my-org/validator:latest
          ports:
            - containerPort: 8080
```

Wasm 워크로드는 이를 지원하는 Node에만 격리되어 나머지 Cluster에 영향을 주지 않습니다.

## Wasm Container가 적합한 상황

**Wasm이 강점을 발휘하는 곳:**

- **Edge Function**: CDN Edge에서 Cold start 지연이 하드 제약인 환경
- **Plugin System**: 신뢰할 수 없는 서드파티 로직을 Sandbox 환경에서 실행
- **고빈도 단명 프로세스**: Validation, 변환, 라우팅처럼 시작 비용이 중요한 처리
- **Cross-platform 라이브러리 배포**: 단일 `.wasm` 파일로 어떤 언어에서도 호출 가능

**Wasm이 적합하지 않은 곳:**

- Node.js, Python, Ruby 에코시스템에 의존하는 Application
- 복잡한 파일시스템 접근이나 OS 상호작용이 필요한 서비스
- GPU 워크로드 — WASI를 통한 CUDA/ROCm 접근은 아직 미성숙

## webhani의 현시점 평가

Docker의 네이티브 Wasm 지원은 실험 장벽을 크게 낮췄습니다. Stateless하고 CPU 바운드인 프로세스로, 시작 시간과 바이너리 크기가 중요한 상황이라면 Production 적용도 충분히 검토할 수 있는 수준입니다.

새로운 Edge Function이나 Plugin System을 설계할 때는 Linux Container와 함께 Wasm Container를 진지한 선택지로 평가해볼 가치가 있습니다. 일반적인 웹 애플리케이션 Backend라면 Linux Container가 여전히 실용적인 기본값입니다.
