## Cilium이 주류가 된 배경

2026년 현재 Cilium은 Kubernetes 프로덕션 환경에서 가장 널리 채택된 CNI(Container Network Interface)입니다. Google, Microsoft, AWS의 인프라와 수천 개의 기업 클러스터에서 운영되고 있습니다.

몇 년 전만 해도 Flannel과 Calico가 주도하던 분야입니다. 무엇이 바뀌었을까요. 답은 eBPF에 있습니다.

## iptables vs. eBPF: 근본적인 차이

Kubernetes 네트워킹은 오랫동안 iptables에 의존해왔습니다. Pod와 Service 간 트래픽을 정적 규칙 체인으로 라우팅하는 방식입니다.

문제는 알고리즘입니다. Service 수가 늘어날수록 iptables 규칙 수가 선형으로 증가합니다. Service 10,000개면 100,000개 이상의 규칙이 생기고, 모든 패킷이 이를 순차적으로 검사합니다. 지연이 클러스터 규모에 비례해 증가합니다.

```
iptables의 규모 문제:
- O(n) 규칙 순회: Service 수에 비례한 지연 증가
- Kernel/Userspace 간 빈번한 Context 전환
- 규칙 업데이트 시 테이블 전체 리로드 필요
- 제한적인 Observability: 무슨 일이 일어나는지 보기 어려움
```

eBPF는 다른 접근 방식을 취합니다. 검증된 샌드박스 프로그램이 Linux 커널 내부에서 실행되며 패킷이 전체 네트워크 스택에 도달하기 전에 처리합니다. 규칙은 Hash Map에 O(1) 조회로 저장됩니다.

```
eBPF의 특징:
- 커널 네이티브 실행: Userspace 왕복 없음
- O(1) Service 조회: 클러스터 규모에 관계없이 일정한 지연
- 증분 업데이트: 테이블 전체 리로드 불필요
- Hubble을 통한 풍부한 Observability
```

## AI Workload에 eBPF가 필요한 이유

전통적인 Microservice와 AI 추론 서비스는 네트워크 패턴이 근본적으로 다릅니다.

**East-West 트래픽 급증**: 추론 서비스는 GPU 서버 RPC, 모델 캐시 접근, 메시지 큐 연결을 고빈도로 수행합니다. 클러스터 내부(East-West) 트래픽이 외부 수신(North-South)을 크게 초과합니다. iptables의 선형 조회 비용이 일반 웹 서비스에서는 문제가 되지 않더라도 이 트래픽 규모에서는 증폭됩니다.

**Topology 인식**: 멀티 GPU 훈련 작업은 NVLink로 연결된 GPU를 같은 Node나 Rack에 배치해야 합니다. Cilium의 Topology 인식 라우팅은 이 제약을 네트워크 레이어에서 표현하고 강제할 수 있습니다.

**세밀한 마이크로세그멘테이션**: GPU Node 접근을 특정 Workload로만 제한하는 것은 보안 요구사항입니다. eBPF는 무시할 수 있는 Overhead로 이 정책을 강제합니다.

## Cilium 기본 설치

```yaml
# Helm으로 설치 — kube-proxy를 완전히 Cilium으로 교체
helm repo add cilium https://helm.cilium.io/
helm install cilium cilium/cilium --version 1.16.0 \
  --namespace kube-system \
  --set kubeProxyReplacement=true \
  --set hubble.relay.enabled=true \
  --set hubble.ui.enabled=true
```

`kubeProxyReplacement=true`를 설정하면 kube-proxy가 완전히 제거됩니다. Cilium이 eBPF Hash Map 조회로 Service 라우팅을 처리합니다.

## Hubble: 기본 내장 Observability

Cilium은 Hubble이라는 플로우 Observability 도구를 포함합니다. eBPF를 사용해 네트워크 홉이나 Sidecar Proxy 없이 커널 레벨에서 네트워크 활동을 캡처합니다.

```bash
# Namespace의 실시간 플로우 모니터링
hubble observe --namespace production --follow

# 특정 Service로의 드롭된 패킷 찾기
hubble observe \
  --to-pod production/inference-server \
  --verdict DROPPED \
  --last 100

# Service 간 요청률 측정
hubble observe \
  --from-pod production/api-gateway \
  --to-pod production/inference-server \
  --protocol http
```

Hubble Metric은 표준 통합으로 Prometheus와 Grafana에 전달됩니다. 결과: 어떤 Pod가 누구와 얼마나 자주 통신하는지, 어디서 실패가 발생하는지를 보여주는 네트워크 Topology 맵 — 애플리케이션 레벨 계측 없이.

## Network Policy 작성

Cilium은 표준 Kubernetes `NetworkPolicy`와 표준 API로 표현할 수 없는 경우를 위한 `CiliumNetworkPolicy`를 모두 지원합니다.

```yaml
# GPU Node 접근을 인가된 추론 Workload로만 제한
apiVersion: cilium.io/v2
kind: CiliumNetworkPolicy
metadata:
  name: gpu-node-access
  namespace: production
spec:
  endpointSelector:
    matchLabels:
      app: inference-server
  egress:
  - toEndpoints:
    - matchLabels:
        kubernetes.io/hostname: gpu-node-01
    toPorts:
    - ports:
      - port: "50051"  # gRPC 추론 Endpoint
        protocol: TCP
```

`CiliumNetworkPolicy`는 L7 Policy도 지원합니다. 특정 HTTP Method나 Path만 허용하는 것처럼 IP/포트 조합이 아닌 세밀한 제어가 가능합니다. 전체 Service Mesh 없이도 어떤 서비스가 어떤 API Endpoint를 호출할 수 있는지 제한할 수 있습니다.

## VMware Migration에서의 Cilium 역할

2026년 많은 기업이 VMware 환경에서 Kubernetes로 마이그레이션하고 있습니다. 네트워킹은 이 마이그레이션에서 가장 많은 마찰이 발생하는 부분 중 하나입니다. 기존 IP 주소 공간, 정책 모델, 운영 방식이 Kubernetes 기본값으로 깔끔하게 매핑되지 않습니다.

VMware Kubernetes Service(VKS)는 버전 3.6부터 Cilium을 CNI로 공식 지원합니다. 이로써 레거시 VMware 환경과 Kubernetes 클러스터 양쪽에서 동일한 CNI가 트래픽을 처리하여 전환 기간을 단순화합니다.

```
VMware Migration에서 Cilium의 가치:
- BGP 라우팅으로 기존 IP 주소 공간 보존
- VMware NSX-T와 Kubernetes 전체에 일관된 정책 모델
- 점진적 마이그레이션: 하이브리드 환경에서 통합 CNI
- 운영 연속성: 전환 전후에 동일한 Tooling
```

## Tetragon: eBPF 기반 Runtime 보안

Cilium 프로젝트에는 Tetragon이라는 Runtime 보안 도구도 포함됩니다. eBPF를 사용해 컨테이너 수정 없이 커널 레벨에서 프로세스 실행, 파일 접근, 네트워크 연결을 모니터링합니다.

```yaml
# Python 프로세스의 네트워크 연결 추적
apiVersion: cilium.io/v1alpha1
kind: TracingPolicy
metadata:
  name: python-network-audit
spec:
  kprobes:
  - call: "tcp_connect"
    syscall: false
    args:
    - index: 0
      type: "sock"
    selectors:
    - matchBinaries:
      - operator: In
        values:
        - "/usr/bin/python3"
```

이 정책은 Python 프로세스가 TCP 연결을 열 때마다 감사 이벤트를 생성합니다. 추론 서버가 예상치 못한 아웃바운드 연결을 시작하면 — Supply Chain 침해의 일반적인 지표 — Tetragon이 즉시 감지합니다.

## Webhani의 시각

Cilium과 eBPF 채택은 더 나은 iptables를 얻는 것보다 네트워킹 레이어가 할 수 있는 것을 바꾸는 것입니다. Observable, Programmable, AI Workload Ready — 이 세 가지 속성이 클러스터가 규모와 복잡성에서 성장할수록 더 중요해집니다.

iptables와 eBPF 사이의 성능 차이는 측정 가능하지만 주된 주장은 아닙니다. 주된 주장은 운영적입니다. Hubble의 Observability가 팀이 네트워크 문제를 디버깅하는 방식을 바꾸고, `CiliumNetworkPolicy`의 표현력이 팀이 보안 자세를 구현하는 방식을 바꿉니다.

Kubernetes에서 AI 서비스를 실행하거나 VMware Migration을 관리하는 팀에게 Cilium은 미래 투자가 아닌 현재의 프로덕션 표준으로 진지하게 검토할 가치가 있습니다.
