프로덕션 Kubernetes 클러스터 운영은 끊임없는 감시를 요구합니다. 클러스터 상태를 모니터링하고, 문제를 조기에 탐지하고, 이슈가 확산되기 전에 대응해야 합니다. AWS 는 2026 년 7 월 두 가지 기능을 제공하기 시작했습니다. AI 에이전트를 통한 자동 진단과 수정(self-healing), 그리고 EKS 클러스터의 Kubernetes Version 을 7 일 이내에 rollback 할 수 있는 기능입니다. 두 기능 모두 운영 부담을 줄이지만, 자동화와 인간의 판단 사이의 경계를 어디에 둘지에 대한 신중한 고민이 필수입니다. webhani 의 관점에서 이 두 기능의 동작 원리, 승인 게이트 설계 전략, 그리고 프로덕션 운영 안전을 위한 Governance 방안을 정리합니다.

## Amazon EKS 의 AI 에이전트: 자동 진단과 수정

전통적으로 Pod 이 CrashLoopBackOff 상태에 빠지거나 Worker Node 의 메모리가 부족하면, 운영팀은 alert 를 받고 로그를 조사해서 수동으로 문제를 해결합니다. 새로운 AI 수정 에이전트는 이 패턴을 바꿉니다. 문제를 자동으로 탐지하고, 해결 방안을 제시하며, 설정에 따라 인간의 개입 없이 자동으로 문제를 해결할 수 있습니다.

에이전트는 네 단계로 동작합니다.

1. **진단 단계**: Cluster 의 event, metrics, resource 정보를 수집하여 증상 파악
2. **추론 단계**: Language Model 을 활용해 근본 원인 분석 (예: ImagePullBackOff 는 registry credential 부재)
3. **제안 단계**: 수정 방안 제시 (예: ImagePullSecret 적용)
4. **실행 단계**: 설정에 따라 승인 대기 또는 자동 수정 실행

EKS 클러스터에서 `enableAIRemediationAgent` flag 를 활성화하면 managed policy 가 IAM role 에 자동 attach 됩니다. 하지만 기본 설정은 권한이 너무 넓으므로, 프로덕션 환경에서는 custom policy 로 제한하는 것이 필수입니다. 다음 policy 는 agent 의 진단 권한은 read-only 로 제한하고, 실제 수정은 별도의 role 을 assume 하도록 설계합니다.

```
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Sid": "DiagnosticReadOnly",
      "Effect": "Allow",
      "Action": [
        "eks:DescribeCluster",
        "eks:ListNodegroups",
        "eks:DescribeNodegroup",
        "ec2:DescribeInstances",
        "cloudwatch:GetMetricStatistics",
        "logs:GetLogEvents"
      ],
      "Resource": [
        "arn:aws:eks:ap-northeast-1:123456789012:cluster/prod-cluster"
      ]
    },
    {
      "Sid": "RemediationViaAssumeRole",
      "Effect": "Allow",
      "Action": [
        "sts:AssumeRole"
      ],
      "Resource": [
        "arn:aws:iam::123456789012:role/eks-ai-remediation-executor"
      ],
      "Condition": {
        "StringEquals": {
          "sts:ExternalId": "prod-remediation-id"
        }
      }
    }
  ]
}
```

이 구조의 장점은 진단 접근(read-only)과 수정 실행(별도 role)을 분리하는 것입니다. ExternalId 는 confused deputy attack 을 방지합니다.

## 세 단계 승인 전략

프로덕션 워크로드를 운영하는 팀들과의 상담 경험에서, 우리는 세 단계 승인 모델을 추천합니다. 이 모델은 운영 의도와 자동화 범위를 일치시킵니다.

**단계 1: 자동 수정 (승인 불필요)**
- 증상: Pod eviction, 임시 네트워크 복구
- 원인: 일시적 리소스 부족, kubelet 이 관리하는 재시작
- 조치: Pod 강제 삭제 (kubelet 이 새 Pod 시작)
- 이유: 영향 범위가 작음, 수 분 내 복구, 비용 영향 없음

**단계 2: 승인 후 실행**
- 증상: 스케일링 결정, Security Group rule 추가
- 원인: 지속적인 리소스 부족, 과도한 network policy
- 조치: HPA min replica 증가, node group scale up
- 이유: 클러스터 전역 영향, 비용 영향 가능

**단계 3: 수동 개입 전용**
- 증상: Control plane 성능 저하, etcd storage 부족
- 원인: 호환성 문제, 인프라 이슈
- 조치: Version rollback (후술), 노드 수동 복구
- 이유: 영향 범위가 크고, 복구가 어려움

단계 2 이상의 승인은 EKS API 로 처리합니다.

```bash
# 승인 대기 중인 수정 제안 목록 확인
aws eks list-ai-remediation-suggestions \
  --cluster-name prod-cluster \
  --region ap-northeast-1 \
  --query 'suggestions[?status==`PENDING_APPROVAL`]'

# 특정 수정 제안 승인
aws eks approve-ai-remediation-suggestion \
  --cluster-name prod-cluster \
  --remediation-id rem-abc123 \
  --region ap-northeast-1
```

승인을 반자동화하려면 Lambda 로 remediation event 를 구독하고, 저위험 카테고리(예: Pod restart)는 자동 승인하도록 설정할 수 있습니다. 우리의 권장사항: 처음에는 모든 것을 수동 승인으로 시작하고, 3 개월간 false positive rate 를 측정한 후, 안정적인 카테고리만 단계적으로 자동화합니다.

## Kubernetes Version Rollback: 업그레이드의 안전장치

EKS 는 이제 Kubernetes Version 업그레이드를 7 일 이내에 되돌릴 수 있습니다. 이전에는 업그레이드가 프로덕션에서 문제를 일으킬 경우 전체 클러스터를 재구성해야 했습니다. Rollback 은 다음과 같은 상황에서 유용합니다.

- 새 version 의 예상하지 못한 호환성 문제 발생
- Workload 가 새 version 의 regression bug 에 hit
- CNI 또는 네트워크 변경으로 인한 연결성 문제

Rollback flow 는 다음과 같습니다.

```bash
# 현재 클러스터 Version 과 업그레이드 상태 확인
aws eks describe-cluster \
  --name prod-cluster \
  --query 'cluster.{Version:version,UpgradeStatus:status}'

# 7 일 이내라면 rollback 시작
aws eks rollback-cluster-version \
  --cluster-name prod-cluster \
  --region ap-northeast-1
```

Rollback 은 control plane 과 data plane 을 순차적으로 되돌리며, 일반적으로 15 ~ 30 분이 소요됩니다. 사용자에게 미리 공지하세요. Workload 는 control plane 재시작 중 일시적인 disruption 을 경험합니다.

7 일 window 는 의도적입니다. Staging 에서 통합 테스트를 실행하고, 문제를 발견하고, window 가 종료되기 전에 rollback 할 충분한 시간을 제공합니다. 7 일 후에 문제를 발견하면 클러스터 재구성으로 돌아갑니다.

## Governance: 자동화와 인간 감시의 균형

핵심 긴장 관계는 이것입니다. AI 에이전트는 빠르게 움직이고, 인간은 신중하게 판단합니다. 프로덕션 안전을 위해서는 이 간격을 메워야 합니다. 우리는 수정 정책 설계 시 세 가지 영역에 집중합니다.

**1. Blast Radius 정의**

일부 수정은 단일 Pod 에만 영향을 미치고, 다른 것들은 클러스터 전역 동작을 재구성합니다. Security Group ingress rule 을 자동으로 추가하는 것은 모든 cluster egress 에 영향을 미칩니다. 이는 CloudTrail 감사와 사후 security team review 를 필수로 합니다. 반대로 Pod 을 재시작하는 것은 해당 workload 에만 영향을 미칩니다.

수정 타입 → blast radius → 승인 요구도를 매핑하는 matrix 를 작성하세요. 모든 변경은 CloudTrail 에 전체 컨텍스트와 함께 기록되어야 합니다.

**2. 복구 가능성과 감사 추적**

질문: 이를 되돌릴 수 있는가? Pod restart 는 가능하지만, PersistentVolumeClaim 삭제는 아닙니다. 복구 가능한 조치만 단계 1 (승인 불필요)에 해당합니다. 그 외는 다음을 포함한 감사 기록이 필요합니다.

- 에이전트의 결정 근거
- 승인 여부 및 승인자
- 실행 결과 및 부작용

**3. 비용과 성능 가드레일**

Node autoscaling 과 리소스 조정은 월간 비용에 직접 영향을 미칩니다. 에이전트가 스케일링 결정을 실행하기 전에 비용을 추정하게 합니다(예: "scale up 시 월 ~$500 추가 비용"). 임계값을 초과하는 제안은 승인 없이 거부합니다.

```bash
# Remediation 시도 빈도 모니터링
# 높은 빈도는 시스템적 under-provisioning 을 나타냄
aws cloudwatch put-metric-alarm \
  --alarm-name eks-remediation-spike \
  --metric-name RemediationAttemptCount \
  --namespace AWS/EKS \
  --statistic Sum \
  --period 3600 \
  --threshold 20 \
  --comparison-operator GreaterThanThreshold \
  --alarm-actions arn:aws:sns:ap-northeast-1:123456789012:ops-team
```

Remediation 빈도가 급증하면 근본적인 문제(클러스터 chronically under-resourced, 설정 오류 등)를 신호합니다. 자동 수정만 계속하지 말고, 팀에 alert 하고 architecture review 를 예약하세요.

## 구현 패턴

**환경별 단계적 롤아웃**

- **Dev/Staging**: 전체 자동 수정 활성화, false positive/negative rate 측정
- **Production**: 기본값으로 단계 2 (승인 필요) 설정, 6 개월 staging data 후 검증된 수정만 단계 1 로 승격
- **Disaster Recovery**: 7 일 window 내에서 rollback 연습, 분기별 DR drill 에 통합

**관찰성과의 통합**

Remediation 시도를 observability platform 에 feed 합니다. 같은 문제의 remediation 빈도가 높으면 시스템적 문제(capacity, configuration, upstream service failure)를 나타냅니다. 자동 수정과 함께 인간 조사로 escalate 되는 alert 을 설정하세요.

**승인 Workflow Best Practices**

- Pull Request 를 통한 승인―Remediation agent 가 change summary 생성, ops team 이 code 로 검토하고 승인
- 승인 SLO 설정 (예: 단계 2 는 업무 시간 내 15 분, 그 외 시간 1 시간 이내)
- Remediation 감사 로그 유지, 주간 리뷰로 정책 조정 필요성 파악

## 실제 예시

프로덕션 클러스터가 메모리 부족으로 Pod eviction 을 경험하고 있다고 가정합시다. AI 에이전트가 진단합니다.

1. 특정 deployment 에서 반복되는 oom-kill event 탐지
2. HPA min replica count 증가 및 node group size 확대 제안
3. 비용 추정: 월 $300 추가 비용
4. 승인 대기열에 수정 제안 등록

운영팀이 검토합니다.

- 최근 일주일간 메모리 사용률이 80% 이상으로 지속적
- 이 deployment 는 business-critical (수익 영향)
- 비용은 acceptable 수준
- 수정 제안 승인

에이전트가 실행합니다. HPA min replica 를 2 에서 4 로 증가, 새 EC2 instance 시작, Pod eviction 중단.

일주일 후, 그 spike 가 일시적(고객 대량 import)임을 깨닫습니다. 수동으로 스케일을 다시 줄입니다. 에이전트가 데이터를 수집하고 조치를 제안했지만, 최종 결정은 인간이 했습니다. 이것이 균형입니다.

## 정리

AI 기반 자동 수정과 Version rollback 은 운영 부담을 줄이고 incident 복구 시간을 단축합니다. 그러나 둘 다 좋은 클러스터 설계, 용량 계획, 관찰성을 대체할 수 없습니다. 일상적인 문제 해결을 자동화하여 운영자가 근본적인 개선에 집중할 수 있게 하는 defensive tool 입니다.

신중하게 배포하세요. Staging 에서 시작하고, false positive rate 를 측정하고, 승인 게이트를 설정하고, 모든 것을 기록하고, 수개월에 걸쳐 팀 신뢰를 구축하세요. 목표는 인간 운영자를 없애는 것이 아니라, 일상적인 문제 해결에서 해방시켜 더욱 효과적으로 만드는 것입니다.
