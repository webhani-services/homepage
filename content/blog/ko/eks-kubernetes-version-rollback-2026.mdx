Kubernetes 업그레이드는 오랫동안 '일방통행'이었습니다. 버전을 올렸는데 문제가 생기면, 백업에서 복구하거나 클러스터를 처음부터 다시 구축해야 했습니다. 둘 다 느리고 위험한 선택지였고, 특히 본운영에서 한밤중에 해야 한다면 더욱 그랬습니다.

이제 상황이 바뀝니다. Amazon EKS가 Kubernetes 버전 롤백 기능을 출시했습니다. 업그레이드 후 7일 이내라면 control plane을 이전 버전으로 롤백할 수 있습니다. 스냅샷 복구가 아니라, 실제로 운영 중이던 이전 버전으로의 완전한 복귀입니다.

## 롤백은 어떻게 작동하는가

EKS 클러스터를 업그레이드할 때, AWS는 이전 버전을 기록해둡니다. 업그레이드 후 문제가 발생하면 — API 호환성 오류, 워크로드 실패, 예상치 못한 동작 — 롤백을 실행해서 클러스터를 이전 상태로 돌릴 수 있습니다.

롤백에는 내장된 안전장치가 있습니다:

1. **롤백 준비상태 확인**: 롤백을 실행하기 전에 EKS가 안전성을 자동으로 평가합니다. 체크 항목에는 API 호환성, 노드 버전 스큐(skew), add-on 호환성, 클러스터 전체 상태가 포함됩니다.
2. **7일 윈도우**: 업그레이드 후 7일 이내에 롤백을 시작해야 합니다. 이 기간이 지나면 이전 버전은 사용할 수 없습니다.
3. **완전한 버전 복구**: 부분 복구가 아니라 이전 Kubernetes 버전으로 완전히 되돌립니다. 검증되고 운영 경험이 있는 상태입니다.

EKS Auto Mode 클러스터의 경우 control plane과 관리형 노드가 함께 롤백합니다. 노드 롤백은 Pod Disruption Budget(PDB)을 존중하므로 워크로드 설정에 따라 시간이 걸릴 수 있습니다.

## 왜 이것이 중요한가

핵심 문제: **팀들은 지원 종료 전에 Kubernetes를 업그레이드해야 합니다**. 구 버전은 보안 리스크이기 때문입니다. 하지만 업그레이드는 위험성을 갖습니다 — API 변경, 스케줄러 동작 변화, 워크로드 측 예상치 못한 오류 등. 이전에는 철저한 스테이징 검증만이 대책이었고, 그것도 본운영을 완전히 재현할 수 없었습니다.

롤백 기능이 있으면 계산이 달라집니다. 이제 이런 것들이 가능합니다:

- 더 자신감 있게 업그레이드할 수 있습니다
- 새로운 API 서버에 대해 스모크 테스트와 통합 테스트를 실행합니다
- 제한된 기간(보통 24~48시간) 내에 핵심 메트릭을 모니터링합니다
- 추측이 아니라 본운영 신호에 기반한 롤백 판단을 내립니다

## 롤백 준비상태 확인

롤백을 실행하기 전에 EKS가 자동 체크를 실행하고 롤백이 안전한지 보고합니다:

- **API 호환성**: 워크로드 매니페스트가 이전 API 버전에서 작동하는가?
- **Add-on 호환성**: 설치된 EKS add-on들(CoreDNS, VPC CNI, kube-proxy 등)이 이전 control plane 버전과 호환되는가?
- **버전 스큐**: 노드 Kubernetes 버전이 롤백 대상 버전과의 지원 범위 내에 있는가?
- **클러스터 상태**: 미해결 오류나 클러스터 전체 문제는 없는가?

이런 체크들은 자동으로 실행되고 롤백 결정 흐름에서 보고됩니다. 문제가 플래그되면 롤백 전에 처리하거나, 새 버전에서 원인을 해결할지 판단할 수 있습니다.

## 실무적 업그레이드 런북

롤백을 판단 포인트로 포함한 현실적 업그레이드 워크플로우:

1. **Control plane 업그레이드**(유지보수 윈도우 또는 비즈니스 시간 외)
   ```bash
   aws eks update-cluster-version \
     --name my-cluster \
     --region ap-northeast-2 \
     --kubernetes-version 1.31
   ```

2. **업그레이드 진행상황 모니터링**: 클러스터가 새 버전에 도달할 때까지 대기

3. **스모크 테스트**(즉시 실행):
   - 테스트 Pod를 배포해서 API 서버 통신 확인
   - 핵심 워크로드가 실행 중인지 확인
   - 로그에서 API 폐지 예정 경고 확인

4. **24~48시간에 걸쳐 통합 테스트 실행**:
   - 주요 애플리케이션 흐름 검증
   - 관찰성 파이프라인(메트릭, 로그, 트레이스) 검증
   - SLO 대시보드에서 이상 확인

5. **롤백 판단 게이트**(2~3일차):
   - 중대한 문제가 발견되면 롤백 준비상태 확인:
     ```bash
     aws eks describe-cluster-version-rollback-status \
       --cluster-name my-cluster \
       --region ap-northeast-2
     ```
   - 롤백이 안전하면 실행:
     ```bash
     aws eks rollback-cluster-version \
       --cluster-name my-cluster \
       --region ap-northeast-2
     ```
   - 롤백 미권장(add-on 호환성, 상태 문제)이면 새 버전에서 원인 해결

6. **노드 롤백**(필요시): Auto Mode 클러스터의 경우 노드가 control plane과 함께 롤백되며 PDB를 존중합니다.

7. **업그레이드 완료**(7일차): 7일 윈도우가 지나고 롤백 실행이 없으면 이전 버전은 사용 불가능해지고 업그레이드가 확정됩니다.

## 중요한 제한사항

- **7일 윈도우**: 신속한 판단이 필요합니다. 롤백은 장기 탈출구가 아니라 발견한 문제에 대한 안전장치입니다.
- **전체 롤백**: 개별 컴포넌트나 add-on이 아니라 전체 클러스터를 롤백합니다.
- **PDB 인식 노드 롤백**: Auto Mode의 경우 노드는 disruption budget을 지키며 롤백합니다. 엄격한 PDB를 가진 대규모 클러스터는 수 시간이 걸릴 수 있습니다.
- **비용**: 표준 EKS 및 컴퓨트 비용이 발생합니다. 롤백 작업 자체에는 추가 요금이 없습니다.
- **이전 버전만**: 바로 전 버전으로만 롤백 가능합니다. 여러 버전 이전으로는 롤백할 수 없습니다.

## 이용 가능성

이 기능은 오늘부터 추가 비용 없이 EKS를 사용할 수 있는 모든 상용 AWS 리전에서 사용 가능합니다. EKS 콘솔, AWS CLI, AWS SDK에서 실행할 수 있습니다.

## 비즈니스 관점

CNCF의 최근 Cloud Native Survey에 따르면 본운영에서 실행되는 컨테이너 워크로드의 약 82%가 Kubernetes입니다. 그 규모에서 업그레이드는 지속적으로 발생하며, 업그레이드 리스크를 낮추는 메커니즘은 실제 비즈니스 가치가 있습니다.

롤백 기능은 스테이징 검증의 대체가 아니라, 관찰성 위에 구축된 인프라입니다. "기도하며 업그레이드"에서 "빠르게 검증하고 실데이터로 판단"으로 업그레이드의 마인드셋이 바뀝니다.
