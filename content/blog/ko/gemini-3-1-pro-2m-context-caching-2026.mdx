Google이 Vertex AI에서 Gemini 3.1 Pro를 General Availability로 출시했습니다. 주목할 기능은 두 가지입니다 — 200만 Token Context Window와 Document Level Caching. 후자는 책 전체나 코드베이스를 캐시된 Context에 미리 로드하여 여러 요청에 걸쳐 재사용할 수 있는 기능입니다.

이것은 단순히 Context Window가 커진 것이 아닙니다. 200만 Token은 약 1,500페이지 분량의 텍스트, 또는 중간 규모 코드베이스 전체를 하나의 Prompt에 담을 수 있는 규모입니다. Document Level Caching은 처리 비용을 한 번 지불하면 이후 요청마다 다시 지불하지 않아도 된다는 것을 의미합니다.

이 글에서는 이러한 기능들이 실제 워크로드에 어떤 영향을 미치는지, 그리고 어떻게 활용하는지를 살펴봅니다.

## 200만 Token Context의 실제 스케일

200만 Token을 구체적으로 표현하면:

- 텍스트 약 1,500~2,000페이지
- 중간 규모 코드베이스 전체 (5만~10만 줄)
- 기술 문서 세트 전체
- 1fps로 수 시간 분량의 동영상

참고로 GPT-4o는 128K, Claude Opus 4.7은 200K Context를 지원합니다. Gemini 3.1 Pro의 200만 Token은 Document 규모 Task에서 다른 차원의 선택지입니다.

실무적 의미는 명확합니다. 이전에는 문서를 청크로 분할하고, Retrieval 인프라를 유지하며, 부분적인 결과를 조합해야 했던 워크로드를 단일 Prompt로 처리할 수 있게 됩니다. 이는 Engineering 복잡성의 한 계층을 통째로 제거하는 것입니다.

## Document Level Caching

운용 측면에서 더 가치 있는 기능은 Caching입니다. Vertex AI의 Context Caching으로 대규모 문서(코드 Repository, 기술 매뉴얼, 법률 문서)를 캐시된 Context에 로드하고, 이후 쿼리에서는 전체 Context를 매번 처리하지 않고 활용할 수 있습니다.

Vertex AI Python SDK 구현 예시:

```python
import vertexai
from vertexai.generative_models import GenerativeModel, Content, Part
from vertexai.preview.caching import CachedContent
import datetime

vertexai.init(project="your-project-id", location="us-central1")

# 대규모 코드베이스 또는 문서를 캐시된 Context에 로드
with open("codebase_context.txt", "r") as f:
    large_document = f.read()

# Cache 항목 생성 — 한 번 처리하면 24시간 재사용 가능
cached_content = CachedContent.create(
    model_name="gemini-3.1-pro-002",
    system_instruction="이 코드베이스를 분석하는 시니어 엔지니어로 동작하세요.",
    contents=[
        Content(role="user", parts=[Part.from_text(large_document)])
    ],
    ttl=datetime.timedelta(hours=24),
    display_name="codebase-analysis-cache",
)

# 캐시된 Context에 대해 여러 쿼리 실행
model = GenerativeModel.from_cached_content(cached_content=cached_content)

queries = [
    "인증 흐름의 주요 Entry Point는 어디인가요?",
    "의존성이 가장 많은 모듈은 무엇인가요?",
    "데이터베이스 Access Layer에 N+1 쿼리 패턴이 있나요?",
    "결제 처리 코드의 테스트 커버리지는 어느 정도인가요?",
]

for query in queries:
    response = model.generate_content(query)
    print(f"Q: {query}")
    print(f"A: {response.text}\n")

# Cache는 TTL 동안 유지 — 각 호출마다 재처리 비용 없음
```

핵심 이점: 500페이지 기술 매뉴얼에 하루 20번 쿼리한다면, 캐시 없이는 20번 분의 Input Token 비용이 발생하지만 Cache를 사용하면 1번만 지불합니다.

Vertex AI의 Cached Content 요금은 일반 Input Token 비용의 약 25~50%로, 반복적인 Document Analysis 워크로드에서 경제적으로 큰 의미가 있습니다.

## Native Video Understanding

Gemini 3.1 Pro는 1fps Native 동영상 이해도 지원합니다. 단순 이미지 추출이 아닌, 동영상의 시간적 시퀀스를 추론하고 Action과 전환을 이해하며 동영상 전체에 걸쳐 무슨 일이 일어나는지 답할 수 있습니다.

```python
import vertexai
from vertexai.generative_models import GenerativeModel, Part

vertexai.init(project="your-project-id", location="us-central1")
model = GenerativeModel("gemini-3.1-pro-002")

# Google Cloud Storage에서 동영상 처리
video_part = Part.from_uri(
    uri="gs://your-bucket/architecture-demo.mp4",
    mime_type="video/mp4"
)

response = model.generate_content([
    video_part,
    "이 아키텍처 설명 동영상에 등장하는 모든 서비스를 나열하고 상호작용 방식을 설명해주세요."
])

print(response.text)
```

실용적인 활용 예: 아키텍처 리뷰 녹화본 처리, 동영상 튜토리얼에서 구조화된 정보 추출, 애플리케이션 흐름의 화면 녹화를 통한 UI 문제 식별.

## 다른 모델과의 사용 구분

Gemini 3.1 Pro가 적합한 경우:

**대량 문서 분석**: 큰 문서를 반복적으로 쿼리하는 워크로드에서는 Caching 기능으로 이 패턴에서 가장 비용 효율적인 선택이 됩니다.

**코드베이스 전체 분석**: 200만 Token Window로 청크 분할 없이 코드베이스 전체를 로드할 수 있습니다. "전체 Repository에서 X 패턴이 사용된 곳을 모두 찾기" 같은 Task에서 RAG 기반 접근법의 복잡성을 제거할 수 있습니다.

**동영상 콘텐츠 처리**: 동등한 Context 길이의 Native 동영상 이해는 다른 주요 Provider에서 제공하지 않습니다.

**GCP 통합 워크로드**: 이미 Google Cloud에서 운영 중이라면 Vertex AI와 IAM·VPC Service Controls·기존 인프라의 통합으로 운용 오버헤드가 줄어듭니다.

일반적인 용도(코딩 지원, Q&A, 소규모 문서 요약)에서는 Claude나 GPT가 더 단순한 선택입니다. 200만 Context와 Caching 기능은 Document 규모 및 반복 쿼리 패턴에서 주로 가치를 발휘합니다.

## webhani의 실사례

대규모 규제 문서군을 반복적으로 분석하는 Client Use Case에서 Gemini 3.1 Pro를 테스트했습니다. 수백 페이지의 정책 문서에 하루 여러 번 특정 질문에 답해야 하는 작업입니다.

Caching 기능은 실제로 경제성을 바꿔놓았습니다. 400페이지 규제 문서에 하루 50번 쿼리하는 경우, Cache 방식은 매번 전체 Context를 전송하는 방식 대비 약 75% 비용 절감이 가능했습니다. 규모가 커질수록 무시할 수 없는 차이입니다.

200만 Token Context Window는 이전에 구축했던 RAG 인프라도 불필요하게 만들었습니다. 이 워크로드에서는 청크 분할·Embedding·Retrieval의 복잡성이 Latency를 늘리고 때로는 Retrieval 실패도 발생했습니다. 이를 단순한 Document-in-Context 방식으로 대체하면서 신뢰성과 개발자 경험이 모두 개선됐습니다.

주의할 점: 이렇게 큰 Context Window가 항상 정답은 아닙니다. 여러 문서를 횡단해 검색해야 하거나, Cache 재생성 없이 콘텐츠를 자주 업데이트해야 하는 경우에는 RAG가 여전히 뛰어난 아키텍처입니다. Access 패턴에 맞는 설계를 선택하는 것이 중요합니다.
