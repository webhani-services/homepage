## Google I/O 2026 발표 내용

2026년 5월 19일 Google I/O에서 Gemini 3.5 Flash와 Gemini Spark 두 모델이 발표됐습니다. 같은 주에 Alibaba의 Qwen 3.7 Max(5월 20일)와 OpenAI의 업데이트도 있었습니다. LLM 릴리즈 속도는 여전히 빠릅니다.

이 글에서는 개발 현장에서 모델을 선택하고 통합하는 관점에서 이번 발표를 읽어봅니다.

## Gemini 3.5 Flash: 속도와 비용 효율의 중심

Gemini Flash 시리즈는 일관된 포지셔닝을 유지해왔습니다. "속도를 우선하면서 비용 효율이 뛰어난 범용 모델"입니다. Gemini 3.5 Flash는 1.5 Flash 대비 의미 있는 성능 향상을 보여줍니다.

실무에서 잘 맞는 용도:

```python
import google.generativeai as genai

genai.configure(api_key="your-api-key")
model = genai.GenerativeModel("gemini-3.5-flash")

# 대량 분류 작업 — 비용 효율 우선
def classify_support_tickets(tickets: list[str]) -> list[dict]:
    results = []
    for ticket in tickets:
        response = model.generate_content(
            f"""다음 지원 티켓을 아래 카테고리 중 하나로 분류하세요:
            billing | technical | account | other

            티켓: {ticket}

            카테고리만 답하세요."""
        )
        results.append({
            "ticket": ticket,
            "category": response.text.strip()
        })
    return results

# 비정형 텍스트에서 구조화된 데이터 추출
def extract_entities(text: str) -> str:
    response = model.generate_content(
        f"""이 텍스트에서 엔티티를 JSON으로 추출하세요:
        {{"people": [], "organizations": [], "locations": []}}

        텍스트: {text}"""
    )
    return response.text
```

수천 건의 요청을 처리해야 하고 정확도 요구사항이 중간 수준인 경우에 Flash가 적합합니다. 복잡한 추론, 코드 생성, 미묘한 분석이 필요한 경우에는 맞지 않습니다.

## Gemini Spark: 온디바이스 모델

Gemini Spark는 클라우드 API 없이 사용자 기기에서 직접 추론하는 온디바이스 모델입니다. 이것은 다른 카테고리의 사용 사례입니다.

**프라이버시 우선 시나리오**: 사용자 데이터가 기기 밖을 나가지 않습니다. GDPR 및 개인정보보호법 대응이 단순해집니다.

**오프라인 동작**: 인터넷 연결 없이 LLM 기능 제공이 가능합니다. 모바일 앱, 현장 작업 도구, 제한된 네트워크 환경에 유용합니다.

**제로 API 레이턴시**: 네트워크 왕복이 없어 응답 시간이 밀리초 단위입니다.

트레이드오프는 성능입니다. 소형 온디바이스 모델은 복잡한 추론이나 코드 생성에서 클라우드 모델을 따라갈 수 없습니다. Spark는 좁고 명확히 정의된 작업에 맞습니다: 텍스트 분류, 간단한 추출, 자동완성, 짧은 텍스트 요약.

```python
# Spark 온디바이스 추론 (개념적 API — 플랫폼마다 다름)
from gemini_spark import LocalModel

model = LocalModel.load("gemini-spark-2026")

# 네트워크 없이 동작 — 모바일/엣지 앱에 유용
response = model.generate(
    "이 이메일을 한 문장으로 요약해 주세요: ...",
    max_tokens=100
)
```

## 2026년 LLM 경쟁 환경

이번 발표로 주요 모델의 포지셔닝이 더 명확해졌습니다:

| 모델 | 포지셔닝 | 주요 활용처 |
|-----|---------|-----------|
| Claude Opus 4.7 | 최고 정확도, 복잡한 추론 | 코드 생성, 기술 분석 |
| GPT-5 | 고성능 범용 | 광범위한 업무 태스크 |
| Gemini 3.5 Flash | 속도, 비용 효율 | 대량 처리, 분류 태스크 |
| Gemini Spark | 엣지, 프라이버시, 오프라인 | 온디바이스 AI |
| Qwen 3.7 Max | 다국어, 아시아 특화 | 지역 언어 애플리케이션 |

모든 차원에서 하나의 모델이 이기는 경우는 없습니다. 이것이 멀티모델 라우팅이 표준 패턴이 되어가는 이유입니다.

## 멀티모델 라우팅 구현

LLM 시장이 복잡해지면서 모든 것에 한 모델을 사용하는 것은 점점 비효율적입니다:

```python
import anthropic
import google.generativeai as genai
from typing import Literal

TaskType = Literal["complex_reasoning", "bulk_classification", "simple_extraction"]

def route_to_model(task_type: TaskType, content: str) -> str:
    if task_type == "complex_reasoning":
        # 정확도가 중요 — Claude Opus 사용
        client = anthropic.Anthropic()
        response = client.messages.create(
            model="claude-opus-4-7",
            max_tokens=2000,
            messages=[{"role": "user", "content": content}]
        )
        return response.content[0].text

    elif task_type in ("bulk_classification", "simple_extraction"):
        # 비용이 중요 — Gemini Flash 사용
        genai.configure(api_key="your-gemini-key")
        flash = genai.GenerativeModel("gemini-3.5-flash")
        return flash.generate_content(content).text

    else:
        raise ValueError(f"Unknown task type: {task_type}")
```

라우팅 로직이 복잡할 필요는 없습니다. 태스크 유형을 모델에 매핑하는 단순한 규칙 기반 시스템으로도 대부분의 가치를 얻을 수 있습니다.

## Vertex AI를 통한 엔터프라이즈 통합

프로덕션 환경에서는 직접 API보다 Vertex AI를 통해 Google 모델을 사용하는 것이 좋습니다:

```python
from google.cloud import aiplatform
from vertexai.generative_models import GenerativeModel

aiplatform.init(project="your-gcp-project", location="us-central1")

model = GenerativeModel("gemini-3.5-flash")

response = model.generate_content(
    "이 고객 피드백을 분석하고 주요 이슈를 추출해 주세요: ...",
    generation_config={
        "temperature": 0.1,  # 분류 안정성을 위한 낮은 온도
        "max_output_tokens": 500,
    }
)
```

Vertex AI의 이점:
- 엔터프라이즈 계약 시 데이터가 모델 훈련에 사용되지 않음
- VPC Service Controls 지원
- Cloud Monitoring, Cloud Logging과의 네이티브 통합

## 모델별 비용 추적

여러 모델을 운영할 때 비용 가시성이 필수적입니다:

```python
from dataclasses import dataclass, field
from collections import defaultdict

@dataclass
class LLMCostTracker:
    # 1K 토큰당 대략적인 비용 (입력)
    price_per_1k_input: dict = field(default_factory=lambda: {
        "gemini-3.5-flash": 0.001,
        "gemini-3.5-pro": 0.005,
        "claude-opus-4-7": 0.015,
    })
    totals: dict = field(default_factory=lambda: defaultdict(float))

    def record(self, model: str, input_tokens: int):
        cost = (input_tokens / 1000) * self.price_per_1k_input.get(model, 0)
        self.totals[model] += cost

    def report(self):
        for model, total in self.totals.items():
            print(f"{model}: ${total:.4f}")

tracker = LLMCostTracker()
# LLM 호출 후 tracker.record() 실행
```

비용 추적 없이는 대량의 Flash 요청이 예상치 못한 비용을 발생시키거나, Flash로 충분한 곳에 비싼 Opus 호출이 남아있는 상황을 발견하기 어렵습니다.

## webhani의 시각

이번 Google I/O 발표에서 주목하는 것은 모델 라인업의 폭입니다. Spark 같은 엣지 모델부터 Flash의 고속 클라우드 모델까지, 용도별로 선택할 수 있는 옵션이 갖춰지고 있습니다.

실무 권장사항:
1. **먼저 태스크를 분류하세요**: 정확도 중심 vs 비용 중심 vs 프라이버시 중심
2. **자체 데이터로 벤치마크하세요**: 공개 벤치마크보다 실제 유스케이스 평가가 중요합니다
3. **멀티모델 설계를 전제로 하세요**: 특정 프로바이더 의존을 피하고 추상화 레이어를 설계하세요
4. **비용 추적을 도입하세요**: 모델별 비용을 가시화하고 최적화 데이터를 축적하세요

LLM 경쟁은 계속됩니다. 추상화 레이어를 미리 구축해두면, 새로운 모델이 나올 때마다 애플리케이션 로직을 다시 작성하지 않고도 전환하거나 라우팅할 수 있습니다.
