CI/CD 파이프라인은 공급망 공격의 주요 표적입니다. GitHub의 2026 보안 로드맵은 이를 해결하기 위한 여러 기능을 발표했으며, 현재 적용 가능한 핵심 기능은 **Workflow Dependency Locking**과 **OIDC Custom Property Claims(2026년 4월 GA)**입니다.

기존에 다룬 Egress Firewall과는 다른 계층에서 동작하며, 함께 적용하면 보다 완전한 공급망 방어를 구성할 수 있습니다.

## Workflow Dependency Locking

현재 GitHub Actions에서 `actions/checkout@v4`처럼 태그를 사용해 Action을 참조할 때 문제가 있습니다. 태그는 변경 가능하기 때문에 저장소가 침해되면 동일한 태그가 악성 코드를 가리킬 수 있습니다.

Dependency Locking은 Go의 `go.sum`과 유사한 방식으로 워크플로우 YAML의 `dependencies:` 섹션에 모든 직접·간접 의존성을 커밋 SHA로 고정합니다.

```yaml
# .github/workflows/ci.yml
name: CI

on: [push, pull_request]

dependencies:
  actions/checkout:
    version: "v4"
    sha: "11bd71901bbe5b1630ceea73d27597364c9af68"
  actions/setup-node:
    version: "v4"
    sha: "49933ea5288caeca8642d1e84afbd3f7d6820020"
  actions/cache:
    version: "v4"
    sha: "5a3ec84eff668545956fd18022ec81e4c5de7fe6"

jobs:
  build:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: "22"
      - run: npm ci && npm test
```

SHA가 일치하지 않으면 코드 실행 전에 Job이 중단됩니다. 의존성 변경은 PR에 diff로 나타나므로 일반 코드 리뷰 프로세스로 검토할 수 있습니다.

## OIDC Custom Property Claims (현재 GA)

GitHub Actions의 기본 OIDC 토큰에는 저장소 이름, 브랜치, Job 이름 같은 기본 정보가 포함됩니다. 이것만으로는 저장소가 *무엇인지*에 기반한 속성 기반 접근 제어(ABAC)를 표현하기 어렵습니다.

Custom Property Claims는 조직/엔터프라이즈 관리자가 저장소 커스텀 속성을 OIDC 토큰 클레임으로 포함할 수 있게 합니다. `team=backend`, `env=production`으로 태그된 저장소는 해당 값을 토큰 클레임으로 내보냅니다.

```yaml
# .github/workflows/deploy.yml
permissions:
  id-token: write
  contents: read

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789:role/BackendProdRole
          aws-region: ap-northeast-1
```

클라우드 측에서는 조건 기반 신뢰 정책을 적용할 수 있습니다.

```json
{
  "Condition": {
    "StringEquals": {
      "token.actions.githubusercontent.com:sub":
        "repo:myorg/myrepo:ref:refs/heads/main",
      "token.actions.githubusercontent.com:custom_property_team": "backend",
      "token.actions.githubusercontent.com:custom_property_env": "production"
    }
  }
}
```

IAM Role 신뢰 정책이 저장소 이름 하드코딩 없이 속성으로 접근을 제어합니다. 새 저장소를 `backend` 팀에 추가하면 자동으로 해당 AWS Role 접근이 부여되고, 속성을 제거하면 권한이 즉시 취소됩니다.

## 구현 우선순위

보안 기능 도입은 리스크와 비용을 고려해 순서를 정하는 것이 좋습니다.

| 기능 | 보호 대상 | 설정 비용 | 우선순위 |
|------|----------|---------|---------|
| SHA 고정(기존) | Action 변조 | 낮음 | 즉시 적용 권장 |
| Dependency Locking | 전이적 의존성 변조 | 중간 | 로드맵 추적 |
| OIDC Custom Claims | 속성 기반 클라우드 접근 | 중간 | 다중 팀 환경에 효과적 |
| Egress Firewall | Runner 외부 통신 차단 | 높음 | 위험 평가 후 도입 |

지금 바로 할 수 있는 것은 기존 워크플로우에서 태그 참조를 SHA 고정으로 교체하는 것입니다. `step-security/harden-runner` 같은 도구로 자동화할 수 있습니다.

## 정리

Dependency Locking은 CI 공급망의 가변 태그 위험을 선언적 잠금으로 해결합니다. OIDC Custom Property Claims는 저장소 속성 기반의 클라우드 접근 제어로 확장성 있는 권한 관리를 가능하게 합니다.

OIDC Custom Claims는 현재 사용 가능합니다. 다양한 신뢰 수준의 여러 저장소에 걸쳐 접근을 관리하고 있다면, 2026 보안 로드맵의 나머지 기능이 출시되기 전에 도입을 검토할 가치가 있습니다.
