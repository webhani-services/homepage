2026년 1분기, CI/CD 파이프라인을 겨냥한 공급망 공격이 잇따라 발생했습니다. tj-actions/changed-files, trivy-action 침해, 악성 Rust Crate 5건에서 공통된 패턴이 나타났습니다: CI Runner 내부에서 코드를 실행하고, 외부 서버로 Secret을 유출하는 방식입니다.

GitHub는 이에 대응하여 2026년 Actions 보안 로드맵을 공개했습니다. 핵심 기능은 Runner VM **외부**에서 동작하는 Layer 7 **Egress Firewall**입니다.

## VM 외부에서 동작하는 이유

기존 Egress 제어 솔루션(harden-runner 등 포함)은 Runner *내부*에서 실행됩니다. 공격자가 VM 내부에서 root 권한을 획득하면 이를 무력화할 수 있습니다.

GitHub의 Firewall은 네트워크 인프라 레벨, 즉 Runner와 인터넷 사이에 위치합니다. VM 내부의 root 권한으로는 이를 우회할 수 없습니다.

```
┌─────────────────────────────────┐
│  GitHub Actions Runner (VM)     │
│  └── 빌드 스크립트 실행          │
│      └── 외부 통신 시도          │
└─────────────────────────────────┘
           ↓ 트래픽이 VM 외부로
┌─────────────────────────────────┐
│  Egress Firewall (Layer 7)      │  ← VM 외부, 변경 불가
│  ├── 허용: api.github.com       │
│  ├── 허용: registry.npmjs.org   │
│  └── 차단: 나머지 모든 통신      │
└─────────────────────────────────┘
```

## Policy 설정 방법

Egress Policy는 Workflow 파일의 Job 단위로 정의합니다:

```yaml
jobs:
  build:
    runs-on: ubuntu-latest
    egress-policy: block

    allowed-endpoints:
      - api.github.com:443
      - registry.npmjs.org:443
      - objects.githubusercontent.com:443

    steps:
      - uses: actions/checkout@v4
      - run: npm ci
      - run: npm test
```

GitHub은 `audit` 모드로 시작할 것을 권장합니다. 실제 차단 없이 모든 아웃바운드 연결을 로깅하여 Allowlist를 구축한 뒤 `block`으로 전환하는 순서입니다.

조직 레벨에서 중앙화된 Policy를 정의하여 모든 Repository에 일괄 적용할 수도 있습니다.

## Actions Data Stream

로드맵에 포함된 또 다른 기능인 **Actions Data Stream**은 CI/CD 실행 텔레메트리를 Amazon S3 또는 Azure Event Hub로 근실시간 전송합니다. 향후에는 프로세스 레벨 가시성과 파일 시스템 모니터링도 추가 예정입니다.

CI/CD에 프로덕션 시스템과 동일한 수준의 관찰 가능성을 부여하는 기능으로, 기존 SIEM이나 관찰 Platform과 연동할 수 있습니다.

## 지금 당장 할 수 있는 것

Egress Firewall은 3~6개월 내 Public Preview, 6~9개월 내 GA 예정입니다. GA를 기다리는 동안 적용할 수 있는 대책들:

**Action을 특정 SHA로 고정**

```yaml
# 버전 Tag가 아닌 Commit SHA로 지정
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2

# Tag는 공격자가 Action 저장소를 침해하면 이동시킬 수 있음
# SHA는 변경 불가
```

**GITHUB_TOKEN 권한 최소화**

```yaml
# Workflow 파일 상단
permissions:
  contents: read        # Checkout에 필요한 최소 권한
  pull-requests: write  # 필요한 경우에만

# 권한을 미설정하면 기본값이 넓은 범위로 적용됨
```

**`pull_request_target` 사용 시 주의**

이 Trigger는 Base Branch의 Context에서 실행되며 Secret에 접근할 수 있습니다. Fork에서 온 코드를 Checkout하여 실행하는 경우, 공격자가 PR을 통해 Secret에 접근할 수 있습니다. 현재 Workflow를 감사하여 이 패턴이 없는지 확인하세요.

**임시 방편으로 harden-runner 활용**

```yaml
- uses: step-security/harden-runner@v2
  with:
    egress-policy: audit
```

VM 내부에서 동작하므로 GitHub 네이티브 솔루션보다 견고함이 낮지만, 공식 Firewall GA까지의 가시성 확보 수단으로 유효합니다.

## 최근 CI/CD 침해 사례 배경

이번 로드맵을 촉발한 주요 사건:

1. **tj-actions/changed-files** — 수천 개의 Repository가 참조하는 Action이 침해되어, Mutable Tag를 통해 CI Secret이 대규모 유출
2. **Kubernetes-el Emacs 패키지** — `pull_request_target` 잘못된 설정을 이용해 `GITHUB_TOKEN` 탈취 및 파괴적 코드 주입
3. **악성 Rust Crate 5건** — `.env` 파일을 수집하여 공격자 서버로 전송하는 코드 포함, 개발자 환경과 CI 환경 모두 표적

## 정리

Egress Firewall은 이러한 공격의 근본 원인, 즉 CI Runner의 제한 없는 아웃바운드 네트워크 접근을 차단합니다. GA 이후에는 빌드 Pipeline이 통신할 수 있는 대상을 정확히 정의하고 나머지를 인프라 레벨에서 차단할 수 있습니다.

GA 전까지는 Action SHA 고정, Token 권한 최소화, `pull_request_target` 감사 — 이 세 가지만 적용해도 가장 흔한 공격 벡터를 상당 부분 차단할 수 있습니다.
