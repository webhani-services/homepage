GitHub Actions에서 AWS 리소스를 다룰 때 `AWS_ACCESS_KEY_ID`와 `AWS_SECRET_ACCESS_KEY`를 Repository Secret에 등록하는 방식은 여전히 널리 사용됩니다. 빠르게 동작하고 설정이 단순하기 때문입니다. 그러나 이 방식은 만료 시점이 없는 장기 자격증명을 CI 시스템 안에 보관하는 구조입니다. 이 글에서는 그 위험성을 구체적으로 살펴보고, OIDC(OpenID Connect)를 활용해 장기 자격증명 없이 AWS를 인증하는 방법을 Terraform과 실제 Workflow 코드로 설명합니다.

## 장기 IAM Access Key의 보안 위험

IAM Access Key는 생성 시점부터 명시적으로 삭제하거나 비활성화하지 않는 한 유효합니다. CI 환경에서 이 특성이 어떤 문제를 만드는지 구체적으로 살펴보겠습니다.

**노출 경로가 다양합니다.** GitHub의 Secret 마스킹은 일반적인 패턴은 가려주지만 Base64 인코딩이나 분할 출력으로 우회할 수 있습니다. 디버깅 목적으로 임시 추가한 `env` 출력 Step이 로그에 자격증명을 노출시키는 사례는 드물지 않습니다. Workflow에 포함된 서드파티 Action도 같은 환경에 접근할 수 있습니다.

**Rotation이 현실적으로 잘 이루어지지 않습니다.** IAM Key를 교체하려면 새 키를 발급하고, 해당 키를 참조하는 모든 Secret Store를 업데이트하고, 동작을 검증한 뒤 기존 키를 삭제해야 합니다. 여러 Repository에 동일한 키를 사용하고 있다면 작업량은 더 늘어납니다. 결과적으로 수개월 혹은 수년이 지난 키가 활성 상태로 남아있는 경우가 많습니다.

**유출 즉시 악용이 가능합니다.** IAM Access Key는 어디서든 즉시 사용 가능합니다. MFA 없이 동작하고, 특정 IP 제한도 기본적으로 없습니다. Public Repository에 키가 한 번이라도 노출되면 자동화된 Scanner가 수 분 내에 탐지합니다. AWS가 직접 Public GitHub Repository를 스캔해 탐지 알림을 보내기도 하지만, 그 시점에는 이미 키가 외부에 공개된 상태입니다.

**사고 발생 시 Blast Radius 파악이 어렵습니다.** 키가 유출됐을 때 어떤 리소스에 어떤 작업이 이루어졌는지 CloudTrail에서 추적해야 합니다. 오래된 키일수록 추적 범위가 넓어지고, Incident 상황에서 이 분석을 수행하는 것은 부담이 큽니다.

## OIDC Token 교환 흐름 설명

OIDC 인증은 장기 자격증명을 CI 시스템에 저장하는 대신, "이 Job이 정말 GitHub Actions에서 실행된 것인가"를 실시간으로 검증합니다.

흐름을 단계별로 설명하면 다음과 같습니다.

1. GitHub Actions Job이 시작되면 GitHub의 내장 OIDC Provider가 해당 Job에 대한 **단기 서명 JWT**를 발행합니다.
2. JWT에는 검증 가능한 Claim이 포함됩니다. `repo`, `ref`, `sha`, `actor`, `workflow`, `event_name`, `environment` 등입니다.
3. Workflow가 AWS STS의 `AssumeRoleWithWebIdentity`를 호출하며 JWT를 전달합니다.
4. AWS가 GitHub의 공개 JWKS Endpoint에서 공개키를 가져와 Token 서명을 검증합니다. 이 과정에서 공유 비밀키가 사용되지 않습니다.
5. IAM Role의 Trust Policy 조건과 Token의 Claim을 비교합니다.
6. 조건을 만족하면 STS가 **최대 1시간 유효한 임시 자격증명**을 반환합니다.
7. Job 종료 후 자격증명은 자동 만료됩니다. 별도의 정리 작업이 필요 없습니다.

Workflow 파일에 노출되는 IAM Role ARN은 민감한 정보가 아닙니다. ARN만 알아서는 Trust Policy 조건을 만족하지 않으면 해당 Role을 assume할 수 없습니다. 비밀번호가 아니라 Resource 식별자입니다.

## AWS IAM Identity Provider 설정

GitHub OIDC Endpoint를 AWS에 등록하는 작업은 Account당 한 번만 수행합니다. 이후에는 여러 Role이 동일한 Provider를 참조할 수 있습니다. Terraform으로 구성하면 다음과 같습니다.

```hcl
# terraform/iam-oidc.tf

# GitHub Actions OIDC Provider를 AWS에 등록
resource "aws_iam_openid_connect_provider" "github" {
  url             = "https://token.actions.githubusercontent.com"
  client_id_list  = ["sts.amazonaws.com"]

  # token.actions.githubusercontent.com의 Root CA 인증서 Thumbprint
  # 아래 명령어로 최신값 확인 가능:
  # openssl s_client -connect token.actions.githubusercontent.com:443 \
  #   -showcerts < /dev/null 2>/dev/null | openssl x509 -fingerprint -noout -sha1
  thumbprint_list = ["1b511abead59c6ce207077c0bf0e0043b1382612"]
}

# 배포용 IAM Role
resource "aws_iam_role" "github_actions_deploy" {
  name        = "github-actions-deploy"
  description = "GitHub Actions OIDC 인증으로 Assume하는 배포 Role. 장기 Key 없음."

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Principal = {
          Federated = aws_iam_openid_connect_provider.github.arn
        }
        Action = "sts:AssumeRoleWithWebIdentity"
        Condition = {
          StringEquals = {
            # audience는 반드시 sts.amazonaws.com으로 고정
            "token.actions.githubusercontent.com:aud" = "sts.amazonaws.com"
            # 특정 Repository의 main Branch만 허용 (정확한 일치)
            "token.actions.githubusercontent.com:sub" = "repo:your-org/your-repo:ref:refs/heads/main"
          }
        }
      }
    ]
  })

  # Session 최대 지속 시간 (초). 기본값 3600초로 충분
  max_session_duration = 3600
}

# Role에 부여할 권한 Policy (필요 최소한으로 제한)
resource "aws_iam_role_policy" "github_actions_deploy" {
  name = "deploy-policy"
  role = aws_iam_role.github_actions_deploy.id

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect = "Allow"
        Action = [
          "s3:PutObject",
          "s3:DeleteObject",
          "s3:ListBucket",
          "s3:GetObject"
        ]
        Resource = [
          "arn:aws:s3:::my-deployment-bucket",
          "arn:aws:s3:::my-deployment-bucket/*"
        ]
      },
      {
        Effect = "Allow"
        Action = ["cloudfront:CreateInvalidation"]
        # 특정 Distribution ID로 제한하는 것이 더 안전
        Resource = "arn:aws:cloudfront::123456789012:distribution/*"
      }
    ]
  })
}
```

## GitHub Actions Workflow 구성

Workflow 측 설정에서 핵심은 `permissions` 블록에서 `id-token: write`를 명시적으로 허용하는 것입니다. 이 권한 없이는 GitHub가 해당 Job에 대한 OIDC Token을 발행하지 않습니다.

```yaml
# .github/workflows/deploy.yml
name: Deploy to AWS

on:
  push:
    branches: [main]

# OIDC Token 발행에 필요한 권한 명시
permissions:
  id-token: write  # Job이 GitHub OIDC Token을 요청할 수 있도록 허용
  contents: read   # actions/checkout에 필요

jobs:
  deploy:
    runs-on: ubuntu-latest
    # GitHub Environments와 조합하면 수동 승인 Gate를 추가할 수 있음
    environment: production

    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Configure AWS credentials via OIDC
        uses: aws-actions/configure-aws-credentials@v4
        with:
          role-to-assume: arn:aws:iam::123456789012:role/github-actions-deploy
          aws-region: ap-northeast-1
          # run_id를 포함하면 CloudTrail에서 Session 추적이 쉬워짐
          role-session-name: deploy-${{ github.run_id }}

      # Role Assumption이 올바르게 동작했는지 확인 (디버깅 시 유용)
      - name: Verify AWS identity
        run: aws sts get-caller-identity

      - name: Build
        run: npm ci && npm run build

      - name: Sync to S3
        run: |
          # 버전이 지정된 Asset은 장기 Cache, HTML은 Cache 없음
          aws s3 sync ./dist s3://my-deployment-bucket \
            --delete \
            --exclude "*.html" \
            --cache-control "public, max-age=31536000, immutable"
          aws s3 sync ./dist s3://my-deployment-bucket \
            --exclude "*" \
            --include "*.html" \
            --cache-control "no-cache, no-store, must-revalidate"

      - name: Invalidate CloudFront cache
        run: |
          aws cloudfront create-invalidation \
            --distribution-id ${{ secrets.CLOUDFRONT_DISTRIBUTION_ID }} \
            --paths "/*"
```

`aws-actions/configure-aws-credentials@v4`가 OIDC Token 요청과 STS 호출을 내부적으로 처리합니다. Workflow에서는 `role-to-assume`에 Role ARN만 지정하면 됩니다.

## Branch/Repository 제한 Trust Policy

`sub` Claim 조건은 가장 중요한 접근 제어 지점입니다. 상황에 따른 패턴을 정리합니다.

```hcl
# 패턴 1: 특정 Repository의 main Branch만 허용 (Production 용도)
Condition = {
  StringEquals = {
    "token.actions.githubusercontent.com:sub" = "repo:your-org/your-repo:ref:refs/heads/main"
  }
}

# 패턴 2: 특정 Repository의 모든 Branch 허용 (Staging/Dev 용도)
Condition = {
  StringLike = {
    "token.actions.githubusercontent.com:sub" = "repo:your-org/your-repo:*"
  }
}

# 패턴 3: GitHub Environment 기반 제한
# Workflow에 environment: production이 선언된 경우만 허용
Condition = {
  StringEquals = {
    "token.actions.githubusercontent.com:sub" = "repo:your-org/your-repo:environment:production"
  }
}

# 패턴 4: 복수 Repository 허용 (배열로 지정)
Condition = {
  StringEquals = {
    "token.actions.githubusercontent.com:sub" = [
      "repo:your-org/app-repo:ref:refs/heads/main",
      "repo:your-org/infra-repo:ref:refs/heads/main"
    ]
  }
}
```

`StringEquals`와 `StringLike`의 차이에 주의하세요. `StringLike`는 와일드카드(`*`)를 지원하지만, 잘못 사용하면 의도보다 넓은 권한이 생깁니다. Production 환경 Role에는 가능하면 `StringEquals`로 정확히 지정하는 것을 권장합니다.

`"repo:your-org/*"` 같은 조건은 절대 Production Role에 사용하지 않아야 합니다. 조직 내 어떤 Repository에서든 해당 Role을 assume할 수 있게 되기 때문입니다.

## 다중 Environment 구성

Production, Staging, Development 환경을 분리해 운영하는 경우, 환경별로 별도의 Role을 생성하고 각 Role의 Trust Policy를 해당 Branch로 제한하는 것이 권장 패턴입니다.

```hcl
locals {
  deploy_environments = {
    production = {
      branch = "main"
      bucket = "my-app-prod"
    }
    staging = {
      branch  = "develop"
      bucket  = "my-app-staging"
    }
  }
}

resource "aws_iam_role" "github_actions" {
  for_each = local.deploy_environments

  name        = "github-actions-${each.key}"
  description = "${each.key} 배포 Role (OIDC 인증)"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { Federated = aws_iam_openid_connect_provider.github.arn }
      Action    = "sts:AssumeRoleWithWebIdentity"
      Condition = {
        StringEquals = {
          "token.actions.githubusercontent.com:aud" = "sts.amazonaws.com"
          "token.actions.githubusercontent.com:sub" = "repo:your-org/your-repo:ref:refs/heads/${each.value.branch}"
        }
      }
    }]
  })
}
```

Workflow에서는 GitHub Environment Variable(Secret이 아닌 일반 Variable)에 Role ARN을 저장해 참조할 수 있습니다.

```yaml
jobs:
  deploy:
    environment: production
    steps:
      - uses: aws-actions/configure-aws-credentials@v4
        with:
          # vars는 Secret이 아닌 일반 Variable. ARN은 민감 정보가 아니므로 적절
          role-to-assume: ${{ vars.AWS_DEPLOY_ROLE_ARN }}
          aws-region: ap-northeast-1
```

## CloudTrail 감사 로그 품질 향상

OIDC 방식의 부수적 이점 중 하나는 CloudTrail 로그 품질입니다. `AssumeRoleWithWebIdentity` 호출마다 Token의 `sub` Claim이 CloudTrail에 기록됩니다.

```json
{
  "eventName": "AssumeRoleWithWebIdentity",
  "requestParameters": {
    "roleArn": "arn:aws:iam::123456789012:role/github-actions-deploy",
    "roleSessionName": "deploy-9876543210"
  },
  "userIdentity": {
    "sessionContext": {
      "webIdFederationData": {
        "federatedProvider": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com",
        "attributes": {
          "sub": "repo:your-org/your-repo:ref:refs/heads/main"
        }
      }
    }
  }
}
```

이 정보를 활용하면 "지난달 S3 Bucket에 접근한 Workflow Run이 어떤 Branch에서 트리거됐는가"를 별도의 CI 로그 없이 CloudTrail에서 직접 조회할 수 있습니다.

## 마무리

OIDC 인증 도입은 설정 작업이 다소 늘어나지만, 유지 관리 부담은 오히려 줄어듭니다.

- **보관해야 할 장기 Secret이 없습니다.** GitHub Secrets에서 IAM Key를 삭제할 수 있습니다.
- **임시 자격증명은 자동 만료됩니다.** Rotation 작업이 필요 없습니다.
- **Branch와 Environment 단위로 권한을 분리할 수 있습니다.** Production Role은 main Branch에서만 assume 가능합니다.
- **CloudTrail 감사 로그 품질이 향상됩니다.** 어떤 Workflow Run이 어떤 리소스에 접근했는지 추적하기 쉬워집니다.

기존 프로젝트를 마이그레이션하는 경우 권장 순서는 다음과 같습니다. 먼저 OIDC Provider와 새 Role을 생성합니다. 별도 Branch에서 새 Workflow를 테스트하고 `aws sts get-caller-identity`로 올바른 Role이 반환되는지 확인합니다. Production에서 동작을 검증한 후 기존 Secret을 삭제합니다. 이 순서로 진행하면 중단 없이 안전하게 전환할 수 있습니다.

장기 자격증명을 없애는 것 자체가 목적이 아닙니다. 새로운 서비스나 Role을 추가할 때마다 "이 권한이 실제로 필요한가"를 묻는 습관과 함께할 때 의미가 있습니다. OIDC는 그 습관을 지속하기 쉽게 만드는 기반 구조입니다.
