GitHub Actions에서 Cloud로 Keyless 인증하는 구조는 이제 표준 구성입니다. 장기 Access Key를 CI에 두는 대신, GitHub가 발급하는 OIDC Token을 Cloud 측에서 검증해 일시적인 권한을 넘겨줍니다. 이 OIDC Token의 `sub`(subject) Claim 형식이 2026년 7월 15일을 기점으로 바뀝니다. 이 글에서는 무엇이 바뀌고, 어떤 Trust Policy에 영향을 주는지 정리합니다.

## 무엇이 바뀌는가

지금까지 OIDC Token의 `sub` Claim은 Repository 경로와 Branch 이름 등으로 구성되어 있었습니다. 예를 들어 다음과 같은 형태입니다.

```text
repo:octocat/my-repo:ref:refs/heads/main
```

새 형식에서는 여기에 Immutable한 Owner ID와 Repository ID가 부여됩니다.

```text
repo:octocat@123456/my-repo@456789:ref:refs/heads/main
```

`@123456`이 Owner의 Immutable ID, `@456789`가 Repository의 Immutable ID입니다. 이들은 숫자 내부 ID이며 이름을 변경해도 불변입니다. 2026년 7월 15일 이후 생성되는 모든 Repository에서 이 새 형식이 Default가 됩니다(출처: GitHub Changelog).

## 왜 Immutable ID가 필요한가

기존 이름 기반 `sub`에는 사칭 Risk가 있었습니다. 조직이나 Repository 이름은 변경할 수 있으므로 다음과 같은 공격 경로가 이론적으로 성립합니다. 어떤 Repository를 삭제하고 같은 이름으로 다른 Repository를 다시 만들면, 이름 기반 Claim은 같은 문자열이 됩니다. Trust Policy가 이름만으로 판정하면, 다시 만들어진 별개의 Repository가 원래 권한을 이어받을 여지가 있었습니다.

Immutable ID를 부여하면 Claim이 원래 Repository에 영구적으로 결합됩니다. 이름을 재사용해도 내부 ID는 일치하지 않으므로, 삭제·재생성에 의한 탈취를 막을 수 있습니다. 이는 Supply Chain 공격 대책의 일환이며, GitHub가 2025년부터 2026년에 걸쳐 진행하고 있는 일련의 강화 중 하나입니다.

## 영향을 받는 것은 Trust Policy 쪽

문제는 Cloud 측에서 `sub` Claim을 문자열 Match로 검증하고 있는 설정입니다. 새 형식에서는 문자열이 바뀌므로, 기존 형식으로 Match하고 있는 Trust Policy는 신규 Repository로부터의 인증을 막게 됩니다.

AWS의 IAM Role을 예로 들겠습니다. 기존 Trust Policy는 다음과 같이 작성되어 있는 경우가 많을 것입니다.

```json
{
  "Effect": "Allow",
  "Principal": {
    "Federated": "arn:aws:iam::123456789012:oidc-provider/token.actions.githubusercontent.com"
  },
  "Action": "sts:AssumeRoleWithWebIdentity",
  "Condition": {
    "StringEquals": {
      "token.actions.githubusercontent.com:aud": "sts.amazonaws.com"
    },
    "StringLike": {
      "token.actions.githubusercontent.com:sub": "repo:octocat/my-repo:ref:refs/heads/*"
    }
  }
}
```

이 `sub`의 `StringLike` 패턴은 기존 형식의 문자열을 전제로 합니다. 새 형식에서는 `octocat@123456`처럼 ID가 끼어들기 때문에 이 패턴에는 일치하지 않습니다. 신규 Repository를 만들어 이 Role을 사용하려 하면 AssumeRole이 실패합니다.

## Migration 방법

대응 방향은 두 가지입니다.

첫 번째는 Immutable ID를 포함한 새 형식에 맞춰 패턴을 갱신하는 방법입니다. 더 엄격한 Matching이 되며, 이름 재사용에 의한 탈취를 막을 수 있습니다.

```json
"StringLike": {
  "token.actions.githubusercontent.com:sub": "repo:octocat@123456/my-repo@456789:ref:refs/heads/*"
}
```

두 번째는 `sub` 대신 개별 Claim(`repository_id`나 `repository_owner_id`)으로 검증하는 방법입니다. OIDC Token에는 이들이 독립된 Claim으로 포함되어 있으므로, `sub`의 형식 변경에 의존하지 않는 설계로 만들 수 있습니다.

```json
"StringEquals": {
  "token.actions.githubusercontent.com:repository_id": "456789"
}
```

`repository_id`는 이름 변경에 대해 불변이므로, 애초에 이름 기반 Match를 그만두는 것이 가장 견고합니다. 기존 Trust Policy가 `sub`의 문자열 패턴에 의존하고 있다면, 이 기회에 `repository_id` 기반으로 Migration하는 것을 검토할 가치가 있습니다.

## 기존 Repository는 어떻게 되는가

이 변경은 "2026년 7월 15일 이후 생성되는 Repository"가 대상입니다. 그 이전부터 존재하는 Repository의 `sub` 형식이 즉시 바뀌는 것은 아닙니다. 다만 GitHub는 Immutable 식별자로의 이행을 단계적으로 진행하고 있으므로, 기존 Repository도 장래적으로 새 형식으로 옮겨 갈 방향으로 보는 것이 자연스럽습니다.

따라서 실무상 권장은 다음과 같습니다. 지금 당장 모든 Repository의 Trust Policy를 다시 쓸 필요는 없지만, 신규 Repository를 만드는 운영이 있는 조직에서는 7월 15일 이후 "왜인지 새 Repository만 CI의 Cloud 인증이 통과하지 않는" 문제가 생길 수 있습니다. 이 증상을 미리 알아 두는 것만으로도 원인 분리에 걸리는 시간이 크게 달라집니다.

## webhani의 관점 — ID 기반 검증으로 옮기기

저희가 고객의 Cloud 인증을 설계·Review할 때, OIDC Trust Policy는 가능한 한 Immutable ID로 구성하기를 권장합니다. 이름 기반 Match는 읽기 쉬운 반면, 이름이라는 가변 속성에 안전성을 맡기게 됩니다. 이번 GitHub 측 변경은 바로 그 약점을 막기 위한 것입니다.

구체적인 Action은 세 단계입니다.

1. 기존 Trust Policy를 점검하고, `sub`의 문자열 패턴에 의존하는 부분을 식별한다.
2. 가능한 것부터 `repository_id` / `repository_owner_id` 기반 검증으로 Migration한다.
3. 신규 Repository를 만드는 운영이 있는 팀에는 7월 15일 이후의 `sub` 형식 변경을 공지한다.

## 정리

GitHub Actions OIDC의 Immutable subject claim은 이름 재사용에 의한 사칭을 막기 위한 Security 강화입니다. 2026년 7월 15일 이후 생성되는 Repository에서는 `sub`가 새 형식이 되며, 기존 형식으로 문자열 Match하는 Trust Policy는 신규 Repository로부터의 인증을 막습니다. 가장 견고한 대응은 `sub`의 문자열 Match를 그만두고 `repository_id` 같은 Immutable Claim으로 검증하는 것입니다. 지금 미리 Trust Policy를 점검하고 ID 기반으로 옮겨 두는 것이 안전합니다.
