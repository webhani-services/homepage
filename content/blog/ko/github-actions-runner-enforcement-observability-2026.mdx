GitHub가 2026년에 추진하는 Actions 보안 강화가, Self-hosted Runner를 쓰는 팀에게는 무시할 수 없는 마감일을 동반하며 움직이기 시작했습니다. GitHub Enterprise Cloud(Data Residency 이용)에서는 Runner 버전 강제의 brownout이 6월 29일부터 시작되어 7월 31일에 완전 강제로 전환됩니다([The GitHub Blog](https://github.blog/news-insights/product-news/whats-coming-to-our-github-actions-2026-security-roadmap/)). 이 글에서는 이 마감일의 의미와, 함께 짚어 둘 관측 기능을 운영 관점으로 정리합니다.

## 무엇이 강제되는가

완전 강제 이후, 최소 필요 버전을 밑도는 Self-hosted Runner는 신규 등록·재등록이 불가능해지고, Job 실행에 필요한 최소 버전을 밑도는 기존 Runner는 Job을 받지 않게 됩니다. 즉 방치해 둔 오래된 Runner가 "어느 날 갑자기 Job을 집지 않는" 상태가 발생할 수 있습니다.

Supply Chain 보호 관점에서는 타당한 변경입니다. 오래된 Runner는 Patch가 적용되지 않은 취약점을 안고 있기 쉽고, CI는 Source Code와 인증 정보에 접근하는 가장 민감한 경로 중 하나이기 때문입니다. 다만 운영 측에서는 마감일 전에 재고 조사를 해 두지 않으면 CI가 멈추는 위험을 안게 됩니다.

## 마감일 전에 해야 할 일

먼저 자사 Runner의 버전을 파악합니다. Label이나 등록 상태는 API로도 확인할 수 있지만, 현장에서 효과적인 것은 각 Runner Host 위에서의 버전 확인입니다.

```bash
# 각 Runner Host에서 버전 확인
cat /opt/actions-runner/.runner | grep -i version 2>/dev/null \
  || ./run.sh --version
```

자동 업데이트를 켜 둔 Runner는 기본적으로 따라가지만, Firewall이나 Proxy로 외부 통신을 조인 환경에서는 자동 업데이트가 실패하고 있는 경우가 있습니다. 다음과 같은 재고 조사 Script로 관리 대상 Host를 일람화하여 업데이트 상태를 확인해 두면 안전합니다.

```bash
#!/usr/bin/env bash
# 관리 중인 Runner Host를 순회하며 버전 집계
set -euo pipefail

HOSTS=(runner-a runner-b runner-c)
for host in "${HOSTS[@]}"; do
  version=$(ssh "$host" 'cd /opt/actions-runner && ./run.sh --version' 2>/dev/null || echo "UNREACHABLE")
  printf '%-12s %s\n' "$host" "$version"
done
```

자동 업데이트가 멈춰 있다면 Runner Package를 최신 버전으로 다시 받아 교체합니다. Container나 IaC로 Runner를 관리하고 있다면, Base Image 버전을 올려 재배포하는 것이 더 깔끔한 방법입니다.

## 함께 들어오는 관측 기능: Actions Data Stream

이번 Roadmap에서는 강제뿐 아니라 CI/CD를 "운영 System과 똑같이 관측"하기 위한 구조도 함께 진행되고 있습니다. Actions Data Stream은 Workflow 실행 데이터를 외부로 흘려 분석할 수 있게 해 주는 것으로, 지금까지 Black Box가 되기 쉬웠던 CI 상태를 다른 운영계와 같은 관측 가능성의 무대에 올립니다.

실무에서의 활용처는 명확합니다. 예를 들어 Job의 소요 시간이나 Queue 대기 시간을 시계열로 집계하면, "요즘 CI가 느리다"는 체감을 수치로 뒷받침할 수 있습니다. Runner의 가동률이나 실패율을 지속적으로 추적하면, 이번 버전 강제 같은 변경이 Job 성공률에 준 영향도 추측이 아니라 데이터로 확인할 수 있습니다.

```yaml
# 관측 가능성의 전 단계로, Job에 타이밍 정보를 남기는 예
- name: Mark job timing
  if: always()
  run: |
    echo "job=${{ github.job }} status=${{ job.status }} \
      run_id=${{ github.run_id }} ts=$(date -u +%FT%TZ)" >> "$GITHUB_STEP_SUMMARY"
```

외부 Stream 도입을 기다리지 않더라도, 우선 Step Summary나 Artifact에 실행 Metadata를 남겨 두면 나중에 경향을 추적하기 쉬워집니다.

## webhani의 관점

CI/CD는 "동작하는 동안에는 아무도 보지 않는" Infrastructure의 전형입니다. 이번 버전 강제는 그 방치되기 쉬운 영역에 외부에서 마감일을 끊어 들어오는 변경이며, 대응 자체는 어렵지 않지만 대상의 재고 조사를 미루면 7월 말에 CI가 멈추는 형태로 표면화됩니다.

저희가 고객의 CI 환경을 볼 때 권하는 것은 2단 구성의 정리입니다. 단기적으로는 마감일 전에 Self-hosted Runner의 버전을 재고 조사하고, 자동 업데이트가 실제로 기능하는지 확인하는 것. 중기적으로는 Actions Data Stream 같은 구조로 CI의 Metric을 지속적으로 관측하여, "망가지고 나서 알아차리는" 운영에서 "징후 단계에서 알아차리는" 운영으로 옮기는 것입니다. 보안 강화의 물결은 앞으로도 계속된다는 전제로, 관측 가능한 CI를 먼저 만들어 두는 것이 이런 마감일 동반 변경에 휘둘리지 않기 위한 가장 좋은 대비입니다.
