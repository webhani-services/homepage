GitHub는 2026년 7월 1일, GitHub에 통합해 제공하던 Model Catalog·Playground·Inference API·BYOK(Bring Your Own Key) 기능인 GitHub Models를 7월 30일 완전히 종료한다고 발표했습니다. 이미 7월 16일과 23일에 예고된 Brownout(일시 중단)이 진행되어, 종료 이후의 상황을 미리 체감한 Team도 있을 겁니다. 30일이 지나면 어떤 Customer에게도 복구되지 않으며, 이는 Production에서 실사용 중인 경우도 예외가 아닙니다.

"무료 Tier인 데다 별도 Billing Account 없이 기존 PAT를 그대로 쓸 수 있어서" 라는 이유만으로 CI 일부나 Prototype, 사내 Tool을 GitHub Models에 연결해둔 Team이라면 이 체크리스트를 확인해야 합니다.

## 실제로 무엇이 깨지는가

GitHub Models가 제공하던 것은 크게 세 가지입니다.

1. GitHub Personal Access Token으로 직접 호출하는 **Inference API** (`models.github.ai` / `models.inference.ai.azure.com` Endpoint)
2. 별도 Provider Account 없이 빠르게 Prompt를 테스트할 수 있는 **Playground**
3. Request는 GitHub Endpoint로 가지만 과금은 직접 제공한 Key로 이루어지는 **BYOK Routing**

이는 GitHub Copilot과는 별개의 제품입니다. Copilot의 IDE 연동, Chat, Agent Mode는 이번 종료의 영향을 받지 않습니다. 이번에 종료되는 것은 GitHub가 Platform 위에 별도로 얹어둔 독립적인 Model Catalog와 API에 한정됩니다. 연동 지점이 Copilot뿐이었다면 별도로 할 일은 없습니다.

## Step 1: 의존하는 곳을 전부 찾는다

대체재를 정하기 전에, 지금 실제로 무엇이 이에 의존하고 있는지부터 파악합니다.

```bash
# 전체 Repository에서 GitHub Models Endpoint 검색
grep -rn "models.github.ai\|models.inference.ai.azure.com" .
grep -rln "GITHUB_MODELS\|GH_MODELS_TOKEN" .

# GitHub Actions Workflow는 별도로 확인
grep -rn "models.github.ai" .github/workflows/
```

놓치기 쉬운 곳은 PR 요약을 자동 생성하는 GitHub Actions Step, 해커톤용으로 누군가 급하게 만든 Next.js API Route, `scripts/` 아래에서 릴리스 노트를 생성하는 스크립트 등입니다. GitHub Models는 "무료에 이미 인증까지 되어 있다"는 이유로 이런 눈에 잘 안 띄는 곳에서 기본값처럼 쓰이는 경우가 많으므로, 전부 점검할 가치가 있습니다.

## Step 2: 대체재를 정한다

Microsoft의 공식 안내는 **Azure AI Foundry**로의 이전을 권장합니다. 이미 Azure/GitHub Ecosystem 안에서 운영 중이고 Billing과 Quota를 한곳에서 관리하고 싶다면 가장 유사한 대체재입니다. 다만 GitHub PAT가 아닌 Entra ID나 API Key를 이용한 별도 인증과 독립된 Azure Subscription이 필요하므로, Model Catalog 자체는 비슷해도 Setup에는 시간이 어느 정도 필요합니다.

기존에 Azure에 종속되어 있지 않은 대부분의 Team에게는 Model Provider에 직접 연결하는 편이 더 단순하며, 중간 단계 없이 가격 정책과 Rate Limit을 스스로 관리할 수 있습니다.

```typescript
// Before: GitHub Models Inference Endpoint
const response = await fetch(
  "https://models.github.ai/inference/chat/completions",
  {
    method: "POST",
    headers: {
      Authorization: `Bearer ${process.env.GITHUB_TOKEN}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      model: "claude-sonnet-5",
      messages: [{ role: "user", content: prompt }],
    }),
  }
);

// After: Anthropic API 직접 호출
import Anthropic from "@anthropic-ai/sdk";

const anthropic = new Anthropic({ apiKey: process.env.ANTHROPIC_API_KEY });

const message = await anthropic.messages.create({
  model: "claude-sonnet-5",
  max_tokens: 1024,
  messages: [{ role: "user", content: prompt }],
});
```

이미 Vercel AI SDK를 사용 중인 Project라면 변경 범위는 더 작아집니다. `generateText()`, `streamText()` 호출부는 그대로 두고 Provider 문자열만 바꾸면 됩니다.

```typescript
import { generateText } from "ai";

// 지금까지는 자체 fetch Wrapper를 통해 GitHub Models로 Routing
// 이제는 AI SDK를 통해 Provider에 직접 연결
const { text } = await generateText({
  model: "anthropic/claude-sonnet-5",
  prompt,
});
```

## Step 3: 30일이 되기 전에 CI부터 고친다

가장 위험한 지점은 `.github/workflows/*.yml` 안의 호출부입니다. 여기가 깨지면 특정 Feature가 아니라 Merge Queue 전체가 조용히 멈춥니다. Secret 이름을 갱신하고(`GITHUB_TOKEN`은 다른 Provider 인증에 쓸 수 없습니다) 새 Provider의 API Key를 Repository 또는 Organization Secret으로 추가한 뒤, 전환일 전에 테스트용 PR로 Workflow를 한 번 돌려보시기 바랍니다. 7월 30일 실제 중단이 벌어지고 나서야 필수 Status Check가 더 이상 응답하지 않는 Endpoint에 의존하고 있었다는 걸 알게 되는 상황은 피해야 합니다.

## 더 큰 교훈

이번 종료는 완만한 Deprecation이 아니라, 6월에 신규 가입 중단을 발표하고 7월 말까지 전체 Customer에 대한 제공을 끝내는 상당히 짧은 일정이었습니다. 무료이면서 명확히 Preview로 명시된 제품 기준으로는 합리적인 기간이지만, Vendor가 함께 묶어 제공하는 Bundled·Promotional한 AI 기능에 의존할 때 참고할 만한 교훈입니다. 자사 Pipeline의 어떤 부분이 스스로 Roadmap을 통제할 수 없는 Catalog 위에 얹혀 있는지 파악하고, Production에서 실제로 중요한 처리에는 Provider 직접 연동이나 Vercel AI Gateway 같은 Routing Layer를 우선하는 편이 좋습니다. 마이그레이션 작업 자체는 보통 몇 시간이면 끝나지만, 그 사실을 Deploy 장애 도중에 처음 알게 되는 것은 피해야 합니다.
