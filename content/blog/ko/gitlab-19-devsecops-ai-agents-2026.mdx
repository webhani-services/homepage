2026년 5월 22일 GitLab 19.0이 출시됐습니다. 이번 Release의 핵심은 DevSecOps의 완전한 통합입니다. 보안 스캔, 컴플라이언스 검사, AI Coding이 하나의 Platform에서 분절 없이 연동되는 Orchestration 기능이 크게 강화됐습니다.

또한 AI Coding이 보편화되면서 피할 수 없어진 문제 — Coding Agent에게 Secret을 노출하는 문제 — 에 대한 실용적인 답변이 포함된 것도 주목할 만합니다.

## DevSecOps를 First-class Pipeline 단계로

기존 GitLab에서는 SAST, Dependency Scanning, Container Scanning을 추가하려면 개발자가 직접 Template을 포함하고 Job 설정을 연결해야 했습니다. GitLab 19.0부터는 `security:` 블록으로 Project 레벨에서 선언적으로 관리할 수 있습니다.

```yaml
# .gitlab-ci.yml (GitLab 19.0)
include:
  - template: Security/SAST.gitlab-ci.yml
  - template: Security/Dependency-Scanning.gitlab-ci.yml

security:
  policy:
    require_sast: true
    block_on_critical: true      # Critical 발견 시 Merge 차단
    auto_fix_minor: true         # Minor는 자동 Patch 제안
  scan_on:
    - merge_request
    - default_branch

stages:
  - build
  - test
  - security    # GitLab이 자동 삽입
  - deploy
```

`block_on_critical: true`를 설정하면 Critical 취약점이 해결되지 않은 MR은 Merge할 수 없습니다. 여러 위치에서 설정을 분산 관리할 필요 없이 한 곳에 선언적으로 작성하면 됩니다.

## Agent Vault: Agent가 읽을 수 없는 Secret

더 새로운 기능은 Agent Vault입니다. AI Coding Agent가 보편화되면서 Agent가 코드를 Push하고 API를 호출하고 Deploy하려면 Credential이 필요해집니다. 문제는 Secret을 Agent에게 직접 전달하면 LLM Context에 평문으로 포함될 수 있다는 점입니다.

Agent Vault는 GitLab의 기존 CI/CD Secret 관리를 Agent용으로 확장한 것입니다. Agent는 실제 Secret 값을 받지 않고 Vault를 통해 위임된 권한으로만 작업합니다.

```yaml
# .gitlab-ci.yml — Agent Vault 설정
coding_agent:
  stage: implement
  image: registry.gitlab.com/gitlab-org/coding-agent:latest
  variables:
    GITLAB_AGENT_VAULT_ENABLED: "true"
    GITLAB_AGENT_PERMISSIONS: "read_repository,create_mr"
  secrets:
    AWS_CREDENTIALS:
      vault: aws/creds/deploy-role   # HashiCorp Vault 경유 위임
      token: $VAULT_TOKEN
  script:
    - gitlab-agent run --task="$TASK_DESCRIPTION"
```

Agent는 코드 읽기, 변경, MR 생성 작업을 완료하지만 `AWS_CREDENTIALS`의 실제 Access Key나 Secret Key 값은 Environment Variable로 받지 않습니다. Vault가 Agent를 대신해 인증을 처리합니다. LLM Context에 Secret이 섞이는 위험을 구조적으로 차단합니다.

## Security Policy as Code 실용 예시

19.0에서 추가된 Security Policy를 코드 출처에 따라 다르게 적용하는 예시입니다.

```yaml
# security-policy.yml (프로젝트 루트에 배치)
version: "2"
scan_execution_policy:
  - name: "Protected Branch Scan"
    description: "main 또는 release 브랜치를 타겟으로 하는 MR에 풀 스캔 실행"
    enabled: true
    rules:
      - type: pipeline
        branches:
          - main
          - release/*
    actions:
      - scan: sast
        variables:
          SAST_EXCLUDED_PATHS: "spec,test,docs"
      - scan: container_scanning
      - scan: dependency_scanning

  - name: "AI Agent Code Review"
    description: "Coding Agent가 작성한 Commit에 추가 스캔 적용"
    enabled: true
    rules:
      - type: merge_request
        branch_type: protected
        commits:
          author_email_regex: ".*agent.*@gitlab\\.com"
    actions:
      - scan: sast
        severity_levels:
          - critical
          - high
```

두 번째 Policy는 Author 이메일이 Agent 패턴과 일치하는 Commit을 타겟으로 합니다. AI 생성 코드는 Protected Branch에 도달하기 전에 추가 Security 검사를 통과해야 합니다. Reviewer가 수동으로 AI 작성 여부를 확인할 필요 없이 자동으로 처리됩니다.

## 19.0 업그레이드 시 주의사항

### Template 경로 변경

```yaml
# ❌ 19.0에서 동작하지 않음
include:
  - template: SAST.gitlab-ci.yml

# ✅ 올바른 경로
include:
  - template: Security/SAST.gitlab-ci.yml
```

### only/except 지원 중단

```yaml
# ❌ 19.0에서 경고 발생
deploy:
  script: ./deploy.sh
  only:
    - main

# ✅ rules로 통일
deploy:
  script: ./deploy.sh
  rules:
    - if: $CI_COMMIT_BRANCH == "main"
```

### Runner 버전 요구사항

Agent Vault 기능을 사용하려면 GitLab Runner 17.5 이상이 필요합니다. Self-hosted Runner를 사용 중이라면 미리 버전을 확인하세요.

```bash
# Runner 버전 확인
gitlab-runner --version

# Docker 기반 Runner 업데이트
docker pull gitlab/gitlab-runner:latest
docker stop gitlab-runner && docker rm gitlab-runner
docker run -d --name gitlab-runner \
  --restart always \
  -v /srv/gitlab-runner/config:/etc/gitlab-runner \
  -v /var/run/docker.sock:/var/run/docker.sock \
  gitlab/gitlab-runner:latest
```

## webhani가 주목하는 부분

GitLab 19.0의 변경 중에서 AI Coding 도구를 활발히 사용하는 팀에 즉각적인 영향이 있는 것은 두 가지입니다.

**Security Policy as Code**: 보안 요건을 코드로 관리하게 되어 여러 Repository 간 보안 기준을 표준화하기 쉬워졌습니다. 특히 AI 생성 코드와 인간 작성 코드의 스캔 기준을 분리 관리할 수 있는 것은 AI Coding 비중이 늘어나는 팀에 직접적으로 유용합니다.

**Agent Vault**: AI Agent에게 Secret을 전달하지 않고 작동시키는 구조는, CI/CD Pipeline에서 Coding Agent를 본격적으로 사용하는 팀에게 단기적으로 적용해야 할 우선순위 높은 기능입니다. 기존 방식이 치명적이었다기보다, LLM Context에서 Credential 노출을 구조적으로 없애는 것이 명백히 더 나은 접근이기 때문입니다.

GitLab 19.0은 기능 추가보다 "통합의 완성도"에 집중한 Release입니다. 기존 DevSecOps 설정이 정리된 팀이라면 비교적 원활하게 업그레이드할 수 있을 것입니다.
