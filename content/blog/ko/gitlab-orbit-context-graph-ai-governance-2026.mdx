2026년 6월 10일, GitLab은 Transcend 행사에서 Agent 주도 Delivery를 축으로 재설계한 Platform을 공개했습니다. 중요한 것은 새 모델이 아니라 Infrastructure입니다. Context Graph인 Orbit, AI Governance Framework, 그리고 재설계된 Source Code 계층. 공통 주제는 분명히 짚을 가치가 있습니다. Coding Agent의 Bottleneck이 추론에서 **Context와 통제**로 옮겨갔다는 것입니다.

## Coding Agent의 진짜 실패 원인은 'Context 부족'

유능한 모델도 전체 그림을 보지 못하면 틀린 답을 냅니다. Bug를 고치는 Agent는 가장 최근에 실패한 Pipeline, 의도한 동작을 기술한 Work Item, 그리고 Production에서 가동 중인 Deployment를 알아야 합니다. 이를 흩어진 도구에서 짜맞추면 익숙한 문제가 생깁니다. 부분적인 정보 위에 자신만만한 출력이 얹히는 것입니다.

GitLab의 답인 Orbit(Public Beta)은 Code, Work Item, Pipeline, Deployment, Production Signal을 Software Lifecycle 전체를 아우르는 단일 Context Graph로 매핑합니다. 엔지니어도 Agent도 이를 유일한 진실로 조회합니다. 보고된 수치는 인상적입니다. Orbit을 쓰는 Agent는 응답이 최대 11배 빠르고, Token 소비가 최대 4.5분의 1, Hallucination이 최대 45분의 1이었다고 합니다.

Vendor Benchmark는 건강한 회의로 봐야 합니다. 그러나 **방향성**은 저희가 현장에서 보는 것과 일치합니다. 저희가 Debug하는 Agent 실패의 다수는 추론 실패가 아니라 Context 실패입니다. "이 시스템에 대해 지금 무엇이 참인가"를 해결하는 Graph는 바로 그 본질적인 문제를 겨냥합니다.

```text
Context Graph가 없을 때:
  Agent → 열린 File에서 추측 → 증상에 Patch

Orbit 식 Context가 있을 때:
  Agent → Graph에 조회: 실패한 Pipeline #4821,
        Issue #233에 연결됨(기대: 멱등 Retry), 마지막 Deploy = v2.7.1
        → 의도한 동작에 맞춰 근본 원인에 Patch
```

## Governance: Agent에게 필요한 것은 Access만이 아니라 '신원'

두 번째는 Agent를 도입하는 모든 팀이 결국 마주치는 질문에 대한 답입니다. **누가 무엇을 했고, 그것이 허가되었는가?** GitLab의 AI Governance Framework(Private Beta)는 Agent의 각 Action에 신원, Policy 경로, 감사 기록을 부여합니다. DevSecOps 팀은 Agent의 입력, 추론, Tool 호출, 이상 패턴을 실시간으로 가시화할 수 있습니다.

이는 Agent를 규모 있게 운영하기 위한, 화려하지 않지만 필수인 전제입니다. 자율 Process가 Merge Request를 열고 Pipeline을 건드리고 Deployment를 trigger할 수 있게 된 순간, "AI가 했다"는 감사 답변이 될 수 없습니다. 모든 Action에는 실행 주체, 허용된 범위, 추적 가능한 기록이 필요합니다. 저희는 지난 1년간 고객에게 이렇게 전해왔습니다. Agent에게는 입사 첫날의 Junior 계약자에게 줄 이상의 권한을 주지 말라고. Action마다 신원과 Policy를 강제하는 구조는 그 원칙을 이상론에서 운영으로 바꿉니다.

## 왜 Source 계층을 다시 만들었는가

세 번째 발표는 놓치기 쉽습니다. 분산 Architecture 기반의 차세대 Source Code Management 구현으로, GitLab은 Agent의 Task 완료를 최대 50배 빠르게 한다고 말합니다(Private Beta). 논리는 단순합니다. 사람이 Git을 다루던 시절에는 작업당 Latency가 사람의 사고 시간에 묻혔습니다. Agent가 분당 수백 건의 작업을 수행하게 되면, 토대가 되는 Git 계층이 Bottleneck이 됩니다. Protocol을 Machine 속도의 Access에 맞게 다시 만드는 것은 Agent 규모의 Delivery가 요구하는, 화려하지 않은 Infrastructure 작업입니다.

## GitLab을 도입하지 않아도 의미가 있는 이유

개별 제품보다, 그것들이 뒷받침하는 패턴이 중요합니다. AI Agent를 Delivery Process에 통합한다면, 다음 세 가지는 타협 불가가 되어가고 있습니다.

1. **통합된 Context 계층.** Agent는 단편적인 정보 위에서 추론을 잘 못합니다. Vendor Graph든 자체 통합이든, "이 시스템에 대해 무엇이 참인가"에 대한 단일하고 조회 가능한 답을 주십시오.
2. **Action 단위의 신원과 감사.** Agent의 Action에는 사람과 동일한 책임성이 필요합니다. 첫 Incident 이후가 아니라, 쓰기 권한을 주기 **전에** 감사 추적을 설계하십시오.
3. **Machine 속도를 위한 Infrastructure.** 사람의 Cadence에 최적화된 도구는 Agent 부하 아래에서 Bottleneck이 됩니다. 사람만이 운영자였던 시절의 전제를 다시 볼 각오를 하십시오.

## webhani의 권장

Agent 주도 Delivery를 시범 운영하는 팀에는 의도적인 순서를 권합니다. 먼저, 가벼운 것이라도 감사·신원 계층을 확립해 첫날부터 모든 Agent Action을 귀속 가능하게 만드십시오. 다음으로, Agent의 자율성을 넓히기 전에 Context 통합에 투자하십시오. 넓은 권한과 빈약한 Context를 가진 Agent는 양쪽의 단점만 모은 것입니다. 그런 다음에야 Agent가 무인으로 해도 되는 범위를 넓히십시오.

Transcend의 교훈은 Agentic Engineering의 최전선이 더 이상 모델 품질 단독이 아니라는 것입니다. 핵심은 그것을 둘러싼 시스템 — 공유 Context, 강제되는 Governance, 따라잡는 Infrastructure — 에 있습니다. 이것들은 많은 팀이 투자를 게을리하기 쉬운 계층이며, 다음 신뢰성 문제가 터져 나올 자리이기도 합니다.
