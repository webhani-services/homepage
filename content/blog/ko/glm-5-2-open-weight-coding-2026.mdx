## Closed 독주 구도가 흔들리고 있습니다

지난 몇 년간 Coding용 최첨단 LLM이라 하면 Claude나 GPT 계열의 Closed Model이 주 전장이었습니다. Open-Weight Model은 "쓸 만한 장면도 있지만 본격적인 개발 업무에는 아직 부족하다"는 위치가 암묵적 합의처럼 자리 잡고 있었습니다.

Zhipu AI(국제 Brand명 Z.ai)가 공개한 GLM-5.2는 이 전제에 균열을 냅니다. MIT License로 Weight가 공개되었고, SWE-bench Pro나 Terminal-Bench 같은 실무 중심 Benchmark에서 Closed Frontier Model에 근접한 결과를 냈습니다. webhani가 이 Model이 갖는 의미를 Enterprise 도입 시각에서 정리해보겠습니다.

## GLM-5.2의 Spec과 위치

GLM-5.2는 약 7,530억 Parameter의 Mixture-of-Experts(MoE) 구성이며, Context Window는 100만 Token에 달합니다. API 가격은 Input 기준 100만 Token당 약 1.40달러, Output 기준 약 4.40달러로, Closed Frontier Model의 몇 분의 1 수준입니다.

가장 주목할 점은 Weight 자체가 MIT License로 Hugging Face에 공개되어 있다는 것입니다. API 호출만이 아니라 자체 Infra에 Model을 내려받아 추론을 돌릴 수 있습니다. 이는 "Model을 Service로 빌려 쓴다"에서 "Model을 자산으로 보유한다"는 선택지가 현실적으로 가능해졌다는 뜻입니다.

```bash
# Hugging Face에서 Weight를 받는 예시
huggingface-cli download zai-org/GLM-5.2 --local-dir ./models/glm-5.2

# 추론 Server로 구동하는 예시 (vLLM 등 추론 Engine 사용)
vllm serve ./models/glm-5.2 --tensor-parallel-size 8
```

실무에서는 이 정도 규모의 Model을 자체적으로 운용하려면 상당한 GPU Cluster가 필요하다는 점을 잊으면 안 됩니다. "Open-Weight니까 저비용"이라고 단순화하는 것은 성급하며, 추론 Infra 비용까지 포함한 총소유비용(TCO)으로 비교해야 합니다.

## Benchmark 격차 축소가 중요한 이유

지금까지 Open-Weight Model과 Closed Model 사이에는 단순한 "성능 차이" 이상의 의미가 있었습니다. Closed Model을 선택하는 이유의 상당수는 성능 그 자체보다 "이 정도 품질이 아니면 실무에서 못 쓴다"는 Threshold의 문제였기 때문입니다. GLM-5.2처럼 장시간 Task 대응 Coding Benchmark에서 Frontier급에 근접한 Model이 등장하면서, 그 Threshold를 Open-Weight로도 넘을 수 있는 장면이 실무 차원에서 늘어나고 있습니다.

이것은 단일 Model의 승패 문제가 아니라, "Closed Model API를 호출하는 것" 외의 선택지가 실무 Level에서 성립하기 시작했다는 구조적 변화로 봐야 합니다.

## Enterprise 도입 판단 기준

webhani가 Client에게 조언할 때 확인하는 논점은 다음과 같습니다.

- **Data Residency와 기밀성**: Self-Hosting이면 Code나 설계 자료가 외부 API를 거치지 않습니다. 금융·의료 등 규제가 엄격한 업계에서는 이 한 가지만으로도 검토 가치가 생깁니다.
- **추론 Infra 운영 체력**: 수천억 Parameter급 Model을 자체적으로 안정 운영하려면 GPU 조달, 운영 노하우, 이중화 설계가 필요합니다. 중소 규모 조직이라면 Managed 추론 Hosting Service를 병행하는 선택이 현실적입니다.
- **License 출처와 제공사 Risk 평가**: MIT License 자체는 Business 이용에 관대하지만, Model 제공사의 소재지와 Supply Chain Risk는 Security Review 대상으로 별도 평가해야 합니다.
- **기존 Workflow와의 통합 비용**: Claude Code 같은 기존 Coding Agent 기반에 다른 Model을 편입할 경우, Tool Call 호환성과 Prompt 설계를 다시 손봐야 하는 경우가 많습니다.

## webhani의 견해

GLM-5.2 단독으로 "도입해야 하는가"를 이분법으로 판단하는 것은 성급합니다. 더 중요한 것은 Open-Weight Model이 실무 Level의 선택지로 현실화되었다는 사실 그 자체입니다. 용도와 Data 기밀도에 따라 Closed 계열과 Open-Weight 계열을 나눠 쓰는 Multi-Model 전략을 검토하는 조직이 앞으로 늘어날 것입니다.

현시점의 실무적인 첫걸음으로는, 기밀성이 낮은 사내 Tool이나 실험적 Workflow에서 GLM-5.2 같은 Open-Weight Model을 시험하며 성능과 비용의 실측치를 쌓아가는 접근을 권합니다. Frontier Model과의 성능 차이가 계속 좁혀지는 지금, 선택지를 넓혀 두는 것 자체가 경쟁력이 됩니다.

## 정리

GLM-5.2는 Open-Weight Model이 "실무에서 쓸 수 있는 수준"에 도달하고 있음을 보여주는 상징적 존재입니다. MIT License에 따른 Self-Hosting의 자유도와 Closed 계열에 근접한 Benchmark 성능의 조합은, 특히 Data Residency나 비용에 민감한 조직이라면 검토할 가치가 있습니다. webhani는 단일 Model 의존을 피하고, 용도에 맞게 Model을 나눠 쓰는 Multi-Model 전략의 일환으로 이런 Open-Weight Model 평가를 이어가겠습니다.
