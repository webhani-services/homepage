## Model에게 JSX를 쓰게 하지 마십시오

2026년 7월, Google이 **A2UI v0.9**를 공개했습니다. a2ui.org에서 Spec과 Migration Guide, 그리고 React / Flutter / Lit / Angular용 공식 Renderer를 배포하고 있고, Agent 쪽 SDK는 PyPI를 통해 Python에서 사용할 수 있습니다. InfoQ와 CopilotKit이 이 Release를 다뤘습니다. 다만 중요한 건 A2UI가 무엇을 추가했는지가 아니라, 무엇을 **금지**했는지입니다.

핵심 주장은 한 문장입니다. **Agent는 UI Code를 생성해서는 안 된다. Agent는 UI의 의도만 선언하고, Rendering은 Host Application이 자기 Design System으로 수행한다.**

이건 API 취향 문제가 아니라, 지금 Generative UI를 만들고 있는 대부분의 팀이 밟고 있는 지뢰를 구조적으로 피하는 방법입니다. webhani의 입장은 명확합니다. A2UI 자체를 지금 Production에 넣지는 마십시오. 다만 A2UI가 그은 **경계선만큼은 이번 주에 여러분 Codebase로 옮겨 심을 가치가 있습니다.**

## 지금의 Generative UI가 가진 구조적 결함

현재 지배적인 Pattern은 단순합니다. LLM에게 JSX나 HTML을 뱉게 하고, Client로 보내서 평가합니다. Demo는 화려하지만, 진지하게 Production을 검토하는 순간 문제가 줄지어 나옵니다.

**Security.** Model의 출력이 곧 DOM이 된다는 건, Model의 출력이 XSS Payload가 될 수 있다는 뜻입니다. Prompt Injection이 한 번만 통과해도 공격자는 여러분의 Origin에서 실행되는 Markup을 작성하게 됩니다. 사후 Sanitizing으로 막겠다는 전략은 "위험한 모든 구문을 열거할 수 있다"는 전제 위에 서 있는데, 이 전제는 지난 20년간 계속 무너져 왔습니다.

**Consistency.** Model은 태연하게 `<button class="bg-blue-500">`을 씁니다. 여러분의 Primary Color가 amber라는 사실을 알 리가 없습니다. Design Token은 무시되고, Accessibility 속성은 운에 맡겨지고, Dark Mode는 고객이 Screenshot을 보내올 때까지 조용히 깨져 있습니다.

**Test 불가능성.** 출력이 임의의 Markup이라면, "Model이 Rendering할 수 있는 것의 집합"은 정의되지 않습니다. 정의되지 않은 집합에 대해서는 Test를 쓸 수 없습니다. 우연히 관찰한 몇 개의 출력에 대한 Test만 쓸 수 있을 뿐입니다.

여기서 많은 팀이 iframe Sandbox로 도피합니다. XSS는 격리되지만, 대신 Application 본체와의 통합이 죽습니다. Routing, 인증 상태, 공유 Store, Focus 관리, Theming이 전부 iframe 경계에서 끊깁니다. Security 문제를 임베드 Widget 문제로 교환한 셈입니다.

## Sanitizing이 아니라 출력 공간을 좁히기

A2UI의 답은 "더 강하게 Sanitizing한다"가 아니라 **"위험한 것을 애초에 표현할 수 없게 만든다"**입니다. Agent가 보내는 것은 Schema로 제약된 선언적 JSON입니다. "이 두 가지 선택지를 가진 확인 Dialog를 띄워라"라고는 말할 수 있지만, "이 Raw HTML을 삽입하라"고는 말할 수 없습니다. 그런 Field가 존재하지 않기 때문입니다.

이 구조는 익숙할 겁니다. **Parameterized Query와 문자열 연결 SQL의 관계**와 정확히 같습니다. 연결된 SQL을 Escaping으로 지키는 건 공격자가 항상 한 수 앞서는 군비 경쟁입니다. 반면 Placeholder 기반 Query가 안전한 이유는 검사가 엄격해서가 아니라, **사용자 입력이 구문으로 해석될 문법 자체가 없기** 때문입니다. 안전성이 구조에서 나옵니다.

A2UI는 이 원리를 Render Surface에 적용합니다. Rendering 가능한 표면은 여러분이 이미 소유하고 감사한 Component의 닫힌 집합으로 한정됩니다. 결과는 세 가지입니다.

- Model의 출력을 검증할 수 있습니다. DOM에 닿기 전에 거절할 수 있습니다.
- Rendering 가능한 UI 공간이 유한하고 열거 가능하므로, Snapshot Test와 Unit Test를 실제로 쓸 수 있습니다.
- Design System 이탈이 권고 사항이 아니라 구조적으로 불가능해집니다.

## 양방향성이 진짜 분기점입니다

v0.9는 Minor Update가 아닙니다. 핵심 철학도, JSON 구조도, Schema도 전부 바뀌었고, 결정적으로 **Protocol이 양방향이 되었습니다.** UI는 Render Tree를 받기만 하는 게 아니라 Agent에게 다시 말을 걸 수 있습니다.

배관 얘기처럼 들리지만, Demo와 Product를 가르는 지점입니다. 단방향 Render Tree로 표현할 수 있는 건 "Agent가 무언가를 표시했다"까지입니다. 대화는 표현하지 못합니다. 현실의 Flow는 왕복입니다. 사용자가 주소를 입력하고, Agent가 배송 가능 여부를 판정하고, 선택지를 좁히고, 확인을 요청하고, 사용자가 최종 승인을 누릅니다. 이 Loop가 Protocol의 일부가 아니라면 결국 별도의 RPC Channel을 덧붙이게 되고, Schema가 준 보증은 그 지점에서 전부 사라집니다.

양방향성은 Generative UI 뒤에 Checkout Flow를 붙이기 위한 최소 조건입니다. 거꾸로, 단방향 Generative UI 제안을 보게 된다면 결제나 예약 같은 진지한 Flow에는 쓸 수 없다고 판단해도 됩니다.

## webhani의 결론: Library가 아니라 Pattern을 채택하십시오

직설적으로 말하면, **오늘 v0.9 위에 Production Checkout을 올리지 마십시오.** pre-1.0이고, 심지어 직전 Release에서 철학과 구조와 Schema를 한꺼번에 갈아엎은 이력이 있습니다. 그것이 마지막 Breaking Change라는 근거는 없습니다.

하지만 Architecture를 얻기 위해 Library가 필요하진 않습니다. 필요한 건 세 가지입니다.

1. **Schema로 제약된 UI Intent Payload**
2. Intent 종류를 자사 Component에 대응시키는 **Registry**
3. Registry에 없는 것은 그리지 않고 떨어지는 **Fail-closed Renderer**

이 세 가지를 갖춰 두면 어떤 표준이 이기든 상관없어집니다. A2UI가 1.0에 도달해 시장을 가져가면 Migration은 "Intent JSON의 형태를 바꾸는 것"으로 끝납니다. 다른 표준이 이겨도 답은 같습니다.

## 자사 Codebase에 구현한 Pattern

아래는 A2UI의 API가 아닙니다. 확정되지 않은 Spec의 Signature를 추측하는 건 Documentation Bug를 만드는 지름길입니다. 아래는 **같은 Pattern을 React 19 / TypeScript / Next.js로 직접 구현한 것**이며, 정확하고 그대로 쓸 수 있습니다.

먼저 Model에게 허용할 어휘를 zod로 정의합니다. 이 Schema가 곧 Model이 표현할 수 있는 세계 전부입니다.

```ts
// lib/agent-ui/schema.ts
import { z } from "zod";

const actionSchema = z.object({
  id: z.string().min(1),
  label: z.string().min(1).max(40),
  intent: z.enum(["confirm", "cancel", "retry"]),
});

export const uiIntentSchema = z.discriminatedUnion("type", [
  z.object({
    type: z.literal("confirmation"),
    title: z.string().min(1).max(80),
    body: z.string().max(400),
    actions: z.array(actionSchema).min(1).max(3),
  }),
  z.object({
    type: z.literal("choice"),
    prompt: z.string().min(1).max(120),
    options: z
      .array(z.object({ value: z.string(), label: z.string().max(60) }))
      .min(2)
      .max(6),
  }),
  z.object({
    type: z.literal("summary"),
    heading: z.string().max(80),
    // Raw HTML은 받지 않습니다. Plain Text 줄만 허용합니다.
    lines: z.array(z.string().max(200)).max(10),
  }),
]);

export type UiIntent = z.infer<typeof uiIntentSchema>;
```

여기 없는 것에 주목하십시오. `html`도, `className`도, `style`도 없습니다. Model은 색을 고를 수 없고, Element를 만들 수 없습니다. 문자열 길이 제한까지 Contract의 일부입니다. 닫힌 출력 공간이란 이런 모양입니다.

다음은 Intent 종류를 자사 Component에 대응시키는 Registry입니다.

```tsx
// lib/agent-ui/registry.tsx
import type { ComponentType } from "react";
import type { UiIntent } from "./schema";
import { ConfirmationCard } from "@/components/agent/ConfirmationCard";
import { ChoiceList } from "@/components/agent/ChoiceList";
import { SummaryPanel } from "@/components/agent/SummaryPanel";

type IntentOf<T extends UiIntent["type"]> = Extract<UiIntent, { type: T }>;

type Registry = {
  [T in UiIntent["type"]]: ComponentType<{
    intent: IntentOf<T>;
    onReply: (payload: unknown) => void;
  }>;
};

export const registry: Registry = {
  confirmation: ConfirmationCard,
  choice: ChoiceList,
  summary: SummaryPanel,
};
```

Registry 타입이 `UiIntent["type"]`에 대해 망라적이기 때문에, Schema에 새 Intent를 추가하고 Component 등록을 잊으면 Production 장애가 아니라 **Type Error**로 잡힙니다.

마지막으로 Renderer입니다. 규칙은 하나뿐입니다. 모르는 것은 그리지 않는다.

```tsx
// components/agent/AgentSurface.tsx
"use client";

import { uiIntentSchema } from "@/lib/agent-ui/schema";
import { registry } from "@/lib/agent-ui/registry";

type Props = {
  payload: unknown; // Agent에서 온 미검증 JSON
  onReply: (payload: unknown) => void;
};

export function AgentSurface({ payload, onReply }: Props) {
  const parsed = uiIntentSchema.safeParse(payload);

  if (!parsed.success) {
    // fail closed: 검증에 실패하면 아무것도 그리지 않습니다
    console.error("rejected agent ui intent", parsed.error.issues);
    return <FallbackNotice />;
  }

  const intent = parsed.data;
  const Component = registry[intent.type];

  // Type System상 도달 불가능하지만, Runtime에서도 막습니다
  if (!Component) return <FallbackNotice />;

  return <Component intent={intent as never} onReply={onReply} />;
}
```

양방향성은 `onReply`에서 들어옵니다. 사용자가 `ConfirmationCard`의 "승인"을 누르면 그 결과가 같은 Channel로 Agent에게 돌아가고, Agent는 다음 Intent를 보냅니다. Render Tree를 단방향으로 흘려보내는 게 아니라, 두 참여자가 있는 State Machine을 돌리는 구조가 됩니다.

## 실무에서 따라오는 부수 효과

**Test가 평범해집니다.** `uiIntentSchema`를 만족하는 모든 Payload는 유한한 형태이므로, Intent별 Snapshot Test를 작성할 수 있습니다. Model을 호출하지 않고도 UI를 Test할 수 있다는 뜻입니다.

**Model을 교체해도 UI가 깨지지 않습니다.** Model의 책임은 Schema에 맞는 JSON을 뱉는 것뿐입니다. Provider를 바꿔도 영향 범위는 생성 품질일 뿐, Markup 습관에 의존하지 않습니다.

**Audit Log가 사람이 읽을 수 있는 형태로 남습니다.** Agent가 사용자에게 무엇을 보여줬는지가 DOM Dump가 아니라 의미 있는 JSON으로 기록됩니다. 새벽 2시 장애 조사 때 이 차이가 큽니다.

**Prompt Injection 피해에 천장이 생깁니다.** 최악의 경우에도 공격자가 할 수 있는 건 여러분의 Component를, 여러분이 허용한 글자 수 안에서, 의도치 않은 문구로 표시하게 만드는 정도입니다. 임의 Script 실행까지 도달하지 못합니다. 근절은 아니지만, 피해의 상한선을 Sanitizer가 아니라 Architecture가 결정한다는 점이 중요합니다.

## 마무리

A2UI v0.9의 가치는 Google이 Renderer를 더 배포했다는 데 있지 않습니다. **Agent와 UI 사이의 경계선이 어디여야 하는지를 업계 표준으로 문서화했다**는 데 있습니다. Agent는 의도를 제공하고, Rendering 권한은 여러분의 Design System이 독점합니다.

Spec은 아직 pre-1.0이고 계속 움직입니다. webhani의 권고는 A2UI를 지켜보되, **Schema 제약 Intent + Component Registry + Fail-closed Renderer**라는 3단 구성을 먼저 여러분 Code에 넣어 두는 것입니다. 어떤 표준이 이기든, 여러분의 Architecture는 이미 그 형태를 하고 있을 겁니다.
