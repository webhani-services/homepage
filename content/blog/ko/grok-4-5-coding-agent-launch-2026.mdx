## 또 새로운 Model, 또 하나의 평가 Cycle

xAI가 이번 주 Grok 4.5를 출시했습니다. 범용 Chat Model이 아니라 명확히 Coding과 Agentic Task를 겨냥한 위치입니다. xAI API, xAI 자체의 Grok Build Agent 제품, 그리고 모든 Plan에서 사용 가능한 Cursor를 통해 제공되며, 알려진 바로는 Cursor Team과 공동으로 Training되었다고 합니다 — 몇 주 뒤에 덧붙여진 Third-Party Provider가 아니라 Launch 시점부터 통합이 긴밀한 이유가 여기에 있습니다.

이미 Coding Assistant를 표준화한 Team이라면 발표를 대충 훑고 넘어가고 싶을 수 있습니다. 하지만 저희는 그것이 잘못된 반응이라고 봅니다. Grok 4.5가 반드시 여러분의 Stack에 들어가야 해서가 아니라, 새로운 Frontier Model이 나올 때마다 취해야 할 올바른 대응은 매번 같은 '가벼운 평가'이지, 무시하거나 전면 교체하는 것이 아니기 때문입니다.

## 이번에 실제로 새로운 점

이번 Release에서 일반적인 Marketing Noise와 구분해서 봐야 할 부분은 세 가지입니다.

**가격**: Grok 4.5는 Input 100만 Token당 $2, Output 100만 Token당 $6로 출시되었습니다. 비교 대상이 되는 Frontier Coding Model 대비 공격적인 가격이며, 긴 Context와 짧은 출력이라는 구조(대규모 Codebase Context를 입력하고 범위가 좁은 Diff를 출력하는, 대부분의 Agentic Coding Task에 해당하는 형태)에 유리한 비대칭적 가격 구조를 갖고 있습니다.

**처리량**: xAI는 기본 설정에서 초당 약 80 Token을 보고하고 있으며, 추론 강도(Low, Medium, High)도 설정할 수 있습니다. 대화형 Coding Session에서는 생성 속도가 순수 성능만큼이나 체감 반응성에 영향을 미칩니다. Benchmark Score가 높아도 느린 Model은 실제로 써보면 더 나쁘게 느껴질 수 있습니다.

**배포 방식**: 직접 설정해야 하는 Third-Party Model이 아니라 Launch 시점부터 Cursor 안에 내장되어 있다는 점은 겉보기보다 중요합니다. Model Provider들이 "API로 직접 연결해서 쓰게 하는 것"보다 "개발자가 이미 열어 둔 Tool 안에서 바로 쓸 수 있게 하는 것"을 점점 더 우선시하고 있다는 신호입니다.

## 저희가 새로운 Model Release를 평가하는 방법

저희는 Launch 시점의 Benchmark만 보고 새로운 Frontier Model을 Client 실무에 도입하지 않습니다. 그런 Benchmark는 Provider가 선택한 것이며 실제 여러분의 Codebase나 Task 구성을 반영하지 못하는 경우가 많기 때문입니다. 저희의 평가 Checklist는 다음과 같습니다.

1. **합성 Benchmark가 아니라 자신의 Repository로 실행한다**: Backlog에서 실제 Ticket 3~5개를 고릅니다(Bug 수정, 작은 기능 추가, Refactoring 등). 같은 Ticket을 현재 쓰는 Model과 새 Model 양쪽에서 돌립니다.
2. **Pass/Fail뿐 아니라 Working Diff까지의 Token 수를 측정한다**: 더 적은 반복으로 올바른 해법에 도달하는 Model은 단순 정확도가 시사하는 것 이상의 가치가 있습니다. Agentic Workflow에서는 반복 비용이 누적되기 때문입니다.
3. **Tool 호출 신뢰성을 추론 품질과 별도로 확인한다**: Agentic Coding은 File 읽기/쓰기, Shell 실행 같은 Tool을 순서대로 정확히 호출하는 능력에 크게 좌우됩니다. 이는 Code 품질과는 별개의 능력이므로 독립적인 Test가 필요합니다.
4. **헤드라인 요금이 아니라 실제 사용 패턴 기준으로 가격을 계산한다**: Turn마다 큰 Context Window를 다시 읽는 Input 중심의 Agentic Session은 같은 가격표 아래에서도 짧고 독립적인 Completion과는 전혀 다르게 동작합니다.

```typescript
// 공개 Benchmark가 아니라 자신의 Ticket으로
// Model을 비교하기 위한 최소한의 Harness
interface EvalTicket {
  id: string;
  prompt: string;
  repoSnapshot: string;
  expectedDiffPattern: RegExp;
}

async function evaluateModel(
  model: string,
  tickets: EvalTicket[]
): Promise<{ id: string; iterations: number; passed: boolean }[]> {
  const results = [];
  for (const ticket of tickets) {
    const { diff, iterations } = await runAgenticSession(model, ticket);
    results.push({
      id: ticket.id,
      iterations,
      passed: ticket.expectedDiffPattern.test(diff),
    });
  }
  return results;
}
```

## 대부분의 Team이 이미 살고 있는 Multi-Model 현실

저희가 함께 일하는 Team 중 단일 Model만 계속 쓰는 곳은 거의 없습니다. 일반적인 패턴은 일상적인 Agentic Coding에는 주력 Model을 쓰고, 특정 Task 유형은 다른 Model로 Routing하는 방식입니다 — Cost에 민감한 대량 작업은 더 저렴한 Model로, 복잡한 Architecture 판단은 그 시점에 해당 축에서 가장 앞선 Model로 보내는 식입니다. Grok 4.5의 가격 정책과 Cursor Native한 제공 방식은 이런 구성에서 'Cost 민감·고 Volume' 자리의 유력한 후보가 될 수 있습니다. 가장 어려운 추론 Task를 담당하는 Model을 반드시 대체할 필요는 없습니다.

바로 이런 이유로 통합 Layer에서의 Vendor Lock-in은 적극적으로 피할 가치가 있습니다. Agentic Coding 구성이 특정 Vendor의 SDK에 직접 연결되어 있다면, Grok 4.5 같은 신규 진입자를 평가하는 것만으로도 전환할 가치가 있는지 확인하기도 전에 통합 Code부터 다시 작성해야 합니다. Model에 종속되지 않는 Interface — Gateway 형태의 추상화든, Provider 전반에 걸쳐 일관된 Prompt/Tool Call 형식이든 — 를 거치도록 하면 평가 비용을 낮게 유지할 수 있습니다.

## webhani의 권장 사항

- **분기마다 반나절을 새 Model 평가에 배정한다**: 매번 동일한 고정 Ticket Set을 사용해, Provider 자체의 Benchmark 주장만이 아니라 Release 간 비교 가능한 결과를 확보합니다.
- **모든 Release를 좇지 않는다**: 의미 있는 차별화가 있는 출시(신규 Provider, 눈에 띄는 가격 변화, 새로운 Tool 사용 능력)만 평가 대상으로 삼고, 세부 Point Release까지 추적하지 않습니다.
- **"Coding을 잘하는가"와 "Agentic한 Tool 사용을 잘하는가"를 구분한다**: 둘은 상관관계는 있지만 동일하지 않으며, Agentic Workflow는 전자보다 후자 축에서 더 자주 실패합니다.
- **통합 Layer를 Model에 종속되지 않게 유지한다**: Frontier Model 지형이 아무리 빠르게 바뀌어도 평가 비용을 낮게 유지할 수 있습니다.

## 정리

Grok 4.5는 Marketing용 이벤트가 아니라 정당한 신규 진입자입니다 — 경쟁력 있는 가격, Cursor Native한 배포, Coding 우선 설계 모두 실무적으로 의미가 있습니다. 하지만 이 분야를 Release마다 지켜보며 얻는 더 근본적인 교훈은, 개별 Model 자체보다 최신 Model을 저렴하고 반복적으로 평가할 수 있는 Team의 역량이 더 중요하다는 것입니다. 그 역량을 한 번 갖춰두면, 앞으로 나올 Grok이든 GPT든 Claude든 새로운 Release는 논쟁거리가 아니라 반나절이면 내릴 수 있는 판단이 됩니다.
