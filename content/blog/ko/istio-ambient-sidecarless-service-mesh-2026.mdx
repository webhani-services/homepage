Service Mesh 도입에는 항상 근본적인 긴장감이 있었습니다. 제공되는 기능(mTLS, Traffic 관리, Observability)은 분명히 가치 있지만, 모든 워크로드 Pod 옆에 Sidecar Proxy를 실행하는 운영 비용이 지속적인 장벽이 되어왔습니다. Memory 오버헤드, Injection 설정의 복잡성, Upgrade 조정 어려움, Debug 난이도 — 이것들은 실제 비용이며 많은 팀이 Istio를 대규모로 채택하지 못하도록 막아온 요인입니다.

Istio의 Ambient Mode는 이 문제를 Architecture 수준에서 해결합니다. 각 Pod에 Envoy Sidecar를 주입하는 대신, 노드별 Layer 4 Proxy(`ztunnel`)와 선택적 Namespace별 Layer 7 Waypoint Proxy라는 두 개의 공유 컴포넌트를 사용합니다. 실질적인 결과: Resource 비용을 대폭 줄이면서 Service Mesh 기능을 제공합니다.

2026년 KubeCon EU(암스테르담)에서 Istio 프로젝트는 Ambient Mode를 한층 더 프로덕션 영역으로 밀어넣는 3가지 신기능을 발표했습니다. Ambient Multicluster Beta, Gateway API Inference Extension Beta, Agentgateway(실험적). 이를 종합하면 Ambient Mode가 더 이상 실험적 트랙이 아니라 권장 Deployment 모델로 자리잡아가고 있음을 알 수 있습니다.

## Ambient Mode의 작동 방식

Architecture를 이해하면 어디에 적합하고 어디에 적합하지 않은지를 판단할 수 있습니다.

**ztunnel(Layer 4)**: 각 Kubernetes 노드는 단일 `ztunnel` 인스턴스를 실행합니다. 이는 노드 레벨에서 동작하며, 해당 노드의 모든 워크로드에 대한 Identity 검증, mTLS 암호화, 기본 Traffic Routing을 처리합니다. Pod 레벨이 아닌 노드 레벨에서 동작하기 때문에 Per-pod Injection 없이 모든 Pod 간 통신에 Zero-Trust Networking이 적용됩니다.

**Waypoint Proxy(Layer 7)**: HTTP Traffic 관리(Routing 규칙, Retry, Header 조작, 고급 Authorization Policy)가 필요한 Service에는 Namespace 레벨에 Waypoint Proxy를 배포합니다. 내부적으로는 Envoy이지만 Pod마다 복제되는 것이 아니라 Namespace 내 모든 워크로드가 공유합니다.

**Resource 소비에 대한 실질적 영향**: Sidecar Mode에서는 각 Pod가 풀 Envoy 인스턴스를 갖습니다. 수백 개의 Service와 여러 Replica를 가진 Cluster에서는 Cluster 전체에 걸쳐 Gigabyte 수준의 Steady-state Memory를 소비하게 됩니다. Ambient Mode는 이를 크게 줄입니다. 프로덕션 평가에서 Sidecar Mode 대비 Memory 90% 이상, CPU 50% 이상 절감이 보고되었습니다.

## KubeCon EU 2026 발표

3가지 신기능은 각각 Ambient Mode 스토리에서 다른 공백을 채웁니다.

**Ambient Multicluster Beta**: 이전에는 Ambient Mode가 Multi-cluster 시나리오(고가용성이나 지리적 분산을 위한 일반적인 패턴)에 대한 지원이 제한적이었습니다. Multicluster Beta는 동일한 ztunnel + Waypoint 모델을 사용하여 Ambient Mode의 Traffic 관리와 mTLS를 Cross-cluster 트래픽까지 확장합니다.

**Gateway API Inference Extension Beta**: AI/ML 워크로드를 실행하는 팀에 특히 관련이 있습니다. GPU Backed Inference Service가 Kubernetes Cluster에서 더 일반화됨에 따라, Inference Endpoint의 Traffic Routing은 특정 요건을 갖습니다. 모델 인식 Load Balancing, Request Priority, 오래 실행되는 Inference Request를 위한 Connection Pooling이 그것입니다. Inference Extension은 이런 기능들을 Gateway API Resource에 추가하여 Istio Ambient Mode를 AI Serving Infrastructure를 위한 더 유능한 플랫폼으로 만듭니다.

**Agentgateway(실험적)**: Agentgateway는 Ambient Mode의 Traffic 관리를 AI Agent 간 통신으로 확장합니다. Istio가 Agent 간 트래픽을 Routing, 인증, 관찰할 수 있게 합니다. 아직 초기 단계이지만, Agentic AI 워크로드가 프로덕션으로 이동함에 따라 Kubernetes Networking이 어디로 향하는지를 반영합니다.

## Sidecar Mode에서의 Migration 경로

현재 Sidecar로 Istio를 실행 중이라면, Ambient Mode로의 전환은 Lift-and-Shift가 아닙니다. 두 Mode는 전환 기간 동안 Cluster 내에 공존할 수 있어 단계적 Migration이 가능합니다.

실용적인 Migration 순서:

1. **Ambient 컴포넌트 설치**: `ztunnel`을 DaemonSet으로 배포하고 Ambient Mode를 지원하는 버전의 istiod를 설치합니다.
2. **Namespace에 Ambient 레이블 지정**: Ambient Mode를 원하는 Namespace에 `istio.io/dataplane-mode: ambient`를 추가합니다. 기존 Sidecar가 주입된 Pod는 계속 작동합니다.
3. **Waypoint Proxy 선택적 배포**: L7 Policy가 필요한 Service는 Namespace에 Waypoint Proxy를 배포합니다. L4만 필요한 경우(mTLS, 기본 Authorization)는 ztunnel만으로 충분합니다.
4. **Sidecar Injection 제거**: Namespace의 모든 워크로드가 Ambient Mode로 커버되면 해당 Namespace의 Sidecar Injection을 비활성화하고 Pod를 재시작하여 기존 Sidecar를 제거합니다.

Red Hat이 OpenShift Service Mesh 3.2에 Ambient Mode를 GA로 포함시킨 것은 여기서 유용한 신호를 제공합니다. Enterprise Kubernetes Distribution이 기능을 GA로 출시한다는 것은 일반적으로 Migration 도구와 운영 플레이북이 프로덕션 사용에 충분히 성숙했다는 의미입니다.

## 실용적 고려 사항

Ambient Mode를 채택하기 전에 확인해야 할 몇 가지 사항이 있습니다.

**Istio 버전**: Ambient Multicluster와 최신 기능에는 Istio 1.22 이상이 필요합니다. 현재 버전을 확인하고 필요하다면 Upgrade 경로를 계획하세요.

**CNI 호환성**: Ambient Mode의 ztunnel은 eBPF 기반 Traffic Interception을 사용하며 CNI Plugin과의 호환성 요건이 있습니다. Calico, Cilium, Flannel은 모두 지원되지만, Istio의 호환성 Matrix에서 특정 버전 조합을 확인하세요.

**L7 Policy 인벤토리**: 현재 Istio 설정 중 Layer 7 처리가 필요한 것(VirtualService, HTTP Routing 규칙, Header 기반 Auth)을 정리하세요. 이것들은 Ambient Mode에서 Waypoint Proxy 배포가 필요합니다. L4 Policy(mTLS의 PeerAuthentication, 기본 AuthorizationPolicy)는 ztunnel만으로 동작합니다.

**Observability 연속성**: 기존 Kiali, Jaeger, Prometheus 통합은 Ambient Mode에서 계속 작동해야 합니다. Telemetry 모델은 호환됩니다. 단, Dashboard Query가 Sidecar 특정 Metrics 레이블에 의존하는지 확인하세요.

## webhani의 관점

Service Mesh는 많은 고객과의 Architecture 논의에서 반복적으로 등장하는 주제였습니다. 현재 규모에 비해 운영 비용이 이점에 비해 높다는 이유로 종종 미뤄졌습니다. Ambient Mode는 그 계산을 근본적으로 바꿉니다.

새로운 Kubernetes 배포에 대해서는 Service 간 mTLS가 필요한 모든 Cluster에 Ambient Mode를 기본 권고사항으로 삼고 있습니다. Resource 절감이 크고, 운영 모델이 단순하며, Feature Set이 대부분의 프로덕션 요건을 커버합니다.

기존 Sidecar 기반 배포에 대해서는 다음 Infrastructure 사이클에서 Migration을 계획하는 것을 권장합니다. Namespace별 단계적 Migration 경로 덕분에 Big-bang 전환 없이 진행할 수 있습니다.

KubeCon EU 2026의 발표, 특히 Inference Extension은 Istio Ambient가 Kubernetes의 AI/ML 워크로드를 위한 표준 Networking Layer로 자리매김하고 있음을 보여줍니다. Infrastructure 로드맵에 GPU Inference Service 확장이 포함되어 있다면, 지금부터 Ambient Mode에 익숙해지는 것이 나중에 분명히 도움이 될 것입니다.
