2026년 JavaScript 생태계는 지루합니다. 이는 불평이 아니라 웹 개발에 일어난 가장 좋은 변화입니다.

5년 전만 해도 "Framework Fatigue"는 실제 문제였습니다. 분기마다 새로운 Bundler나 Meta-framework이 등장했고, 따라잡는 것 자체가 거의 전업이었습니다. 그 시대는 끝났습니다.

2026년 5월 The JavaScript Doctor의 보고서에 따르면, 신규 프로젝트의 권장 스택은 **Next.js + TypeScript + Tailwind CSS + Prisma + PostgreSQL + Supabase**로 수렴했습니다. 특정 벤더의 주장이 아니라, 커뮤니티가 Production 경험을 통해 독립적으로 도달한 합의입니다.

## 전환점을 상징하는 두 가지 사건

**React Compiler의 Stable 도달.** React 19.2에서 React Compiler가 Stable 상태가 됐습니다. 오랜 숙제였던 수동 Memoization 문제에 대한 공식 답변입니다. 이제 `useMemo`와 `useCallback` 배치를 고민하지 않아도 됩니다.

```tsx
// React Compiler 이전: 수동 최적화 필요
const SortedList = React.memo(({ items }: { items: string[] }) => {
  const sorted = useMemo(() => [...items].sort(), [items]);
  return <ul>{sorted.map(i => <li key={i}>{i}</li>)}</ul>;
});

// React Compiler 이후: 자연스럽게 작성, Compiler가 최적화
const SortedList = ({ items }: { items: string[] }) => {
  const sorted = [...items].sort();
  return <ul>{sorted.map(i => <li key={i}>{i}</li>)}</ul>;
};
```

**TypeScript의 필수화.** TypeScript 6.0 RC 출시와 함께, TypeScript 채택은 "할지 말지"의 논쟁이 아니게 됐습니다. 신규 프로젝트에서 순수 JavaScript 코드베이스를 선택하는 것은 의도적으로 기술 부채를 만드는 것과 동일하게 여겨집니다.

## Next.js가 표준 위치를 유지하는 이유

Next.js 16은 `use cache` Directive를 도입해 페이지, 컴포넌트, 함수 단위로 캐시를 선언적으로 제어합니다:

```typescript
async function getProducts() {
  "use cache";
  return await db.products.findMany(); // 기본적으로 캐시됨
}

async function getWeatherForecast() {
  "use cache";
  cacheLife("hours"); // 명시적 TTL 설정
  return await fetch("https://api.weather.example/forecast").then(r => r.json());
}
```

안정화된 App Router, 성숙한 React Server Components 지원, Production에서 검증된 Server Actions — 이것들이 쌓여 Next.js는 2010년대 Rails가 가졌던 "이걸 선택하면 안전하다"는 포지션을 확립했습니다.

## Bundler 공존의 시대

Turbopack이 Next.js 프로젝트의 기본 Dev Server가 됐고, 비 Next.js React 애플리케이션에서는 Vite 6가 주류입니다. 두 도구 간의 경쟁이 양쪽 모두의 성능을 끌어올렸고, 개발자는 Framework 선택에 따라 자동으로 최적의 도구를 갖게 됩니다.

| 용도 | 도구 |
|---|---|
| Full-stack Framework | Next.js (App Router) |
| SPA / Library 개발 | Vite + React |
| Type 안전성 | TypeScript 6 (strict) |
| Styling | Tailwind CSS 4 |
| Database ORM | Prisma 또는 Drizzle |
| Database / BaaS | PostgreSQL + Supabase |

## '지루한 기술 선정'의 가치

스택 논쟁을 멈추면 제품 개발에 집중할 수 있습니다. 2~3년 전에 안정적이고 관습적인 도구를 채택한 팀들이 지금 앞서 있습니다. 도구가 더 뛰어나서가 아니라, 도메인 로직과 사용자 경험에 시간을 쏟았기 때문입니다.

webhani에서도 신규 Client 프로젝트의 기본 스택으로 이 수렴된 구성을 사용합니다. 이유는 실용적입니다. 새 엔지니어 온보딩이 쉽고, 채용 후보자의 스킬셋과 일치하며, 장기적인 유지보수 비용을 예측하기 쉽습니다.

## 마무리

JavaScript 생태계의 수렴은 정체가 아니라 Production 경험을 통한 자연 선택의 결과입니다. 살아남은 도구들은 다양한 팀 규모와 프로젝트 유형에서 스스로를 증명한 것들입니다.

새 프로젝트의 기술 스택으로 고민 중이라면, 커뮤니티 합의를 출발점으로 삼으세요. 차별화는 무엇으로 만드느냐가 아니라 무엇을 만드느냐에서 나옵니다.
