## 들어가며

2026년 3월 23일부터 26일까지 암스테르담에서 KubeCon + CloudNativeCon Europe 2026이 개최됩니다. Cloud Native 생태계는 이미 성숙기에 접어들었고, 논의의 초점은 "도입 여부"에서 "운영 품질을 어떻게 높일 것인가"로 이동했습니다.

이 글에서는 올해 KubeCon에서 특히 주목할 만한 세 가지 트렌드 — Platform Engineering, Agentic DevOps, Software Supply Chain Security — 를 실무 관점에서 정리합니다.

## Platform Engineering의 본격화

### IDP가 표준 운영 모델로 자리잡다

Internal Developer Platform(IDP)은 개발자가 인프라 세부사항을 직접 다루지 않고도 Service를 프로비저닝하고 배포할 수 있게 해주는 플랫폼입니다. 2026년에는 중규모 이상 조직에서 IDP가 표준적인 운영 모델로 정착하고 있습니다.

Kubernetes가 계속 복잡해지는 상황에서 Application 개발자에게 그 복잡도를 그대로 노출하는 것은 비효율적입니다. IDP는 그 사이의 추상화 레이어 역할을 합니다.

### Backstage를 활용한 IDP 구축

CNCF 프로젝트인 Backstage는 IDP 기반으로 가장 널리 사용되고 있습니다. 아래는 Software Catalog에 Kubernetes Component를 등록하는 설정 예시입니다.

```yaml
# catalog-info.yaml
apiVersion: backstage.io/v1alpha1
kind: Component
metadata:
  name: order-service
  description: 주문 처리 마이크로서비스
  annotations:
    backstage.io/kubernetes-id: order-service
    backstage.io/techdocs-ref: dir:.
  tags:
    - go
    - grpc
spec:
  type: service
  lifecycle: production
  owner: team-backend
  system: ecommerce
  dependsOn:
    - resource:default/postgres-order-db
    - component:default/payment-service
```

### Scorecard로 서비스 성숙도 측정

IDP의 핵심 가치는 단순한 프로비저닝이 아니라, 조직 전체의 Service 품질을 지속적으로 가시화하는 데 있습니다. Backstage Scorecards를 사용하면 Production 준비 상태를 정량적으로 평가할 수 있습니다.

```yaml
# scorecard.yaml
apiVersion: backstage.io/v1alpha1
kind: Scorecard
metadata:
  name: production-readiness
spec:
  checks:
    - id: has-owner
      name: Owner 팀이 지정되어 있는가
      rule:
        metadata.annotations['team/owner']: { exists: true }
    - id: has-runbook
      name: Runbook이 정비되어 있는가
      rule:
        metadata.links[?title=='Runbook']: { exists: true }
    - id: has-slo
      name: SLO가 정의되어 있는가
      rule:
        metadata.annotations['slo/availability']: { exists: true }
```

## Agentic DevOps — CI/CD에 AI 통합

### 76%의 팀이 이미 AI를 CI/CD에 도입

2025년 업계 조사에 따르면, DevOps 팀의 약 76%가 CI/CD Pipeline에 어떤 형태로든 AI를 통합한 것으로 나타났습니다. 단순한 자동화를 넘어, AI Agent가 Pipeline 내에서 맥락에 맞는 판단을 내리는 "Agentic DevOps" 패턴이 확산되고 있습니다.

구체적으로는 Test 실패 원인 분석, Deploy 위험도 판단, Security Scan 결과 분류 등에서 AI가 역할을 담당합니다.

### 실전 예시: 변경 위험도 기반 Pipeline 분기

아래는 변경 내용의 위험도에 따라 Pipeline 동작을 조정하는 GitHub Actions Workflow 예시입니다.

```yaml
# .github/workflows/ai-assisted-review.yaml
name: AI-Assisted Pipeline

on:
  pull_request:
    branches: [main]

jobs:
  ai-review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: 변경 위험도 분석
        id: risk-assessment
        run: |
          CHANGED_FILES=$(git diff --name-only origin/main...HEAD)
          HIGH_RISK_PATTERNS="migration|security|auth|payment"
          RISK_LEVEL="low"
          for file in $CHANGED_FILES; do
            if echo "$file" | grep -qiE "$HIGH_RISK_PATTERNS"; then
              RISK_LEVEL="high"
              break
            fi
          done
          echo "risk_level=$RISK_LEVEL" >> "$GITHUB_OUTPUT"

      - name: 고위험 변경에 대한 추가 보안 스캔
        if: steps.risk-assessment.outputs.risk_level == 'high'
        run: |
          echo "고위험 변경이 감지되었습니다. 추가 보안 스캔을 실행합니다."
          trivy fs --severity HIGH,CRITICAL .
```

변경 맥락에 따라 Pipeline 동작을 동적으로 변경하는 이 패턴은 Agentic DevOps의 실용적인 시작점입니다. 단순한 것부터 시작해서 점진적으로 정교한 분석을 추가하면 됩니다.

## Software Supply Chain Security

### SBOM과 SLSA가 필수 요소로

Supply Chain 공격의 증가로 인해 SBOM(Software Bill of Materials) 생성과 SLSA(Supply-chain Levels for Software Artifacts) Framework 준수가 필수적인 보안 관행이 되고 있습니다. Kubernetes 생태계에서는 Container Image 서명 및 검증이 표준적인 Practice로 자리잡고 있습니다.

### Cosign을 이용한 Image 서명

```dockerfile
# Dockerfile (멀티스테이지 빌드)
FROM golang:1.22-alpine AS builder
WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download
COPY . .
RUN CGO_ENABLED=0 go build -o /app/server ./cmd/server

FROM gcr.io/distroless/static-debian12:nonroot
COPY --from=builder /app/server /server
USER nonroot:nonroot
ENTRYPOINT ["/server"]
```

```bash
# Image 서명
cosign sign --key cosign.key ghcr.io/myorg/order-service:v1.2.0

# 서명 검증
cosign verify --key cosign.pub ghcr.io/myorg/order-service:v1.2.0

# SBOM 생성 및 첨부
syft ghcr.io/myorg/order-service:v1.2.0 -o spdx-json > sbom.json
cosign attach sbom --sbom sbom.json ghcr.io/myorg/order-service:v1.2.0
```

### Cluster 수준에서 서명 강제

Admission Controller와 결합하면 서명되지 않은 Image의 Deploy를 Cluster 수준에서 차단할 수 있습니다.

```yaml
# Kyverno ClusterPolicy
apiVersion: kyverno.io/v1
kind: ClusterPolicy
metadata:
  name: verify-image-signature
spec:
  validationFailureAction: Enforce
  rules:
    - name: check-signature
      match:
        any:
          - resources:
              kinds:
                - Pod
      verifyImages:
        - imageReferences:
            - "ghcr.io/myorg/*"
          attestors:
            - entries:
                - keys:
                    publicKeys: |-
                      -----BEGIN PUBLIC KEY-----
                      ...
                      -----END PUBLIC KEY-----
```

## 중규모 프로젝트에 미치는 영향

webhani가 Consulting으로 참여하는 중규모 프로젝트(5~20명 규모)에서 이 트렌드들은 다음과 같이 적용됩니다.

- **Platform Engineering**: 처음부터 완전한 IDP를 구축할 필요는 없습니다. Helm Chart 템플릿화와 ArgoCD를 활용한 GitOps 도입만으로도 개발자 경험이 크게 개선됩니다
- **Agentic DevOps**: CI/CD Pipeline에 Security Scan과 변경 위험도 분석을 추가하는 것부터 시작하는 게 현실적입니다. 위의 Risk 기반 분기 패턴은 외부 AI 벤더 의존 없이도 바로 적용 가능합니다
- **Supply Chain Security**: Container Image 서명은 도입 비용이 낮고 효과가 큽니다. SBOM 생성도 CI에 한 단계만 추가하면 됩니다. 둘 다 빠르게 적용 가능한 항목입니다

## 마치며

KubeCon Europe 2026의 주제들은 Cloud Native가 "기본값"이 된 이후, 운영 품질을 어떻게 높일 것인가에 집중하고 있습니다. Platform Engineering으로 개발자 경험을 향상시키고, Agentic DevOps로 Pipeline을 지능화하며, Supply Chain Security로 소프트웨어 전달 과정을 강화하는 흐름입니다.

세 가지 모두 점진적으로 도입할 수 있습니다. 작게 시작해서 효과를 확인하고 확장하는 방식이, 한 번에 대규모로 전환하는 것보다 현실적이고 효과적입니다.
