KubeCon + CloudNativeCon Europe 2026에서 CNCF와 SlashData의 Q1 2026 State of Cloud Native Development 리포트가 공개됐습니다. 글로벌 Cloud Native 개발자 수는 1,990만 명에 달하며, 조사 대상 기업의 93%가 Kubernetes를 사용 중, 파일럿 중, 또는 평가 중이고 그 중 80%가 프로덕션 환경에서 운영하고 있습니다.

이 숫자가 의미하는 것은 간단합니다. Kubernetes는 더 이상 선진적인 선택이 아닌 업계 표준 Infrastructure입니다. 이 포스트에서는 Kubernetes 1.35의 주요 변경 사항, Docker의 2026 로드맵, AI Native Infrastructure 트렌드의 실질적 의미를 정리합니다.

## Kubernetes 1.35: DRA Binding Conditions 베타 승격

Kubernetes 1.35에서 가장 주목할 기술적 변화는 Dynamic Resource Allocation(DRA)의 Binding Conditions가 알파에서 베타로 승격된 것입니다. DRA는 GPU, FPGA 같은 특수 하드웨어 리소스를 Pod에 동적으로 할당하는 메커니즘으로, 기존의 정적 할당 방식을 대체합니다.

AI/ML 워크로드를 Kubernetes에서 운영하는 팀에게 이 변화는 실질적인 의미가 있습니다. 기존 `resources.limits`로 GPU를 지정하는 방식은 경직돼 있어서 GPU가 특정 Pod에 할당되거나 안 되거나 둘 중 하나입니다. DRA는 조건을 포함한 구조화된 요청을 지원합니다.

```yaml
apiVersion: resource.k8s.io/v1alpha3
kind: ResourceClaim
metadata:
  name: gpu-resource
spec:
  devices:
    requests:
    - name: gpu
      deviceClassName: nvidia-gpu
      selectors:
      - cel:
          expression: device.attributes["memory"].isGreaterThan(quantity("16Gi"))
```

이 방식은 "16GB 이상 메모리를 가진 GPU"처럼 조건을 표현할 수 있어, 스케줄러가 사용 가능한 하드웨어를 동적으로 매칭합니다. 여러 Pod가 GPU를 공유하는 추론 서비스나 가변적인 리소스 요구를 가진 학습 파이프라인에서 실질적인 효율 개선이 가능합니다.

```yaml
# ML 워크로드용 Namespace 설정 예시
apiVersion: v1
kind: Namespace
metadata:
  name: ml-workloads
  labels:
    nvidia.com/gpu.present: "true"
---
apiVersion: v1
kind: LimitRange
metadata:
  name: gpu-limits
  namespace: ml-workloads
spec:
  limits:
  - type: Container
    default:
      nvidia.com/gpu: "1"
    max:
      nvidia.com/gpu: "4"
```

베타 승격은 프로덕션 테스트에 충분히 안정적이라는 의미이지만, GA 전까지 API 변경 가능성이 있으므로 팀의 Migration 계획도 함께 세워두는 것이 좋습니다.

## Docker 2026 로드맵: 3가지 축

Docker의 2026년 방향은 AI Native 개발, Security by Design, WebAssembly 통합의 세 가지를 중심으로 합니다.

**AI Native 개발**은 Docker Desktop에 AI 기능을 직접 통합하는 것입니다. Dockerfile 최적화 제안, 보안 스캔 결과 해석, 설정 자동 생성이 포함됩니다. Multi-stage Build는 AI 보조 최적화가 실질적으로 도움이 되는 영역 중 하나입니다.

```dockerfile
# 최적화된 Multi-stage Build 예시
FROM node:24-alpine AS deps
WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production

FROM node:24-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
RUN npm run build

FROM node:24-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
COPY --from=builder /app/.next ./.next
COPY --from=deps /app/node_modules ./node_modules
EXPOSE 3000
CMD ["npm", "start"]
```

**Security by Design**는 빌드 시점에 취약점을 발견하는 방향입니다. 알려진 CVE가 포함된 베이스 이미지 사용 시 경고, 최소 권한 원칙 기반 설정 권장이 강화됩니다.

**WebAssembly(Wasm) 통합**은 경량 워크로드나 보안이 중요한 처리에서 Linux Container의 대안으로 Wasm을 활용하는 것입니다. Container보다 시작이 빠르고 Sandbox가 강력해 Edge Computing에서 활용이 늘고 있습니다.

## Serverless Container와 Kubernetes 복잡성 추상화

KubeCon에서 반복된 주제 중 하나는 Kubernetes의 운영 복잡성을 추상화하는 Managed Control Plane으로의 전환입니다. AWS Fargate, Google Cloud Run, Azure Container Apps 같은 Serverless Container 서비스를 이용하면 Cluster Node, Control Plane, etcd 관리 없이 Container를 실행할 수 있습니다.

트레이드오프는 명확합니다.

| 상황 | 권장 방식 |
|------|----------|
| 세밀한 제어가 필요한 대규모 Cluster | 직접 Kubernetes 운영 |
| Kubernetes 전문가가 없는 팀 | Managed Service (GKE, EKS, AKS) |
| Infrastructure 관리를 최소화하고 싶은 경우 | Serverless Container |
| GPU 스케줄링이 필요한 복잡한 AI/ML 파이프라인 | Kubernetes + DRA |

대부분의 웹 애플리케이션 워크로드에서는 Managed Kubernetes 서비스나 Serverless Container가 더 나은 기본값입니다. DRA 수준의 GPU 제어, 특정 Network Policy, 또는 Managed Service 비용이 정당화되지 않는 규모에서만 직접 Cluster 관리를 선택하는 것이 합리적입니다.

## YAML을 넘어선 Infrastructure as Code

Pulumi처럼 Kubernetes 설정을 범용 프로그래밍 언어로 관리하는 도구의 채택이 늘고 있습니다. YAML 기반 설정의 한계(루프 없음, 조건문 없음, 타입 없음, 재사용 어려움)를 TypeScript나 Python으로 극복하는 접근입니다.

```typescript
// Pulumi: TypeScript로 Kubernetes Deployment 정의
import * as k8s from "@pulumi/kubernetes"

const appLabels = { app: "api-service" }

const deployment = new k8s.apps.v1.Deployment("api-service", {
  spec: {
    replicas: 3,
    selector: { matchLabels: appLabels },
    template: {
      metadata: { labels: appLabels },
      spec: {
        containers: [{
          name: "api",
          image: "myapp/api:1.2.0",
          resources: {
            requests: { cpu: "100m", memory: "128Mi" },
            limits: { cpu: "500m", memory: "512Mi" },
          },
          readinessProbe: {
            httpGet: { path: "/health", port: 3000 },
            initialDelaySeconds: 10,
            periodSeconds: 5,
          },
        }],
      },
    },
  },
})
```

대부분의 팀에서 Helm을 완전히 대체하지는 않지만, YAML의 한계가 유지보수 문제를 일으키는 복잡한 설정에서 실용적인 대안이 됩니다.

## AIOps 기반 FinOps

Cloud 비용 최적화에 AI가 결합되고 있습니다. AI/ML 워크로드로 리소스 소비가 증가하면서 프로비저닝된 리소스와 실제 사용량 간의 격차가 커지고 있습니다. AIOps 도구는 실제 사용 패턴을 분석해 리소스 `requests`와 `limits` 조정을 권장하거나 자동으로 적용합니다.

해결하려는 문제는 명확합니다. 팀들은 안전 마진을 두고 리소스를 과도하게 할당하는 경향이 있습니다. 512Mi 메모리 Limit으로 설정됐지만 항상 180Mi만 사용하는 Container가 수백 개의 Replica로 실행된다면, 불필요하게 예약된 용량이 상당합니다. 관찰된 패턴 기반의 자동 Right-sizing은 안정성을 희생하지 않고 Cluster 비용을 실질적으로 줄일 수 있습니다.

## 실무 포인트

**새 Infrastructure 프로젝트를 시작하는 팀**: 특별한 이유가 없다면 Self-managed Cluster보다 Managed Kubernetes Service(GKE, EKS, AKS)를 선택하세요. Control Plane 관리 부담은 대부분의 Application 워크로드에서 정당화되기 어렵습니다.

**AI/ML 워크로드를 운영하는 팀**: Kubernetes 1.35의 DRA 베타를 주목하세요. 현재 정적 설정으로 GPU 할당 한계를 우회하고 있다면, DRA가 더 깔끔한 경로를 제공합니다. 베타 단계인 지금부터 테스트를 시작하세요.

**Docker 로드맵을 평가 중인 팀**: AI 보조 Dockerfile 최적화 기능은 Container 모범 사례 온보딩에 유용합니다. Security by Design 스캔 기능은 Build 시점에 CVE를 잡는 것이 프로덕션에서 잡는 것보다 항상 낫기 때문에 우선적으로 활용할 가치가 있습니다.

Cloud Native Infrastructure는 성숙 단계에 접어들었습니다. 이제 "Kubernetes를 써야 하나"가 아닌 "GPU 가속 AI 워크로드를 비용 효율적으로 어떻게 운영할 것인가"가 핵심 과제입니다. 그리고 도구들은 그 문제를 향해 빠르게 수렴하고 있습니다.
