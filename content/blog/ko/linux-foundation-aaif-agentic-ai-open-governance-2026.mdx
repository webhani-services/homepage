Linux Foundation은 2026년 4월, **Agentic AI Foundation(AAIF)** 설립을 발표했습니다. 초기 프로젝트 기증으로 Anthropic의 Model Context Protocol(MCP), Block의 goose Agent 프레임워크, 그리고 AGENTS.md 스펙이 포함됩니다. AI Agent 기반 애플리케이션을 개발하는 팀이라면 이 거버넌스 변화가 처음 보기보다 훨씬 중요한 의미를 가집니다.

## AAIF란 무엇인가

Agentic AI Foundation은 AI Agent가 의존하는 툴링과 표준에 중립적이고 오픈된 거버넌스를 제공하기 위해 Linux Foundation이 설립한 새 조직입니다. Cloud Native 세계에서 CNCF가 맡았던 역할과 같이, AI Agent 에코시스템의 표준화 기반이 될 것으로 기대됩니다. "개별 기업이 관리하는 흥미로운 툴"에서 "커뮤니티가 유지하는 오픈 표준"으로의 전환점이라 할 수 있습니다.

기증되는 주요 프로젝트:

- **Model Context Protocol(MCP)** — Anthropic이 공개한 AI Model을 툴, 데이터 소스, External Service에 연결하는 표준. 이미 2,300개 이상의 Public Server가 있고, Claude, Cursor, VS Code, 그 외 200개 이상의 Client가 네이티브 지원합니다.
- **goose** — Block의 Open Source AI Agent 프레임워크. 로컬에서 코딩 및 엔지니어링 태스크를 자율 실행하도록 설계됩니다.
- **AGENTS.md** — AI Agent의 동작, 제약, Capability를 기계 판독 가능한 형식으로 기술하는 Specification.

## 오픈 거버넌스가 바꾸는 것

MCP가 Anthropic 단독 관리 하에 있을 때 Enterprise 채택에는 합당한 우려가 있었습니다. 단일 벤더가 자사 제품 이익에 맞게 스펙을 바꿀 수 있다는 문제였습니다. Linux Foundation으로의 오픈 거버넌스 이전은 이 리스크를 직접 해소합니다.

CNCF의 사례가 참고가 됩니다. 2016년 Kubernetes가 CNCF로 이전되기 전, Enterprise 팀들은 실질적으로 Google이 관리하는 프로젝트에 프로덕션 Infrastructure를 구축하는 것을 꺼렸습니다. 중립 거버넌스로의 이전이 본격적인 투자와 표준화를 열었고, 현재의 Cloud Native 에코시스템으로 이어졌습니다.

AAIF는 Agentic AI 툴링에 동일한 궤적을 가져옵니다. 지금 MCP 위에 구축하는 팀들은 Anthropic의 로드맵만이 아니라 커뮤니티 Input으로 진화하는 스펙 위에 서게 됩니다.

## MCP 구현에 대한 실질적인 영향

이미 MCP를 사용 중인 팀에게 단기적인 실무 영향은 제한적입니다. 프로토콜이 첫날부터 바뀌는 건 아닙니다. 하지만 장기적 함의는 큽니다:

**멀티 벤더 안정성** — 주요 Cloud Provider와 Tooling 벤더는 자신이 거버넌스에 참여할 수 있는 스펙에 더 깊이 투자합니다. MCP Integration은 더욱 빠르게 확산될 것입니다.

**RFC를 통한 스펙 진화** — 보안 수정, 인증 개선, 새 기능은 Anthropic의 일방적 결정이 아닌 공개 RFC 프로세스를 거쳐 구현됩니다.

**Enterprise 채택 가속** — 대기업 Legal 및 조달 팀은 오픈 거버넌스 표준을 선호합니다. 규제 산업에서의 MCP 채택에서 흔히 나오는 반론이 제거됩니다.

## 주목해야 할 AGENTS.md

MCP보다 덜 논의되지만 AI Agent의 실제 동작을 정의한다는 의미에서 동등하게 중요합니다. Agent의 허용 Action, Tool Access, 범위 제약, 동작 가이드라인을 구조화해 기술하는 방법을 제공합니다:

```markdown
# AGENTS.md

## Identity
name: webhani-deploy-agent
version: 1.0.0
description: webhani 프로덕션 환경의 Deployment 자동화를 담당하는 Agent

## Capabilities
tools:
  - name: read_file
    scope: ./src/**
  - name: run_command
    allowed: ["npm run build", "npm run test"]
    denied: ["rm -rf", "git push --force"]

## Constraints
- ./src와 ./config 외의 파일을 수정하지 않을 것
- 프로덕션 환경 Deploy에는 반드시 Human 승인을 받을 것
- 모든 Tool Invocation을 Audit Log에 기록할 것

## Escalation
on_uncertainty: request_human_review
on_error: halt_and_notify
```

이런 구조화된 Specification으로 Orchestration 프레임워크는 Prompt 지시에 의존하지 않고 제약을 프로그래밍적으로 적용할 수 있습니다. 적대적 조건에서 LLM이 Prompt를 무시하거나 잘못 해석할 리스크를 크게 줄일 수 있습니다.

## 지금 AI Agent 스택을 검토할 이유

AAIF 설립은 현재 또는 계획 중인 AI Agent Infrastructure를 점검하는 좋은 계기입니다:

1. **Tool Connectivity에 MCP를 사용하고 있나요?** 커스텀 Integration Layer를 구축 중이라면, 장기 거버넌스가 명확해진 지금 MCP 전환을 검토할 가치가 있습니다.

2. **제약이 코드로 정의되어 있나요?** Agent 동작에 대한 즉흥적인 Prompt 지시는 취약합니다. AGENTS.md 방식의 Specification으로 감사 가능하고 적용 가능한 정의를 확보하세요.

3. **거버넌스 리스크는?** 규제 산업에 속한다면 Linux Foundation 출처의 표준은 내부 채택 논의에서 실질적인 근거가 됩니다.

4. **어떤 프레임워크가 AAIF에 정렬되어 있나요?** Claude Code, Cursor, Windsurf, VS Code는 이미 MCP를 네이티브 지원합니다. AAIF 멤버십 확대와 함께 goose 주변 에코시스템도 성장할 것입니다.

## 마무리

Agentic AI Foundation은 MCP가 오늘 하는 것을 바꾸는 게 아니라, 그것이 무엇이 될지를 누가 통제하는지를 바꿉니다. 프로덕션 AI Agent 시스템을 구축하는 팀에게 오픈 거버넌스는 장기적인 벤더 종속 리스크를 줄이고, 이 스택 레이어가 진지한 투자에 값하는 성숙 단계에 도달했음을 알립니다. AGENTS.md를 기존 문서와 나란히 두는 First-class Artifact로 취급하기 시작할 때입니다.
