2026년 5월 11일, The Register는 "Local LLM이 Compute 부담을 줄일 준비가 됐다"는 기사를 게재했습니다. 2년 전까지만 해도 "정확도가 낮다", "설정이 너무 복잡하다"는 평가를 받던 Local LLM이 이제 Enterprise Production 환경에서 실질적인 선택지가 됐습니다.

이 변화의 핵심은 모델 소형화와 정확도 향상이 동시에 이뤄졌다는 점입니다. Alibaba Qwen 팀이 출시한 27B Parameter Coding 특화 모델은 단일 A100 GPU(또는 RTX 4090 수준의 Consumer GPU)로 추론이 가능하면서도 Frontier Model에 근접한 성능을 냅니다.

## 기업이 Self-Hosting을 선택하는 세 가지 이유

**비용 예측 가능성.** Cloud API 비용은 사용량에 비례합니다. 코드 리뷰 자동화, 문서 분류, 지원 티켓 처리 등 반복성이 높은 대량 처리 작업에서는 기존 Hardware에서 운영하는 Self-Hosted 모델이 더 경제적일 수 있습니다.

**데이터 주권.** 소스 코드나 고객 데이터를 외부 API로 전송하면 규정 준수 리스크가 생깁니다. 자체 호스팅은 데이터를 조직의 통제 범위 내에 유지합니다.

**레이턴시.** Cloud API 호출에는 네트워크 왕복 비용이 붙습니다. 내부 Network이나 VPC에서 추론하면 이 오버헤드가 없어집니다.

## Inference Stack 설정

Ollama는 Local LLM 추론의 사실상 표준 Runtime입니다. Docker Compose 최소 구성:

```yaml
# docker-compose.yml
services:
  ollama:
    image: ollama/ollama:latest
    ports:
      - "11434:11434"
    volumes:
      - ollama_data:/root/.ollama
    deploy:
      resources:
        reservations:
          devices:
            - driver: nvidia
              count: 1
              capabilities: [gpu]

volumes:
  ollama_data:
```

모델을 Pull하고 테스트합니다:

```bash
docker exec -it ollama ollama pull qwen2.5-coder:32b

curl http://localhost:11434/api/chat -d '{
  "model": "qwen2.5-coder:32b",
  "messages": [{"role": "user", "content": "TypeScript로 fetch 래퍼를 작성해 주세요"}],
  "stream": false
}'
```

## OpenAI 호환 API 활용

Ollama는 `/v1` 경로에서 OpenAI 호환 Endpoint를 제공합니다. 기존 코드의 `baseURL`만 바꾸면 됩니다:

```typescript
import OpenAI from "openai";

const client = new OpenAI({
  baseURL: "http://localhost:11434/v1",
  apiKey: "ollama", // SDK 필수값이지만 Ollama는 검증하지 않음
});

async function chat(message: string): Promise<string> {
  const response = await client.chat.completions.create({
    model: "qwen2.5-coder:32b",
    messages: [{ role: "user", content: message }],
  });
  return response.choices[0].message.content ?? "";
}
```

Production 환경에서는 `localhost` 대신 내부 Hostname이나 Service DNS 이름을 사용합니다.

## 작업 유형별 모델 크기 선택

모든 작업에 최대 모델을 쓸 필요는 없습니다. 작업과 모델 크기를 매칭하면 비용과 성능을 최적화할 수 있습니다:

| 작업 유형 | 권장 모델 크기 | VRAM |
|---|---|---|
| 코드 완성 / Snippet 생성 | 7B~14B | 8~16 GB |
| 문서 요약 / 분류 | 7B~14B | 8~16 GB |
| 코드 리뷰 / 리팩터링 | 32B | 24~32 GB |
| 복잡한 다단계 추론 | 70B+ | 멀티 GPU 또는 Cloud |

반복적인 경량 작업은 Local Model로, 복잡하고 중요한 작업은 Cloud API로 라우팅하는 Hybrid 구성이 현실적입니다.

## 보안 고려사항

Self-Hosting은 운영 책임도 함께 가져옵니다:

- Ollama는 기본적으로 `localhost`에만 Bind됩니다. 내부 Network에 노출할 경우 앞에 인증 Proxy를 추가하세요
- 모델 Weight는 검증된 출처(Ollama 공식 Library, Hugging Face 공식 Organization)에서만 다운로드합니다
- 추론 Container는 별도의 Docker Network로 격리하고 Application Service만 접근 가능하도록 합니다
- 모델 파일도 Dependency처럼 버전을 고정하고 업데이트를 검토한 후 Pull합니다

## 마무리

Local LLM은 이제 실험적 선택지가 아닌 엔지니어링 설계 결정의 하나입니다. 도입 여부는 모델 능력의 한계가 아니라 사용량, 보유 Hardware, 데이터 민감도 요건에 따라 판단해야 합니다.

대량의 반복적 AI 작업을 처리하는 팀이라면 내부 Hardware에서 27B 모델을 운영하는 것을 Prototype해볼 만합니다. CI/CD 특정 Job, 사내 Document 검색, 티켓 분류 같은 범위가 명확한 유스케이스부터 시작하는 것이 좋습니다.
