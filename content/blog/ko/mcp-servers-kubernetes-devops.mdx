Model Context Protocol(MCP)은 AI Assistant 통합에서 Infrastructure Tooling으로 영역을 확장하고 있습니다. 2026년에는 AI Agent가 Kubernetes 클러스터에 직접 접근할 수 있는 DevOps MCP 서버 카테고리가 성장하고 있습니다. "이 Pod는 왜 실패하고 있나요?" 또는 "이번 주 재시작이 가장 많았던 Deployment는?" 같은 자연어 질의가 수동 `kubectl` 명령 없이 가능해지고 있습니다.

이 글에서는 DevOps MCP 서버가 무엇인지, 기존 Kubernetes 워크플로우에 어떻게 맞는지, 그리고 이 패턴이 실제로 가치를 발휘하는 상황과 주의해야 할 상황을 정리합니다.

## MCP와 DevOps에서의 의미

MCP는 AI Agent가 외부 Tool이나 데이터 소스에 연결하는 방법을 표준화하는 Open Protocol입니다. MCP 서버는 AI Model이 대화나 Agentic 워크플로우 중에 호출할 수 있는 일련의 Operation(Tool Call과 Resource Read)을 노출합니다.

Kubernetes에 적용하면, MCP 서버는 Kubernetes API에 대한 Structured Interface로 기능합니다. `kubectl` 명령을 직접 작성하거나 YAML Manifest를 수동으로 파싱하는 대신, AI Agent는 `list_pods`, `get_events`, `describe_deployment`, `get_logs` 같은 MCP Tool을 호출하고 서버가 API 상호작용을 처리합니다.

가치는 편의성만이 아닙니다. AI Model이 클러스터 상태를 추론 능력과 결합할 수 있다는 점입니다. "이 Pod는 왜 실패하고 있나요?"라고 물으면, MCP 서버가 관련 Event, Log, Resource Status를 가져옵니다. Model은 그 데이터를 통합해 일관된 진단을 생성합니다 — 일반적으로 여러 `kubectl` 명령과 수동 상관관계 분석이 필요한 작업입니다.

## 실제 워크플로우 예시

클러스터에 연결된 MCP 서버를 전제로, Claude Code 등의 클라이언트 내에서의 워크플로우는 다음과 같습니다:

```
User: staging의 api-service Deployment가 20분째 Crashloop 중입니다. 무슨 일인가요?

Model: [get_pod_events("api-service-*", namespace="staging") 호출]
       [get_pod_logs("api-service-7f8d9b", namespace="staging", tail=50) 호출]
       [describe_deployment("api-service", namespace="staging") 호출]

Model: api-service Pod가 OOMKilled 신호로 실패하고 있습니다.
       Memory Request는 256Mi로 설정되어 있지만,
       최근 3번의 재시작 사이클에서 일관되게 400Mi를 초과하고 있습니다.
       23분 전 최신 Rollout에서 Resource Limit이 강화되었습니다.
       권장사항: Memory Limit을 512Mi로 늘리거나 메모리 사용량 증가를 Profiling하세요.
```

OOMKill 신호, Log 출력, 최근 Deployment 변경을 연결하는 이 상관관계 분석은 일반적으로 수분의 수동 조사가 필요합니다. MCP 패턴은 이를 하나의 Prompt로 줄여줍니다.

## Kubernetes MCP 서버 Setup

여러 Open Source Kubernetes MCP 서버를 사용할 수 있습니다. Helm 기반 서버를 활용한 일반적인 Setup:

```bash
helm repo add mcp-k8s https://charts.mcp-k8s.io
helm install mcp-k8s mcp-k8s/mcp-k8s \
  --namespace mcp-system \
  --create-namespace \
  --set rbac.readOnly=true
```

MCP 클라이언트(Claude의 `mcp_settings.json` 등)에서 설정:

```json
{
  "mcpServers": {
    "kubernetes": {
      "command": "kubectl",
      "args": ["mcp-proxy", "--namespace", "production"],
      "env": {
        "KUBECONFIG": "/path/to/kubeconfig"
      }
    }
  }
}
```

## Access Control: Read-Only 부터 시작

가장 중요한 운영상의 고려사항은 Access Scope입니다. 전체 클러스터 Write 권한을 가진 MCP 서버를 AI Agent에 연결하면 상당한 위험이 생깁니다 — Model이 Namespace 삭제, Deployment Scale Down, RBAC 수정을 수행하도록 유도될 수 있습니다.

안전한 기본 패턴:

1. MCP 서버의 Service Account에 Read-Only RBAC부터 시작
2. 사용 사례와 관련된 특정 Namespace로 Scope를 제한
3. 저위험 작업(Replica Scale, Deployment 재시작)에만 Write Operation을 추가하고 명시적 확인 단계를 설정
4. 파괴적인 작업(삭제, Namespace 전체 변경)은 MCP 서버에서 완전히 제외

```yaml
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: mcp-k8s-readonly
rules:
  - apiGroups: ["", "apps", "batch"]
    resources: ["pods", "deployments", "services", "events", "replicasets", "jobs"]
    verbs: ["get", "list", "watch"]
  - apiGroups: [""]
    resources: ["pods/log"]
    verbs: ["get"]
```

## 이 패턴이 효과를 발휘하는 상황

운영 조사 사용 사례가 가장 명확한 성과를 낼 수 있습니다: Crashloop 진단, Rollout 실패 파악, 리소스 간 Event 상관관계 분석. 이들은 Read 중심의 시간 제약이 있는 Task이며, 클러스터 상태와 추론을 결합하는 능력이 진정한 가치를 발휘합니다.

Resource Utilization Review도 강력한 사용 사례입니다 — 실제 사용 데이터를 기반으로 리소스가 부족하거나 과도하게 Provisioning된 Workload를 식별하고 조정안을 제안하게 하는 방식입니다.

Onboarding과 Documentation은 부드러운 성과입니다: 신규 팀원이 `kubectl`의 전체 표면을 즉시 배우지 않아도 자연어로 클러스터 상태를 조회할 수 있습니다.

## 주의해야 할 상황

명시적 확인 없는 자동 Write Operation — 안전해 보이는 것도 — 은 공유 클러스터에서 상당한 위험을 가집니다. 현재로서의 적절한 경계는 AI를 Intelligent Query Interface로 사용하고 변경 사항은 사람이 승인하는 것입니다.

프로덕션 환경의 MCP 서버 호출 Audit Logging은 필수입니다. 사용자, 작업, 파라미터를 포함한 모든 Tool 호출을 로그에 기록해야 합니다 — 권한 있는 API 접근에 적용하는 동일한 기준으로요.

## 요약

- DevOps MCP 서버는 AI Agent에게 Kubernetes 클러스터 상태에 대한 Structured Access를 제공합니다
- 주요 가치는 Event, Log, Resource 상태를 상관관계 분석해 실행 가능한 진단을 생성하는 것
- Read-Only RBAC와 Namespace-Scope Access부터 시작하고, Write Operation은 확인 단계와 함께 점진적으로 추가
- MCP Tool 호출의 Audit Logging은 프로덕션 클러스터 사용 전 필수
- 이 패턴은 조사 시간을 크게 단축하지만, 수정 대응에 대한 운영자 판단을 대체하지는 않음
