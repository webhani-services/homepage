지금 대부분의 MCP 서버는 REST endpoint를 만들 때와 똑같은 방식으로 설계되어 있습니다. Client가 request를 보내면 서버가 작업을 처리하고 response를 돌려주는 구조입니다. 파일을 읽거나 database를 조회하는 정도라면 이 방식으로 충분합니다. 문제는 어떤 tool 하나가 20분씩 걸리는 작업을 해야 할 때 시작됩니다.

MCP steering committee가 발표한 2026년 로드맵은 이 문제를 정면으로 다룹니다. 겉보기엔 소박해 보이지만 실제로는 MCP 서버 architecture를 근본적으로 바꾸는 두 가지 변화입니다. 장시간 실행되는 작업을 위한 **Tasks primitive**, 그리고 **stateless HTTP transport** variant입니다. 저희도 클라이언트 프로젝트에서 MCP 연동을 구축하면서 이미 부딪혔던 문제들이라 계속 지켜보고 있었습니다.

## 동기식 Request/Response가 Agent Workload에 맞지 않는 이유

HTTP request에는 암묵적인 계약이 하나 있습니다. 서버가 응답할 때까지 connection이 열려 있어야 하고, 그 응답은 몇 분이 아니라 몇 초 안에 온다는 전제입니다. MCP는 JSON-RPC over stdio와 SSE에서 이 구조를 그대로 물려받았고, "이 record 찾아줘", "이 JSON 포맷 맞춰줘" 같은 tool call에는 합리적인 기본값입니다.

하지만 agent가 실행하려는 작업은 점점 이 계약에 들어맞지 않는 경우가 많아지고 있습니다. Model training, 여러 단계로 이루어진 data pipeline, 응답이 느린 서드파티 batch API 대기, 대용량 문서 crawling 같은 작업들입니다. 이런 작업을 단일 동기식 `tools/call`로 모델링하려고 하면 다음 세 가지 나쁜 결과 중 하나로 귀결됩니다.

- Client가 timeout 되어 실제로는 job이 계속 돌고 있는데도 agent는 tool이 실패했다고 오판합니다.
- 20분 동안 connection을 붙잡고 있어야 하는데, 이는 네트워크 장애에 취약하고 connection을 열어둔 만큼 compute 비용도 듭니다.
- MCP 위에 자체적인 polling 방식을 즉흥적으로 만들게 되는데, 이러면 표준 protocol을 쓰는 의미가 없어집니다.

상시 작동하는 persistent agent라면 문제가 더 심각해집니다. 몇 시간 동안 무인으로 돌아가야 하는 agent는 tool call 하나 때문에 block되는 것을 감당할 수 없습니다. 작업을 던져놓고 다른 일을 하다가 나중에 결과를 확인할 수 있어야 합니다.

## Tasks Primitive가 서버 설계에 실제로 바꾸는 것

Tasks primitive는 장시간 실행되는 작업을 단일 request/response 쌍이 아니라, 자체 lifecycle을 가진 하나의 object로 취급합니다. 개념적인 흐름은 이렇습니다.

```typescript
// 1. Client가 작업을 던지고 즉시 task handle을 돌려받습니다
const { taskId } = await mcpClient.callTool("run-data-pipeline", {
  source: "s3://raw-events/2026-07-09/",
  transform: "dedupe-and-enrich",
});

// 2. Block 하는 대신 polling 합니다 (client 또는 agent의 scheduler가 수행)
async function waitForTask(taskId: string) {
  while (true) {
    const status = await mcpClient.getTaskStatus(taskId);

    if (status.state === "completed") return status.result;
    if (status.state === "failed") throw new Error(status.error);

    // 아직 "running" 이거나 "queued" — 잠시 대기 후 다시 확인
    await sleep(status.suggestedPollIntervalMs ?? 5000);
  }
}
```

중요한 건 polling loop 자체가 아니라, 이게 서버 설계에 무엇을 요구하느냐입니다. Production 수준의 MCP 서버라면 다음이 필요해집니다.

- Task 상태를 durable 하게 추적하는 방법 (데모라면 in-memory로 충분하지만, production에서는 서버 재시작 이후에도 살아남는 database row나 queue entry가 필요합니다)
- Client가 한 시간 뒤에 재접속해도 "task `abc123` 어떻게 됐어?"라고 물을 수 있는 식별자 체계
- Cancel 경로 — agent가 마음을 바꾸거나 실행 도중 강제 종료될 수 있기 때문입니다
- 합리적인 status 의미 체계 (`queued`, `running`, `completed`, `failed` — 장시간 작업이라면 부분 진행률 보고까지 있으면 이상적입니다)

이건 "얇은 RPC wrapper로서의 MCP 서버"에서 "작은 job orchestration 시스템으로서의 MCP 서버"로의 실질적인 전환입니다. 이미 Celery, Sidekiq, 클라우드 task queue 같은 background job 인프라를 운영 중이라면 Tasks는 그 위에 얇은 adapter를 얹는 정도로 끝날 수 있습니다. 아직 그런 인프라가 없다면, 지금이 하나 갖춰야 할 시점입니다.

## Stateless HTTP — 다른 서비스처럼 MCP 서버를 Scaling 합니다

두 번째 축은 stateless HTTP transport 옵션입니다. 현재 많은 MCP 서버는 Server-Sent Events를 사용하는데, 이는 client마다 서버가 persistent connection을 유지해야 한다는 뜻입니다. 규모가 작을 땐 문제없지만, load balancer가 sticky session을 처리해야 하고, 서버는 연결된 agent 수만큼 file descriptor를 붙잡고 있어야 하며, horizontal scaling이 까다로워집니다. 한 instance가 처리 중인 SSE session을 다른 instance가 가로챌 수 없기 때문입니다.

Stateless HTTP는 각 request가 처리에 필요한 모든 정보를 스스로 담고 있어서, call 사이에 connection이나 in-memory session을 유지할 필요가 없다는 뜻입니다. 개념적인 handler는 평범한 HTTP endpoint와 같은 모양입니다.

```typescript
// Stateless MCP HTTP handler — session object도, 붙잡고 있는 connection도 없습니다
export async function handleMcpRequest(req: Request): Promise<Response> {
  const rpcMessage = await req.json();

  // Task 상태, resource cache 같은 state가 필요하면
  // 서버 메모리가 아니라 외부 store에서 조회합니다
  const result = await routeMcpMethod(rpcMessage, {
    store: sharedTaskStore, // 예: Redis, Postgres — process 메모리가 아닙니다
  });

  return Response.json(result);
}
```

어떤 instance도 특별하지 않기 때문에 표준 load balancer가 이 request들을 round-robin으로 여러 instance에 분산시킬 수 있습니다. 이를 Tasks primitive와 결합하면 깔끔한 패턴이 나옵니다. Client가 어느 instance로 task를 던지든, 다음 polling을 다른 instance가 받아도 상관없습니다. Task 상태가 특정 process가 아니라 공유 storage에 있기 때문입니다.

이건 10년 전 web backend가 sticky session 기반 app server에서 load balancer 뒤의 stateless service로 넘어가면서 배웠던 것과 같은 교훈입니다. MCP가 agent tooling 영역에서 이제야 그걸 따라잡고 있는 셈입니다.

## webhani의 관점 — 언제 도입하고, 언제 도입하지 말아야 할까

모든 팀에게 "내일 당장 Tasks와 stateless HTTP로 MCP 서버를 재작성하세요"라고 말하지는 않습니다. 실제로 구축하면서 얻은 몇 가지 실무 기준을 정리하면 다음과 같습니다.

**Tool이 정말로 빠르다면 단순하게 유지하세요.** 서버 안의 모든 tool call이 몇 초 안에 끝난다면 — CRUD 작업, 단순 조회, 간단한 변환 등 — 동기식 request/response가 여전히 맞는 기본값입니다. 필요 없는 작업에 task polling 메커니즘을 얹는 건 순전한 overhead일 뿐입니다.

**Client의 timeout을 넘길 가능성이 있는 작업이 있다면 Tasks를 도입하세요.** 특히 latency를 예측할 수 없는 외부 API 호출, batch 처리, 앞 단계가 느린 upstream 시스템에 의존하는 multi-step chain이 그 대상입니다. 긴 호출을 버티려고 `setTimeout` 기반 keep-alive나 수동 retry logic을 이미 만들고 있다면, 그게 바로 신호입니다.

**실제로 horizontal scaling이 필요한 시점에 stateless HTTP로 전환하세요.** 미리 선제적으로 할 필요는 없습니다. 단일 instance로 돌아가는 사내 도구라면 필요 없습니다. 여러 팀이 공유하는 MCP 서버, 또는 의미 있는 수준의 request 양을 처리하는 서버라면 필요합니다. 그리고 in-memory session state를 이미 잔뜩 쌓아 놓은 뒤 나중에 손보는 것보다, 처음부터 stateless로 설계하는 편이 훨씬 쉽습니다.

**Task storage layer는 필요해지기 전에 미리 정해두세요.** Tasks primitive 자체는 task 상태를 어떻게 영속화할지 규정하지 않습니다. 그건 구현하는 쪽의 몫입니다. Redis TTL 기반 store로 할지, Postgres table로 할지, 기존 job queue의 status table을 재사용할지 초반에 결정해두세요. Migration 도중 agent가 실행 중인 job을 놓친 다음에야 이걸 설계하는 건 좋은 타이밍이 아닙니다.

**스펙을 계속 지켜보되, 방어적으로 구현하세요.** 로드맵 항목들은 아직 확정되지 않았고, SDK 지원도 언어별로 고르지 않게 들어올 겁니다. 지금 구현한다면, transport와 task 추적 logic을 자체 interface 뒤에 숨겨두세요. 그러면 공식 primitive가 확정된 뒤에도 tool handler를 다시 쓰지 않고 교체할 수 있습니다.

두 변화 모두를 관통하는 흐름은 하나입니다. MCP가 "하나의 assistant가 몇 개의 tool을 호출하는 protocol"에서 "production 수준 agent infrastructure를 위한 protocol"로 성숙해지고 있다는 것입니다. 방향 자체는 좋지만, 그만큼 우리가 만드는 서버도 단순한 request handler가 아니라 진짜 backend service에 가까운 모습을 갖춰야 한다는 뜻이기도 합니다.
