## PostgreSQL 일변도 논의에서 한 걸음 물러서서

지난 몇 년간 이 분야의 논의는 PostgreSQL의 신기능 — 비동기 I/O, UUIDv7, OAuth 인증 지원 등 — 에 치우치는 경향이 있었습니다. 실제로 확장성이나 Modern 기능 추가 속도감에서 PostgreSQL이 화제의 중심에 있는 것은 사실입니다. 다만 기존 자산으로 MySQL을 계속 쓰는 조직도 많고, 그 진화를 놓치지 않고 "PostgreSQL 일변도"로 결론짓는 것은 현장 감각과 어긋납니다.

MySQL 9.3(Innovation Release)은 화려함은 없지만 JavaScript Stored Routine 확장과 Optimizer 개선 등 실무에 효과가 있는 변경을 포함합니다. 한편 놓치기 쉬운 운영상의 주의점도 존재합니다. webhani가 이번 Innovation Release를 실무 관점에서 정리해보겠습니다.

## JavaScript Stored Routine 확장

MySQL은 비교적 최근 JavaScript로 Stored Procedure·Function을 작성할 수 있는 기능을 도입했습니다. 9.3에서는 이 기능이 더 확장되어, JavaScript Routine 내에서 DECIMAL Type을 완전히 지원하게 되었습니다. 금액 계산처럼 정밀도가 중요한 처리를 JavaScript Routine으로 다룰 때의 장벽이 하나 줄어든 셈입니다.

또한 `ALTER PROCEDURE` / `ALTER FUNCTION`에 `USING` 절이 추가되어, JavaScript Stored Routine이 Import하는 Library의 추가·교체·삭제를 Routine 자체를 다시 만들지 않고도 할 수 있게 되었습니다. `CREATE LIBRARY`에도 `COMMENT` 절이 추가되어 `SHOW CREATE LIBRARY`나 `SHOW LIBRARY STATUS` 출력에서 내용을 확인할 수 있습니다.

```sql
-- JavaScript Routine이 의존하는 Library를 교체하는 예시
ALTER PROCEDURE calculate_tax
  USING REPLACE LIBRARY tax_rules_v1 WITH tax_rules_v2;

-- Library에 Comment를 달아 이후 용도 추적을 쉽게 한다
CREATE LIBRARY tax_rules_v2
  LANGUAGE JAVASCRIPT
  COMMENT '2026년도 개정판 세율 계산 로직'
  AS $$
    export function calc(amount) { /* ... */ }
  $$;
```

Library 관리가 Version 관리하기 쉬워진 점은, Business Logic 일부를 Database 계층에 두는 조직에게 눈에 띄지 않지만 중요한 개선입니다.

## Optimizer 개선: Subquery 처리 범위 확대

9.3에서는 `subquery_to_derived`라는 최적화가 확장되어, `>ANY` `>=ANY` `<ANY` `<=ANY`, `>ALL` `>=ALL` `<ALL` `<=ALL`, `=ANY` `<>ALL` 같은 양화 비교 연산자 전체를 지원하게 되었습니다. 더불어 이 변환이 `SELECT` 절과 `WHERE` 절 모두에서 사용 가능해졌습니다.

```sql
-- 양화 비교를 포함한 Subquery 예시
-- 9.3에서는 이런 형태의 Query도 Derived Table 변환 대상이 될 수 있다
SELECT product_id, price
FROM products
WHERE price >ALL (
  SELECT price FROM products WHERE category = 'discontinued'
);
```

이런 변환은 실행 계획이 Hash Join 등 더 효율적인 결합 방식을 선택하기 쉬워진다는 뜻입니다. 기존 Query를 다시 쓰지 않아도 Upgrade만으로 실행 계획이 개선될 가능성이 있다는 점은 운영 비용 관점에서 놓칠 수 없습니다.

## 놓치기 쉬운 함정: Innovation Series 간 Downgrade 불가

이번 변경에서 실무적으로 가장 주의해야 할 것은 기능 추가가 아니라 제약 추가입니다. MySQL 9.3부터 Innovation Series의 개별 Release 간 Downgrade가, 같은 Series 내에서도 불가능해졌습니다.

지금까지 검증 환경에서 문제가 발견되면 이전 Version으로 되돌리는 운영을 취한 Team도 있을 것입니다. 9.3부터는 이 선택지가 사실상 사라집니다. Upgrade 전 Snapshot 확보와, 되돌릴 수 없다는 전제하의 Rollout 계획 재검토가 필수가 됩니다.

## webhani의 권장 사항

- **Production 반영 전 Staging 환경에서 Full Backup을 필수화한다**: Downgrade 불가라는 제약이 있는 이상, "일단 적용하고 문제가 있으면 되돌린다"는 운영은 통하지 않습니다.
- **JavaScript Stored Routine을 사용 중이라면 Library 관리를 재점검한다**: `USING` 절로 Library 교체가 가능해진 만큼, Routine의 의존 관계를 명시적으로 관리하는 운영으로 옮겨갈 좋은 기회이기도 합니다.
- **기존의 복잡한 Subquery를 포함한 Query의 실행 계획을 확인한다**: `subquery_to_derived` 확장으로 `EXPLAIN` 결과가 바뀌는 Query가 나올 수 있습니다. Performance 회귀가 없는지 Upgrade 후 주요 Query를 확인해야 합니다.
- **Innovation Release와 LTS 운영 방침을 재확인한다**: 최신 기능을 좇을지 안정성을 우선할지, 조직 방침에 따라 Innovation Series를 따라갈 속도를 정해 두어야 합니다.

## 정리

MySQL 9.3은 JavaScript Stored Routine의 실용성 향상과 Optimizer의 꾸준한 개선을 포함한, 실무에 효과 있는 Update입니다. 다만 Innovation Series 간 Downgrade가 불가능해졌다는 제약은 Upgrade 운영의 전제를 바꿉니다. 화려한 신기능에 시선을 빼앗기지 말고 Rollout 계획과 Backup 체계를 먼저 재점검하는 것이, 이번 Release를 안전하게 받아들이는 첫걸음입니다.
