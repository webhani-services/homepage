2026년 3월 18일 Next.js 16.2가 릴리즈됐습니다. 새로운 추상화나 대규모 API 변경보다는 개발자가 매일 체감할 수 있는 성능 개선에 집중한 업데이트입니다. `next dev` 시작 속도 최대 400% 향상, 렌더링 속도 약 50% 개선이라는 수치가 눈에 띄는데, 실제로 프로젝트 규모가 클수록 효과가 두드러집니다.

## 개발 서버 시작 속도: 왜 빨라졌나

핵심은 Lazy Compilation 도입입니다. 기존에는 `next dev` 실행 시 모든 Route를 파싱하고 번들링한 뒤 서버를 시작했지만, 16.2부터는 실제로 접근된 Route만 그 시점에 컴파일합니다. 덕분에 초기 시작은 훨씬 빨라졌고, 처음 방문하는 Route는 약간의 추가 지연이 생기는 트레이드오프가 있습니다.

```bash
# 16.1 — 약 150개 페이지 프로젝트 기준
$ next dev
▶ Next.js 16.1
Starting... 23.8s

# 16.2 — 동일한 프로젝트
$ next dev
▶ Next.js 16.2
Starting... 5.7s
```

소규모 프로젝트에서는 차이가 크지 않을 수 있지만, 페이지 수가 많은 프로젝트를 운영하는 팀이라면 즉시 체감할 수 있습니다.

## 렌더링 속도 50% 향상

SSR 렌더링 시간이 약 50% 개선됐습니다. Next.js와 React의 Streaming 처리 연동이 최적화되면서 TTFB(Time to First Byte)가 전반적으로 낮아졌습니다. 특히 여러 Suspense 경계를 사용하는 페이지에서 효과가 두드러집니다.

```tsx
// app/dashboard/page.tsx
import { Suspense } from 'react'
import { DashboardStats } from './stats'
import { RecentActivity } from './activity'

export default function Dashboard() {
  return (
    <div>
      <Suspense fallback={<StatsSkeleton />}>
        <DashboardStats />
      </Suspense>
      <Suspense fallback={<ActivitySkeleton />}>
        <RecentActivity />
      </Suspense>
    </div>
  )
}
```

이런 구조에서는 초기 HTML Shell이 더 빨리 클라이언트에 전달되어 JavaScript 실행 전에도 사용자가 인식하는 로딩 속도가 개선됩니다.

## Turbopack Server Fast Refresh

개발 경험에 가장 직접적인 영향을 주는 변경입니다. Turbopack이 이제 Server Component 변경에도 Fast Refresh를 지원합니다. 이전에는 Server Component를 수정하면 페이지 전체가 리로드됐지만, 16.2부터는 변경된 컴포넌트만 업데이트됩니다.

```tsx
// app/components/ServerWidget.tsx (Server Component)
// 이 파일을 수정하면 페이지 전체 리로드 없이 변경 사항이 반영됩니다
export async function ServerWidget() {
  const data = await fetchData()
  return <div className="widget">{data.value}</div>
}
```

복잡한 상태 의존성이 있는 경우 전체 리로드가 발생하는 경우도 있지만, 대부분의 일상적인 편집 작업에서는 Client Component 수정과 비슷한 빠른 피드백을 받을 수 있습니다.

## AGENTS.md 지원

`create-next-app`으로 생성된 프로젝트에 이제 `AGENTS.md` 파일이 포함됩니다. 이 파일은 Claude Code, Cursor, Codex 등 AI 코딩 도구가 프로젝트 컨텍스트와 규약을 이해하는 데 사용합니다.

```markdown
# AGENTS.md

## 프로젝트 개요
Next.js 16.2 App Router 기반 애플리케이션

## 코딩 규약
- Server Component를 기본으로 사용하고, 'use client'는 필요한 경우만 추가
- cookies(), headers(), params는 반드시 await 사용
- 데이터 변경은 Server Action 사용 (Route Handler 지양)

## 디렉토리 구조
- `app/` — 라우팅 및 레이아웃
- `components/` — 공통 UI
- `lib/` — 유틸리티 및 데이터 접근
```

이미 `CLAUDE.md`나 `.cursorrules`를 관리하고 있다면, `AGENTS.md`와의 역할 중복을 정리해두는 것이 좋습니다. 도구마다 다른 지침을 받으면 생성 코드의 일관성이 깨질 수 있습니다.

## Next.js Across Platforms

3월 25일, Vercel이 "Next.js Across Platforms"를 발표했습니다. 안정적인 Adapter API 제공, 공유 테스트 스위트 구성, OpenNext 등 커뮤니티 프로젝트와의 협력을 통해 AWS Lambda, Cloudflare Workers, 기타 Edge 환경에서의 Next.js 동작 품질을 개선하겠다는 내용입니다.

Vercel 이외 환경에 배포하는 팀에게는 지속 가능한 공식 인터페이스가 생겼다는 점에서 중요한 변화입니다.

## 업그레이드 전 확인 사항

**비동기 Request API**: 16.0부터 필수인 `await cookies()`, `await headers()`, `await params` 패턴이 아직 적용되지 않은 코드가 있다면 먼저 정리해야 합니다.

```ts
// 16.x에서 필수
import { cookies } from 'next/headers'

export async function GET() {
  const cookieStore = await cookies()
  const token = cookieStore.get('token')
  // ...
}
```

**Turbopack 설정 위치**: `next.config.ts` 최상위에 배치해야 합니다. `experimental.turbopack`은 Deprecated 됐습니다.

```ts
// next.config.ts
import type { NextConfig } from 'next'

const config: NextConfig = {
  turbopack: {
    resolveAlias: {
      '@': './src',
    },
  },
}

export default config
```

**Type 체크 강화**: 16.2에서 일부 Type 검사가 엄격해졌습니다. 업그레이드 후 `tsc --noEmit`을 실행해 새로운 타입 에러가 없는지 확인하세요.

## 정리

Next.js 16.2는 추가 학습 비용 없이 즉시 체감할 수 있는 개선들로 구성된 업데이트입니다. 개발 서버 시작 속도 향상은 대형 프로젝트에서 가장 두드러지며, Turbopack Server Fast Refresh는 Server Component 개발 시 반복 주기를 눈에 띄게 단축시킵니다.

이미 16.x를 사용 중이라면 비동기 API 사용 현황과 Turbopack 설정 위치를 확인하고, 타입 체크를 한 번 돌려보는 것으로 업그레이드 준비는 충분합니다.
