Next.js 16.2가 2026년 3월 18일에 릴리즈되었습니다. 이번 버전의 핵심은 `next dev` 시작 속도 약 400% 향상과 렌더링 속도 약 50% 개선입니다. 성능 개선 외에도 AI 기반 개발 워크플로우를 위한 새로운 도구들이 추가되었습니다.

## 주요 변경사항

### 1. dev 서버 시작 속도 개선

Turbopack의 모듈 분석 파이프라인이 전면 개선되면서 초기 컴파일 시 의존성 해석이 병렬화되었습니다. 500개 이상의 모듈을 가진 대형 프로젝트에서도 몇 초 안에 개발 서버가 시작됩니다.

실무에서 dev 서버 시작에 20초가 걸렸다면, 이제 5초 이내로 줄어드는 수준입니다. 하루 종일 누적되면 상당한 시간을 절약할 수 있습니다.

### 2. Agent-ready `create-next-app`

AI Coding Assistant가 프로젝트 구조를 빠르게 파악할 수 있도록 `--agent-ready` 옵션이 추가되었습니다:

```bash
npx create-next-app@latest my-app --agent-ready
```

이 옵션을 사용하면 `agent.json`(프로젝트 구조 요약)과 `.agent/context.md`(주요 아키텍처 결정사항)가 자동으로 생성됩니다. Claude Code나 GitHub Copilot Workspace 같은 도구가 이 파일을 읽어 탐색 과정 없이 바로 의미 있는 코드 제안을 할 수 있게 됩니다.

### 3. 브라우저 로그 전달 기능

개발 중 브라우저 콘솔 출력을 터미널로 전달할 수 있게 되었습니다:

```typescript
// next.config.ts
const nextConfig = {
  experimental: {
    browserLogForwarding: true,
  },
};
```

AI Agent가 개발 세션을 진행할 때 클라이언트 사이드 에러를 서버 로그와 동일한 스트림에서 확인할 수 있어, 별도의 브라우저 접근 없이도 더 완전한 디버깅 컨텍스트를 제공받을 수 있습니다.

### 4. 실험적 Agent DevTools

`--experimental-agent-devtools` 플래그로 dev 서버를 시작하면 AI Agent가 Next.js 내부 상태를 실시간으로 조회할 수 있는 IPC 엔드포인트가 활성화됩니다:

```bash
next dev --experimental-agent-devtools
```

라우팅 트리, Server/Client Component 경계, 캐시 상태 등을 조회할 수 있습니다. 아직 실험 단계이지만, AI 기반 개발 루프를 단축할 수 있는 가능성을 보여줍니다.

## Turbopack 개선사항

### Server Fast Refresh

기존 HMR은 클라이언트 사이드에만 적용되었지만, Server Fast Refresh를 통해 서버 사이드 모듈도 세밀하게 리로드할 수 있게 되었습니다. Server Component를 편집할 때마다 전체 페이지가 리로드되던 문제가 해결됩니다.

### Web Worker에서 WASM 지원 강화

Workers 내에서 WASM 라이브러리 사용을 막던 오리진 제약이 완화되었습니다. `crypto-wasm`, `@tensorflow/tfjs-backend-wasm` 같은 라이브러리를 Web Worker 내에서 추가 설정 없이 사용할 수 있습니다.

### JavaScript에 대한 Subresource Integrity(SRI) 지원

생성된 JavaScript 청크에 SRI 해시가 자동으로 추가됩니다:

```html
<script
  src="/_next/static/chunks/main.js"
  integrity="sha256-..."
  crossorigin="anonymous"
></script>
```

CDN을 통해 서비스를 제공하는 경우 스크립트 변조 위험을 줄일 수 있습니다. 엄격한 CSP 요구사항이 있는 엔터프라이즈 프로젝트에 특히 유용합니다.

### Dynamic Import 트리 쉐이킹

동적으로 임포트된 모듈에서 사용하지 않는 익스포트가 빌드 시 제거됩니다. 이전에는 동적 임포트가 불투명한 번들로 처리되었지만, 이제 각 모듈에서 실제로 사용하는 부분만 포함됩니다. 많은 `import()` 호출을 가진 대형 앱에서 번들 크기 감소 효과를 기대할 수 있습니다.

### 500 에러 페이지 재설계

기본 500 에러 페이지가 새롭게 디자인되었습니다. 개발 환경에서는 소스맵이 적용된 스택 트레이스가 표시되고, 프로덕션 환경에서는 내부 정보를 노출하지 않는 깔끔한 사용자 메시지가 표시됩니다.

## 마이그레이션 시 주의사항

16.1에서 16.2로의 마이그레이션에는 파괴적 변경사항이 없습니다. 다음 두 가지만 확인하세요:

**1. Turbopack 설정 위치 변경**

`experimental.turbopack`은 16.2부터 완전히 무시됩니다. 설정을 최상위 `turbopack` 키로 이동해야 합니다:

```typescript
// 이전 방식 (16.0 이전)
const nextConfig = {
  experimental: {
    turbopack: { /* ... */ },
  },
};

// 올바른 방식 (16.0+, 16.2부터 필수)
const nextConfig = {
  turbopack: { /* ... */ },
};
```

**2. Server Fast Refresh 충돌 확인**

커스텀 HMR 로직이 있다면 업그레이드 후 테스트가 필요합니다. 필요한 경우 비활성화할 수 있습니다:

```typescript
const nextConfig = {
  turbopack: {
    hotUpdate: {
      serverComponents: false,
    },
  },
};
```

## webhani의 관점

이번 릴리즈에서 가장 즉각적인 실무 가치는 dev 서버 시작 속도 개선입니다. 특히 페이지 수가 많은 대형 프로젝트에서 체감 효과가 큽니다.

SRI 지원은 조용하지만 중요한 보안 강화입니다. CDN을 통해 Next.js 앱을 서비스하고 있다면 이 기능을 활성화하는 것을 권장합니다. Agent DevTools는 아직 실험 단계이지만 AI 코딩 어시스턴트를 적극적으로 활용하는 팀이라면 지금부터 테스트해볼 가치가 있습니다.

업그레이드는 `npm install next@16.2.0`으로 간단히 진행할 수 있습니다.
