## '빠름'이 아니라 '즉각'을 목표로

2026년 6월 29일에 릴리스된 Next.js 16.3은 Turbopack의 메모리 절감과 빌드 캐시가 주목받았지만([Next.js Blog](https://nextjs.org/blog/next-16-3-turbopack)), 실제 사용자 경험에 직결되는 것은 'Instant Navigation' 개선입니다. 더 똑똑한 Prefetch와 streaming으로 링크를 클릭한 순간 화면이 반응하는 것처럼 느껴지게 하는 방향입니다.

여기서 중요한 것은 '빠름'과 '즉각'의 차이입니다. 페이지 전체 로딩이 1초에 끝나는 것과, 클릭한 순간 무언가 움직이는 것은 체감상 완전히 다른 이야기입니다. 사람의 지각에서는 100밀리초 이내에 반응이 있으면 "직접 조작하고 있다"고 느끼고, 그것을 넘으면 "기다리게 된다"고 느끼기 시작합니다. Instant Navigation은 이 지각의 경계를 의식한 설계입니다.

이 글에서는 이 구조를 떠받치는 Prefetch와 streaming의 개념, 그리고 Next.js를 쓰는 개발자로서 무엇에 주의해야 하는지를 정리합니다.

## Prefetch: 클릭 전에 준비를 끝낸다

Next.js의 App Router에서는 `<Link>` 컴포넌트가 뷰포트에 들어오면 이동할 곳의 데이터를 미리 가져오려 합니다. 사용자가 아직 클릭하지 않은 단계에서 필요한 것을 갖춰 두기 때문에, 실제로 클릭했을 때는 표시에 필요한 재료가 이미 손에 있다는 발상입니다.

```tsx
import Link from "next/link";

// 뷰포트에 들어온 시점에 prefetch가 실행된다
export function ArticleCard({ slug, title }: { slug: string; title: string }) {
  return (
    <Link href={`/blog/${slug}`} prefetch>
      <article className="card">
        <h2>{title}</h2>
      </article>
    </Link>
  );
}
```

16.3 개선의 핵심은 이 Prefetch가 '더 똑똑해졌다'는 점입니다. 모든 것을 일률적으로 미리 읽는 것이 아니라, 무엇을 우선해 가져올지 판단이 정교해져 불필요한 통신을 줄이면서 체감 속도를 높이는 방향으로 나아가고 있습니다.

다만 Prefetch가 만능은 아닙니다. 뷰포트에 링크가 많은 목록 페이지에서 전부를 미리 읽으면 오히려 대역폭을 압박합니다. 저희가 구현을 리뷰할 때는 목록의 각 카드에서 무조건 prefetch를 활성화하지 않았는지 반드시 확인합니다.

## Streaming: 준비된 부분부터 먼저 보여준다

또 하나의 축이 streaming입니다. 페이지 전체 데이터가 갖춰지길 기다리지 않고, 준비된 부분부터 순서대로 HTML을 내보내는 구조로, React Server Components와 Suspense의 조합으로 구현합니다.

```tsx
import { Suspense } from "react";

export default function BlogPage() {
  return (
    <main>
      {/* 바로 낼 수 있는 헤더는 즉시 표시 */}
      <BlogHeader />

      {/* 무거운 목록은 Suspense로 감싸고 준비되는 대로 흘려보낸다 */}
      <Suspense fallback={<PostListSkeleton />}>
        <PostList />
      </Suspense>
    </main>
  );
}
```

이 구조로 만들어 두면 `PostList`의 데이터 취득에 시간이 걸려도 헤더와 레이아웃은 즉시 표시됩니다. 사용자는 "아무것도 일어나지 않는 흰 화면"을 보지 않아도 되고, 대기 시간 자체는 바뀌지 않아도 체감 속도는 크게 개선됩니다.

Instant Navigation은 이 Prefetch(클릭 전 준비)와 streaming(준비된 부분부터 표시)을 조합해, 전환이 끊기지 않고 매끄럽게 느껴지게 하는 설계라고 이해하면 알기 쉽습니다.

## 실무에서 주의할 3가지

### 1. Suspense 경계의 입도를 설계한다

Suspense로 감싸는 범위가 너무 크면 streaming의 혜택이 옅어집니다. 반대로 너무 잘게 나누면 스켈레톤이 깜빡여 어수선합니다. "먼저 보여주고 싶은 부분"과 "늦어도 괜찮은 부분"을 나눠 경계를 긋는 것이 실무의 요령입니다.

### 2. Prefetch 대상을 좁힌다

앞서 말했듯 무조건적인 prefetch는 대역폭을 낭비합니다. 중요한 동선(글 상세로의 전환 등)에 좁히고, 우선도가 낮은 링크는 절제하는 것이 정석입니다.

### 3. 체감 속도를 계측으로 확인한다

Instant Navigation의 효과는 주관적으로 느껴지지만, 인상으로 끝내지 말고 계측하는 것이 중요합니다. Core Web Vitals의 INP(Interaction to Next Paint)는 클릭부터의 반응 속도를 재는 지표로 특히 유효합니다. 개선 전후로 수치를 비교하는 습관을 들이면 시책의 효과를 객관적으로 판단할 수 있습니다.

## webhani의 관점

Instant Navigation 같은 개선은 Framework 버전을 올리는 것만으로 자동으로 작동하는 부분과, 이쪽 구현에 따라 효과가 달라지는 부분이 있습니다. Prefetch와 streaming은 후자입니다. Suspense 경계를 긋는 방식이나 Prefetch 대상 선정 같은 설계 판단이 체감 속도를 크게 좌우합니다.

저희가 고객의 Next.js 프로젝트를 지원할 때는 먼저 INP를 계측해 현재 전환 경험을 수치화하고, 그 위에서 Suspense 경계와 Prefetch 전략을 재검토하는 순서로 진행합니다. "왠지 빨라진 것 같다"로 끝내지 않고 계측에 근거해 개선하는 것이 확실합니다.

## 정리

Next.js 16.3의 Instant Navigation은 Prefetch로 클릭 전에 준비를 끝내고, streaming으로 준비된 부분부터 표시함으로써 전환을 '즉각'으로 느끼게 하는 설계입니다. Framework가 토대를 마련해 주는 한편, Suspense 경계의 입도나 Prefetch 대상 선정 같은 구현 판단이 효과를 좌우합니다.

버전을 올렸다면 먼저 INP를 계측해 보세요. 수치로 현재 상태를 파악하는 것이 체감 속도 개선의 첫걸음입니다.
