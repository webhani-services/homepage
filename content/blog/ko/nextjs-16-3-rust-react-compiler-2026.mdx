Next.js 16.3의 Turbopack에는 Build 쪽에서 실무에 효과적인 변경 세 가지가 들어갔습니다. 개발 시 Memory Eviction(Default 활성화), Build용 영속 File Cache, 그리고 실험적인 Rust 버전 React Compiler입니다. 이 글에서는 특히 App이 커질수록 효과가 커지는 Rust 버전 React Compiler와 영속 Build Cache 두 가지에 집중해, 동작 방식과 활성화 절차를 정리합니다.

## 지금까지 React Compiler의 과제

React Compiler는 `useMemo`나 `useCallback`을 손으로 작성하지 않아도 Compile 시점에 자동으로 메모이제이션을 삽입해 주는 Tool입니다. Re-render 최적화를 Code 외형을 해치지 않고 구현할 수 있어 채택이 넓어졌습니다.

다만 지금까지 React Compiler는 Babel 변환으로만 제공되었습니다. 여기에 대규모 App에서의 문제가 있습니다. Babel 변환은 JavaScript 실행 Resource를 사용하기 때문에, Code 양이 늘수록 Build를 느리게 만드는 요인이 되었습니다. 모처럼 Turbopack이 Rust로 빠르게 Build해도 React Compiler 부분이 JS 실행 대기로 발목을 잡는 구조였습니다.

## Rust 버전 React Compiler

React 팀이 Compiler의 Native Rust Port를 공개했고, Vercel이 이를 Turbopack에 통합했습니다. 이로써 React Compiler 처리가 Turbopack의 Build Pipeline 안에서 Rust Native로 실행됩니다. JS 실행 Resource 경합이 해소되므로 대규모 App일수록 효과가 큽니다.

Vercel의 초기 Test에서는 v0 같은 대규모 React App에 대해 20~50%의 Compile 시간 단축이 보고되었습니다(출처: Next.js 공식 Blog). 효과는 App 규모와 Component 수에 의존하므로, 작은 App에서는 차이가 작고 큰 App일수록 이점이 커집니다.

16.3 시점에서는 실험적 기능이므로 명시적으로 활성화합니다. 설정은 `next.config.ts`에서 합니다.

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default nextConfig;
```

React Compiler 자체를 단계적으로 도입하고 싶다면 대상 Directory를 좁히는 운영이 현실적입니다. 우선 영향 범위가 작은 기능부터 활성화하고, Render 동작이 바뀌지 않는지 확인한 뒤 넓혀 갑니다. React Compiler는 메모이제이션을 자동 삽입하므로, Side Effect에 의존한 구현이나 참조 동일성에 의존한 Code에서는 동작이 바뀔 수 있습니다. 도입 시 기존 Test로 Regression을 확인하는 것이 전제입니다.

## 영속 Build Cache

또 하나의 실무적 변경이 Build용 영속 File Cache입니다. `turbopackFileSystemCacheForBuild` Flag로 활성화합니다.

```typescript
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  experimental: {
    turbopackFileSystemCacheForBuild: true,
  },
};

export default nextConfig;
```

이를 활성화하면 생성된 `.next` Directory를 실행 간에 재사용할 수 있습니다. Local에서의 연속 Build는 물론, CI에서의 효과가 큰 기능입니다. 많은 CI Provider가 Cache Step을 가지고 있으므로, `.next`를 Cache 대상에 추가하면 변경되지 않은 부분의 재Compile을 피할 수 있습니다.

예를 들어 GitHub Actions에서는 다음과 같이 Build 출력을 Cache합니다.

```yaml
- name: Cache Next.js build
  uses: actions/cache@v4
  with:
    path: |
      .next/cache
    key: nextjs-${{ hashFiles('**/package-lock.json') }}-${{ github.sha }}
    restore-keys: |
      nextjs-${{ hashFiles('**/package-lock.json') }}-
```

작은 PR을 하루에 여러 번 Merge하는 팀에서는 이 효과가 쌓입니다. App의 95%가 바뀌지 않았다면, 그 95%의 Cold Compile Cost를 매번 지불하지 않아도 되기 때문입니다. Merge할 때마다 Full Build하던 Pipeline에서는 체감되는 단축이 됩니다.

## 개발 시 Memory Eviction

세 번째 변경으로 개발 Server의 Memory Eviction이 Default로 활성화되었습니다. 16.1에서 도입된 File System 영속화를 토대로, Turbopack은 Memory 상의 Cache 결과를 File로 내보내 Memory에서 제거할 수 있게 되었습니다. 장시간 개발 Session에서 Memory가 끝없이 늘어나는 문제가 완화됩니다.

이는 설정 불필요하며 Default 활성화이므로 16.3으로 올리기만 하면 혜택을 받을 수 있습니다. 장시간 `next dev`를 켜 두는 개발 Style에서는 Memory 사용량의 정체가 체감될 것입니다.

## 도입 시 정해 둘 것 — Cache 운영이 먼저

이 기능들을 넣을 때 먼저 정해 둬야 하는 것은 Cache 운영 정책입니다. 영속 Build Cache는 강력하지만, Cache Key 설계를 잘못하면 "오래된 Cache를 끌어와 변경이 반영되지 않는" 문제로 이어집니다.

webhani에서는 다음 순서를 권장합니다.

1. 우선 `turbopackFileSystemCacheForBuild`를 Local에서 활성화하고, 연속 Build에서 기대대로 Cache가 듣는지 확인한다.
2. CI의 Cache Key에 의존 관계 Hash(`package-lock.json` 등)를 반드시 포함한다. 의존이 바뀌면 Cache를 다시 만드는 설계로 한다.
3. Rust 버전 React Compiler는 Regression Test가 갖춰진 Project부터 단계적으로 활성화한다.

Rust 버전 React Compiler는 "빨라지는" 기능이지만, 영속 Build Cache는 "운영을 정한 뒤 쓰는" 기능입니다. 순서를 뒤집으면 속도를 위해 넣은 Cache가 Debug의 원인이 됩니다. 먼저 Cache 동작을 이해하고 Key 설계를 굳힌 뒤 CI에 올리는 것이 안전합니다.

## 정리

Next.js 16.3의 Build 계열 변경은 대규모 App 실무에 곧바로 효과가 있습니다. Rust 버전 React Compiler는 JS 실행 대기를 해소하고, Component 수가 많을수록 단축 폭이 커집니다. 영속 Build Cache는 작은 PR을 자주 Merge하는 팀에서 효과를 냅니다. 개발 시 Memory Eviction은 Default 활성화라 Upgrade만으로 혜택이 있습니다. 도입 순서로는 Cache 운영 정책을 먼저 굳히고, React Compiler는 Regression Test가 갖춰진 곳부터 단계적으로가 현실적입니다.
