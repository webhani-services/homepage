Next.js 16 Beta가 출시되었으며, Turbopack이 공식 기본 Bundler로 채택되었습니다. Typed Routes와 Route Export Validation도 안정 버전으로 전환되었습니다. 이 변경 사항들은 빌드 속도, 타입 안전성, 설정 정확성 등 Next.js 애플리케이션 개발의 일상적인 작업에 직접적인 영향을 미칩니다.

## Turbopack이 기본값으로

Turbopack은 Vercel이 webpack의 장기적인 대안으로 개발한 Rust 기반 Bundler입니다. Next.js 15까지는 실험적 옵션으로 제공되었지만, Next.js 16 Beta부터는 기본값이 됩니다.

대표적인 벤치마크는 대규모 코드베이스에서 최대 700배 빠른 속도입니다. 실제로는 프로젝트가 성장할수록 차이가 더욱 뚜렷해집니다. 핵심적인 차이는 단순한 속도뿐만 아니라, Turbopack의 빌드 시간이 코드베이스 크기에 따라 훨씬 완만하게 증가한다는 점입니다.

```bash
# webpack (Next.js 15) — 대규모 프로젝트
$ next dev
  Ready in 8.3s

# Turbopack (Next.js 16) — 동일한 프로젝트
$ next dev
  Ready in 0.9s
```

HMR(Hot Module Replacement)도 빨라졌습니다. 복잡한 컴포넌트 트리에서도 변경 사항이 수초가 아닌 밀리초 단위로 브라우저에 반영됩니다. 하루 종일 개발하다 보면 이 속도 차이가 실제로 쌓입니다.

## TypeScript 개선 사항

두 가지 TypeScript 기능이 Next.js 16에서 안정 버전으로 전환됩니다.

### Typed Routes

`next.config.ts`에서 활성화합니다.

```typescript
import type { NextConfig } from "next";

const config: NextConfig = {
  experimental: {
    typedRoutes: true,
  },
};

export default config;
```

활성화하면 `Link` 컴포넌트의 `href` props가 실제 Route 구조에 맞게 타입 검사됩니다.

```typescript
// ✅ 유효한 Route — 컴파일 통과
<Link href="/blog/my-post">더 읽기</Link>

// ❌ Route 오타 — 컴파일 오류
<Link href="/blg/my-post">더 읽기</Link>
```

내부 링크 오타는 실제로 발견하기 어려운 버그의 원인이 됩니다. 컴파일 타임에 이를 잡아내는 것은 여러 개발자가 Route를 추가하는 팀에서 구체적인 개선 사항입니다.

### Route Export Validation

이제 Page 파일이 빌드 시 내보내는 항목을 검증합니다.

```typescript
// app/blog/[slug]/page.tsx

// ✅ 유효한 export
export default function Page() { ... }
export async function generateStaticParams() { ... }

// ❌ 유효하지 않은 export — 빌드 오류
export const someRandomExport = "value";
```

잘못 구성된 Route 파일이 빌드를 조용히 통과하여 런타임 문제를 일으키는 것을 방지합니다.

## 마이그레이션 방법

대부분의 프로젝트에서 Next.js 15에서 16으로의 업그레이드는 간단합니다.

```bash
npm install next@beta react@latest react-dom@latest
```

주요 확인 사항은 커스텀 webpack 설정입니다. 모든 webpack 플러그인이 Turbopack과 호환되지는 않습니다. 커스텀 webpack 로더나 플러그인을 사용하는 경우, 업그레이드 전에 호환성을 확인하세요.

```js
// next.config.ts
const config: NextConfig = {
  // Next.js 16에서 Turbopack은 기본값
  // 커스텀 webpack 설정은 여전히 동작하지만 webpack 사용 시에만 적용됨
  webpack: (config) => {
    // webpack 모드에서만 적용 (--no-turbopack 플래그 사용 시)
    return config;
  },
};
```

## webhani의 관점

Turbopack을 기본값으로 채택한 실질적인 가치는 "Dev Server 대기 시간" 문제가 불만으로 쌓이기 전에 해결한다는 점입니다. Next.js 16부터 프로젝트를 시작하는 팀은 느린 빌드라는 기술 부채를 축적하지 않아도 됩니다.

Typed Routes는 실제로 겪는 문제를 해결합니다: 런타임까지 보이지 않는 깨진 내부 링크. Route 정확성에 컴파일 타임 검사를 추가하는 것은 TypeScript 생태계의 "오류를 더 일찍 잡자"는 방향과 일치합니다.

현재 운영 중인 Next.js 15 프로젝트는 안정 버전 16 릴리스를 기다렸다가 마이그레이션하는 것을 권장합니다. 새로운 프로젝트나 내부 도구라면 지금 Beta를 테스트해볼 가치가 있습니다.
