Next.js 16.2 릴리즈 노트에는 두 가지 성능 수치가 눈에 띕니다. `next dev` 시작 속도 약 400% 향상과 React Server Components Payload 역직렬화 약 350% 고속화입니다. dev 시작 속도 개선은 주로 Turbopack의 병렬화 작업 덕분이지만, RSC 역직렬화 개선은 더 흥미로운 배경이 있습니다. JSON 파싱의 특정 알고리즘을 바꾼 것이 직접적인 원인입니다.

## RSC Payload와 역직렬화 비용

React Server Component가 렌더링되면 서버는 출력을 JSON 형태의 Wire Format으로 직렬화하고, 클라이언트가 이를 역직렬화해서 컴포넌트 트리를 재구성합니다. 이 작업은 페이지 내비게이션, Server Action 응답, Streaming 업데이트마다 실행됩니다. 컴포넌트 트리가 깊거나 데이터 페칭이 많은 애플리케이션에서는 이 비용이 쌓입니다.

역직렬화는 클라이언트 측, 즉 브라우저의 JavaScript 엔진에서 실행됩니다. 16.1 이전 구현에서 이 처리가 측정 가능한 수준으로 느렸던 것이 이번 개선의 출발점입니다.

## Reviver 콜백이 있는 JSON.parse의 문제

기존 구현은 `JSON.parse`에 Reviver 콜백을 전달하는 방식이었습니다.

```javascript
// 기존 구현의 개념적 표현
function deserializeRSCPayload(rawJson) {
  return JSON.parse(rawJson, (key, value) => {
    // 파싱 중 모든 키-값 쌍에 대해 호출됨
    if (isSpecialRSCType(value)) {
      return reconstructRSCNode(value);
    }
    return value;
  });
}
```

Reviver 콜백은 JSON의 모든 키-값 쌍(깊게 중첩된 것 포함)에 대해 엔진이 호출합니다. 이 콜백은 임의의 JavaScript 함수이고 엔진의 C++ 파싱 루프 안에서 실행되기 때문에, 엔진이 이 경로를 최적화하기가 어렵습니다. 호출당 오버헤드는 작지만 RSC Payload는 밀도가 높습니다. 컴포넌트 50개짜리 페이지라도 수천 개의 중첩 키를 가진 Payload가 생성될 수 있습니다.

## 해결책: 먼저 파싱, 그 다음 워크

16.2에서는 파싱과 변환을 분리하는 방식으로 변경되었습니다.

```javascript
// 새 구현의 개념적 표현
function deserializeRSCPayload(rawJson) {
  // Step 1: 순수 JSON.parse — 엔진이 완전히 최적화, 콜백 없음
  const parsed = JSON.parse(rawJson);

  // Step 2: 순수 JavaScript로 재귀적 워크
  return walkRSCTree(parsed);
}

function walkRSCTree(node) {
  if (node === null || typeof node !== "object") {
    return node;
  }
  if (isSpecialRSCType(node)) {
    return reconstructRSCNode(node);
  }
  if (Array.isArray(node)) {
    return node.map(walkRSCTree);
  }
  const result = {};
  for (const key of Object.keys(node)) {
    result[key] = walkRSCTree(node[key]);
  }
  return result;
}
```

핵심은 Reviver 없는 `JSON.parse`가 V8, SpiderMonkey, JavaScriptCore 모두에서 가장 공격적으로 최적화된 코드 경로 중 하나라는 점입니다. 엔진은 이를 거의 네이티브 연산으로 처리합니다. Reviver를 전달하는 순간 모든 노드에 대해 인터프리터 JavaScript로 돌아갑니다.

먼저 순수한 `JSON.parse`를 실행하고, 결과 오브젝트 트리를 별도의 JavaScript 패스에서 워크하는 방식으로 엔진 최적화 파싱의 이점을 최대한 누릴 수 있습니다. 워크 자체도 일반 JavaScript 오브젝트 순회, 즉 JIT 컴파일 가능하고 예측 가능한 처리입니다. C++ 파싱 상태 머신과 뒤섞이지 않습니다.

결과적으로 대표적인 RSC Payload에서 역직렬화가 350% 빨라졌습니다.

## Streaming에 미치는 영향

RSC Payload는 종종 Streaming HTTP 응답으로 청크 단위로 도착합니다. 클라이언트는 각 청크가 도착할 때마다 처리하기 때문에, 역직렬화 레이턴시는 Streaming 페이지의 Time to Interactive에 직접 영향을 줍니다.

16.1에서는 10개 청크의 RSC 데이터를 Streaming하는 페이지가 Reviver 기반 역직렬화를 10번 실행했습니다. 각 호출은 노드마다 Reviver 호출 오버헤드를 지불했습니다. 16.2에서는 각 청크가 빠른 `JSON.parse` 패스와 별도 워크를 거쳐 청크당 레이턴시가 줄고 Streaming 페이지의 인터랙티브화가 빨라집니다.

이 개선 효과가 가장 두드러지는 경우는 다음과 같습니다.

- 컴포넌트 트리가 깊고 Server Component가 많은 페이지
- 대규모 직렬화 데이터 구조를 처리하는 대시보드나 데이터 집약 페이지
- Streaming을 사용하는 라우트 (`loading.tsx`, `Suspense` 경계)

## dev 시작 속도 개선

`next dev` 400% 향상은 RSC와 별개의 개선으로, Turbopack 모듈 번들링 파이프라인 최적화 결과입니다. 의존성 그래프 해석이 CPU 코어를 가로질러 더 적극적으로 병렬화되고, 증분 캐시 레이어도 반복 시작 시 더 효과적으로 작동합니다.

Next.js 16.1과의 비교에서는 기본 애플리케이션 기준 약 87% 개선이 보고되어 있습니다. 400%라는 수치는 더 넓은 베이스라인과의 비교로, 측정 대상이 달라 모순되는 것이 아닙니다.

실무 감각으로는 16.1에서 12~15초 걸리던 프로젝트가 16.2에서 2~4초로 줄어드는 경우가 많습니다.

## 이번 릴리즈에 포함된 보안 패치

16.2에는 여러 서브시스템에 걸친 보안 패치도 포함되어 있습니다.

| 대상 영역 | 문제 유형 |
|---|---|
| React Server Components | XSS, SSRF |
| App Router | Cache Poisoning, 인가 우회 |
| Middleware | 요청 우회 |
| Proxy 설정 | DoS |

App Router의 Cache Poisoning과 인가 우회는 실제 공격 패턴이 확립된 취약점 유형입니다. 프로덕션에서 Next.js를 운영하고 있다면, 성능 개선과 별개로 보안 패치만으로도 업그레이드 이유가 충분합니다.

## 업그레이드 시 확인사항

16.1에서 16.2로의 업그레이드에는 파괴적 변경사항이 없습니다. RSC 역직렬화 변경은 프레임워크 내부 변경이라 애플리케이션 코드 수정이 필요하지 않습니다.

```bash
npm install next@latest
# 또는
pnpm update next@latest
```

업그레이드 후 확인해야 할 사항은 다음과 같습니다.

**1. Server Component를 많이 사용하는 라우트를 테스트한다.** 역직렬화 로직이 변경되었습니다. Next.js 팀이 정확성을 보장하지만, 커스텀 RSC Payload 처리나 컴포넌트 트리의 엣지케이스가 있다면 명시적으로 테스트해야 합니다.

**2. 배포 전에 전체 테스트 스위트를 실행한다.** 성능 개선은 수동적으로 적용되어 별도 설정이 필요 없습니다. 하지만 보안 패치를 감안하면 프로덕션에 신속하게 배포하는 것이 권장됩니다.

**3. Turbopack 설정 위치를 확인한다.** `next.config.ts`의 `experimental.turbopack` 아래 설정이 남아 있다면 최상위 `turbopack` 키로 이동해야 합니다.

```typescript
// next.config.ts — 16.2 기준 올바른 형식
const nextConfig = {
  turbopack: {
    // Turbopack 설정을 여기에 작성
  },
};
```

## 프로덕션 앱에 미치는 실제 영향

RSC 역직렬화 개선은 개발 경험뿐 아니라 클라이언트 사이드 렌더링 성능에 직접 영향을 줍니다.

- **내비게이션 속도**: App Router의 클라이언트 사이드 내비게이션은 RSC Payload 역직렬화를 수반합니다. 350% 향상은 내비게이션 완료와 인터랙티브화가 빨라진다는 것을 의미합니다.
- **Server Actions**: Action 응답에는 RSC Payload가 포함됩니다. Server Actions를 사용하는 폼과 Mutation이 클라이언트 측에서 더 빠르게 반응합니다.
- **초기 페이지 로드**: SSR + Streaming 페이지에서는 각 Streaming 청크가 빠른 파싱 경로의 이점을 받습니다.

효과의 크기는 Payload 크기에 비례합니다. Server Component가 적은 단순한 페이지는 개선 폭이 작지만, 대규모 데이터셋을 Server Component를 통해 직렬화하는 대시보드 페이지에서는 눈에 띄는 개선이 기대됩니다.

## 마치며

RSC 역직렬화 수정은 되돌아보면 단순한 최적화입니다. Reviver 사용을 중단하고 별도 패스에서 처리하는 것. 하지만 JavaScript 엔진이 파싱 콜백을 어떻게 처리하는지 이해하지 못하면 이 개선 기회를 발견할 수 없습니다. 350% 향상은 실측 수치이고, 엔진을 가로질러 재현 가능하며, 애플리케이션 코드 변경도 필요 없습니다.

보안 패치를 위해 업그레이드하고, 성능 개선은 같은 패키지로 받아 가세요.
