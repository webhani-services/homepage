Production AI 시스템의 아키텍처가 하나의 패턴으로 수렴하고 있습니다. AI/ML 로직은 Python, API 오케스트레이션은 Go, 신뢰할 수 없는 코드 실행은 Rust의 WebAssembly Sandbox. 이는 언어 선호도가 아니라, 각 언어의 특성과 성숙한 에코시스템에 근거한 합리적인 선택입니다.

Go 개발자보다 $25K~$30K 높은 연봉을 받는 시니어 Rust 개발자들이 Rust를 고집하는 이유는 미학이 아닙니다. Production 환경에서 LLM이 생성한 코드를 안전하게 실행할 때, Rust만큼 강력한 보장을 제공하는 언어가 없기 때문입니다.

## 단일 언어 AI 시스템이 무너지는 이유

AI 시스템의 각 레이어는 서로 충돌하는 요구사항을 가집니다. 하나의 런타임에 강제로 담으려 하면 문제가 생깁니다.

**ML 레이어**는 PyTorch, TensorFlow, Hugging Face, Anthropic SDK를 필요로 합니다. 이 라이브러리들은 Python 우선입니다. 다른 언어로 이 에코시스템을 재현하려면 이미 잘 동작하는 도구를 다시 만드는 셈입니다.

**API 레이어**는 예측 가능한 Latency와 낮은 리소스 소비로 수천 개의 동시 연결을 처리해야 합니다. Python의 GIL은 진정한 동시성을 제한하고, asyncio는 복잡성을 더합니다. Go의 Goroutine 모델이 이를 깔끔하게 처리합니다.

**실행 레이어**(사용자나 LLM이 코드를 제출해 실행하는 경우)는 런타임에 패치로 해결할 수 없는 메모리 안전성 보장이 필요합니다. Rust의 소유권 모델은 이를 컴파일 시점에 제공합니다.

```
         ┌─────────────────────────────────────┐
         │         클라이언트 / 프론트엔드        │
         └──────────────┬──────────────────────┘
                        │
         ┌──────────────▼──────────────────────┐
         │          Go API Gateway              │
         │  인증 · Rate Limiting · 라우팅        │
         │  ML 서비스 Fan-out                   │
         └──────┬───────────────────┬──────────┘
                │                   │
   ┌────────────▼──────┐  ┌─────────▼────────────┐
   │  Python ML Layer  │  │  Rust WASM Sandbox   │
   │  RAG · LLM 호출   │  │  코드 실행            │
   │  임베딩 생성        │  │  신뢰 불가 입력 처리  │
   └───────────────────┘  └──────────────────────┘
```

## Python: ML 로직에 집중

Python Service는 한 가지 역할에 집중해야 합니다. ML 로직입니다. 인증, Rate Limiting, 라우팅은 Go에 맡기세요. FastAPI와 async/await로 I/O 바운드 LLM 호출의 처리량을 확보합니다.

```python
from fastapi import FastAPI, HTTPException
from anthropic import AsyncAnthropic
from sentence_transformers import SentenceTransformer
import numpy as np

app = FastAPI()
client = AsyncAnthropic()
embedder = SentenceTransformer("all-MiniLM-L6-v2")

@app.post("/rag/query")
async def rag_query(query: str, documents: list[str]) -> dict:
    if not documents:
        raise HTTPException(status_code=400, detail="문서가 제공되지 않았습니다")

    query_emb = embedder.encode(query)
    doc_embs = embedder.encode(documents)

    # 코사인 유사도로 Top-3 선택
    scores = np.dot(doc_embs, query_emb) / (
        np.linalg.norm(doc_embs, axis=1) * np.linalg.norm(query_emb) + 1e-10
    )
    top_k = np.argsort(scores)[-3:][::-1]
    context = "\n\n".join(documents[i] for i in top_k)

    response = await client.messages.create(
        model="claude-sonnet-4-6",
        max_tokens=1024,
        messages=[{"role": "user", "content": f"컨텍스트:\n{context}\n\n질문: {query}"}]
    )
    return {"answer": response.content[0].text, "sources": top_k.tolist()}
```

이 Service는 내부 Service로 배포합니다. 인터넷에서 직접 접근할 수 없어야 하며, Go Gateway가 유일한 진입점이 됩니다.

## Go: Gateway와 오케스트레이션

Go는 클라이언트와 Python 레이어 사이에 위치한 API Gateway 역할을 합니다. Goroutine의 경량성 덕분에 여러 ML Service에 병렬로 요청을 보내는 Fan-out 패턴을 콜백 복잡성 없이 구현할 수 있습니다.

```go
package main

import (
    "context"
    "encoding/json"
    "fmt"
    "net/http"
    "sync"
    "time"
)

type MLResult struct {
    ServiceID string
    Answer    string
    Error     error
}

// 여러 ML Service 레플리카에 병렬 요청
func queryMLServices(ctx context.Context, query string, docs []string) []MLResult {
    endpoints := []string{
        "http://ml-service-1:8000/rag/query",
        "http://ml-service-2:8000/rag/query",
    }

    results := make([]MLResult, len(endpoints))
    var wg sync.WaitGroup

    for i, ep := range endpoints {
        wg.Add(1)
        go func(idx int, url string) {
            defer wg.Done()

            reqCtx, cancel := context.WithTimeout(ctx, 15*time.Second)
            defer cancel()

            answer, err := callMLService(reqCtx, url, query, docs)
            results[idx] = MLResult{
                ServiceID: fmt.Sprintf("svc-%d", idx),
                Answer:    answer,
                Error:     err,
            }
        }(i, ep)
    }

    wg.Wait()
    return results
}

func handleQuery(w http.ResponseWriter, r *http.Request) {
    // 인증과 Rate Limiting은 Python으로 전달하기 전에 처리
    if !authenticate(r) {
        http.Error(w, "unauthorized", http.StatusUnauthorized)
        return
    }

    results := queryMLServices(r.Context(), extractQuery(r), extractDocs(r))
    json.NewEncoder(w).Encode(results)
}
```

Go의 Context 취소는 모든 Goroutine에 자동으로 전파됩니다. HTTP Request Context가 취소되면(클라이언트 연결 끊김, Gateway 타임아웃) 하위 ML 호출도 함께 취소됩니다.

## Rust: WASM 샌드박스 실행

LLM이 생성한 코드나 사용자 제공 코드를 실행해야 한다면, Rust + WebAssembly가 가장 안전한 선택입니다. Wasmtime 같은 WASM Runtime을 사용하면 코드를 완전히 격리된 환경에서 실행할 수 있습니다. 실행 중인 코드는 호스트 파일시스템, 네트워크, 할당된 영역 외의 메모리에 접근할 수 없습니다.

```rust
use wasmtime::{Engine, Module, Store, Linker, Config};
use anyhow::Result;

pub struct Sandbox {
    engine: Engine,
}

impl Sandbox {
    pub fn new() -> Result<Self> {
        let mut config = Config::new();
        config.consume_fuel(true);  // 연료 소비 추적 활성화
        Ok(Sandbox {
            engine: Engine::new(&config)?,
        })
    }

    pub fn run(&self, wasm_bytes: &[u8], fuel_limit: u64) -> Result<String> {
        let module = Module::new(&self.engine, wasm_bytes)?;
        let mut store = Store::new(&self.engine, ());

        // 연료 한도가 실행 단계를 제한 — 무한 루프 방지
        store.add_fuel(fuel_limit)?;

        let linker = Linker::new(&self.engine);
        let instance = linker.instantiate(&mut store, &module)?;

        let run = instance.get_typed_func::<(), ()>(&mut store, "run")?;
        run.call(&mut store, ())?;

        let consumed = fuel_limit - store.fuel_consumed().unwrap_or(0);
        Ok(format!("완료, 소비된 연료: {}", consumed))
    }
}
```

연료 메커니즘이 연산 예산 역할을 합니다. 무한 루프는 연료가 소진되면 서비스를 중단시키지 않고 깔끔하게 종료됩니다.

## 서비스 간 통신 설계

이 아키텍처가 제대로 동작하려면 명확한 인터페이스를 가진 독립 서비스로 각 언어 레이어를 취급해야 합니다. 내부 모듈이 아닙니다.

| 레이어 | 프로토콜 | 이유 |
|-------|---------|------|
| Go → Python | HTTP/JSON (내부) | 단순하고 디버그하기 쉬움 |
| Go → Rust | HTTP 또는 stdin/stdout | WASM 출력은 텍스트/바이너리 |
| 외부 → Go | HTTPS + 인증 | 유일한 진입점 |

LLM 작업처럼 응답 시간이 긴 경우, Go와 Python 사이에 Message Queue(Kafka, RabbitMQ)를 두면 좋습니다. LLM 응답을 30초 동안 Go Goroutine을 블로킹하며 기다리는 대신, 작업을 큐에 넣고 Python Service가 완료되면 Webhook으로 결과를 받습니다.

## webhani의 시각

모든 AI 프로젝트에 이 아키텍처가 필요한 것은 아닙니다. 대부분의 팀에게는 단일 FastAPI Service가 올바른 출발점입니다. Polyglot 분업이 의미있는 시점은 구체적인 문제가 생겼을 때입니다. 높은 부하에서 Python 동시성 한계에 부딪혔을 때, 신뢰할 수 없는 코드를 안전하게 실행해야 할 때, ML 모델과 API 로직의 배포 주기를 분리해야 할 때입니다.

핵심은 세 가지 언어를 사용하는 것이 아니라, 이 세 관심사가 서로 다른 특성을 가지고 있음을 인식하고 그에 맞게 설계하는 것입니다. 처음부터 복잡하게 만들 필요는 없지만, 규모가 커졌을 때 어떻게 분리할지 그림을 가지고 시작하는 것이 중요합니다.
