## PostgreSQL이 OAuth를 지원한다는 의미

PostgreSQL 18에서 표준 기능으로 OAuth 2.0 기반 인증이 지원되었습니다. 소소해 보이지만 운영의 핵심을 크게 바꾸는 변경입니다. 그동안 Database 인증은 Password나 인증서, 또는 LDAP 연동으로 처리하는 것이 일반적이었습니다. 하지만 조직 전체가 IdP(Identity Provider)를 이용한 SSO(Single Sign-On)를 도입한 경우, "Database만 별도의 인증 체계"인 상태가 남기 쉬웠습니다.

PostgreSQL 18의 OAuth 지원은 이 분단을 메우는 첫걸음입니다. OpenID Connect(OIDC)를 통해 Keycloak이나 Ping Identity 같은 IdP가 발급한 Token으로 Database에 접속할 수 있게 됩니다. 이 글에서는 그 설계 사상과 실무 관점을 정리합니다.

## 왜 "외부 Validator"에 위임하는 설계인가

PostgreSQL 18의 OAuth 구현에서 특징적인 점은 Token 검증을 Core 본체에서 끝내지 않고 외부 Validator Library에 위임한다는 것입니다. 언뜻 돌아가는 것처럼 보이지만 이유가 있습니다.

OIDC는 Specification으로는 공통이어도 IdP마다 Token 형식과 Claim을 담는 방식, 검증 절차가 미묘하게 다릅니다. 만약 PostgreSQL 본체가 특정 IdP의 방식을 전제로 구현하면, 다른 IdP와의 호환성 문제나 Specification 변경 추종이 부담이 됩니다. 그래서 PostgreSQL은 "Token을 어떻게 검증할 것인가"를 교체 가능한 Plugin으로 분리했습니다.

이 역할을 맡는 구현 중 하나가 Percona의 `pg_oidc_validator`입니다. PostgreSQL과 임의의 OIDC 준수 IdP를 잇는 Extension으로, deb/rpm Package가 배포됩니다.

## 구성의 전체 그림

OAuth 인증을 활성화할 때의 흐름을 개념적으로 정리합니다(실제 Parameter 이름은 환경과 버전에 따라 다르므로 반드시 공식 Documentation을 확인하세요).

1. **Server 측 준비**: PostgreSQL 18 Server Package에 더해 Validator(예: `pg_oidc_validator`)를 도입합니다.
2. **설정 파일에서 Validator Load**: 어떤 Validator를 쓸지, 어떤 Claim을 PostgreSQL 사용자명에 매핑할지(예: `authn_field`)를 지정합니다.
3. **인증 설정과 Identity Map 정의**: `pg_hba.conf`에 해당하는 인증 설정에서 OAuth를 지정하고, IdP상의 사용자와 Database 사용자를 매핑하는 Identity Map을 만듭니다.

```text
# 개념도 (실제 표기는 Documentation 참조)
[Client (psql 등)]
      │  IdP 발급 Token 제시
      ▼
[PostgreSQL 18] --위임--> [OIDC Validator (예: pg_oidc_validator)]
      │                          │
      │  <--검증 결과·사용자 Mapping--
      ▼
접속 허용 (IdP 사용자 → DB 사용자로 Map)
```

Client 측에서도 대응이 필요합니다. `psql` 같은 Command-line Client와 Validator를 조합하면 현시점에서도 시험해볼 수 있습니다.

## 도입 판단을 위한 현실적 시각

새 기능이긴 하지만 Webhani로서는 다음을 고려해 단계적으로 검토할 것을 권합니다.

### 1. 아직 "실험 단계"임을 전제로

OAuth 지원은 PostgreSQL 18 이상에서만 이용 가능하며 Client 측 대응과 외부 Validator를 필요로 합니다. 본격적인 확산에는 시간이 걸릴 전망입니다. 우선 검증 환경에서 자사 IdP와 조합해 동작을 확인하는 데서 시작하는 것이 현실적입니다.

### 2. 얻는 이점을 구체적으로 평가

OIDC 지원의 가치는 인증의 일원화에 있습니다. IdP 측에서 사용자의 Lifecycle(입사·퇴사·권한 변경)을 관리하고 있다면 Database 접근도 그 관리에 태울 수 있습니다. Password를 Database별로 배포·Rotation하는 운영에서 벗어날 수 있다는 것은 감사 대응이나 Zero Trust 설계 관점에서도 큰 이점입니다.

### 3. 보안 설정 초기값에 주의

인증의 일원화는 보안을 끌어올리지만, 설정을 잘못 짜면 역효과입니다. Claim Mapping을 느슨하게 설정하면 의도치 않은 사용자가 권한을 얻을 위험이 있습니다. Identity Map은 최소 권한 원칙에 따라 신중하게 설계하세요.

## Webhani의 견해

Database 인증을 IdP에 붙이는 흐름은 다른 Middleware에서 일어났던 일의 연장선상에 있습니다. Application은 진작에 SSO로 이전했는데 Database만 Password 운영으로 남겨져 있는 상태 — 이는 보안상의 약점이 되기 쉽습니다. PostgreSQL 18의 OAuth 지원은 그 마지막 조각을 메우는 움직임이라고 봅니다.

현시점에서는 성숙 도중이지만 방향성으로는 본류입니다. 특히 금융·의료처럼 감사 요건이 엄격한 영역이나 Zero Trust를 추진하는 조직에서는 미리 검증해둘 가치가 있습니다. Webhani도 고객의 Infrastructure 설계에서 "Database 인증의 SSO 통합"을 선택지 중 하나로 제안해 나갈 방침입니다.
