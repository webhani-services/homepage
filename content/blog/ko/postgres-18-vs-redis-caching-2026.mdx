## 캐싱 계층의 재평가

지난 10년간, 답은 자동이었습니다. Redis를 추가하세요. 빠른 읽기, 간단한 API, 애플리케이션 로직을 데이터베이스 레이턴시로부터 분리합니다. 하지만 2026년, 이 반사적인 판단은 재검토할 가치가 있습니다. PostgreSQL 18에 도달하면서, 특히 스타트업과 중견기업의 엔지니어들이 정말로 "Redis 운영 비용이 그만한 가치가 있나?"라고 질문하기 시작했습니다.

이 트렌드는 실제입니다. PostgreSQL의 개선는 상당합니다. PgBouncer 연결 풀링, BRIN 인덱스, 부분 인덱스를 통해 적절히 설정된 PostgreSQL은 일반적인 액세스 패턴에서 **5ms 이하의 읽기 레이턴시**를 제공할 수 있습니다. 한편 Redis 8.2는 명령 실행 속도를 35% 향상시켰고, Valkey 8.1은 메모리 사용량을 28% 감소시켰습니다. PostgreSQL과 Redis 모두 진화했습니다. 이제 문제는 "캐시 사용 여부"가 아니라 "무엇을 캐시에 보관할 것인가, 무엇을 데이터베이스에 보관할 것인가"입니다.

Webhani에서는 실제 시스템에서 이 결정을 여러 번 내렸습니다. 계산 구조가 바뀌었고, 이것이 왜인지—그리고 각각 언제 선택해야 하는지 이해할 가치가 있습니다.

## PostgreSQL 18이 캐시 부하를 줄이는 이유

### 연결 풀링이 게임을 바꾼다

PostgreSQL의 연결당 오버헤드는 상당합니다. 각 연결은 메모리를 확보하고, 버퍼 관리자에 참여하며, 쿼리 실행 시 레이턴시를 추가합니다. PgBouncer는 애플리케이션과 데이터베이스 사이에 위치하여, 많은 애플리케이션 연결을 작은 수의 장수명 데이터베이스 연결로 멀티플렉싱합니다.

```sql
-- pgbouncer.ini 설정
[databases]
myapp = host=localhost port=5432 dbname=myapp

[pgbouncer]
pool_mode = transaction
max_client_conn = 1000
default_pool_size = 25
reserve_pool_size = 5
reserve_pool_timeout = 3
```

영향: 한때 200개의 동시 애플리케이션 연결을 처리하던 하드웨어 구성이 이제 1000+개를 처리할 수 있습니다. 연결 획득에 소비되는 시간이 줄어들기 때문에 읽기 레이턴시가 떨어집니다. 많은 I/O 바운드 워크로드의 경우, 이것만으로도 Redis가 필요한 경우를 제거합니다.

### 인덱스의 진화

BRIN (Block Range INdex)은 인덱스 크기와 정밀도를 절충합니다. 수백만 행이 있는 시계열 테이블에서 BRIN 인덱스는 B-tree의 디스크 공간의 일부를 차지하면서 빠른 순차 액세스를 유지합니다. 부분 인덱스를 사용하면 실제로 쿼리하는 행만 인덱싱할 수 있습니다—활성 세션, 미해결 티켓, 만료되지 않은 토큰입니다.

```sql
-- 부분 인덱스: 활성 세션만
CREATE INDEX idx_sessions_active ON user_sessions (user_id)
  WHERE status = 'active'
  AND updated_at > NOW() - INTERVAL '1 day';

-- 시계열 데이터용 BRIN (B-tree보다 훨씬 작음)
CREATE INDEX idx_events_time ON events USING BRIN (created_at)
  WITH (pages_per_range = 128);
```

PostgreSQL의 개선된 쿼리 플래너와 함께, 이러한 인덱스는 Redis에 접근하지 않고도 핫 데이터를 가져올 수 있습니다. 잘 조정된 쿼리와 인메모리 캐시 페치 사이의 레이턴시 차이가 크게 좁혀졌습니다.

### Materialized View를 중간 지점으로

모든 것을 미리 계산할 필요는 없습니다. 하지만 일부는 필요합니다. Materialized View를 통해 비용이 많이 드는 집계를 SQL에 한 번 정의하고, 일정에 따라 새로 고쳐서, 별도의 캐싱 시스템 없이 최신 결과를 제공할 수 있습니다.

```sql
CREATE MATERIALIZED VIEW daily_revenue AS
SELECT 
  DATE(created_at) as date,
  COUNT(*) as orders,
  SUM(amount) as revenue
FROM transactions
WHERE created_at > NOW() - INTERVAL '90 days'
GROUP BY DATE(created_at);

CREATE INDEX ON daily_revenue (date);

-- 야간 또는 온디맨드 새로 고침
REFRESH MATERIALIZED VIEW CONCURRENTLY daily_revenue;
```

애플리케이션은 일반 테이블처럼 뷰를 쿼리합니다. PostgreSQL이 새로 고침 일정을 처리합니다. 클라이언트 측 캐싱 로직 없음, TTL 버그 없음, 스테일 데이터 놀람 없음입니다.

## Postgres만으로 Job Queue 구현하기

실전에서 성공적으로 마이그레이션한 패턴: 경량 작업 큐입니다. 전통적으로 팀은 작업 큐용 Redis를 사용했습니다 (신뢰성, 지속성). 하지만 PostgreSQL도 이것을 할 수 있습니다—저렴하게—UNLOGGED 테이블과 SKIP LOCKED 의미로.

```sql
-- UNLOGGED 테이블: 지속되지만 WAL 오버헤드 없음
CREATE UNLOGGED TABLE job_queue (
  id BIGSERIAL PRIMARY KEY,
  job_type VARCHAR(50) NOT NULL,
  payload JSONB NOT NULL,
  status VARCHAR(20) DEFAULT 'pending',
  created_at TIMESTAMP DEFAULT NOW(),
  started_at TIMESTAMP,
  completed_at TIMESTAMP
);

-- 워커가 다른 것을 차단하지 않고 다음 작업을 선택
SELECT * FROM job_queue
WHERE status = 'pending'
ORDER BY id
LIMIT 10
FOR UPDATE SKIP LOCKED;

-- 완료로 표시
UPDATE job_queue
SET status = 'completed', completed_at = NOW()
WHERE id = $1;
```

**장점:**
- UNLOGGED는 WAL 오버헤드를 건너뜁니다; 로깅된 테이블보다 3-5배 빠릅니다
- SKIP LOCKED는 워커가 서로를 잠그는 것을 방지합니다
- 지속성: 서버 재시작 후에도 데이터가 유지됩니다 (주의 사항 참고—아래 참조)
- 단일 도구: PostgreSQL만

**중요한 주의:** UNLOGGED 테이블은 비정상 종료 시 **절단됩니다**. 미션 크리티컬한 작업의 경우, 별도의 내구성 큐와 결합하거나 공격적인 아카이빙이 있는 로깅된 테이블을 사용하십시오.

## Redis가 여전히 이기는 경우

이것은 Redis를 대규모로 교체하는 것이 아닙니다. 각 사용 사례마다 올바른 도구를 선택하는 것입니다.

### 실시간 카운터 및 레이트 제한

세션 활동 수, API 호출 제한, 라이브 사용자 수, 요청 레이트. 이들은 **서브 밀리초 응답 시간**이 필요하고 디스크 I/O를 허용하지 않습니다. Redis의 인메모리 구조가 결정적으로 이깁니다.

```lua
-- Redis Lua for atomic rate-limit check
if redis.call('INCR', KEYS[1]) == 1 then
  redis.call('EXPIRE', KEYS[1], 60)
end
```

Postgres는 내구성을 희생하지 않고는 이 레이턴시를 달성할 수 없습니다.

### Pub/Sub 및 스트리밍

실시간 채팅, 활동 피드, 라이브 알림. Redis Streams와 Pub/Sub은 다대다 통신 패턴을 위해 구축되었습니다. PostgreSQL의 LISTEN/NOTIFY는 시스템 내 알림을 위해 설계되었으며, 고팬아웃 메시징을 위한 것이 아닙니다. 단일 토픽에서 수천 명의 구독자가 필요한 경우, Redis가 답입니다.

### 분산 잠금

여러 프로세스 또는 서비스 간의 작업 조율. PostgreSQL 잠금은 트랜잭션입니다—트랜잭션이 종료되면 해제됩니다. 장수명 분산 잠금(진행 중인 데이터베이스 마이그레이션, 실행 중인 작업)의 경우, Redis가 필요합니다.

## 의사결정 프레임워크

| 요구 사항 | PostgreSQL 권장 | Redis 더 나음 |
|---------|---|---|
| **레이턴시** | 밀리초 범위 (앱 로직 포함) | 서브 밀리초 (원자적 연산) |
| **지속성** | 필수 | 선택 사항 (임시) |
| **쿼리 복잡성** | 중간 이상 (JOIN, WHERE) | 단순 (키 조회, 리스트 연산) |
| **데이터 용량** | 기가바이트부터 테라바이트 | 실제 한계 100GB까지 |
| **운영** | 단일 도구 | 별도 모니터링, 백업, 페일오버 |
| **Pub/Sub** | 확장에 부적합 | 주요 사용 사례 |

## 우리의 실전 경험

여러 Webhani 클라이언트 참여 프로젝트에서 Redis를 더 이상 필요하지 않은 아키텍처에서 제거했습니다.

1. **중규모 SaaS 플랫폼**: 세션 캐시용 Redis 제거, PostgreSQL + PgBouncer + 활성 세션의 부분 인덱스로 교체. 결과: 인프라 비용 20% 감소, 모니터링 및 업그레이드할 도구 하나 적음.

2. **로그 집계 서비스**: 세션 수준 집계 (시간당 고유 사용자)를 Redis에서 PostgreSQL Materialized View(10분마다 새로 고침)로 이동. 결과: Redis를 폴링하는 cron 작업 제거, 데이터 일관성 보증 단순화.

3. **실시간 알림 시스템**: Pub/Sub용 Redis 유지 (필수), 다른 캐시를 PostgreSQL로 이동. 결과: Redis 메모리 압력 감소, Redis 운영 단순화.

패턴: 먼저 측정한 후 결정하세요. Redis를 유행이라고 삭제하지 마세요. 프로파일링이 데이터베이스로 충분하다는 것을 증명하기 때문에 삭제하세요.

## 결론

PostgreSQL 18은 더 이상 "데이터베이스"만이 아닙니다. 캐싱, 큐 관리, 분석 워크로드를 트랜잭션 쿼리와 함께 처리할 수 있는 데이터 플랫폼으로 성숙했습니다. PgBouncer, BRIN 인덱스, Materialized View는 성능을 희생하지 않으면서 인프라를 통합하도록 합니다.

Redis 8.2와 Valkey 8.1은 제외되지 않았습니다. 그들의 장소를 찾았습니다: 실시간 연산, Pub/Sub, 서브 밀리초 레이턴시가 필수 불가결한 초고스루풋입니다.

2026년 모범 사례는 "항상 Redis를 사용하세요" 또는 "Redis를 사용하지 마세요"가 아닙니다. 그것은 **당신의 워크로드를 측정하고, 쿼리를 프로파일링하고, 데이터가 결정하도록 하세요**입니다. PostgreSQL로 시작하세요. 프로파일링이 필요함을 보여주는 곳에만 Redis를 추가하세요. 이 훈련된 접근 방식은 운영 오버헤드를 줄이고 스택을 단순하게 유지합니다—이는 프로덕션에서 종종 최고의 성능 향상입니다.
