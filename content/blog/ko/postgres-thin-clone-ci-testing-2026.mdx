CI 파이프라인에서 데이터베이스 테스트는 오래된 딜레마를 안고 있습니다. 적은 양의 시드 데이터로 구성한 Fixture 데이터베이스는 빠르지만, 본환경의 Query Plan이나 Index 동작을 완전히 재현할 수 없습니다. 반대로 본환경 스냅샷 전체를 복원하면 현실적인 데이터 분포를 얻지만, 매번 복원에 수십 분이 소요되어 CI 실행 시간이 엄청나게 늘어납니다. Copy-on-Write Thin Clone 기술이 이 딜레마를 실질적으로 해결합니다.

이 글에서는 GitHub Actions와의 통합 사례를 보여주면서, 이 기법이 왜 Migration 테스트와 Integration 테스트의 품질을 개선하는지 정리하겠습니다.

## Fixture 데이터만으로는 잡지 못하는 문제

일반적인 소규모 Fixture 테스트의 한계를 구체적으로 봅시다.

```sql
-- 본환경에서는 users 테이블이 5000만 행
-- 일반적인 SELECT는 문제없지만,
-- 특정 Index 삭제 판단이 잘못된 경우
-- 시딩된 100행 테스트에서는 절대 감지되지 않음

-- 본환경 Query Plan (Full Scan 감지 가능)
EXPLAIN ANALYZE 
SELECT * FROM users 
WHERE region = ? AND status = ? AND created_at > ?;
-- 수십만 행을 스캔해야 하므로 Timeout 발생
```

Fixture 데이터베이스에서는 이런 「규모 의존적」 문제가 숨겨진 채로 남습니다. 또한 Migration 실행 중 Lock 점유 시간, 테이블 스캔 지연, Connection Pool 소비 패턴도 소규모 데이터에서는 다른 동작을 보입니다.

본환경 스냅샷에서 복원하면 확실하지만, 300 GB 데이터베이스를 테스트할 때마다 복원하면 CI는 비현실적이 됩니다.

## Thin Clone의 작동 원리와 이점

Copy-on-Write Thin Clone은 기저 블록 스토리지의 기능을 활용합니다. 대체로 다음과 같은 메커니즘으로 작동합니다.

1. **초기 스냅샷**: 본환경 데이터베이스의 읽기 전용 스냅샷(메타데이터)을 생성. 이 시점에는 물리적 복제가 일어나지 않음.
2. **분기**: CI Job이 스냅샷에서 「Clone」을 생성. 새 Clone은 원본 스냅샷의 데이터 블록을 참조.
3. **Copy-on-Write**: Clone이 데이터를 쓸 때만 영향을 받은 블록을 복제해 새 복사본에 변경 적용. 읽기 전용 블록은 공유된 채로 유지되어 메모리와 스토리지 절약.
4. **삭제**: CI Job 종료 후 Clone을 제거. 원본 스냅샷은 영향받지 않음.

이 패턴으로 본환경 규모의 SQL 데이터베이스를 초 단위로 개별 프로비저닝할 수 있습니다. Migration 실행, Integration 테스트 데이터 준비, 여러 병렬 테스트 Job이 모두 독립된 환경을 얻을 수 있습니다.

## GitHub Actions에서의 구현 예시

다음은 Thin Clone을 사용하는 실무적 GitHub Actions 워크플로우 예시입니다. 본인의 환경에 맞춰 커스터마이즈하세요.

```yaml
name: Database Migration & Integration Tests

on:
  pull_request:
    paths:
      - 'migrations/**'
      - 'src/db/**'
      - '.github/workflows/ci-db.yml'

jobs:
  test-with-thin-clone:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4

      - name: Python 설정
        uses: actions/setup-python@v5
        with:
          python-version: '3.12'

      - name: 의존성 설치
        run: |
          pip install psycopg2-binary alembic pytest sqlalchemy

      - name: Thin Clone 데이터베이스 프로비저닝
        env:
          DB_SNAPSHOT_ID: ${{ secrets.PROD_DB_SNAPSHOT_ID }}
          CLONE_ID: ci-test-${{ github.run_id }}-${{ github.run_attempt }}
          DB_ADMIN_HOST: db-admin.internal.example.com
          DB_ADMIN_USER: postgres
          DB_ADMIN_PASSWORD: ${{ secrets.DB_ADMIN_PASSWORD }}
        run: |
          # Clone 프로비저닝 스크립트 실행
          # 실제 구현은 사용하는 Clone Engine에 따라 달라짐
          python scripts/provision_clone.py \
            --snapshot-id "$DB_SNAPSHOT_ID" \
            --clone-id "$CLONE_ID" \
            --admin-host "$DB_ADMIN_HOST"
          
          # Clone 연결 정보 추출
          CLONE_HOST=$(python scripts/get_clone_host.py --clone-id "$CLONE_ID")
          echo "TEST_DB_HOST=$CLONE_HOST" >> $GITHUB_ENV
          echo "TEST_CLONE_ID=$CLONE_ID" >> $GITHUB_ENV

      - name: Database Migration 실행
        env:
          DATABASE_URL: postgresql://postgres:${{ secrets.DB_PASSWORD }}@${{ env.TEST_DB_HOST }}/postgres
        run: |
          alembic upgrade head

      - name: Integration 테스트 실행
        env:
          DATABASE_URL: postgresql://postgres:${{ secrets.DB_PASSWORD }}@${{ env.TEST_DB_HOST }}/postgres
        run: |
          pytest tests/integration/ -v --tb=short

      - name: Query Plan 검증
        env:
          DATABASE_URL: postgresql://postgres:${{ secrets.DB_PASSWORD }}@${{ env.TEST_DB_HOST }}/postgres
        run: |
          # 실제 데이터를 대상으로 Query 실행해 회귀 검증
          python scripts/verify_query_plans.py

      - name: Thin Clone 정리
        if: always()
        env:
          DB_ADMIN_HOST: db-admin.internal.example.com
          DB_ADMIN_PASSWORD: ${{ secrets.DB_ADMIN_PASSWORD }}
          CLONE_ID: ${{ env.TEST_CLONE_ID }}
        run: |
          python scripts/cleanup_clone.py \
            --clone-id "$CLONE_ID" \
            --admin-host "$DB_ADMIN_HOST"
```

주요 특징:

- **빠른 프로비저닝**: Clone 생성이 2-5초, 5-10분이 아님.
- **격리**: 각 CI Job은 고유의 쓰기 가능 데이터베이스 인스턴스를 가짐. 테스트 간 간섭 없음.
- **현실적 규모**: Migration과 Query Planning이 실제 데이터 분포를 상대로 실행.
- **확실한 정리**: 테스트 실패 시에도 Clone이 제거됨.

여러 Job이 동일한 스냅샷에서 각각 독립된 Clone을 생성해 병렬로 실행되므로 전체 파이프라인 시간이 대폭 단축됩니다.

## 실운영에서의 고려사항

### PII와 본환경 데이터 민감성

CI 파이프라인에서 본환경 스냅샷을 사용하면 데이터 보호 위험이 생깁니다. 고객 이메일, 전화번호, 거래 정보가 테스트 로그에 노출될 수 있습니다.

두 가지 접근 방식이 있습니다.

1. **사전 마스킹 스냅샷**: 스냅샷 생성 시 PII 컬럼을 Hash 또는 익명화. 일회성 처리 후 모든 CI 실행에서 재사용. 장점: 단순함. 단점: 초기 마스킹 작업 필요.
2. **합성 본환경 규모 데이터**: 실제 고객 데이터 없이 완전히 생성된 데이터로 본환경 규모 데이터베이스 유지. 장점: 최대 보안. 단점: Schema 동기화 노력.

Webhani는 대부분의 프로젝트에서 두 번째 접근을 선호합니다. Thin Clone의 가치는 특정 고객 레코드 복제가 아니라, 규모에서의 Query Plan 테스트에 있습니다. 5000만 행의 결정론적 생성 데이터는 Index 회귀, Lock Timeout, Sequential Scan 문제를 잡기에 충분합니다.

### 비용과 실행 시간 예산

Thin Clone은 미미한 인프라 투자로 CI 시간을 크게 절약합니다.

- **스토리지**: 본환경 스냅샷(일회성) + 병렬 Clone들의 Delta 저장. 200 GB 스냅샷 + 30개 Clone이 각 500 MB 쓰면 약 215 GB 임시 스토리지 필요. 대부분의 클라우드는 탄력적으로 프로비저닝 가능.
- **스냅샷 최신성**: 본환경 Schema가 월별·분기별로 변경? 정기적으로 스냅샷 갱신(주 1회 권장).
- **정리 자동화**: 정리 없으면 버려진 Clone이 스토리지를 계속 점유. CI와 정기 Orphan 감지 Job에 확실한 정리 로직 구축.

Thin Clone이 모든 팀에 필수는 아닙니다. 신뢰할 수 있는 Fixture 테스트와 작은 데이터 규모를 가진 팀은 가치를 못 느낄 수 있습니다. 강력한 채택 사례는:

- Migration 실패가 본환경에까지 도달한 경험이 있음
- Integration 테스트 실패가 본환경에서만 발견되는 정기적 현상
- 데이터 규모가 GB 이상이고 증가 추세
- 안정성 요구도가 인프라 투자를 정당화할 수준

### 병렬 실행의 이득

Thin Clone으로는 데이터베이스 의존 테스트를 충돌 없이 병렬로 실행 가능합니다.

```yaml
strategy:
  matrix:
    test-suite: [migrations, api-integration, background-jobs]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      # 각 Matrix Job이 독립된 Clone을 얻음
      # 전체 시간: max(suite_time) + Clone Overhead
      # Clone 미사용 시: sum(모든 suite) + 각 Job당 복원 Overhead
```

이 병렬화가 가장 큰 CI 시간 이득입니다.

## 결론

Thin Clone은 본환경 규모 데이터베이스 테스트를 CI 파이프라인의 현실적 범위 안으로 가져옵니다. Migration은 현실적 Query Plan을 대상으로 실행됩니다. Integration 테스트는 Index 회귀와 Lock 경합 버그를 잡습니다. 여러 CI Job이 데이터 충돌 없이 병렬 실행됩니다.

기술적 효과는 명확합니다. 본환경 데이터베이스 사건 감소, 개발자 신뢰도 향상, CI 파이프라인 가속화.

만약 조직이 Migration 회귀, 규모에서의 Index 서프라이즈, CI 통과 후 본환경 데이터베이스 문제를 경험했다면, Thin Clone은 데이터베이스 신뢰도를 향한 실용적인 다음 단계입니다.
