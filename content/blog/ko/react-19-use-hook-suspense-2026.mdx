React 19가 정식 출시되면서 데이터 페칭 방식에 실질적인 변화가 생겼습니다. 그 중심에 있는 것이 `use` 훅입니다. 이름은 단순하지만, 기존 Hook 규칙의 예외가 적용되는 특별한 API입니다. `Suspense`와 결합하면 비동기 처리를 선언적으로 표현할 수 있어 코드 구조가 크게 개선됩니다.

이 글에서는 `use` 훅이 내부적으로 어떻게 동작하는지, 기존 `useEffect` 패턴과 무엇이 다른지, 그리고 실제 프로젝트에서 바로 쓸 수 있는 설계 패턴을 정리합니다.

## `use` 훅이란?

`use`는 Promise나 Context를 인자로 받아 그 값을 반환하는 함수입니다. 핵심 동작은 다음과 같습니다.

- Promise가 **pending** 상태이면: 컴포넌트 렌더링을 중단(suspend)하고, 가장 가까운 `<Suspense>` fallback을 표시합니다.
- Promise가 **resolved** 상태이면: 즉시 resolved 값을 반환합니다.
- Promise가 **rejected** 상태이면: 에러를 throw하고, 가장 가까운 `ErrorBoundary`가 이를 캐치합니다.

```tsx
import { use } from "react";

type User = {
  id: string;
  name: string;
  email: string;
  avatarUrl: string;
};

function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise);
  // 이 줄에 도달했다면 user는 반드시 resolved 상태입니다
  return (
    <div className="profile">
      <img src={user.avatarUrl} alt={user.name} />
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </div>
  );
}
```

컴포넌트는 데이터가 항상 존재한다는 전제로 작성합니다. Loading 상태 처리는 외부의 `Suspense`에 완전히 위임합니다.

## 기존 useEffect 패턴과 비교

React 18 이전에는 데이터 페칭 컴포넌트가 대부분 이런 형태였습니다.

```tsx
// React 18 이전의 전형적인 패턴
function UserProfile({ userId }: { userId: string }) {
  const [user, setUser] = useState<User | null>(null);
  const [loading, setLoading] = useState(true);
  const [error, setError] = useState<Error | null>(null);

  useEffect(() => {
    let cancelled = false;

    fetchUser(userId)
      .then((data) => {
        if (!cancelled) {
          setUser(data);
          setLoading(false);
        }
      })
      .catch((err) => {
        if (!cancelled) {
          setError(err);
          setLoading(false);
        }
      });

    return () => {
      cancelled = true;
    };
  }, [userId]);

  if (loading) return <Spinner />;
  if (error) return <div>에러: {error.message}</div>;
  return (
    <div>
      <h2>{user!.name}</h2>
      <p>{user!.email}</p>
    </div>
  );
}
```

이 패턴에는 구조적인 문제가 있습니다.

- `loading`, `user`, `error` 세 가지 State를 직접 동기화해야 합니다.
- `cancelled` 플래그는 unmount 후 State 업데이트를 막기 위한 것인데, 작성을 빠뜨리기 쉽습니다.
- Loading UI와 Error UI가 컴포넌트 본체에 섞여 있어 가독성이 떨어집니다.
- TypeScript에서 `user!`처럼 non-null assertion이 필요합니다.

`use` 훅은 이런 문제를 전부 제거합니다.

## Suspense와 결합한 기본 패턴

`use` 훅이 Promise를 중단시키려면 반드시 위쪽 트리에 `<Suspense>`가 있어야 합니다.

```tsx
import { use, Suspense, useMemo } from "react";

async function fetchUser(userId: string): Promise<User> {
  const res = await fetch(`/api/users/${userId}`);
  if (!res.ok) throw new Error(`User fetch failed: ${res.status}`);
  return res.json();
}

function UserProfile({ userPromise }: { userPromise: Promise<User> }) {
  const user = use(userPromise);
  return (
    <div>
      <h2>{user.name}</h2>
      <p>{user.email}</p>
    </div>
  );
}

function UserPage({ userId }: { userId: string }) {
  // Promise는 렌더마다 재생성되지 않도록 useMemo로 메모이제이션
  const userPromise = useMemo(() => fetchUser(userId), [userId]);

  return (
    <Suspense fallback={<div className="skeleton h-24 w-full rounded" />}>
      <UserProfile userPromise={userPromise} />
    </Suspense>
  );
}
```

`useMemo` 없이 `const userPromise = fetchUser(userId)`를 렌더 함수 본체에 직접 쓰면, 재렌더링마다 새 Promise가 생성되어 무한 suspend 루프에 빠집니다. Promise는 항상 컴포넌트 외부 또는 `useMemo`로 안정화해야 합니다.

## Parent에서 Promise를 전달하는 설계

`use` 훅이 만들어내는 자연스러운 설계 패턴 중 하나는 Promise를 Parent에서 생성하고 Child에 전달하는 구조입니다. 단순한 스타일 선택이 아니라 **병렬 페칭**을 가능하게 하는 구조적 이점이 있습니다.

```tsx
type Post = {
  id: string;
  title: string;
  slug: string;
  publishedAt: string;
};

async function fetchUserPosts(userId: string): Promise<Post[]> {
  const res = await fetch(`/api/users/${userId}/posts`);
  if (!res.ok) throw new Error("Posts fetch failed");
  return res.json();
}

function UserPostList({ postsPromise }: { postsPromise: Promise<Post[]> }) {
  const posts = use(postsPromise);
  return (
    <ul className="space-y-2">
      {posts.map((post) => (
        <li key={post.id}>
          <a href={`/blog/${post.slug}`} className="hover:underline">
            {post.title}
          </a>
          <time className="ml-2 text-sm text-gray-500">{post.publishedAt}</time>
        </li>
      ))}
    </ul>
  );
}

function UserDashboard({ userId }: { userId: string }) {
  // 두 Request가 동시에 시작됩니다 — Waterfall 없음
  const userPromise = useMemo(() => fetchUser(userId), [userId]);
  const postsPromise = useMemo(() => fetchUserPosts(userId), [userId]);

  return (
    <div className="space-y-6">
      <Suspense fallback={<ProfileSkeleton />}>
        <UserProfile userPromise={userPromise} />
      </Suspense>
      <Suspense fallback={<PostListSkeleton />}>
        <UserPostList postsPromise={postsPromise} />
      </Suspense>
    </div>
  );
}
```

`useEffect` 방식에서는 첫 번째 페치가 완료된 후에야 두 번째 페치를 시작하는 Waterfall이 발생하기 쉽습니다. Parent에서 Promise를 동시에 생성하는 이 패턴으로 그 문제를 자연스럽게 해결합니다.

## Error Boundary 연동

`use`에 전달한 Promise가 reject되면, 렌더링 중 에러가 throw됩니다. 이 에러는 가장 가까운 `ErrorBoundary`가 캐치합니다.

```tsx
import { Component, type ReactNode } from "react";

type ErrorBoundaryProps = {
  children: ReactNode;
  fallback: (error: Error, reset: () => void) => ReactNode;
};

type ErrorBoundaryState = { error: Error | null };

class ErrorBoundary extends Component<ErrorBoundaryProps, ErrorBoundaryState> {
  state: ErrorBoundaryState = { error: null };

  static getDerivedStateFromError(error: Error): ErrorBoundaryState {
    return { error };
  }

  reset = () => this.setState({ error: null });

  render() {
    if (this.state.error) {
      return this.props.fallback(this.state.error, this.reset);
    }
    return this.props.children;
  }
}

// 중첩 순서: ErrorBoundary가 Suspense를 감쌉니다
function UserDashboard({ userId }: { userId: string }) {
  const userPromise = useMemo(() => fetchUser(userId), [userId]);

  return (
    <ErrorBoundary
      fallback={(error, reset) => (
        <div className="rounded border border-red-300 bg-red-50 p-4">
          <p className="font-medium text-red-700">데이터를 불러오지 못했습니다.</p>
          <code className="mt-1 block text-sm text-red-600">{error.message}</code>
          <button
            onClick={reset}
            className="mt-3 rounded bg-red-600 px-3 py-1 text-sm text-white"
          >
            다시 시도
          </button>
        </div>
      )}
    >
      <Suspense fallback={<ProfileSkeleton />}>
        <UserProfile userPromise={userPromise} />
      </Suspense>
    </ErrorBoundary>
  );
}
```

중첩 순서가 중요합니다. `ErrorBoundary`가 `Suspense`를 감싸야 합니다. 반대로 하면 에러가 예상치 못한 위치로 전파될 수 있습니다.

`reset` 콜백은 Retry 흐름에 유용합니다. `reset()`을 호출하면서 Promise를 재생성하면 새로운 페치 시도가 트리거됩니다.

## 조건부로 호출 가능한 `use`

모든 React 개발자가 숙지하고 있는 Hook 규칙이 있습니다. "조건문이나 루프 안에서 Hook을 호출하지 마라." `use`는 이 규칙의 유일한 예외입니다.

```tsx
function OptionalUserCard({
  userId,
  userPromise,
}: {
  userId: string | null;
  userPromise: Promise<User> | null;
}) {
  if (!userId || !userPromise) {
    return <div className="text-gray-400">선택된 사용자가 없습니다.</div>;
  }

  // early return 이후에도 use 호출 가능 — 다른 Hook은 불가
  const user = use(userPromise);
  return (
    <div className="card">
      <h3>{user.name}</h3>
      <p>{user.email}</p>
    </div>
  );
}
```

`use`는 Context를 읽는 데도 사용할 수 있어 `useContext`의 대안으로도 동작합니다.

```tsx
import { use } from "react";
import { ThemeContext } from "./ThemeContext";

function ThemedButton({ label }: { label: string }) {
  const theme = use(ThemeContext);
  return (
    <button
      style={{ backgroundColor: theme.primaryColor, color: theme.textColor }}
      className="rounded px-4 py-2 font-medium"
    >
      {label}
    </button>
  );
}
```

## Server Components와 구분

Next.js App Router를 사용 중이라면 `use` 훅이 항상 최선의 선택은 아닙니다. Server Components에서는 Promise를 서버에서 직접 `await`할 수 있습니다.

```tsx
// app/users/[id]/page.tsx — Server Component
// use 훅 없이 async/await 직접 사용 가능
export default async function UserPage({
  params,
}: {
  params: { id: string };
}) {
  const user = await fetchUser(params.id);
  // HTML이 생성되기 전에 데이터가 준비됩니다
  return <UserProfile user={user} />;
}
```

판단 기준을 정리하면 다음과 같습니다.

| 상황 | 접근 방식 |
|---|---|
| 정적 데이터, Client 인터랙션 불필요 | Server Component에서 `await` |
| Client State에 따라 동적으로 변하는 데이터 | `use` 훅 + Suspense |
| 버튼 클릭, 폼 제출 등 사용자 액션으로 트리거되는 페치 | Server Action 또는 Event Handler |
| API Key, DB 쿼리 등 민감한 처리 | 항상 Server Component |

Server Components를 활용하면 JavaScript Bundle 크기를 줄이고, 페치 로직이 브라우저에 노출되지 않습니다. `use` 훅은 데이터가 Client State에 의존하는 경우를 위해 남겨두는 것이 좋습니다.

## Next.js App Router와의 통합 예시

Client Component에서 `use`를 쓰는 실제 Next.js 패턴입니다.

```tsx
// components/LiveUserStatus.tsx
"use client";

import { use, Suspense, useMemo } from "react";

type UserStatus = {
  userId: string;
  status: "online" | "offline" | "busy";
  lastSeen: string;
};

async function fetchUserStatus(userId: string): Promise<UserStatus> {
  const res = await fetch(`/api/users/${userId}/status`, {
    // 30초마다 재검증 — 정적 캐시 없이 최신 상태 유지
    next: { revalidate: 30 },
  });
  if (!res.ok) throw new Error("Status fetch failed");
  return res.json();
}

function StatusBadge({ statusPromise }: { statusPromise: Promise<UserStatus> }) {
  const { status } = use(statusPromise);
  const colors: Record<UserStatus["status"], string> = {
    online: "bg-green-500",
    offline: "bg-gray-400",
    busy: "bg-yellow-500",
  };
  return (
    <span className={`inline-block h-2.5 w-2.5 rounded-full ${colors[status]}`} />
  );
}

export function LiveUserStatus({ userId }: { userId: string }) {
  const statusPromise = useMemo(() => fetchUserStatus(userId), [userId]);
  return (
    <Suspense fallback={<span className="inline-block h-2.5 w-2.5 rounded-full bg-gray-200" />}>
      <StatusBadge statusPromise={statusPromise} />
    </Suspense>
  );
}
```

이처럼 실시간으로 변하는 Client 상태는 `use` 훅이 적합합니다. 반면 초기 페이지 데이터는 Server Component에서 처리하는 것이 효율적입니다.

## 마무리

React 19의 `use` 훅은 특정 상황을 위한 정밀한 도구입니다. 모든 데이터 페칭을 대체하는 만능 해결책이 아닙니다.

핵심 내용을 정리합니다.

- `use(promise)`는 pending 중에는 렌더링을 중단하고, resolve되면 값을 반환하며, reject되면 에러를 throw합니다.
- Promise는 `useMemo`나 컴포넌트 외부에서 안정적으로 생성해야 합니다.
- Parent에서 여러 Promise를 동시에 생성하면 Waterfall 없이 병렬 페칭이 가능합니다.
- `ErrorBoundary`는 `Suspense` 바깥에 배치합니다.
- 다른 Hook과 달리 조건문 안에서도 호출할 수 있습니다.
- Next.js App Router에서는 Server Components와 역할을 명확히 나눠야 합니다.

아직 `useEffect`로 데이터 페칭을 작성하는 코드가 남아있다면, `use` 훅으로 전환해볼 것을 권장합니다. Loading과 Error 처리가 컴포넌트 외부로 분리되면서 코드의 가독성이 눈에 띄게 개선됩니다.
