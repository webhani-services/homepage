React Compiler v1.0은 2025년 10월에 출시되었습니다. 2026년 초에는 React 19를 사용하는 신규 프로젝트의 표준 설정이 되어가고 있습니다. 실질적인 변화는 `useMemo`, `useCallback`, `React.memo` — 수년간 React 개발자의 통과의례였던 수동 Performance 최적화 도구들 — 이 점점 Legacy Pattern으로 취급되고 있다는 점입니다.

이 글에서는 Compiler의 동작 원리, 기존 Codebase에서 안전하게 제거할 수 있는 코드, 그리고 여전히 수동 개입이 의미 있는 상황을 살펴보겠습니다.

## Compiler가 하는 일

React Compiler는 Build 시 실행되는 Babel/SWC Plugin입니다. Component와 Hook을 정적으로 분석하고, 자동으로 Memoize해도 안전한 값을 식별한 후, 컴파일된 출력에 `useMemo`/`useCallback`에 해당하는 처리를 삽입합니다 — 개발자의 Annotation 없이도요.

Compiler가 분석하는 대상:
- Props와 State에만 의존하는 순수한 계산
- Event Handler나 Dependency로 전달되는 안정적인 Function Reference
- 관련 없는 State 변경 시 재Render가 불필요한 Component Subtree

핵심은 React의 Rule을 따르는 코드(순수한 Rendering, Render 내 Side Effect 없음)라면 Compiler가 이를 정확하게 판단할 수 있다는 점입니다. Type System이나 Annotation 없이 코드 구조에서 추론합니다.

## Before / After

React Compiler 도입 전 일반적인 패턴:

```typescript
function ProductList({ products, onSelect }: Props) {
  const sortedProducts = useMemo(
    () => [...products].sort((a, b) => a.name.localeCompare(b.name)),
    [products]
  );

  const handleSelect = useCallback(
    (id: string) => onSelect(id),
    [onSelect]
  );

  return (
    <ul>
      {sortedProducts.map((p) => (
        <ProductItem key={p.id} product={p} onSelect={handleSelect} />
      ))}
    </ul>
  );
}
```

React Compiler를 활성화하면 컴파일된 출력이 Memoization을 담당합니다. Source는 다음과 같이 됩니다:

```typescript
function ProductList({ products, onSelect }: Props) {
  const sortedProducts = [...products].sort((a, b) =>
    a.name.localeCompare(b.name)
  );

  return (
    <ul>
      {sortedProducts.map((p) => (
        <ProductItem key={p.id} product={p} onSelect={onSelect} />
      ))}
    </ul>
  );
}
```

Performance 특성은 동일합니다. Source의 노이즈가 크게 줄어듭니다.

## React Compiler Setup

```bash
npm install --save-dev babel-plugin-react-compiler
```

```json
// babel.config.json
{
  "plugins": ["babel-plugin-react-compiler"]
}
```

Next.js 15 이상에서는 `next.config.ts`에 기본 내장되어 있습니다:

```typescript
import type { NextConfig } from "next";

const config: NextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default config;
```

Compiler는 기본적으로 Opt-in 모드로 동작합니다. 안전하게 최적화할 수 있는 Component만 처리하며, React의 Rule을 위반하는 Component는 건너뜁니다 — 앱이 깨지지 않습니다.

## 기존 Codebase에서 제거할 수 있는 것

기존 Codebase에 도입할 때의 현실적인 절차:

1. Compiler를 활성화하고 Test Suite 실행 — 문제가 없는지 확인
2. React Compiler의 ESLint Plugin으로 불필요한 수동 Memoization을 감지:

```bash
npm install --save-dev eslint-plugin-react-compiler
```

3. Props와 State에만 의존하는 순수 계산을 감싸는 `useMemo` 제거
4. 컴파일된 Child Component에 전달되는 Handler의 `useCallback` 제거
5. Compiler가 자동으로 처리하는 Component의 `React.memo` 제거

수동 Memoization을 유지해야 하는 경우:
- 참조 안정성이 필요한 외부 Library 연동 (복잡한 Chart Configuration 등)
- 의도적으로 지연되는 무거운 계산 (On-demand 데이터 변환 등)
- `"use no memo"` Directive로 Opt-out한 Component

## Opt-out이 필요한 경우

의도적으로 비순수한 동작을 하는 Component를 위한 Escape Hatch:

```typescript
function LegacyWidget({ data }: Props) {
  "use no memo";
  // 이 Component는 Compiler의 최적화 대상에서 제외됩니다.
  // 특수한 참조 요구사항이 있는 Third-party 연동에 사용하세요.
}
```

## 실제 이점

수동 Memoization을 제거하는 가장 큰 이점은 Raw Performance보다 가독성과 인지 부하 감소입니다. `useMemo`가 5개, `useCallback`이 3개 있는 Component는 읽는 사람이 Dependency Array를 추적하고 각 값의 재계산 시점을 추론해야 합니다.

Compiler가 이를 처리하면, Component 코드는 "어떻게 최적화할지"가 아니라 "무엇을 Render할지"를 표현하는 방향으로 돌아옵니다. 이는 버그 발생 가능성이 있는 Surface Area의 유의미한 감소입니다.

## 요약

- React Compiler v1.0은 Production 준비 완료, React 19 신규 프로젝트의 표준 경로로
- React의 Rule을 따르는 Component에 대해 Memoization을 자동으로 처리
- `useMemo`, `useCallback`, `React.memo`는 Edge Case에서 여전히 유효 — 단, 기본값이어선 안 됨
- 기존 Codebase 도입은 ESLint Plugin과 Test Suite를 활용해 점진적으로 진행
- 주요 이점은 더 깔끔하고 읽기 쉬운 코드 — Performance는 유지됨
