2025년 10월 React Compiler v1.0이 안정 버전으로 출시된 지 약 6개월이 지났습니다. 2026년 초 개발 커뮤니티는 예고된 변화를 확인하고 있습니다. `useMemo`, `useCallback`, `React.memo`를 직접 작성하는 것이 대부분의 Component에서 불필요해졌습니다. 이는 React 코드를 작성하고 검토하는 방식에 실질적인 변화를 가져옵니다.

## React Compiler가 하는 일

React Compiler는 Build 시점에 동작하는 Babel Plugin으로, Component 코드를 정적으로 분석하여 Memoization을 자동으로 삽입합니다. React의 규칙(순수 Render 함수, 안정적인 Reference, 의존성 추적)을 이해하고, 이전에는 개발자가 직접 작성하던 최적화를 Build 단계에서 처리합니다.

```tsx
// 개발자가 작성하는 코드
function ProductList({ products, onAddToCart }: Props) {
  const filtered = products.filter(p => p.inStock);

  return (
    <ul>
      {filtered.map(product => (
        <ProductItem
          key={product.id}
          product={product}
          onAdd={onAddToCart}
        />
      ))}
    </ul>
  );
}

// Compiler가 Build 시점에 자동으로 처리하는 것:
// - products가 변경되지 않으면 filtered 계산을 Memoize
// - 자식 Component에 전달하는 onAddToCart Reference 안정화
// - 두 Props 모두 변경되지 않으면 ProductItem 재렌더링 방지
```

직접 `useMemo`, `useCallback`, `React.memo`를 작성할 필요가 없습니다.

## 실제 코드의 변화

Compiler 이전의 성능 최적화된 React 코드:

```tsx
// Compiler 이전 방식 (2024년 스타일)
function ProductList({ products, onAddToCart }: Props) {
  const filtered = useMemo(
    () => products.filter(p => p.inStock),
    [products]
  );

  const handleAdd = useCallback(
    (id: string) => onAddToCart(id),
    [onAddToCart]
  );

  return (
    <ul>
      {filtered.map(product => (
        <MemoizedProductItem
          key={product.id}
          product={product}
          onAdd={handleAdd}
        />
      ))}
    </ul>
  );
}
```

Compiler를 활용하면 Annotation 없이 동일한 Component를 작성할 수 있습니다:

```tsx
// 2026년 스타일 — 로직에 집중, 최적화는 Compiler에게
function ProductList({ products, onAddToCart }: Props) {
  const filtered = products.filter(p => p.inStock);

  return (
    <ul>
      {filtered.map(product => (
        <ProductItem
          key={product.id}
          product={product}
          onAdd={onAddToCart}
        />
      ))}
    </ul>
  );
}
```

코드량이 줄고, 의존성 배열 실수(흔한 React Bug의 원인)가 사라지며, 가독성이 높아집니다.

## 여전히 주의가 필요한 부분

Compiler가 모든 최적화 문제를 해결하는 것은 아닙니다.

**부작용이 있는 계산**: 파생 값 계산에 Side Effect(외부 Store 쓰기, Analytics Event 트리거 등)가 포함된 경우 Compiler는 안전을 위해 최적화를 건너뜁니다. 이런 케이스는 명시적으로 구조화가 필요합니다.

**Third-party Library Hook**: Zustand, Jotai, TanStack Query 등 주요 Library는 Compiler 호환성을 확인했습니다. 오래된 Custom Hook이 많은 기존 코드베이스는 Compiler 활성화 후 예상치 못한 재렌더링이 발생하는지 검토가 필요합니다.

**성능 측정 습관 유지**: Compiler가 특정 종류의 성능 문제를 해결해 주지만, 아키텍처 문제(과도한 데이터 Fetch, 깊은 Component 트리, 과도한 상태 구독)는 여전히 개발자의 책임입니다. React DevTools Profiler를 계속 활용하세요.

## Compiler 활성화 방법

Next.js 16에서는 기본적으로 활성화되어 있습니다. 명시적으로 설정하려면:

```typescript
// next.config.ts
import type { NextConfig } from 'next';

const nextConfig: NextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default nextConfig;
```

기존 프로젝트에서 단계적으로 적용하려면 Annotation Mode를 사용합니다:

```typescript
// next.config.ts — Component별 선택적 적용
const nextConfig: NextConfig = {
  experimental: {
    reactCompiler: {
      compilationMode: 'annotation',
    },
  },
};
```

그런 다음 개별 Component 상단에 `"use memo"`를 추가해 선택적으로 최적화를 활성화합니다:

```tsx
"use memo";

function MyComponent({ data }: Props) {
  // Compiler 최적화가 이 Component에 적용됨
}
```

이 방식으로 신뢰가 쌓이면 전체 프로젝트로 확장할 수 있습니다.

## 정리

신규 프로젝트는 Memoization Annotation 없이 간단하게 Component를 작성하고 Compiler에 맡기는 방식을 권장합니다. React DevTools Profiler로 Regression이 없는지 확인하세요.

기존 프로젝트는 서두를 필요가 없습니다. Annotation Mode로 먼저 활성화하여 몇 개의 Component를 검증한 후, 확신이 생기면 전체 적용을 고려하세요. 기존의 `useMemo`와 `useCallback`은 Compiler와 공존할 수 있으며, 단지 중복될 뿐입니다.

가장 큰 이점은 코드 감소뿐만 아니라, 신규 개발자가 올바른 React를 작성하기 위한 전제 조건으로 `useMemo` 사용 시점을 학습할 필요가 없어진다는 점입니다. 인지적 부담이 줄고 Component가 로직 중심으로 읽기 쉬워집니다.
