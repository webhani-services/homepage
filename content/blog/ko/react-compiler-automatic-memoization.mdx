2025년 10월 React Compiler 1.0이 Stable로 릴리스된 이후, "더 이상 `useMemo`를 직접 쓸 필요가 없다"는 이야기가 점점 당연한 말이 되어가고 있습니다. Meta Quest Store에서는 초기 로딩 속도 최대 12% 향상, 일부 인터랙션에서 2.5배 빠른 응답 속도가 보고됐고, Next.js 16에서는 설정 파일에 한 줄만 추가하면 바로 활성화됩니다.

이 글에서는 React Compiler가 내부적으로 어떻게 동작하는지, 기존 프로젝트에 어떻게 안전하게 도입할 수 있는지를 실제 코드와 함께 살펴봅니다.

## React Compiler가 해결하는 문제

React는 상태나 Props가 변경되면 해당 컴포넌트와 모든 하위 컴포넌트를 다시 렌더링합니다. 자식 컴포넌트의 Props가 실제로 변경되지 않았더라도 부모가 렌더링되면 함께 실행됩니다. 이를 막기 위해 개발자들은 오랫동안 다음과 같은 코드를 작성해왔습니다:

```tsx
// 기존 방식 — 로직보다 최적화 코드가 더 많아지는 문제
const sortedItems = useMemo(() => [...items].sort(compareFn), [items]);
const handleDelete = useCallback((id: string) => removeItem(id), [removeItem]);
const MemoizedCard = React.memo(ItemCard);
```

React Compiler는 이 최적화를 빌드 타임에 자동으로 처리합니다. 컴파일러가 컴포넌트 코드를 정적 분석하여 어떤 값과 함수를 캐시해도 안전한지 판단하고 Memoization을 자동으로 주입합니다.

## 도입 방법

### Next.js 16

`next.config.ts`에 한 줄 추가합니다:

```ts
import type { NextConfig } from "next";

const nextConfig: NextConfig = {
  reactCompiler: true,
};

export default nextConfig;
```

### Vite / Babel 프로젝트

```bash
npm install -D babel-plugin-react-compiler
```

```json
{
  "plugins": ["babel-plugin-react-compiler"]
}
```

### 컴포넌트 단위로 점진적 도입

`"use memo"` 디렉티브로 특정 컴포넌트에만 먼저 적용할 수 있습니다:

```tsx
"use memo";

export function ProductList({ items }: { items: Item[] }) {
  const filtered = items.filter((item) => item.inStock);
  return (
    <ul>
      {filtered.map((item) => (
        <li key={item.id}>{item.name}</li>
      ))}
    </ul>
  );
}
```

## 기존 useMemo와 useCallback은 어떻게 되나요?

Compiler는 기존에 작성된 `useMemo`, `useCallback`을 그대로 존중합니다. React 팀의 공식 가이드는 "새로 작성하는 코드에서는 Compiler에 맡기고, 정밀한 제어가 필요한 경우에만 수동으로 작성하라"입니다.

수동 Memoization이 여전히 유효한 경우:
- **캐시 수명 주기를 명시적으로 제어해야 할 때**
- **서드파티 라이브러리 제약** — Compiler가 분석할 수 없는 외부 Hook과의 조합
- **부수 효과 순서 보장**

## React 규칙 준수가 핵심

Compiler는 React의 규칙을 따르는 코드만 최적화합니다. ESLint 플러그인을 먼저 설치하는 것이 중요합니다:

```bash
npm install -D eslint-plugin-react-compiler
```

```json
{
  "plugins": ["react-compiler"],
  "rules": {
    "react-compiler/react-compiler": "error"
  }
}
```

Lint 규칙 추가 → 위반 사항 수정 → Compiler 활성화 순서로 진행하면 기존 프로젝트에서도 안전하게 도입할 수 있습니다.

## 적용 효과 확인

React DevTools Profiler로 Compiler 덕분에 재렌더링을 건너뛴 컴포넌트를 확인할 수 있습니다. `why-did-you-render`와 함께 사용하면 불필요한 재렌더링이 남아있는 컴포넌트를 빠르게 파악할 수 있습니다:

```tsx
if (process.env.NODE_ENV === "development") {
  const { default: wdyr } = await import("@welldone-software/why-did-you-render");
  wdyr(React, { trackAllPureComponents: true });
}
```

## 정리

React Compiler 1.0은 Performance 최적화를 런타임 보일러플레이트에서 빌드 타임 자동화로 전환하는 큰 변화입니다. 도입 위험도는 낮습니다. ESLint 플러그인으로 규칙 위반을 먼저 정리하고, Config 파일에 한 줄 추가하면 됩니다.

실질적인 효과는 컴포넌트당 인지 부하 감소입니다. "이걸 Memoize 해야 할까?" 고민 대신, 비즈니스 로직에 집중하고 최적화는 Compiler에 맡길 수 있습니다.
