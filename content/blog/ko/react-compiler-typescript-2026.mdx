React 19.2가 출시되면서 React Compiler가 Stable 상태로 승격되었습니다. 단순한 버전 업이 아닙니다. React의 성능 최적화 모델이 근본적으로 바뀌는 전환점입니다. 개발자가 직접 메모이제이션을 관리하는 시대가 서서히 마무리됩니다. 그와 동시에 2026년 프론트엔드 현장에서 TypeScript는 이제 "도입할 것인가"가 아닌 "얼마나 잘 쓰고 있는가"의 문제가 되었습니다.

## React Compiler가 해결하는 문제

React Compiler는 빌드 타임에 컴포넌트 코드를 정적 분석해서, 개발자가 수동으로 작성했던 `useMemo`, `useCallback`, `React.memo`를 자동으로 삽입합니다. 목적은 간단합니다. 불필요한 리렌더링을 방지하는 최적화의 책임을 개발자에서 프레임워크로 이전하는 것입니다.

### 기존 방식의 문제점

React Compiler 이전에는 성능 최적화를 위해 이런 코드를 직접 작성해야 했습니다:

```tsx
// 컴파일러 이전: 수동 메모이제이션
const UserCard = React.memo(({ user, onDelete }: Props) => {
  const formattedDate = useMemo(
    () => formatDate(user.createdAt),
    [user.createdAt]
  );
  const handleDelete = useCallback(
    () => onDelete(user.id),
    [onDelete, user.id]
  );

  return (
    <div>
      <span>{user.name}</span>
      <span>{formattedDate}</span>
      <button onClick={handleDelete}>삭제</button>
    </div>
  );
});
```

이 코드의 문제는 작성이 어렵다는 게 아닙니다. 의존성 배열을 잘못 작성하기 쉽다는 점입니다. 배열에 빠진 의존성이 있으면 stale closure 버그가 생기고, 불필요한 값이 들어가면 최적화 효과가 사라집니다. React에 익숙하지 않은 팀원이 작성한 코드에서 특히 자주 발생하는 오류였습니다.

### 컴파일러 이후

React Compiler를 활성화하면 이렇게 작성해도 됩니다:

```tsx
// 컴파일러 이후: 메모이제이션 없이 직관적으로
const UserCard = ({ user, onDelete }: Props) => {
  const formattedDate = formatDate(user.createdAt);
  const handleDelete = () => onDelete(user.id);

  return (
    <div>
      <span>{user.name}</span>
      <span>{formattedDate}</span>
      <button onClick={handleDelete}>삭제</button>
    </div>
  );
};
```

컴파일러가 어떤 값이 함께 변경되는지 분석해서, 빌드 결과물에 최적화된 코드를 자동으로 추가합니다.

## Next.js에서 활성화하기

Next.js 15.x 이상에서는 React Compiler 지원이 내장되어 있습니다:

```js
// next.config.mjs
/** @type {import('next').NextConfig} */
const nextConfig = {
  experimental: {
    reactCompiler: true,
  },
};

export default nextConfig;
```

기존 코드의 `useMemo`나 `useCallback`을 즉시 삭제할 필요는 없습니다. 컴파일러는 이런 코드가 남아 있어도 정상 동작합니다. 불필요한 훅 제거는 컴파일러가 안정적으로 동작한다는 확신이 생긴 뒤 단계적으로 진행하면 됩니다.

### 컴파일러가 최적화하지 못하는 패턴

컴파일러는 React의 규칙을 따르는 코드에서만 제대로 동작합니다. 다음 패턴은 컴파일러 최적화가 적용되지 않습니다:

- Props나 State를 직접 변경 (immutability 위반)
- `useEffect` 내부에 관련 없는 여러 부작용 혼재
- 같은 입력에 대해 다른 참조를 반환하는 Custom Hook

`eslint-plugin-react-compiler`를 사용하면 이런 패턴을 미리 감지할 수 있습니다:

```bash
npm install --save-dev eslint-plugin-react-compiler
```

```json
// .eslintrc.json
{
  "plugins": ["react-compiler"],
  "rules": {
    "react-compiler/react-compiler": "error"
  }
}
```

이 Lint 규칙을 먼저 클린하게 만드는 것이 컴파일러 도입의 전제 조건입니다.

## TypeScript: "쓸 것인가"에서 "잘 쓰고 있는가"로

TypeScript 도입률이 높아진 지금, 더 중요한 질문은 TypeScript를 제대로 활용하고 있는지입니다. TypeScript를 켜놓고 `any`로 타입 오류를 막아놓으면 JavaScript와 다를 게 없습니다.

TypeScript의 진짜 가치는 타입으로 코드 설계를 강제할 때 나옵니다.

### 타입으로 설계하기

```typescript
// TypeScript를 제대로 쓰지 않는 예시
function processPayment(data: any) {
  return fetch('/api/payment', {
    method: 'POST',
    body: JSON.stringify(data),
  });
}

// TypeScript를 제대로 쓰는 예시
type PaymentRequest = {
  amount: number;
  currency: 'JPY' | 'USD' | 'EUR';
  customerId: string;
  description?: string;
};

async function processPayment(
  data: PaymentRequest
): Promise<{ transactionId: string }> {
  const res = await fetch('/api/payment', {
    method: 'POST',
    body: JSON.stringify(data),
    headers: { 'Content-Type': 'application/json' },
  });
  return res.json();
}
```

`PaymentRequest` 타입을 정의하면, 호출 측에서 잘못된 통화 코드를 넘기거나 숫자/문자 혼용 실수를 컴파일 타임에 잡을 수 있습니다. 별도 문서 없이도 타입 자체가 API 명세 역할을 합니다.

### React 19의 타입 정의 개선

React 19에서는 Server Components와 Actions API를 위한 타입 정의가 크게 개선되었습니다:

```typescript
// Server Action 타입 정의
'use server';

export async function createPost(
  prevState: { error: string | null },
  formData: FormData
): Promise<{ error: string | null }> {
  const title = formData.get('title');
  if (typeof title !== 'string' || title.length === 0) {
    return { error: '제목은 필수 항목입니다' };
  }
  // DB 저장 처리...
  return { error: null };
}
```

`useActionState`와 조합하면 서버 사이드 유효성 검사 오류를 폼에 반영하는 타입 안전한 패턴을 구현할 수 있습니다.

## 실제 프로젝트 적용 전략

### 신규 프로젝트

React 19.2, TypeScript strict mode, React Compiler를 모두 처음부터 활성화한 상태로 시작하는 것을 권장합니다:

```bash
npx create-next-app@latest my-app \
  --typescript \
  --tailwind \
  --eslint \
  --app
```

`tsconfig.json`에서 `"strict": true`를 반드시 활성화하세요. 나중에 strict mode로 전환하는 비용은 처음부터 켜는 것보다 훨씬 큽니다.

### 기존 프로젝트

단계적 적용이 현실적입니다:

1. **React Compiler 활성화** (저위험): `next.config.mjs`에 한 줄 추가, 기존 코드 변경 불필요
2. **TypeScript 점진적 강화**: 전체 strict mode보다 `noImplicitAny`부터 시작, 파일 단위로 개선
3. **불필요한 메모이제이션 정리**: 컴파일러가 안정적으로 동작한다는 확인 후 단계적으로 정리

세 가지를 동시에 진행하면 문제 발생 시 원인 파악이 어렵습니다.

## 정리

React Compiler의 Stable 승격은 React가 "성능 최적화는 프레임워크가 담당한다"는 방향을 명확히 한 중요한 이정표입니다. TypeScript 표준화와 함께 프론트엔드 개발의 기준선이 높아지고 있습니다.

기존 코드베이스를 당장 전면 수정할 필요는 없습니다. 중요한 것은 새로운 프로젝트의 출발점이 바뀌었다는 점입니다. 수동 메모이제이션과 TypeScript 선택적 사용은 점점 레거시 접근 방식이 되어가고 있습니다.
