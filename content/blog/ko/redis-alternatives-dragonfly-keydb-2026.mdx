## Redis 라이선스 변경과 그 여파

Redis는 10년 이상 웹 애플리케이션 캐싱 레이어의 표준이었습니다. 2024년 BSD에서 SSPL/RSALv2 라이선스로의 전환이 그 지위를 흔들고 있습니다. Redis를 내부 캐시로만 사용하는 팀에게 직접적인 영향은 크지 않지만, Redis를 기반으로 상용 서비스를 구축하거나 SaaS로 제공하는 경우 법적 노출이 현실적인 우려가 됩니다.

2026년 현재, 클라우드 프로바이더 대비 8~12% 더 나은 비용 효율성을 보고하는 Dragonfly와 KeyDB가 대안으로 주목받고 있습니다.

## Dragonfly

Dragonfly는 멀티스레드 아키텍처를 핵심으로 하는 Redis 호환 솔루션입니다:

```bash
# Docker로 즉시 시작 — Redis 드롭인 대체
docker run --rm -p 6379:6379 docker.dragonflydb.io/dragonflydb/dragonfly

# 기존 Redis CLI로 바로 연결
redis-cli -h localhost -p 6379 ping
# PONG
```

```python
# 기존 클라이언트 코드 변경 없이 사용
import redis

client = redis.Redis(host="dragonfly-host", port=6379)
client.set("session:user123", json.dumps(user_data), ex=3600)
data = client.get("session:user123")
```

주요 특징:
- Redis Protocol (RESP2/RESP3) 완전 호환
- 멀티스레드: Redis의 싱글스레드 모델과 달리 모든 CPU 코어 활용
- Business Source License (BSL) — 4년 후 Apache 2.0으로 전환
- 모든 Redis 데이터 구조 지원 (Strings, Hashes, Lists, Sets, Sorted Sets, Streams)

**BSL 라이선스 주의사항**: Dragonfly는 4년 기간이 지나기 전까지 완전한 오픈소스가 아닙니다. 엄격한 오픈소스 정책을 가진 팀은 이 점을 확인해야 합니다.

## KeyDB

KeyDB는 Redis의 완전한 포크로, 활성-활성 복제(Active-Active Replication)가 핵심 기능입니다:

```conf
# keydb.conf
server-threads 4
active-replica yes
repl-diskless-sync yes
```

```bash
keydb-server /etc/keydb/keydb.conf
```

활성-활성 복제는 복제 쌍의 두 노드 모두 쓰기를 받을 수 있어 가용성을 높이고 Primary 부하를 분산합니다. Redis Sentinel이나 Cluster 없이도 높은 가용성을 구현할 수 있습니다.

## 정직한 성능 측정

8~12% 비용 효율성 수치는 동일 클라우드 인스턴스에서 고동시성 환경의 처리량 비교 결과입니다. 실제 개선폭은 워크로드에 따라 크게 다릅니다. 벤더 벤치마크가 아닌 자체 워크로드로 측정해야 합니다:

```python
# 실제 워크로드 기반 측정 예시
import redis
import time
import statistics

def measure_latency(client, n=10000):
    writes, reads = [], []

    for i in range(n):
        start = time.perf_counter()
        client.set(f"bench:{i}", f"val:{i}")
        writes.append(time.perf_counter() - start)

    for i in range(n):
        start = time.perf_counter()
        client.get(f"bench:{i}")
        reads.append(time.perf_counter() - start)

    return {
        "write_p50_ms": statistics.median(writes) * 1000,
        "write_p99_ms": sorted(writes)[int(n * 0.99)] * 1000,
        "read_p50_ms": statistics.median(reads) * 1000,
        "read_p99_ms": sorted(reads)[int(n * 0.99)] * 1000,
    }

# 동일 클라이언트 라이브러리로 세 가지 모두 측정 가능
redis_client = redis.Redis(host="redis-host")
dragonfly_client = redis.Redis(host="dragonfly-host")
keydb_client = redis.Redis(host="keydb-host")

print("Redis:", measure_latency(redis_client))
print("Dragonfly:", measure_latency(dragonfly_client))
print("KeyDB:", measure_latency(keydb_client))
```

성능 향상은 고동시성 + 멀티코어 인스턴스 환경에서 가장 두드러집니다. Redis 인스턴스가 이미 CPU-idle 상태라면 성능 차이는 미미합니다.

## 전환 판단 기준

**전환이 명확한 경우:**
- Redis를 핵심 컴포넌트로 포함한 상용 SaaS를 제공하는 경우 (SSPL 적용 가능성)
- 멀티코어 인스턴스에서 Redis가 싱글스레드로 CPU 병목이 된 경우
- Redis Enterprise 가격 없이 Active-Active 복제가 필요한 경우

**전환이 급하지 않은 경우:**
- Redis를 순수 내부 캐시로만 사용하는 경우 (SSPL 영향 없음)
- ElastiCache, Memorystore 등 매니지드 서비스를 사용하는 경우 (프로바이더가 라이선스 처리)
- 기존 Redis 클러스터가 안정적으로 운영 중이고 CPU 병목이 없는 경우

## Blue/Green 전환 방식

```yaml
# Kubernetes에서 Dragonfly 병렬 운영
apiVersion: apps/v1
kind: Deployment
metadata:
  name: dragonfly
spec:
  replicas: 1
  template:
    spec:
      containers:
        - name: dragonfly
          image: docker.dragonflydb.io/dragonflydb/dragonfly
          resources:
            requests:
              memory: "4Gi"
              cpu: "2"
            limits:
              memory: "8Gi"
              cpu: "4"
```

권장 전환 단계:
1. 기존 Redis 옆에 Dragonfly/KeyDB 배포
2. 먼저 읽기 트래픽을 새 인스턴스로 전환 (검증)
3. 데이터 일관성과 레이턴시 검증 후 쓰기 트래픽 전환
4. Redis를 스탠바이로 유지
5. 문제 없으면 Redis 제거

## webhani의 시각

Redis에서 Dragonfly나 KeyDB로의 기술적 전환 경로는 비교적 단순합니다. Redis Protocol 호환성 덕분에 대부분의 기존 클라이언트 코드를 그대로 사용할 수 있습니다.

핵심 질문은 전환이 정당화되는가입니다.

대부분의 팀이 Redis를 표준 내부 캐시로 사용하는 경우: 급할 이유가 없습니다. SSPL 노출이 있거나 Redis가 CPU 병목인 팀: 전환 비용이 낮고 이점이 구체적입니다.

결정 전에 자체 워크로드로 측정하세요. 8~12% 효율성 수치는 유리한 조건의 벤치마크 상한선이지, 실제 워크로드에서 보장되는 최솟값이 아닙니다.
