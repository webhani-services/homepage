InfoQ가 2026년 3월에 보도한 2025년 State of JavaScript 설문 결과는 JavaScript 생태계의 성숙을 수치로 확인해줍니다. 핵심은 두 가지입니다. TypeScript를 전용으로 사용하는 개발자가 40%에 달했고, Meta-Framework가 대부분 Web 프로젝트의 표준 진입점이 되었습니다. 남은 논쟁은 주변부에 있습니다.

## TypeScript의 압도적인 보급

설문의 주요 수치:

- TypeScript **전용 사용**: 40%
- TypeScript + JavaScript **혼용**: 46%
- JavaScript **전용**: 14%

응답자의 86%가 어떤 형태로든 TypeScript를 사용합니다. 2026년에 "TypeScript를 도입해야 하는가"라는 논쟁은 이미 끝났습니다. 현재의 논점은 "TypeScript 설정을 얼마나 엄격하게 유지하는가"입니다.

신규 프로젝트를 시작할 때 권장하는 기본 설정:

```typescript
// tsconfig.json — 2026년 신규 프로젝트 기본 설정
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitReturns": true,
    "target": "ES2024",
    "moduleResolution": "bundler",
    "paths": {
      "@/*": ["./src/*"]
    }
  }
}
```

`strict` 모드는 프로젝트 시작 시점부터 반드시 활성화해야 합니다. 기존 Codebase에 나중에 적용하면 수정 비용이 매우 커집니다. 변경 사항 하나하나는 단순하지만 수량이 방대하기 때문입니다.

TypeScript 보급을 뒷받침하는 세 가지 요인:

**에디터 지원의 성숙**: VSCode + TypeScript Language Server 조합으로 타입 검사, Auto-complete, Refactoring이 높은 수준으로 동작합니다.

**Framework의 기본 TypeScript화**: Next.js, NestJS, Remix는 모두 기본으로 TypeScript 프로젝트를 생성합니다. 신규 프로젝트에서 JavaScript를 선택할 이유가 줄었습니다.

**AI Coding Tool과의 시너지**: 타입 정보는 AI Coding Assistant의 Context 품질을 향상시킵니다. 명확한 타입 정의가 있으면 Claude Code와 GitHub Copilot이 더 정확한 코드를 제안합니다.

## Meta-Framework 표준화 시대

Next.js는 응답자의 59%가 사용하며 가장 높은 채택률을 기록했습니다. 설문 리포트의 핵심 분석:

> "2026년에 Next.js 같은 Meta-Framework는 대부분의 전문 Web 프로젝트에서 표준 진입점이 되었다. Routing, Data Fetching, Caching, Rendering 전략, API Layer를 처리하며, 많은 Web App에서 Backend는 Frontend Repository 안의 한 폴더가 되었다."

Server Component와 Server Action이 Frontend/Backend 경계를 흐리면서, 소중형 프로젝트는 별도 Backend Service 없이 단일 Next.js Repository로 구현이 가능해졌습니다:

```typescript
// app/actions/order.ts — Frontend Repository의 Server 로직
"use server";

import { db } from "@/lib/db";
import { stripe } from "@/lib/stripe";
import { revalidatePath } from "next/cache";

export async function createOrder(
  cartId: string,
  paymentMethodId: string
): Promise<{ success: boolean; orderId?: string }> {
  const cart = await db.cart.findUnique({
    where: { id: cartId },
    include: { items: { include: { product: true } } },
  });

  if (!cart) throw new Error("장바구니를 찾을 수 없습니다");

  const total = cart.items.reduce(
    (sum, item) => sum + item.quantity * item.product.price,
    0
  );

  const paymentIntent = await stripe.paymentIntents.create({
    amount: total,
    currency: "jpy",
    payment_method: paymentMethodId,
    confirm: true,
  });

  const order = await db.order.create({
    data: {
      cartId,
      total,
      stripePaymentIntentId: paymentIntent.id,
      status: "confirmed",
    },
  });

  revalidatePath("/orders");
  return { success: true, orderId: order.id };
}
```

```typescript
// app/checkout/page.tsx — Server Action을 직접 호출하는 UI
"use client";

import { createOrder } from "@/actions/order";
import { useRouter } from "next/navigation";

export default function CheckoutPage({ cartId }: { cartId: string }) {
  const router = useRouter();

  const handleSubmit = async (paymentMethodId: string) => {
    const result = await createOrder(cartId, paymentMethodId);
    if (result.success) router.push(`/orders/${result.orderId}`);
  };

  return <CheckoutForm onSubmit={handleSubmit} />;
}
```

Database 접근, 결제 처리, Cache 무효화가 모두 Server Side에서 처리됩니다. 이 패턴에서는 별도 API Layer가 필요 없습니다.

## Next.js에 대한 솔직한 비판

Next.js는 설문에서 가장 많은 코멘트를 받은 Framework이기도 합니다. 주요 비판:

**Caching 동작의 복잡성**: Next.js 13~15에서 기본으로 활성화된 자동 Cache가 예측하기 어려운 동작을 자주 일으켰습니다. Next.js 16은 이를 `'use cache'` 지시자로 명시적 Opt-in 방식으로 전환했습니다:

```typescript
// Next.js 16 — 명시적이고 예측 가능한 Cache
import { unstable_cache as cache } from "next/cache";

const getProducts = cache(
  async (category: string) => {
    return db.product.findMany({ where: { category } });
  },
  ["products"],
  { revalidate: 3600, tags: ["products"] }
);
```

**Vendor 의존성**: Next.js의 일부 기능이 Vercel Platform에 최적화되어 있어 Self-hosted 환경에서 동작 차이가 발생할 수 있습니다. 배포 환경을 선택하기 전에 어떤 Vercel 전용 기능을 사용하는지 파악해두세요.

## TypeScript 6.0 주요 변경사항

2026년에 TypeScript 6.0 RC가 출시되었습니다:

```typescript
// 더 엄격한 Type-only Import 강제
import type { User, Post } from "@/types";

// Const Type Parameter 추론 개선
function createConfig<const T extends Record<string, unknown>>(config: T): T {
  return config;
}

const config = createConfig({ mode: "production", retries: 3 });
// config.mode는 string이 아닌 "production"으로 추론됨
// config.retries는 number가 아닌 3으로 추론됨
```

기존 프로젝트를 업그레이드하는 경우 6.0의 엄격성 개선 사항을 단계적으로 활성화하는 것을 권장합니다.

## webhani의 권장 사항

2026년 신규 프로젝트를 시작하는 팀에게:

**TypeScript strict 모드는 처음부터.** 프로토타입 이상의 프로젝트라면 타협 없이 적용하세요. Ergonomics는 그 어느 때보다 좋아졌고 에러 예방 효과는 높습니다. 팀 내에서 이를 두고 논의 중이라면 이미 업계가 결론을 내린 논쟁입니다.

**대부분의 Web 프로젝트는 Next.js App Router.** Server Component 모델은 Production에서 충분히 성숙했습니다. `'use client'` 경계는 Component Tree 깊숙이 밀어 넣으세요. 변이 작업에는 Route Handler 대신 Server Action을 사용하세요(Public API를 만드는 경우 제외).

**Vercel 종속성을 의식적으로 파악.** 아키텍처에서 Vercel 전용 인프라에 의존하는 기능이 무엇인지 파악하세요. Self-hosted 배포가 가능성으로 남아 있다면 초기에 테스트해보세요.

저희가 현재 권장하는 Stack:
- **Language**: TypeScript (strict mode)
- **Framework**: Next.js 16 (App Router)
- **AI Tool**: Claude Code + GitHub Copilot
- **Testing**: Vitest + Playwright

## 정리

2025년 State of JavaScript는 JavaScript 생태계가 실험기에서 성숙기로 진입했음을 보여줍니다. TypeScript와 Meta-Framework는 이제 전문 Web 개발의 기준선입니다. TypeScript 전용 사용 40%, Next.js 채택률 59%는 단순한 통계가 아닙니다. 팀의 결정 피로를 줄여주는 정착된 합의를 반영합니다. 생태계의 안정화는 지식이 축적되고 채용 시장이 안정되며 도구가 성숙해지는 좋은 신호입니다.
