## TanStack/router에서 무슨 일이 있었나

2026년 5월, GitHub의 TanStack/router 리포지토리에 부모 커밋이 없는 고아 커밋(orphaned commit)이 불법으로 push되는 사건이 발생했습니다. 보안 업계에서 "Mini Shai-Hulud 공격"이라고 불리는 이 사건은 정상적인 컨트리뷰터로 위장한 인증 정보를 이용한 것으로 분석됩니다.

실제 피해는 보고되지 않았지만, 이 사건이 시사하는 바는 무겁습니다. TanStack Router는 React 생태계에서 가장 많이 다운로드되는 라우팅 라이브러리 중 하나입니다. 악성 코드가 릴리즈에 포함되었다면, 누군가 발견하기 전에 이미 수많은 프로덕션 애플리케이션에 전파되었을 것입니다.

## CI/CD 파이프라인이 표적이 되는 이유

공격자가 빌드 및 배포 파이프라인을 노리는 이유는 명확합니다:

1. **높은 권한으로 실행** — 배포 자격 증명, 클라우드 접근 권한, 서명 키를 보유
2. **자동 실행** — push나 PR마다 사람의 검토 없이 코드가 실행됨
3. **복잡한 의존성 체인** — 서드파티 Actions, npm 패키지, Docker 이미지 등이 공격 표면을 확장

TanStack 공격의 벡터는 파이프라인 자체보다는 리포지토리 접근 권한이었지만, 리스크 패턴은 동일합니다. 리포지토리 침해는 빌드 아티팩트 침해로 이어집니다.

## GitHub의 2026년 Actions 보안 로드맵

GitHub은 2026년 Actions 보안 로드맵을 공개했으며, 다음 기능들을 단계적으로 제공하고 있습니다:

**워크플로 의존성 락파일:** Actions 버전을 SHA로 고정하는 기존 보안 관행을 공식 락파일 기능으로 지원할 예정입니다.

**Native Layer 7 Egress Firewall:** 워크플로 실행 중 아웃바운드 네트워크 트래픽을 제어합니다. 공격자 인프라로의 데이터 유출 시도를 차단합니다.

**Scoped Secrets:** 시크릿 접근을 특정 워크플로나 잡으로 제한할 수 있습니다. 현재는 리포지토리의 모든 잡이 모든 시크릿을 읽을 수 있어 대부분의 팀이 인식하는 것보다 접근 범위가 넓습니다.

**OIDC Custom Property Claims (GA):** Actions OIDC 토큰에 리포지토리 커스텀 속성을 클레임으로 포함할 수 있게 되었습니다. 클라우드 프로바이더와 연동 시 더 세밀한 신뢰 정책을 구현할 수 있습니다.

## 지금 바로 적용할 수 있는 방어 조치

### SHA 핀닝으로 Actions 고정

```yaml
# 위험: 태그는 다른 커밋으로 변경될 수 있습니다
- uses: actions/checkout@v4

# 안전: 특정 커밋 SHA로 핀닝
- uses: actions/checkout@11bd71901bbe5b1630ceea73d27597364c9af683  # v4.2.2
```

태그는 변경 가능합니다. SHA 핀닝을 사용하면 참조하는 코드가 명시적인 업데이트 없이는 절대 변경되지 않습니다.

### 최소 권한 설정

```yaml
# 워크플로 레벨에서 제한적인 기본값 설정
permissions:
  contents: read

jobs:
  deploy:
    permissions:
      contents: read
      id-token: write  # OIDC만, 정적 자격 증명 없음
```

기존 워크플로를 감사해보세요. `contents: write`가 필요 없는데 설정된 경우가 많습니다. 권한 범위가 작을수록 자격 증명이 침해되었을 때 피해 범위도 작아집니다.

### 정적 자격 증명을 OIDC로 교체

```yaml
permissions:
  id-token: write
  contents: read

steps:
  - name: AWS 인증 (OIDC)
    uses: aws-actions/configure-aws-credentials@010d0da01d0b5a38af31e9c3470dbfdabdecca3a  # v4
    with:
      role-to-assume: arn:aws:iam::123456789012:role/github-actions-role
      aws-region: ap-northeast-1
      # 액세스 키 없음 — OIDC를 통한 단기 토큰 자동 발급
```

토큰은 잡 범위로 몇 분간만 유효합니다. 유출되더라도 공격 가능한 시간 창이 매우 좁습니다. 시크릿으로 저장된 장기 유효 AWS 액세스 키보다 근본적으로 안전한 보안 자세입니다.

### 브랜치 보호 및 커밋 서명

TanStack 사건은 알려진 컨트리뷰터처럼 보이는 불법 커밋이 관여했습니다. 서명된 커밋을 요구하면 이 공격 유형을 직접 차단할 수 있습니다:

```
GitHub Repository Settings → Branches → Branch protection rules:
✅ Require pull request reviews before merging
✅ Require review from Code Owners
✅ Restrict who can push to matching branches
✅ Require signed commits  ← 이 공격 유형 직접 차단
```

GPG 또는 SSH 커밋 서명을 강제하면, 인증된 키의 유효한 서명이 없는 커밋은 병합 전에 거부됩니다.

### 자동화된 의존성 업데이트

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: "github-actions"
    directory: "/"
    schedule:
      interval: "weekly"
    commit-message:
      prefix: "ci"
```

SHA 핀닝과 Dependabot을 함께 사용하세요. Dependabot이 핀닝된 SHA를 최신 태그 릴리즈로 유지해주어, SHA 핀의 불변성과 정기적 업데이트의 최신성을 동시에 확보할 수 있습니다.

## 우선순위별 조치

| 우선순위 | 조치 | 이유 |
|----------|------|------|
| 1 | 클라우드 인증 OIDC 전환 | 장기 정적 자격 증명 제거; 효과 크고 설정 비용 적음 |
| 2 | 서명된 커밋 요구 | 불법 커밋 삽입 직접 차단 |
| 3 | 모든 Actions SHA 핀닝 | 의존성 그래프 고정; Dependabot이 관리 부담 줄여줌 |
| 4 | 권한 제한 적용 | 자격 증명 침해 시 피해 범위 축소 |

## 이상 탐지를 위한 Actions Data Stream

GitHub 로드맵에는 Actions Data Stream도 포함되어 있습니다. 워크플로 실행 텔레메트리를 외부 시스템으로 실시간 스트리밍하는 기능으로, SIEM 통합과 비정상적인 파이프라인 동작(예상치 못한 네트워크 호출, 비정상적인 환경 변수 접근, 비정상적인 시간대의 잡 실행)에 대한 커스텀 알림을 설정할 수 있습니다.

현재 베타 단계이지만, GitHub UI를 넘어서는 CI/CD 감사 로깅이 필요한 팀에게는 주목할 만한 기능입니다.

## 마무리

잘 알려진 OSS 리포지토리를 대상으로 한 공급망 공격은 이론적 위협이 아니라 현실입니다. TanStack 사건은 확인된 피해가 없었지만, 공격을 가능하게 한 방식 — 정상적으로 보이는 불법 커밋 — 은 다른 곳에서 실제 심각한 피해를 일으킨 것과 동일한 패턴입니다.

GitHub의 2026년 보안 로드맵이 플랫폼 레벨에서 구조적인 문제를 해결하고 있습니다. 그 사이에도 여기 제시한 방어 조치들은 오늘 바로 적용 가능하며, 파이프라인을 크게 다시 작성하지 않고도 가장 일반적인 공격 벡터를 차단할 수 있습니다. 커밋 서명과 OIDC 도입, 이 두 가지가 이번 주 대부분의 팀이 적용할 수 있는 가장 효과적인 변경입니다.
