LLM을 Production에서 운영하는 팀에게 Token 소비 관리는 이제 무시할 수 없는 Infrastructure 과제입니다. 2026년 5월 The New Stack이 "Tokenmaxxing" 현상을 보도하며, AI 예산이 조용히 그리고 확실하게 불어나는 실태를 알렸습니다.

Claude Opus 4.7이나 GPT-5.5 같은 Frontier Model은 정밀도가 높은 만큼 비용도 높습니다. 요청당 Token 소비가 2배가 되면 월간 API 비용도 2배가 됩니다. 하루 수백만 건의 API 호출이 발생하는 시스템에서는 작은 차이가 큰 금액으로 이어집니다.

## Tokenmaxxing이란 무엇인가

"Tokenmaxxing"은 단일한 문제가 아니라 여러 비효율이 쌓여 Token 소비가 눈덩이처럼 증가하는 현상의 총칭입니다.

**주요 원인:**

- **Context 오염**: 관련성이 낮은 과거 대화 이력을 모두 Request에 포함
- **장황한 System Prompt**: 같은 내용을 여러 방식으로 반복 기술한 System Prompt
- **재전송 비효율**: Network 오류로 인한 Partial Response 재전송으로 Input Token이 이중 계산
- **Agent 간 중복 전달**: Multi-agent 시스템에서 각 Agent에 전체 상태를 통째로 전달

개발 단계에서는 이러한 문제가 잘 보이지 않습니다. Scale이 커질수록 비용으로 드러납니다.

## 먼저 소비량을 측정합니다

최적화 전에 현황을 파악합니다. API Client에 간단한 Wrapper를 추가하면 어느 Endpoint에서 얼마나 Token이 사용되는지 파악할 수 있습니다.

```typescript
interface UsageSample {
  model: string;
  endpoint: string;
  inputTokens: number;
  outputTokens: number;
  cacheHitTokens: number;
  timestamp: Date;
}

class TokenTracker {
  private samples: UsageSample[] = [];
  private alertThreshold: number;

  constructor(alertThreshold = 50_000) {
    this.alertThreshold = alertThreshold;
  }

  record(sample: UsageSample): void {
    this.samples.push(sample);
    const total = sample.inputTokens + sample.outputTokens;
    if (total > this.alertThreshold) {
      console.warn(
        `[Token Alert] ${sample.endpoint}: ${total.toLocaleString()} tokens`
      );
    }
  }

  summarizeByEndpoint(): Record<string, number> {
    return this.samples.reduce((acc, s) => {
      const key = `${s.model}::${s.endpoint}`;
      acc[key] = (acc[key] ?? 0) + s.inputTokens + s.outputTokens;
      return acc;
    }, {} as Record<string, number>);
  }
}
```

## 전략 1: Prompt Caching 활성화

Anthropic의 Prompt Caching은 반복 사용되는 System Prompt나 긴 Document Context의 비용을 일반 Input Token의 약 10분의 1로 줄여줍니다. System Prompt가 1,000 Token을 넘는다면 Caching을 활성화하지 않을 이유가 거의 없습니다.

```typescript
import Anthropic from "@anthropic-ai/sdk";

const client = new Anthropic();

const SYSTEM_PROMPT = `당신은 webhani Inc.의 Technical Consultant입니다.
고객의 기술적 질문에 실용적이고 구체적인 조언을 제공합니다.
다음 원칙을 따라 답변하세요:
...（수백 Token에 걸친 상세 지시）`;

async function queryWithCache(userMessage: string) {
  const response = await client.messages.create({
    model: "claude-opus-4-7",
    max_tokens: 1024,
    system: [
      {
        type: "text",
        text: SYSTEM_PROMPT,
        cache_control: { type: "ephemeral" }, // Cache 활성화
      },
    ],
    messages: [{ role: "user", content: userMessage }],
  });

  const cached = response.usage.cache_read_input_tokens ?? 0;
  const written = response.usage.cache_creation_input_tokens ?? 0;
  console.log(`Cache: ${cached} read, ${written} written`);
  return response;
}
```

첫 번째 요청에서 Cache가 생성되고, 이후 요청부터는 `cache_read_input_tokens`로 계산되어 저렴하게 처리됩니다.

## 전략 2: 선택적 Context Trimming

대화 이력을 그대로 축적하지 말고 중요도에 따라 메시지를 필터링합니다.

```typescript
interface ScoredMessage {
  role: "user" | "assistant";
  content: string;
  relevanceScore: number; // 0.0~1.0
  tokenCount: number;
}

function buildOptimizedContext(
  history: ScoredMessage[],
  maxContextTokens: number,
  keepRecentCount = 4
): ScoredMessage[] {
  // 최근 대화는 항상 포함
  const recent = history.slice(-keepRecentCount);
  const recentTokens = recent.reduce((sum, m) => sum + m.tokenCount, 0);

  // 남은 Budget에서 관련성 높은 순으로 추가
  const older = history.slice(0, -keepRecentCount);
  const sorted = [...older].sort((a, b) => b.relevanceScore - a.relevanceScore);

  const selected: ScoredMessage[] = [];
  let remaining = maxContextTokens - recentTokens;

  for (const msg of sorted) {
    if (remaining < msg.tokenCount) continue;
    selected.push(msg);
    remaining -= msg.tokenCount;
  }

  // 시간순으로 재정렬
  return [
    ...selected.sort((a, b) => older.indexOf(a) - older.indexOf(b)),
    ...recent,
  ];
}
```

## 전략 3: Structured Output으로 Response 압축

다운스트림 소비자가 코드라면 자연어 대신 구조화된 출력을 요청합니다. 동일한 정보량이라도 JSON Response는 자연어 설명보다 30~50% Token이 적은 경향이 있습니다.

```typescript
// ❌ 자연어로 상세 설명을 요청 (Token 소비 과다)
const verbosePrompt = `
  이 코드를 자세히 분석해서 버그가 있다면 줄 번호와 원인,
  수정 방법을 친절하게 설명해 주세요.
`;

// ✅ 구조화 출력 요청 (Token 효율 좋음)
const structuredPrompt = `
아래 코드의 버그를 분석하고 이 JSON 형식으로만 반환하세요.
설명 없이 JSON만 반환합니다.

{"issues":[{"line":number,"severity":"error"|"warning","summary":"<10단어 이내>","fix":"<수정 코드>"}]}

코드:
\`\`\`
${codeBlock}
\`\`\`
`;
```

## Multi-agent 시스템에서의 Budget 배분

여러 Agent를 조합하는 시스템에서는 역할별로 Token Budget을 명시적으로 할당합니다. 무제한으로 Token을 소비할 수 있는 Agent는 Tokenmaxxing을 일으키기 쉽습니다.

```typescript
const AGENT_BUDGETS = {
  planner: 8_000,    // 계획 수립: 간결함이 요구됨
  executor: 20_000,  // 실행: 가장 복잡한 작업 담당
  reviewer: 5_000,   // 리뷰: Diff 확인에 집중
  summarizer: 3_000, // 요약: 짧은 Output이 전제
} as const;

type AgentRole = keyof typeof AGENT_BUDGETS;

async function runAgentWithBudget(role: AgentRole, prompt: string) {
  return client.messages.create({
    model: "claude-opus-4-7",
    max_tokens: AGENT_BUDGETS[role],
    messages: [{ role: "user", content: prompt }],
  });
}
```

`max_tokens`를 역할별로 설정하면 각 Agent가 자신의 범위 내에서 작동하도록 강제됩니다.

## 실행 우선순위

Production에서 비용 절감을 도입할 때의 권장 순서입니다.

1. **계측 기반 구축** — 변경 전에 현황 파악
2. **Prompt Caching 활성화** — 긴 System Prompt에 즉각적인 효과
3. **Context 구성 검토** — 대부분의 팀은 필요 이상의 이력을 포함
4. **Output 형식 최적화** — Structured Output으로 Response량 압축
5. **Budget Alert 설정** — 청구서에 반영되기 전에 이상 감지

LLM 비용은 Production Scale이 커질수록 영향이 커집니다. 요청당 소소한 개선이 수백만 건의 일일 API 호출에서 크게 누적됩니다. Token Budget 관리는 일회성 최적화가 아니라 LLM 운영의 지속적인 규율입니다.
