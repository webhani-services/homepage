## TypeScript 7.0 RC 출시 — 컴파일러의 전면 재구현

2026년 6월 18일, TypeScript 7.0이 Release Candidate 단계에 진입했습니다. Microsoft 발표에 따르면 안정판 GA(General Availability)는 RC 이후 약 1개월을 목표로 하고 있습니다. 이 마일스톤의 중요한 점은 새로운 언어 기능이 아니라 근본적인 아키텍처 변화입니다. 컴파일러가 Go로 재구현되었기 때문입니다.

5개 메이저 버전 동안 TypeScript는 JavaScript로 작성된 컴파일러를 사용했습니다. 7.0부터는 Go로 빌드된 네이티브 바이너리를 도입합니다. 성능 개선은 눈에 띕니다. **타입 체크 속도가 약 10배 향상**됩니다(Microsoft 벤치마크 기준). 구체적인 예로, VS Code 코드베이스(약 150만 줄의 TypeScript)의 타입 체크 시간이 78초에서 7.5초로 단축됩니다.

이는 빌드 시간에서 수 밀리초를 깎는 것이 아닙니다. 대규모 팀 환경에서는 타입 체크 속도가 피드백 루프, 에디터 응답성, 그리고 CI 파이프라인의 효율성을 크게 변화시킵니다.

## 재구현 과정 — "Strada" 프로젝트

내부적으로 이 작업은 "Strada"라는 프로젝트명으로 진행되었습니다. Microsoft 컴파일러 팀은 기존 JavaScript 구현을 파일 단위로 Go로 이식하면서 동일한 타입 체크 로직과 출력을 유지했습니다. 목표는 처음부터의 재작성이 아니라 동일한 동작을 보장하는 것입니다.

Go 기반 구현이 기존 TypeScript 컴파일러의 약 85%에 도달했을 때 RC가 릴리스되었습니다. 중요한 점은 **모든 진단(에러, 경고, 제안)이 기존 JavaScript 컴파일러와 완전히 동일**해야 한다는 것입니다. 새로운 컴파일러는 RC 릴리스 전에 Microsoft 자체 코드베이스에서 진단 동등성을 검증했습니다.

속도 개선은 두 가지 요인에서 비롯됩니다. 네이티브 바이너리 실행(JavaScript 런타임 오버헤드 제거)과 공유 메모리를 활용한 멀티스레드 병렬 처리입니다.

## RC 시도하기 — 설치 및 Shadow 테스트

TypeScript 7.0 RC를 로컬에서 테스트하려면 npm에서 설치합니다.

```bash
npm install -D typescript@rc
```

설치되는 `tsc` 바이너리는 Go 기반 네이티브 컴파일러입니다. 별도의 명령어가 필요하지 않습니다. `node_modules/.bin/tsc`에 위치하는 바이너리가 이미 새로운 구현입니다.

프로덕션에 사용하려면 **Shadow CI 전략**을 권장합니다. 즉시 전환하기보다는 안정판과 RC를 병렬로 실행하여 진단 결과를 비교한 후 동일함을 확인했을 때 전환하는 방식입니다.

### Shadow CI 예제

`package.json`에 두 버전을 모두 실행하는 스크립트를 추가합니다.

```json
{
  "scripts": {
    "type-check": "tsc --noEmit",
    "type-check:rc": "npm install -D typescript@rc && tsc --noEmit"
  }
}
```

CI에서는 두 명령을 별도의 작업으로 실행하고 출력을 수집합니다.

```yaml
# GitHub Actions 예제
jobs:
  type-check-stable:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm install
      - run: npm run type-check > stable.txt 2>&1
      - uses: actions/upload-artifact@v4
        with:
          name: stable-diagnostics
          path: stable.txt

  type-check-rc:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
      - run: npm install
      - run: npm run type-check:rc > rc.txt 2>&1
      - uses: actions/upload-artifact@v4
        with:
          name: rc-diagnostics
          path: rc.txt

  compare-diagnostics:
    needs: [type-check-stable, type-check-rc]
    runs-on: ubuntu-latest
    steps:
      - uses: actions/download-artifact@v4
      - run: diff stable-diagnostics/stable.txt rc-diagnostics/rc.txt || true
```

최소 1주일 동안 주 CI와 함께 이 작업을 실행합니다. 진단 결과가 동일하면 마이그레이션할 준비가 완료된 것입니다.

### Nightly 빌드 — native-preview와 tsgo

최신 개발 버전을 추적하고 싶은 팀을 위해 nightly 패키지인 `@typescript/native-preview`가 `tsgo` 명령을 제공합니다.

```bash
npm install -D @typescript/native-preview
npx tsgo --version
```

최신 구현을 시도할 수 있습니다. 하지만 RC보다 안정성이 낮으므로 로컬 개발이나 실험적 프로젝트에만 사용하고, 프로덕션 CI에는 사용하지 않기를 권장합니다.

## 10배 성능 개선의 실무적 의미

타입 체크 속도 10배는 벤치마크 수치처럼 들릴 수 있습니다. 실제로는 대규모 팀의 개발 방식을 근본적으로 변화시킵니다.

**에디터 응답성이 극적으로 향상됩니다.** 매 입력마다 백그라운드에서 타입 체크가 실행됩니다. 전체 VS Code 코드베이스에 대한 7.5초의 타입 체크는 자동완성 지연을 눈에 띄는 수준에서 거의 즉시 응답으로 줄입니다.

**CI 피드백 루프가 압축됩니다.** 모노레포 환경에서 타입 체크는 흔히 critical path입니다. 5분에서 30초로 단축하면 개발자의 커밋 빈도와 사전 푸시 체크 방식이 달라집니다.

**프리커밋 훅이 실용적이 됩니다.** `tsc --noEmit`이 초 단위로 완료되면, 개발자 경험을 해치지 않으면서도 더 엄격한 타입 체크 규칙을 도입할 수 있습니다.

**더 엄격한 체크를 자주 실행할 수 있습니다.** 시간이 제약이 아니면 추가 진단을 활성화하고, 더 엄격한 `tsconfig` 설정을 적용하거나, 더 공격적인 타입 좁히기 검사를 할 수 있습니다. 속도가 엄격함을 감당할 수 있게 만듭니다.

## 플랫폼별 바이너리 — 알아야 할 사항

네이티브 바이너리는 본질적으로 플랫폼별로 다릅니다. `typescript@rc` npm 패키지는 다음 플랫폼을 위해 미리 컴파일된 바이너리를 포함합니다.

- Windows (x86_64)
- macOS (Intel 및 Apple Silicon)
- Linux (x86_64 및 ARM)

대부분의 CI 환경에서 이 중 하나와 일치합니다. 하지만 ARM 기반 엣지 러너나 특수한 CI 플랫폼을 사용한다면, RC로 변경하기 전에 해당 플랫폼용 바이너리가 존재하는지 반드시 확인해야 합니다.

### 확인 방법

CI 환경에서 `npm install typescript@rc`를 실행하고 `tsc --version`을 시도합니다. 플랫폼용 바이너리가 없으면 npm이 설치 중에 명확한 에러를 표시합니다. 메인 파이프라인을 변경하기 전에 shadow 작업에서 이를 테스트하세요.
