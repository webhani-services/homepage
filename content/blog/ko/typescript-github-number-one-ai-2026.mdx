## 무슨 일이 있었나

GitHub Octoverse 2025 리포트에 따르면, TypeScript가 2025년 8월에 Python과 JavaScript 모두를 제치고 GitHub에서 가장 많이 사용되는 언어가 되었습니다. 2025년 한 해 동안 100만 명 이상의 신규 개발자가 TypeScript를 채택했으며, 전년 대비 66% 성장이라는 기록적인 수치를 기록했습니다. 활성 기여자 수는 총 263만 명에 달합니다.

GitHub은 이 변화를 "10년 이상 중 가장 중요한 언어 전환"이라고 표현했습니다. 그런데 더 흥미로운 것은 숫자 자체가 아니라 "왜 이런 일이 일어났는가"입니다.

---

## Convenience Loop란 무엇인가

TypeScript의 급성장에는 AI 코딩 도구와의 자기 강화적 피드백 루프가 깊이 관여하고 있습니다. 이를 「Convenience Loop」라고 합니다.

구조는 단순합니다.

1. AI 도구가 TypeScript 코드를 잘 작성함
2. TypeScript 프로젝트에서 AI 지원이 더 매끄럽게 느껴짐
3. 개발자들이 AI 지원 프로젝트에 TypeScript를 선택
4. GitHub에 더 많은 TypeScript 코드가 쌓임
5. LLM 학습 데이터에 TypeScript가 늘어남
6. AI가 TypeScript를 더 잘 다루게 됨
7. → 1번으로 돌아감

AI가 어떤 언어를 "마찰 없이" 느끼게 만들면 개발자들이 그 언어로 몰립니다. 이것이 더 많은 학습 데이터를 만들고, AI는 해당 언어를 더 잘 다루게 되며, 마찰은 더욱 줄어듭니다. 이 자기 강화 사이클이 TypeScript를 가속시키고 있습니다.

---

## TypeScript와 AI의 기술적 궁합

AI 도구가 TypeScript를 선호하는 데는 명확한 기술적 이유가 있습니다. **타입 오류가 AI의 실수를 자동으로 잡아냅니다.**

GitHub 조사에 따르면, LLM이 생성한 코드의 컴파일 오류 중 94%는 타입 체크 실패입니다. Claude나 Copilot이 인터페이스를 잘못 사용하거나, 잘못된 타입을 함수에 전달하거나, 존재하지 않는 프로퍼티에 접근하는 코드를 생성할 때, TypeScript는 컴파일 시점에 즉시 이를 포착합니다. JavaScript에서는 같은 오류가 런타임에 프로덕션에서야 나타날 수 있습니다.

```typescript
// TypeScript가 AI 생성 실수를 컴파일 시점에 잡아내는 예시
interface User {
  id: number;
  name: string;
  email: string;
}

function sendWelcome(user: User) {
  // TS2339: Property 'mail' does not exist on type 'User'
  // TypeScript가 즉시 오류 표시 — JS는 런타임에야 알 수 있음
  mailer.send(user.mail, "환영합니다!");
}
```

타입 오류 감지 외에도, 타입 정의는 AI 도구에게 코드 의도에 대한 풍부한 컨텍스트를 제공합니다. `interface Order { items: OrderItem[]; total: number; currency: "KRW" | "USD"; }` 같은 정의가 있으면 AI는 도메인 모델을 정확히 이해하고 관련 코드를 더 정확하게 생성합니다.

## Framework 기본값 효과

TypeScript 채택을 강화하는 구조적 요인이 하나 더 있습니다. 거의 모든 주요 프론트엔드 프레임워크가 이제 TypeScript를 기본으로 스캐폴드합니다.

Next.js, SvelteKit, Qwik, Astro, Remix — 프로젝트 생성 도구를 실행하면 TypeScript가 기본 선택입니다. 개발자들은 이제 TypeScript를 선택하는 것이 아니라, TypeScript에서 벗어나기를 선택해야 하는 상황이 되었습니다.

```bash
# 모든 주요 프레임워크: TypeScript가 이제 기본 스캐폴드
npx create-next-app@latest my-app
# ✓ TypeScript를 사용하시겠습니까? … Yes (기본값)

npx sv create my-app
# ✓ TypeScript로 타입 체크를 추가하시겠습니까? … Yes (기본값)
```

이것은 채택 곡선을 근본적으로 바꿉니다. TypeScript의 장벽은 더 이상 "새로운 도구를 배우기"가 아닙니다. 의도적으로 TypeScript에서 벗어나는 마찰이 되었습니다.

## AI + TypeScript 실전 패턴

AI 도구와 함께 TypeScript 프로젝트를 개발할 때 AI의 정확도를 크게 높이는 몇 가지 패턴이 있습니다.

**구현 전에 타입을 먼저 작성하기**  
AI에게 구현을 요청하기 전에 Interface와 타입 시그니처를 먼저 정의하세요. 타입이 AI 출력을 제약하고 안내하는 사양서 역할을 합니다.

```typescript
// 계약을 먼저 정의하고 AI에게 구현 요청
interface PaymentProcessor {
  charge(
    amount: number,
    currency: "KRW" | "USD" | "JPY",
    customerId: string
  ): Promise<{ success: boolean; transactionId: string }>;
  
  refund(
    transactionId: string,
    amount?: number
  ): Promise<{ success: boolean; refundId: string }>;
}

// 이 Interface가 있으면 AI 생성 코드는 계약을 만족하거나
// 타입 체크에서 실패함 — 조용히 오작동하지 않음
```

**Strict 모드 활성화**  
`tsconfig.json`에서 `strict: true`를 설정하면 TypeScript의 전체 검사 배터리가 활성화됩니다. 엄격 모드 하에서 AI 생성 코드는 타입 버그가 줄어듭니다.

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true
  }
}
```

**런타임 경계에 Zod 사용하기**  
TypeScript 타입은 런타임에서 사라집니다. API 응답과 외부 데이터에는 TypeScript 타입과 Zod 스키마를 함께 사용하여 경계에서 검증하세요.

```typescript
import { z } from "zod";

const OrderSchema = z.object({
  id: z.string(),
  total: z.number().positive(),
  currency: z.enum(["KRW", "USD", "JPY"]),
  items: z.array(z.object({
    sku: z.string(),
    quantity: z.number().int().positive(),
  })),
});

type Order = z.infer<typeof OrderSchema>; // 스키마에서 TypeScript 타입 생성
```

## TypeScript vs. JavaScript: 역할 분담

TypeScript의 부상은 JavaScript의 종말을 의미하지 않습니다. 두 언어는 공존하며 역할이 나뉘고 있습니다.

| 사용 사례 | TypeScript | JavaScript |
|---------|------------|------------|
| 다인원 장기 프로젝트 | ◎ | △ |
| AI 지원 개발 | ◎ | ○ |
| 스크립트·자동화 | ○ | ◎ |
| 빠른 프로토타이핑 | △ | ◎ |
| 빌드 단계 없는 브라우저 실행 | △ | ◎ |

TypeScript는 타입이 사양서가 되는 상황, 즉 여러 사람이 장기간 유지보수하는 프로젝트에서 강합니다. AI 도구를 활용할 때 특히 TypeScript의 우위가 두드러집니다.
