2025년 8월, TypeScript가 Python과 JavaScript를 제치고 GitHub에서 가장 많이 사용되는 프로그래밍 언어가 되었습니다. 단순한 언어 경쟁의 결과가 아닙니다. 이 변화의 핵심에는 AI Coding Assistant의 확산이 있습니다.

## 데이터로 보는 변화

GitHub Octoverse 2025 보고서의 주요 수치입니다.

- TypeScript가 **GitHub 리포지토리 사용 1위** 언어로 등극 (Python, JavaScript 추월, 2025년 8월)
- 신규 GitHub 개발자의 **80%가 첫 주에 Copilot 사용**
- LLM SDK를 사용하는 Public Repository **110만 개 이상** (전년 대비 178% 증가)
- 개발자의 **84%가 AI 도구를 사용 중이거나 사용 예정**, 51%는 매일 사용

GitHub 팀은 이를 "Convenience Loop"라고 명명했습니다. AI 도구가 TypeScript에서 더 잘 작동하고, 이것이 개발자를 TypeScript로 유인하며, TypeScript 환경이 AI 도구를 더 효과적으로 만드는 선순환 구조입니다. 언어 선택이 도구 선택에서 비롯되는 것입니다.

## TypeScript가 AI와 잘 맞는 이유

이유는 기계적으로 명확합니다. LLM이 코드를 생성할 때는 컨텍스트를 바탕으로 타입, 형태, 계약에 대해 추론합니다. 타입이 명시적일수록 모델이 추측해야 하는 범위가 줄어듭니다.

2025년 학술 연구에 따르면, **LLM이 생성한 코드의 컴파일 오류 중 94%가 타입 체크 실패**였습니다. 타입 어노테이션이 AI 지원 코딩에서 가장 흔한 실패 유형을 직접 해결하는 셈입니다.

```typescript
// TypeScript: AI가 정확히 무엇을 전달하고 무엇을 기대해야 하는지 파악
async function processOrder(
  customerId: string,
  items: Array<{ productId: string; quantity: number }>,
  currency: 'KRW' | 'USD' | 'JPY'
): Promise<{ orderId: string; totalAmount: number }> {
  // AI가 생성하는 코드가 타입 제약 안에서 작동
}

// JavaScript: AI가 currency에 'krw', 0, undefined를 전달할 가능성이 있음
function processOrder(customerId, items, currency) {
  // 런타임에서 발견되는 오류들
}
```

TypeScript가 이론적으로 우월하다는 주장이 아닙니다. AI 도구가 더 많은 신호를 받을 때 더 나은 결과물을 생성한다는 실용적인 관찰입니다.

## Convenience Loop의 실제 작동 방식

GitHub의 분석은 Copilot을 일찍 도입한 개발자들에게서 일관된 행동 패턴을 보여줍니다.

1. 새 개발자가 첫날부터 Copilot을 사용
2. TypeScript 프로젝트에서 Copilot이 더 많은 컨텍스트를 받아 자동완성이 더 정확
3. 수정이 덜 필요한 경험 → TypeScript가 더 쉽게 느껴짐
4. 다음 프로젝트에서도 TypeScript 선택

이 Loop는 "쉽다"의 기준을 재설정합니다. AI 도움을 받으며 코딩을 배운 개발자는 JavaScript의 유연함을 자유로 경험하지 않고, TypeScript의 엄격함을 낮은 마찰감으로 경험합니다.

## JavaScript 프로젝트의 단계적 마이그레이션

한 번에 모든 것을 바꿀 필요는 없습니다.

### 1단계: 기존 JavaScript와 병행하여 TypeScript 활성화 (1~2일)

```bash
npm install -D typescript @types/node
npx tsc --init
```

시작을 위한 최소 `tsconfig.json`:

```json
{
  "compilerOptions": {
    "target": "ES2022",
    "module": "ESNext",
    "moduleResolution": "bundler",
    "strict": false,
    "allowJs": true,
    "checkJs": false,
    "outDir": "./dist",
    "baseUrl": ".",
    "paths": {
      "@/*": ["./src/*"]
    }
  },
  "include": ["src/**/*"]
}
```

`allowJs: true`를 설정하면 기존 `.js` 파일을 수정 없이 컴파일할 수 있습니다. 새 파일은 처음부터 `.ts`로 작성합니다.

### 2단계: 모든 신규 코드를 TypeScript로 작성 (지속)

기존 파일 마이그레이션은 우선순위가 낮은 기술 부채로 처리합니다. 새 기능과 모듈은 즉시 `.ts`로 시작합니다. 대규모 리팩토링 없이도 코드베이스가 자연스럽게 전환됩니다.

### 3단계: Strict Mode 활성화 (3~6개월 후)

```bash
# strict 활성화 전 오류 수 파악 — 예상치를 설정
npx tsc --noEmit --strict 2>&1 | grep "error TS" | wc -l

# 오류가 많은 파일 파악 — 여기서 시작
npx tsc --noEmit --strict 2>&1 | grep "error TS" \
  | awk -F'(' '{print $1}' \
  | sort | uniq -c | sort -rn \
  | head -20
```

`tsconfig.json`에서 `strict: true`를 활성화하고 트래픽이 많은 모듈부터 파일 단위로 오류를 수정합니다.

## AI 도구와 함께 달라지는 것들

TypeScript 코드베이스에서 AI 도구를 활용하면 구체적으로 체감되는 차이가 있습니다.

**자동완성 정확도**: AI가 올바른 인수 타입, 반환 타입, Nullable 처리를 정확하게 제안합니다. 빠르게 읽으면 맞아 보이지만 런타임에 실패하는 코드의 빈도가 줄어듭니다.

**리팩토링 안전성**: 타입 체커가 AI 제안 변경 사항을 자동으로 검증합니다. "테스트를 실행해야 오류를 알 수 있는" 상태에서 "실행 전에 오류가 보이는" 상태로 피드백 루프가 빨라집니다.

**온보딩 속도**: 새 개발자와 AI Assistant 모두 모든 경계에서 타입 계약이 명시적일 때 코드베이스를 더 빠르게 파악합니다. 타입은 코드 주석이 거의 그렇지 못한 방식으로 항상 최신 상태를 유지하는 문서 역할을 합니다.

## TypeScript가 적합하지 않은 경우

균형 잡힌 시각을 위해 덧붙입니다.

- **일회성 스크립트**: 한 번 실행하고 삭제되는 스크립트라면 타입 설정 비용이 불필요합니다.
- **매우 작은 프로젝트**: 200줄 유틸리티에는 전체 TypeScript 툴체인이 과도합니다.
- **활발히 개발 중이지 않은 안정적인 JavaScript 코드베이스**: 마이그레이션을 위한 마이그레이션은 피하세요.

TypeScript가 가장 효과적인 곳은 성장할 프로젝트, 여러 개발자가 유지보수하는 코드, AI 지원 개발의 혜택을 받을 코드입니다. 대부분의 Production Web Application이 여기에 해당합니다.

## 마치며

GitHub에서 TypeScript가 1위를 차지한 것은 단순한 언어 트렌드가 아니라 코드 작성 방식의 구조적 변화를 반영합니다. AI 도구 효과성과 정적 타이핑의 연관성은 이 흐름이 더욱 가속될 가능성이 높다는 것을 시사합니다.

팀이 아직 Production Application에 JavaScript를 사용하고 있다면, 지금이 마이그레이션을 시작하기에 가장 좋은 시기입니다. `allowJs: true`로 새 모듈만 TypeScript로 작성하는 것부터 시작하세요. 코드베이스가 자연스럽게 전환되면서 AI 도구와의 시너지를 즉시 체감할 수 있습니다.
