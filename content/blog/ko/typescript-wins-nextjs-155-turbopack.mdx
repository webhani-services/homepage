## TypeScript는 이미 기본값이 되었다

JavaScript 생태계 조사에서 주목받는 발언이 있었습니다. Nuxt 코어 팀 리더의 말입니다: "TypeScript가 이겼다. 번들러로서가 아니라 언어로서."

조사 데이터도 이를 뒷받침합니다. 응답자의 40%가 이미 TypeScript만으로 코딩하고 있으며, 이 비율은 계속 증가 중입니다. 더 실질적인 변화는 주요 라이브러리들이 TypeScript를 기본으로 설계되고, AI 코딩 도구들이 TypeScript를 기본 출력으로 사용한다는 점입니다. 2026년에 순수 JavaScript로 새 프로젝트를 시작한다는 것은 표준 툴체인에서 의도적으로 벗어나는 것을 의미합니다.

이제 트렌드가 아니라 기준선입니다. 중요한 질문은 TypeScript를 사용할지 여부가 아니라, 어떻게 올바르게 설정하는가로 이동했습니다.

## Next.js 15.5: 중요한 변경사항

### Turbopack 프로덕션 빌드 (베타)

Turbopack은 개발 서버에서 한동안 사용 가능했습니다. Next.js 15.5에서는 프로덕션 빌드도 베타로 제공됩니다:

```bash
next build --turbopack
```

대규모 코드베이스에서 Webpack 대비 700배 빠르다고 알려져 있습니다. 실제 프로젝트에서 이 수치가 그대로 적용될지는 규모와 설정에 따라 다르지만, 방향성은 맞습니다. 개발 서버 시작 시간과 HMR 속도 개선은 이미 많은 팀이 체감하고 있습니다.

프로덕션 빌드 전환 전 확인사항:

- Turbopack 출력에 대해 전체 테스트 스위트를 실행하세요 — 청크 분할 방식이 Webpack과 다를 수 있습니다
- 번들 크기를 비교하세요. Turbopack은 다른 출력 형태를 생성할 수 있습니다
- 사용 중인 커스텀 Webpack 플러그인의 Turbopack 지원 여부를 확인하세요

```javascript
// next.config.ts — --turbopack 플래그 사용 시 추가 설정 불필요
// 커스텀 alias 해석이 필요한 경우:
const nextConfig = {
  experimental: {
    turbo: {
      resolveAlias: {
        underscore: "lodash",
      },
    },
  },
};

export default nextConfig;
```

Turbopack 프로덕션 빌드는 아직 베타 단계입니다. 스테이징과 비중요 워크로드에서 먼저 사용하고 프로덕션 적용은 신중하게 판단하세요. 개발 서버는 프로덕션 준비가 완료되어 있지만, 빌더는 아직입니다.

### Node.js Middleware 안정화

Next.js의 Middleware는 기존에 Edge Runtime에서만 실행되어 제약이 있었습니다. 네이티브 Node.js API를 사용할 수 없고 패키지 호환성도 제한적이었습니다. 기본적인 요청 조작을 넘어서는 작업에 있어 실질적인 제약이었습니다.

Next.js 15.5에서 Middleware에 Node.js Runtime이 안정화되었습니다:

```typescript
// middleware.ts
import { NextRequest, NextResponse } from "next/server";
import { verifyToken } from "@/lib/auth"; // 전체 Node.js crypto 사용 가능

export async function middleware(request: NextRequest) {
  const token = request.cookies.get("session")?.value;

  if (!token) {
    return NextResponse.redirect(new URL("/login", request.url));
  }

  // Edge 호환 crypto 우회 방법 불필요
  const payload = await verifyToken(token);

  const response = NextResponse.next();
  response.headers.set("x-user-id", payload.userId);
  return response;
}

export const config = {
  matcher: ["/((?!_next|api/public).*)"],
};
```

실질적인 영향이 가장 큰 변경사항입니다. Edge Runtime 제약으로 인해 별도 Route Handler를 만들거나 경량 crypto 라이브러리를 따로 도입했던 인증 Middleware를 이제 직접 작성할 수 있습니다. 런타임 제약에 대한 정신적 부담이 사라집니다.

### `next lint` 비권장화

`next lint` 커맨드가 15.5에서 비권장(deprecated)되었습니다. ESLint 또는 Biome을 직접 사용하도록 권장합니다.

```json
{
  "scripts": {
    "lint": "eslint . --ext .ts,.tsx,.js",
    "lint:fix": "eslint . --ext .ts,.tsx,.js --fix"
  }
}
```

CI 파이프라인에서 `next lint`를 호출하고 있다면 업데이트가 필요합니다. 이미 ESLint를 직접 사용하고 있었다면 변경사항이 없습니다.

## ESLint/Prettier 대안으로서의 Biome

Biome(구 Rome)이 Rust로 작성된 통합 포매터 및 린터로 실질적인 주목을 받고 있습니다. 하나의 도구로 포매터와 린터를 통합하며 빠른 실행 속도가 특징입니다.

```bash
npm install --save-dev @biomejs/biome
npx @biomejs/biome init
```

```json
// biome.json
{
  "$schema": "https://biomejs.dev/schemas/1.x.x/schema.json",
  "formatter": {
    "indentStyle": "space",
    "indentWidth": 2,
    "lineWidth": 100
  },
  "linter": {
    "rules": {
      "recommended": true,
      "correctness": {
        "noUnusedVariables": "error"
      }
    }
  },
  "javascript": {
    "formatter": {
      "quoteStyle": "double",
      "semicolons": "always"
    }
  }
}
```

트레이드오프: Biome의 규칙 세트는 아직 ESLint 플러그인 생태계만큼 풍부하지 않습니다. `eslint-plugin-react-hooks`, `@typescript-eslint/strict` 등 특정 플러그인에 의존하고 있다면 마이그레이션 비용이 클 수 있습니다. 신규 프로젝트나 마이그레이션을 감수할 의향이 있는 팀에게 적합합니다.

## 실질적으로 중요한 TypeScript 설정

TypeScript가 기본값이 된 지금, 설정의 품질이 중요합니다. strict 모드를 활성화하지 않은 `tsconfig.json`은 거짓된 안전감을 줍니다.

```json
{
  "compilerOptions": {
    "strict": true,
    "noUncheckedIndexedAccess": true,
    "exactOptionalPropertyTypes": true,
    "noImplicitReturns": true,
    "noFallthroughCasesInSwitch": true,
    "forceConsistentCasingInFileNames": true
  }
}
```

`noUncheckedIndexedAccess`는 대부분의 팀이 놓치는 옵션입니다. 배열과 객체 인덱스 접근 시 `T` 대신 `T | undefined`를 반환하도록 강제합니다. 실제 버그를 타입 레벨에서 잡아내며, 추가되는 `?.`와 null 체크는 그만한 가치가 있습니다.

`strict: true`는 프로젝트 시작 시 활성화하세요. 나중에 활성화하면 코드베이스 전체에서 수백 개의 타입 에러를 해결해야 합니다. 빠를수록 좋습니다.

## 2026년 권장 스택

| 카테고리 | 선택 | 비고 |
|----------|------|------|
| Framework | Next.js 15.x | App Router + RSC 안정화 |
| 언어 | TypeScript (strict) | 처음부터 strict 모드로 |
| Bundler | Turbopack(신규) / Webpack(기존) | Turbopack 프로덕션 빌드는 베타 |
| Linter/Formatter | ESLint + Prettier 또는 Biome | 신규 프로젝트는 Biome 고려 |

## 마치며

TypeScript는 이미 기본값입니다. 가장 실용적인 변경사항은 Next.js 15.5의 Node.js Middleware 안정화입니다. Edge Runtime 제약 없이 인증 Middleware를 작성할 수 있게 된 것은 실질적인 개발 경험 개선입니다. Turbopack 프로덕션 빌드는 주목할 가치가 있지만 아직 프로덕션 전환 단계는 아닙니다.

바로 할 수 있는 것: `tsconfig.json`을 열고 `strict: true`와 `noUncheckedIndexedAccess: true`가 설정되어 있는지 확인하세요. 설정되어 있지 않다면 지금 추가하고 에러를 해결하세요. 기다릴수록 비용이 커집니다.
