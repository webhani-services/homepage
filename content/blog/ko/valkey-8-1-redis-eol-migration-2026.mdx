## 기한이 판단을 재촉하고 있다

Redis OSS 7.2 계열 지원이 2026년 2월 말로 종료되었습니다. EOL을 지난 Version을 그대로 Production에서 계속 쓴다는 것은 Security 수정이 더 이상 도착하지 않는다는 뜻입니다. 많은 Team이 이 시점에 "다음은 어디로 옮길까"를 결정해야 하는 상황에 놓였습니다.

선택지는 크게 둘입니다. License 변경 이후의 Redis를 따라갈지, BSD License Fork인 Valkey로 옮길지. Valkey 8.1은 Amazon ElastiCache에서 일반 제공(GA)되어 Cloud Cache 선택지로서 일급 시민의 지위를 확립했습니다. 보고된 가격 차이는 Redis OSS 대비 약 33% 저렴하다는 것으로, Cache 계층 Cost가 쌓이는 환경에서는 무시할 수 없는 숫자입니다.

webhani도 여러 Client의 Cache 기반에서 이 판단에 관여했습니다. 결론부터 말하면 많은 Workload에서 Valkey Migration은 기술적으로 매끄럽습니다. 다만 "매끄럽다"는 것과 "검증 없이 옮겨도 된다"는 것은 다릅니다.

## Valkey란 무엇인가

Valkey는 Redis OSS에서 Fork된 Project로 Command 호환성을 유지합니다. 기존 Redis Client Library, 기존 `redis-cli`에 해당하는 Tool, 기존 Data 구조(String, Hash, List, Set, Sorted Set, Stream 등)를 그대로 쓸 수 있습니다. Application 쪽 Code를 고치지 않고 접속 대상만 바꿀 수 있다는 점이 Migration이 쉬운 근거입니다.

Security 면에서도 Valkey는 9.0 계열 Release Candidate에서 CVE를 수정하며 유지보수가 이어지고 있습니다. EOL을 맞은 7.2를 계속 쓰는 것보다 Patch가 도착하는 계열로 옮기는 편이 운영상 건전합니다.

## Migration 전에 측정해 둘 것

Command 호환이 있다고 해도 Production 투입 전에 확인할 점이 있습니다.

**영속화와 Replication 동작** — RDB Snapshot이나 AOF 설정, Replica로의 전파 Timing은 Version 간에 미묘하게 다를 수 있습니다. Failover 시 동작은 반드시 Staging에서 확인하세요.

**Client Library Version** — 오래된 Library가 특정 Version 문자열을 전제로 Handshake하면 접속 시 예상치 못한 분기로 들어갈 수 있습니다. Library를 최신으로 올린 뒤 Migration하는 것이 안전합니다.

**Module 의존** — RediSearch나 RedisJSON 같은 추가 Module에 의존한다면 그대로 옮길 수 있다고 단정할 수 없습니다. 이 부분은 사전 조사가 필수입니다. 순수 Cache 용도(Key-Value와 TTL)라면 문제가 잘 생기지 않지만, Module 의존이 있으면 이야기가 달라집니다.

## 안전한 전환 절차

Production 접속 대상을 곧바로 갈아치우지 말고 단계적으로 진행합니다.

```bash
# 1. 현행 Redis의 Data 규모와 Connection 수 파악
redis-cli INFO keyspace
redis-cli INFO clients

# 2. Valkey Instance를 별도로 띄우고 접속 확인
valkey-cli -h <valkey-host> -p 6379 PING
# → PONG

# 3. 주요 Command의 호환성을 Smoke Test
valkey-cli -h <valkey-host> SET smoke:test "ok" EX 60
valkey-cli -h <valkey-host> GET smoke:test
valkey-cli -h <valkey-host> TTL smoke:test
```

Application 쪽에서는 접속 대상을 환경 변수로 전환할 수 있게 해두면 Rollback이 쉬워집니다.

```typescript
// 접속 대상을 환경 변수로 제어해 문제가 있으면 즉시 되돌릴 수 있게 한다
import { createClient } from "redis"; // Valkey와도 Protocol 호환

const client = createClient({
  url: process.env.CACHE_URL, // Redis / Valkey 어느 쪽이든 가리키기만 하면 된다
  socket: {
    reconnectStrategy: (retries) => Math.min(retries * 50, 2000),
  },
});

client.on("error", (err) => {
  // Migration 기간에는 Error를 반드시 가시화한다
  console.error("[cache] connection error:", err.message);
});

await client.connect();

// Cache는 Fallback 가능한 설계로 만들어 둔다
async function getCached<T>(
  key: string,
  loader: () => Promise<T>,
  ttlSeconds = 300
): Promise<T> {
  try {
    const hit = await client.get(key);
    if (hit) return JSON.parse(hit) as T;
  } catch (err) {
    // Cache 장애 시 원본 Data 취득으로 넘긴다
    console.warn("[cache] read failed, falling back:", (err as Error).message);
  }

  const value = await loader();

  try {
    await client.set(key, JSON.stringify(value), { EX: ttlSeconds });
  } catch (err) {
    console.warn("[cache] write failed:", (err as Error).message);
  }

  return value;
}
```

이처럼 Cache를 "있으면 빠르고, 없어도 동작하는" 계층으로 설계해 두면 Migration 중 접속이 불안정해져도 본체 기능은 멈추지 않습니다. Migration에 국한하지 않고 Cache 계층은 이 전제로 짜야 합니다.

## Cost만으로 결정하지 않는다

Valkey가 약 33% 저렴하다는 숫자는 매력적이지만, Migration 판단을 Cost만으로 내리는 것은 성급합니다. webhani가 Client에게 권하는 판단 축은 다음 세 가지입니다.

1. **현행이 EOL에 도달했는가** — Security 수정이 멈추는 쪽이 대개 Cost보다 큰 Risk
2. **Module 의존 유무** — 순수 Cache 용도라면 Migration은 쉽고, Module 의존이 있으면 정밀 검토 필요
3. **운영 Team의 숙련도** — Managed Service(ElastiCache 등)를 쓰면 운영 부하는 작지만, 자체 운영이라면 Toolchain 검증이 필요

많은 Web Application의 Cache 용도는 1번과 2번을 통과합니다. 그 경우 Cost 절감과 지원 지속을 모두 얻는 Valkey는 합리적인 선택입니다.

## 정리

Redis 7.2의 EOL은 Cache 계층을 재검토할 좋은 기회입니다. Valkey는 Command 호환을 유지하면서 Patch가 도착하는 계열로서, 그리고 더 저렴한 선택지로서 현실적입니다. 다만 "호환되니 검증 없이 옮길 수 있다"는 뜻은 아니며, 영속화·Replication·Module 의존 세 가지는 Staging에서 반드시 확인해야 합니다. Cache를 Fallback 가능하게 설계하고 접속 대상을 환경 변수로 전환할 수 있게 해두면 Migration Risk는 충분히 관리할 수 있습니다.
