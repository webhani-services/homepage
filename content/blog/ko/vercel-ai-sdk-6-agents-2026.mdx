Vercel AI SDK 6는 v5에서 실험적(experimental)으로 제공되던 기능들을 안정화하고, Agent 개발에 필요한 기반을 정비한 버전입니다. `createAgent()`로 재사용 가능한 Agent 객체를 만들고, Tool 단위로 사람의 승인을 요구하는 흐름을 구현하며, MCP 서버를 직접 연결할 수 있는 Client가 SDK에 내장되었습니다. 각 기능의 사용법과 v5 마이그레이션 시 주의할 Breaking Change를 정리합니다.

## createAgent() — 재사용 가능한 Agent 객체

v5까지는 Agent 로직을 `generateText()`나 `streamText()` 호출마다 작성해야 했습니다. API Route, Background Job, Script 등 여러 곳에서 동일한 Agent를 사용하면 모델 설정, System Prompt, Tool 목록이 중복되었습니다. `createAgent()`는 이를 한 곳에 정의하고 여러 Context에서 재사용하는 방식을 제공합니다.

```typescript
import { createAgent } from "ai";
import { openai } from "@ai-sdk/openai";
import { searchWeb, fetchPage, summarize } from "./tools";

const researchAgent = createAgent({
  model: openai("gpt-4o"),
  system: "You are a research assistant. Retrieve information and summarize findings concisely.",
  tools: {
    searchWeb,
    fetchPage,
    summarize,
  },
  maxSteps: 10,
});

// API Route에서 사용
export async function POST(req: Request) {
  const { topic } = await req.json();
  const result = await researchAgent.generate(`Research this topic: ${topic}`);
  return Response.json({ summary: result.text });
}

// Background Job에서 동일한 Agent 재사용
export async function runDailyDigest() {
  const result = await researchAgent.generate("오늘의 AI 관련 뉴스를 요약해줘");
  await sendDigestEmail(result.text);
}
```

`createAgent()`는 내부적으로 `generateText()`와 동일하게 동작합니다. 핵심 가치는 설정을 한 곳에 모아서, 모델이나 System Prompt가 바뀔 때 모든 호출 지점을 찾아 수정하지 않아도 되는 데 있습니다.

## needsApproval — Tool 단위 승인 흐름 구현

DB 업데이트, 이메일 발송, 배포 실행처럼 외부 시스템에 쓰기 작업을 수행하는 Tool을 Agent가 자율적으로 실행하게 두는 것은 위험할 수 있습니다. SDK 6는 Tool 단위로 `needsApproval: true` 플래그를 설정해 실행을 일시 중단하고 사람의 승인을 기다리는 흐름을 구현할 수 있습니다.

```typescript
import { createAgent, tool } from "ai";
import { openai } from "@ai-sdk/openai";
import { z } from "zod";

const deployAgent = createAgent({
  model: openai("gpt-4o"),
  system: "You manage infrastructure deployments.",
  tools: {
    analyzeChanges: tool({
      description: "PR diff를 분석하고 배포 위험도를 추정합니다",
      parameters: z.object({ prUrl: z.string() }),
      execute: async ({ prUrl }) => analyzeChanges(prUrl),
    }),
    triggerDeploy: tool({
      description: "Production 배포를 실행합니다",
      parameters: z.object({ prUrl: z.string(), environment: z.string() }),
      needsApproval: true, // 이 Tool은 사람의 승인이 필요
      execute: async ({ prUrl, environment }) => triggerDeploy(prUrl, environment),
    }),
  },
});

export async function POST(req: Request) {
  const { pr } = await req.json();
  const result = await deployAgent.generate(`${pr}을 Production에 배포해줘`);

  if (result.status === "requires-approval") {
    // 승인 대기 상태를 DB에 영속화합니다
    await db.pendingApprovals.create({
      agentState: result.state,
      toolCall: result.pendingToolCall,
      requestedAt: new Date(),
    });
    return Response.json({ status: "pending", id: result.state.id });
  }

  return Response.json({ result: result.text });
}
```

승인 후 Agent를 재개할 때는 저장한 `state`를 사용합니다.

```typescript
// 승인 처리 API Route
export async function POST(req: Request) {
  const { approvalId, approved } = await req.json();
  const pending = await db.pendingApprovals.findById(approvalId);

  const resumed = await deployAgent.resume({
    state: pending.agentState,
    approved,
  });

  return Response.json({ result: resumed.text });
}
```

`needsApproval`은 Tool 단위로 적용됩니다. `analyzeChanges`처럼 읽기 전용 Tool은 자유롭게 실행하고, `triggerDeploy`처럼 쓰기 작업이 있는 Tool만 승인 흐름에 태울 수 있습니다. 같은 Agent 안에 두 가지 성격의 Tool을 자연스럽게 공존시킬 수 있습니다.

## createMcpClient() — MCP Server 직접 연결

Model Context Protocol(MCP) Server를 SDK 호출에 연결하려면 기존에는 별도 패키지가 필요했습니다. SDK 6에는 `createMcpClient()`가 내장되어 추가 의존성 없이 MCP Server를 Agent에 연결할 수 있습니다.

```typescript
import { createAgent, createMcpClient } from "ai";
import { openai } from "@ai-sdk/openai";

const githubMcp = createMcpClient({
  transport: {
    type: "stdio",
    command: "npx",
    args: ["-y", "@modelcontextprotocol/server-github"],
    env: { GITHUB_PERSONAL_ACCESS_TOKEN: process.env.GITHUB_TOKEN! },
  },
});

const codeReviewAgent = createAgent({
  model: openai("gpt-4o"),
  system: "You review pull requests and provide specific, actionable feedback.",
  tools: await githubMcp.getTools(), // MCP Server가 제공하는 Tool 목록을 가져옵니다
});

const review = await codeReviewAgent.generate(
  "webhani/homepage 저장소의 PR #123을 리뷰해줘"
);

await githubMcp.close();
```

HTTP 기반 MCP Server도 동일한 방식으로 연결합니다.

```typescript
const remoteMcp = createMcpClient({
  transport: {
    type: "http",
    url: "https://mcp.internal.example.com/api",
    headers: { Authorization: `Bearer ${process.env.MCP_TOKEN}` },
  },
});
```

`getTools()`는 MCP Server의 Tool Manifest를 가져와서 SDK의 Tool 형식으로 자동 변환합니다. GitHub, Slack, 내부 API 등 MCP 호환 Server라면 어떤 것이든 Adapter 코드 없이 Agent에 바로 연결할 수 있습니다.

## OpenTelemetry 트레이싱

멀티 스텝 Agent의 실행 흐름을 가시화하기 위한 OpenTelemetry 연동이 정비되었습니다.

```typescript
// instrumentation.ts (Next.js) 또는 앱 진입점
import { registerOTel } from "@vercel/otel";

registerOTel({ serviceName: "webhani-ai-app" });
```

```typescript
const researchAgent = createAgent({
  model: openai("gpt-4o"),
  tools: { searchWeb, fetchPage, summarize },
  experimental_telemetry: {
    isEnabled: true,
    functionId: "research-agent",
    metadata: {
      version: "1.0",
    },
  },
});

const result = await researchAgent.generate("Vercel 최신 발표 내용을 찾아줘", {
  experimental_telemetry: {
    metadata: { userId: ctx.userId, sessionId: ctx.sessionId },
  },
});
```

각 LLM Request, Tool 호출, Step 전환이 별도의 Span으로 기록됩니다. Jaeger, Honeycomb, Grafana Tempo 등 OTLP 호환 Backend로 전송하면 Agent 실행 경로와 Latency를 Timeline으로 확인할 수 있습니다.

## Per-Tool Strict Mode

JSON Schema Validation 방식을 Tool 단위로 설정할 수 있습니다. `strict: true`는 예상치 못한 Property가 포함된 응답을 거부하고, `strict: false`는 이를 허용합니다.

```typescript
const agent = createAgent({
  model: openai("gpt-4o"),
  tools: {
    createInvoice: tool({
      description: "새 Invoice 레코드를 생성합니다",
      parameters: invoiceSchema,
      strict: true, // 정확한 Schema 준수 필요
      execute: async (params) => createInvoice(params),
    }),
    searchDocuments: tool({
      description: "내부 문서를 검색합니다",
      parameters: searchSchema,
      strict: false, // 추가 Context Field 허용
      execute: async (params) => searchDocuments(params),
    }),
  },
});
```

하나의 Agent 안에서 strict와 lenient를 혼용할 수 있다는 점이 SDK 6의 특징입니다.

## v5에서의 Breaking Change

**1. `experimental_tools` → `tools` 리네임**

```typescript
// v5
const result = await generateText({
  model,
  experimental_tools: { searchWeb },
});

// v6
const result = await generateText({
  model,
  tools: { searchWeb },
});
```

동작 차이는 없고 이름만 변경되었습니다. 코드베이스 전체 치환으로 대응합니다.

**2. Chat SDK 트랜스크립트 형식 변경**

`useChat()`이 반환하는 메시지 객체의 Schema가 변경되었습니다. 채팅 트랜스크립트를 DB에 영속화하고 있다면, 업그레이드 전에 마이그레이션 Script가 필요합니다. 공식 문서에 참고 Script가 제공됩니다.

**3. `LanguageModel` 타입 변경**

`LanguageModel` 타입을 직접 참조하는 코드는 업데이트가 필요합니다.

```typescript
// v6
import type { LanguageModelV2 } from "ai";

function runWithModel(model: LanguageModelV2) {
  return createAgent({ model, tools: {} });
}
```

## 정리

SDK 6의 핵심은 실험적 기능의 안정화와 Agent 개발 패턴의 정비입니다.

- `createAgent()`는 Agent 설정이 여러 파일에 흩어지는 문제를 해결합니다. 새 프로젝트라면 처음부터 사용하는 것을 권장합니다.
- `needsApproval`은 쓰기 작업이 있는 Tool에 먼저 적용하고, 운영 환경에서의 자율 실행 범위를 점진적으로 넓혀가는 접근이 안전합니다.
- MCP Client는 이미 MCP 호환 Service를 사용 중이라면 기존 Adapter 코드를 대체할 수 있습니다.
- v5에서 마이그레이션할 때는 `experimental_tools` 리네임과 Chat SDK 트랜스크립트 형식 변경 두 가지를 우선 확인합니다. 트랜스크립트를 영속화하고 있다면 데이터 마이그레이션 계획을 먼저 세우고 업그레이드를 진행합니다.
